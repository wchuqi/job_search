## 一、场景题和生产实践

### 1. 如何把一个 JavaScript 项目渐进迁移到 TypeScript？

**参考答案：**

先建立类型检查基线，再分阶段迁移。可以从 `allowJs`、`checkJs`、JSDoc 或少量核心模块开始，优先迁移公共 API、数据模型和高频变更代码。开启严格规则时不要一次性全部推进，应按错误类型和目录分批处理。

> **重点：** 迁移成功依赖节奏和门禁，不只是改后缀名。

### 2. 接口返回的数据类型和实际不一致，如何处理？

**参考答案：**

不能只依赖静态类型。应把外部响应视为 `unknown`，用类型保护或 schema 校验。校验失败返回错误结果、抛出明确异常或记录监控。

```ts
type Result<T> =
  | { ok: true; value: T }
  | { ok: false; error: string };
```

> **易错：** 用 `as` 隐藏不可信数据。

### 3. 项目里很多 `any`，你会怎么治理？

**参考答案：**

先分类：外部输入、第三方库缺类型、历史代码、复杂类型暂时无法表达。外部输入改为 `unknown` 并校验；第三方缺类型补声明；历史代码按模块逐步替换；短期保留的 `any` 要加注释和跟踪。

> **重点：** 治理 `any` 要先找来源，再决定替代方案。

### 4. 升级 TypeScript 版本后大量报错，如何排查？

**参考答案：**

先阅读 release notes，确认是否有推断、标准库、模块解析或严格检查变化。然后分离业务错误、第三方声明错误和配置错误。对公共库要验证生成的 `.d.ts` 和下游项目。

> **难点：** 版本升级错误经常不是代码坏了，而是旧代码的隐患被新检查暴露。

### 5. 线上出现“类型检查通过但运行失败”的模块问题，怎么查？

**参考答案：**

先确认运行时报错的真实导入路径，再用 `tsc --traceResolution` 查看编译期解析。检查 `paths`、`exports`、`type`、`module`、`moduleResolution`、打包器 alias 和运行时加载器是否一致。

> **重点：** 类型解析和运行时加载必须分别排查。

### 6. 如何避免复杂类型拖慢项目？

**参考答案：**

减少递归深度、巨大联合和过度条件类型；为复杂类型添加中间命名类型；避免在高频公共类型中做昂贵运算；必要时用显式类型换取可读性和性能。

> **易错：** 认为类型越自动越好。大型项目里可维护性和编译性能同样重要。
