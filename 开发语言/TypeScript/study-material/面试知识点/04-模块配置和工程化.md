## 一、模块、配置和工程化

### 1. `.d.ts` 文件是什么？

**参考答案：**

`.d.ts` 是声明文件，用来描述 JavaScript 模块或全局变量的类型，不包含运行时实现。它让 TypeScript 能理解已有 JS 库的 API。

> **重点：** 声明文件必须和真实运行时 API 一致。
>
> **易错：** 以为写了声明就创造了运行时代码。

### 2. `declare` 的作用是什么？

**参考答案：**

`declare` 告诉 TypeScript 某个变量、函数、模块或类型在其他地方存在。它只影响类型检查，不生成 JavaScript。

```ts
declare const APP_VERSION: string;
```

> **易错：** `declare` 只让编译器相信，不保证运行时存在。

### 3. `target`、`module`、`moduleResolution` 有什么区别？

**参考答案：**

`target` 决定输出 JS 语法级别；`module` 决定输出模块格式；`moduleResolution` 决定编译器如何查找导入模块和类型声明。

> **重点：** 输出格式、类型解析和运行时加载是三件事。

### 4. 为什么配置了 `paths` 后运行时仍可能找不到模块？

**参考答案：**

`paths` 只告诉 TypeScript 编译期如何解析模块，不一定改变编译输出，也不让 Node 自动理解别名。运行时需要打包器、加载器、路径重写或包导出配套。

> **易错：** 编辑器能跳转就以为运行时也能加载。

### 5. CI 中应该如何检查 TypeScript 项目？

**参考答案：**

至少运行 `tsc --noEmit` 做类型检查，运行测试验证行为，运行 Lint 检查风险模式。库项目还应验证声明文件输出和示例项目引用。

```json
{
  "scripts": {
    "typecheck": "tsc --noEmit",
    "test": "vitest run",
    "lint": "eslint ."
  }
}
```

> **重点：** 打包成功不等于完整类型检查成功。
