## 一、基础概念和运行时边界

### 1. TypeScript 和 JavaScript 是什么关系？

**参考答案：**

TypeScript 是 JavaScript 的类型化超集。它在 JavaScript 语法基础上增加类型语法、类型检查和更强的编辑器能力，最终通常编译成 JavaScript 运行。运行时执行的是 JavaScript，类型信息大多会被擦除。

> **重点：** TypeScript 增强开发期，不替代 JavaScript 运行时。
>
> **易错：** 以为 TypeScript 是独立运行时语言。

### 2. 什么是类型擦除？它有什么工程影响？

**参考答案：**

类型擦除指 TypeScript 类型标注在编译输出中通常不存在。例如 `function add(a: number)` 编译后参数上没有 `number`。因此 TS 能在编译期发现调用错误，但不能在运行时自动校验 JSON、表单、环境变量和数据库数据。

工程影响是：外部输入必须做运行时校验，不能只写 `as SomeType`。

> **重点：** 编译期类型安全不等于运行时数据可信。

### 3. `any` 和 `unknown` 有什么区别？

**参考答案：**

`any` 会绕过类型检查，任何操作都被允许，并且会污染后续表达式。`unknown` 表示未知类型，使用前必须通过 `typeof`、`instanceof`、自定义类型保护或 schema 校验收窄。

```ts
function handle(value: unknown) {
  if (typeof value === "string") {
    return value.toUpperCase();
  }
  return "";
}
```

> **重点：** 外部输入优先用 `unknown`。
>
> **易错：** 用 `any` 快速消除报错，导致类型系统失效。

### 4. `void` 和 `never` 有什么区别？

**参考答案：**

`void` 表示函数没有有意义的返回值；`never` 表示函数不会正常返回，或某个分支不可能出现。`never` 常用于穷尽检查。

```ts
function fail(message: string): never {
  throw new Error(message);
}
```

> **难点：** `never` 是“没有可能的值”，不是“没有返回值”。

### 5. TypeScript 能否保证接口返回值一定符合类型？

**参考答案：**

不能。接口返回值来自运行时外部系统，TypeScript 无法自动验证它。正确做法是把响应先作为 `unknown`，通过类型保护或 schema 校验后再进入业务代码。

> **易错：** `await res.json() as User` 只是断言，不是校验。
