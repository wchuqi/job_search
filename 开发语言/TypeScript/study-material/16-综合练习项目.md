# TypeScript学习资料：综合练习项目

[返回索引](../TypeScript学习资料.md)

## 项目目标

构建一个类型安全的 API client 练习项目，覆盖类型建模、运行时校验、异步错误处理、模块导出、测试和构建发布。

## 项目结构

```text
typed-api-client/
  package.json
  tsconfig.json
  src/
    index.ts

    http.ts
  tests/
    user.test.ts
```

## 任务 1：定义结果类型

```ts
export type Result<T, E = string> =
  | { ok: true; value: T }
  | { ok: false; error: E };
```

## 任务 2：定义业务模型和运行时校验

```ts
export type User = {
  id: string;
  name: string;
  email?: string;
};

export function isUser(value: unknown): value is User {
  return (
    typeof value === "object" &&
    value !== null &&
    "id" in value &&
    "name" in value
  );
}
```

## 任务 3：实现 API client

```ts
export async function getJson<T>(
  url: string,
  validate: (value: unknown) => value is T
): Promise<Result<T>> {
  const response = await fetch(url);
  const payload: unknown = await response.json();

  if (!validate(payload)) {
    return { ok: false, error: "invalid payload" };
  }

  return { ok: true, value: payload };
}
```

## 任务 4：测试样本

至少覆盖：

- 正常用户对象。
- 缺少 `id`。
- `name` 类型错误。
- API 返回非对象。
- fetch 抛错或网络失败。
- 成功和失败结果分支的消费代码。

## 进阶任务

1. 用 Zod 或其他 schema 库替换手写类型保护。
2. 增加 `ApiError` 类型，区分网络错误、HTTP 错误和校验错误。
3. 为 `getJson` 增加超时和取消。
4. 加入类型测试，确保 `Result<T>` 能正确收窄。
5. 发布为内部 npm 包并验证 `.d.ts`。

## 验收

- API 返回值不会直接被断言为业务类型。
- 成功/失败通过判别联合清晰表达。
- 类型检查、测试和构建都能在 CI 中运行。
- 对外导出有声明文件。
- 能说明哪些问题由 TS 处理，哪些必须运行时处理。

## 重点

- 综合项目的核心不是“写复杂类型”，而是把类型、校验、错误处理和工程流程连起来。

## 难点

- 运行时校验和静态类型要保持同步，否则会出现“类型说可以，校验说不行”或反过来的分裂。

## 易错

> **易错：** `await res.json() as User` 后直接使用。
>
> 正确做法：先校验，再返回 `Result<User>` 或抛出明确错误。
