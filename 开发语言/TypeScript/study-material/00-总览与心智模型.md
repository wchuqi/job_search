# TypeScript学习资料：总览与心智模型

[返回索引](../TypeScript学习资料.md)

## 学习目标

- 理解 TypeScript 的定位：为 JavaScript 增加静态类型、编辑器能力和重构安全。
- 建立“类型系统是编译期约束，运行时仍是 JavaScript”的心智模型。
- 区分类型检查、编译转换、运行时校验和构建打包。
- 知道 TypeScript 的核心收益和边界。

## 理论导读

TypeScript 不是一门完全脱离 JavaScript 的新语言。它在 JavaScript 语法基础上增加类型语法和类型检查，然后编译为 JavaScript 运行。类型信息通常会在编译后被擦除，因此 TypeScript 能帮助开发者在提交代码前发现大量契约错误，但不能自动验证网络请求、用户输入、数据库数据或 JSON 配置在运行时一定符合类型。

TypeScript 的核心价值在团队协作中更明显：函数签名成为可检查的契约，编辑器能自动补全和重构，复杂数据结构可以被显式表达，升级依赖时能更早发现破坏性变更。

## 核心心智模型

把 TypeScript 看成一套“开发期协议检查器”：

```text
源代码 + 类型声明 + tsconfig
  -> 类型检查器验证契约
  -> 编译器输出 JavaScript
  -> JavaScript 引擎执行
```

类型检查器不会替你执行业务逻辑，也不会替你检查外部输入。凡是来自网络、文件、环境变量、用户输入、第三方库的值，都要在运行时边界做校验。

## 知识点详解

### TypeScript 解决的问题

| 问题 | TypeScript 的帮助 |
| --- | --- |
| 函数参数传错 | 用函数签名提前报错 |
| 对象字段拼错 | 用对象类型和接口约束 |
| 重构风险大 | 通过类型引用和编辑器重构发现影响面 |
| API 契约不清 | 用类型别名、接口、泛型表达数据结构 |
| 复杂状态难维护 | 用判别联合表达有限状态机 |

### TypeScript 不解决的问题

- 不自动保证 API 返回的数据真实符合类型。
- 不自动避免所有 `undefined`、空字符串、非法日期、越权数据。
- 不替代单元测试、集成测试和运行时校验。
- 不替代打包器、运行时模块加载器和部署配置。

### 类型擦除

TypeScript 代码：

```ts
function add(a: number, b: number): number {
  return a + b;
}
```

编译后的 JavaScript 中通常没有 `number`：

```js
function add(a, b) {
  return a + b;
}
```

这意味着运行时传入字符串仍可能发生，除非你在入口处做校验。

## 例子

```ts
type User = {
  id: string;
  name: string;
};

function greet(user: User): string {
  return `hello ${user.name}`;
}

greet({ id: "u1", name: "Alice" });
```

这个例子能防止调用方漏传 `name`，但不能证明后端返回的 JSON 一定有 `name`。

## 练习

1. 写一个 `User` 类型和 `formatUser(user)` 函数。
2. 把一个来自 `JSON.parse` 的值标为 `unknown`，再手动校验字段。
3. 解释为什么 `as User` 不等于运行时校验。

## 验收

- 能说明 TypeScript 和 JavaScript 的关系。
- 能解释类型擦除。
- 能区分静态类型检查和运行时校验。
- 能说出 TypeScript 在团队项目中的价值。

## 重点

- TypeScript 是 JavaScript 的静态类型层。
- 类型主要在编译期生效，运行时仍要面对真实数据。
- 好的 TS 代码会把“不确定性”限制在边界，把内部业务代码变成类型明确的世界。

## 难点

- 很多线上问题不是类型系统没能力，而是开发者把运行时不可信数据直接断言成可信类型。

## 易错

> **易错：** 认为 `const data = await res.json() as User` 就完成了校验。
>
> 正确做法：把外部数据先当成 `unknown`，在运行时检查结构，再转成内部可信类型。
