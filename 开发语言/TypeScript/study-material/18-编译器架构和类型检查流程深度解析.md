# TypeScript学习资料：编译器架构和类型检查流程深度解析

[返回索引](../TypeScript学习资料.md)

## 学习目标

- 理解 TypeScript 编译器从源码到诊断和输出的大致流水线。
- 掌握 scanner、parser、binder、checker、emitter 的职责边界。
- 能解释为什么有些错误是语法错误，有些是类型错误，有些只在模块解析阶段出现。
- 能从编译流程角度排查“编辑器报错但命令行不报错”或反过来的问题。

## 理论导读

TypeScript 编译器不是简单地“读一遍文件然后输出 JS”。它需要先把文本转成 token，再解析成 AST，建立符号表，做类型检查，最后根据配置输出 JavaScript 和声明文件。理解这条流水线能帮助你区分问题发生在哪一层。

工程中常见的困惑包括：为什么语法错误不依赖类型也会报；为什么有些导入路径错误不是运行时报，而是类型检查阶段报；为什么编辑器和 CI 的诊断不同；为什么 `skipLibCheck` 影响的是声明文件检查而不是你的业务代码类型正确性。

## 核心心智模型

```text
Source text
  -> Scanner: 文本切 token
  -> Parser: token 组 AST
  -> Binder: 建符号和作用域
  -> Checker: 推断、兼容性、控制流、诊断
  -> Emitter: 输出 JS / .d.ts / source map
```

每一层只解决一类问题。语法不合法时，不会等到类型检查；模块找不到时，checker 无法完整理解导入符号；类型正确也不意味着 emitter 输出能在目标运行时正确加载。

## 知识点详解

### Scanner：把文本变成 token

Scanner 负责识别关键字、标识符、字符串、数字、标点、注释等。它不理解业务语义，只做词法层处理。

例子：

```ts
const age: number = 20;
```

会被切成类似 `const`、`age`、`:`、`number`、`=`、`20`、`;` 的 token。

### Parser：构造 AST

Parser 根据语法规则把 token 组成语法树。AST 描述代码结构，例如变量声明、函数声明、类型引用、导入语句。

语法错误如少括号、非法 token、类型参数位置不合法，通常在这一层暴露。

### Binder：建立符号表

Binder 会遍历 AST，建立作用域和符号关系。它知道一个变量、函数、类、接口、模块导出在符号表里的位置。

如果你在同一作用域重复声明不允许重复的变量，或者模块导出冲突，binder/checker 阶段会给出诊断。

### Checker：类型系统核心

Checker 做最多复杂工作：

- 类型推断。
- 泛型推断。
- 结构化兼容性判断。
- 联合和交叉类型归约。
- 控制流收窄。
- 条件类型求值。
- 重载解析。
- 访问修饰符和可见性检查。
- 模块导入导出类型检查。

大多数“为什么这个类型不能赋值”的问题都在 checker 层。

### Emitter：输出阶段

Emitter 根据 `target`、`module`、`declaration` 等配置输出 JavaScript、`.d.ts`、source map。注意：emitter 不负责保证运行时依赖一定存在。

## 例子

```ts
type User = { id: string };

function print(user: User) {
  console.log(user.name);
}
```

排查流程：

1. Scanner 能识别 token。
2. Parser 能生成合法 AST。
3. Binder 能找到 `User` 和 `print`。
4. Checker 发现 `User` 上没有 `name`。
5. Emitter 通常不会在类型错误时继续输出，取决于配置和命令。

## 工程场景

### 编辑器报错但命令行不报错

常见原因：

- 编辑器打开了不同的 workspace 根目录。
- 编辑器使用了内置 TypeScript 版本，命令行使用项目本地版本。
- 编辑器加载了不同的 `tsconfig.json`。
- 文件不在命令行 `include` 范围里，但编辑器单文件分析仍提示。

排查：

```powershell
npx tsc --showConfig
npx tsc --noEmit
```

并在编辑器里确认 TypeScript SDK 版本。

### 命令行报错但编辑器不报错

常见原因：

- 编辑器尚未刷新 tsserver。
- 项目引用或生成声明文件未构建。
- CI 使用更严格的 tsconfig。
- 依赖安装版本不一致。

## 练习

1. 写一个语法错误、一个模块解析错误、一个类型错误，分别判断发生在哪一层。
2. 用 `npx tsc --showConfig` 查看实际配置。
3. 故意切换 TypeScript 版本，观察诊断变化。
4. 开启 `declaration`，观察 emitter 输出 `.d.ts`。

## 验收

- 能画出 TypeScript 编译器主要阶段。
- 能区分语法诊断、模块解析诊断和类型诊断。
- 能解释 tsserver 和 `tsc` 为什么可能诊断不同。
- 能从编译流程定位问题层级。

## 重点

- Checker 是类型系统核心，但不是编译器全部。
- `tsconfig` 决定项目边界，边界不同诊断就可能不同。
- 类型检查通过只说明编译期契约成立，不说明运行时加载和数据一定正确。

## 难点

- TypeScript 项目中的问题经常跨越 checker、module resolution、emitter 和 runtime，需要分层排查。

## 易错

> **易错：** 看到 TypeScript 报错就只盯着类型本身。
>
> 正确做法：先判断报错发生在语法、绑定、模块解析、类型检查还是输出/运行时阶段。
