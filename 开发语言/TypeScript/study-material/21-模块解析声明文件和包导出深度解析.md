# TypeScript学习资料：模块解析、声明文件和包导出深度解析

[返回索引](../TypeScript学习资料.md)

## 学习目标

- 深入理解 TypeScript 的模块解析决策。
- 掌握 `package.json` 中 `type`、`exports`、`types` 对类型和运行时的影响。
- 能排查 ESM/CJS 互操作、默认导入、声明入口错误。
- 能设计可被下游稳定消费的包导出结构。

## 理论导读

模块问题是 TypeScript 工程里最容易跨层的故障。TypeScript 需要解析类型，Node 或浏览器需要加载运行时代码，打包器可能还会改写导入。一个包能在编辑器里跳转，不代表 Node 能加载；一个包运行没问题，不代表声明文件描述正确。

## 核心心智模型

```text
导入语句
  -> TS moduleResolution 找类型入口
  -> package.json exports/types/typesVersions 参与选择
  -> 编译输出保留或转换导入
  -> Node/打包器按运行时规则加载
```

## 知识点详解

### 相对路径和包名路径

```ts
import { a } from "./a";
import { z } from "zod";
```

相对路径通常从当前文件出发。包名路径会进入 `node_modules`，读取包的 `package.json`，再结合解析策略寻找入口。

### `type` 字段

```json
{
  "type": "module"
}
```

在 Node 中，它影响 `.js` 文件按 ESM 还是 CJS 解释。TypeScript 的 `NodeNext` 模式会参考这个字段。

### `exports` 字段

```json
{
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    },
    "./utils": {
      "types": "./dist/utils.d.ts",
      "import": "./dist/utils.js"
    }
  }
}
```

`exports` 限制和定义外部可导入路径。没有暴露的子路径，下游不应依赖。

### `types` 字段

```json
{
  "types": "./dist/index.d.ts"
}
```

它指定主类型入口。对于复杂 `exports`，最好在每个导出条件中提供 `types`。

### 默认导入陷阱

CJS 包和 ESM 默认导入的行为受 `esModuleInterop`、`allowSyntheticDefaultImports`、打包器和运行时共同影响。类型能写 `import foo from "foo"`，不代表运行时一定有默认导出。

## 例子：类型入口和运行时入口不一致

```json
{
  "main": "./dist/index.cjs",
  "types": "./dist/index.d.ts"
}
```

如果 `.d.ts` 声明导出了 `foo`，但 CJS 运行时代码没有导出 `foo`，下游会出现类型通过、运行失败。

## 排障步骤

1. 看报错发生在类型检查、构建还是运行时。
2. 用 `npx tsc --traceResolution` 查看 TS 找到的文件。
3. 查看包的 `package.json`：`type`、`exports`、`main`、`module`、`types`。
4. 查看编译输出导入语句是否被转换。
5. 用 Node 实际 import/require 验证运行时入口。
6. 检查打包器 alias 和 tsconfig `paths` 是否一致。

## 练习

1. 写一个同时支持 ESM/CJS 的包导出配置。
2. 故意让 `.d.ts` 和运行时导出不一致，观察下游错误。
3. 用 `--traceResolution` 分析一个第三方包。
4. 比较 `esModuleInterop` 开关对导入写法的影响。

## 验收

- 能解释 TypeScript 和 Node 模块解析的差异。
- 能读懂 `exports` 中的 `types/import/require`。
- 能排查默认导入和 CJS 互操作问题。
- 能设计稳定的包类型入口。

## 重点

- 声明文件是契约，必须和运行时代码一致。
- `exports` 会限制子路径导入。
- `paths`、打包器 alias、Node 加载规则必须统一。

## 难点

- 同一个包在 Node、bundler、test runner、tsserver 下可能走不同解析路径。

## 易错

> **易错：** 下游能从 `package/dist/internal` 导入就认为这是稳定 API。
>
> 正确做法：只依赖 `exports` 明确暴露的入口；内部路径不作为兼容承诺。
