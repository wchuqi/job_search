# TypeScript学习资料：深度实验手册和能力验收

[返回索引](../TypeScript学习资料.md)

## 学习目标

- 通过实验验证 TypeScript 深层机制。
- 把类型系统、模块解析、运行时边界和工程治理串起来。
- 建立可展示的作品和面试讲解素材。
- 用明确验收标准判断是否达到中高级水平。

## 实验 1：类型擦除和运行时边界

任务：

1. 写一个 `User` 类型。
2. 用 `JSON.parse` 解析错误数据。
3. 用 `as User` 直接使用，观察运行时错误。
4. 改成 `unknown + isUser`。

验收：

- 能解释为什么类型检查没发现 JSON 问题。
- 能说明 `as` 和校验的区别。

## 实验 2：判别联合和穷尽检查

任务：

1. 定义 `RequestState<T>`。
2. 写 `renderState`。
3. 新增一种状态但不修改 `renderState`。
4. 用 `assertNever` 让编译器报错。

验收：

- 能证明判别联合能防止漏处理状态。

## 实验 3：泛型推断和条件类型分发

任务：

1. 实现 `get<T, K extends keyof T>`。
2. 实现 `ToArray<T>` 和 `ToArrayNoDistribute<T>`。
3. 实现 `MyReturnType<T>`。
4. 写类型测试样例。

验收：

- 能解释每个类型工具的输入输出。
- 能说明分发和不分发的区别。

## 实验 4：模块解析排障

任务：

1. 配置 `paths` 别名。
2. 让 `tsc` 类型检查通过。
3. 直接用 Node 运行制造失败。
4. 配置打包器或运行时别名修复。
5. 使用 `--traceResolution` 记录 TS 解析路径。

验收：

- 能说清编译期解析和运行时加载的差异。

## 实验 5：声明文件发布

任务：

1. 写一个小工具库。
2. 开启 `declaration`。
3. 配置 `exports.types`。
4. 新建下游项目引用它。
5. 故意让声明和运行时不一致，观察问题。

验收：

- 能设计稳定的类型入口。
- 能解释 `.d.ts` 和运行时代码的一致性要求。

## 实验 6：strict 迁移

任务：

1. 准备一段包含隐式 any、空值、索引访问风险的代码。
2. 逐步开启 `noImplicitAny`、`strictNullChecks`、`noUncheckedIndexedAccess`。
3. 记录每类错误和修复策略。

验收：

- 能制定真实项目 strict 迁移计划。

## 实验 7：类型性能观察

任务：

1. 写一个复杂递归类型。
2. 用 `--extendedDiagnostics` 记录耗时。
3. 简化类型或增加中间类型别名。
4. 对比前后诊断数据。

验收：

- 能用数据说明类型复杂度影响。

## 最终能力验收

| 能力 | 验收标准 |
| --- | --- |
| 基础机制 | 能解释类型擦除、结构化类型、控制流收窄 |
| 类型建模 | 能用判别联合和泛型表达业务状态和 API 契约 |
| 类型运算 | 能读写条件类型、映射类型、`infer`、模板字面量类型 |
| 工程配置 | 能排查 tsconfig、模块解析、路径别名、声明文件问题 |
| 运行时安全 | 能为外部输入设计 schema 或类型保护 |
| 大型项目 | 能制定 strict 迁移、版本升级和类型性能治理方案 |
| 面试表达 | 能用机制、例子、风险和工程方案回答问题 |

## 重点

- 深度学习 TypeScript 必须通过实验验证，不应只背工具类型。
- 能排障和能设计迁移方案，是中高级能力分水岭。

## 难点

- TypeScript 的难点常在“编译期类型”和“运行时 JavaScript”之间的边界，而不是语法本身。

## 易错

> **易错：** 学完高级类型后忽略运行时校验、模块解析和团队治理。
>
> 正确做法：把类型系统能力放回真实项目上下文中，用实验和项目验证。
