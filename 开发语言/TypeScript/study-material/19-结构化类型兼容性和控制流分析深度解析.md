# TypeScript学习资料：结构化类型兼容性和控制流分析深度解析

## 学习目标

- 深入理解结构化类型兼容性的判断方式。
- 掌握新鲜对象字面量、多余属性检查、函数参数兼容和方差风险。
- 理解控制流分析如何改变变量类型。
- 能设计可收窄、可穷尽、可维护的业务状态类型。

## 理论导读

TypeScript 的类型系统为了贴合 JavaScript，采用结构化兼容。它不会要求你显式 implements 某个接口，而是检查形状是否满足。这个设计很灵活，但也让很多人误以为 TS 是“宽松不可靠”。实际上，它有一套兼容性规则：对象属性要满足、函数参数和返回值要满足、私有/受保护成员会影响类兼容、新鲜对象字面量会触发额外检查。

控制流分析则是 TypeScript 类型体验的关键。编译器会根据分支、返回、赋值、异常、类型保护来更新变量在不同位置的类型。

## 核心心智模型

```text
结构化兼容：这个值至少具备目标类型要求的结构吗？
控制流分析：走到这一行时，编译器已经知道哪些条件成立？
```

## 知识点详解

### 对象兼容：多可以赋给少

```ts
type HasName = { name: string };

const full = { id: "u1", name: "Alice" };
const named: HasName = full;
```

`full` 有额外字段，但它满足 `HasName` 需要的字段，因此兼容。

### 新鲜对象字面量

```ts
function print(value: { name: string }) {}

print({ name: "Alice", nmae: "typo" }); // 报错，可能是拼写错误

const value = { name: "Alice", nmae: "typo" };
print(value); // 通常允许
```

这是 TypeScript 在对象字面量上加的一层防错机制，不等同于名义类型。

### 函数兼容

函数兼容要看参数和返回值。返回值一般要求实际返回类型能赋给期望返回类型；参数位置更复杂，涉及严格函数类型、回调位置和历史兼容。

```ts
type Handler = (value: string) => void;

const h: Handler = (value) => {
  console.log(value.toUpperCase());
};
```

### 控制流收窄

```ts
function format(value: string | number) {
  if (typeof value === "string") {
    return value.toUpperCase();
  }
  return value.toFixed(2);
}
```

进入 `if` 分支后，`value` 被收窄为 `string`；离开后，根据控制流剩余可能性变成 `number`。

### 自定义类型保护

```ts
type User = { id: string; name: string };

function isUser(value: unknown): value is User {
  return typeof value === "object" && value !== null && "id" in value && "name" in value;
}
```

类型保护必须和运行时检查一致，否则会制造虚假安全。

### 判别字段和穷尽检查

```ts
type Event =
  | { type: "created"; id: string }
  | { type: "deleted"; id: string; reason: string };

function handle(event: Event) {
  switch (event.type) {
    case "created":
      return event.id;
    case "deleted":
      return event.reason;
  }
}
```

判别字段让控制流分析更稳定，比依赖字段存在性更适合长期维护。

## 例子：把脆弱状态改成判别联合

脆弱写法：

```ts
type RequestState<T> = {
  loading?: boolean;
  data?: T;
  error?: string;
};
```

它允许 `{ loading: true, data: user, error: "failed" }` 这种矛盾状态。

更好的写法：

```ts
type RequestState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "failed"; error: string };
```

## 验收

- 能解释结构化兼容和多余属性检查的区别。
- 能说明控制流分析根据哪些语句收窄。
- 能设计判别联合状态。
- 能指出错误类型保护带来的风险。

## 重点

- 结构化类型不是“随便兼容”，而是按形状和上下文规则兼容。
- 控制流分析依赖真实代码路径。
- 类型保护必须诚实，否则比 `as` 更隐蔽。

## 难点

- 函数参数兼容和方差问题在回调、事件系统、泛型容器中容易引发中高级面试题。

## 易错

> **易错：** 类型保护只检查 `typeof value === "object"` 就返回 `value is User`。
>
> 正确做法：至少检查对象非空、关键字段存在和字段类型；复杂结构用 schema 工具更稳。
