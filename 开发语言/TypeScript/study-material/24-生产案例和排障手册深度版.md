# TypeScript学习资料：生产案例和排障手册深度版

[返回索引](../TypeScript学习资料.md)

## 学习目标

- 能用排障剧本定位 TypeScript 工程问题。
- 掌握类型错误、模块错误、声明错误、运行时边界错误的处理路径。
- 能把生产事故转化为类型和流程改进。
- 能准备中高级面试中的工程场景题。

## 理论导读

TypeScript 生产问题往往不是“语法不会”，而是配置、依赖、运行时、构建和类型声明的交界处出了问题。排障时应先分层：错误发生在编辑器、类型检查、构建、测试、运行时还是线上数据边界。不同层的工具和证据不同。

## 排障总流程

```text
1. 确认错误层级
2. 收集最小复现
3. 查看实际 tsconfig
4. 查看 TypeScript 和依赖版本
5. 分析模块解析或类型推断
6. 修复并补测试/类型测试/CI 门禁
```

## 案例 1：类型通过但运行时 Cannot find module

现象：

```text
Error: Cannot find module '@/user'
```

常见原因：

- `tsconfig.paths` 只配置了编译期别名。
- Node 运行时没有别名解析。
- 测试框架没有同步 alias。
- 打包输出没有重写路径。

排查：

```powershell
npx tsc --traceResolution
```

修复：

- 配置打包器 alias。
- 配置测试框架 alias。
- 使用相对路径或包导出。
- 运行前通过工具重写路径。

## 案例 2：第三方库类型升级导致大量报错

处理：

1. 锁定升级前后版本。
2. 查看 changelog 和声明文件 diff。
3. 判断是业务调用不符合新类型，还是库声明回归。
4. 优先修复真实不兼容。
5. 临时 workaround 要集中封装，不要散落在业务代码。

## 案例 3：API 返回数据变更导致线上错误

根因：

- 客户端用 `as User` 信任响应。
- 后端字段从 `name` 改为 `displayName`。
- 编译期类型没有覆盖真实响应。

改进：

- API 边界 schema 校验。
- 契约测试或 OpenAPI 生成。
- 错误结果结构化。
- 监控校验失败率。

## 案例 4：类型检查越来越慢

常见原因：

- 单个 tsconfig 包含整个 monorepo。
- 复杂递归类型在公共路径中频繁实例化。
- 生成文件或 dist 被 include。
- 依赖声明冲突导致重复检查。

处理：

```powershell
npx tsc --extendedDiagnostics
npx tsc --build --verbose
```

修复方向：

- 收窄 `include`。
- 排除生成目录。
- 拆项目引用。
- 简化复杂类型。
- 对公共类型增加中间别名。

## 案例 5：`@ts-ignore` 越来越多

风险：

- 真错误被永久掩盖。
- 代码升级后注释仍然存在。
- 团队不知道为什么忽略。

治理：

- 改用 `@ts-expect-error`。
- 要求说明原因。
- CI 检查新增数量。
- 定期清理。

## 练习

1. 构造一个路径别名运行时失败的最小项目，并修复。
2. 用 `--traceResolution` 分析一个导入问题。
3. 设计 API payload 校验失败的监控字段。
4. 对一个复杂类型工具做性能简化。
5. 写一个 `@ts-expect-error` 治理规范。

## 验收

- 能按层级定位 TS 工程问题。
- 能用命令收集诊断证据。
- 能把事故修复转成长期工程规则。
- 能回答中高级 TypeScript 场景题。

## 重点

- 类型问题要分层排查，不要只盯报错文本。
- 运行时问题不能靠类型断言修复。
- 生产事故应沉淀为边界校验、测试和 CI 规则。

## 难点

- 同一个错误可能同时涉及 TypeScript、Node、打包器、测试框架和包声明。

## 易错

> **易错：** 用 `as any` 或 `@ts-ignore` 快速压掉生产问题。
>
> 正确做法：先止血，再定位边界和契约问题，最后补校验、测试和门禁。
