## 必须掌握

- TypeScript 类型会擦除，运行时仍是 JavaScript。
- `any` 会关闭检查，`unknown` 强制使用前收窄。
- 结构化类型系统按形状兼容，不按声明名称兼容。
- 判别联合适合表达有限状态和互斥数据。
- 泛型表达类型关系，不是随便绕过类型检查。
- `keyof`、`typeof`、索引访问能减少重复类型定义。
- 条件类型和映射类型是工具类型的基础。
- `.d.ts` 只描述类型，不提供运行时代码。
- `paths` 只解决编译期解析，不解决运行时加载。
- 外部输入必须做运行时校验。
- TypeScript 编译器流水线包括 scanner、parser、binder、checker、emitter。
- 条件类型分发发生在裸类型参数遇到联合类型时。
- 模块问题要分别检查 TypeScript 解析、编译输出和运行时加载。
- 类型性能问题可能来自深层递归类型、巨大联合、过宽 include 和项目边界不清。

## 进阶和容易遗漏

- 多余属性检查只对新鲜对象字面量更严格。
- 可选属性和 `undefined` 在 `exactOptionalPropertyTypes` 下语义更精细。
- `noUncheckedIndexedAccess` 会让索引访问返回可能为 `undefined`。
- 条件类型对联合类型的分发行为。
- `infer` 的使用位置限制。
- `as const` 对字面量推断的影响。
- `satisfies` 适合校验形状又保留字面量推断。
- ESM/CJS 互操作和 `package.json` 的 `type`、`exports`、`types`。
- `skipLibCheck` 不是解决依赖类型冲突的根治方案。
- 大型项目需要控制类型复杂度和编译性能。
- tsserver 和命令行 `tsc` 可能因为 TypeScript 版本、workspace 根、tsconfig 边界不同而诊断不同。
- `declare` 只提供类型承诺，不生成运行时代码。
- `.d.ts` 与运行时代码不一致会导致“类型通过、运行失败”。
- Schema 的价值不只是校验，还包括解析、转换和错误报告。

## 常见错误清单

| 错误 | 后果 | 正确做法 |
| --- | --- | --- |
| 到处写 `as any` | 类型系统失效 | 用收窄、泛型或运行时校验 |
| 把 API JSON 直接断言成类型 | 运行时脏数据进入业务层 | 外部输入先作为 `unknown` |
| 用大量可选字段表达状态 | 非法状态可表示 | 使用判别联合 |
| 配置 `paths` 后不处理运行时别名 | 编译通过但运行失败 | 打包器/加载器/发布路径同步配置 |
| 复杂类型工具无测试 | 重构退化成 `any` 不易发现 | 加类型测试和示例 |
| 一次性 strict 全开改老项目 | 迁移失控 | 分阶段开启严格选项 |
| 类型保护内部只写断言 | 制造虚假安全 | 基于真实运行时检查 |
| 发布包时声明入口和运行时入口不一致 | 下游类型通过但运行失败 | 验证 `.d.ts` 和产物导出 |
| 复杂类型工具无边界 | 编辑器和 CI 变慢 | 简化类型、加中间别名和类型测试 |
