# TypeScript学习资料：编译模式、模块解析和配置层

[返回索引](../TypeScript学习资料.md)

## 学习目标

- 掌握 `target`、`lib`、`module`、`moduleResolution`、`baseUrl`、`paths`。
- 理解 TypeScript 如何查找模块和类型声明。
- 能区分编译期路径解析和运行时模块加载。
- 能处理 monorepo、项目引用和增量构建。

## 理论导读

模块解析是 TypeScript 工程排障中最高频的问题之一。编译器要回答两个问题：导入的模块在哪里，模块的类型是什么。运行时又要回答另一个问题：JavaScript 引擎或加载器能不能找到对应文件。两套规则相似但不等价。

## 核心心智模型

```text
import "@/user"
  -> TS 根据 paths 找到源码或声明
  -> 编译输出可能仍保留 "@/user"
  -> Node 如果不知道 @ 别名就会运行失败
```

## 知识点详解

### `target` 和 `lib`

- `target` 决定输出 JavaScript 语法级别。
- `lib` 决定类型环境里有哪些内置 API，例如 DOM、ES2022。

### `module` 和 `moduleResolution`

- `module` 决定输出模块格式。
- `moduleResolution` 决定编译器如何解析导入路径。

Node 现代项目常见组合是 `NodeNext`，但具体选择要和运行环境、包类型、打包器配套。

### `baseUrl` 和 `paths`

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"]
    }
  }
}
```

这只告诉 TypeScript 如何解析类型，不保证运行时理解别名。

### 项目引用

```json
{
  "references": [
    { "path": "../shared" }
  ]
}
```

项目引用适合大型仓库拆分构建边界，提高增量构建速度。

## 例子

```powershell
npx tsc --showConfig
npx tsc --traceResolution
npx tsc --build
```

`--traceResolution` 是排查模块解析的关键工具。

## 练习

1. 配置一个 `@/*` 别名，并让运行时也能识别。
2. 使用 `--traceResolution` 查看一个包的类型入口。
3. 建立两个项目引用：`shared` 和 `app`。
4. 比较 `moduleResolution: bundler` 和 `NodeNext` 的适用场景。

## 验收

- 能解释 `target`、`lib`、`module` 的区别。
- 能说明 TypeScript 模块解析流程。
- 能定位路径别名运行失败问题。
- 能理解项目引用的价值。

## 重点

- 编译期能解析不等于运行期能加载。
- `--traceResolution` 是模块解析排障入口。
- 配置应服务真实运行环境，而不是复制模板。

## 难点

- ESM/CJS、`exports`、`typesVersions`、路径别名和打包器规则叠加时，问题往往不在单个配置项。

## 易错

> **易错：** 为了消除类型错误随意打开 `skipLibCheck`，却忽略依赖类型不兼容。
>
> 正确做法：`skipLibCheck` 可用于加速和规避第三方噪声，但核心依赖的类型冲突仍要定位根因。
