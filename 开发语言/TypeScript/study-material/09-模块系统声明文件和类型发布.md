# TypeScript学习资料：模块系统、声明文件和类型发布

[返回索引](../TypeScript学习资料.md)

## 学习目标

- 理解 TypeScript 中 ESM、CJS 和类型声明的关系。
- 掌握 `.d.ts`、`declare`、全局声明和模块声明。
- 能发布带类型的包。
- 能排查第三方库缺类型、类型错误或导入方式不兼容的问题。

## 理论导读

TypeScript 编译器既要理解源代码模块，也要理解类型声明。对于 TS 源码，编译器可以直接分析类型；对于 JS 库，则依赖 `.d.ts` 声明文件描述它的公共 API。声明文件是“类型世界中的头文件”，不提供运行时代码。

模块系统问题往往同时涉及三层：源码导入写法、编译输出格式、运行时加载规则。ESM 和 CJS 的差异会影响默认导入、命名导入、包导出字段和 Node 解析。

## 核心心智模型

```text
import 语句
  -> TypeScript 按 moduleResolution 找源码或 .d.ts
  -> 编译器按 module 输出 JS
  -> 运行时按 Node/浏览器/打包器规则加载
```

任意一层不一致，都可能出现“类型能过但运行失败”。

## 知识点详解

### 声明文件

```ts
// math.d.ts
export function add(a: number, b: number): number;
```

声明文件只描述类型，不实现逻辑。

### `declare`

```ts
declare const APP_VERSION: string;
```

`declare` 告诉编译器某个值在运行时会存在，但它不生成代码。

### 模块声明

```ts
declare module "*.css" {
  const classes: Record<string, string>;
  export default classes;
}
```

### 类型发布

包发布时常见字段：

```json
{
  "main": "./dist/index.cjs",
  "module": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    }
  }
}
```

## 例子

```ts
// src/index.ts
export type User = {
  id: string;
  name: string;
};

export function formatUser(user: User): string {
  return `${user.id}:${user.name}`;
}
```

开启 `declaration: true` 后可生成 `.d.ts`。

## 练习

1. 为一个没有类型的 JS 模块写 `declare module`。
2. 开启 `declaration` 输出声明文件。
3. 比较默认导入和命名导入在 CJS 包上的差异。
4. 给 CSS module 写声明。

## 验收

- 能解释 `.d.ts` 不包含实现。
- 能写简单模块声明。
- 能说明 `types` 和 `exports.types` 的作用。
- 能排查常见导入兼容问题。

## 重点

- 类型声明必须和运行时代码真实 API 保持一致。
- `declare` 不会创造运行时变量。
- 包发布要同时考虑 ESM、CJS 和类型入口。

## 难点

- ESM/CJS 互操作涉及 Node 版本、`package.json type`、编译选项和打包器行为。

## 易错

> **易错：** 写了 `declare const X` 后以为运行时一定有 `X`。
>
> 正确做法：`declare` 只让类型检查器相信它存在，运行时仍必须由脚本、环境或依赖真实提供。
