# Python学习资料：CPython 运行时、对象模型和内存管理

[返回索引](../Python学习资料.md)

## 学习目标

- 理解 Python 代码在 CPython 中从源码到字节码再到解释执行的主要路径。
- 掌握对象身份、引用计数、垃圾回收、内存分配和常见缓存机制。
- 能解释常见“看起来奇怪”的行为：小整数缓存、字符串驻留、列表扩容、字典有序、循环引用。
- 能用 `dis`、`sys`、`gc`、`tracemalloc` 做基础运行时观察。

## 理论导读

多数人说“Python 很慢”“Python 有 GIL”“Python 是解释型语言”时，默认说的是 CPython。CPython 是最常用的 Python 实现，它把 `.py` 源码编译成字节码，再由解释器循环执行。理解这个过程不是为了写 C 源码，而是为了在遇到性能、内存、对象生命周期和面试深问时有稳定的解释框架。

Python 的对象模型很统一：几乎一切运行时值都是对象，对象有类型、引用计数、值和一组行为。变量名只是指向对象的绑定。很多语言特性，包括函数调用、属性访问、运算符、迭代、上下文管理，最终都会落到对象协议和解释器调度上。

## 核心心智模型

把 CPython 想成一个带仓库、账本和调度员的执行系统：

- 源码是订单。
- 编译器把订单翻译成字节码指令。
- 解释器逐条执行指令。
- 对象都放在堆内存里。
- 变量名、容器元素、栈帧局部变量都是对象引用。
- 引用计数是账本，没人引用的对象可以立即回收。
- 循环引用由周期性垃圾回收器补充处理。

## 从源码到执行

| 阶段 | 发生什么 | 常见观察工具 |
| --- | --- | --- |
| 解析 | 源码被解析为抽象语法树。语法错误在此阶段暴露。 | `ast.parse` |
| 编译 | AST 编译为 code object 和字节码。 | `compile`、`dis` |
| 执行 | 解释器执行字节码，维护栈帧和运行时对象。 | `dis`、`sys.settrace` |
| 缓存 | 模块字节码可缓存到 `__pycache__`。 | 查看 `__pycache__` |

```python
import dis

def add(a, b):
    total = a + b
    return total

dis.dis(add)
```

你会看到类似 `LOAD_FAST`、`BINARY_OP`、`RETURN_VALUE` 的指令。它们说明 Python 函数执行时会从局部变量槽加载对象引用，对对象执行运算，再返回结果。

> **重点：** CPython 不是逐字符执行源码，而是先编译成字节码。

## code object、function object 和 frame object

函数定义后，Python 会创建函数对象。函数对象内部引用一个 code object，code object 保存字节码、常量表、变量名等静态信息。每次调用函数时，会创建新的栈帧 frame，保存本次调用的局部变量、执行位置和调用上下文。

```python
def f(x):
    y = x + 1
    return y

print(f.__code__.co_varnames)  # ('x', 'y')
print(f.__code__.co_consts)    # (None, 1)
```

这个模型能解释递归、异常栈、闭包和调试器。调用栈就是一串 frame；异常 traceback 记录的也是这些 frame 的路径。

## 对象身份、类型和值

每个对象至少可以从三个角度观察：

- 身份：`id(obj)`，在 CPython 中通常和对象内存地址相关。
- 类型：`type(obj)`，决定对象支持哪些操作。
- 值：对象表示的数据内容。

```python
a = [1, 2]
b = a
c = [1, 2]

print(a is b)  # True
print(a == c)  # True
print(a is c)  # False
```

> **易错：** 用 `id` 推断长期稳定地址。对象销毁后，内存地址可能被后续对象复用。

## 引用计数

CPython 对多数对象使用引用计数管理生命周期。每当有一个新引用指向对象，计数增加；引用消失，计数减少；计数降到 0，对象通常会立即释放。

```python
import sys

obj = []
print(sys.getrefcount(obj))  # 注意：传参给 getrefcount 本身会临时增加一次引用
```

常见会增加引用的地方：

- 变量名绑定对象。
- 容器保存对象。
- 函数参数绑定对象。
- 闭包、默认参数、全局缓存保存对象。

> **难点：** 引用计数解释了很多“对象为什么还没释放”的问题，但不能单独解决循环引用。

## 循环引用和垃圾回收

两个或多个对象互相引用时，即使外部已经没有引用，它们的引用计数也不会降到 0。这时 CPython 的循环垃圾回收器会检测容器对象之间的引用环，并在安全时回收。

```python
import gc

a = []
b = [a]
a.append(b)

del a
del b
print(gc.collect())
```

垃圾回收关注的是“容器对象”的引用图，不是每个整数、字符串都频繁扫描。生产中排查内存问题时，要区分：

- 对象还被全局变量、缓存、闭包、日志 handler 等引用。
- 对象形成循环引用。
- 内存释放给 Python 分配器，但未立刻还给操作系统。

## `__del__`、弱引用和资源释放

不要把文件、网络连接、锁这类外部资源的释放完全寄托在对象析构上。对象什么时候被回收受实现和引用关系影响，`__del__` 还可能让循环引用处理更复杂。资源管理应优先使用上下文管理器。

```python
from contextlib import closing
from urllib.request import urlopen

with closing(urlopen("https://example.com")) as response:
    data = response.read()
```

弱引用允许你引用对象但不增加其引用计数，常用于缓存、观察者模式、对象注册表。

```python
import weakref

class User:
    pass

u = User()
ref = weakref.ref(u)
print(ref() is u)
```

> **重点：** 外部资源生命周期用 `with` 管理，内存对象生命周期由引用关系管理。

## 小整数缓存和字符串驻留

CPython 会缓存一部分常用不可变对象，例如小整数和部分字符串。这是实现优化，不是语言语义。它会导致一些 `is` 比较在交互环境中看似成立。

```python
a = 256
b = 256
print(a is b)  # CPython 中常见为 True

x = 10_000
y = 10_000
print(x == y)  # 永远应该用值比较
```

> **易错：** 用 `is` 判断数字或字符串值。正确做法是使用 `==`。

## 列表的内存模型和扩容

Python 列表是动态数组，内部保存对象引用。追加元素时，如果容量不足，会申请更大的连续引用数组并搬迁引用。由于有预留容量，连续 `append` 的摊还复杂度通常是 O(1)。

适合列表的场景：

- 按顺序存放元素。
- 频繁尾部追加。
- 按索引访问。

不适合列表的场景：

- 大量头部插入删除，考虑 `collections.deque`。
- 大量成员判断，考虑 `set`。
- 大量数值计算，考虑 `array`、`numpy`。

## 字典的哈希表和插入有序

Python 3.7+ 语言层面保证字典保持插入顺序。字典平均查找接近 O(1)，依赖键的哈希值和相等性判断。键必须可哈希，并且作为键期间哈希和相等性结果应稳定。

```python
data = {"a": 1, "b": 2}
data["c"] = 3
print(list(data))  # ['a', 'b', 'c']
```

高质量字典键要满足：

- `__hash__` 返回稳定。
- `__eq__` 与 `__hash__` 协调：相等对象哈希必须相等。
- 不要用可变字段影响哈希。

> **难点：** 自定义对象如果重写 `__eq__`，默认可能变为不可哈希，需要明确设计。

## `__slots__` 的作用和边界

普通实例通常有 `__dict__` 保存动态属性。`__slots__` 可以声明固定属性，减少每个实例的内存开销，并阻止随意添加未声明属性。

```python
class Point:
    __slots__ = ("x", "y")

    def __init__(self, x, y):
        self.x = x
        self.y = y
```

适合场景：

- 会创建大量小对象。
- 属性集合稳定。
- 对内存敏感。

不适合场景：

- 需要动态添加属性。
- 依赖 `__dict__`、某些序列化工具或多继承复杂场景。

## 排查内存问题

```python
import tracemalloc

tracemalloc.start()

data = [str(i) for i in range(100_000)]

snapshot = tracemalloc.take_snapshot()
for stat in snapshot.statistics("lineno")[:5]:
    print(stat)
```

排查思路：

1. 先确认是对象数量增长、单个对象变大，还是外部资源泄漏。
2. 用 `tracemalloc` 找分配热点。
3. 用 `gc.get_referrers` 谨慎查看谁还引用对象。
4. 检查全局缓存、LRU 缓存、单例、闭包、日志 handler、长生命周期队列。
5. 区分 Python 堆内存下降和操作系统 RSS 下降。

## 练习

3. 创建 100 万个普通对象和 100 万个 `__slots__` 对象，用 `tracemalloc` 比较内存。
4. 自定义一个可哈希对象，再故意让参与哈希的字段可变，观察字典查找异常行为。

- 能解释引用计数和循环垃圾回收各解决什么问题。

## 重点

- CPython 运行机制是理解性能、内存、GIL、调试器和异常栈的基础。
- Python 变量名绑定对象，不保存对象本身。
- 引用计数能及时回收多数对象，循环 GC 处理引用环。
- 内置容器的复杂度优势来自底层数据结构。

## 难点

- 内存释放给 Python 分配器不等于立刻还给操作系统。
- 对象生命周期由引用图决定，不由代码块直觉决定。
- Python 语言规范和 CPython 实现细节要区分。

## 易错

> **易错：** 把 CPython 的缓存行为当成 Python 语义。
>
> 正确做法：值比较用 `==`，身份比较只用于对象同一性或 `None`。

> **易错：** 看到 RSS 不下降就断定对象没释放。
>
> 正确做法：结合对象数量、`tracemalloc` 和分配器行为分析。

> **易错：** 用 `__del__` 管理文件和网络连接。
>
> 正确做法：使用上下文管理器和显式关闭。
