# Python学习资料：导入系统、打包发布和依赖解析

[返回索引](../Python学习资料.md)

## 学习目标

- 深入理解 `import` 的查找、加载、缓存、包初始化和循环导入。
- 理解 `sys.path`、finder、loader、module spec、`sys.modules` 的职责边界。
- 掌握现代 Python 项目的打包结构、`pyproject.toml`、wheel、入口点和版本约束。
- 能排查依赖冲突、环境错乱、模块遮蔽和安装后行为不一致。

## 理论导读

导入系统是 Python 项目规模变大后最容易出问题的机制之一。很多“本地能跑、部署不能跑”“明明安装了包却找不到”“导入的是错误文件”“循环导入时对象不存在”的问题，都来自对导入路径和模块初始化过程缺少清晰模型。

打包和依赖解析则解决“代码如何被别人安装和复现”的问题。一个脚本可以靠相对路径运行，但一个团队项目需要明确源码布局、构建配置、运行入口、依赖版本和兼容范围。

## 核心心智模型

导入可以理解成四步：

1. 查缓存：`sys.modules` 里是否已有模块对象。
2. 找规格：根据 `sys.path` 和 finder 找到 module spec。
3. 加载执行：loader 创建模块并执行模块顶层代码。
4. 写缓存：模块对象放入 `sys.modules`，供后续导入复用。

## `import` 查找顺序

导入顶层模块时，Python 会按 `sys.path` 顺序查找。典型来源包括：

- 当前脚本目录或当前工作目录相关路径。
- `PYTHONPATH`。
- 标准库路径。
- 虚拟环境的 `site-packages`。
- `.pth` 文件注入的路径。

```python
import sys
from pprint import pprint

pprint(sys.path)
```

> **易错：** 项目里有 `json.py`、`typing.py`、`requests.py`，遮蔽标准库或第三方库。

## `sys.modules` 和部分初始化模块

模块执行前后都会和 `sys.modules` 交互。为支持循环引用，模块对象可能在执行完成前就被放入缓存。循环导入时，另一个模块拿到的可能是“部分初始化”的模块。

```python
# a.py
import b
value = 1

# b.py
import a
print(a.value)  # 可能因为 a 尚未执行到 value 赋值而失败
```

解决循环导入的常见方法：

- 把共享类型或常量移到第三个模块。
- 延迟导入到函数内部。
- 反转依赖，让高层组装，底层不互相导入。
- 使用接口或回调降低双向依赖。

## 包初始化和 `__init__.py`

导入包时会执行包目录下的 `__init__.py`。这个文件适合放少量公共 API 暴露和版本信息，不适合放复杂业务初始化。

```python
# mypkg/__init__.py
from .service import run

__all__ = ["run"]
```

> **重点：** `__init__.py` 执行成本会影响导入包的成本。

## 相对导入和绝对导入

包内部可以使用相对导入：

```python
from .parser import parse_line
```

但相对导入依赖包上下文。直接运行包内文件常导致 `ImportError: attempted relative import with no known parent package`。推荐用模块方式运行：

```powershell
python -m mypkg.cli
```

## 命名空间包

Python 支持没有 `__init__.py` 的命名空间包，允许多个路径贡献同一个包名。这在大型组织或插件系统中有用，但也会让包来源更复杂。

排查时可以查看：

```python
import package_name
print(package_name.__spec__)
print(package_name.__path__)
```

## 现代项目结构

推荐库项目或中大型应用使用 `src` 布局：

```text
project/
  pyproject.toml
  README.md
  src/
    package_name/
      __init__.py
      cli.py
      service.py
  tests/
```

`src` 布局的价值：

- 避免测试时意外导入仓库根目录中的源码。
- 更接近安装后的真实导入行为。
- 迫使你正确配置包发现和测试路径。

## `pyproject.toml`

简化示例：

```toml
[project]
name = "log-report"
version = "0.1.0"
requires-python = ">=3.10"
dependencies = []

[project.scripts]
log-report = "log_report.cli:main"

[build-system]
requires = ["setuptools>=68"]
build-backend = "setuptools.build_meta"
```

入口点安装后，用户可以直接运行：

```powershell
log-report --help
```

## wheel、源码分发和可编辑安装

| 形式 | 用途 |
| --- | --- |
| wheel `.whl` | 构建后的二进制分发格式，安装快，生产常用。 |
| sdist `.tar.gz` | 源码分发，安装时可能需要构建。 |
| editable install | 开发时安装项目，源码改动即时生效。 |

```powershell
python -m pip install -e .
python -m pip wheel .
```

## 依赖解析和版本约束

依赖解析器会尝试找到一组满足所有版本约束、Python 版本、平台标签的包版本。冲突发生时，可能是两个依赖要求不兼容，或者当前 Python 版本不满足包要求。

版本约束示例：

```toml
dependencies = [
  "requests>=2.31,<3",
]
```

生产项目需要区分：

- 直接依赖：你代码直接 import 的包。
- 传递依赖：直接依赖再依赖的包。
- 版本范围：声明兼容范围。
- 锁定版本：部署时固定具体版本，保证复现。

> **重点：** 库项目通常不要过度锁死依赖；应用项目要追求可复现部署。

## 排障清单

| 症状 | 检查项 |
| --- | --- |
| 安装了却导入不了 | `sys.executable`、虚拟环境、`python -m pip show 包名`。 |
| 导入了错误模块 | `模块.__file__`、同名文件、`sys.path` 顺序。 |
| 直接运行包内文件失败 | 改用 `python -m 包.模块`。 |
| 循环导入 | 查看 traceback 中两个模块是否互相在顶层导入。 |
| 本地测试能过，安装后失败 | 检查包数据文件、`src` 布局、未声明依赖。 |
| 依赖冲突 | 新建干净环境，逐个安装，查看版本约束。 |

## 练习

1. 创建一个 `src` 布局项目，配置 `pyproject.toml` 和命令行入口点。
2. 故意创建一个 `json.py`，观察标准库导入被遮蔽的错误，再修复。
3. 构造循环导入例子，用第三个模块拆出共享定义解决。
4. 使用 `python -m pip install -e .` 做可编辑安装。
5. 写一份排障记录，说明你如何确认当前运行的是哪个解释器和哪个包文件。

## 验收

- 能说清 `sys.path`、finder、loader、`sys.modules` 的作用。
- 能解释循环导入为什么会得到部分初始化模块。
- 能设计清晰的 `src` 项目结构。
- 能写基础 `pyproject.toml` 和控制台入口。
- 能排查模块遮蔽、环境错乱和依赖冲突。

## 重点

- 导入是查找、加载、执行、缓存的组合。
- 运行方式会影响导入上下文。
- 打包发布解决的是复现和分发，不只是“能 import”。

## 难点

- 依赖解析受 Python 版本、平台、版本范围和传递依赖共同影响。
- 循环导入本质是模块边界设计问题，不应只靠调整 import 顺序掩盖。

## 易错

> **易错：** 在当前目录下测试通过，就认为打包安装也一定通过。
>
> 正确做法：在干净虚拟环境中安装项目后运行测试。

> **易错：** 应用项目不锁版本，生产部署每次装到不同依赖。
>
> 正确做法：应用部署使用锁定策略并记录 Python 版本。
