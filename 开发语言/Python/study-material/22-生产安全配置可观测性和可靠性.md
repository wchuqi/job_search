# Python学习资料：生产安全、配置、可观测性和可靠性

[返回索引](../Python学习资料.md)

## 学习目标

- 理解 Python 应用从本地脚本到生产运行需要补齐的安全、配置、日志、错误处理和交付能力。
- 掌握配置分层、密钥管理、输入校验、日志脱敏、超时重试、退出码和健康检查。
- 能设计一个可观测、可排障、可复现、可回滚的 Python 工程。

## 理论导读

生产代码的标准不是“能跑”，而是别人能安装、运行、监控、定位、升级和回滚。Python 很适合快速开发，但也容易因为脚本习惯留下隐患：硬编码路径、裸异常、无超时请求、日志泄密、依赖不锁、全局状态混乱。

可靠性来自边界清晰：输入边界要校验，外部依赖要超时，错误要分层处理，日志要有上下文，配置要可覆盖，依赖要可复现，危险操作要可审计。

## 核心心智模型

把生产程序看成一个长期运行的服务或可重复执行的任务：

- 配置决定它在哪里运行。
- 输入决定它处理什么。
- 依赖决定它能调用谁。
- 日志和指标说明它发生了什么。
- 错误处理决定失败时怎么收场。
- 版本和依赖记录决定能否复现和回滚。

## 配置分层

常见配置来源优先级：

1. 命令行参数。
2. 环境变量。
3. 配置文件。
4. 默认值。

高优先级覆盖低优先级。这样本地、测试、生产可以共用同一份代码，只在运行时注入差异。

```python
import os
from dataclasses import dataclass

@dataclass(frozen=True)
class Settings:
    log_level: str
    output_dir: str

def load_settings() -> Settings:
    return Settings(
        log_level=os.getenv("LOG_LEVEL", "INFO"),
        output_dir=os.getenv("OUTPUT_DIR", "reports"),
    )
```

> **重点：** 配置是运行环境差异，不应硬编码在业务逻辑里。

## 密钥和敏感信息

不要把密码、Token、私钥写入代码、配置样例或日志。密钥应通过环境变量、密钥管理服务或部署平台注入。

敏感信息包括：

- 密码、Token、API Key。
- 身份证、手机号、邮箱等个人信息。
- Cookie、Authorization header。
- 数据库连接串。

日志脱敏示例：

```python
def mask_token(token: str) -> str:
    if len(token) <= 8:
        return "***"
    return token[:4] + "***" + token[-4:]
```

> **易错：** 异常日志里直接打印完整请求头或配置对象。

## 输入校验和危险 API

外部输入不可信。文件路径、命令参数、JSON 字段、HTTP 参数都要校验。

高风险 API：

- `eval`、`exec`
- `pickle.loads` 处理不可信数据
- 拼接 shell 命令
- 不限制路径的文件读写
- 解压不可信压缩包

危险示例：

```python
import os

def bad(filename):
    os.system("type " + filename)
```

更安全做法是避免 shell 拼接，使用结构化参数和白名单。

```python
import subprocess

subprocess.run(["python", "--version"], check=True)
```

## 路径安全

处理用户传入路径时，要防止目录穿越。

```python
from pathlib import Path

def safe_join(base: Path, user_path: str) -> Path:
    base = base.resolve()
    target = (base / user_path).resolve()
    if base not in target.parents and target != base:
        raise ValueError("path escapes base directory")
    return target
```

## 外部请求：超时、重试和幂等

任何网络、数据库、消息队列调用都可能慢、失败或返回错误。默认没有超时的请求会拖死程序。

设计原则：

- 必须设置连接和读取超时。
- 只对可重试错误重试。
- 重试要有次数上限和退避。
- 非幂等操作谨慎重试。
- 记录请求目标、耗时、状态和错误类型。

## 日志设计

日志要回答：什么时候、哪个请求或任务、发生了什么、关键参数是什么、结果如何、错误根因是什么。

```python
import logging

logger = logging.getLogger(__name__)

def process_file(path):
    logger.info("process_file_start", extra={"path": str(path)})
    try:
        ...
    except Exception:
        logger.exception("process_file_failed", extra={"path": str(path)})
        raise
```

日志级别：

| 级别 | 用途 |
| --- | --- |
| DEBUG | 开发和深度排查。 |
| INFO | 关键业务流程和状态变化。 |
| WARNING | 可恢复异常或风险信号。 |
| ERROR | 当前操作失败。 |
| CRITICAL | 服务不可用或严重故障。 |

## 错误边界和退出码

CLI 程序应把内部异常转换为用户可读错误和退出码。

```python
def main() -> int:
    try:
        run()
        return 0
    except ConfigError as exc:
        print(f"config error: {exc}")
        return 2
    except RuntimeError as exc:
        print(f"runtime error: {exc}")
        return 1

if __name__ == "__main__":
    raise SystemExit(main())
```

错误分层：

- 底层：抛出具体异常，保留根因。
- 中层：添加业务上下文。
- 入口：记录日志，输出可读信息，返回退出码。

## 可观测性

可观测性通常包括三类信号：

- 日志：离散事件和上下文。
- 指标：可聚合的数值，如耗时、错误率、队列长度。
- Trace：跨服务调用链路。

即使是 CLI 工具，也应至少记录耗时、处理数量、失败数量和输出位置。

```python
from time import perf_counter

start = perf_counter()
count = run_job()
logger.info("job_finished", extra={"count": count, "cost": perf_counter() - start})
```

## 依赖和供应链安全

实践建议：

- 使用虚拟环境隔离依赖。
- 记录 Python 版本和依赖版本。
- 定期升级依赖并运行测试。
- 不安装来源不明的包。
- 检查包名拼写，防止 typosquatting。
- 生产构建使用可复现依赖来源。

## 部署和回滚

最小交付清单：

- 明确运行命令。
- 明确 Python 版本。
- 明确依赖安装方式。
- 有配置说明。
- 有日志位置和日志等级。
- 有测试命令。
- 有版本号或构建标识。
- 有回滚方式。

## 练习

1. 给 `log_report` 项目加配置加载，支持环境变量覆盖默认值。
2. 给所有外部文件操作加清晰错误信息和异常链。
3. 写一个日志脱敏函数并加测试。
4. 给 CLI 加退出码，区分配置错误和运行错误。
5. 写一份部署说明，要求新机器可以复现运行。

## 验收

- 能解释配置分层和覆盖优先级。
- 能识别常见危险 API 和安全替代方案。
- 能写出有上下文、不泄密的日志。
- 能为外部调用设置超时和重试边界。
- 能把 Python 工程交付给别人复现运行。

## 重点

- 生产代码要可复现、可观测、可排障。
- 外部输入永远不可信。
- 密钥不进仓库，不进日志。
- 外部依赖必须有超时、错误处理和恢复策略。

## 难点

- 日志既要有足够上下文，又不能泄露敏感信息。
- 重试可能放大故障，尤其是非幂等操作。
- 配置分层过多会增加排查难度，需要明确优先级。

## 易错

> **易错：** 本地脚本直接上线，路径、Token、依赖全靠本机状态。
>
> 正确做法：把环境差异配置化，把依赖和运行入口文档化。

> **易错：** 捕获异常后只打印“失败了”。
>
> 正确做法：保留异常链，记录上下文，入口层输出对使用者有意义的信息。
