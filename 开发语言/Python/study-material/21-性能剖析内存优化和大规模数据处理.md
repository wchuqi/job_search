# Python学习资料：性能剖析、内存优化和大规模数据处理

[返回索引](../Python学习资料.md)

## 学习目标

- 建立“先测量，再优化”的性能工作流。
- 掌握时间剖析、内存剖析、复杂度分析和大文件处理策略。
- 能区分算法问题、IO 问题、对象分配问题、序列化问题和并发调度问题。
- 能在不牺牲可维护性的前提下做有效优化。

## 理论导读

Python 性能优化最常见的错误是凭感觉改代码。正确顺序是：定义指标、准备可复现数据、测量基线、定位瓶颈、选择最小改动、再次测量、记录取舍。没有基线，就无法判断优化是否有效。

性能问题通常不只来自语言本身。糟糕的算法、过多 IO、重复解析、一次性加载大数据、无界缓存、错误并发模型，都可能比 Python 字节码执行慢更致命。

## 核心心智模型

把性能看成管道吞吐：

- 算法复杂度决定理论上限。
- 数据结构决定访问成本。
- IO 决定等待时间。
- 对象分配决定内存压力。
- 并发模型决定等待期间能否做其他事。
- 序列化和拷贝决定跨边界成本。

## 性能剖析流程

1. 明确目标：延迟、吞吐、CPU、内存、启动时间还是成本。
2. 固定输入：使用代表性数据，不用随手编的小样本代替生产规模。
3. 测量基线：记录运行时间、内存、环境和版本。
4. 定位热点：用剖析工具找到最耗时或分配最多的位置。
5. 做最小优化：优先算法、数据结构、IO 次数。
6. 回归验证：保证结果正确且性能确实改善。
7. 记录取舍：优化可能增加复杂度，要说明原因。

## 时间测量

粗粒度：

```python
from time import perf_counter

start = perf_counter()
result = do_work()
cost = perf_counter() - start
print(f"cost={cost:.3f}s")
```

函数级剖析：

```python
import cProfile
import pstats

cProfile.run("do_work()", "profile.out")
stats = pstats.Stats("profile.out")
stats.sort_stats("cumtime").print_stats(20)
```

看 `tottime` 和 `cumtime`：

- `tottime`：函数自身耗时，不含子函数。
- `cumtime`：函数和所有子调用累计耗时。

## 内存剖析

```python
import tracemalloc

tracemalloc.start()
do_work()
snapshot = tracemalloc.take_snapshot()
for stat in snapshot.statistics("lineno")[:10]:
    print(stat)
```

常见内存问题：

- 一次性读入大文件。
- 中间列表过多。
- 缓存无上限。
- 任务队列无界。
- 全局变量或闭包持有大对象。
- pandas 或 JSON 处理时复制过多。

## 算法和数据结构优先

错误示例：列表成员判断嵌套循环。

```python
blocked = ["a", "b", "c"]
items = ["a", "x", "y"]

result = [item for item in items if item not in blocked]
```

数据量大时应改成集合：

```python
blocked_set = set(blocked)
result = [item for item in items if item not in blocked_set]
```

比较：

| 场景 | 低效做法 | 更合适做法 |
| --- | --- | --- |
| 成员判断 | `x in list` | `x in set` |
| 计数 | 手写 dict 分支 | `collections.Counter` |
| 队列头部弹出 | `list.pop(0)` | `collections.deque` |
| 排名前 N | 全量排序 | `heapq.nlargest` |
| 字符串大量拼接 | 循环 `+=` | `''.join(parts)` |

## 减少 IO 次数

IO 通常比内存计算慢很多。优化方向：

- 批量读写。
- 减少重复打开文件。
- 避免循环里逐条网络请求，能批量就批量。
- 设置超时和重试，防止慢请求拖死整体。
- 对重复外部查询做有界缓存。

## 流式处理大文件

错误做法：

```python
lines = open("big.log", encoding="utf-8").readlines()
```

正确做法：

```python
from pathlib import Path

def iter_errors(path: Path):
    with path.open("r", encoding="utf-8") as f:
        for line in f:
            if "ERROR" in line:
                yield line
```

流式处理原则：

- 输入逐行或分块。
- 中间转换使用生成器。
- 聚合状态尽量小。
- 输出可以增量写。
- 对异常行记录并跳过或进入错误通道。

## 缓存策略

`functools.lru_cache` 适合纯函数或输入稳定的计算。

```python
from functools import lru_cache

@lru_cache(maxsize=1024)
def load_user(user_id: int):
    ...
```

缓存要回答四个问题：

- 键是什么。
- 值何时失效。
- 最大占用多少内存。
- 错误结果是否缓存。

> **易错：** 无上限缓存会把性能问题变成内存问题。

## 避免不必要对象分配

优化方向：

- 用生成器代替中间列表。
- 合并多次遍历。
- 避免在循环里重复编译正则。
- 避免频繁构造大字典。
- 对热点小对象考虑 `__slots__` 或更紧凑结构。

```python
import re

pattern = re.compile(r"ERROR\s+(\w+)")
for line in lines:
    match = pattern.search(line)
```

## 并发不是性能万能药

并发有额外成本：

- 线程上下文切换。
- 锁竞争。
- 进程序列化。
- 事件循环调度。
- 外部系统限流。
- 内存占用上升。

只有当等待时间或可并行计算足够大时，并发才值得。

## 生产性能指标

常见指标：

- P50/P95/P99 延迟。
- 吞吐量。
- 错误率。
- CPU 使用率。
- 内存峰值和增长趋势。
- 队列长度。
- 外部依赖耗时。

单次本地耗时不能代表生产表现。生产要结合日志、指标、trace 和压测。

## 练习

1. 写一个低效日志统计程序，先用列表成员判断，再改成集合，记录数据规模和耗时。
2. 用 `cProfile` 找出函数热点，并写出优化前后对比。
3. 用 `tracemalloc` 比较 `readlines()` 和逐行生成器的内存差异。
4. 给一个重复计算函数加 `lru_cache`，测试不同 `maxsize` 的内存和命中率。
5. 对同一任务分别使用同步、线程池、进程池，解释性能差异。

## 验收

- 能说明一次性能优化的指标、基线、瓶颈和结果。
- 能使用 `cProfile` 和 `tracemalloc` 定位基础问题。
- 能识别常见低效数据结构选择。
- 能写流式大文件处理代码。
- 能说明缓存和并发的副作用。

## 重点

- 先测量，再优化。
- 算法、数据结构和 IO 次数优先于微优化。
- 大数据处理优先控制内存峰值。
- 优化要保留正确性测试。

## 难点

- 性能数据容易受环境、输入规模、缓存状态影响。
- 局部最快的代码不一定让系统整体更快。
- 某些优化会降低可读性，必须有数据支撑。

## 易错

> **易错：** 为了优化把清晰代码改成难懂代码，但没有测量证明收益。
>
> 正确做法：每次优化都保留基线和对比数据。

> **易错：** 只看平均耗时，不看尾延迟和异常路径。
>
> 正确做法：生产服务至少关注 P95/P99 和错误率。
