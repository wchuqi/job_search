## 一、模块、包、环境和工程化

### 1. Python 导入模块时发生了什么？

**参考答案：**

解释器根据 `sys.path` 查找模块，找到后创建模块对象、执行模块顶层代码，并把模块放入 `sys.modules` 缓存。后续导入同一模块时通常直接复用缓存。

```python
import sys
print(sys.path)
print(sys.modules.get("json"))
```

> **重点：** 导入会执行模块顶层代码。
> **易错：** 在模块顶层写网络请求、数据库连接或文件写入，导致导入就产生副作用。

### 2. `ModuleNotFoundError` 如何排查？

**参考答案：**

先确认当前解释器和虚拟环境，再确认包是否安装、运行目录是否正确、导入路径是否在 `sys.path` 中、是否用错直接运行包内文件的方式。必要时打印 `sys.executable`、`sys.path` 和模块 `__file__`。

```python
import sys
print(sys.executable)
print(sys.path)
```

> **重点：** 很多导入问题本质是运行方式和环境问题。
> **易错：** 盲目反复安装包，而不检查当前 Python 解释器。

### 3. `python script.py` 和 `python -m package.module` 有什么区别？

**参考答案：**

`python script.py` 直接以文件脚本方式运行，脚本所在目录会进入搜索路径；`python -m package.module` 以模块方式运行，会基于包上下文解析导入，通常更适合包内入口。

> **重点：** 包项目推荐使用 `python -m 包.模块` 验证入口。
> **难点：** 相对导入依赖包上下文，直接运行包内文件可能失败。

### 4. 为什么需要虚拟环境？

**参考答案：**

虚拟环境隔离项目依赖，避免不同项目互相污染。它让每个项目有独立的解释器环境和 site-packages，便于复现、升级、删除和排查依赖问题。

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
```

> **重点：** 每个项目一个虚拟环境是安全默认做法。
> **易错：** 全局安装所有依赖，导致版本冲突不可控。

### 5. `requirements.txt` 和 `pyproject.toml` 有什么区别？

**参考答案：**

`requirements.txt` 常用于记录要安装的依赖列表，简单直接；`pyproject.toml` 是现代 Python 项目的标准配置入口，可描述构建系统、项目元数据、依赖和工具配置。应用部署常配合锁定文件使用，库项目通常需要 `pyproject.toml`。

> **重点：** 二者用途不同，不是简单替代关系。
> **难点：** 生产复现要关注版本锁定，而不仅是声明宽泛依赖范围。

### 6. 一个可维护 Python 项目应该如何组织？

**参考答案：**

常见结构是 `src/包名` 放源码，`tests/` 放测试，`README.md` 写运行说明，`pyproject.toml` 管理项目和工具配置。入口、业务逻辑、IO 和测试应分离。

```text
project/
  pyproject.toml
  src/
    project_name/
      cli.py
      service.py
  tests/
```

> **重点：** 项目结构服务可测试和可复现。
> **易错：** 所有逻辑写进一个脚本，后续无法测试和复用。

### 7. 如何避免配置和密钥管理问题？

**参考答案：**

密钥不应写入代码仓库；路径、环境差异、Token 等应通过环境变量、配置文件或部署平台注入。日志中不要输出敏感信息，配置读取失败要给出清晰错误。

> **重点：** 配置是运行时差异，代码是稳定逻辑。
> **易错：** 把本机绝对路径、账号密码、Token 写死在代码里。
