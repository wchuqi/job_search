# Python学习资料：并发、异步和性能

[返回索引](../Python学习资料.md)

## 学习目标

- 理解线程、进程、协程、异步 IO 和 GIL 的适用场景。
- 能区分 IO 密集、CPU 密集和内存密集任务。
- 掌握基础性能分析、缓存、流式处理和安全优化思路。

## 理论导读

并发解决“同时处理多个任务”的组织问题，性能优化解决“单位资源做更多事”的效率问题。Python 中线程适合许多阻塞 IO 场景，进程适合 CPU 密集任务，协程适合大量可等待的 IO 任务。CPython 的 GIL 会影响多线程执行 Python 字节码的 CPU 并行能力，但不代表线程没有价值。

优化前先测量。没有测量的优化容易把代码变复杂，却没有改善真正瓶颈。常见工具包括 `time.perf_counter`、`cProfile`、日志耗时、内存采样和基准测试。

## 核心心智模型

把任务瓶颈分成三类：等外部系统、算 CPU、占内存。等外部系统时并发能提高吞吐；算 CPU 时需要进程、向量化或原生扩展；占内存时要流式处理和减少中间对象。

## 知识点详解

| 模型 | 适用场景 | 注意点 |
| --- | --- | --- |
| 线程 | 文件、网络、数据库等阻塞 IO | 共享内存，需要锁保护共享状态。 |
| 进程 | CPU 密集计算 | 进程间通信和序列化有成本。 |
| 协程 | 大量网络 IO、异步客户端 | 不能在事件循环中调用阻塞函数。 |
| GIL | CPython 全局解释器锁 | 限制多线程同时执行 Python 字节码。 |
| 缓存 | 重复计算或重复查询 | 要考虑失效、内存和输入是否可哈希。 |
| 流式处理 | 大文件、大数据流 | 避免一次性读入全部数据。 |

## 例子

线程池处理 IO 任务：

```python
from concurrent.futures import ThreadPoolExecutor
from pathlib import Path

def count_lines(path: Path) -> int:
    with path.open("r", encoding="utf-8") as f:
        return sum(1 for _ in f)

paths = list(Path("logs").glob("*.log"))
with ThreadPoolExecutor(max_workers=8) as pool:
    results = list(pool.map(count_lines, paths))
```

异步示意：

```python
import asyncio

async def fetch_mock(name):
    await asyncio.sleep(1)
    return f"done {name}"

async def main():
    results = await asyncio.gather(fetch_mock("a"), fetch_mock("b"))
    print(results)

asyncio.run(main())
```

性能测量：

```python
from time import perf_counter

start = perf_counter()
result = sum(range(1_000_000))
print(perf_counter() - start, result)
```

## 练习

写三个版本的目录文件统计工具：同步版、线程池版、进程池版。用同一批测试数据比较耗时，并解释差异。再写一个异步模拟版本，理解事件循环如何调度等待任务。

## 验收

- 能为 IO 密集任务选择线程或协程。
- 能为 CPU 密集任务选择进程或更合适的计算库。
- 能解释 GIL 对 CPython 多线程 CPU 并行的影响。
- 能先测量再优化，并给出优化前后数据。

## 重点

- 并发模型要由瓶颈决定，不要凭感觉选择。
- 异步代码链路中不能混入阻塞调用。
- 性能优化优先考虑算法、数据结构和 IO 次数。

## 难点

- 共享状态、锁、死锁、任务取消和异常传播会增加并发代码复杂度。
- 异步生态要求调用链上的库都支持异步，否则收益会被阻塞点抵消。

## 易错

> **易错：** 认为多线程一定能让 CPU 密集任务变快。
>
> 正确做法：在 CPython 中 CPU 密集任务优先考虑多进程、向量化、C 扩展或更合适的算法。

> **易错：** 不做性能剖析就重写代码。
>
> 正确做法：先用数据定位瓶颈，再做最小范围优化。
