# Python学习资料：类型提示、泛型和静态分析

[返回索引](../Python学习资料.md)

## 学习目标

- 理解类型提示的目标：提升可读性、工具检查和接口契约，而不是改变运行时动态类型本质。
- 掌握 Python 3.x 常用类型标注、联合类型、泛型、协议、`TypedDict`、`Literal`。
- 能为业务代码、库代码和测试代码选择合适的标注粒度。
- 理解 mypy、pyright、Ruff 等工具能发现什么，发现不了什么。

## 理论导读

Python 类型提示是渐进式类型系统。它允许你从关键接口开始标注，而不是要求整个项目一次性严格类型化。类型提示的价值在于把“函数需要什么、返回什么、哪些状态可能为空”写成机器可检查的契约。

类型提示默认不在运行时强制检查。`def f(x: int) -> int` 并不会阻止你传入字符串，除非额外使用运行时校验库或手写校验。静态检查工具在运行前分析代码，尽早发现不匹配、空值风险、错误属性访问和不可达分支。

## 核心心智模型

把类型提示看成接口说明书。运行时仍然能把错误零件塞进机器，但静态检查会在启动前提醒你“这个零件形状不对”。

## 基础标注

```python
def normalize_name(name: str) -> str:
    return " ".join(part.capitalize() for part in name.split())

def find_user(user_id: int) -> dict[str, object] | None:
    ...
```

Python 3.10+ 推荐用 `X | Y` 表示联合类型。旧版本常见写法是 `typing.Optional[X]` 或 `typing.Union[X, Y]`。

| 写法 | 含义 |
| --- | --- |
| `list[str]` | 字符串列表。 |
| `dict[str, int]` | 字符串到整数的映射。 |
| `str | None` | 字符串或空值。 |
| `tuple[int, str]` | 固定结构元组。 |
| `Callable[[str], int]` | 接收字符串返回整数的可调用对象。 |

## `Any`、`object` 和类型边界

`Any` 表示放弃检查，任何操作都被允许。`object` 表示任意对象，但不能随便调用具体方法。二者差别很大。

```python
from typing import Any

def bad(value: Any):
    return value.not_exist().whatever()

def safer(value: object):
    return str(value)
```

> **重点：** `Any` 是类型系统里的逃生门，应该限制在边界层，例如 JSON 解析、第三方动态数据入口。

## 泛型

泛型用于表达“输入和输出之间的类型关系”。

```python
from typing import TypeVar

T = TypeVar("T")

def first(items: list[T]) -> T:
    return items[0]
```

这里 `first(list[int])` 返回 `int`，`first(list[str])` 返回 `str`。如果写成 `object`，就丢失了这种关系。

## 协议 Protocol

`Protocol` 用结构化子类型表达“只要长得像就可以”。这非常适合 Python 的鸭子类型。

```python
from typing import Protocol

class Writer(Protocol):
    def write(self, text: str) -> int:
        ...

def save_report(writer: Writer, text: str) -> None:
    writer.write(text)
```

任何实现了 `write(str) -> int` 的对象都可以传入，不需要显式继承 `Writer`。

> **重点：** 面向接口编程在 Python 中可以用 Protocol 表达，而不是强迫继承抽象基类。

## `TypedDict`、`Literal` 和结构化数据

外部 JSON 或字典数据可以用 `TypedDict` 表达结构。

```python
from typing import Literal, TypedDict

class UserPayload(TypedDict):
    id: int
    name: str
    role: Literal["admin", "member"]

def handle_user(user: UserPayload) -> str:
    return user["name"]
```

适合场景：

- JSON API 入参出参。
- 配置字典。
- 不想立刻引入数据类或模型库。

边界：

- 运行时仍然只是普通字典。
- 深层复杂结构可读性会下降。
- 数据校验仍需运行时逻辑。

## `dataclass`、`NamedTuple` 和字典如何选择

| 结构 | 适合场景 | 优点 | 风险 |
| --- | --- | --- | --- |
| `dict` | 动态数据、外部 JSON | 灵活 | 键名错误运行时才发现 |
| `TypedDict` | 固定结构字典 | 静态检查键和值 | 运行时不校验 |
| `dataclass` | 业务领域对象 | 明确字段、方法、默认值 | 需要构造转换 |
| `NamedTuple` | 轻量不可变记录 | 可解包、不可变 | 表达复杂行为较弱 |

## 类型收窄

静态检查器会根据 `if value is None`、`isinstance`、`return` 等控制流缩小类型范围。

```python
def length(value: str | None) -> int:
    if value is None:
        return 0
    return len(value)
```

如果不做收窄，直接 `len(value)` 会被类型检查器提醒，因为 `None` 没有长度。

## 运行时校验和静态类型的边界

类型提示不能替代输入校验。用户输入、网络响应、数据库数据、文件内容都必须在运行时校验。

```python
def parse_age(raw: object) -> int:
    if not isinstance(raw, int):
        raise TypeError("age must be int")
    if raw < 0:
        raise ValueError("age must be non-negative")
    return raw
```

> **易错：** 以为加了 `age: int` 就不需要校验外部输入。

## 工具链建议

| 工具 | 用途 |
| --- | --- |
| Ruff | 快速 lint 和部分格式化能力。 |
| Black / Ruff format | 统一代码格式。 |
| mypy | 静态类型检查，生态成熟。 |
| pyright | 快速、严格、编辑器集成好。 |
| pytest | 测试行为，弥补类型检查无法证明的业务逻辑。 |

渐进策略：

1. 先标注公共函数、模块边界、复杂返回值。
2. 对外部数据入口使用 `object` 或具体解析模型，不滥用 `Any`。
3. 开启基础类型检查。
4. 对新代码逐步提高严格度。
5. 将类型检查接入 CI。

## 练习

1. 为日志分析项目的 `LogRecord`、`parse_line`、`write_report` 添加类型标注。
2. 用 `TypedDict` 表达 JSON 输出结构。
3. 用 `Protocol` 抽象一个 `ReportWriter`，支持 CSV 和 JSON writer。
4. 故意传错类型，用 mypy 或 pyright 观察报错。
5. 把一个使用大量 `Any` 的函数改成泛型函数。

## 验收

- 能解释类型提示不会自动做运行时校验。
- 能区分 `Any` 和 `object`。
- 能用泛型表达输入输出类型关系。
- 能用 `Protocol` 表达鸭子类型接口。
- 能选择 `TypedDict`、`dataclass`、普通字典的合适场景。

## 重点

- 类型提示是接口契约和工具输入。
- 外部输入必须运行时校验。
- `Any` 要收敛在边界，不要在核心业务层扩散。

## 难点

- 泛型和 Protocol 的价值在小脚本里不明显，在长期维护和多人协作中更明显。
- 静态类型无法证明业务规则正确，测试仍然必要。

## 易错

> **易错：** 为了通过类型检查到处加 `Any`。
>
> 正确做法：在边界层解析未知数据，核心逻辑使用明确类型。

> **易错：** 把类型提示当作文档装饰，不接入检查工具。
>
> 正确做法：至少在 CI 或本地任务中运行一种类型检查器。
