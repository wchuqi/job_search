# Pandas学习资料：性能优化、内存诊断和大数据边界

[返回索引](../Pandas学习资料.md)

## 学习目标

- 能定位慢操作和内存过高问题。
- 掌握 dtype 优化、分块读取、category、列裁剪。
- 知道何时切换到数据库、DuckDB、Polars、Dask 或 Spark。

## 理论导读

Pandas 适合单机内存数据处理。性能问题常见来源包括 object dtype、逐行 apply、重复 merge、大量临时副本、读取无关列、未过滤就全量加载。优化时先看数据规模、字段类型、操作复杂度和内存峰值。

## 示例

```python
import pandas as pd

df = pd.read_csv(
    "orders.csv",
    usecols=["user_id", "amount", "city"],
    dtype={"user_id": "Int64", "amount": "float32", "city": "category"},
)

print(df.memory_usage(deep=True))
```

## 优化策略

| 问题 | 策略 |
| --- | --- |
| object 字符串占内存 | string/category |
| CSV 太大 | usecols、dtype、chunksize、Parquet |
| apply 慢 | 向量化、map、merge、NumPy |
| merge 慢或爆内存 | 检查键、裁剪列、预聚合 |
| 超出内存 | DuckDB、Polars、Dask、Spark |

## 练习

读取一个大 CSV，比较默认读取和显式 dtype、usecols、category 后的内存占用。

## 验收

- 能使用 `memory_usage(deep=True)`。
- 能指出 object dtype 的风险。
- 能说明 Pandas 不适合的场景。

## 重点

- 优先减少数据量和不必要列，再考虑微优化。

## 难点

- Pandas 3.x 生态中 Copy-on-Write 等机制会影响拷贝行为，但仍需关注表达式和中间对象。

## 易错

> **易错：** 数据已超过内存还强行用 Pandas。
>
> 正确做法：先明确数据规模和计算模式，必要时使用数据库、DuckDB、Polars、Dask 或 Spark。
