# FastAPI学习资料：部署、容器、进程模型和可观测性

[返回索引](../FastApi学习资料.md)

## 学习目标

- 理解开发运行和生产运行的差异。
- 会设计容器镜像、健康检查、进程数和资源配置。
- 掌握日志、指标、追踪和错误监控的基本接入点。

## 理论导读

部署 FastAPI 不是简单把本地命令搬到服务器。生产环境要考虑 HTTPS、反向代理、进程管理、重启策略、worker 数、内存占用、数据库连接池、迁移时机、健康检查和可观测性。

`fastapi dev` 面向开发，带自动 reload；生产应使用合适的 ASGI server 命令、容器入口和进程管理。多 worker 能提高并发处理能力，但每个 worker 都会创建自己的内存和连接池。

## 核心心智模型

生产服务是一个长时间运行的进程组。能启动只是最低要求，还要能被探活、能优雅关闭、能定位错误、能承受重启和扩容。

## Dockerfile 示例

```dockerfile
FROM python:3.12-slim

WORKDIR /app
COPY pyproject.toml ./
RUN pip install --no-cache-dir "fastapi[standard]"
COPY app ./app

EXPOSE 8000
CMD ["fastapi", "run", "app/main.py", "--port", "8000"]
```

## 健康检查

```python
@app.get("/health", tags=["system"])
async def health():
    return {"status": "ok"}


@app.get("/ready", tags=["system"])
async def ready():
    return {"database": "ok"}
```

## 可观测性清单

| 能力 | 目的 |
| --- | --- |
| 请求 ID | 串联单次请求日志 |
| 结构化日志 | 便于检索和告警 |
| 指标 | 观察 QPS、延迟、错误率 |
| tracing | 定位跨服务慢调用 |
| 健康检查 | 支持部署平台探活 |
| 错误监控 | 捕获未处理异常 |

## 练习

为示例项目添加 Dockerfile、`/health`、`/ready`、请求 ID 日志，并写出启动命令。

## 验收

- 容器能启动并访问 `/docs`。
- 健康检查能区分进程存活和依赖可用。
- 日志能定位一次请求的入口、错误和耗时。

## 重点

- worker 数、连接池大小和数据库最大连接数要一起规划。
- 迁移脚本不要在每个 worker 启动时并发执行。

## 难点

- 反向代理、根路径、HTTPS 终止和客户端 IP 头会影响文档地址、重定向和日志。

## 易错

> **易错：** 开多个 worker 后忘记数据库连接池按进程复制。
>
> 正确做法：按 `worker数 * 每进程连接池` 估算数据库连接上限，并预留管理连接。
