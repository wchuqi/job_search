# FastAPI学习资料：性能优化、排障和生产规范

[返回索引](../FastApi学习资料.md)

## 学习目标

- 能定位慢接口、错误率升高、连接池耗尽和事件循环阻塞。
- 掌握常见性能优化方向。
- 建立生产规范和排障流程。

## 理论导读

FastAPI 框架本身通常不是性能瓶颈。真实瓶颈常在数据库查询、外部 HTTP 调用、序列化大对象、日志阻塞、连接池配置、worker 数、网络和容器资源限制。优化前先测量，不要凭感觉修改。

## 核心心智模型

性能排障是“缩小范围”的过程：先看请求是否进入服务，再看路由耗时、依赖耗时、数据库耗时、外部调用耗时和响应序列化耗时。

## 常见问题和定位

| 现象 | 可能原因 | 排查方向 |
| --- | --- | --- |
| P95 延迟升高 | 慢 SQL、外部服务慢 | tracing、SQL 日志、超时配置 |
| 偶发 500 | 未处理异常、连接断开 | 错误监控、异常栈 |
| 连接池耗尽 | 事务过长、连接泄露 | 池指标、Session 生命周期 |
| CPU 飙高 | 大量 JSON 序列化、CPU 计算 | profiling、拆任务 |
| 并发低 | 阻塞事件循环 | 检查 sync 调用和线程池 |
| 422 增多 | 客户端请求格式变化 | 请求样本、Schema diff |

## 生产规范

- 所有外部 HTTP 调用必须设置超时。
- 数据库查询要有索引和分页。
- 大响应要限制字段和数量。
- 关键接口要有请求 ID、耗时日志和错误码。
- 配置通过环境变量或配置中心注入。
- 升级 FastAPI、Pydantic、Starlette 前运行回归测试。

## 练习

给一个分页接口添加：

- 最大 `page_size` 限制。
- SQL 查询耗时日志。
- 外部调用超时。
- 统一错误码。

## 验收

- 能用数据说明瓶颈在哪里。
- 能写出慢接口排障步骤。
- 能解释为什么“框架快”不等于“业务接口快”。

## 重点

- 先观测，再优化。
- 超时、限流、分页、连接池和日志是生产 API 的基本功。

## 难点

- 多因素叠加时，单点优化可能无效，需要从完整请求链路看问题。

## 易错

> **易错：** 接口慢就盲目增加 worker。
>
> 正确做法：先判断瓶颈是 CPU、IO、数据库、连接池还是外部依赖。worker 增加可能放大数据库压力。
