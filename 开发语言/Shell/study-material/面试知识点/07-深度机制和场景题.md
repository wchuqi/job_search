## 一、深度机制

### 1. `cmd >out 2>&1` 和 `cmd 2>&1 >out` 为什么不同？

**参考答案：**

重定向从左到右处理。`>out 2>&1` 先把 stdout 指向 out，再把 stderr 复制到当前 stdout，所以二者都进 out。`2>&1 >out` 先把 stderr 复制到原 stdout，随后 stdout 才指向 out，因此 stderr 仍到终端。

> **重点：** `2>&1` 是复制当时 FD 1 的目标，不是建立动态绑定。

### 2. `while read` 放在管道后，循环内变量为什么带不出来？

**参考答案：**

很多 Shell 中管道的各段在子 Shell 中执行，循环内变量修改发生在子 Shell，父 Shell 看不到。可用 process substitution 或输入重定向避免。

```bash
while IFS= read -r line; do
  count=$((count + 1))
done < <(printf '%s\n' a b)
```

### 3. `set -e` 为什么没有让脚本在 `if grep ...` 时退出？

**参考答案：**

`set -e` 在条件判断、`&&`、`||`、`while/until` 条件、管道非最后命令等上下文有例外。`if grep ...` 中 grep 的非 0 是条件结果，不是未处理异常。

### 4. `FOO=bar command` 会不会改变当前 Shell 的 FOO？

**参考答案：**

对外部命令通常只影响该命令环境，不改变当前 Shell。对函数、内建命令、special builtin 时可能有不同副作用，尤其 special builtin 前的赋值可能保留。因此生产脚本不要依赖复杂临时赋值副作用。

### 5. `> file` 单独一行做了什么？

**参考答案：**

这是 redirection-only command，会创建或截断 file。它不是无操作。若 file 是重要文件，会造成数据损失。

### 6. alias 为什么不适合生产脚本？

**参考答案：**

alias 是词法替换，默认主要用于交互 Shell，非交互脚本默认不展开 alias，时机也早于函数运行。生产脚本应使用函数或显式命令。

## 二、生产场景

### 7. 一个 root 定时脚本 source 配置文件，有什么风险？

**参考答案：**

source 会执行配置文件中的 Shell 代码。如果配置文件可被低权限用户修改，等于低权限用户获得 root 代码执行。应检查所有者和权限，或按白名单 key/value 解析。

### 8. 如何设计一个安全的删除脚本？

**参考答案：**

命令行参数先解析和校验；配置必填；路径 realpath 规范化；确认目标在允许根目录内；dry-run 默认开启；删除前打印计划；使用 `rm -rf -- "$target"`；记录日志；测试覆盖空变量、符号链接、空格路径。

### 9. 脚本在 Alpine 容器中失败，但 Ubuntu 正常，可能原因？

**参考答案：**

Alpine 默认 `/bin/sh` 是 BusyBox ash，可能没有 Bash；工具也是 BusyBox 精简实现，sed、awk、find、xargs 选项可能不同；musl 和 GNU libc 行为也可能影响部分命令。应确认 shebang、安装 Bash 或改写 POSIX sh，并测试工具参数兼容性。

### 10. 如何判断 Shell 脚本性能瓶颈？

**参考答案：**

先看是否在循环中频繁启动外部命令，是否多次扫描大文件，是否用命令替换读取大输出，是否 sort 大量数据。用 `time`、`/usr/bin/time -v`、阶段日志或 `strace -f -c` 分析。优化方向是一次 awk、减少进程、流式处理，复杂任务换更合适语言。
