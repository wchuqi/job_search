## 速记清单

- Shell 命令执行前会解析、展开、分词、glob。
- 变量默认加双引号：`"$var"`。
- 参数转发用 `"$@"`。
- Bash 特性不能写在 `/bin/sh` 脚本里。
- 条件判断本质是看退出状态。
- `[` 是命令，`[[` 是 Bash 关键字。
- `set -e` 有例外，不能替代显式错误处理。
- 管道默认取最后命令退出码，生产常用 `pipefail`。
- 文件批处理用 `find -print0` 和 `xargs -0`。
- 读行用 `while IFS= read -r line`。
- 删除前校验 realpath 和允许根目录。
- `eval` 默认视为危险。
- cron 环境不是交互终端环境。
- ShellCheck 是生产脚本基本门槛。

## 自测题

### 1. 为什么 `for f in $(ls)` 不可靠？

**参考答案：**

命令替换输出会被按 IFS 分词，文件名中的空格、换行会被拆坏，glob 还可能再次展开。应使用 glob、`find -print0` 或数组。

### 2. `"$@"` 为什么适合转发参数？

**参考答案：**

它保留每个位置参数为独立参数，不会把多个参数拼成一个字符串，也不会因为空格分裂。

### 3. 如何判断一个变量必填？

**参考答案：**

使用 `${var:?message}`：

```bash
: "${DB_URL:?missing DB_URL}"
```

### 4. `while read` 从管道读数据后变量为什么没变？

**参考答案：**

很多 Shell 中管道段在子 Shell 中执行，循环内变量修改不会回到父 Shell。可改用 process substitution 或重定向输入。

### 5. 如何写安全删除？

**参考答案：**

变量加引号只是第一步，还要检查非空、类型、realpath 后在允许根目录下，支持 dry-run，并使用 `rm -rf -- "$target"`。

### 6. 什么时候不应该用 Shell？

**参考答案：**

复杂业务逻辑、复杂数据结构、大规模并发、复杂 JSON/YAML、长期维护的大型程序不适合 Shell，应考虑 Python、Go 或专用工具。

## 最终验收

- 能现场解释展开顺序。
- 能写参数解析和日志模板。
- 能处理带空格文件名。
- 能说明 `set -euo pipefail` 的边界。
- 能用 awk 写一个日志统计命令。
