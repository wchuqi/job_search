## 一、安全

### 1. eval 为什么危险？

**参考答案：**

`eval` 会让字符串再次经过 Shell 解析，相当于把数据当代码执行。不可信输入进入 eval 会造成命令注入。多数场景应使用数组、case、函数映射或显式参数替代。

### 2. 写 `rm -rf "$dir"` 就安全吗？

**参考答案：**

不够。加引号只能防止分词和 glob，还要校验变量非空、路径存在、类型正确、realpath 后位于允许根目录下，最好支持 dry-run 和删除前打印计划。

### 3. Shell 脚本有哪些 secret 泄漏路径？

**参考答案：**

命令行参数可能被 `ps` 看到；`set -x` 会打印命令；日志可能输出环境变量；`.env` 可能被打包或提交；临时文件权限过宽；CI 日志可能打印 token。

## 二、工程化

### 4. 一个生产 Shell 脚本应包含哪些基本部分？

**参考答案：**

明确 shebang、`set` 策略、usage、参数解析、日志函数、错误处理、main 入口、依赖检查、dry-run、trap 清理、锁、ShellCheck、测试和 README。

### 5. ShellCheck 的价值是什么？

**参考答案：**

ShellCheck 能静态发现未引用变量、错误数组展开、无效 POSIX 写法、read 缺 `-r`、命令替换误用等常见问题。它不能证明脚本完全正确，但能显著降低低级错误。

### 6. cron 中脚本失败，手动执行成功，怎么排查？

**参考答案：**

检查 PATH、当前目录、执行用户、权限、环境变量、解释器、locale 和输出日志。用 `env -i PATH=/usr/bin:/bin bash script.sh` 模拟最小环境。

### 7. 如何防止脚本重复运行？

**参考答案：**

使用 `flock` 做文件锁，比手写 lock 文件更可靠：

```bash
exec 9>/tmp/job.lock
flock -n 9 || exit 1
```

手写检查 lock 文件存在有竞态条件。
