# Shell学习资料：可移植性、性能和生产运维

[返回索引](../Shell学习资料.md)

## 学习目标

- 知道 Bash、POSIX sh、GNU/BSD 工具差异。
- 掌握 Shell 性能边界和优化方向。
- 设计可运行在 cron、CI、容器和生产主机上的脚本。

## 理论导读

Shell 可移植性不仅取决于解释器，还取决于外部命令版本。GNU `sed`、BSD `sed`、BusyBox `sed` 参数不完全一致；`date`、`realpath`、`readlink`、`xargs` 都可能有差异。生产脚本要声明运行环境，或写成保守 POSIX 风格。

## 可移植性层级

| 层级 | 选择 | 适合 |
| --- | --- | --- |
| 严格 POSIX | `/bin/sh` + POSIX 工具 | 跨 Unix、最小容器 |
| Bash 脚本 | Bash 4+ 或 5+ | 受控 Linux 环境 |
| Linux/GNU | Bash + GNU coreutils | 大多数服务器 |
| 特定平台 | 依赖 systemd、GNU sed、Linux procfs | 运维脚本 |

## GNU/BSD 差异例子

`sed -i`：

```bash
sed -i.bak 's/a/b/g' file     # GNU/BSD 都较稳
sed -i '' 's/a/b/g' file      # BSD 常见写法
```

`date`：

```bash
date -d 'yesterday' '+%F'     # GNU
date -v-1d '+%F'              # BSD/macOS
```

## 性能边界

Shell 性能瓶颈常来自：

- 在循环中频繁启动外部进程。
- 对大文件反复 grep/sed/awk 多遍扫描。
- 用 Shell 处理复杂字符串和数组。
- 命令替换读取超大输出到内存。

低效：

```bash
while read -r line; do
  echo "$line" | awk '{print $1}'
done < big.log
```

更好：

```bash
awk '{print $1}' big.log
```

## 生产脚本要求

- 固定解释器和依赖命令。
- 明确 `PATH`。
- 不依赖当前工作目录。
- 输出结构化日志。
- 失败有非 0 退出码。
- 支持 dry-run。
- 幂等或有锁。
- 清理临时资源。
- 对危险操作有保护。

## 容器中的 Shell

容器镜像可能只有 `/bin/sh`，没有 Bash、awk、sed 完整版本。Alpine 使用 BusyBox 工具和 musl，脚本兼容性要单独验证。

```bash
docker run --rm alpine:3.20 sh -c 'echo "$0"; sed --help | head'
```

## 练习

1. 把一个 Bash 脚本改写成 POSIX sh。
2. 在 Ubuntu、Alpine、macOS 或 BusyBox 中运行同一脚本，记录差异。
3. 优化一个循环中反复调用 awk 的脚本。

## 验收

- 能说明脚本依赖的 Shell 和外部命令版本。
- 能识别 GNU/BSD 工具差异。
- 能判断任务是否应该换 Python/Go 等语言。

## 重点

- 可移植性要声明目标环境，不要假装所有系统一样。
- 大数据处理尽量让 awk/sed/sort 一次完成。
- 生产脚本要为 cron、CI、容器的最小环境设计。

## 易错

> **易错：** 本机 macOS 上调好的脚本直接丢到 Linux 生产，或反过来。
>
> 正确做法：用目标环境容器或虚拟机验证，并固定依赖版本或写兼容分支。
