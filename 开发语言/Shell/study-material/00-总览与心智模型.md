# Shell学习资料：总览与心智模型

[返回索引](../Shell学习资料.md)

## 学习目标

- 理解 Shell 的三重角色：交互命令解释器、脚本语言、系统工具胶水层。
- 建立“文本输入 -> 解析 -> 展开 -> 执行 -> 退出状态”的运行模型。
- 明确 Shell 适合什么、不适合什么。

## 理论导读

Shell 不是普通编程语言，也不是简单地把命令按行执行。Shell 的核心工作是把一行文本解析成命令结构，对变量、命令替换、算术、通配符等进行展开，再调用内建命令、函数或外部程序。很多 Shell bug 的根源不是命令本身，而是命令执行之前发生了意外的展开、分词或路径匹配。

Shell 的优势是胶水能力：把文件、进程、标准输入输出、系统命令、文本流和退出码组合起来。它适合自动化、批处理、运维脚本、部署脚本和轻量数据处理。但它不适合复杂业务逻辑、复杂数据结构、大型并发程序或需要强类型和可维护抽象的系统。

## 核心心智模型

Shell 运行一条命令大致经过：

```text
读取文本
  -> 解析语法结构
  -> 执行各种展开
  -> 按 IFS 分词
  -> 路径名展开 glob
  -> 设置重定向
  -> 查找命令
  -> 执行命令
  -> 得到退出状态
```

这个顺序非常重要。例如：

```bash
rm $dir/*
```

如果 `dir` 为空，可能变成 `rm /*`；如果 `dir` 包含空格，可能被拆成多个参数；如果 glob 匹配了意外文件，删除范围会扩大。

## Shell 的适用边界

适合：

- 启停服务、部署、备份、巡检。
- 文件批处理。
- 日志扫描和简单统计。
- 调用系统命令组合工作流。
- CI/CD 中的轻量步骤。

不适合：

- 大型业务系统。
- 复杂 JSON/YAML 处理，除非配合 `jq`、`yq`。
- 复杂并发和错误恢复。
- 需要严格类型、复杂测试和长期演进的逻辑。

## 例子

```bash
#!/usr/bin/env bash
set -Eeuo pipefail

log() {
  printf '[%s] %s\n' "$(date '+%F %T')" "$*" >&2
}

main() {
  local path=${1:-}
  if [[ -z "$path" ]]; then
    log "usage: $0 <path>"
    return 2
  fi

  if [[ ! -e "$path" ]]; then
    log "not found: $path"
    return 1
  fi

  log "file exists: $path"
}

main "$@"
```

这个脚本体现了几个基本原则：明确解释器、错误选项、日志走 stderr、变量加引号、入口函数接收 `"$@"`。

## 练习

1. 写一个脚本，接收一个目录参数，判断它是否存在并列出其中普通文件数量。
2. 故意传入带空格的路径，验证脚本是否正确。
3. 在脚本里打印 `$0`、`$#`、`$@`、`$?` 的含义。

## 验收

- 能解释 Shell 的执行模型。
- 能说明为什么变量展开通常要加双引号。
- 能判断一个任务是否适合用 Shell 写。

## 重点

- Shell 的难点是隐式处理：展开、分词、glob、退出码。
- 脚本应默认把输入看成不可信文本。
- 小脚本也要有基本错误处理。

## 难点

- Shell 在命令执行前已经做了很多转换，错误常发生在“还没执行程序”之前。
- 同一段脚本在 Bash、dash、zsh 下行为可能不同。

## 易错

> **易错：** 把 Shell 脚本当成“命令备忘录”，不处理参数、错误、路径和日志。
>
> 正确做法：脚本是程序，至少要有入口、参数校验、失败退出和可读日志。
