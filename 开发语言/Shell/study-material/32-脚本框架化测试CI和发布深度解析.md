# Shell学习资料：脚本框架化、测试、CI 和发布深度解析

[返回索引](../Shell学习资料.md)

## 学习目标

- 把单个脚本提升为可维护的脚本项目。
- 设计测试、静态检查、版本发布和兼容性验证。
- 掌握脚本库化和入口分离。

## 理论导读

当 Shell 脚本超过几十行或被多人使用时，就需要项目治理。否则参数、日志、错误处理、安全检查会散落各处。框架化不是引入复杂框架，而是抽出稳定的库函数、统一入口、测试和发布流程。

## 项目结构

```text
scripts/
  bin/
    backup
    deploy
  lib/
    log.sh
    args.sh
    safe-path.sh
  test/
    backup.bats
    safe-path.bats
  shellcheckrc
  README.md
```

入口脚本负责解析参数和调用业务函数，库文件只定义函数，不自动执行。

```bash
# lib/log.sh
log_info() { printf '[INFO] %s\n' "$*" >&2; }
log_error() { printf '[ERROR] %s\n' "$*" >&2; }
```

```bash
# bin/backup
SCRIPT_DIR=$(cd -- "$(dirname -- "${BASH_SOURCE[0]}")" && pwd)
# shellcheck source=../lib/log.sh
. "$SCRIPT_DIR/../lib/log.sh"
main "$@"
```

## source 边界

source 库文件时：

- 路径必须受控。
- 库文件不要执行业务动作。
- 库内变量加命名空间前缀。
- 避免库修改全局 Shell 选项，或修改后恢复。

## 测试层级

- 语法：`bash -n`。
- 静态：ShellCheck。
- 单元：测试函数。
- 集成：临时目录中运行真实脚本。
- 兼容：bash/dash/alpine/macOS 矩阵。
- 破坏性操作：只在 sandbox 目录和 dry-run 中测试。

## CI gate

```bash
find bin lib test -type f -name '*.sh' -print0 | xargs -0 shellcheck
find bin lib -type f -print0 | xargs -0 -n1 bash -n
bats test
```

## 版本和发布

- 在 README 写清脚本版本、兼容环境、依赖命令。
- release tag 对应脚本内容。
- 生产机器不要 curl 最新 main 分支执行。
- 关键脚本发布前做 checksum 或包管理。

## 可观测性

生产脚本输出应包含：

- 版本。
- 参数摘要，secret 脱敏。
- 开始和结束时间。
- 操作计划。
- 失败命令上下文。
- 退出码。

## 练习

1. 把一个大脚本拆成 `bin/` 和 `lib/`。
2. 用 bats 或临时目录测试 safe path 函数。
3. 写 CI 脚本执行 bash -n、ShellCheck、测试。
4. 为脚本写 release checklist。

## 验收

- 库文件 source 后不自动执行业务动作。
- CI 能阻止未引用变量和语法错误。
- README 说明兼容环境和高风险操作。

## 易错

> **易错：** 把所有函数 source 到当前 Shell 后依赖全局变量互相传递。
>
> 正确做法：函数参数显式传递，库变量命名空间化，入口集中组装配置。
