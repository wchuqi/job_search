# Shell学习资料：大型日志处理、性能剖析和替代工具深度解析

[返回索引](../Shell学习资料.md)

## 学习目标

- 知道 Shell 脚本性能瓶颈来自哪里。
- 能减少外部进程、避免多次扫描和大输出命令替换。
- 判断何时应换 awk、perl、python、Go、jq 或日志系统。

## 理论导读

Shell 本身不擅长大量循环计算。每次调用外部命令都有 fork/exec 成本，循环中对每行调用 grep/awk/sed 会非常慢。大文件处理应尽量流式、单次扫描、把逻辑推到 awk/sed/sort 这类专门工具里。再复杂就不该继续堆 Shell。

## 性能坏味道

```bash
while read -r line; do
  echo "$line" | awk '{print $1}'
done < big.log
```

每行启动 awk，代价巨大。

改成：

```bash
awk '{print $1}' big.log
```

## 命令替换大输出

危险：

```bash
data=$(cat huge.log)
```

这会把大文件读入内存，还会删除尾随换行。应使用流式管道。

## 一次扫描

多次扫描：

```bash
grep ' 500 ' access.log | wc -l
grep ' 404 ' access.log | wc -l
grep ' 200 ' access.log | wc -l
```

一次 awk：

```bash
awk '{code[$9]++} END {for (c in code) print c, code[c]}' access.log
```

## 排序成本

`sort` 可能占用大量临时磁盘。大数据时要考虑：

```bash
LC_ALL=C sort -S 50% -T /fast/tmp file
```

如果只是 Top N，可以考虑 awk 先聚合再 sort，减少数据量。

## 剖析方法

```bash
time ./script.sh
/usr/bin/time -v ./script.sh
strace -f -c ./script.sh
```

在没有 strace 的环境下，可用日志统计每阶段耗时。

## 替代工具选择

| 场景 | 更合适工具 |
| --- | --- |
| JSON | jq、python |
| YAML | yq、python |
| CSV | csvkit、python |
| 复杂正则和状态机 | perl、python |
| 大规模日志 | awk、ripgrep、日志平台 |
| 长期维护 CLI | Python、Go |
| 高并发网络任务 | Go、Python asyncio、专用工具 |

## 练习

1. 写一个逐行启动 awk 的慢脚本，再改为单 awk。
2. 统计 access log 状态码、Top IP、Top URL，尽量一次扫描。
3. 用 `/usr/bin/time -v` 比较内存和时间。
4. 用 jq 处理 JSON 日志，比较 grep/sed 方案风险。

## 验收

- 能指出循环中外部命令启动成本。
- 能把多次扫描合并为一次 awk。
- 能判断何时停止使用 Shell。

## 易错

> **易错：** 为了“只用 Shell”而解析复杂 JSON、CSV 和大型数据。
>
> 正确做法：Shell 负责调度，复杂解析交给专用工具或更合适语言。
