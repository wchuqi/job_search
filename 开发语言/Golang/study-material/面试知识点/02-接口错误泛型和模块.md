## 一、接口、错误、泛型和模块

### 1. Go 接口为什么是隐式实现？

**参考答案：**

Go 类型只要拥有接口要求的方法，就自动实现该接口，不需要显式声明。这降低了包之间的耦合，让调用方可以定义自己需要的小接口，便于测试和替换。

> **重点：** 小接口通常放在使用方，而不是实现方。

### 2. 为什么接口变量保存 nil 指针后不等于 nil？

**参考答案：**

接口值由动态类型和动态值组成。只有动态类型和动态值都为空时接口才等于 nil。如果接口保存了一个动态类型为 `*T`、动态值为 nil 的值，接口本身不为 nil。

```go
var b *bytes.Buffer = nil
var r io.Reader = b
fmt.Println(r == nil) // false
```

> **易错：** 返回具体 nil 指针作为接口值，调用方判断 `err == nil` 或接口是否为空时可能误判。

### 3. Go 推荐怎样处理错误？

**参考答案：**

普通失败用 `error` 返回值；需要保留上下文时用 `fmt.Errorf("...: %w", err)` 包装；上层用 `errors.Is` 判断错误类别，用 `errors.As` 提取具体错误类型。`panic` 只用于不可恢复的程序错误或边界层统一保护。

### 4. 泛型和接口如何选择？

**参考答案：**

如果需要对多种静态类型复用同一算法或容器，用泛型；如果需要按行为替换实现，用接口。泛型强调编译期类型安全，接口强调运行期行为多态。

> **难点：** 不要把业务分派强行泛型化，也不要用 `any` 加大量类型断言伪装泛型。

### 5. Go modules 的 MVS 是什么？

**参考答案：**

MVS 是 Minimal Version Selection。对同一个模块路径，Go 选择依赖图中被要求的最高版本。它不是在约束范围内自动选择最新版本，因此升级依赖需要显式执行。

> **重点：** v2+ 模块通常需要在模块路径中带 `/v2`、`/v3`。
