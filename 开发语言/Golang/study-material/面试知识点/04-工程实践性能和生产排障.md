## 一、工程实践、性能和生产排障

### 1. Go HTTP 服务上线前要检查什么？

**参考答案：**

检查服务端超时、客户端超时、优雅关闭、健康检查、结构化日志、指标、trace、pprof 开关、请求体大小限制、TLS 配置、依赖漏洞和配置脱敏。默认 HTTP client 没有整体超时，不应直接用于生产外部调用。

> **重点：** 超时、取消和连接池是网络服务可靠性的基础。

### 2. 如何定位 Go 服务 CPU 飙高？

**参考答案：**

先看监控确认时间段和实例，再抓 CPU profile，使用 `go tool pprof` 查看 top、list、flamegraph。结合最近发布、流量、错误率、goroutine 数和下游延迟判断是业务热点、死循环、锁竞争、序列化开销还是重试风暴。

### 3. 如何定位内存持续上涨？

**参考答案：**

收集 heap profile，对比不同时间点的 inuse 对象；同时观察 RSS、GC 次数、堆目标、goroutine 数和缓存大小。常见原因包括无界 map、切片持有大数组、goroutine 泄漏、连接未关闭和大对象缓存。

> **易错：** 把 RSS 上涨直接等同于 Go 堆泄漏。需要结合 heap profile 和 runtime 指标判断。

### 4. 逃逸分析有什么用？

**参考答案：**

逃逸分析决定对象能否放在栈上。如果对象在函数返回后仍可能被引用，或被接口、闭包等方式延长生命周期，可能逃逸到堆。可用 `go build -gcflags=-m` 查看提示。

### 5. 如何设计 Go 项目的包边界？

**参考答案：**

入口放 `cmd/`，业务私有代码放 `internal/`，只有稳定可复用 API 才放公开包。包应该围绕职责和依赖方向组织，避免按 controller/service/utils 机械拆分。接口通常在使用方定义，小而明确。

> **难点：** 包边界一旦被外部依赖，就会成为长期兼容承诺。
