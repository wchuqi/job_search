## 一、运行时、GC、HTTP 和生产深水区

### 1. Go 程序启动时发生了什么？

**参考答案：**

进程入口先进入 runtime 启动代码，而不是直接进入 `main.main`。runtime 会初始化栈、调度器、GC、类型元数据、命令行参数和环境等，然后按 import 依赖顺序初始化包，执行包级变量初始化和 `init`，最后调用 `main.main`。`main.main` 返回后进程结束，其他 goroutine 不会阻止退出。

> **重点：** 包初始化顺序是依赖包先于导入方，一个包只初始化一次。

### 2. Go 调度器的 G/M/P 如何协作？

**参考答案：**

G 是 goroutine，M 是 OS 线程，P 是执行 Go 代码需要的调度资源和本地队列。G 必须在绑定了 P 的 M 上运行。P 有本地队列，空闲时可以从其他 P 窃取任务。阻塞系统调用时 runtime 会尽量把 P 移交给其他 M，减少整体停顿。

> **难点：** `GOMAXPROCS` 控制 P 的数量，不是 goroutine 数，也不是 OS 线程绝对数量。

### 3. Go GC 为什么需要写屏障？

**参考答案：**

Go GC 主要使用并发标记。应用 goroutine 和 GC 同时运行时，对象引用关系会变化。写屏障用于记录引用写入或维护标记不变量，避免 GC 漏标仍然可达的对象。没有屏障，并发标记可能错误回收活对象。

> **重点：** 三色标记是理解写屏障的模型，不是单纯背颜色。

### 4. 线上内存高，你如何判断是不是泄漏？

**参考答案：**

先区分 RSS、Go heap、容器 working set 和非 Go 堆。抓 heap profile 对比两个时间点，看存活对象类型和引用路径；同时看 goroutine profile、缓存大小、连接数、CGO/mmap 使用。若 heap 中某类对象持续增长且无业务上限，才更接近泄漏；若是缓存或 GC 目标堆增长，需要按容量和策略处理。

### 5. HTTP 客户端为什么不能直接用默认配置？

**参考答案：**

默认 `http.Client` 没有整体超时。下游卡住时，请求可能长期占用 goroutine 和连接。生产客户端应配置整体 timeout、Transport 连接池、TLS/Dial/ResponseHeaderTimeout，并通过 context 传递请求 deadline。

> **易错：** 只设置服务端超时，忘记所有外部调用也必须有超时。

### 6. response body 不关闭会怎样？

**参考答案：**

不关闭 body 会泄露响应资源，并影响连接复用。连接无法回到空闲池，可能导致连接数上涨、fd 耗尽、延迟升高和 goroutine 堆积。正确做法是在确认 `err == nil` 后 `defer resp.Body.Close()`，并根据场景读取或丢弃 body。

### 7. 如何回答“Go 服务 P99 延迟突然升高”？

**参考答案：**

先按链路拆解：入口排队、业务处理、下游调用、数据库、锁等待、GC、调度、网络。查看请求量、错误率、下游耗时、goroutine 数、GC 指标和资源使用。抓 trace 或 pprof，必要时看 mutex/block profile。不要直接猜 GC；P99 往往是队列、下游慢、重试或锁竞争放大的结果。

### 8. MVS 的生产风险是什么？

**参考答案：**

MVS 选择依赖图中要求的最高版本，但不会自动选择最新安全补丁。因此依赖漏洞修复需要显式升级、测试和发布。`go.sum` 校验内容完整性，不负责自动升级。私有模块还要配置 `GOPRIVATE`，避免路径泄露到公共代理或校验库。

### 9. sync.Pool 适合做什么，不适合做什么？

**参考答案：**

`sync.Pool` 适合缓存临时对象，降低高频分配压力，例如临时 buffer。它不适合作为业务缓存，因为 GC 可以清理池中对象，池中对象生命周期不稳定。放回 Pool 的对象必须清理敏感数据和旧状态。

### 10. 如何证明一次性能优化有效？

**参考答案：**

先用 benchmark 或生产 profile 定位瓶颈，记录基线 `ns/op`、`B/op`、`allocs/op` 或线上指标。修改后在相同输入和环境复测，并确认功能测试、竞态检测和真实流量压测没有回归。只展示代码更短或理论上更快，不算证明。
