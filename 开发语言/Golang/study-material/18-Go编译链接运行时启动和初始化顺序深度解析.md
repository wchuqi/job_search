# Golang 学习资料：Go 编译、链接、运行时启动和初始化顺序深度解析

[返回索引](../Golang学习资料.md)

## 学习目标

- 理解 `go build` 从源码到可执行文件的主要阶段。
- 掌握包初始化顺序、`init`、全局变量初始化和 `main.main` 的执行边界。
- 能解释构建缓存、构建标签、链接信息注入和交叉编译。
- 能排查“本地能编译、CI 不行”“init 顺序不符合预期”“构建产物不可复现”等问题。

## 理论导读

Go 的构建不是简单把所有 `.go` 文件拼起来编译。`go` 命令会先根据当前模块、构建标签、目标操作系统和架构筛选文件，再解析 import 图，按包为单位编译，最后由 linker 生成可执行文件。运行时启动也不是直接进入 `main.main`：进程先进入 runtime 的启动代码，完成栈、调度器、GC、命令行参数、环境变量、模块数据等初始化，再按依赖顺序初始化包，最后调用用户的 `main.main`。

这个链路重要，是因为很多“玄学问题”都发生在边界处：构建标签导致文件没参与编译，`init` 依赖隐藏顺序，`-ldflags` 注入版本信息失败，CGO 改变交叉编译条件，构建缓存让旧结果看起来像新代码。

## 核心心智模型

```text
源码目录
  -> go 命令读取 go.mod 和环境
  -> 根据 GOOS/GOARCH/tags 选择文件
  -> 解析 import 图并编译每个 package
  -> linker 合并符号、runtime、元数据
  -> 进程启动 runtime
  -> 初始化依赖包
  -> 执行 main.main
```

## 编译阶段

### 1. 文件选择

Go 会根据文件名后缀和 build constraints 选择文件：

- `foo_linux.go` 只在 `GOOS=linux` 时参与。
- `foo_amd64.go` 只在 `GOARCH=amd64` 时参与。
- 文件顶部 `//go:build prod` 需要 `-tags prod`。
- `_test.go` 只参与测试包构建。

> **易错：** 修改了一个带构建标签的文件，却没有在当前构建参数下启用它，结果代码完全没进二进制。

### 2. 包级编译

Go 以 package 为编译单元。同一个目录内的非测试文件通常属于同一个包。编译器会生成对象文件，包含机器码、类型信息、导出数据和调试信息。导出数据让依赖包无需重新解析源码也能知道公开 API。

### 3. 构建缓存

构建缓存按源码内容、编译参数、环境、依赖和工具链版本计算。缓存命中能加速构建，但如果你依赖生成文件、环境变量或外部命令副作用，可能出现缓存结果与预期不一致。

常用排查：

```powershell
go clean -cache
go clean -testcache
go build -x ./...
go list -json ./...
```

## 链接阶段

链接器把各包对象、runtime、符号表和元数据组合成可执行文件。常见生产用法是注入版本信息：

```powershell
go build -ldflags "-X 'main.version=1.2.3' -X 'main.commit=abc123'" ./cmd/app
```

`-X` 只能设置字符串变量，变量必须可被 linker 定位。若变量被优化、路径写错或不是 string，注入会失败。

## 包初始化顺序

初始化顺序是 Go 深水区高频点：

1. 先初始化被导入的包。
2. 一个包只初始化一次。
3. 包内先初始化包级变量，再执行 `init`。
4. 多个文件内的初始化顺序由编译器按文件名顺序处理，但不应依赖跨文件细节。
5. 同一文件内按声明顺序初始化。
6. 所有依赖包初始化完成后，才执行 `main` 包初始化，最后进入 `main.main`。

示例：

```go
var a = f("a")

func init() {
	fmt.Println("init")
}

func main() {
	fmt.Println("main")
}
```

> **重点：** `init` 适合注册、校验和一次性准备，不适合隐藏复杂业务流程、发网络请求或启动后台 goroutine。

## 运行时启动

程序入口先进入 runtime。runtime 会准备：

- 主 goroutine。
- 调度器结构和 P。
- 栈管理。
- GC 元数据。
- 类型和模块信息。
- defer/panic 支持。
- 网络轮询所需结构。

之后 runtime 创建并运行执行 `main.main` 的 goroutine。`main.main` 返回后，进程结束；其他未退出 goroutine 不会阻止进程结束。

## 例子：构建信息

```go
package main

import "fmt"

var (
	version = "dev"
	commit  = "none"
)

func main() {
	fmt.Println(version, commit)
}
```

```powershell
go build -ldflags "-X main.version=1.0.0 -X main.commit=abc123" .
```

## 排障剧本

| 现象 | 检查点 | 命令 |
| --- | --- | --- |
| CI 编译失败，本地成功 | GOOS/GOARCH、tags、Go 版本、CGO | `go env` |
| 文件修改不生效 | 构建标签、文件后缀、缓存 | `go build -x` |
| 版本号没注入 | 变量路径、变量类型、包名 | `go tool nm app` |
| init 顺序异常 | import 图、全局变量依赖 | `go list -deps` |

## 练习

1. 写两个互相依赖的包，打印包级变量和 `init` 执行顺序。
2. 创建 `feature_prod.go` 和 `feature_dev.go`，用 build tags 切换实现。
3. 用 `-ldflags -X` 注入版本号，并用 `go version -m` 查看二进制模块信息。

## 验收

- 能完整描述 `go build` 的文件选择、编译和链接阶段。
- 能解释包初始化顺序和 `init` 的风险。
- 能用命令定位构建标签、缓存、版本注入和模块信息问题。

## 易错

> **易错：** 在 `init` 中启动后台 goroutine 或访问远程服务。
>
> 正确做法：把副作用放到显式启动流程中，让错误可返回、可测试、可配置。
