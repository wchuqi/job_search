## 项目：任务队列 HTTP 服务

实现一个小型任务队列服务，用于提交任务、查询状态、并发执行任务和观察运行状态。

## 功能要求

- `POST /tasks` 提交任务，返回任务 ID。
- `GET /tasks/{id}` 查询状态：pending、running、success、failed、canceled。
- Worker pool 限制并发数。
- 支持服务关闭时停止接收新任务，等待运行中任务收尾。
- 使用 context 控制超时和取消。
- 持久化可先用内存 map，再扩展为数据库。

## 工程要求

- 目录包含 `cmd/`、`internal/server`、`internal/service`、`internal/repository`。
- 表驱动测试覆盖状态流转、错误路径和并发提交。
- 使用 `go test -race ./...` 无竞态。
- 提供 benchmark 衡量提交和查询性能。
- 提供 pprof 或指标接口观察 goroutine 数、队列长度和任务耗时。

## 验收标准

- 能在 1 分钟内说明包边界设计。
- 能解释任务取消如何传播到 worker。
- 能证明 map 访问被锁保护或由单 goroutine 串行管理。
- 能说明 HTTP client/server 超时设置。
- 能用 pprof 定位一个人为制造的性能热点。

## 重点

- 这个项目综合检查语言、并发、测试、HTTP 和生产意识。
- 不追求功能多，追求边界清楚、行为可测、故障可查。

## 难点

- 状态机、取消、错误和并发同步必须统一设计，否则容易出现任务卡死或状态回退。

## 易错

> **易错：** 只实现正常提交和查询，不处理关闭、超时、失败和重复请求。
>
> 正确做法：先画状态机，再写测试，最后实现。
