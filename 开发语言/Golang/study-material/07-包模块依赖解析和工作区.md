# Golang 学习资料：包、模块、依赖解析和工作区

[返回索引](../Golang学习资料.md)

## 学习目标

- 掌握 package、module、semantic import versioning、`go.mod`、`go.sum`、`replace`、`exclude`、workspace。
- 理解 Go modules 的依赖选择规则。
- 能排查依赖下载、版本冲突和私有仓库问题。

## 理论导读

包是代码组织单位，模块是版本发布单位。一个模块可以包含多个包。Go modules 的关键目标是可复现构建：同样的源码和依赖要求，应在不同机器上选择同样的模块版本。`go.sum` 不是 lockfile，但保存模块内容校验，防止依赖被篡改。

## 核心心智模型

依赖解析像合并多张采购清单。Go 使用 Minimal Version Selection：对每个模块路径，选择依赖图中被要求的最高版本，而不是像某些包管理器那样求“满足约束的最新版本”。

## 知识点详解

### 版本选择

- `require` 声明模块最低版本要求。
- 如果 A 要求 X v1.2.0，B 要求 X v1.4.0，构建选择 X v1.4.0。
- 主版本 v2+ 通常必须体现在模块路径中，例如 `example.com/lib/v2`。
- `replace` 只在当前主模块生效，不会传递给依赖的使用者。

### 工作区

`go.work` 适合本地同时开发多个模块。它不应被用来掩盖模块边界不清的问题。

## 例子

```powershell
go mod init example.com/app
go get github.com/google/uuid@latest
go mod tidy
go list -m all
go mod why github.com/google/uuid
```

## 练习

创建两个模块 `lib` 和 `app`，使用 `go work` 本地联调。然后移除 workspace，改用正式版本或 `replace` 验证行为差异。

## 验收

- 能解释 MVS 的“取最高要求版本”规则。
- 能说明 `go.sum` 与 lockfile 的差异。
- 能排查私有模块需要的 `GOPRIVATE`、`GONOSUMDB`、`GOPROXY`。

## 重点

- 模块路径是公开契约，尤其 v2+ 路径不能随意设计。
- `replace` 是本地和迁移工具，不是长期依赖治理方案。

## 难点

- 间接依赖升级可能改变行为，但 Go 不会自动求最新补丁，必须主动升级。

## 易错

> **易错：** 提交指向本地磁盘路径的 `replace`。
>
> 正确做法：提交前检查 `go.mod`，共享分支只保留团队都能解析的模块路径。
