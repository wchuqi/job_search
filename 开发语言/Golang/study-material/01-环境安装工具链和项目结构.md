# Golang 学习资料：环境安装、工具链和项目结构

[返回索引](../Golang学习资料.md)

## 学习目标

- 安装 Go，理解 `GOROOT`、`GOPATH`、module cache 和 workspace。
- 掌握 `go run`、`go build`、`go test`、`go fmt`、`go vet`、`go env`。
- 能设计清晰的 Go 项目目录。

## 理论导读

Go 的工具链把“写代码到交付”的基础流程收进一个 `go` 命令。它不是只负责编译，还负责解析模块、下载依赖、运行测试、生成覆盖率、格式化、检查常见问题和交叉编译。统一工具链的价值是让团队成员在不同机器上得到更接近的构建结果。

`GOROOT` 是 Go 自身安装位置，通常不需要手工改。`GOPATH` 在 module 时代主要用于保存下载缓存和二进制工具。项目真正的身份由 `go.mod` 决定。

## 核心心智模型

把 `go` 命令理解成项目管家：它先从当前目录向上找 `go.mod`，确定模块根目录，再根据 import 路径、版本要求和环境变量决定源码、依赖、缓存和输出。

## 知识点详解

### 常用命令

| 命令 | 作用 | 常见场景 |
| --- | --- | --- |
| `go version` | 查看版本 | 确认环境 |
| `go env` | 查看工具链配置 | 排查代理、缓存、交叉编译 |
| `go mod init` | 初始化模块 | 新项目 |
| `go mod tidy` | 同步依赖清单 | 增删 import 后 |
| `go test ./...` | 跑全部测试 | 提交前 |
| `go test -race ./...` | 竞态检测 | 并发代码 |
| `go build ./...` | 编译所有包 | CI |
| `go install pkg@version` | 安装工具 | 安装 CLI |

### 推荐目录

```text
myservice/
  go.mod
  cmd/myservice/main.go
  internal/config/
  internal/server/
  internal/service/
  internal/repository/
  pkg/client/
```

`cmd/` 放入口，`internal/` 放本模块私有包，`pkg/` 只放确实希望外部复用的 API。不要为了显得“架构完整”提前创建一堆空目录。

## 例子

```powershell
go mod init example.com/hello
go run ./cmd/hello
go test ./...
go env GOPROXY GOMOD GOMODCACHE
```

## 练习

创建一个模块，写一个 `cmd/app` 入口和 `internal/config` 包。配置包从环境变量读取端口，入口包只调用配置包和启动函数。

## 验收

- 能解释 `go.mod` 与 `go.sum` 的职责。
- 能用 `go env` 排查代理和缓存位置。
- 能说明为什么 `internal/` 能限制外部导入。

## 重点

- `go fmt` 是团队风格基线，不应在格式上自定义争论。
- `go test ./...` 是 Go 项目的最小质量门槛。

## 难点

- 多模块仓库、workspace 和 replace 容易让本地环境与 CI 不一致。

## 易错

> **易错：** 把业务公共代码随手放到 `pkg/`，导致 API 被外部依赖后难以修改。
>
> 正确做法：默认放 `internal/`，只有稳定且有复用价值的包才公开。
