# Golang 学习资料：生产事故案例和排障剧本深度版

[返回索引](../Golang学习资料.md)

## 学习目标

- 掌握 Go 服务常见生产事故的症状、根因和处置顺序。
- 能把 goroutine、heap、CPU、HTTP、数据库和发布问题串成排障证据链。
- 能形成复盘和预防清单。

## 事故 1：goroutine 持续上涨

### 症状

- goroutine 数从几千涨到几十万。
- RSS 上涨，P99 延迟升高。
- goroutine profile 大量停在 channel receive 或 HTTP client。

### 常见根因

- 下游请求没有超时。
- worker 退出后上游发送者仍阻塞。
- ticker 没 stop。
- response body 没关闭。
- 后台 goroutine 没监听 context。

### 处置

1. 抓 goroutine profile。
2. 按堆栈聚类，找数量最多的等待点。
3. 查该等待点是否有 context、超时和关闭路径。
4. 临时限流或重启止血。
5. 修复取消传播和资源关闭。

## 事故 2：内存上涨并触发 OOM

### 症状

- 容器被 OOMKill。
- Go heap profile 显示某 map 或 slice 持续增长。
- 或 heap 不高但 RSS 高。

### 常见根因

- 无界缓存。
- 切片保留大数组。
- goroutine 泄漏间接保留对象。
- CGO 或 mmap 占用非 Go 堆。
- 大响应体一次性读入内存。

### 处置

1. 分清 RSS、heap、container working set。
2. 抓 heap profile 和 goroutine profile。
3. 对比增长对象。
4. 加缓存上限、流式处理、释放引用。
5. 设置合理 `GOMEMLIMIT`，但不能替代修复泄漏。

## 事故 3：HTTP 调用放大故障

### 症状

- 下游变慢后，上游线程或 goroutine 堆积。
- 错误率和重试量同时上升。
- 连接池耗尽，整体吞吐下降。

### 根因

- 没有整体 timeout。
- 重试没有幂等约束和退避。
- 没有限流和熔断。
- 上游 deadline 大于网关或下游 timeout，造成资源滞留。

### 处置

1. 暂停或降低重试。
2. 加客户端整体超时和响应头超时。
3. 对高成本接口限流。
4. 按链路统一 timeout budget。
5. 给重试增加指数退避、抖动和最大次数。

## 事故 4：发布后 CPU 飙高

### 症状

- 发布后 CPU 突增，吞吐未增加。
- pprof 指向 JSON、正则、日志格式化或锁竞争。

### 根因

- 新增高频日志格式化。
- 正则或解析逻辑进入热点路径。
- 锁粒度扩大。
- 反复 string/[]byte 转换。
- 重试风暴。

### 处置

1. 回滚或降级开关止血。
2. 抓 CPU profile。
3. 对比发布差异。
4. 用 benchmark 复现热点。
5. 修复后压测验证。

## 事故 5：map 并发读写崩溃

### 症状

运行时报：

```text
fatal error: concurrent map read and map write
```

### 根因

普通 map 被多个 goroutine 读写且没有同步。

### 处置

1. 定位 map 访问路径。
2. 用 `go test -race` 补测试。
3. 根据场景使用 Mutex、RWMutex、sync.Map 或单 goroutine owner。
4. 将检查和写入放进同一临界区，避免逻辑竞争。

## 通用复盘模板

- 触发条件：流量、发布、配置、下游、数据形态。
- 影响范围：接口、实例、用户、时间。
- 直接原因：代码、配置或依赖。
- 深层原因：测试、监控、容量、发布流程。
- 止血动作：回滚、限流、扩容、开关。
- 根治动作：代码修复、测试补充、告警、演练。

## 验收

- 能根据 profile 堆栈提出明确根因假设。
- 能区分止血和根治。
- 能写出事故复盘里的预防动作。

## 易错

> **易错：** 事故中一边猜测一边改多个变量。
>
> 正确做法：先止血，再保留证据；每次只验证一个主要假设。
