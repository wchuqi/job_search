# Golang 学习资料：GC、三色标记、写屏障、GOGC 和内存限制深度解析

[返回索引](../Golang学习资料.md)

## 学习目标

- 理解 Go GC 的目标、三色标记、写屏障和 STW 阶段。
- 掌握 `GOGC`、`GOMEMLIMIT`、heap profile、runtime metrics 的含义。
- 能区分内存泄漏、缓存增长、GC 目标堆、RSS 和容器内存压力。

## 理论导读

Go GC 的核心目标是低停顿和可接受吞吐，而不是最小内存。GC 回收的是不可达对象，但程序内存占用还受缓存、栈、mspan/mcache、操作系统归还策略、CGO、文件映射和容器统计影响。线上看到 RSS 高，不等于 Go 堆泄漏；看到 GC 频繁，也不一定是 GC 参数问题，可能是分配速率太高。

## 三色标记

对象颜色：

- 白色：尚未发现，标记结束仍为白则可回收。
- 灰色：已发现，子对象未扫描完。
- 黑色：自己和子引用已扫描。

并发标记时，用户 goroutine 仍在修改对象引用。写屏障用于记录引用变化，避免“黑色对象指向白色对象但 GC 不知道”这类错误。

## Go GC 周期

简化流程：

1. STW 短暂停顿，开启写屏障。
2. 并发标记，大部分工作和应用同时运行。
3. 标记终止，短暂停顿完成收尾。
4. 并发清扫，把未使用 span 归还给分配器。

> **重点：** Go GC 的主要停顿通常不随堆大小线性增长，但 GC CPU 成本和扫描工作仍与活对象、指针数量、分配速率有关。

## GOGC 和 GOMEMLIMIT

`GOGC` 控制下一次 GC 目标堆增长比例。默认 100 表示当新分配堆大约达到上次存活堆大小的 100% 时触发下一轮。值越低，GC 越频繁，内存更低；值越高，GC 更少，内存更高。

`GOMEMLIMIT` 是 runtime 的软内存限制，适合容器环境。它不是硬限制，不能替代容器 limit 和容量规划。

## 排查指标

| 指标 | 含义 | 常见误判 |
| --- | --- | --- |
| heap inuse | Go 堆正在使用 | 不等于 RSS |
| heap alloc | 当前已分配且存活对象 | 上升可能是缓存 |
| heap objects | 存活对象数量 | 多小对象会增加 GC 成本 |
| GC CPU fraction | GC CPU 占比 | 高通常先看分配速率 |
| RSS | 进程常驻内存 | 包含非 Go 堆 |

## 例子：开启 GC trace

```powershell
$env:GODEBUG="gctrace=1"
go run ./cmd/app
```

输出可观察 GC 触发、堆大小、暂停时间和 CPU 消耗。生产中更推荐 runtime metrics、pprof 和监控系统。

## 内存上涨排障剧本

1. 确认是 RSS、Go heap、容器 working set 还是业务指标上涨。
2. 抓两份 heap profile，对比增长对象。
3. 查看 goroutine profile，排除 goroutine 泄漏。
4. 检查无界 map、缓存、slice 持有大数组、连接未关闭。
5. 检查 CGO、mmap、压缩库、数据库驱动等非 Go 堆。
6. 决定是修代码、限制缓存、调 `GOGC/GOMEMLIMIT`，还是扩容。

## 练习

1. 构造无界 map 缓存，观察 heap profile。
2. 构造 `small := big[:10]` 保留大数组，修复后比较。
3. 调整 `GOGC=50/100/300`，比较 GC 次数和内存。

## 验收

- 能解释三色标记和写屏障为什么存在。
- 能说明 `GOGC` 调低和调高的代价。
- 能从 profile 区分泄漏和缓存。

## 易错

> **易错：** 线上内存高时第一反应是把 `GOGC` 调低。
>
> 正确做法：先确认增长来源。若是无界缓存或泄漏，调 GC 只能增加 CPU，不能解决根因。
