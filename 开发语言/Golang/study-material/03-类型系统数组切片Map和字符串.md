# Golang 学习资料：类型系统、数组、切片、Map 和字符串

[返回索引](../Golang学习资料.md)

## 学习目标

- 掌握基本类型、命名类型、类型别名、数组、切片、map、字符串。
- 理解切片三元组、底层数组、容量、扩容和共享。
- 理解 map 的 nil、并发不安全和迭代无序。

## 理论导读

Go 的类型系统强调清楚的内存和行为边界。数组是定长值，切片是对数组片段的描述，map 是运行时管理的哈希表，字符串是不可变字节序列。很多线上 bug 来自误以为“切片就是动态数组”或“map 读写天然安全”。

## 核心心智模型

切片不是数据本体，而是一个包含指针、长度和容量的窗口。多个切片可能看向同一块底层数组，一个窗口写入会影响另一个窗口能看到的数据。

## 知识点详解

### 切片

```go
s := []int{1, 2, 3}
t := s[:2]
t[0] = 99
fmt.Println(s) // [99 2 3]
```

`append` 可能复用原数组，也可能分配新数组。依赖它是否扩容会造成隐蔽 bug。

### Map

```go
counts := map[string]int{}
counts["go"]++
```

nil map 可以读，不能写。map 迭代顺序不稳定，不要把它用于生成有序输出。

### 字符串

字符串按字节存储，`len("中")` 返回字节数，不是字符数。遍历 Unicode 字符应使用 `range` 得到 rune。

## 练习

实现一个 `TopNWords(text string, n int) []string`，统计词频并按次数和字典序排序。

## 验收

- 能画出切片的 ptr、len、cap。
- 能解释 `nil` 切片和空切片在 JSON 编码中的差异。
- 能说明 map 为什么不能并发读写。

## 重点

- 切片共享底层数组是 Go 高频考点。
- map 不是线程安全容器。

## 难点

- `append` 扩容策略不是业务契约，不能写依赖具体容量增长的逻辑。

## 易错

> **易错：** 返回大数组上的小切片，导致大数组长期不能被 GC。
>
> 正确做法：需要保留小片段时复制到新切片。
