# Java设计模式学习资料：深度实验手册和能力验收

[返回索引](../Java设计模式学习资料.md)

## 学习目标

- 通过实验验证设计模式的运行时行为。
- 用测试、调用链和失败用例证明自己真正掌握模式。
- 建立面试和项目都能复用的能力验收标准。

## 实验 1：策略注册表

### 任务

实现一个 `PaymentStrategyRegistry`：

- 支持多个支付策略。
- key 大小写归一化。
- 重复 key 启动失败。
- 未知 key 抛明确异常。
- 支持默认策略，但默认策略必须显式配置。

### 验收

- `ALI`、`ali` 都能命中同一策略。
- 两个策略返回同一 key 时测试失败。
- 未知渠道不会静默走空逻辑。
- 新增策略类不改注册表代码。

## 实验 2：责任链顺序和短路

### 任务

实现风控链：

- 黑名单规则 order=10。
- 限额规则 order=20。
- 频率规则 order=30。
- 命中拒绝后短路。
- 每条规则记录耗时。

### 验收

- 规则按 order 执行。
- 黑名单命中时后续规则不执行。
- 相同 order 的行为有明确规则。
- 异常规则能记录规则名和异常原因。

## 实验 3：代理边界

### 任务

实现一个 JDK 动态代理，为接口方法记录日志和耗时。再构造一个内部自调用案例，观察代理是否生效。

### 验收

- 外部通过代理调用时日志生效。
- 目标对象内部调用自身方法时代理不生效。
- `InvocationTargetException` 被正确解包。
- finally 中一定记录耗时。

## 实验 4：装饰器顺序

### 任务

实现 `FileStorage`：

- `RemoteStorage`
- `CompressStorageDecorator`
- `EncryptStorageDecorator`
- `MetricsStorageDecorator`

分别测试：

```text
compress -> encrypt -> remote
encrypt -> compress -> remote
```

### 验收

- 能解释两种顺序的数据格式差异。
- 能证明外层装饰器先接收调用。
- 能避免重复装饰导致重复压缩或重复加密。

## 实验 5：状态迁移并发

### 任务

实现订单状态迁移：

- PENDING -> PAID
- PENDING -> CANCELED
- PAID -> SHIPPED

用乐观锁模拟两个线程同时支付或取消。

### 验收

- 只有一个线程能成功迁移状态。
- 重复支付回调可以幂等成功。
- 非法迁移有明确错误。
- 状态迁移表有测试覆盖。

## 实验 6：事务事件和 Outbox

### 任务

实现支付成功后事件记录：

- 一个事务内更新订单状态并写 `outbox_event`。
- 后台任务扫描未发送事件。
- 消费者处理短信和积分。
- 消费者按 `eventId` 幂等。

### 验收

- 订单更新失败时不产生可发送事件。
- 事件发送失败可重试。
- 消费者重复消费不会重复加积分。
- 能说明这是观察者模式在分布式系统中的工程化版本。

## 实验 7：从大方法重构

### 任务

准备一个包含支付、优惠、风控、通知的大方法，分 4 次提交重构：

1. 提取支付策略。
2. 提取优惠规则。
3. 提取风控责任链。
4. 提取支付成功事件。

### 验收

- 每次提交测试都通过。
- 每次只改变结构，不改变外部行为。
- 最终新增支付渠道不改核心流程。
- 最终新增风控规则不改核心流程。

## 综合能力验收

| 源码理解 | 能解释 JDK/Spring/MyBatis 中的模式组合 |

| 生产意识 | 能处理事务、并发、缓存、事件和幂等 |

## 面试自测题

1. 策略注册表如何处理重复 key？
2. 责任链中一个处理器异常后是否继续？由谁决定？
3. Spring AOP 为什么内部自调用不生效？
4. 缓存装饰器先删缓存还是先写数据库？为什么不能简单回答？
5. 支付成功事件为什么建议事务提交后发布？
6. 状态模式在并发支付回调下如何保证幂等？
7. Java IO 装饰器顺序为什么会影响行为？
8. 什么时候不应该使用设计模式？

## 最终验收

完成学习后，你应该能拿一个业务需求，输出：

- 类图。
- 调用链。
- 状态迁移表。
- 失败处理矩阵。
- 测试清单。
- 模式选择说明。
- 反模式风险说明。

如果只能说出模式定义，说明还停留在浅层；如果能解释运行时顺序、失败路径和扩展成本，才算真正掌握。
