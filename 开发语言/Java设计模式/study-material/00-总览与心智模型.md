# Java设计模式学习资料：总览与心智模型

[返回索引](../Java设计模式学习资料.md)

## 学习目标

- 理解设计模式的本质：在特定上下文中，解决一类反复出现的设计问题。
- 建立 Java 视角的心智模型：接口定义稳定边界，组合承载变化，多态负责分派。
- 先掌握变化点，再学习模式名称。

## 理论导读

设计模式不是语法，也不是框架 API。它更像“代码结构的经验配方”：当系统里某个点会频繁变化时，直接写死会让调用方、被调用方、测试和扩展都互相牵连。模式的作用，是把变化封装到清晰的位置，让稳定部分依赖抽象，让变化部分可以替换。

在 Java 中，模式通常依赖这些基础能力：

- 接口：定义调用方依赖的稳定协议。
- 抽象类：复用骨架流程或部分默认实现。
- 多态：运行时根据具体对象执行不同行为。
- 组合：把对象拼起来，而不是靠继承层级堆功能。
- 泛型：保持类型安全，同时复用算法或容器。
- 反射和动态代理：在框架中延迟绑定、增强行为、实现 AOP。

## 核心心智模型

把软件想成“稳定骨架 + 变化插槽”：

- 稳定骨架：业务流程、调用入口、公共协议、生命周期。
- 变化插槽：算法、实现类、渠道、状态、第三方系统、扩展规则。
- 模式：给插槽命名并约束连接方式。

例子：订单支付流程通常稳定，但支付渠道会变化。直接写：

```java
if ("ALI".equals(type)) {
    payByAli(order);
} else if ("WECHAT".equals(type)) {
    payByWechat(order);
}
```

当渠道增加时，调用方持续修改。策略模式或工厂模式可以把“选择支付方式”和“执行支付行为”隔离出来。

## 学习路线

1. 先学 SOLID 和组合优于继承。
2. 再学创建型模式，理解对象创建的隔离。
3. 接着学结构型模式，理解对象连接方式。
4. 然后学行为型模式，理解职责分配和流程调度。
5. 最后读 JDK/Spring 源码中的模式组合。

## 重点

- 模式是为了控制变化，不是为了增加抽象层。
- Java 模式实现的核心是接口、多态、组合、生命周期和依赖注入。
- 框架中的模式通常是组合使用，不要期待源码长得和教材完全一样。

## 难点

- 同一个业务问题可以用多个模式解决，关键是识别主变化点。
- 代理、装饰器、适配器、外观很容易混淆，因为它们都包了一层对象。
- 策略、状态、命令也容易混淆，因为它们都把行为封装成对象。

## 易错

> **易错：** 先背模式定义，再硬套项目。
>
> 正确做法：先问“哪里在变化、谁不该知道这个变化、扩展时会改哪些代码”。

> **易错：** 认为模式越多代码越好。
>
> 正确做法：模式必须降低真实复杂度。若抽象层让阅读、调试、测试更困难，就要重新评估。

## 练习

找一个你熟悉的业务模块，写出：

- 哪些代码稳定。
- 哪些代码经常变化。
- 当前变化会影响哪些类。
- 可以用哪个模式把变化收拢。

## 验收

- 能不用背定义，解释 23 种模式分别解决什么变化问题。
- 能画出一个模式的参与者、依赖方向和运行时调用链。
- 能说出每个模式的代价和不适用场景。
