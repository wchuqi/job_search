## 一、深度机制

### 1. 策略模式在 Spring 中如何优雅注册？要注意什么？

**参考答案：**

可以让每个策略实现统一接口，并由 Spring 注入 `List<Strategy>` 或 `Map<String, Strategy>`。更推荐策略自己暴露业务 key，然后构建不可变注册表。

要注意重复 key、大小写归一、默认策略、未知 key、初始化失败和策略是否无状态。不要直接把 Spring Bean 名当业务 key，除非团队明确约定。

> **重点：** 面试官通常想听到“注册、选择、冲突、默认、线程安全”，而不只是 `Map`。

### 2. 责任链和策略模式如何选择？

**参考答案：**

如果一次请求只选择一个算法，通常是策略；如果多个处理器按顺序执行，可能短路或叠加处理，通常是责任链。优惠规则如果互斥是策略，如果可叠加是责任链或规则集合。

> **易错：** 看到多个实现就答策略，不分析是否会执行多个。

### 3. Spring AOP 为什么是代理模式？为什么会不生效？

**参考答案：**

Spring AOP 通过代理对象拦截方法调用，在目标方法前后执行增强逻辑。不生效常见原因：内部自调用没有经过代理；方法不是可代理方法；对象不是 Spring 管理的 Bean；切点表达式不匹配；final 类或 final 方法影响 CGLIB 代理。

> **难点：** 注解不是增强本身，调用经过代理才是关键。

### 4. 装饰器和代理代码很像，如何从源码中区分？

**参考答案：**

看意图。装饰器主要是叠加能力，通常可以多层组合，例如缓存、压缩、加密。代理主要是控制访问或横切增强，例如权限、事务、远程调用、延迟加载。两者都可能实现同一接口并持有目标对象。

### 5. 观察者模式用于支付成功通知，有什么生产风险？

**参考答案：**

同步通知会拖慢主流程，通知失败可能导致支付失败；异步通知要处理消息丢失、重复消费、顺序、幂等、事务提交前发布等问题。生产上常用事务消息或 Outbox，把订单状态更新和事件记录放在同一事务里。

## 二、系统设计场景

### 6. 设计一个多支付渠道系统，你会组合哪些模式？

**参考答案：**

支付渠道执行用策略模式，支付客户端创建用工厂或抽象工厂，第三方接口不一致用适配器，统一支付流程可用模板方法，支付前校验用责任链，支付成功副作用用观察者，核心入口可以用外观。

还要补充工程规则：渠道 key 冲突处理、配置加载、签名验签、超时重试、幂等号、回调状态机、事件一致性。

### 7. 设计一个风控系统，责任链如何落地？

**参考答案：**

每个规则实现统一接口，规则按配置或 order 排序执行。上下文记录命中规则、拒绝原因、耗时和中间数据。需要定义是否短路、异常是否拒绝、远程规则超时如何降级、规则是否允许写数据库。

> **重点：** 深度回答要包含排序、短路、异常、可观测性和降级。

### 8. 如何把一个大 if-else 方法重构成设计模式？

**参考答案：**

先补测试，确认行为。再识别分支背后的变化点：算法变化用策略，流程步骤变化用模板方法，多个校验按顺序执行用责任链，状态驱动行为用状态模式。每次只抽一个变化点，保持行为不变，最后删除旧分支。

### 9. 设计模式会不会影响性能？

**参考答案：**

会。额外抽象会增加调用层级，动态代理和反射有额外成本，责任链过长会增加延迟，装饰器链可能重复处理数据。但多数业务系统里可维护性收益更重要。高频路径要压测，用指标证明问题，而不是凭感觉优化。

### 10. 什么时候不应该使用设计模式？

**参考答案：**

变化点不明确、分支很少且稳定、团队理解成本明显高于收益、只是为了展示技巧、当前没有测试保护时，都不应贸然引入复杂模式。设计模式的目标是降低变化成本，不是增加类数量。

## 三、深水区追问

### 11. 缓存装饰器更新数据时如何保证一致性？

**参考答案：**

没有一个简单顺序能在所有场景绝对保证一致。常见做法是先更新数据库，再删除缓存；配合事务提交后删除、延迟双删、订阅 binlog、版本号、短 TTL 和幂等重试。要结合读写并发、业务容忍度和数据新鲜度要求选择。

### 12. 状态模式如何处理并发回调？

**参考答案：**

用状态迁移表定义合法迁移，并在数据库更新时带上当前状态或版本号条件。重复回调做幂等，非法迁移返回业务错误，并发冲突按业务重试或忽略。

### 13. Outbox 和观察者模式是什么关系？

**参考答案：**

观察者模式解决发布者和订阅者解耦，Outbox 是它在分布式事务场景下的工程化落地。它把“业务数据更新”和“事件记录”放在同一事务里，再异步投递，避免事务提交前发事件或提交后发事件失败。

## 面试判断标准

- 初级回答：能说模式定义和简单例子。
- 中级回答：能写 Java 代码并区分相似模式。
- 高级回答：能说运行时顺序、失败路径、并发事务边界和框架源码里的组合方式。
