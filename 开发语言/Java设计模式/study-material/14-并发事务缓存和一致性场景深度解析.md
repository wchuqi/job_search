# Java设计模式学习资料：并发、事务、缓存和一致性场景深度解析

[返回索引](../Java设计模式学习资料.md)

## 学习目标

- 理解设计模式在生产场景中会遇到的并发、事务、缓存、消息和一致性问题。
- 能判断某个模式实现是否线程安全、是否破坏事务边界、是否存在重复执行风险。
- 能为观察者、责任链、策略、代理、装饰器设计失败处理和可观测性。

## 理论导读

设计模式不是只在内存里组织类。一旦进入 Web 服务和分布式系统，模式会和这些问题绑定：

- 多线程访问单例对象。
- 数据库事务提交和事件发布顺序。
- 缓存装饰器和数据库更新的一致性。
- 责任链中某个处理器失败后的补偿。
- 策略热更新和运行中请求的可见性。
- 代理事务和内部调用边界。

如果这些问题不处理，模式结构再漂亮也会出生产事故。

## 单例、策略和线程安全

策略类通常应该设计成无状态：

```java
class VipDiscountStrategy implements DiscountStrategy {
    public BigDecimal apply(Order order) {
        return order.amount().multiply(new BigDecimal("0.8"));
    }
}
```

危险写法：

```java
class BadDiscountStrategy implements DiscountStrategy {
    private Order currentOrder;

    public BigDecimal apply(Order order) {
        this.currentOrder = order;
        return calculate();
    }
}
```

如果策略被 Spring 单例管理，多线程请求会共享 `currentOrder`。

安全原则：

- 单例 Bean 尽量无状态。
- 可变状态放局部变量。
- 需要缓存时使用线程安全结构并定义过期策略。
- 使用 ThreadLocal 必须在 finally 中清理。

## 事务代理和模式边界

典型支付流程：

```java
@Transactional
public void pay(String orderId) {
    orderRepository.markPaid(orderId);
    eventPublisher.publishEvent(new OrderPaidEvent(orderId));
}
```

问题：事件可能在事务提交前被监听器处理。如果监听器查询数据库，可能读不到已提交数据。

更稳的做法：

- 使用事务提交后事件。
- 或写 Outbox 表，再异步投递消息。

```text
事务内：更新订单 + 插入 outbox_event
事务外：任务扫描 outbox_event -> 发送 MQ -> 标记已发送
消费者：幂等处理通知、积分、发票
```

## 观察者和事件一致性

同步观察者适合进程内轻量扩展：

```text
创建订单 -> 同步校验 -> 同步记录审计日志
```

异步观察者适合非关键链路：

```text
支付成功 -> 发短信、加积分、生成发票、数据分析
```

设计矩阵：

| 问题 | 同步事件 | 异步事件 |
| --- | --- | --- |
| 发布者是否等待 | 等待 | 不等待 |
| 失败影响主流程 | 通常影响 | 通常不影响 |
| 一致性 | 强一些 | 最终一致 |
| 复杂度 | 低 | 高 |
| 必须处理 | 异常传播 | 重试、幂等、顺序、死信 |

## 缓存装饰器和一致性

缓存常用装饰器实现：

```java
class CachedProductRepository implements ProductRepository {
    private final ProductRepository delegate;
    private final Cache cache;

    public Product findById(long id) {
        String key = "product:" + id;
        Product cached = cache.get(key, Product.class);
        if (cached != null) {
            return cached;
        }
        Product product = delegate.findById(id);
        cache.set(key, product, Duration.ofMinutes(10));
        return product;
    }

    public void update(Product product) {
        delegate.update(product);
        cache.delete("product:" + product.id());
    }
}
```

深层问题：

- 先删缓存还是先写数据库。
- 数据库提交失败时缓存是否已删除。
- 并发读写下是否出现旧值回填。
- 缓存穿透、击穿、雪崩如何处理。
- key 设计是否包含租户、语言、权限维度。

> **重点：** 缓存装饰器不是只包一层 Map，生产场景要处理一致性和失效时序。

## 责任链和事务边界

风控责任链中每个规则可能读取数据库、调用远程服务或写审计日志。

要明确：

- 规则是否允许写数据库。
- 写操作是否在主事务内。
- 远程调用失败是否拒绝、放行还是降级。
- 规则执行时间是否有限制。
- 每个规则是否记录命中和耗时。

推荐上下文：

```java
class RiskContext {
    private final List<String> matchedRules = new ArrayList<>();
    private final Map<String, Object> attributes = new HashMap<>();
    private boolean rejected;
    private String rejectReason;

    void reject(String rule, String reason) {
        this.rejected = true;
        this.rejectReason = reason;
        this.matchedRules.add(rule);
    }
}
```

但上下文不要变成隐式全局变量。处理器之间依赖上下文字段时，要写入契约文档或封装成类型安全方法。

## 策略热更新和可见性

如果策略可由配置中心热更新，不能随意修改共享 Map：

```java
class RoutingTable {
    private final AtomicReference<Map<String, RouteStrategy>> ref =
            new AtomicReference<>(Map.of());

    RouteStrategy get(String key) {
        return ref.get().get(key);
    }

    void reload(List<RouteStrategy> strategies) {
        Map<String, RouteStrategy> newTable = buildImmutableMap(strategies);
        ref.set(newTable);
    }
}
```

好处：

- 请求看到的是完整旧表或完整新表。
- 不会读到更新一半的 Map。
- 不需要对每次读加锁。

## 状态模式和并发更新

订单状态流转必须防止并发事件重复处理：

```sql
update orders
set status = 'PAID', version = version + 1
where id = ? and status = 'PENDING' and version = ?
```

如果返回 0 行，说明状态已变，应该根据业务判断：

- 重复支付回调：幂等返回成功。
- 非法状态迁移：抛业务异常。
- 并发冲突：重试或提示稍后再试。

## 代理和可观测性

代理常用于日志、指标、链路追踪：

```java
class MetricsProxy implements InvocationHandler {
    private final Object target;

    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        long start = System.nanoTime();
        try {
            return method.invoke(target, args);
        } catch (InvocationTargetException e) {
            throw e.getTargetException();
        } finally {
            long cost = System.nanoTime() - start;
            record(method.getName(), cost);
        }
    }
}
```

注意：

- 要解包 `InvocationTargetException`。
- 不要吞异常。
- 指标标签不要包含高基数字段，如用户 id、订单 id。
- ThreadLocal traceId 必须清理。

## 重点

- 模式实现要考虑线程安全、事务边界、失败处理和可观测性。
- 同步观察者简单，但会影响主流程；异步观察者解耦，但要处理最终一致。
- 缓存装饰器必须设计失效顺序和并发回填。

## 难点

- 事务提交和事件发布的顺序是高频事故点。
- 状态模式在单线程内很简单，在并发回调下必须做幂等。
- 策略热更新要保证读到完整快照。

## 易错

> **易错：** 在单例策略中保存请求临时字段。
>
> 正确做法：策略无状态化，临时数据放方法参数或局部变量。

> **易错：** 支付事务中直接同步发送短信，短信失败导致支付回滚。
>
> 正确做法：支付核心链路只处理核心状态，通知类副作用异步化并幂等。

## 练习

设计“支付成功事件”：

1. 订单状态更新和事件记录必须在同一事务。
2. 短信、积分、发票异步消费。
3. 消费者支持幂等。
4. 消息失败可重试并进入死信。
5. 每个处理器记录耗时和结果。

## 验收

- 能说清事务边界和事件发布顺序。
- 能设计幂等 key。
- 能解释缓存更新和删除顺序。
- 能指出哪些模式实现必须无状态。
