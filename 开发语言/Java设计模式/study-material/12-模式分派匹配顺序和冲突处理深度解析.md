# Java设计模式学习资料：模式分派、匹配顺序和冲突处理深度解析

[返回索引](../Java设计模式学习资料.md)

## 学习目标

- 掌握策略、责任链、代理、装饰器、状态、观察者在运行时的分派和排序规则。
- 能设计清晰的冲突处理机制，避免“看类图没问题，运行时一团乱”。
- 能在面试中解释模式的执行路径、异常传播和边界条件。

## 理论导读

设计模式的浅层是静态结构，深层是运行时决策。一个系统是否可维护，往往取决于这些问题是否清楚：

- 多个实现同时存在时选谁。
- 多个处理器按什么顺序执行。
- 一个处理器失败后是否继续。
- 包装对象的内外层顺序是什么。
- 事件监听器是否同步、是否有事务边界。
- 状态迁移冲突如何处理。

## 策略模式的分派规则

策略模式必须有明确选择算法。常见方式：

| 选择方式 | 例子 | 优点 | 风险 |
| --- | --- | --- | --- |
| 枚举 switch | `PayChannel.ALI` | 显式、简单 | 扩展要改分支 |
| Map 注册 | `Map<String, Strategy>` | 扩展方便 | key 冲突、默认策略 |
| 注解扫描 | `@PayType("ALI")` | 声明式 | 反射和扫描复杂 |
| Spring 注入 | `Map<String, PayStrategy>` | 容器管理 | Bean 名和业务 key 混淆 |
| 配置路由 | 配置中心决定 | 可动态调整 | 热更新一致性 |

### 推荐的策略注册表

```java
interface TypedStrategy {
    String type();
}

final class StrategyRegistry<T extends TypedStrategy> {
    private final Map<String, T> strategies;

    StrategyRegistry(List<T> strategies) {
        this.strategies = strategies.stream().collect(Collectors.toUnmodifiableMap(
                strategy -> normalize(strategy.type()),
                Function.identity(),
                (a, b) -> {
                    throw new IllegalStateException("duplicate strategy type: " + a.type());
                }));
    }

    T require(String type) {
        T strategy = strategies.get(normalize(type));
        if (strategy == null) {
            throw new IllegalArgumentException("unsupported strategy type: " + type);
        }
        return strategy;
    }

    private String normalize(String type) {
        return type.trim().toUpperCase(Locale.ROOT);
    }
}
```

这个注册表比普通 Map 更可靠，因为它定义了：

- key 规范化。
- 重复 key 启动失败。
- 找不到策略时抛明确异常。
- 对外不暴露可变 Map。

## 责任链的执行模型

责任链不只是 `for` 循环。它至少有四种模型：

| 模型 | 执行规则 | 典型场景 |
| --- | --- | --- |
| 全量执行链 | 所有处理器都执行 | 日志、数据补充 |
| 短路链 | 命中条件后停止 | 风控、权限校验 |
| 过滤器链 | 前置和后置包围执行 | Servlet Filter |
| 拦截器链 | pre/post/afterCompletion | Spring MVC |

### 短路链示例

```java
interface RiskRule {
    RiskDecision check(Order order);
}

record RiskDecision(boolean rejected, String reason) {
    static RiskDecision pass() { return new RiskDecision(false, ""); }
    static RiskDecision reject(String reason) { return new RiskDecision(true, reason); }
}

class RiskEngine {
    private final List<RiskRule> rules;

    RiskEngine(List<RiskRule> rules) {
        this.rules = rules;
    }

    RiskDecision check(Order order) {
        for (RiskRule rule : rules) {
            RiskDecision decision = rule.check(order);
            if (decision.rejected()) {
                return decision;
            }
        }
        return RiskDecision.pass();
    }
}
```

### 责任链排序

排序必须显式：

```java
interface OrderedRiskRule extends RiskRule {
    int order();
}

this.rules = rules.stream()
        .sorted(Comparator.comparingInt(OrderedRiskRule::order))
        .toList();
```

排序冲突要定义：

- 相同 order 是否允许。
- 是否按类名二次排序保证稳定。
- 配置缺失时默认排到哪里。
- 紧急规则是否可以插队。

## 代理的分派边界

代理模式的关键不是“有一个 Proxy 类”，而是调用是否经过代理对象。

```text
Controller
  -> userServiceProxy.transactionalMethod()
      -> TransactionInterceptor
      -> target.transactionalMethod()
```

内部自调用问题：

```java
class UserService {
    public void outer() {
        inner(); // 不经过代理，事务/缓存/AOP 可能不生效
    }

    @Transactional
    public void inner() {}
}
```

解决方式：

- 把 `inner` 移到另一个 Bean。
- 通过代理对象调用。
- 调整事务边界，让入口方法直接承载事务。

> **重点：** 代理模式的执行边界由“对象引用是谁”决定，不由注解本身决定。

## 装饰器的包装顺序

装饰器顺序会改变语义。

```text
new MetricsStorage(
  new CompressStorage(
    new EncryptStorage(
      new RemoteStorage()
    )
  )
)
```

调用顺序：

```text
Metrics.save
  -> Compress.save
    -> Encrypt.save
      -> Remote.save
```

如果先压缩再加密，压缩率好；如果先加密再压缩，压缩几乎失效。设计装饰器时必须写清楚顺序规则。

## 状态模式的迁移表

状态模式不能只写几个状态类，还要有迁移表。

| 当前状态 | 事件 | 下一个状态 | 动作 |
| --- | --- | --- | --- |
| PENDING | PAY | PAID | 记录支付流水 |
| PENDING | CANCEL | CANCELED | 释放库存 |
| PAID | SHIP | SHIPPED | 创建物流单 |
| PAID | REFUND | REFUNDING | 发起退款 |
| SHIPPED | CONFIRM | FINISHED | 完成订单 |

状态迁移要处理：

- 非法事件抛业务异常还是忽略。
- 并发事件谁先成功。
- 状态更新是否需要乐观锁。
- 动作失败时状态是否回滚。

```java
int updated = orderRepository.updateStatus(
        orderId,
        expectedStatus,
        nextStatus
);
if (updated == 0) {
    throw new ConcurrentModificationException("order status changed");
}
```

## 观察者的顺序和一致性

本地观察者：

```text
pay success
  -> update order
  -> publish event
  -> send sms
  -> add points
```

如果短信失败，支付是否失败？如果积分失败，是否回滚订单？这些必须设计清楚。

生产系统常用拆法：

```text
本地事务：更新订单 + 写 outbox_event
异步任务：扫描 outbox_event -> 发送 MQ
消费者：短信、积分、发票各自幂等消费
```

观察者模式进入分布式系统后，重点变成：

- 事件是否在事务提交后发布。
- 消费者是否幂等。
- 失败是否重试和告警。
- 消息顺序是否有要求。
- 是否需要补偿任务。

## 模板方法的扩展点冲突

模板方法固定流程时，要明确定义钩子语义：

```java
abstract class ImportTemplate {
    public final ImportResult run(Path file) {
        beforeRead(file);
        List<Row> rows = read(file);
        validate(rows);
        ImportResult result = save(rows);
        afterSave(result);
        return result;
    }

    protected void beforeRead(Path file) {}
    protected abstract List<Row> read(Path file);
    protected abstract void validate(List<Row> rows);
    protected abstract ImportResult save(List<Row> rows);
    protected void afterSave(ImportResult result) {}
}
```

需要规定：

- 子类能否跳过校验。
- 钩子异常是否中断流程。
- 资源打开关闭由谁负责。
- 模板方法是否 `final`，避免子类破坏骨架。

## 重点

- 模式的深层质量体现在运行时规则，而不是类名。
- 策略要定义 key、重复、默认、缺失。
- 责任链要定义顺序、短路、异常、上下文。
- 代理要定义调用是否经过代理对象。
- 观察者要定义事务边界、幂等和失败处理。

## 难点

- 多个模式组合时，顺序会叠加，例如代理外面还有责任链和事件发布。
- 异步观察者很容易从设计模式问题变成一致性问题。
- 状态迁移在并发下必须配合版本号或数据库条件更新。

## 易错

> **易错：** 设计类图时没有设计运行时顺序。
>
> 正确做法：为每个模式补一张调用链或时序图。

> **易错：** 策略缺失时静默走默认逻辑。
>
> 正确做法：只有业务明确允许默认策略时才默认，否则快速失败。

## 练习

为订单系统写出这些规则：

1. 支付策略 key 如何匹配。
2. 风控责任链如何排序，是否短路。
3. 缓存装饰器和权限代理谁在外层。
4. 支付成功事件同步还是异步。
5. 状态迁移并发冲突如何处理。

## 验收

- 能画出完整调用链。
- 能列出每个模式的冲突处理规则。
- 能写测试覆盖重复策略、责任链短路、代理不生效、非法状态迁移。
