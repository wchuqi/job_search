# Java设计模式学习资料：框架源码中的模式组合深度解析

[返回索引](../Java设计模式学习资料.md)

## 学习目标

- 从源码结构理解 JDK、Spring、MyBatis、Servlet 中的模式组合。
- 能说明一个请求或一次方法调用经过哪些工厂、代理、模板、责任链和策略。
- 避免只背“某框架用了某模式”的浅层答案。

## 理论导读

真实框架很少把某个模式孤立呈现。它们通常把多个模式组合成一条运行时管线：

```text
元数据读取 -> 工厂创建 -> 生命周期回调 -> 代理增强 -> 策略匹配 -> 责任链执行 -> 模板封装资源 -> 事件通知
```

因此读源码时要看“谁在什么时候调用谁”，而不是只看类名。

## Java IO：装饰器链

典型写法：

```java
try (InputStream in = new BufferedInputStream(
        new GZIPInputStream(
                Files.newInputStream(path)))) {
    byte[] bytes = in.readAllBytes();
}
```

运行时调用：

```text
BufferedInputStream.read
  -> GZIPInputStream.read
    -> FileInputStream.read
```

深层理解：

- 每一层都实现同一接口或继承同一抽象类。
- 外层控制调用入口。
- 内层提供真实数据源或下一级转换。
- 关闭资源时也按包装链传播。

> **易错：** 只说 Java IO 是装饰器，不说明包装顺序会影响功能语义。

## JDK 动态代理：代理模式和调用分派

```java
UserService proxy = (UserService) Proxy.newProxyInstance(
        classLoader,
        new Class<?>[]{UserService.class},
        (p, method, args) -> {
            long start = System.nanoTime();
            try {
                return method.invoke(target, args);
            } finally {
                System.out.println(method.getName() + " cost " + (System.nanoTime() - start));
            }
        });
```

调用链：

```text
client.call(proxy)
  -> generated proxy method
  -> InvocationHandler.invoke
  -> Method.invoke(target)
  -> target method
```

关键边界：

- JDK 动态代理基于接口。
- 代理对象和目标对象不是同一个对象。
- `equals`、`hashCode`、`toString` 也会进入 `InvocationHandler`。
- 反射调用异常会被包装，需要解包处理。

## Spring BeanFactory：工厂模式的生产级形态

`BeanFactory` 不只是 `create()`，它还管理：

- Bean 定义解析。
- 单例缓存。
- 依赖注入。
- 初始化回调。
- 后置处理器。
- 作用域。
- 销毁回调。

简化调用链：

```text
ApplicationContext.getBean
  -> BeanFactory.getBean
  -> doGetBean
  -> createBean
  -> populateBean
  -> initializeBean
  -> BeanPostProcessor
  -> exposedObject may be proxy
```

对应模式组合：

| 环节 | 模式 |
| --- | --- |
| 根据名称和类型创建 Bean | 工厂 |
| singleton 缓存 | 单例/注册表 |
| BeanPostProcessor 链 | 责任链 |
| AOP 包装 | 代理 |
| 初始化流程固定 | 模板方法 |
| 生命周期事件 | 观察者 |

## Spring AOP：代理、策略和责任链

一次 AOP 方法调用大致是：

```text
proxy.method
  -> Advisor 匹配
  -> Interceptor 链
  -> TransactionInterceptor
  -> MethodBeforeAdviceInterceptor
  -> target.method
  -> AfterReturning/AfterThrowing
```

深层问题：

- 切点如何匹配方法。
- 多个增强如何排序。
- 异常时哪些 after 逻辑执行。
- 返回值是否被增强修改。
- 事务提交发生在目标方法之后、返回给调用方之前。

> **重点：** Spring AOP 是代理模式，但执行过程更像代理对象内部维护了一条拦截器责任链。

## Spring MVC：策略、适配器和拦截链

请求进入 Spring MVC 后，核心流程简化为：

```text
DispatcherServlet
  -> HandlerMapping 找 Handler
  -> HandlerAdapter 适配调用方式
  -> Interceptor.preHandle
  -> Controller method
  -> Interceptor.postHandle
  -> ViewResolver/MessageConverter
  -> Interceptor.afterCompletion
```

模式组合：

- `DispatcherServlet`：前端控制器，也像外观。
- `HandlerMapping`：策略匹配。
- `HandlerAdapter`：适配器。
- `HandlerInterceptor`：责任链。
- `ViewResolver` / `HttpMessageConverter`：策略。

排序和冲突：

- 多个 `HandlerMapping` 有 order。
- 多个拦截器按注册顺序 preHandle，返回时逆序执行 afterCompletion。
- `preHandle` 返回 false 会短路 Controller。

## JdbcTemplate：模板方法和回调

不用模板时，JDBC 代码重复：

```text
获取连接 -> 创建语句 -> 绑定参数 -> 执行 -> 遍历结果 -> 关闭资源 -> 转换异常
```

`JdbcTemplate` 固定资源管理，把变化点交给回调：

```java
List<User> users = jdbcTemplate.query(
        "select id, name from users where status = ?",
        (rs, rowNum) -> new User(rs.getLong("id"), rs.getString("name")),
        "ACTIVE");
```

深层价值：

- 统一异常转换。
- 统一资源关闭。
- 统一事务绑定连接。
- 让调用方只关心 SQL 和行映射。

## MyBatis Mapper：动态代理和模板执行器

Mapper 调用简化链路：

```text
mapper.findById(1)
  -> MapperProxy.invoke
  -> MapperMethod.execute
  -> SqlSession.selectOne
  -> Executor.query
  -> StatementHandler
  -> JDBC
```

模式组合：

- Mapper 接口：代理。
- `SqlSessionFactory`：工厂。
- Executor 类型：策略。
- 插件机制：代理/责任链。
- SQL 执行骨架：模板。

深层问题：

- Mapper 方法如何映射到 statement id。
- 参数如何解析。
- 一级缓存和二级缓存什么时候命中。
- 插件拦截顺序如何影响 SQL 和结果。

## Servlet Filter：责任链的标准形态

```java
class TraceFilter implements Filter {
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        try {
            setTraceId(request);
            chain.doFilter(request, response);
        } finally {
            clearTraceId();
        }
    }
}
```

深层理解：

- `chain.doFilter` 之前是前置逻辑。
- `chain.doFilter` 之后是后置逻辑。
- 不调用 `chain.doFilter` 就短路。
- finally 很重要，尤其是 ThreadLocal 清理。

## 重点

- 框架源码中的模式通常是组合：工厂 + 代理 + 责任链 + 模板 + 策略。
- 读源码要画运行时调用链。
- 面试回答要落到具体类和执行顺序。

## 难点

- AOP、事务、缓存、异步等注解背后都依赖代理边界。
- MVC 和 MyBatis 的“适配”不是普通接口转换，还涉及方法参数、返回值和异常处理。
- 拦截器和过滤器都像责任链，但所在层次和执行时机不同。

## 易错

> **易错：** 答“Spring 用了工厂、代理、单例”，到此为止。
>
> 正确做法：说明 Bean 创建链、AOP 调用链、MVC 请求链中的具体执行顺序。

> **易错：** 把 Filter 和 Interceptor 混为一谈。
>
> 正确做法：Filter 属于 Servlet 容器层，Interceptor 属于 Spring MVC 层，执行时机和可获得上下文不同。
