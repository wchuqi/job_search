# Java设计模式学习资料：结构型模式

[返回索引](../Java设计模式学习资料.md)

## 学习目标

- 掌握对象之间的组合、包装、转接和访问控制方式。
- 能区分适配器、装饰器、代理和外观。
- 能在 Java 项目中设计可扩展的接口边界。

## 理论导读

结构型模式关注“对象怎么连接”。当已有接口不兼容、功能需要叠加、访问需要控制、对象结构是树形或对象数量很大时，结构型模式可以把复杂连接关系整理成稳定结构。

## 适配器模式

适配器把一个已有接口转换成目标接口。重点是解决接口不兼容。

```java
interface SmsGateway {
    void sendSms(String phone, String text);
}

class ThirdPartySmsApi {
    void push(String mobile, String message, String token) {
        System.out.println("send by third party");
    }
}

class ThirdPartySmsAdapter implements SmsGateway {
    private final ThirdPartySmsApi api;
    private final String token;

    ThirdPartySmsAdapter(ThirdPartySmsApi api, String token) {
        this.api = api;
        this.token = token;
    }

    public void sendSms(String phone, String text) {
        api.push(phone, text, token);
    }
}
```

## 桥接模式

桥接让抽象和实现两个维度独立变化。它常用于“类型维度”和“实现维度”都会扩展的场景。

```java
interface Renderer {
    void render(String data);
}

abstract class Report {
    protected final Renderer renderer;

    protected Report(Renderer renderer) {
        this.renderer = renderer;
    }

    abstract void export();
}

class SalesReport extends Report {
    SalesReport(Renderer renderer) { super(renderer); }

    void export() {
        renderer.render("sales data");
    }
}
```

报表类型可以扩展，渲染方式也可以扩展，二者不需要互相继承组合出大量类。

## 组合模式

组合模式用统一接口处理树形结构的整体和部分。

```java
interface MenuComponent {
    void print(int level);
}

class MenuItem implements MenuComponent {
    private final String name;
    MenuItem(String name) { this.name = name; }

    public void print(int level) {
        System.out.println(" ".repeat(level) + "- " + name);
    }
}

class MenuGroup implements MenuComponent {
    private final String name;
    private final List<MenuComponent> children = new ArrayList<>();

    MenuGroup(String name) { this.name = name; }

    void add(MenuComponent child) { children.add(child); }

    public void print(int level) {
        System.out.println(" ".repeat(level) + "+ " + name);
        children.forEach(child -> child.print(level + 2));
    }
}
```

## 装饰器模式

装饰器在不修改原对象的情况下动态叠加职责。

```java
interface DataService {
    String query(String id);
}

class RemoteDataService implements DataService {
    public String query(String id) {
        return "remote:" + id;
    }
}

class CacheDataService implements DataService {
    private final DataService delegate;
    private final Map<String, String> cache = new ConcurrentHashMap<>();

    CacheDataService(DataService delegate) {
        this.delegate = delegate;
    }

    public String query(String id) {
        return cache.computeIfAbsent(id, delegate::query);
    }
}
```

## 外观模式

外观为复杂子系统提供统一入口。它不一定增强功能，而是降低调用方认知成本。

```java
class OrderFacade {
    private final InventoryService inventory;
    private final PaymentService payment;
    private final DeliveryService delivery;

    OrderFacade(InventoryService inventory, PaymentService payment, DeliveryService delivery) {
        this.inventory = inventory;
        this.payment = payment;
        this.delivery = delivery;
    }

    void placeOrder(Order order) {
        inventory.reserve(order);
        payment.pay(order);
        delivery.ship(order);
    }
}
```

## 代理模式

代理控制对真实对象的访问，可以做权限、延迟加载、远程调用、事务、日志、缓存。

```java
interface DocumentService {
    Document load(String id);
}

class PermissionProxy implements DocumentService {
    private final DocumentService target;
    private final CurrentUser currentUser;

    PermissionProxy(DocumentService target, CurrentUser currentUser) {
        this.target = target;
        this.currentUser = currentUser;
    }

    public Document load(String id) {
        if (!currentUser.canRead(id)) {
            throw new SecurityException("forbidden");
        }
        return target.load(id);
    }
}
```

## 享元模式

享元共享大量对象的内部状态，减少内存占用。典型例子是 Java 字符串常量池、包装类缓存、棋子类型缓存。

```java
record IconStyle(String color, String shape) {}

class IconStyleFactory {
    private final Map<String, IconStyle> pool = new ConcurrentHashMap<>();

    IconStyle get(String color, String shape) {
        return pool.computeIfAbsent(color + ":" + shape, k -> new IconStyle(color, shape));
    }
}
```

## 易混模式对比

| 模式 | 包一层的目的 | 调用方是否感知原接口 | 典型关键词 |
| --- | --- | --- | --- |
| 适配器 | 转换接口 | 通常不感知 | 兼容、转换、接入第三方 |
| 装饰器 | 叠加职责 | 使用同一接口 | 增强、组合、多层包装 |
| 代理 | 控制访问 | 使用同一接口 | 权限、远程、延迟、事务 |
| 外观 | 简化子系统 | 调用统一入口 | 门面、聚合流程、降低复杂度 |

## 重点

- 结构型模式多依赖组合而不是继承。
- 包装对象时，要清楚“包装目的”。
- 树形结构优先考虑组合模式。

## 难点

- 代理和装饰器代码形态相似，差异在意图。
- 桥接模式要有两个独立变化维度才值得使用。
- 享元模式要区分内部状态和外部状态。

## 易错

> **易错：** 把外观写成上帝类，所有业务流程都塞进去。
>
> 正确做法：外观只负责协调子系统，不吞掉子系统自身职责。

> **易错：** 装饰器链过长，调试时不知道实际执行顺序。
>
> 正确做法：明确包装顺序，并在配置或构造处保持可读。

## 练习

设计一个文件存储模块：

- 统一接口 `FileStorage`。
- 用适配器接入本地文件、OSS、S3。
- 用装饰器增加压缩和加密。
- 用代理增加权限校验。
- 用外观提供上传头像、上传合同两个业务入口。

## 验收

- 能解释每层包装的目的。
- 能画出调用链。
- 能指出哪些场景不该使用桥接或享元。
