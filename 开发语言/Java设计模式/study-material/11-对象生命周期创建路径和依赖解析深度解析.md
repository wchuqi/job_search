# Java设计模式学习资料：对象生命周期、创建路径和依赖解析深度解析

[返回索引](../Java设计模式学习资料.md)

## 学习目标

- 从运行时角度理解创建型模式，而不是只看 `new` 被藏在哪里。
- 掌握对象从类加载、实例化、依赖注入、初始化、代理包装到销毁的完整路径。
- 理解 Spring 容器为什么是工厂、单例、代理、模板和观察者的综合体。
- 能分析循环依赖、构造器选择、重复 Bean、懒加载和作用域带来的设计问题。

## 理论导读

创建型模式的浅层理解是“封装对象创建”。深层理解应该是：系统如何把“描述对象的元数据”转化成“可被调用的运行时对象”，以及这个过程中谁有权决定实现类、生命周期、作用域、依赖和增强逻辑。

在普通 Java 代码里，对象创建通常是：

```java
PaymentClient client = new AliPayClient(config);
```

在框架里，对象创建更接近：

```text
配置/注解/BeanDefinition
  -> 类型解析
  -> 构造器选择
  -> 实例化
  -> 属性注入
  -> 初始化回调
  -> BeanPostProcessor
  -> AOP 代理包装
  -> 注册到容器
  -> 被业务代码注入和调用
```

设计模式进入框架后，重点就从“类图”变成“生命周期管线”。

## 对象创建路径：从 new 到容器工厂

### 1. 直接创建

直接创建最简单，调用方强依赖具体类：

```java
class OrderService {
    private final PaymentClient client = new AliPayClient("appId", "secret");
}
```

问题：

- 调用方知道具体实现。
- 配置散落在业务类中。
- 单元测试难以替换。
- 新增渠道时修改业务代码。

### 2. 简单工厂

```java
class PaymentClientFactory {
    PaymentClient create(String channel) {
        return switch (channel) {
            case "ALI" -> new AliPayClient(loadAliConfig());
            case "WECHAT" -> new WeChatPayClient(loadWechatConfig());
            default -> throw new IllegalArgumentException(channel);
        };
    }
}
```

简单工厂把创建集中起来，但工厂本身会随着类型增加而修改。它适合类型少、变化可控、创建规则集中管理的场景。

### 3. 工厂方法

工厂方法把具体创建交给具体工厂：

```java
interface PaymentClientFactory {
    String channel();
    PaymentClient create();
}

class AliPayClientFactory implements PaymentClientFactory {
    public String channel() { return "ALI"; }
    public PaymentClient create() { return new AliPayClient(loadConfig()); }
}
```

再通过注册表选择：

```java
class PaymentClientFactories {
    private final Map<String, PaymentClientFactory> factories;

    PaymentClientFactories(List<PaymentClientFactory> factories) {
        this.factories = factories.stream()
                .collect(Collectors.toUnmodifiableMap(PaymentClientFactory::channel, Function.identity()));
    }

    PaymentClient create(String channel) {
        PaymentClientFactory factory = factories.get(channel);
        if (factory == null) {
            throw new IllegalArgumentException("unsupported channel: " + channel);
        }
        return factory.create();
    }
}
```

这里的深层问题不是“有没有工厂接口”，而是注册表要定义这些规则：

- key 是否区分大小写。
- 重复 key 如何处理。
- 找不到实现时抛错还是走默认实现。
- 创建失败是否重试。
- 创建出来的是新对象、缓存对象还是容器单例。

## Spring Bean 创建生命周期

Spring 的 Bean 创建可以理解为一个强大的工厂管线。

```text
读取 BeanDefinition
  -> merge 定义
  -> 推断构造器
  -> 实例化对象
  -> 提前暴露单例工厂
  -> 属性填充
  -> Aware 回调
  -> BeanPostProcessor 前置处理
  -> 初始化方法
  -> BeanPostProcessor 后置处理
  -> 可能生成代理对象
  -> 放入 singletonObjects
```

### 关键参与者

| 参与者 | 作用 | 对应模式 |
| --- | --- | --- |
| `BeanFactory` | 根据 Bean 定义创建对象 | 工厂 |
| `BeanDefinition` | 对象创建的元数据 | 配置对象 |
| `BeanPostProcessor` | 在初始化前后扩展处理 | 责任链/模板回调 |
| AOP AutoProxyCreator | 判断是否需要代理 | 代理/策略 |
| Scope | 决定对象作用域 | 策略 |
| ApplicationEvent | 生命周期事件通知 | 观察者 |

## 构造器选择和依赖解析

当一个类有多个构造器时，容器必须选择一个。简化规则可以理解为：

1. 优先使用显式标注的构造器。
2. 没有标注时，若只有一个构造器就使用它。
3. 多个构造器时，根据可解析参数、是否必需、优先级推断。
4. 参数依赖再按类型、名称、限定符、主 Bean 等规则解析。

这说明“依赖注入”内部其实包含匹配算法和冲突解决。

```java
class OrderService {
    private final PaymentService paymentService;
    private final RiskService riskService;

    OrderService(PaymentService paymentService, RiskService riskService) {
        this.paymentService = paymentService;
        this.riskService = riskService;
    }
}
```

如果存在两个 `PaymentService` 实现，容器必须解决：

- 按类型找到多个候选怎么办。
- 是否存在 `@Primary`。
- 是否有 `@Qualifier`。
- 参数名能否匹配 Bean 名。
- 是否允许注入集合或 Map。

## 循环依赖和设计模式

循环依赖不是纯技术问题，通常说明职责边界有问题。

```text
OrderService -> PaymentService -> OrderService
```

常见修复方式：

- 提取共同依赖，例如 `OrderRepository` 或 `OrderDomainService`。
- 使用事件解耦，例如支付完成发布事件，订单监听后更新状态。
- 使用外观协调两个服务，而不是让服务互相调用。
- 延迟依赖只能作为权宜之计，不应掩盖模型问题。

> **重点：** 循环依赖往往是中介者、观察者、外观、领域服务边界设计不清的信号。

## 单例作用域和线程安全

Spring 默认 singleton 是容器级单例。它解决的是对象数量，不解决线程安全。

危险写法：

```java
class DiscountService {
    private BigDecimal currentDiscount;

    BigDecimal calculate(Order order) {
        currentDiscount = loadDiscount(order);
        return order.amount().subtract(currentDiscount);
    }
}
```

在多线程 Web 请求中，字段会被并发覆盖。正确做法是让单例服务无状态：

```java
class DiscountService {
    BigDecimal calculate(Order order) {
        BigDecimal discount = loadDiscount(order);
        return order.amount().subtract(discount);
    }
}
```

## 对象生命周期和模式选择

| 生命周期问题 | 推荐模式/机制 | 判断标准 |
| --- | --- | --- |
| 对象创建复杂 | Builder/工厂 | 构造参数多、规则复杂 |
| 实现类由配置决定 | 工厂/策略注册表 | 配置 key 能映射实现 |
| 一组对象必须配套 | 抽象工厂 | 产品族一致性重要 |
| 对象需要增强 | 代理/装饰器 | 是控制访问还是叠加功能 |
| 对象状态需要恢复 | 原型/备忘录 | 是否需要快照 |
| 对象生命周期由容器管理 | DI/模板/观察者 | 是否涉及初始化和销毁 |

## 例子：支付客户端创建的生产级规则

```java
record PaymentClientConfig(
        String channel,
        String appId,
        String secret,
        Duration timeout,
        int maxRetry
) {}

interface PaymentClientFactory {
    String channel();
    PaymentClient create(PaymentClientConfig config);
}

class PaymentClientRegistry {
    private final Map<String, PaymentClientFactory> factories;
    private final PaymentConfigRepository configRepository;

    PaymentClientRegistry(List<PaymentClientFactory> factories,
                          PaymentConfigRepository configRepository) {
        this.factories = factories.stream().collect(Collectors.toMap(
                PaymentClientFactory::channel,
                Function.identity(),
                (left, right) -> {
                    throw new IllegalStateException("duplicate payment channel: " + left.channel());
                }));
        this.configRepository = configRepository;
    }

    PaymentClient getClient(String channel) {
        PaymentClientFactory factory = factories.get(channel);
        if (factory == null) {
            throw new IllegalArgumentException("unsupported payment channel: " + channel);
        }
        PaymentClientConfig config = configRepository.load(channel);
        return factory.create(config);
    }
}
```

这个版本比简单工厂更深入，因为它定义了：

- 重复渠道直接启动失败。
- 未知渠道抛明确异常。
- 配置由仓库统一加载。
- 客户端创建和配置加载分离。

## 重点

- 创建型模式的深层问题是生命周期、依赖解析、作用域和失败处理。
- DI 容器不是简单 Map，它有构造器选择、依赖匹配、初始化、代理包装和销毁流程。
- Spring 单例不等于线程安全。

## 难点

- 对象最终被注入的可能是代理对象，而不是原始目标对象。
- 循环依赖常常暴露模块职责边界不清。
- 工厂注册表要定义重复、缺省、失败、热更新和作用域规则。

## 易错

> **易错：** 只把 `new` 移进工厂类，就认为完成了解耦。
>
> 正确做法：同时定义选择规则、配置来源、生命周期、失败策略和测试替换方式。

> **易错：** 在 Spring 单例 Bean 中保存请求级临时状态。
>
> 正确做法：请求状态放局部变量、方法参数、请求作用域对象或显式上下文中。

## 练习

设计一个支付客户端注册中心：

1. 支持支付宝、微信、余额。
2. 每个渠道配置独立加载。
3. 重复渠道实现启动时报错。
4. 未知渠道返回明确错误码。
5. 客户端创建失败时记录异常并允许降级到“不可用客户端”。

## 验收

- 能画出对象从配置到运行时实例的创建链。
- 能解释 Spring Bean 被代理前后的对象差异。
- 能指出单例作用域下的线程安全风险。
- 能通过测试覆盖重复 key、未知 key、配置缺失和创建失败。
