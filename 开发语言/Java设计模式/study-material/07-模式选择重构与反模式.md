# Java设计模式学习资料：模式选择、重构与反模式

[返回索引](../Java设计模式学习资料.md)

## 学习目标

- 能根据问题选择模式，而不是按记忆套模式。
- 能从坏味道出发逐步重构。
- 能识别过度设计和常见反模式。

## 模式选择决策表

| 问题信号 | 优先考虑 | 判断问题 |
| --- | --- | --- |
| 创建逻辑复杂或实现类很多 | 工厂、Builder | 调用方是否依赖太多具体类 |
| 构造参数多且有校验 | Builder | 对象是否应不可变 |
| 第三方接口不兼容 | 适配器 | 是否只是接口转换 |
| 需要叠加缓存、日志、压缩 | 装饰器 | 是否保持同一接口 |
| 需要权限、事务、远程访问控制 | 代理 | 是否控制访问 |
| 子系统调用复杂 | 外观 | 是否给调用方一个简洁入口 |
| 算法经常变化 | 策略 | 是否一组可替换算法 |
| 流程稳定但步骤变化 | 模板方法 | 流程骨架是否稳定 |
| 多个处理器按顺序处理 | 责任链 | 顺序和短路是否明确 |
| 状态决定行为 | 状态 | 状态迁移是否复杂 |
| 状态变化通知多个模块 | 观察者 | 是否需要解耦发布者和订阅者 |

## 重构路径

### 从条件分支到策略

适用条件：

- 分支代表同一业务能力的多个实现。
- 分支数量会增长。
- 每个分支逻辑较长或需要独立测试。

步骤：

1. 提取公共接口。
2. 每个分支变成一个实现类。
3. 建立策略注册表。
4. 调用方根据业务 key 选择策略。
5. 为每个策略写独立测试。

### 从大流程到模板方法或责任链

如果流程固定、步骤变化，用模板方法。如果处理器数量可变、顺序可配置、可能短路，用责任链。

### 从继承爆炸到桥接或装饰器

如果有两个变化维度，例如“报表类型”和“导出格式”，用桥接。如果是在同一能力上叠加功能，例如“读取 + 缓存 + 加密 + 统计”，用装饰器。

## 常见反模式

| 反模式 | 表现 | 后果 | 修正 |
| --- | --- | --- | --- |
| 模式驱动开发 | 先决定用模式，再找业务点 | 抽象层无意义 | 从变化点和痛点出发 |
| 单例滥用 | 全局状态到处访问 | 测试困难、并发风险 | 使用 DI 管理依赖 |
| 上帝外观 | 一个 Facade 调用所有服务 | 职责失控 | 按业务边界拆分 |
| 抽象泄漏 | 接口暴露实现细节 | 替换实现困难 | 重新定义稳定协议 |
| 策略黑箱 | Map 里塞大量匿名 Lambda | 难排查、难测试 | 复杂策略命名成类 |
| 责任链副作用 | 上下文被隐式修改 | 顺序依赖脆弱 | 明确输入输出和字段语义 |

## 重点

- 模式选择的起点是变化点，不是模式定义。
- 重构要小步进行，先提接口，再迁移分支，再删除旧逻辑。
- 设计模式应降低测试难度，而不是让测试更难。

## 难点

- 真实项目常常是多个模式组合，不能机械地给整个模块贴一个模式标签。
- 判断“现在抽象”还是“等变化出现再抽象”需要经验。
- 过度设计短期看起来优雅，长期会带来认知负担。

## 易错

> **易错：** 为了符合 OCP，完全不允许修改旧代码。
>
> 正确做法：OCP 是控制常见扩展的修改范围，不是禁止合理重构。

> **易错：** 业务规则未稳定时就抽象成很多策略类。
>
> 正确做法：先让规则清晰，再围绕稳定边界抽象。

## 练习

找一个包含 5 个以上 `if-else` 分支的业务方法：

1. 判断分支是否代表同一抽象。
2. 写出重构前的坏味道。
3. 选择策略、状态、责任链或保持原状。
4. 重构其中两个分支。
5. 写测试证明行为不变。

## 验收

- 能说明选择某个模式的理由和放弃其他模式的理由。
- 能识别至少 5 种模式误用。
- 能用测试保护重构。
