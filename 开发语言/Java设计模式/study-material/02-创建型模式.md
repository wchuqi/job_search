# Java设计模式学习资料：创建型模式

[返回索引](../Java设计模式学习资料.md)

## 学习目标

- 掌握对象创建的封装方式。
- 能区分单例、工厂方法、抽象工厂、建造者和原型。
- 能在 Java 项目里写出线程安全、可测试、不过度抽象的创建逻辑。

## 理论导读

创建型模式处理的核心问题是：调用方不应该总是知道“具体创建哪个类、怎么创建、创建过程有哪些步骤”。当对象构造复杂、实现类很多、创建规则会变化时，把创建逻辑从业务流程中抽离出来，可以降低耦合。

## 单例模式

单例保证一个类只有一个实例，并提供全局访问点。Java 中推荐枚举单例或静态内部类单例。

```java
public enum ConfigRegistry {
    INSTANCE;

    private final Map<String, String> config = new ConcurrentHashMap<>();

    public String get(String key) {
        return config.get(key);
    }
}
```

适用场景：无状态工具、全局配置、注册表、重量级共享资源。

不适用场景：有用户上下文、有请求状态、需要频繁替换依赖、需要并行测试隔离。

## 工厂方法

工厂方法把创建对象的职责交给具体工厂，调用方只依赖产品接口。

```java
interface MessageSender {
    void send(String target, String content);
}

class SmsSender implements MessageSender {
    public void send(String target, String content) {
        System.out.println("sms to " + target + ": " + content);
    }
}

interface SenderFactory {
    MessageSender create();
}

class SmsSenderFactory implements SenderFactory {
    public MessageSender create() {
        return new SmsSender();
    }
}
```

## 抽象工厂

抽象工厂创建一族相关对象。重点不是“一个工厂创建多个对象”，而是这些对象属于同一产品族。

```java
interface CloudFactory {
    StorageClient storage();
    QueueClient queue();
}

class AwsFactory implements CloudFactory {
    public StorageClient storage() { return new S3Client(); }
    public QueueClient queue() { return new SqsClient(); }
}

class AliyunFactory implements CloudFactory {
    public StorageClient storage() { return new OssClient(); }
    public QueueClient queue() { return new MnsClient(); }
}
```

如果项目要按云厂商整体切换存储、队列、消息等组件，抽象工厂比多个零散工厂更合适。

## 建造者模式

建造者用于分步骤构造复杂对象，特别适合参数多、可选项多、需要校验的不可变对象。

```java
public final class HttpRequest {
    private final String url;
    private final String method;
    private final Map<String, String> headers;
    private final String body;

    private HttpRequest(Builder builder) {
        this.url = builder.url;
        this.method = builder.method;
        this.headers = Map.copyOf(builder.headers);
        this.body = builder.body;
    }

    public static class Builder {
        private String url;
        private String method = "GET";
        private Map<String, String> headers = new HashMap<>();
        private String body;

        public Builder url(String url) {
            this.url = Objects.requireNonNull(url);
            return this;
        }

        public Builder method(String method) {
            this.method = method;
            return this;
        }

        public Builder header(String key, String value) {
            headers.put(key, value);
            return this;
        }

        public Builder body(String body) {
            this.body = body;
            return this;
        }

        public HttpRequest build() {
            if (url == null || url.isBlank()) {
                throw new IllegalStateException("url required");
            }
            return new HttpRequest(this);
        }
    }
}
```

## 原型模式

原型通过复制已有对象创建新对象，适合对象创建成本高或初始状态复杂的场景。Java 中要谨慎使用 `Cloneable`，更推荐复制构造器或显式 copy 方法。

```java
record ReportTemplate(String title, List<String> sections) {
    ReportTemplate copyWithTitle(String newTitle) {
        return new ReportTemplate(newTitle, List.copyOf(sections));
    }
}
```

## 对比表

| 模式 | 关注点 | 典型场景 | 易错点 |
| --- | --- | --- | --- |
| 单例 | 唯一实例 | 注册表、配置、无状态工具 | 隐藏全局状态 |
| 工厂方法 | 单个产品创建 | 按类型创建发送器、解析器 | 工厂类膨胀 |
| 抽象工厂 | 产品族创建 | 同一厂商的一组组件 | 产品族边界不清 |
| 建造者 | 复杂构造过程 | DTO、请求、配置对象 | Builder 不校验不变量 |
| 原型 | 复制已有对象 | 模板、快照、复杂初始对象 | 浅拷贝导致共享可变状态 |

## 重点

- 创建型模式的核心是隔离 `new` 和复杂构造逻辑。
- Spring 容器本身就是大型对象工厂，但业务中仍可使用工厂表达领域创建规则。
- Builder 适合不可变对象，尤其能减少构造器参数顺序错误。

## 难点

- 工厂方法和抽象工厂的区别在于产品族，不在于方法数量。
- 单例的线程安全、序列化、反射攻击和测试隔离需要额外考虑。
- 原型模式里的深拷贝和浅拷贝边界必须明确。

## 易错

> **易错：** 所有工具类都写成单例。
>
> 正确做法：无状态工具可以用静态方法；有依赖的对象交给 DI 容器管理。

> **易错：** Builder 只负责 set 值，不校验业务不变量。
>
> 正确做法：在 `build()` 中集中校验必填字段、互斥字段和默认值。

## 练习

实现一个 `NotificationClient` 创建模块：

- 支持短信、邮件、企业微信。
- 每种客户端有不同配置。
- 使用 Builder 构造 `NotificationRequest`。
- 写出单元测试验证新增渠道时核心发送服务不用修改。

## 验收

- 能写出线程安全单例并说明为什么安全。
- 能区分工厂方法和抽象工厂。
- 能说明 Builder 和 JavaBean setter 的差异。
