# Java设计模式学习资料：行为型模式

[返回索引](../Java设计模式学习资料.md)

## 学习目标

- 掌握对象之间职责分配、流程控制、状态切换和算法替换。
- 能区分策略、状态、命令、责任链、模板方法和观察者。
- 能把行为型模式应用到 Java 业务代码和框架源码阅读中。

## 理论导读

行为型模式关注“谁来做、何时做、按什么顺序做、变化如何替换”。它们比创建型和结构型更贴近业务流程，也更容易被滥用。判断是否使用行为型模式时，要先看流程是否稳定、步骤是否可变、处理者是否可插拔、状态是否驱动行为。

## 策略模式

策略模式封装一组可互换算法，调用方通过抽象使用它们。

```java
interface ShippingFeeStrategy {
    String region();
    BigDecimal calculate(BigDecimal weight);
}

class EastRegionShipping implements ShippingFeeStrategy {
    public String region() { return "EAST"; }
    public BigDecimal calculate(BigDecimal weight) {
        return weight.multiply(new BigDecimal("6.5"));
    }
}

class ShippingFeeService {
    private final Map<String, ShippingFeeStrategy> strategies;

    ShippingFeeService(List<ShippingFeeStrategy> strategies) {
        this.strategies = strategies.stream()
                .collect(Collectors.toMap(ShippingFeeStrategy::region, Function.identity()));
    }

    BigDecimal fee(String region, BigDecimal weight) {
        ShippingFeeStrategy strategy = strategies.get(region);
        if (strategy == null) {
            throw new IllegalArgumentException("unsupported region: " + region);
        }
        return strategy.calculate(weight);
    }
}
```

## 模板方法

模板方法在父类中固定流程骨架，把可变步骤交给子类。

```java
abstract class ImportTemplate {
    public final void run(Path file) {
        List<String> rows = read(file);
        validate(rows);
        save(rows);
        afterSave(rows);
    }

    protected List<String> read(Path file) { return List.of(); }
    protected abstract void validate(List<String> rows);
    protected abstract void save(List<String> rows);
    protected void afterSave(List<String> rows) {}
}
```

适合流程稳定、步骤可变的场景。若流程本身也常变，模板方法会变脆。

## 观察者模式

观察者让一个对象状态变化时通知多个订阅者。

```java
record OrderPaidEvent(String orderId) {}

interface OrderPaidListener {
    void onPaid(OrderPaidEvent event);
}

class OrderEventPublisher {
    private final List<OrderPaidListener> listeners = new CopyOnWriteArrayList<>();

    void register(OrderPaidListener listener) {
        listeners.add(listener);
    }

    void publish(OrderPaidEvent event) {
        listeners.forEach(listener -> listener.onPaid(event));
    }
}
```

业务中常见演进是：内存观察者 -> Spring 事件 -> MQ 事件。

## 责任链模式

责任链让请求沿处理器链依次处理。重点是顺序、短路和错误处理。

```java
interface RiskCheck {
    void check(Order order, RiskContext context);
}

class RiskChain {
    private final List<RiskCheck> checks;

    RiskChain(List<RiskCheck> checks) {
        this.checks = checks;
    }

    void check(Order order) {
        RiskContext context = new RiskContext();
        for (RiskCheck check : checks) {
            check.check(order, context);
            if (context.rejected()) {
                break;
            }
        }
    }
}
```

责任链的调度规则必须明确：顺序由谁决定，失败是否继续，处理器之间是否共享上下文。

## 命令模式

命令把请求封装成对象，便于排队、撤销、记录日志、重试。

```java
interface Command {
    void execute();
}

class CreateOrderCommand implements Command {
    private final OrderService orderService;
    private final OrderRequest request;

    CreateOrderCommand(OrderService orderService, OrderRequest request) {
        this.orderService = orderService;
        this.request = request;
    }

    public void execute() {
        orderService.create(request);
    }
}
```

## 状态模式

状态模式让对象在不同状态下表现出不同行为，避免状态判断散落各处。

```java
interface OrderState {
    void pay(Order order);
    void cancel(Order order);
}

class PendingState implements OrderState {
    public void pay(Order order) {
        order.setState(new PaidState());
    }

    public void cancel(Order order) {
        order.setState(new CanceledState());
    }
}

class PaidState implements OrderState {
    public void pay(Order order) {
        throw new IllegalStateException("already paid");
    }

    public void cancel(Order order) {
        throw new IllegalStateException("paid order cannot cancel directly");
    }
}
```

## 其他行为型模式

| 模式 | 适用场景 | Java 中的常见体现 |
| --- | --- | --- |
| 迭代器 | 隐藏集合结构并遍历 | `Iterator`、`Iterable` |
| 中介者 | 多对象复杂交互集中协调 | GUI 组件协调、聊天室 |
| 备忘录 | 保存和恢复状态 | 编辑器撤销、快照 |
| 解释器 | 简单规则语言解析 | 表达式、规则引擎雏形 |
| 访问者 | 对稳定对象结构增加新操作 | AST 遍历、报表导出 |

## 易混模式对比

| 模式 | 核心变化点 | 典型问题 |
| --- | --- | --- |
| 策略 | 算法可替换 | 同一输入，不同算法 |
| 状态 | 状态驱动行为 | 同一方法，不同状态下行为不同 |
| 命令 | 请求对象化 | 延迟、队列、撤销、重试 |
| 责任链 | 多处理器顺序执行 | 过滤、审批、校验 |
| 模板方法 | 流程骨架稳定 | 导入、导出、任务执行流程 |

## 重点

- 行为型模式必须讲清楚调度顺序。
- 策略关注算法替换，状态关注状态迁移。
- 责任链一定要定义短路、异常和顺序规则。

## 难点

- 观察者异步化后会引入一致性、重试、幂等和消息顺序问题。
- 状态模式如果状态太多，可能需要状态机表或专门状态机框架。
- 访问者适合对象结构稳定的场景，如果元素类频繁变化会很痛苦。

## 易错

> **易错：** 把所有 `if-else` 都改成策略或状态。
>
> 正确做法：只有当分支代表稳定抽象、且未来会扩展时才值得抽离。

> **易错：** 责任链中处理器偷偷修改共享上下文，后续处理器依赖隐式副作用。
>
> 正确做法：约定上下文字段含义，明确处理器输入输出。

## 练习

实现订单处理流程：

- 用策略模式计算不同会员等级优惠。
- 用责任链执行风控校验。
- 用状态模式管理待支付、已支付、已取消、已发货。
- 用观察者发布支付成功事件。

## 验收

- 能说明每个行为模式的调度规则。
- 能从业务流程中找出算法变化、步骤变化和状态变化。
- 能解释模式误用会带来的维护成本。
