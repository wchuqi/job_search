# Java设计模式学习资料：从坏代码到模式的重构案例深度解析

[返回索引](../Java设计模式学习资料.md)

## 学习目标

- 掌握从真实坏代码逐步重构到模式组合的过程。
- 学会用测试保护行为，而不是凭感觉重构。
- 能解释每一步为什么做、风险是什么、何时停止抽象。

## 案例背景

电商订单支付方法越来越大：

- 支持支付宝、微信、余额。
- 不同会员等级有不同优惠。
- 支付前要做风控。
- 支付成功要改订单状态、发短信、加积分、清缓存。
- 后续还要新增 PayPal 和企业转账。

## 重构前代码

```java
class OrderApplicationService {
    void pay(String orderId, String channel, String memberType) {
        Order order = orderRepository.findById(orderId);
        if (!"PENDING".equals(order.status())) {
            throw new IllegalStateException("order cannot pay");
        }

        if (blacklistService.contains(order.userId())) {
            throw new RiskException("black user");
        }
        if (order.amount().compareTo(new BigDecimal("5000")) > 0) {
            throw new RiskException("amount too large");
        }

        BigDecimal finalAmount = order.amount();
        if ("VIP".equals(memberType)) {
            finalAmount = finalAmount.multiply(new BigDecimal("0.8"));
        } else if ("NEW".equals(memberType)) {
            finalAmount = finalAmount.subtract(new BigDecimal("20"));
        }

        if ("ALI".equals(channel)) {
            aliPayClient.pay(orderId, finalAmount);
        } else if ("WECHAT".equals(channel)) {
            wechatPayClient.pay(orderId, finalAmount);
        } else if ("BALANCE".equals(channel)) {
            balanceService.debit(order.userId(), finalAmount);
        } else {
            throw new IllegalArgumentException("unsupported channel");
        }

        orderRepository.markPaid(orderId);
        smsClient.send(order.phone(), "paid");
        pointsService.add(order.userId(), finalAmount);
        cache.delete("order:" + orderId);
    }
}
```

## 坏味道分析

| 坏味道 | 表现 | 后果 |
| --- | --- | --- |
| 分支膨胀 | 支付渠道、优惠、风控都在一个方法 | 新增规则必改核心方法 |
| 职责混杂 | 查询、校验、支付、通知、缓存都在一起 | 测试难、回滚难 |
| 状态散落 | 订单状态用字符串判断 | 非法流转难控制 |
| 副作用混入主链路 | 短信和积分同步执行 | 非核心失败影响支付 |
| 不可观测 | 不知道哪个规则拒绝或耗时 | 排障困难 |

## 第一步：补测试保护行为

重构前先写测试：

- 非待支付订单不能支付。
- 黑名单用户拒绝。
- VIP 优惠 8 折。
- 不支持渠道抛异常。
- 支付成功后订单变为已支付。

测试是重构的护栏。没有测试时，大重构会变成重写。

## 第二步：提取支付策略

```java
interface PaymentStrategy {
    String channel();
    void pay(Order order, BigDecimal amount);
}

class AliPaymentStrategy implements PaymentStrategy {
    private final AliPayClient client;

    public String channel() { return "ALI"; }

    public void pay(Order order, BigDecimal amount) {
        client.pay(order.id(), amount);
    }
}
```

注册表：

```java
class PaymentStrategyRegistry {
    private final Map<String, PaymentStrategy> strategies;

    PaymentStrategyRegistry(List<PaymentStrategy> strategies) {
        this.strategies = strategies.stream().collect(Collectors.toMap(
                PaymentStrategy::channel,
                Function.identity(),
                (a, b) -> {
                    throw new IllegalStateException("duplicate channel: " + a.channel());
                }));
    }

    PaymentStrategy require(String channel) {
        PaymentStrategy strategy = strategies.get(channel);
        if (strategy == null) {
            throw new IllegalArgumentException("unsupported channel: " + channel);
        }
        return strategy;
    }
}
```

收益：

- 新增支付渠道不改核心流程。
- 每个渠道可独立测试。
- 重复渠道启动失败。

## 第三步：提取优惠策略

优惠可以是互斥策略，也可以是叠加规则。先问业务：

- VIP 和新人是否能同时享受？
- 多张券是否可叠加？
- 是否有最优价计算？

如果互斥，用策略。如果叠加，用责任链或规则集合。

```java
interface DiscountRule {
    boolean supports(Order order, Member member);
    BigDecimal apply(BigDecimal amount, Order order, Member member);
}
```

如果规则可叠加：

```java
class DiscountCalculator {
    private final List<DiscountRule> rules;

    BigDecimal calculate(Order order, Member member) {
        BigDecimal amount = order.amount();
        for (DiscountRule rule : rules) {
            if (rule.supports(order, member)) {
                amount = rule.apply(amount, order, member);
            }
        }
        return amount.max(BigDecimal.ZERO);
    }
}
```

## 第四步：提取风控责任链

```java
interface RiskRule {
    String name();
    int order();
    RiskDecision check(Order order);
}

class RiskEngine {
    private final List<RiskRule> rules;

    RiskEngine(List<RiskRule> rules) {
        this.rules = rules.stream()
                .sorted(Comparator.comparingInt(RiskRule::order))
                .toList();
    }

    void check(Order order) {
        for (RiskRule rule : rules) {
            RiskDecision decision = rule.check(order);
            if (decision.rejected()) {
                throw new RiskException(rule.name() + ": " + decision.reason());
            }
        }
    }
}
```

关键不是写链，而是定义：

- order 是否允许重复。
- 命中是否短路。
- 远程规则超时如何处理。
- 是否记录每条规则耗时。

## 第五步：状态模式或状态迁移服务

如果状态少，可以用状态迁移表，不必一开始就上状态模式。

```java
enum OrderStatus {
    PENDING, PAID, CANCELED, SHIPPED
}

class OrderStateMachine {
    private final Set<Transition> transitions = Set.of(
            new Transition(PENDING, PAY, PAID),
            new Transition(PENDING, CANCEL, CANCELED),
            new Transition(PAID, SHIP, SHIPPED)
    );

    OrderStatus transit(OrderStatus current, OrderEvent event) {
        return transitions.stream()
                .filter(t -> t.from() == current && t.event() == event)
                .map(Transition::to)
                .findFirst()
                .orElseThrow(() -> new IllegalStateException("illegal transition"));
    }
}
```

当每个状态有大量不同行为时，再拆成状态类。

## 第六步：事件化副作用

支付成功后的短信、积分、发票不应直接绑死在主流程：

```java
record OrderPaidEvent(String orderId, long userId, BigDecimal amount) {}

class OrderPaidEventPublisher {
    void publish(OrderPaidEvent event) {
        // 本地事件或写 outbox
    }
}
```

主流程只负责：

- 风控通过。
- 调用支付。
- 原子更新订单状态。
- 记录支付成功事件。

通知和积分异步处理，消费者幂等。

## 重构后核心流程

```java
class OrderPaymentService {
    private final RiskEngine riskEngine;
    private final DiscountCalculator discountCalculator;
    private final PaymentStrategyRegistry paymentStrategies;
    private final OrderStateMachine stateMachine;
    private final OrderPaidEventPublisher eventPublisher;

    @Transactional
    void pay(String orderId, String channel, Member member) {
        Order order = orderRepository.findByIdForUpdate(orderId);
        stateMachine.assertCan(order.status(), OrderEvent.PAY);

        riskEngine.check(order);
        BigDecimal amount = discountCalculator.calculate(order, member);

        paymentStrategies.require(channel).pay(order, amount);

        orderRepository.updateStatus(orderId, OrderStatus.PENDING, OrderStatus.PAID);
        eventPublisher.publish(new OrderPaidEvent(orderId, order.userId(), amount));
    }
}
```

## 重构效果评估

| 变化 | 重构前 | 重构后 |
| --- | --- | --- |
| 新增支付渠道 | 修改大方法 | 新增策略类 |
| 新增风控规则 | 修改大方法 | 新增规则类 |
| 支付成功副作用 | 同步写死 | 事件订阅 |
| 非法状态 | 字符串判断 | 状态迁移表 |
| 测试 | 只能测大流程 | 可分别测策略、规则、状态 |

## 何时停止抽象

不要继续抽象到这些程度：

- 每个一行逻辑都一个类。
- 为未来不存在的渠道预留复杂插件系统。
- 用反射扫描替代简单构造。
- 在没有动态配置需求时引入复杂规则引擎。

## 重点

- 重构路径必须从测试开始。
- 每一步只抽一个变化点。
- 模式组合要服务业务边界：策略处理支付，责任链处理风控，状态机处理状态，观察者处理副作用。

## 难点

- 优惠规则到底用策略还是责任链，取决于规则互斥还是叠加。
- 状态模式和状态机表不是敌人，复杂行为用状态类，规则清晰用迁移表。
- 事件化会引入最终一致性，需要幂等和补偿。

## 易错

> **易错：** 一次性把所有模式都引进来，导致行为变化难定位。
>
> 正确做法：每次重构只改变结构，不改变行为，并用测试验证。

> **易错：** 为了 OCP 把业务流程切得太碎。
>
> 正确做法：核心流程应保持可读，变化点才抽出去。

## 练习

拿一个真实项目中的大方法：

1. 标记所有变化点。
2. 按变化类型分类：创建、结构、行为、副作用、状态。
3. 先写测试。
4. 只抽第一个变化点。
5. 对比重构前后新增需求的修改范围。

## 验收

- 能写出重构报告：坏味道、模式选择、替代方案、风险、测试。
- 能证明新增一个渠道或规则时不修改核心流程。
- 能说明哪些地方刻意没有继续抽象，以及原因。
