# Rust学习资料：Send、Sync、原子内存序和并发正确性深度解析

[返回索引](../Rust学习资料.md)

## 学习目标

- 理解 `Send`、`Sync`、auto trait 和内部字段如何决定线程安全。
- 掌握 Mutex、RwLock、atomic、channel 的选择边界。
- 理解原子内存序的基本语义和使用风险。

## 理论导读

Rust 的并发安全分两层。第一层是类型系统防止数据竞争：没有同步就不能多线程共享可变状态。第二层是逻辑正确性：死锁、饥饿、ABA、错误内存序、锁粒度过大、背压缺失仍然需要工程设计解决。

`Send` 和 `Sync` 是自动推导的 marker trait。一个类型是否 `Send`/`Sync` 通常由字段决定；包含 `Rc`、`RefCell`、裸指针等字段时，推导结果会改变。unsafe 手动实现这些 trait 时，你就是在向编译器承诺线程安全不变量。

## 核心心智模型

`Send` 是“能不能把所有权搬到另一个线程”，`Sync` 是“能不能把共享引用给多个线程看”。原子内存序是“多个 CPU 核心看到读写顺序的契约”。

## 机制拆解

### 1. Send

如果 `T: Send`，表示 `T` 的所有权可以安全转移到另一个线程。`String`、`Vec<T>` 在元素满足条件时通常是 Send。`Rc<T>` 不是 Send，因为引用计数非原子，多线程递增递减会数据竞争。

### 2. Sync

如果 `T: Sync`，表示 `&T: Send`，也就是共享引用可以跨线程。`Mutex<T>` 可以让非线程安全的内部可变状态通过锁安全共享，但要求 `T: Send`。

### 3. Auto trait 推导

```rust
use std::rc::Rc;

struct NotThreadSafe {
    value: Rc<String>,
}
```

因为字段 `Rc<String>` 不是 Send/Sync，外层结构体也不会自动是 Send/Sync。

### 4. Mutex 与 RwLock

`Mutex` 适合写多、临界区短或简单共享状态。`RwLock` 适合读多写少，但在写竞争、读锁长期持有、实现策略偏向读或写时可能表现更差。

### 5. Atomic 和内存序

常见内存序：

| 内存序 | 含义 | 典型用途 |
| --- | --- | --- |
| Relaxed | 只保证单个原子变量操作原子性，不建立跨变量顺序 | 计数器、统计 |
| Acquire | 后续读写不能重排到它之前 | 读取发布的数据 |
| Release | 之前读写不能重排到它之后 | 发布数据 |
| AcqRel | 读改写同时具备 acquire 和 release | CAS、fetch_update |
| SeqCst | 全局最强顺序，最容易推理但可能更贵 | 简化复杂同步推理 |

原子不是“无锁就简单”。错误内存序可能在 x86 上偶尔正常，在 ARM 上暴露问题。

### 6. ABA 和 Compare-Exchange

CAS 只看到值从 A 变到 B。如果值经历 A -> C -> A，CAS 可能误以为没有变化。无锁结构常需要版本号、hazard pointer、epoch GC 等技术。

## 决策规则

| 需求 | 推荐工具 | 说明 |
| --- | --- | --- |
| 单生产单消费消息 | channel | 降低共享状态复杂度 |
| 共享配置只读 | `Arc<T>` | 不需要锁 |
| 小状态互斥修改 | `Arc<Mutex<T>>` | 简单可靠 |
| 读多写少缓存 | `RwLock` 或并发 map | 压测验证 |
| 计数器 | atomic relaxed | 只统计不发布数据 |
| 发布初始化数据 | release/acquire | 需要成对设计 |
| 复杂无锁结构 | 成熟库 | 不建议手写 |

## 故障模式

- 锁顺序不一致导致死锁。
- 在持锁状态下 await，导致任务互相等待。
- RwLock 读锁长期占用导致写饥饿。
- 使用 Relaxed 发布数据，另一个线程看见标志但看不见数据。
- 手动 `unsafe impl Send`，但内部裸指针生命周期无法保证。

## 诊断方法

- 给锁加作用域，缩短 guard 生命周期。
- 用 tracing 记录锁等待时间。
- 用 loom 测试并发交错。
- 用 tokio-console 观察异步任务阻塞。
- 对无锁代码优先使用成熟 crate，并查看其 unsafe 文档。

## 练习

1. 实现 `Arc<Mutex<HashMap<K, V>>>` 缓存，并记录锁等待时间。
2. 用 atomic 计数器统计请求数，说明为什么 Relaxed 足够。
3. 用错误锁顺序制造死锁，再写出锁顺序规范。

## 验收

- 能解释 `Send`/`Sync` 如何自动推导。
- 能说明 Relaxed 不适合发布数据。
- 能选择 channel、Mutex、RwLock、atomic 的适用场景。

## 重点

- Rust 防数据竞争，不防所有并发逻辑错误。

## 难点

- 原子内存序的正确性是跨 CPU、编译器优化和数据结构不变量共同决定的。

## 易错

> **易错：** 看到 Mutex 就觉得性能差，过早写无锁结构。
>
> 正确做法：先用锁写出正确性，再测量瓶颈；无锁代码只有在证据充分且审查能力足够时才值得做。
