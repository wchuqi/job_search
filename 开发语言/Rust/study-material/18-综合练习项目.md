## 项目：异步任务追踪服务

实现一个名为 `taskhub` 的 Rust workspace，包含核心库、CLI 和 HTTP 服务。它用于管理任务、保存状态、查询统计，并提供命令行和 Web API 两种入口。

## 项目结构

```text
taskhub/
  Cargo.toml
  crates/
    taskhub-core/
      src/lib.rs
    taskhub-cli/
      src/main.rs
    taskhub-server/
      src/main.rs
  tests/
```

## 功能要求

- `taskhub-core`：
  - 定义 `TaskId`、`Task`、`TaskStatus`。
  - 使用 enum 表达任务状态。
  - 提供 `Repository` Trait。
  - 错误类型使用 `thiserror`。
- `taskhub-cli`：
  - 使用 clap 支持 `add`、`done`、`list`。
  - 支持 JSON 文件存储。
- `taskhub-server`：
  - 使用 axum 提供 `/health`、`/tasks`、`/tasks/:id`。
  - 使用 Tokio 异步运行。
  - 使用 tracing 输出请求日志。

## 分阶段任务

### 阶段 1：核心模型

- 定义结构体和枚举。
- 为任务状态写状态迁移方法。
- 写单元测试覆盖合法和非法迁移。

> **验收：** `cargo test -p taskhub-core` 通过，错误分支有测试。

### 阶段 2：错误和存储

- 定义 `TaskError`。
- 实现内存存储 `MemoryRepository`。
- 扩展 JSON 文件存储。

> **验收：** Repository Trait 有集成测试，文件损坏时返回可诊断错误。

### 阶段 3：CLI

- 用 clap 定义子命令。
- 输出格式支持 human 和 json。
- 对用户输入做错误提示。

> **验收：** 能通过命令添加、完成、列出任务。

### 阶段 4：HTTP 服务

- 用 axum 暴露接口。
- 使用 `Arc` 共享 Repository。
- 增加 tracing span。
- 增加超时和优雅关闭。

> **验收：** `/health` 返回 200，任务接口能被 curl 调用，日志中有请求 ID。

### 阶段 5：质量和发布

- 配置 fmt、clippy、test。
- 增加文档测试。
- 增加 cargo audit 或 cargo deny。
- 编写 release 构建说明。

> **验收：** CI 命令全通过，README 能让别人从零运行。

## 加分任务

- 增加 SQLite 存储。
- 增加分页和过滤。
- 增加并发写入压测。
- 增加 criterion benchmark。
- 用 FFI 封装一个简单 C 函数，并写 Safety 文档。

## 易错

> **易错：** 把所有逻辑写在 `main.rs`。
>
> 正确做法：核心领域逻辑放在库 crate，CLI 和 HTTP 只是适配层。

> **易错：** HTTP handler 直接 `unwrap()`。
>
> 正确做法：实现统一错误响应，把内部错误映射成合适状态码和用户可读信息。
