# Rust学习资料：性能、生产排障、安全和发布

[返回索引](../Rust学习资料.md)

## 学习目标

- 掌握 release 构建、profile、benchmark、profiling。
- 理解日志、tracing、指标、错误上下文和故障定位。
- 掌握供应链安全、交叉编译、发布和运行基线。

## 理论导读

Rust 性能好不等于自动最优。实际瓶颈可能来自算法、分配、锁竞争、序列化、IO、日志、数据库或网络。生产 Rust 项目要同时关注可观测性、可复现构建、安全公告和运行参数。

## 核心心智模型

优化像排队系统分析：先找最长队伍，再看是服务速度慢、排队入口多、锁住了窗口，还是日志和序列化把通道堵住。

## 知识点详解

### 构建和 profile

```toml
[profile.release]
lto = "thin"
codegen-units = 1
panic = "abort"
strip = "symbols"
```

这些配置要按服务类型权衡：吞吐、延迟、二进制体积、构建时间、调试能力都可能互相影响。

### Benchmark 和 Profiling

常用 `criterion` 做微基准，`flamegraph`、perf、heaptrack 或平台工具看热点。微基准不能替代端到端压测。

### 可观测性

`tracing` 比普通日志更适合异步和分布式场景，span 可以串起一次请求的上下文。生产服务至少要有请求 ID、错误链、关键耗时、依赖调用结果和资源指标。

### 安全和供应链

关注 `cargo audit`、`cargo deny`、许可证、crate 来源、构建脚本、默认 feature、unsafe 使用和 CVE。2026 年 Cargo 曾发布安全修复来加强 crate 包解压时的符号链接处理，说明供应链边界不能忽略。

### 发布

发布时要明确目标平台、glibc/musl 差异、交叉编译、配置注入、健康检查、日志格式、崩溃策略和回滚方案。

## 例子

```rust
use tracing::{info, instrument};

#[instrument(skip(password))]
fn login(user: &str, password: &str) -> bool {
    info!(user, "login requested");
    !password.is_empty()
}
```

敏感字段不能进入日志。`skip(password)` 避免密码被 span 自动记录。

## 练习

1. 用 criterion 对两个解析函数做 benchmark。
2. 为 Web 服务增加 tracing span、请求 ID 和错误上下文。
3. 用 `cargo tree` 检查依赖树，用 `cargo audit` 检查安全公告。

## 验收

- 能解释 debug 和 release 性能差异。
- 能用 profiling 数据说明瓶颈，而不是猜测。
- 能列出 Rust 服务发布前检查项。

## 重点

- 性能优化先测量，再定位，再修改，再回归验证。

## 难点

- 异步服务的延迟问题常来自锁、阻塞调用、连接池、背压和任务调度，而不是单个函数慢。

## 易错

> **易错：** 因为 Rust 快，就不做压测和可观测性。
>
> 正确做法：Rust 减少一类内存错误，但生产系统仍会遇到容量、依赖、配置、网络和数据问题。
