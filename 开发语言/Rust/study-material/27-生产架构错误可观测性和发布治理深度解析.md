# Rust学习资料：生产架构、错误、可观测性和发布治理深度解析

[返回索引](../Rust学习资料.md)

## 学习目标

- 能设计可运行、可排障、可升级的 Rust 服务或工具。
- 掌握错误分层、配置优先级、日志脱敏、tracing、metrics、健康检查和优雅关闭。
- 理解发布治理、依赖安全、MSRV、回滚和事故复盘。

## 理论导读

Rust 的类型系统能挡住很多内存和并发数据竞争问题，但生产事故更多来自配置、容量、依赖、重试、超时、数据不一致、日志缺失、错误吞没和发布流程。生产级 Rust 项目必须把“类型安全”扩展成“运行安全”：错误有上下文，调用有超时，队列有容量，配置有优先级，日志能串联请求，发布能回滚。

## 核心心智模型

一个 Rust 服务可以分成四层：领域核心、适配器、运行时平台、治理体系。领域核心用类型表达不变量；适配器处理 HTTP/DB/MQ；运行时平台处理并发、配置和观测；治理体系处理发布、安全和事故。

## 架构分层

```text
crates/
  domain/       # 纯业务类型、状态机、不依赖 Web/DB
  app/          # 用例编排、事务边界、错误映射
  adapters/     # DB、HTTP client、MQ、外部服务
  server/       # axum/actix 路由、配置、运行时、观测
```

这种结构能避免 handler 里堆满业务逻辑，也能让核心逻辑通过单元测试覆盖。

## 错误分层

| 层级 | 目标 | 示例 |
| --- | --- | --- |
| 领域错误 | 表达业务规则失败 | 余额不足、状态不可迁移 |
| 应用错误 | 表达用例失败和依赖失败 | DB 超时、幂等冲突 |
| 传输错误 | 映射到 HTTP/gRPC 响应 | 400、404、409、500 |
| 诊断错误 | 给日志和排障使用 | request_id、source chain、backtrace |

库层适合稳定 enum 错误，应用层可用 `anyhow` 汇总上下文。不要把数据库错误原样暴露给 HTTP 调用方。

## 配置治理

明确优先级：

```text
默认值 < 配置文件 < 环境变量 < 命令行参数
```

配置必须在启动时校验，失败要快速退出。不要在第一次请求时才发现数据库 URL、线程数或证书路径错误。

## 可观测性

### tracing

每个请求至少携带：

- request_id / trace_id。
- 路由、方法、状态码、耗时。
- 关键业务 ID。
- 依赖调用耗时和结果。
- 错误链和分类。

```rust
use tracing::{error, info_span, Instrument};

async fn handle(id: String) {
    async move {
        // business logic
    }
    .instrument(info_span!("handle_request", task_id = %id))
    .await;
}
```

### metrics

核心指标：

- 请求量、错误率、延迟分位数。
- 队列长度、连接池占用、任务数。
- 外部依赖超时、重试、熔断次数。
- 内存、CPU、文件描述符、线程数。

### 日志脱敏

密码、token、cookie、身份证、手机号、银行卡等不得进入日志。derive Debug 的结构体如果包含 secret，需要手动实现 Debug 或使用 secrecy 类库。

## 运行时治理

- 外部请求必须有 timeout。
- 重试必须有退避、上限和幂等保护。
- 任务队列必须有容量。
- 优雅关闭要停止接收新请求、等待在途请求、flush 日志。
- DB 事务边界要和业务状态机一致。
- 后台任务要有生命周期管理，不能 fire-and-forget 后无人监控。

## 发布治理

### 构建

- 固定 toolchain 或声明 MSRV。
- 应用提交 `Cargo.lock`。
- 记录 `rustc --version`、git commit、构建参数。
- 产物包含版本信息。

### CI

```powershell
cargo fmt --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all-features
cargo audit
cargo deny check
```

### 灰度和回滚

- 支持配置开关。
- 数据库迁移向前兼容。
- 新旧版本能短时间共存。
- 指标异常可自动停止发布。
- 回滚不依赖重新构建。

## 事故复盘模板

```markdown
# 事故复盘：标题

## 时间线
## 影响范围
## 触发条件
## 技术根因
## 为什么类型系统没有覆盖
## 诊断证据
## 修复方案
## 回归测试
## 长期治理
```

“为什么类型系统没有覆盖”是 Rust 项目复盘的关键问题：是外部系统、配置、资源耗尽、逻辑状态、unsafe 边界，还是异步取消和重试造成的？

## 常见生产失败模式

- 所有错误映射成 500，调用方无法判断是否可重试。
- `unwrap()` 在边缘输入下触发 panic。
- 日志没有 request_id，无法串联链路。
- timeout 缺失导致任务堆积。
- 重试没有幂等保护，放大写入。
- 连接池上限和运行时 worker 配置不匹配。
- 配置热更新没有版本和校验。
- 依赖升级改变默认 feature 或 MSRV。

## 练习

1. 为综合项目设计错误 enum，并映射成 HTTP Problem Details。
2. 给服务增加 tracing、metrics 和 request_id。
3. 写一个优雅关闭流程，验证 Ctrl+C 后不丢在途请求。
4. 模拟依赖超时和重试风暴，观察队列和延迟。
5. 写一份 Rust 服务发布前检查清单。

## 验收

- 能把 Rust 类型安全延伸到生产运行安全。
- 能设计错误、日志、指标和发布治理。
- 能写出有证据链的生产故障复盘。

## 重点

- 生产 Rust 的核心不是“没有 GC 所以快”，而是用类型、测试、观测和治理共同守住系统边界。

## 难点

- 很多生产问题跨越语言边界：数据库、网络、容器、证书、依赖服务和发布流程都不受 Rust 编译器保护。

## 易错

> **易错：** 把 Rust 编译通过当成生产就绪。
>
> 正确做法：编译通过只是起点，生产就绪还需要错误边界、可观测性、容量验证、安全审计和回滚能力。
