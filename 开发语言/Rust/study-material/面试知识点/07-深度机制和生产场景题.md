## 一、深度机制和生产场景题

### 1. 借用检查器为什么要基于 MIR 和控制流分析？

**参考答案：**

因为源码语法可能隐藏临时变量、分支、move、drop 和 reborrow。MIR 把代码降成更明确的控制流和赋值操作，借用检查器可以基于“值何时被借用、何时最后使用、何时 drop、是否被 move”来判断引用是否有效。NLL 让借用范围按最后一次使用收缩，而不是机械持续到词法作用域结尾。

> **重点：** 借用错误本质上是引用有效性、别名规则或释放顺序无法被证明。
> **易错：** 只通过加生命周期标注修错，而不调整所有权和 API。

### 2. Trait object 的 vtable 里大概有什么？为什么有对象安全限制？

**参考答案：**

`dyn Trait` 通常通过胖指针表示：数据指针和 vtable 指针。vtable 中包含方法地址，以及类型大小、对齐、drop 等元信息。对象安全限制来自动态分发时必须在运行时通过统一表项调用方法；如果方法返回 `Self` 或有泛型参数，调用点需要编译期知道具体类型或生成不同版本，无法放进统一 vtable。

> **难点：** `impl Trait` 隐藏的是一个具体类型，`dyn Trait` 擦除具体类型并走动态分发。

### 3. Cargo feature unification 在生产中会造成什么问题？

**参考答案：**

features 是加法合并模型，同一 crate 被不同依赖路径启用的 feature 会合并。生产中可能导致额外 TLS、压缩、数据库驱动、默认运行时或 unsafe 代码被间接启用，带来编译变慢、二进制变大、平台兼容问题或安全边界扩大。应使用 `cargo tree -e features` 定位 feature 来源。

> **重点：** 实际构建看的是最终 feature 图，不是你在当前 crate 直接写了哪些 feature。

### 4. Future 被取消为什么可能导致数据不一致？

**参考答案：**

Rust 中取消 Future 通常就是 drop Future。任何 `.await` 都可能是取消点。如果一个操作在第一个 await 后已经修改了外部状态，但第二个 await 前被取消，就可能留下半完成状态。解决方式包括事务、幂等设计、补偿动作、显式状态机和把不可中断区间缩小。

```rust
async fn transfer() -> Result<(), Error> {
    debit().await?;
    credit().await?;
    Ok(())
}
```

> **易错：** 认为 async 取消只是不返回结果，不影响外部状态。

### 5. 什么时候 Relaxed 原子内存序足够？什么时候不够？

**参考答案：**

Relaxed 只保证单个原子变量操作本身是原子的，不建立其他内存读写的可见顺序。统计计数器、请求量累加这类不用于发布数据的场景通常足够。如果一个线程写入数据后通过原子标志通知另一个线程读取数据，就需要 Release/Acquire 或更强顺序，否则另一个线程可能看到标志但看不到数据。

> **重点：** 原子操作不是自动同步所有内存。

### 6. 如何审查 `unsafe impl Send`？

**参考答案：**

先看类型内部是否包含裸指针、外部资源、非线程安全引用计数、内部可变性或 FFI 句柄。再确认跨线程移动后析构、访问、回调和释放都安全。必须说明所有权、生命周期、同步策略和底层库线程安全承诺。没有明确不变量和测试工具支撑的 `unsafe impl Send` 应视为高风险。

> **难点：** `unsafe impl Send` 是向编译器承诺线程安全，错误承诺会污染所有安全调用方。

### 7. 线上 Rust 服务 p99 延迟升高，你怎么排查？

**参考答案：**

先确认变更和影响范围，再看指标：请求量、错误率、p50/p95/p99、CPU、内存、连接池、队列长度、外部依赖耗时。Rust/Tokio 服务重点查阻塞调用、锁等待、无限并发、连接池耗尽、日志过量、重试风暴、任务泄漏。用 tracing span 串联请求，用 flamegraph 看 CPU，用 tokio-console 看任务调度和阻塞。

> **重点：** 不要先改代码；先用指标和 trace 缩小范围。

### 8. Rust 项目发布前你会设置哪些质量门禁？

**参考答案：**

至少包括 `cargo fmt --check`、`cargo clippy --all-targets --all-features -- -D warnings`、`cargo test --all-features`、安全审计、许可证检查、MSRV 测试、依赖树检查、release profile 审查、配置校验、日志脱敏、健康检查、优雅关闭、压测和回滚验证。

> **易错：** 只跑单元测试，不检查依赖、安全、发布和运行时配置。
