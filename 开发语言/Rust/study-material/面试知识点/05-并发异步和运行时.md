## 一、并发异步和运行时

### 1. Send 和 Sync 分别是什么？

**参考答案：**

`Send` 表示值可以安全移动到另一个线程；`Sync` 表示 `&T` 可以安全在线程间共享。它们是 marker trait，很多并发 API 会通过 trait bound 要求类型满足这些约束。

> **重点：** `Rc<T>` 通常不是 Send/Sync，跨线程共享用 `Arc<T>`。

### 2. `Arc<Mutex<T>>` 表达什么？

**参考答案：**

`Arc` 提供多线程引用计数共享所有权，`Mutex` 提供互斥访问内部值。两者组合表示多个线程共享同一份可变状态，但每次修改必须先获得锁。

> **易错：** Rust 防数据竞争，不自动防死锁。

### 3. async fn 为什么返回 Future？

**参考答案：**

`async fn` 会被编译成状态机，调用时返回 Future。Future 需要 executor poll 才会推进，遇到未就绪 IO 会挂起并等待 waker 唤醒。

```rust
async fn value() -> u32 { 1 }
```

> **难点：** Future 被 drop 就可能取消，涉及状态修改时要考虑取消安全。

### 4. Tokio 中为什么不能随便阻塞？

**参考答案：**

Tokio worker 线程负责调度很多异步任务。长时间阻塞会占住 worker，让其他任务无法推进，导致延迟上升甚至超时。阻塞 IO 或 CPU 密集任务应使用异步 API、专门线程池或 `spawn_blocking`。

> **重点：** async 是协作式调度，不是自动无限并行。
