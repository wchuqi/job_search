## 一、unsafe FFI 和生产实践

### 1. unsafe 允许做哪些事情？

**参考答案：**

unsafe 允许解引用原始指针、调用 unsafe 函数或方法、访问或修改可变静态变量、实现 unsafe trait、访问 union 字段。unsafe 不关闭类型检查和借用检查，只是允许少数编译器无法证明安全的操作。

> **重点：** unsafe 代码必须维护额外安全不变量。

### 2. 如何审查一个 unsafe 封装？

**参考答案：**

看 unsafe 块是否足够小，外部 API 是否安全，是否写明 `# Safety`，是否说明指针有效性、对齐、别名、生命周期、线程安全、释放责任和 panic 边界。还要看测试、Miri、sanitizer 或 fuzz 是否覆盖关键路径。

> **难点：** unsafe 正确性常依赖跨函数、跨线程或跨语言的不变量。

### 3. FFI 主要风险有哪些？

**参考答案：**

主要风险包括 ABI 不匹配、数据布局不一致、字符串编码和空终止、缓冲区长度错误、所有权和释放责任不清、panic 穿过 C 边界、回调生命周期和线程安全问题。

> **重点：** `#[repr(C)]` 只解决布局的一部分，不解决所有权和生命周期。

### 4. Rust 服务上线前要检查什么？

**参考答案：**

至少检查 release profile、配置优先级、错误上下文、tracing 和指标、依赖安全审计、secret 脱敏、健康检查、优雅关闭、压测、容量、镜像体积、目标平台和回滚方案。

> **易错：** 只因为 Rust 内存安全就忽略生产工程。
