# Java 学习资料：安全、编码规范、可维护性和工程实践

[返回索引](../Java学习资料.md)

## 学习目标

完成本章后，你应该能：

- 识别 Java 常见安全风险。
- 写出可维护、可测试、可演进的代码。
- 管理配置、依赖、日志和敏感信息。
- 理解代码规范、分层和重构原则。

## 理论导读：安全和可维护性是在为未知变化预留余地

安全和工程实践不是项目末尾才补的检查项，而是代码每天面对外部输入、依赖变化、团队协作和线上故障时的基本生存能力。外部输入可能恶意或格式错误，依赖可能有漏洞，日志可能泄露敏感信息，过度耦合的代码会让一个小需求牵动大片修改。好的工程实践是在一开始就把风险隔离到可控边界。

安全可以从“信任边界”理解：凡是跨过系统边界进来的数据，都不能默认可信。HTTP 参数、JSON、文件、消息、数据库旧数据、第三方接口响应，都可能带来注入、越权、反序列化、SSRF、XSS 或资源耗尽风险。输入校验、参数化 SQL、类型白名单、大小限制、密钥管理、依赖扫描，本质上都是在边界上设置检查站。

可维护性则关心代码多年后是否还能被理解和修改。清晰命名、短方法、单一职责、不可变对象、明确异常边界、稳定分层，让代码像分区明确的建筑：哪里接收请求、哪里表达业务、哪里访问数据库、哪里处理外部系统，都有固定位置。重构不是为了好看，而是为了让下一次变化不至于破坏已有行为。

## 一、输入校验

所有外部输入都不可信：

- HTTP 参数
- JSON body
- Header
- 文件上传
- 消息队列
- 数据库内容
- 第三方接口响应

示例：

```java
if (email == null || !email.contains("@")) {
    throw new IllegalArgumentException("invalid email");
}
```

Web 项目使用 Bean Validation：

```java
public record RegisterRequest(
        @Email String email,
        @Size(min = 8, max = 64) String password
) {}
```

## 二、SQL 注入防护

使用参数化查询：

```java
PreparedStatement ps = connection.prepareStatement(
        "select * from users where email = ?"
);
ps.setString(1, email);
```

不要拼接：

```java
"select * from users where email = '" + email + "'"
```

## 三、反序列化风险

不要反序列化不可信输入。Java 原生序列化尤其危险。

建议：

- 使用 JSON / Protobuf 等格式。
- 对类型白名单。
- 限制输入大小。
- 对依赖库保持更新。

## 四、敏感信息

不要提交：

- 密码
- token
- 私钥
- `.env`
- 生产配置

日志中不要输出：

- Authorization header
- password
- access token
- 身份证号
- 银行卡号

## 五、依赖安全

建议：

- 定期升级依赖。
- 使用依赖漏洞扫描。
- 锁定版本。
- 删除不用依赖。
- 关注 transitive dependency。

Maven：

```bash
mvn dependency:tree
```

## 六、编码规范

建议：

- 类名名词，方法名动词。
- 方法短小，职责单一。
- 变量名表达业务含义。
- 避免魔法值。
- 避免过深嵌套。
- 公共 API 明确异常和边界。

## 七、不可变性

优先使用不可变对象：

```java
public record Money(BigDecimal amount, String currency) {
    public Money {
        Objects.requireNonNull(amount);
        Objects.requireNonNull(currency);
    }
}
```

好处：

- 线程安全。
- 易测试。
- 易推理。

## 八、空值策略

建议：

- 方法参数明确是否允许 null。
- 返回集合时优先空集合，不返回 null。
- Optional 适合返回值，不适合字段和参数滥用。

```java
return users == null ? List.of() : List.copyOf(users);
```

## 九、错误处理规范

建议：

- 边界层统一处理异常。
- 业务异常有错误码。
- 保留原始 cause。
- 不吞异常。
- 不重复打印异常。

## 十、日志规范

```java
log.info("order created, orderId={}", orderId);
log.warn("payment retry, orderId={}, attempt={}", orderId, attempt);
log.error("payment failed, orderId={}", orderId, e);
```

避免：

```java
log.info("password={}", password);
```

## 十一、分层和依赖方向

建议依赖方向：

```text
controller -> application -> domain <- infrastructure
```

核心业务不应依赖具体 Web 或数据库框架。

## 十二、重构原则

常见坏味道：

- 超长方法。
- 上帝类。
- 重复代码。
- 过度继承。
- 过多 if/else 类型判断。
- 数据类没有行为。

重构方向：

- 提取方法。
- 提取对象。
- 用多态替代条件。
- 用组合替代继承。
- 引入值对象。

## 十三、兼容性和升级

JDK 升级检查：

- 编译插件版本。
- 依赖是否支持新 JDK。
- 反射访问是否受模块限制。
- GC 参数是否仍有效。
- CI / Docker 镜像。
- 字符编码和 locale。
- 测试覆盖。

## 练习

审查一个用户注册接口：

- 输入校验。
- 密码不写日志。
- SQL 参数化。
- 业务异常统一返回。
- 单元测试覆盖非法输入。

- 能设计清晰分层。
