# Java 学习资料：异常处理、断言和错误边界

[返回索引](../Java学习资料.md)

## 学习目标

完成本章后，你应该能：

- 区分 checked exception、unchecked exception 和 error。
- 正确使用 try-catch-finally 和 try-with-resources。
- 设计业务异常和错误边界。
- 避免吞异常、滥用异常和错误日志重复打印。

## 理论导读：异常是在正常路线旁边铺一条可追踪的故障通道

程序运行不是只有“成功返回”一条路。文件可能不存在，数据库可能断开，参数可能非法，内存也可能耗尽。异常机制的意义，是把这些偏离正常路线的情况用明确的对象携带出去，让调用链知道哪里出错、为什么出错、还能不能恢复。它像一张带原因和现场信息的故障单，而不是一句模糊的“失败了”。

checked exception、unchecked exception 和 `Error` 的区别，本质上是在表达责任边界。checked exception 通常告诉调用方“这个问题你应该考虑处理或继续声明”，例如 IO 失败；unchecked exception 更多表示代码假设被破坏，例如参数非法、状态不对；`Error` 则通常是 JVM 或系统层面的严重问题，业务代码不应该把它当普通异常吞掉。

好的异常设计要保留因果链，也要控制传播边界。底层可以把 `SQLException` 包装成业务可理解的 `UserLoadException`，但必须保留 cause；Web、消息消费、定时任务、CLI 入口等边界层负责统一记录日志和转换响应。否则异常会变成两种坏结果：要么一路丢失上下文，要么每层都打印日志造成噪音。

## 一、异常层次

```text
Throwable
  |-- Error
  |-- Exception
        |-- RuntimeException
```

| 类型 | 含义 | 是否必须捕获或声明 |
| --- | --- | --- |
| checked exception | 可恢复或调用方应处理的问题 | 是 |
| unchecked exception | 编程错误或运行期异常 | 否 |
| Error | JVM 或系统级严重错误 | 否 |

例子：

- `IOException`：checked exception。
- `IllegalArgumentException`：unchecked exception。
- `OutOfMemoryError`：Error。

> **重点：** 不要捕获 `Throwable` 作为常规异常处理。

## 二、try-catch-finally

```java
try {
    risky();
} catch (IllegalArgumentException e) {
    System.err.println("bad argument: " + e.getMessage());
} finally {
    cleanup();
}
```

多个 catch：

```java
try {
    read();
} catch (java.io.FileNotFoundException e) {
    throw new IllegalStateException("file not found", e);
} catch (java.io.IOException e) {
    throw new IllegalStateException("read failed", e);
}
```

> **易错：** catch 顺序要从具体到宽泛，否则具体异常无法匹配。

## 三、try-with-resources

适合自动关闭资源：

```java
try (var reader = java.nio.file.Files.newBufferedReader(path)) {
    return reader.readLine();
}
```

资源必须实现 `AutoCloseable`。

> **重点：** 文件、流、数据库连接、HTTP 响应等资源应优先使用 try-with-resources。

## 四、异常转换

底层异常不一定适合直接暴露给上层。

```java
public User loadUser(long id) {
    try {
        return repository.findById(id);
    } catch (java.sql.SQLException e) {
        throw new UserLoadException("failed to load user: " + id, e);
    }
}
```

保留 cause：

```java
throw new UserLoadException("failed", e);
```

> **易错：** 包装异常时丢失原始 cause，会严重影响排障。

## 五、业务异常设计

```java
public class BusinessException extends RuntimeException {
    private final String code;

    public BusinessException(String code, String message) {
        super(message);
        this.code = code;
    }

    public String code() {
        return code;
    }
}
```

建议：

- 异常信息包含关键上下文，但不要包含敏感信息。
- 业务异常带错误码。
- 不要用异常控制正常流程。
- 边界层统一转换为错误响应。

## 六、断言 assert

```java
assert amount > 0 : "amount must be positive";
```

启用断言：

```bash
java -ea App
```

> **易错：** 断言默认关闭，不应用于业务参数校验。

## 七、日志和异常

不建议每层都打印日志：

```java
try {
    service.process();
} catch (Exception e) {
    log.error("process failed", e);
    throw e;
}
```

如果上层还会记录，这会造成重复日志。

建议：

- 在边界层记录异常。
- 中间层补充上下文并抛出。
- 不要吞异常。

坏例子：

```java
try {
    save();
} catch (Exception e) {
    // ignore
}
```

## 八、常见异常

| 异常 | 常见原因 |
| --- | --- |
| `NullPointerException` | 空引用访问 |
| `IllegalArgumentException` | 参数非法 |
| `IllegalStateException` | 对象状态不允许当前操作 |
| `IndexOutOfBoundsException` | 下标越界 |
| `ClassCastException` | 类型转换错误 |
| `ConcurrentModificationException` | 迭代时结构修改 |
| `IOException` | 文件或网络 IO 失败 |
| `SQLException` | 数据库访问失败 |

## 九、错误边界

错误边界是系统中统一处理异常的位置，例如：

- Web Controller Advice
- 消息消费入口
- 定时任务入口
- CLI main 方法
- 线程任务包装器

示例：

```java
public static void main(String[] args) {
    try {
        new App().run(args);
    } catch (BusinessException e) {
        System.err.println(e.getMessage());
        System.exit(1);
    }
}
```

## 练习

实现一个文件读取服务：

- 文件不存在时抛出业务异常。
- 读取失败时保留原始 cause。
- 使用 try-with-resources。
- 不在底层重复打印日志。
