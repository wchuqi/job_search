# Java 学习资料：面向对象、封装、继承和多态

[返回索引](../Java学习资料.md)

## 学习目标

完成本章后，你应该能：

- 用类和对象表达领域模型。
- 正确使用封装、继承、组合和多态。
- 理解接口、抽象类、默认方法。
- 正确实现 `equals`、`hashCode`、`toString`。

## 理论导读：对象不是数据袋，而是带规则的业务角色

面向对象的核心不是“把字段和方法写进同一个 class”，而是让程序中的概念拥有边界、责任和规则。一个 `Account` 不应该只是暴露 `balance` 的数据袋，它应该知道“余额不能随便变成负数”“充值金额必须大于 0”“转账需要检查状态”。封装就是把状态和保护状态的规则放在一起，让对象像一个有门禁的房间：外部不能随意改里面的东西，只能通过被允许的入口操作。

继承和多态解决的是“相同协议下的不同行为”。当代码声明 `PaymentMethod method` 时，它关心的是“能支付”这个能力，而不是具体是信用卡、支付宝还是余额支付；运行期实际对象决定调用哪个实现。多态像插座和电器：插座只规定接口，台灯、电脑、充电器各自完成自己的工作。这样可以减少 `if-else` 分支，让新增行为更自然。

但继承也会把父类和子类强绑定在一起。父类像一份会被子类共享的合同和内部结构，一旦父类变化，所有子类都可能被牵动。因此真实项目里更常使用组合：把可替换的能力作为成员对象注入进来，让对象通过协作完成任务，而不是靠漫长继承链复用一点代码。

## 一、类和对象

类像一张对象蓝图，描述对象有哪些状态和行为；对象则是按这张蓝图创建出来的具体实例。`User` 类说明“用户应该有 id、name、rename 行为”，每个 `new User(...)` 才是内存里真实存在的用户对象。字段保存对象状态，方法表达对象能做什么，构造器负责让对象一出生就处在合法状态。

设计类时不要只问“需要哪些字段”，还要问“这些字段必须满足什么规则”。如果一个对象创建后马上可能处于非法状态，后续每个方法都要防御它；如果构造器和方法入口已经守住规则，对象内部就更容易保持一致。

```java
public class User {
    private final long id;
    private String name;

    public User(long id, String name) {
        this.id = id;
        this.name = name;
    }

    public long id() {
        return id;
    }

    public String name() {
        return name;
    }

    public void rename(String newName) {
        if (newName == null || newName.isBlank()) {
            throw new IllegalArgumentException("name is blank");
        }
        this.name = newName;
    }
}
```

> **重点：** 字段通常应 private，通过方法维护对象不变量。

## 二、封装

封装的价值是把“状态”和“能改变状态的规则”关在同一个边界里。外部代码不应该绕过规则直接改余额、订单状态或用户权限，否则业务对象就会变成任何人都能乱写的共享表格。好的封装让对象自己保护不变量，例如余额不能为负、订单取消后不能支付、用户名不能为空。

`private` 只是封装的技术手段，不是封装本身。如果一个类把所有字段设为 private，却给每个字段生成无脑 getter/setter，外部仍然可以随意把对象改坏。真正的封装应该暴露业务动作，例如 `deposit`、`cancel`、`rename`，而不是暴露内部字段的任意读写开关。

封装不是简单地把字段设为 private，而是把状态和规则放在一起。

坏例子：

```java
public class Account {
    public int balance;
}
```

较好例子：

```java
public class Account {
    private long balance;

    public void deposit(long amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("amount must be positive");
        }
        balance += amount;
    }
}
```

> **易错：** getter / setter 自动生成不等于良好封装。对象应保护自己的业务规则。

## 三、构造器和初始化

构造器是对象合法性的第一道门。对象创建时，字段会先有默认值，再执行字段初始化、初始化块和构造器逻辑。构造器应该把必须存在的依赖和关键字段一次性准备好，并拒绝非法参数，这样对象创建完成后就能被安全使用。

初始化顺序影响 `final` 字段、继承层次和复杂对象构建。不要在构造过程中调用可能被子类重写的方法，因为此时子类字段可能还没有初始化完成。简单对象优先使用明确构造器，参数过多时再考虑 builder 或工厂方法。

```java
public class Product {
    private final String name;
    private final long price;

    public Product(String name, long price) {
        if (name == null || name.isBlank()) {
            throw new IllegalArgumentException("name is blank");
        }
        if (price < 0) {
            throw new IllegalArgumentException("price is negative");
        }
        this.name = name;
        this.price = price;
    }
}
```

字段初始化顺序：

1. 默认值。
2. 字段初始化表达式。
3. 初始化块。
4. 构造器。

## 四、static

`static` 成员属于类，而不是某个对象。可以把它理解成挂在蓝图上的共享信息：所有实例看到的是同一份静态字段，调用静态方法也不需要某个具体对象。工具方法、常量、无状态工厂方法适合 static；可变 static 状态则要非常谨慎。

可变 static 字段像全局共享抽屉，任何地方都可能改它。它会让测试互相影响，也会在并发环境中产生竞态，还会让代码依赖隐藏状态。除非明确需要全局共享，并且已经处理线程安全和生命周期，否则不要轻易使用可变 static 状态。

`static` 成员属于类，不属于某个对象。

```java
public class IdGenerator {
    private static long nextId = 1;

    public static long next() {
        return nextId++;
    }
}
```

> **易错：** 可变 static 状态在并发和测试中容易出问题。

## 五、继承

继承表达的是稳定的 is-a 关系：子类应该在语义上确实是父类的一种，并且能遵守父类对外承诺。`Dog extends Animal` 只有在所有使用 `Animal` 的地方都能安全接收 `Dog` 时才合理。继承不是单纯为了复用几个字段或方法，因为它会把父类实现细节和子类强绑定。

继承层次越深，理解成本越高。读一个子类方法时，可能需要同时查看父类构造器、父类字段、被重写的方法和模板方法调用顺序。业务代码里如果只是想复用能力，通常把能力抽成成员对象组合进来，比继承更灵活。

```java
public class Animal {
    public void speak() {
        System.out.println("...");
    }
}

public class Dog extends Animal {
    @Override
    public void speak() {
        System.out.println("woof");
    }
}
```

Java 只支持单继承，但可以实现多个接口。

> **重点：** 优先使用组合而不是继承，除非确实存在稳定的 is-a 关系。

## 六、多态

多态让代码面向抽象协议，而不是面向具体实现。变量声明为父类或接口类型，运行期实际对象决定调用哪个重写方法。这样调用方只依赖“会做什么”，不依赖“具体怎么做”，新增实现时调用方代码通常不用修改。

动态分派只发生在实例方法上。字段访问看的是引用声明类型，静态方法也不是多态调用。理解这一点，才能避免把“父类字段”和“子类字段”混在一起，或者误以为 static 方法能像普通实例方法一样被运行期对象动态选择。

```java
Animal animal = new Dog();
animal.speak(); // woof
```

编译期看引用类型，运行期动态分派到实际对象方法。

> **难点：** 字段没有多态，方法才有动态分派。

## 七、抽象类和接口

抽象类和接口都能表达抽象，但侧重点不同。抽象类适合描述一类对象的共同骨架，可以包含共享状态、构造器和模板逻辑；接口更像能力协议，强调“实现者必须能做什么”，不关心它内部怎么保存状态。Java 单继承、多接口的设计，也让接口更适合表达可组合能力。

选择时要看你是在建模“同一类事物”，还是建模“某种能力”。`Shape` 有共同的面积计算语义，可以是抽象类或接口；`Payable`、`Serializable`、`Comparable` 更像能力标签或协议，通常用接口。不要因为接口可以有 default 方法，就把大量共享状态和复杂模板塞进接口。

抽象类：

```java
abstract class Shape {
    abstract double area();
}
```

接口：

```java
interface Payable {
    void pay(long amount);

    default boolean supportsRefund() {
        return false;
    }
}
```

选择建议：

| 场景 | 建议 |
| --- | --- |
| 需要共享状态和模板逻辑 | 抽象类 |
| 定义能力或协议 | 接口 |
| 需要多实现 | 接口 |

## 八、重载和重写

重载发生在编译期，编译器根据方法名和参数列表选择调用哪个方法；重写发生在运行期，JVM 根据实际对象类型选择子类实现。一个解决“同名不同参数”的问题，一个解决“同一协议不同行为”的问题。

它们名字相似，但使用场景完全不同。重载可以让 API 更自然，例如 `print(String)` 和 `print(int)`；重写则是多态的基础，例如不同支付方式实现同一个 `pay` 方法。写重写方法时使用 `@Override`，能让编译器帮你发现签名写错导致没有真正重写的问题。

重载 overload：同名方法，参数不同，编译期决定。

```java
void print(String text) {}
void print(int value) {}
```

重写 override：子类重新实现父类方法，运行期动态分派。

```java
@Override
public String toString() {
    return "User";
}
```

> **易错：** 返回值不同但参数相同，不能构成重载。

## 九、equals 和 hashCode

默认情况下，对象相等通常按身份判断，也就是是不是同一个对象。但很多业务对象需要按值相等，例如两个 `UserId(1)` 应该表示同一个用户编号。`equals` 定义业务相等，`hashCode` 定义对象进入哈希结构时的桶位置，两者必须保持一致：相等对象必须有相同 hashCode。

如果只重写 `equals` 不重写 `hashCode`，对象放进 `HashSet` 或作为 `HashMap` key 时会出现“明明相等却找不到”的问题。可以把 HashMap 想成先按 hashCode 找柜子，再在柜子里用 equals 找具体物品；柜子编号错了，后面再怎么比较也找不到。

如果对象需要按业务值比较，应重写：

```java
import java.util.Objects;

public class UserId {
    private final long value;

    public UserId(long value) {
        this.value = value;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof UserId userId)) return false;
        return value == userId.value;
    }

    @Override
    public int hashCode() {
        return Objects.hash(value);
    }
}
```

> **重点：** 重写 `equals` 必须同时重写 `hashCode`，否则 HashMap / HashSet 会出问题。

## 十、toString

`toString` 是对象的可读展示，主要服务于日志、调试和排障。好的 `toString` 应该包含关键业务标识，让人能快速判断对象是什么；但它不是审计日志，也不是完整数据导出，更不应该泄露密码、token、身份证号等敏感字段。

record 会自动生成 `toString`，普通类需要按场景实现。线上日志里对象展示越常见，越要关注敏感信息和输出体积，避免把大集合、二进制内容或隐私字段直接打印出来。

```java
@Override
public String toString() {
    return "User{id=%d, name='%s'}".formatted(id, name);
}
```

注意不要在 `toString` 中输出敏感信息，例如密码、token、身份证号。

## 十一、组合优于继承

组合的思想是“对象通过协作完成任务”。`OrderService` 不需要继承 `PaymentClient` 才能支付，它只需要持有一个 `PaymentClient` 并调用它。这样支付能力可以被替换、mock、扩展，也不会把服务类锁死在某个父类层次里。

组合比继承更接近真实系统的变化方式。需求变化时，我们常常需要替换某个协作者，例如换支付渠道、换通知方式、换存储实现；组合让这些变化集中在依赖装配处，而不是修改继承树。继承仍然有价值，但更适合稳定抽象，不适合为了省几行代码而滥用。

继承容易导致：

- 父类变更影响子类。
- 子类暴露不该暴露的行为。
- 继承层次僵化。

组合示例：

```java
class OrderService {
    private final PaymentClient paymentClient;

    OrderService(PaymentClient paymentClient) {
        this.paymentClient = paymentClient;
    }
}
```
