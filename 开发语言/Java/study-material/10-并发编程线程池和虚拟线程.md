# Java 学习资料：并发编程、线程池和虚拟线程

[返回索引](../Java学习资料.md)

## 学习目标

完成本章后，你应该能：

- 理解线程、共享状态和线程安全。
- 正确使用 synchronized、volatile、Lock、Atomic。
- 掌握线程池和 CompletableFuture。
- 理解 Java 内存模型和 happens-before。
- 使用 JDK 21 虚拟线程处理 IO 密集任务。

## 理论导读：并发是在多个执行线索之间管理共享和等待

并发不是“同时多开几个线程”这么简单，而是在多个执行线索之间管理两件事：共享状态和等待成本。只要多个线程能读写同一个对象，就会出现谁先写、谁后读、读到的是不是最新值、修改是不是被打断这些问题。并发编程的核心，就是让这些不确定性变成可解释、可控制、可验证的规则。

可以把共享变量想象成多人共用的一块白板。没有同步时，每个人可能看的是自己缓存里的旧照片，也可能两个人同时擦写导致内容被覆盖；`synchronized` 像给白板加一把锁，同一时间只允许一个人进入，并且离开时把修改刷新给下一个人；`volatile` 像要求大家每次都看最新公告，但它不负责把“读取、计算、写回”这三个动作合成一个不可分割的整体；Atomic 类则用硬件级 CAS 尝试在无锁情况下完成原子更新。

线程池和虚拟线程解决的是另一类问题：线程创建、调度和阻塞的成本。平台线程像昂贵的工作台，数量过多会压垮系统；线程池通过复用工作台控制成本；虚拟线程则像轻量任务座位，让阻塞式 IO 可以用接近“一任务一线程”的直观模型表达。但虚拟线程不会让 CPU 算得更快，也不能替代数据库连接池、限流和背压。

## 一、线程基础

线程是程序中的一条执行线索。一个进程可以有多个线程，它们共享同一个进程里的堆对象、类元数据和文件句柄，但每个线程有自己的调用栈。可以把多线程程序想成多个人在同一个办公室工作：每个人有自己的待办清单和桌面，但共享文件柜、白板和打印机。

`start()` 和直接调用 `run()` 的区别非常关键。`start()` 会请求 JVM 创建新的执行线索，让任务在新线程里跑；直接调用 `run()` 只是普通方法调用，仍然在当前线程执行。`join()` 则表示当前线程等待目标线程完成，像等另一位同事把手头任务交回来。

```java
Thread thread = new Thread(() -> System.out.println("hello"));
thread.start();
thread.join();
```

Runnable：

```java
Runnable task = () -> doWork();
new Thread(task).start();
```

Callable：

```java
Callable<Integer> task = () -> 42;
```

## 二、线程安全问题

线程安全问题来自共享可变状态。只读共享通常安全，不共享也通常安全；真正危险的是多个线程同时读写同一个可变对象。`value++` 看起来是一行，实际至少包含读取、加一、写回三个步骤，两个线程交错执行时，就可能都读到旧值，最后丢失一次更新。

判断一个类是否线程安全，要问：哪些状态会被多个线程访问，是否可变，每次读写是否需要原子性、可见性和顺序保证。解决方式也不只一种：可以避免共享，可以让对象不可变，可以加锁，可以使用原子类，也可以把状态限定在线程内部。

错误示例：

```java
class Counter {
    private int value;

    void increment() {
        value++;
    }
}
```

`value++` 不是原子操作。

解决：

```java
class Counter {
    private final AtomicInteger value = new AtomicInteger();

    void increment() {
        value.incrementAndGet();
    }
}
```

## 三、synchronized

`synchronized` 同时提供互斥和可见性。互斥意味着同一把锁保护的代码块，同一时间只能有一个线程进入；可见性意味着一个线程释放锁前对共享变量的修改，对随后获得同一把锁的线程可见。它不是简单的“排队执行”，而是在 JVM 内存模型中建立了明确的 happens-before 关系。

锁对象就是门禁卡。同步实例方法默认锁 `this`，同步静态方法默认锁 `Class` 对象，显式同步块可以指定锁对象。锁选错了，保护范围就会失效；锁范围过大，则会降低并发度甚至增加死锁风险。

```java
class SafeCounter {
    private int value;

    synchronized void increment() {
        value++;
    }

    synchronized int value() {
        return value;
    }
}
```

作用：

- 互斥
- 可见性

## 四、volatile

`volatile` 适合表达“这个变量的最新值必须被其他线程及时看见”。它常用于停止标志、状态开关、配置引用等场景。被 `volatile` 修饰的变量，写入会被刷新到主内存，读取会从主内存获取最新值，同时还会限制相关指令重排序。

但 `volatile` 不会把多个操作合成一个原子动作。`count++` 仍然是读、加、写三个步骤，即使每次读写都可见，两个线程也可能交错导致丢更新。可以把 `volatile` 想成“公告栏内容及时刷新”，但不能保证“多人同时修改公告时不会覆盖彼此”。

```java
class Worker {
    private volatile boolean running = true;

    void stop() {
        running = false;
    }
}
```

`volatile` 保证可见性和有序性，但不保证复合操作原子性。

> **易错：** `volatile int count; count++` 仍然不是线程安全的。

## 五、Lock 和 Condition

`Lock` 是比 `synchronized` 更显式的锁工具。它把加锁和解锁写成方法调用，因此必须在 `finally` 中释放，否则异常可能让锁永远不释放。显式锁的优势是能力更丰富：可以尝试加锁、可中断等待、设置公平性，还能配合多个 `Condition` 管理不同等待队列。

`Condition` 像一把锁下的多个休息室。线程拿到锁后，如果条件不满足，可以进入某个条件队列等待；其他线程改变状态后再通知对应队列。它适合生产者消费者、资源池、复杂状态机等需要精细等待通知的场景。

```java
Lock lock = new ReentrantLock();

lock.lock();
try {
    // critical section
} finally {
    lock.unlock();
}
```

适合需要：

- 可中断锁
- 尝试加锁
- 多条件队列

## 六、线程池

线程池的本质是复用平台线程，并控制任务排队和拒绝策略。创建平台线程成本高，数量过多会造成上下文切换、内存占用和调度压力；线程池像一组固定或可伸缩的工作台，任务来了先交给空闲工作台，忙不过来就进入队列，队列也满了就触发拒绝策略。

理解线程池不能只背参数名。核心线程数决定常驻工作台数量，最大线程数决定高峰时最多扩张到多少，队列决定任务等待方式，拒绝策略决定系统过载时如何反馈。生产系统里，线程池大小还要结合 CPU 核数、任务是否阻塞、外部资源容量和响应时间目标来定。

```java
ExecutorService executor = Executors.newFixedThreadPool(4);
try {
    Future<Integer> future = executor.submit(() -> 42);
    System.out.println(future.get());
} finally {
    executor.shutdown();
}
```

自定义线程池：

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
        4,
        8,
        60,
        TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(100),
        new ThreadPoolExecutor.CallerRunsPolicy()
);
```

> **重点：** 生产代码应理解核心线程数、最大线程数、队列、拒绝策略，而不是盲目使用默认工厂方法。

## 七、CompletableFuture

`CompletableFuture` 表达的是异步任务的结果和后续编排。它可以把“任务完成后转换结果”“两个任务都完成后合并”“失败时兜底”这些关系写成链式管道。它适合并行调用多个独立服务、组合异步结果、把回调逻辑变得更结构化。

真正需要注意的是执行器。很多异步方法默认使用 common pool，如果里面跑的是阻塞 IO，可能占满公共线程影响其他任务。业务代码应根据任务性质提供明确的 Executor，并清楚每一步是在当前线程执行，还是提交到线程池执行。

```java
CompletableFuture<String> future = CompletableFuture
        .supplyAsync(() -> loadUser())
        .thenApply(User::name)
        .exceptionally(ex -> "anonymous");
```

组合：

```java
CompletableFuture<User> user = loadUserAsync();
CompletableFuture<Order> order = loadOrderAsync();

CompletableFuture<String> result = user.thenCombine(order,
        (u, o) -> u.name() + ":" + o.id());
```

> **易错：** 默认使用 common pool，不适合所有业务场景。阻塞任务应使用明确的 Executor。

## 八、并发集合

并发集合不是普通集合外面随便包一层锁，而是针对并发访问重新设计的数据结构。`ConcurrentHashMap` 允许多个线程在不同区域上并发操作，`BlockingQueue` 把队列和等待通知结合起来，`CopyOnWriteArrayList` 则通过写时复制换取读操作无锁。

选择并发集合也要看访问模式。读多写少可以考虑 `CopyOnWriteArrayList`，高并发计数和映射更新适合 `ConcurrentHashMap` 配合 `merge` 或 `compute`，生产者消费者适合 `BlockingQueue`。不要因为集合名字里有 Concurrent 就忽略复合操作的线程安全边界。

常用：

- `ConcurrentHashMap`
- `CopyOnWriteArrayList`
- `BlockingQueue`
- `ConcurrentLinkedQueue`
- `ArrayBlockingQueue`
- `LinkedBlockingQueue`

示例：

```java
ConcurrentHashMap<String, Integer> counts = new ConcurrentHashMap<>();
counts.merge("java", 1, Integer::sum);
```

## 九、Java 内存模型

Java 内存模型定义的是线程之间如何看见共享变量的变化，以及编译器和处理器在什么范围内可以重排序。它解决的不是“变量放在哪个物理内存条上”，而是“多线程读写共享数据时，哪些顺序和可见性是有保证的”。没有这套规则，并发程序的正确性只能靠运气。

happens-before 是 JMM 里最重要的判断工具。若 A happens-before B，那么 A 的结果对 B 可见，并且 A 在顺序上先于 B。`synchronized` 解锁到后续加锁、`volatile` 写到后续读、`start` 到新线程动作、线程动作到 `join` 返回，都是常见的 happens-before 关系。

JMM 规定线程间如何可见、如何排序。

关键概念：

- 原子性
- 可见性
- 有序性
- happens-before

常见 happens-before：

- synchronized 解锁 happens-before 后续加锁。
- volatile 写 happens-before 后续读。
- 线程 start happens-before 新线程动作。
- 线程动作 happens-before join 返回。

## 十、虚拟线程

虚拟线程是 JDK 21 中正式可用的轻量线程。它仍然是 `Thread`，但不再一对一长期占用操作系统平台线程；当虚拟线程遇到可挂起的阻塞 IO 时，JVM 可以把它从平台线程上卸载，等 IO 就绪后再继续执行。这让阻塞式代码可以保持直观写法，同时支撑更高并发等待。

虚拟线程解决的是“等待很贵”的问题，不是“计算很慢”的问题。HTTP 请求、数据库查询、文件 IO 这类大部分时间在等外部系统的任务很适合；纯 CPU 计算仍然受核心数限制；数据库连接、远程服务限流、内存容量也不会因为虚拟线程变多而自动增加，所以仍然需要资源隔离和背压。

JDK 21 中虚拟线程正式可用。

```java
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    Future<String> future = executor.submit(() -> fetchData());
    System.out.println(future.get());
}
```

直接创建：

```java
Thread.startVirtualThread(() -> doBlockingIo());
```

适合：

- IO 密集任务
- 阻塞式代码
- 大量并发请求

不适合：

- CPU 密集任务加速
- 长时间持有 synchronized 锁的阻塞操作
- 依赖线程池大小控制资源的旧模型

> **重点：** 虚拟线程降低线程创建和阻塞成本，但不会让 CPU 计算变快。

## 十一、结构化并发

结构化并发的思想是把一组相关并发任务放进同一个作用域管理，而不是让任务散落在各处。父任务可以等待所有子任务完成，也可以在某个子任务失败时取消其他任务，并统一处理结果。它像把多个并发分支装进一个有入口和出口的代码块，让生命周期更清楚。

在 JDK 21 中结构化并发仍是预览 API，因此学习重点是理解思想，而不是默认用于生产。实际项目中可以先用稳定的 Executor、CompletableFuture 或虚拟线程执行器，等团队确认 JDK 版本和预览特性策略后再考虑引入。

结构化并发在 JDK 21 中仍是预览 API。核心思想是把一组并发任务当作一个结构化作用域管理，便于取消、失败传播和结果汇总。

> **易错：** 预览 API 需要 `--enable-preview`，不应默认作为稳定生产 API 使用。

## 十二、常见并发问题

| 问题 | 说明 |
| --- | --- |
| 竞态条件 | 多线程读写共享状态 |
| 死锁 | 线程互相等待锁 |
| 活锁 | 线程不断让步但无进展 |
| 饥饿 | 某线程长期得不到资源 |
| 线程泄漏 | 线程未结束或线程池未关闭 |
