# JVM 学习资料：GC屏障记忆集和低延迟收集器深度解析

[返回索引](../JVM学习资料.md)

## 学习目标

- 理解 GC Roots、可达性分析、三色标记和 STW 的真实作用。
- 能解释写屏障、读屏障、卡表、记忆集为什么存在。
- 能比较 G1、ZGC、Shenandoah 的目标和代价。
- 能从业务目标选择吞吐优先、均衡型或低延迟 GC。
- 能读懂 GC 日志中的关键指标。

## 理论导读

GC 的核心问题不是“怎么删除垃圾”，而是在程序还在运行、对象引用还在变化时，如何判断哪些对象仍然活着，并以可接受的停顿、吞吐和内存成本完成回收。现代 GC 的复杂度，大多来自两个矛盾：一是堆越来越大但停顿不能随堆线性增长，二是应用线程和 GC 线程并发运行时引用关系会变化。

屏障可以理解为 JVM 在读写引用时插入的“记录动作”。写屏障常用于记录跨代引用、维护卡表或保证并发标记正确性；读屏障常用于低延迟 GC 在读取对象引用时处理转发、重映射或颜色状态。它们让 GC 不必每次都扫描整个堆，但代价是每次引用访问路径可能多一点工作。

## 核心心智模型

```text
对象图在变化
  -> GC Roots 提供起点
  -> 标记算法试图找到活对象
  -> 屏障记录应用线程造成的引用变化
  -> 记忆集减少跨区域扫描范围
  -> 收集器用吞吐、停顿、内存占用做取舍
```

## 知识点详解

### 1. 三色标记

三色标记把对象分为：

- 白色：尚未访问，最终可能被回收。
- 灰色：已发现，但子引用还没扫描完。
- 黑色：对象及其子引用已扫描完。

并发标记时，应用线程可能让黑色对象新引用白色对象，导致白色对象被误回收。解决思路：

- 增量更新：记录新增引用。
- SATB：记录被删除前的旧引用，保持“快照开始时活着的对象仍被认为活着”。

> **难点：** 三色标记不是为了面试背颜色，而是为了解释并发标记为什么需要屏障。

### 2. 卡表和记忆集

分代 GC 中，年轻代回收很频繁。如果每次 Minor GC 都扫描整个老年代寻找指向年轻代的引用，成本太高。HotSpot 通过卡表把堆切成小块，当老年代对象写入年轻代引用时，写屏障把对应卡片标脏。Minor GC 只扫描脏卡片即可。

G1 把堆拆成 Region，并为 Region 维护 Remembered Set，用来知道“谁引用了我”。这样回收某些 Region 时，不必全堆扫描。

> **重点：** 记忆集是用额外内存和写屏障开销换更小扫描范围。

### 3. G1 的回收路径

G1 目标是可预测停顿和较好吞吐，常见阶段：

- Young GC：回收年轻 Region。
- Concurrent Mark：并发标记老年代活对象。
- Remark：短暂停顿修正并发期间变化。
- Cleanup：统计 Region 回收价值。
- Mixed GC：选择年轻 Region 和部分收益高的老年代 Region 一起回收。

G1 的关键概念：

- Region：统一大小的堆分区。
- Humongous Object：超过 Region 一半的大对象，处理成本高。
- Pause Target：`-XX:MaxGCPauseMillis` 是目标不是硬保证。

### 4. ZGC 和 Shenandoah

低延迟 GC 的核心目标是让大部分标记、转移和重定位与应用线程并发执行，使停顿主要与 Roots 处理有关，而不是与堆大小强绑定。

| 收集器 | 目标 | 典型特点 | 代价 |
| --- | --- | --- | --- |
| G1 | 均衡吞吐和停顿 | Region、Mixed GC、可预测停顿 | 大堆低延迟不如 ZGC |
| ZGC | 极低停顿 | 并发标记、并发重定位、读屏障 | 吞吐和内存余量成本 |
| Shenandoah | 低停顿 | 并发压缩、转发表/屏障机制 | 实现和平台差异需关注 |
| Parallel | 吞吐优先 | 多线程 STW | 停顿可能较长 |

## 例子

### 开启 GC 日志

```bash
java -Xms2g -Xmx2g -XX:+UseG1GC -Xlog:gc*,safepoint:file=gc.log:time,uptime,level,tags:filecount=5,filesize=50m -jar app.jar
```

### 观察 G1 堆布局

```bash
jcmd <pid> GC.heap_info
jcmd <pid> VM.flags
```

## GC 选择建议

| 场景 | 优先考虑 |
| --- | --- |
| 普通后端服务，延迟和吞吐都要 | G1 |
| 批处理、离线任务、吞吐优先 | Parallel |
| 大堆、低停顿、P99 要求严格 | ZGC 或 Shenandoah |
| 极小工具程序或实验 | Serial |
| 性能测试分配开销 | Epsilon 仅限实验 |

## 练习

1. 用同一个分配压力程序分别运行 G1 和 ZGC，比较 GC 日志。
2. 构造大对象分配，观察 G1 Humongous Object 日志。
3. 用缓存 Map 保留对象，观察老年代增长和 Mixed GC。

## 验收

- 能解释为什么 Minor GC 需要扫描部分老年代引用。
- 能说明 G1 的 Mixed GC 为什么不是简单 Full GC。
- 能根据延迟、吞吐、堆大小、CPU 预算选择 GC。
- 能指出 `MaxGCPauseMillis` 不是绝对 SLA。

## 重点

- 屏障和记忆集是现代 GC 的核心工程手段。
- 低延迟 GC 通过并发化减少停顿，但会消耗吞吐、CPU 和内存余量。

## 难点

- 并发标记和应用线程同时修改对象图时，必须用 SATB、增量更新或其他机制维持正确性。

## 易错

> **易错：** 只看 GC 次数，不看分配速率、停顿分布、老年代趋势和业务延迟。
>
> 正确做法：把 GC 日志、请求 P95/P99、CPU、堆使用曲线和对象分配热点放在一起判断。
