# JVM 学习资料：生产故障案例和排障剧本深度版

[返回索引](../JVM学习资料.md)

## 学习目标

- 能把 JVM 故障从现象转化为证据链。
- 能掌握 CPU 高、内存涨、频繁 GC、死锁、启动慢、容器 OOM 的排查流程。
- 能输出可复盘的故障报告和预防措施。

## 总原则

生产排障的第一步是保留现场。不要在没有 dump、日志、指标和线程栈之前盲目重启。重启可能恢复服务，但会毁掉定位根因的证据。

```text
现象确认 -> 影响范围 -> 保现场 -> 采集证据 -> 假设验证 -> 临时止血 -> 根因修复 -> 复盘预防
```

## 剧本一：CPU 高

### 现象

- Pod 或机器 CPU 使用率接近上限。
- 请求延迟升高，线程池队列堆积。

### 采集

```bash
top -Hp <pid>
jcmd <pid> Thread.print -l > threads.txt
jcmd <pid> JFR.start name=cpu settings=profile duration=120s filename=cpu.jfr
```

把 `top -Hp` 看到的线程 id 转成十六进制，在 thread dump 中查找对应 `nid`。

### 判断

| 线索 | 可能原因 |
| --- | --- |
| 单个业务线程长期 RUNNABLE | 死循环、热方法、正则回溯、JSON 大对象 |
| 多个 GC 线程占用高 | 分配速率高、堆压力、GC 配置不合适 |
| ForkJoinPool 线程高 | 并行流、CompletableFuture、框架异步任务 |
| JIT 编译线程高且启动期明显 | 预热阶段或 Code Cache 压力 |

## 剧本二：内存持续上涨

### 采集

```bash
jcmd <pid> GC.heap_info
jcmd <pid> GC.class_histogram > histo.txt
jcmd <pid> GC.heap_dump /dumps/heap.hprof
jcmd <pid> VM.native_memory summary
```

### 判断

- 堆 used 随 Full GC 后仍上涨：怀疑 Java 对象泄漏。
- 堆稳定但 RSS 上涨：怀疑直接内存、线程栈、Native 库或 mmap。
- 类数量和 Metaspace 上涨：怀疑动态类或 ClassLoader 泄漏。
- 线程数上涨：检查线程池创建、定时任务、连接池。

## 剧本三：频繁 GC 或长停顿

### 采集

```bash
jcmd <pid> VM.flags
jcmd <pid> GC.heap_info
# 查看已经配置的 gc.log
```

### 判断维度

- 分配速率是否过高。
- 老年代是否回收后仍高。
- Humongous Object 是否频繁。
- Mixed GC 是否跟不上晋升。
- 是否存在显式 `System.gc()`。
- 容器 CPU limit 是否导致 GC 线程拿不到足够 CPU。

> **易错：** 看到 Full GC 就直接调大堆。堆变大会降低 GC 频率，但可能增加单次停顿和容器成本。

## 剧本四：死锁和锁竞争

### 采集

```bash
jcmd <pid> Thread.print -l > threads.txt
jcmd <pid> JFR.start name=lock settings=profile duration=120s filename=lock.jfr
```

### 判断

- Java-level deadlock：thread dump 通常直接给出死锁环。
- BLOCKED 线程多：查找共同 monitor。
- WAITING/TIMED_WAITING 多：区分正常等待、连接池耗尽、条件队列等待。

## 剧本五：容器 OOMKilled

### 采集

```bash
kubectl describe pod <pod>
kubectl logs <pod> --previous
```

检查：

- `Reason: OOMKilled`
- exit code 137
- JVM 是否生成 heap dump 或 hs_err
- 容器 memory working set 与 limit
- `-Xmx` 或 `MaxRAMPercentage`

## 练习

1. 本地制造 CPU 空转、死锁、内存泄漏、频繁分配四类故障。
2. 每类故障都采集 thread dump、heap 信息、JFR 或 GC 日志。
3. 按模板写一份完整排障报告。

## 验收

- 能在 10 分钟内完成 JVM 现场证据采集。
- 能区分 Java 堆泄漏和 Native 内存上涨。
- 能从线程栈定位 CPU 热线程。
- 能把排障结论写成可复盘的证据链。

## 重点

- 先保现场，再止血，再根因修复。
- 排障结论必须能被日志、dump、指标或代码证据支持。

## 难点

- JVM 故障经常跨越应用代码、框架、JVM、OS、容器和流量模型，需要多层证据交叉验证。

## 易错

> **易错：** 只拿一份 heap dump 就断定内存泄漏。
>
> 正确做法：结合 Full GC 后趋势、对象保留链、业务缓存策略、容器 RSS 和 Native 内存判断。
