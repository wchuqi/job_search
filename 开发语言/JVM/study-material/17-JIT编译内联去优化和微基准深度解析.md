# JVM 学习资料：JIT编译内联去优化和微基准深度解析

[返回索引](../JVM学习资料.md)

## 学习目标

- 理解解释执行、C1、C2、分层编译和热点探测。
- 能解释方法内联、逃逸分析、锁消除、标量替换和循环优化。
- 能理解去优化、OSR、Code Cache 和类加载对 JIT 的影响。
- 能避免常见微基准错误，使用 JMH 得到可信结果。

## 理论导读

JVM 的性能来自“边运行边观察边优化”。解释器让程序快速启动，JIT 根据运行时 profile 找到热点方法，把它们编译成机器码，并利用真实类型分布、分支概率和调用频率做激进优化。JIT 的优势是拥有运行时信息，劣势是优化假设可能被后来的类加载、反射、动态代理或罕见分支打破。

JIT 不是把所有 Java 代码一次性编译成本地代码。它只优化值得优化的热路径，并且必须保持 Java 语义正确。一旦某个优化假设失效，JVM 会触发去优化，回退到解释执行或重新编译。

## 核心心智模型

```text
解释执行收集 profile
  -> 热点方法达到阈值
  -> C1 快速编译或 C2 深度优化
  -> 内联、逃逸分析、锁消除、循环优化
  -> 假设失效时去优化
```

## 知识点详解

### 1. 分层编译

HotSpot 常见执行层次：

- 解释器：启动快，便于收集 profile。
- C1：编译速度快，适合较早优化。
- C2：优化更激进，适合长期热点。

分层编译让 JVM 在启动速度和峰值性能之间折中。服务刚启动时性能逐渐变好，常见原因就是热点代码被 JIT 编译和内联。

### 2. 方法内联

方法内联把被调用方法体复制到调用点，减少调用开销，并暴露更多优化机会。例如内联后，JIT 可能发现对象不会逃逸，从而消除分配。

内联受影响因素：

- 方法大小。
- 调用频率。
- 虚方法实际类型分布。
- 递归和多态程度。
- JVM 参数和 Code Cache 压力。

### 3. 逃逸分析和标量替换

如果对象不会逃出方法或线程，JIT 可能：

- 消除同步：对象只在本线程使用，锁没有必要。
- 标量替换：对象字段被拆成局部变量，不一定真的分配对象。
- 减少 GC 压力。

> **易错：** “栈上分配”常被简化讲法滥用。HotSpot 更常见的可观察结果是分配被消除或字段被标量替换，而不是一定能在栈上看到对象。

### 4. 去优化

JIT 优化基于假设，例如某个接口调用点长期只有一个实现类。如果后续加载了新实现，原机器码可能不再适用，JVM 需要去优化并重新选择执行方式。

去优化来源：

- 类层次变化。
- 罕见分支被执行。
- 类型 profile 改变。
- 反射、动态代理、invokedynamic 相关调用点变化。

### 5. 微基准陷阱

错误 benchmark 常见问题：

- 没有预热，测到解释执行或编译过程。
- 结果未被使用，被死代码消除。
- 常量折叠让代码根本没执行。
- 数据规模太小，被缓存和分支预测掩盖。
- 在一次 main 方法里手写循环，测到循环优化而不是业务逻辑。

## 例子

### 使用 JMH

```java
@Benchmark
public int sum() {
    int s = 0;
    for (int i = 0; i < data.length; i++) {
        s += data[i];
    }
    return s;
}
```

运行时关注：

```bash
java -jar benchmarks.jar -wi 5 -i 5 -f 2
```

### 查看编译活动

```bash
java -Xlog:jit+compilation=info -jar app.jar
jcmd <pid> Compiler.codecache
```

## 练习

1. 写一个未预热的 main 循环 benchmark，再用 JMH 重写，对比结果稳定性。
2. 写一个局部对象创建实验，尝试观察逃逸分析开启和关闭的吞吐差异。
3. 制造接口单实现和多实现调用，观察内联和性能变化。

## 验收

- 能解释为什么 Java 服务运行一段时间后性能可能提升。
- 能说明 JIT 优化和 Java 语义之间的约束。
- 能说出 JMH 解决了哪些测量问题。
- 能理解去优化不是 bug，而是动态优化的必要成本。

## 重点

- JIT 依赖 profile，不是静态编译器。
- 内联是许多后续优化的入口。

## 难点

- JIT 的优化结果不容易直接从 Java 源码推断，需要结合日志、JFR、JMH 和汇编工具谨慎分析。

## 易错

> **易错：** 用 `System.nanoTime()` 包一段循环就得出“某写法一定更快”。
>
> 正确做法：使用 JMH，设置预热、迭代、fork，并保证结果不会被优化掉。
