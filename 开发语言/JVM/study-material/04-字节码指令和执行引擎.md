# JVM 学习资料：字节码指令和执行引擎

[返回索引](../JVM学习资料.md)

## 学习目标

- 能用 `javap` 阅读常见字节码。
- 理解局部变量表、操作数栈和方法调用指令。
- 建立解释器、JIT、运行时栈帧之间的关系。

## 理论导读

字节码是 JVM 的中间语言。Java、Kotlin、Scala、Groovy 等语言都可以生成 JVM 字节码。执行引擎并不是直接读 Java 源码，而是执行 `.class` 文件中的指令。字节码采用以操作数栈为中心的模型：指令把数据压栈、出栈、计算、存回局部变量表或字段。

读字节码的价值不是为了手写指令，而是理解语法背后的执行成本和语义边界。例如 `i++` 不是原子操作，字符串拼接可能被编译器转成 `StringBuilder` 或 `invokedynamic`，Lambda 通常依赖 `invokedynamic`，异常处理依靠异常表而不是普通跳转。

## 核心心智模型

```text
栈帧
├─ 局部变量表：this、参数、局部变量
├─ 操作数栈：计算中间值
├─ 动态链接：符号引用到运行时目标
└─ 返回地址：方法返回位置
```

## 常见指令

| 类型 | 指令例子 | 含义 |
| --- | --- | --- |
| 加载/存储 | `iload`, `istore`, `aload` | 局部变量和操作数栈之间移动 |
| 常量 | `iconst_0`, `ldc` | 常量入栈 |
| 运算 | `iadd`, `isub`, `imul` | 基本类型运算 |
| 对象 | `new`, `getfield`, `putfield` | 创建对象和访问字段 |
| 调用 | `invokestatic`, `invokevirtual` | 方法调用 |
| 控制流 | `ifeq`, `goto`, `tableswitch` | 条件和跳转 |
| 同步 | `monitorenter`, `monitorexit` | synchronized |
| 动态 | `invokedynamic` | Lambda、字符串拼接等动态链接场景 |

## 方法调用指令

| 指令 | 场景 |
| --- | --- |
| `invokestatic` | 静态方法 |
| `invokespecial` | 构造器、私有方法、父类方法 |
| `invokevirtual` | 普通实例方法，运行时动态分派 |
| `invokeinterface` | 接口方法 |
| `invokedynamic` | 由 bootstrap method 动态决定调用目标 |

## 例子

Java 代码：

```java
public class BytecodeDemo {
    public int add(int a, int b) {
        return a + b;
    }
}
```

查看字节码：

```bash
javac BytecodeDemo.java
javap -c -v BytecodeDemo
```

可能看到：

```text
0: iload_1
1: iload_2
2: iadd
3: ireturn
```

含义：把参数 `a` 和 `b` 入栈，执行整型加法，再返回栈顶结果。

## 练习

- 对比普通方法、静态方法、接口方法、Lambda 的字节码。
- 编译 `try-catch-finally`，观察异常表。
- 编译 `synchronized` 方法和同步代码块，比较字节码差异。

## 验收

- 能解释局部变量表和操作数栈如何配合。
- 能说出五类方法调用指令的主要差异。
- 能用字节码说明 `i++` 为什么不是线程安全的复合操作。

## 重点

- JVM 字节码是栈式指令集。
- 动态分派和 JIT 内联共同影响方法调用性能。
- `synchronized` 代码块对应 `monitorenter` 和 `monitorexit`。

## 难点

- `invokedynamic` 的目标不是编译时固定方法，而是由引导方法建立调用点。
- 字节码层面能解释语义，但最终性能还受 JIT、CPU、对象布局影响。

## 易错

> **易错：** 看到一段 Java 代码就按源码行数估算执行成本。
>
> 正确做法：关键路径要结合字节码、JIT 优化、对象分配和实际 profiling 判断。
