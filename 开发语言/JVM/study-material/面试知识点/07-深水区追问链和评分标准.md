## 一、深水区追问链

这份材料用于把普通 JVM 答案升级成高级工程师回答。每组问题都按“基础问题 -> 机制追问 -> 生产追问 -> 高分判断点”组织。

## 二、GC 追问链

### 1. G1 为什么需要 Remembered Set？

**参考答案：**

G1 以 Region 为单位回收。如果回收某个 Region 时需要知道外部 Region 是否引用它，不能每次扫描全堆，所以为 Region 维护 RSet，记录外部到本 Region 的引用来源。RSet 由写屏障和卡表等机制维护，代价是额外内存和引用写入成本。

**追问：RSet 过大说明什么？**

可能说明跨 Region 引用复杂，例如老对象大量引用年轻对象、大缓存对象图复杂、对象池或全局索引保留过多引用。排查要看 GC phases 中 RSet 扫描成本、对象图和缓存结构。

**高分判断点：**

- 能说出“减少扫描范围”的目的。
- 能说出“写屏障和额外内存”的代价。
- 能联系到缓存对象图和 GC phase。

### 2. 三色标记为什么会漏标？

**参考答案：**

并发标记时，GC 线程和应用线程同时运行。如果黑色对象新增对白色对象引用，同时灰色对象到该白色对象的引用被删除，GC 可能再也扫描不到这个白色对象，从而误回收。解决思路是通过 SATB 或增量更新记录引用变化。

**追问：SATB 和增量更新直觉区别？**

SATB 关注“保留快照开始时活着的对象”，写屏障记录被删除的旧引用；增量更新关注“记录新增引用”，避免黑色对象直接指向未扫描白色对象而无人知道。

**高分判断点：**

- 能描述并发修改对象图导致的问题。
- 能说出两类解决思路的记录对象不同。

## 三、Safepoint 追问链

### 1. STW 一定是 GC 吗？

**参考答案：**

不是。GC 会触发 STW 阶段，但 safepoint 还服务于 thread dump、heap dump、类重定义、JIT 去优化、代码缓存清理等 VM Operation。分析停顿要看 safepoint 日志和 JFR，而不只看 GC 日志。

**追问：GC pause 只有 20ms，接口却卡了 1s，怎么查？**

按时间线对齐业务延迟、GC 日志、safepoint 日志和 JFR。重点看 Time to safepoint、At safepoint、VM Operation、线程是否在 JNI 或长循环、是否有 dump 或诊断操作。

**高分判断点：**

- 不把 STW 简化成 GC。
- 能说出 safepoint 同步时间和操作执行时间。

## 四、类加载追问链

### 1. 为什么 `User cannot be cast to User`？

**参考答案：**

JVM 类型身份由类加载器和全限定类名共同决定。两个不同 ClassLoader 分别加载同名 `User`，类型身份不同，强转会失败。常见于插件系统、Web 容器、fat jar 依赖冲突和公共 API 放错加载器层级。

**追问：怎么修？**

把共享 API 放到父加载器或宿主公共层；插件只加载实现类；避免宿主和插件各自携带一份接口 jar；检查依赖 shading 和容器 lib。

**高分判断点：**

- 能说出类身份公式。
- 能给出插件或 Tomcat 的真实修复方向。

### 2. `NoSuchMethodError` 和 `ClassNotFoundException` 排查路径有什么不同？

**参考答案：**

`ClassNotFoundException` 首先查类是否在当前 ClassLoader 可见路径中；`NoSuchMethodError` 更偏运行期二进制版本冲突，类加载到了，但不是编译期那个版本。排查后者要看依赖树、运行 classpath、类加载日志和 jar 内容。

## 五、JIT 追问链

### 1. 为什么预热后服务变快？

**参考答案：**

解释器先执行并收集 profile，热点方法由 C1/C2 编译，JIT 根据真实类型分布做内联、逃逸分析、锁消除和循环优化。预热后热点路径进入编译代码，性能提升。

**追问：为什么之后又可能变慢？**

可能是 Code Cache 满、新类加载导致去优化、类型 profile 改变、异常路径变热、GC 压力变大或 CPU 限制导致编译线程和应用线程竞争。

**高分判断点：**

- 能说出动态优化和动态失效。
- 能提到 Code Cache、去优化和 profile。

## 六、JMM 追问链

### 1. `volatile` 为什么不能保证 `i++` 安全？

**参考答案：**

`volatile` 保证可见性和有序性，但 `i++` 是读、加、写三个步骤。多个线程可能同时读到相同旧值再写回，导致丢失更新。需要 AtomicInteger、LongAdder、锁或其他同步方案。

**追问：volatile 发布对象为什么有用？**

volatile 写具备 release 语义，读具备 acquire 语义。写 volatile 前的普通写，对后续读到该 volatile 值的线程可见，因此可用于状态标志和安全发布。

**高分判断点：**

- 分开分析原子性、可见性、有序性。
- 能用 happens-before 证明，而不是只背术语。

## 七、生产参数追问链

### 1. 给 2C4G 容器配 JVM 参数，你怎么回答？

**参考答案：**

先问业务目标和画像：QPS、延迟目标、线程数、直接内存、缓存、请求大小、GC 目标。普通服务可先用 G1，堆给 60% 到 70%，保留非堆和系统余量，开启 GC 日志、OOM dump、hs_err。再通过压测校准，而不是直接给固定 `-Xmx`。

**追问：为什么不能 `-Xmx4g`？**

容器限制的是整个进程，不只有堆。还需要线程栈、元空间、直接内存、Code Cache、GC/JIT Native 开销和系统余量。否则可能 Java 堆没 OOM，容器先 OOMKilled。

**高分判断点：**

- 能建立内存预算表。
- 能强调压测验证和证据保留。

## 九、回答禁忌

- 不要把 JVM 规范和 HotSpot 实现混为一谈。
- 不要把所有停顿都归因于 GC。
- 不要把调优说成“调大堆、换 ZGC”。
- 不要只说概念，不给日志、命令、场景和边界。
- 不要把 `volatile` 说成“轻量锁”。
