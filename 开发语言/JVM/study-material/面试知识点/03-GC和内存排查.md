## 一、GC 和内存排查

### 1. Java 如何判断对象可回收？

**参考答案：**

HotSpot 主流使用可达性分析，从 GC Roots 出发遍历对象引用图，无法从 Roots 到达的对象可被回收。GC Roots 包括线程栈中的引用、静态字段引用、常量引用、JNI 引用、活跃 monitor 等。

> **易错：** Java 主流 GC 不是依靠引用计数，因为循环引用会让引用计数方案难以直接判断垃圾。

### 2. 分代假说是什么？

**参考答案：**

分代假说认为大多数对象朝生夕灭，少数对象存活较久。因此堆常分为年轻代和老年代，年轻代频繁回收，老年代较少回收。分代提升了常见应用的回收效率，但跨代引用需要卡表、记忆集和写屏障维护。

### 3. G1 的核心特点是什么？

**参考答案：**

G1 把堆划分为多个 Region，以可预测停顿为目标，通过 Young GC、并发标记、Mixed GC 等阶段优先回收收益高的 Region。它适合大多数后端服务，但 `MaxGCPauseMillis` 只是目标，不是硬保证。

### 4. ZGC 为什么停顿低？

**参考答案：**

ZGC 尽量把标记、转移、重定位等工作并发化，使 STW 阶段主要处理 Roots 等较小工作量。它通过屏障机制保证并发过程中的对象访问正确。代价是更多 CPU、内存余量和屏障开销。

### 5. Full GC 频繁怎么排查？

**参考答案：**

先看 GC 日志确认触发原因和回收效果，再判断分配速率、老年代占用、Humongous Object、元空间、显式 `System.gc()`、容器 CPU 限制等因素。若 Full GC 后老年代仍高，重点查对象保留链和泄漏；若回收效果好但很快又涨，重点查分配速率。

```bash
jcmd <pid> GC.heap_info
jcmd <pid> GC.class_histogram
jcmd <pid> GC.heap_dump /dumps/heap.hprof
```

### 6. 内存泄漏怎么定位？

**参考答案：**

先确认 Full GC 后堆使用是否持续上涨，再抓 heap dump，用 MAT 或 JProfiler 分析 dominator tree 和 GC Roots 保留链。常见来源包括无界缓存、静态集合、ThreadLocal 未清理、监听器未注销、ClassLoader 泄漏。

> **重点：** 泄漏本质是“不再需要的对象仍被可达引用保留”。

### 7. Java OOM 和容器 OOMKilled 如何区分？

**参考答案：**

Java OOM 通常有异常堆栈和 OOM 类型，如 `Java heap space`、`Metaspace`、`Direct buffer memory`。容器 OOMKilled 是进程被宿主机杀掉，Java 进程可能来不及打印异常，Kubernetes 里可看到 `Reason: OOMKilled` 和常见退出码 137。
