## 一、JIT、JMM 和锁

### 1. 为什么 Java 程序运行一段时间后可能变快？

**参考答案：**

HotSpot 先解释执行并收集 profile，热点方法达到阈值后由 C1/C2 JIT 编译为机器码，并进行内联、逃逸分析、锁消除、循环优化等。因此服务预热后性能可能提升。

> **重点：** JIT 是基于运行时热点和 profile 的动态优化。

### 2. 什么是逃逸分析？

**参考答案：**

逃逸分析判断对象是否会逃出方法或线程。如果对象不逃逸，JIT 可能进行标量替换、同步消除或分配消除，从而减少锁和 GC 压力。

> **易错：** 不要绝对说“对象被分配到栈上”，更准确是“分配可能被优化消除或标量替换”。

### 3. 什么是去优化？

**参考答案：**

JIT 优化基于运行时假设，例如某调用点只有一个实现类。当后续类加载或分支行为打破假设，JVM 会废弃已编译代码，回退解释执行或重新编译，这叫去优化。

### 4. JMM 解决什么问题？

**参考答案：**

JMM 定义 Java 多线程下读写共享变量的可见性、有序性和同步规则。它通过 happens-before 关系约束编译器和 CPU 重排序，使正确同步的程序在不同平台上具有一致语义。

### 5. `volatile` 保证什么，不保证什么？

**参考答案：**

`volatile` 保证变量写入对后续读取可见，并通过内存屏障提供有序性约束。但它不保证复合操作原子性，所以 `volatile int count; count++;` 仍然线程不安全。

### 6. `synchronized` 的内存语义是什么？

**参考答案：**

释放同一个 monitor 的锁先行发生于后续获取该锁。也就是说，线程退出 synchronized 块前的写入，对之后进入同一锁保护区域的线程可见。它同时提供互斥和可见性。

### 7. 死锁如何定位？

**参考答案：**

使用 thread dump：

```bash
jcmd <pid> Thread.print -l
```

查看是否有 `Found one Java-level deadlock`，并分析线程持有和等待的 monitor。若是锁竞争但非死锁，需要结合 BLOCKED 线程数量、等待对象、JFR lock profile 和业务代码判断。
