## 一、工具、参数和生产场景

### 1. 常用 JVM 排障工具有哪些？

**参考答案：**

常用 JDK 工具有 `jcmd`、`jstack`、`jmap`、`jstat`、`javap`、JFR、JMC。生产中还会结合 `top`、`pidstat`、`perf`、容器指标、GC 日志、heap dump、thread dump 和 async-profiler。

> **重点：** `jcmd` 是现代 JDK 中很重要的统一诊断入口。

### 2. CPU 高如何排查？

**参考答案：**

先用 OS 工具找高 CPU 线程，再把线程 id 转成十六进制，在 Java thread dump 中定位业务栈。必要时开启 JFR 或 async-profiler 采集 CPU profile。

```bash
top -Hp <pid>
jcmd <pid> Thread.print -l
jcmd <pid> JFR.start name=cpu settings=profile duration=120s filename=cpu.jfr
```

### 3. 线程数过多有什么风险？

**参考答案：**

线程数过多会增加上下文切换、线程栈内存、调度开销和锁竞争风险。在容器中，线程栈会占用进程内存，可能导致 native thread OOM 或 OOMKilled。

### 4. 生产 JVM 参数一般怎么配？

**参考答案：**

至少要明确堆大小或 RAM 百分比、GC、GC 日志、OOM dump、错误日志、编码和必要的可观测性参数。容器内不要只设置 `-Xmx`，还要考虑直接内存、线程栈、元空间、Code Cache 和系统余量。

```bash
-XX:MaxRAMPercentage=70
-XX:+UseG1GC
-Xlog:gc*,safepoint:file=/logs/gc.log:time,uptime,level,tags:filecount=5,filesize=50m
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/dumps
-XX:ErrorFile=/logs/hs_err_pid%p.log
```

### 5. 2C4G 容器如何做 JVM 内存预算？

**参考答案：**

不能直接把 `-Xmx` 设置为 4G。可以先给堆 60% 到 70%，保留 30% 到 40% 给线程栈、元空间、直接内存、Code Cache、GC/JIT 开销和系统余量。如果 Netty、线程数、动态类多或低延迟 GC，需要进一步降低堆比例。最终以压测和生产指标校准。

### 6. heap dump 会不会有风险？

**参考答案：**

有。heap dump 可能很大，生成时可能造成停顿和磁盘压力；里面还可能包含用户数据、token、密码、业务敏感字段。生产应提前配置 dump 路径、磁盘容量、权限和脱敏流程。

### 7. 服务启动慢如何分析？

**参考答案：**

从类加载数量、Spring 或框架初始化、JIT 编译、磁盘 IO、DNS、远程依赖、容器 CPU limit、熵源和配置中心等方面排查。JFR startup profile、类加载日志和应用启动阶段埋点都很有帮助。
