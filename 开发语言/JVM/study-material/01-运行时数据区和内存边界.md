# JVM 学习资料：运行时数据区和内存边界

[返回索引](../JVM学习资料.md)

## 学习目标

- 掌握 JVM 运行时数据区的职责。
- 能区分 Java 堆、栈、元空间、直接内存和进程 RSS。
- 能根据异常文案判断内存问题的大致区域。

## 理论导读

JVM 内存像一个多区域城市。线程私有区域负责“当前正在执行什么”：程序计数器记录位置，虚拟机栈保存每次方法调用的栈帧，本地方法栈服务 Native 调用。线程共享区域负责“程序共同拥有的状态”：堆存放对象实例，元空间存放类元数据，Code Cache 存放 JIT 编译后的机器码。直接内存虽然不属于 Java 堆，但由 Java 程序经 NIO、Unsafe、JNI 等路径使用。

线上内存问题不能只看 `-Xmx`。容器给进程的总内存限制包括堆、非堆、线程栈、直接内存、Code Cache、GC 自身数据结构和 Native 库。如果 `-Xmx` 设得太接近容器上限，GC 还没来得及报 Java OOM，进程可能先被系统 OOM killer 杀掉。

## 核心心智模型

```text
Java 进程内存
├─ Java Heap：对象实例、数组
├─ Metaspace：类元数据、方法元信息
├─ Java Thread Stack：每个线程一个栈
├─ Code Cache：JIT 后的机器码
├─ Direct Memory：ByteBuffer、Netty、NIO
└─ Native Memory：JNI、libc、GC 内部结构、线程本地结构
```

## 知识点详解

| 区域 | 线程私有 | 主要内容 | 常见问题 |
| --- | --- | --- | --- |
| 程序计数器 | 是 | 当前线程字节码执行位置 | 通常不会 OOM |
| Java 虚拟机栈 | 是 | 栈帧、局部变量表、操作数栈 | `StackOverflowError` |
| 本地方法栈 | 是 | Native 方法调用状态 | Native 栈溢出 |
| Java 堆 | 否 | 对象实例、数组 | `Java heap space` |
| 元空间 | 否 | 类元数据 | `Metaspace` OOM |
| 直接内存 | 否 | DirectByteBuffer、Native buffer | `Direct buffer memory` |
| Code Cache | 否 | JIT 编译代码 | 编译受限、性能下降 |

### 栈帧

每次方法调用都会创建栈帧。栈帧包含局部变量表、操作数栈、动态链接和返回地址。递归太深或线程栈太小会导致栈溢出。

### 堆

堆是 GC 管理的主要区域。多数普通 Java 对象在堆上分配，但 JIT 可能通过逃逸分析把部分对象标量替换，使其不必真实分配到堆。

### 元空间

JDK 8 之后，类元数据主要放在本地内存中的 Metaspace。大量动态代理、反射生成类、脚本引擎、热部署 ClassLoader 泄漏都可能造成元空间增长。

### 直接内存

直接内存绕开 Java 堆，适合 IO 场景减少一次数据复制。代价是排查难度更高，并且释放依赖 Cleaner 或框架显式管理。

## 例子

构造栈溢出：

```java
public class StackOverflowDemo {
    static int depth = 0;

    public static void main(String[] args) {
        call();
    }

    static void call() {
        depth++;
        call();
    }
}
```

构造堆 OOM：

```java
import java.util.ArrayList;
import java.util.List;

public class HeapOomDemo {
    public static void main(String[] args) {
        List<byte[]> list = new ArrayList<>();
        while (true) {
            list.add(new byte[1024 * 1024]);
        }
    }
}
```

运行：

```bash
java -Xms64m -Xmx64m HeapOomDemo
```

## 练习

- 分别构造栈溢出、堆 OOM、直接内存 OOM。
- 用 `jcmd <pid> VM.native_memory summary` 观察 Native Memory，前提是启动时打开 NMT。
- 修改 `-Xss`，观察递归深度变化。

## 验收

- 能解释堆 OOM 和系统 OOM killer 的区别。
- 能估算容器中 JVM 内存基线：`堆 + 元空间 + 直接内存 + 线程数 * Xss + Code Cache + 余量`。
- 能根据异常文案选择 heap dump、thread dump、NMT 或 GC 日志。

## 重点

- Java 堆不是 Java 进程的全部内存。
- 每个线程都有栈，线程数过多会显著消耗内存。
- Metaspace 属于 Native Memory，类加载器泄漏会让类无法卸载。

## 难点

- 直接内存和 Native Memory 通常不在普通 heap dump 里完整体现。
- 容器内存限制下，JVM 自适应参数和实际 RSS 之间需要监控验证。

## 易错

> **易错：** 线上看到 `Killed` 就认为是 Java 堆 OOM。
>
> 正确做法：检查容器事件、系统日志、RSS、`-Xmx`、直接内存、线程数和 NMT。

> **易错：** 把局部变量都说成“存在栈上，所以对象也在栈上”。
>
> 正确做法：局部变量表里常保存引用，对象本体通常在堆上。
