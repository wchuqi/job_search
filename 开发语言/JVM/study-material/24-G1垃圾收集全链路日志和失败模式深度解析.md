# JVM 学习资料：G1垃圾收集全链路日志和失败模式深度解析

[返回索引](../JVM学习资料.md)

## 学习目标

- 能解释 G1 Region、RSet、CSet、Young GC、Concurrent Mark、Mixed GC 的完整链路。
- 能读懂 G1 GC 日志中的关键字段。
- 能判断常见失败模式：Humongous、to-space exhausted、evacuation failure、concurrent mode failure。
- 能基于日志提出合理调优方向，而不是背参数。

## 理论导读

G1 的名字是 Garbage First，它的目标不是“最快回收所有垃圾”，而是在停顿预算内优先回收垃圾比例高、收益大的 Region。它把堆拆成等大小 Region，不再把年轻代和老年代理解为物理连续区域，而是逻辑上由一组 Region 组成。G1 的复杂度来自两个核心任务：第一，如何知道每个 Region 的回收收益；第二，如何在不扫描全堆的情况下处理跨 Region 引用。

## 一、核心对象

| 概念 | 含义 |
| --- | --- |
| Region | G1 管理堆的基本单位 |
| RSet | Remembered Set，记录其他 Region 指向本 Region 的引用来源 |
| CSet | Collection Set，本次暂停要回收的 Region 集合 |
| Evacuation | 把存活对象复制到新 Region |
| Humongous | 大对象，超过 Region 一半 |
| IHOP | Initiating Heap Occupancy Percent，触发并发标记的占用阈值 |

## 二、Young GC 链路

```text
Eden Region 分配满
  -> STW
  -> 选择 Eden + Survivor 作为 CSet
  -> 扫描 Roots 和 RSet
  -> 存活对象复制到 Survivor 或 Old
  -> 更新引用
  -> 释放空 Region
```

Young GC 停顿受这些因素影响：

- CSet 中 Region 数量。
- 存活对象大小。
- RSet 扫描成本。
- 引用处理和弱引用数量。
- GC 线程数和容器 CPU limit。

## 三、并发标记和 Mixed GC

G1 老年代回收依赖并发标记：

```text
Initial Mark，一般搭 Young GC
  -> Root Region Scan
  -> Concurrent Mark
  -> Remark
  -> Cleanup
  -> Mixed GC 回收部分老年代 Region
```

Mixed GC 不是一次性清理整个老年代，而是在停顿目标内挑选回收价值高的 Old Region 加入 CSet。G1 会根据 Region 存活率、回收成本和暂停目标做取舍。

> **重点：** Mixed GC 的本质是“收益排序 + 停顿预算内选择”，不是简单老年代 Full GC。

## 四、日志阅读框架

建议参数：

```bash
-Xlog:gc*,gc+heap=debug,gc+ergo=trace,gc+phases=debug,safepoint:file=gc.log:time,uptime,level,tags:filecount=5,filesize=100m
```

阅读顺序：

1. 先看 pause 类型：Young、Mixed、Full、Concurrent Mark。
2. 看回收前后堆容量：是否回收有效。
3. 看 user/sys/real：是否 CPU 不足或系统调用异常。
4. 看 phases：Object Copy、Scan Heap Roots、Update RS、Redirty Cards 哪个高。
5. 看 humongous、to-space、evacuation failure 相关提示。
6. 对齐业务延迟和 safepoint total。

## 五、典型失败模式

### 1. Humongous Object 压力

大对象超过 Region 一半会被当作 Humongous，通常直接进老年代连续 Region。风险：

- 需要连续 Region。
- 回收依赖并发标记结果。
- 大数组、大 JSON、大 byte[]、大缓存值容易触发。

处理方向：

- 限制单请求/单消息大小。
- 流式处理大文件和大 JSON。
- 调整 Region size 只是辅助手段，不应替代对象模型治理。

### 2. to-space exhausted

Evacuation 时需要目标 Region 承接存活对象。如果可用空间不足，会出现 to-space exhausted 或 evacuation failure，可能退化到更重的 Full GC。

常见原因：

- 堆太小或老年代占用过高。
- 存活对象比例过高。
- 并发标记启动太晚。
- 分配速率超过回收速率。

### 3. RSet 成本过高

跨 Region 引用多时，RSet 维护和扫描成本上升。常见于：

- 大量老对象持有年轻对象。
- 大对象图互相引用。
- 缓存结构和对象池设计不合理。

## 六、调优决策

| 现象 | 先查 | 可能方向 |
| --- | --- | --- |
| Young GC 很频繁 | 分配速率、Eden 大小 | 减少临时对象、增加堆、检查批量处理 |
| Mixed GC 后老年代仍高 | 存活率、保留链 | 查泄漏、缓存上限、对象生命周期 |
| Pause 超目标 | phases 明细、CPU limit | 降 CSet、增 CPU、减存活对象 |
| Humongous 多 | 对象大小分布 | 流式处理、拆分大对象、检查 Region |
| Full GC | 触发原因 | 查 to-space、元空间、显式 GC、并发失败 |

## 练习

1. 写一个分配大量短生命周期对象的程序，观察 Young GC。
2. 写一个保留大量 byte[] 的程序，观察 Humongous。
3. 限制容器 CPU，观察 GC real time 与 user time 差异。

## 验收

- 能解释 G1 为什么需要 RSet。
- 能根据日志判断 Young、Mixed、Full 的区别。
- 能说出 G1 三类常见失败模式及处理方向。

## 重点

- G1 调优首先看对象生命周期和日志证据。
- `MaxGCPauseMillis` 是目标，不是 JVM 对业务 SLA 的承诺。

## 难点

- G1 停顿由 CSet 选择、存活对象、RSet、引用处理、CPU 共同决定，不是一个参数能解决。

## 易错

> **易错：** Full GC 频繁就直接把 `-Xmx` 翻倍。
>
> 正确做法：先看回收效果和触发原因。泄漏导致的老年代高，单纯加堆只是延迟故障。
