# JVM 学习资料：常见垃圾收集器和选择策略

[返回索引](../JVM学习资料.md)

## 学习目标

- 掌握常见 HotSpot GC 的特点。
- 能根据吞吐、延迟、堆大小和业务场景选择 GC。
- 理解 G1、ZGC、Shenandoah 的基本机制和边界。

## 理论导读

GC 没有“绝对最好”，只有目标不同。批处理更关心吞吐，允许较长停顿；交互式服务更关心 P99 延迟；大堆服务关心可预测停顿；资源紧张服务还要关注 CPU 和内存开销。选 GC 前要先问：堆多大、对象生命周期如何、延迟目标是多少、CPU 是否富余、容器内存是否严格。

## 收集器对比

| GC | 目标 | 典型场景 | 注意点 |
| --- | --- | --- | --- |
| Serial | 简单、低开销 | 小堆、客户端、测试 | 单线程 STW |
| Parallel | 吞吐优先 | 批处理、计算任务 | 停顿可能较长 |
| G1 | 平衡吞吐和停顿 | 默认后端服务常用 | 需要读懂 Region 和 mixed GC |
| ZGC | 超低延迟 | 大堆、低停顿服务 | CPU/内存开销和版本特性需验证 |
| Shenandoah | 超低延迟 | 低停顿服务 | 可用性取决于发行版 |
| Epsilon | 不回收 | 性能实验、短进程 | 会用完堆 |

## G1 核心机制

G1 把堆切成多个 Region，不要求年轻代和老年代物理连续。它根据回收收益选择 Region，目标是在可控停顿内回收更多垃圾。

关键概念：

- Region。
- Remembered Set。
- Young GC。
- Concurrent Marking。
- Mixed GC。
- Humongous Object。
- Pause target。

## ZGC 核心机制

ZGC 目标是把停顿压得很低。它使用着色指针、读屏障、并发标记和并发重定位，让大部分工作和应用线程并发执行。

适合：

- 大堆。
- 对 P99/P999 延迟敏感。
- 可以接受一定 CPU 开销。

## 选择策略

| 业务目标 | 初始选择 |
| --- | --- |
| 普通 Java 后端 | G1 |
| 批处理吞吐优先 | Parallel |
| 小工具或小内存 | Serial 或 G1 |
| 大堆低延迟 | ZGC |
| 严格延迟但发行版支持 Shenandoah | Shenandoah |

## 参数例子

G1：

```bash
java -XX:+UseG1GC -Xms2g -Xmx2g -Xlog:gc*:file=gc.log:time,uptime,level,tags App
```

ZGC：

```bash
java -XX:+UseZGC -Xms8g -Xmx8g -Xlog:gc*:file=gc.log:time,uptime,level,tags App
```

## 练习

- 同一个分配压力程序分别用 G1、Parallel、ZGC 运行，比较吞吐和停顿。
- 设置不同 `-XX:MaxGCPauseMillis`，观察 G1 是否一定能满足目标。
- 制造大对象分配，观察 G1 Humongous Object 日志。

## 验收

- 能解释 G1 为什么适合通用服务。
- 能说明低延迟 GC 降低停顿的代价。
- 能根据业务目标选择初始 GC，并说明需要验证的指标。

## 重点

- 默认选择不等于无需观测。
- `MaxGCPauseMillis` 是目标，不是保证。
- 大堆、低延迟、高吞吐很难同时极致满足。

## 难点

- G1 的 mixed GC、RSet 维护和 humongous object 会影响实际表现。
- ZGC/Shenandoah 的并发阶段仍可能受 CPU、内存带宽和分配速率影响。

## 易错

> **易错：** 看到 ZGC 低延迟就给所有服务启用。
>
> 正确做法：结合堆大小、CPU、延迟目标、吞吐损失和团队排障能力验证。
