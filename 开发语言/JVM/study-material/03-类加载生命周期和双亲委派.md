# JVM 学习资料：类加载生命周期和双亲委派

[返回索引](../JVM学习资料.md)

## 学习目标

- 掌握类加载、链接、初始化的完整生命周期。
- 理解双亲委派、类加载器命名空间和类冲突。
- 能排查常见类加载异常。

## 理论导读

类加载器是 JVM 的入口检查系统。它不只是把 `.class` 文件读进内存，还决定了类的身份边界。JVM 判断两个类是否相同，不只看全限定类名，还要看是否由同一个类加载器加载。因此插件系统、应用服务器、IDE 热加载、Spring Boot fat jar 都会认真设计类加载策略。

双亲委派像逐级上报：应用类加载器收到请求后先委托父加载器，父加载器无法完成时才由子加载器加载。这样能保证核心类库的唯一性和安全性。但 SPI、JDBC 驱动、应用服务器隔离、OSGi、Tomcat WebAppClassLoader、某些插件系统会调整或打破默认模型。

## 核心心智模型

```text
加载 Loading
  -> 验证 Verification
  -> 准备 Preparation
  -> 解析 Resolution
  -> 初始化 Initialization
  -> 使用 Using
  -> 卸载 Unloading
```

## 知识点详解

### 加载

- 通过类名获取二进制字节流。
- 转换为方法区运行时数据结构。
- 在堆中生成 `java.lang.Class` 对象作为访问入口。

### 验证

验证字节码格式、元数据、控制流、类型安全，防止非法字节码破坏 JVM。

### 准备

为类变量分配内存并设置零值。注意这里不是执行 Java 代码中的赋值逻辑。

```java
static int count = 10;
```

准备阶段 `count` 是 `0`，初始化阶段才变为 `10`。

### 解析

把常量池中的符号引用转换为直接引用。解析可以在初始化前，也可以延迟到首次使用。

### 初始化

执行类构造器 `<clinit>`，包括静态字段赋值和静态代码块。初始化有严格触发条件，例如 `new`、访问静态字段、调用静态方法、反射、初始化子类前先初始化父类。

## 常见异常对比

| 异常 | 典型含义 | 排查方向 |
| --- | --- | --- |
| `ClassNotFoundException` | 显式加载类失败 | classpath、模块、依赖缺失 |
| `NoClassDefFoundError` | 编译时存在，运行时不可用或初始化失败 | 运行依赖、初始化异常 |
| `NoSuchMethodError` | 运行时版本没有该方法 | 依赖冲突、jar 版本不一致 |
| `ClassCastException` | 类型转换失败 | 类加载器隔离、类型错误 |
| `LinkageError` | 链接阶段冲突 | 二进制不兼容、重复类 |

## 例子

查看类加载：

```bash
java -Xlog:class+load=info -cp app.jar com.example.Main
```

类初始化触发：

```java
class Parent {
    static {
        System.out.println("Parent init");
    }
}

class Child extends Parent {
    static final int COMPILE_CONSTANT = 1;
    static int runtimeValue = 2;
    static {
        System.out.println("Child init");
    }
}

public class InitDemo {
    public static void main(String[] args) {
        System.out.println(Child.COMPILE_CONSTANT);
        System.out.println(Child.runtimeValue);
    }
}
```

## 练习

- 写一个自定义 ClassLoader，从指定目录读取 `.class`。
- 用两个不同 ClassLoader 加载同一个类，尝试强转并观察异常。
- 制造依赖版本冲突，观察 `NoSuchMethodError`。

## 验收

- 能说清加载、验证、准备、解析、初始化的区别。
- 能解释为什么“类名相同”不等于“同一个类”。
- 能根据类加载异常判断是缺依赖、版本冲突还是加载器隔离问题。

## 重点

- 类身份 = 类加载器 + 全限定类名。
- 双亲委派保护核心类库一致性。
- 初始化阶段才执行静态代码块和静态字段真实赋值。

## 难点

- SPI 经常需要父加载器反向使用子加载器可见的实现类。
- Spring Boot、Tomcat、OSGi、插件系统的类加载策略不同，不能套一个模型。

## 易错

> **易错：** 认为双亲委派永远不能打破。
>
> 正确做法：默认模型是委派，但框架为了隔离、插件化和 SPI 可能主动调整。
