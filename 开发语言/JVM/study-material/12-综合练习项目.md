# JVM 学习资料：综合练习项目

[返回索引](../JVM学习资料.md)

## 项目目标

构建一个 `jvm-lab` 实验项目，能够主动制造和诊断 JVM 常见问题：CPU 高、死锁、堆泄漏、直接内存泄漏、频繁 GC、类加载异常、JIT 微基准陷阱。

## 项目结构

```text
jvm-lab/
  src/main/java/lab/
    CpuHotLoop.java
    DeadlockDemo.java
    HeapLeakDemo.java
    DirectMemoryDemo.java
    GcPressureDemo.java
    ClassLoadingDemo.java
    JitBenchmarkTrap.java
  scripts/
    run-gc-log.sh
    capture-thread-dump.sh
    capture-jfr.sh
  docs/
    report-template.md
```

## 实验任务

| 任务 | 目标 | 验收 |
| --- | --- | --- |
| CPU 高 | 定位热点线程和调用栈 | 能把 OS 线程 ID 对应到 Java 栈 |
| 死锁 | 用 thread dump 找锁等待环 | 报告中包含死锁线程和锁对象 |
| 堆泄漏 | heap dump 找保留链 | 能指出 GC Root 和 retained size |
| 直接内存 | 用 NMT 观察堆外增长 | 能区分 heap 和 direct memory |
| 频繁 GC | 读 GC 日志 | 能判断分配速率和回收效果 |
| 类加载 | 制造版本冲突 | 能解释异常类型 |
| JIT | 修正错误 benchmark | 能说明预热和死代码消除 |


## 推荐命令

```bash
jps -lv
jcmd <pid> VM.flags
jcmd <pid> Thread.print > thread.txt
jcmd <pid> GC.class_histogram > histo.txt
jcmd <pid> GC.heap_dump heap.hprof
jcmd <pid> JFR.start name=lab settings=profile duration=60s filename=lab.jfr
```

## 重点

- 每个实验必须保留启动参数和证据。
- 结论必须能从日志、dump 或 profile 推导出来。
- 不要只写“可能是内存泄漏”，要指出保留链或增长区域。

## 难点

- 同一现象可能有多个原因，例如“卡顿”可能来自 GC、锁、IO、CPU、线程池或下游。
- 实验环境结果不能直接等同生产环境，但排查方法可以迁移。

## 易错

> **易错：** 实验只跑出异常就结束。
>
> 正确做法：必须完成定位、证据、修复和预防四步。
