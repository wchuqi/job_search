# JVM 学习资料：类加载解析初始化和命名空间深度解析

[返回索引](../JVM学习资料.md)

## 学习目标

- 能精确区分类加载的加载、链接、初始化三个大阶段。
- 能解释准备阶段、解析阶段和初始化阶段的边界。
- 能判断一个类什么时候会被主动使用并触发初始化。
- 能解释双亲委派、线程上下文类加载器、类加载器命名空间和类冲突。
- 能排查 `ClassNotFoundException`、`NoClassDefFoundError`、`NoSuchMethodError`、`ClassCastException` 等问题。

## 理论导读

类加载不是“找到 class 文件然后 new 对象”这么简单。JVM 必须先把二进制字节流变成运行时结构，验证它不会破坏虚拟机安全，再把符号引用逐步解析成可直接使用的引用，最后在规定时机执行类初始化逻辑。这个过程既影响启动速度，也影响框架插件隔离、热部署、依赖冲突和线上兼容性。

类加载器像一组带边界的仓库管理员。同一个全限定类名，如果由不同类加载器加载，在 JVM 看来就是不同类型。Spring Boot fat jar、Tomcat WebAppClassLoader、SPI、JDBC Driver、OSGi、插件系统都会利用或调整这个边界。很多“类明明存在却报错”的问题，本质不是文件不存在，而是加载器视角、版本、初始化失败或二进制兼容性出了问题。

## 核心心智模型

```text
.class 字节流
  -> 加载：找到字节流，生成 Class 对象
  -> 链接
       -> 验证：格式、字节码、安全约束
       -> 准备：给静态变量分配内存并设置零值
       -> 解析：符号引用转直接引用，可延迟
  -> 初始化：执行 <clinit>，给静态变量赋显式值，执行 static 块
  -> 使用
  -> 卸载：类加载器可回收且无活跃引用
```

> **重点：** 准备阶段不是执行 Java 代码，初始化阶段才执行 `<clinit>`。

## 知识点详解

### 1. 准备阶段和初始化阶段

```java
class Config {
    static int port = 8080;
    static final int DEFAULT_PORT = 80;
    static final Integer BOXED_PORT = 443;
}
```

- `port` 在准备阶段先是 `0`，初始化阶段变成 `8080`。
- `DEFAULT_PORT` 是编译期常量，可能直接进入调用方常量池，访问它不一定触发 `Config` 初始化。
- `BOXED_PORT` 不是基本类型或 String 的编译期常量，通常需要初始化阶段创建对象。

> **易错：** 看到 `static final` 就认为一定不触发类初始化。只有编译期常量才可能被调用方内联。

### 2. 主动使用触发初始化

常见主动使用：

- `new` 一个类。
- 读写非编译期常量的静态字段。
- 调用静态方法。
- 反射调用并要求初始化。
- 初始化子类时，父类先初始化。
- JVM 启动时初始化主类。

不会直接触发初始化的典型情况：

- 通过子类引用父类静态字段，只初始化父类。
- 创建数组 `new MyClass[10]` 不初始化 `MyClass`。
- 访问编译期常量不一定初始化定义类。

### 3. 双亲委派和破坏委派

双亲委派的默认过程：

```text
loadClass(name)
  -> 先查 findLoadedClass
  -> 委托 parent.loadClass
  -> 父加载器失败后调用 findClass
```

它解决两个问题：

- 核心类优先由 Bootstrap 加载，避免应用伪造 `java.lang.String`。
- 同一层级内核心类型保持一致，减少类型冲突。

破坏或调整委派的场景：

- JDBC、JNDI、SPI 使用线程上下文类加载器，让父层代码能发现子层实现。
- Tomcat 每个 Web 应用有独立类加载器，隔离不同应用依赖。
- 插件系统常使用 child-first 或自定义查找策略，保证插件依赖独立。

> **难点：** 双亲委派是 HotSpot 常见加载器体系的实践模型，不等于所有框架都必须完全遵守。

### 4. 类身份和命名空间

类身份由两部分决定：

```text
类加载器实例 + 全限定类名
```

如果两个 ClassLoader 分别加载了 `com.example.User`，即使字节码完全一样，也不是同一个类型。强转可能报：

```text
java.lang.ClassCastException: com.example.User cannot be cast to com.example.User
```

这种错误常见于：

- 插件接口类被插件和宿主各加载一份。
- Web 容器公共 lib 和应用 lib 存在相同 jar。
- fat jar 或 shading 处理不当。

## 例子

### 查看类加载日志

```bash
java -Xlog:class+load=info -cp target/classes com.example.App
```

### 观察初始化顺序

```java
class Parent {
    static int x = log("parent x");
    static int log(String s) {
        System.out.println(s);
        return 1;
    }
}

class Child extends Parent {
    static int y = log("child y");
}

public class InitOrder {
    public static void main(String[] args) {
        System.out.println(Child.y);
    }
}
```

输出先父后子，因为子类初始化前必须先初始化父类。

## 排障表

| 错误 | 常见含义 | 第一排查入口 |
| --- | --- | --- |
| `ClassNotFoundException` | 运行时按名称找类失败 | classpath、module path、ClassLoader 视角 |
| `NoClassDefFoundError` | 编译期存在，运行期缺失或初始化曾失败 | 启动依赖、初始化异常历史 |
| `NoSuchMethodError` | 运行期实际加载的类版本没有该方法 | 依赖冲突、jar 版本、二进制兼容 |
| `LinkageError` | 链接阶段发现不兼容 | 重复类、版本冲突、类加载边界 |
| `ClassCastException: A cannot be cast to A` | 同名类被不同加载器加载 | 插件/Web 容器/公共 lib 边界 |

## 练习

1. 写一个父类、子类和编译期常量实验，记录哪些访问会触发初始化。
2. 用两个自定义 ClassLoader 加载同一个 `.class`，验证类型不相等。
3. 制造两个版本 jar，一个有 `foo()` 一个没有，运行时替换旧 jar，观察 `NoSuchMethodError`。

## 验收

- 能画出类加载生命周期。
- 能说明准备阶段和初始化阶段的差异。
- 能解释为什么同名类可能不是同一类型。
- 能给出 `NoSuchMethodError` 的依赖冲突排查路径。

## 重点

- 类加载顺序、初始化触发条件、类加载器命名空间是 JVM 面试和生产排障高频点。
- `类加载器 + 类名` 才是 JVM 中完整的类型身份。

## 难点

- 解析可以延迟，初始化触发也有严格条件，不能把所有动作都理解成启动时一次性完成。
- 框架中的类加载策略经常为隔离和扩展性牺牲简单性。

## 易错

> **易错：** 把 `ClassNotFoundException` 和 `NoClassDefFoundError` 都说成“类不存在”。
>
> 正确做法：前者偏显式查找失败，后者常表示编译时存在但运行时缺失、链接失败或初始化失败后的再次使用。
