# JVM 学习资料：GC 基础、可达性和分代模型

[返回索引](../JVM学习资料.md)

## 学习目标

- 理解 GC 解决的问题和基本术语。
- 掌握可达性分析、GC Roots、分代假说。
- 能区分内存泄漏、内存溢出和正常内存波动。

## 理论导读

GC 的目标不是让程序永远不占内存，而是自动回收不再可达的对象。HotSpot 主流 GC 基于可达性分析：从 GC Roots 出发，沿引用关系遍历，能走到的对象视为存活，走不到的对象才可回收。内存泄漏的核心通常是对象仍然可达，但业务上已经无用。

分代模型来自经验假说：大多数对象朝生夕死，少数对象存活很久。因此 JVM 可以把新对象放在年轻代，用较低成本频繁回收短命对象；活得足够久的对象晋升到老年代。现代 G1 虽然基于 Region 管理，不再是传统连续年轻代和老年代，但仍保留分代思想。

## 核心概念

| 概念 | 说明 |
| --- | --- |
| GC Roots | 可达性分析的起点 |
| STW | Stop-The-World，暂停应用线程 |
| Young GC | 主要回收年轻代 |
| Old GC | 回收老年代 |
| Full GC | 通常涉及整个堆和其他区域，代价较高 |
| 分配速率 | 单位时间创建对象的速度 |
| 晋升 | 对象从年轻代进入老年代 |
| 浮动垃圾 | 并发标记期间产生、本轮可能无法回收的垃圾 |

## GC Roots

常见 GC Roots：

- 虚拟机栈中的引用。
- 静态字段引用。
- 常量引用。
- JNI 引用。
- 活跃线程对象。
- 被同步锁持有的对象。

## 基本算法

| 算法 | 特点 | 代价 |
| --- | --- | --- |
| 标记-清除 | 标记存活，清除垃圾 | 产生碎片 |
| 标记-复制 | 存活对象复制到新区域 | 浪费部分空间 |
| 标记-整理 | 存活对象向一端移动 | 移动成本高 |
| 分代收集 | 不同代用不同策略 | 跨代引用需要额外记录 |

## 跨代引用和写屏障

年轻代 GC 不能每次扫描整个老年代，因此需要记忆集或卡表记录“老年代哪些区域引用了年轻代”。写屏障会在引用写入时维护这些结构。屏障不是业务代码中的屏障，而是 JVM 在内存写操作周围插入的辅助逻辑。

## 例子

短命对象：

```java
public class AllocationDemo {
    public static void main(String[] args) {
        for (int i = 0; i < 1_000_000; i++) {
            byte[] data = new byte[1024];
            consume(data);
        }
    }

    static void consume(byte[] data) {
        // 模拟短生命周期对象
    }
}
```

观察 GC：

```bash
java -Xms128m -Xmx128m -Xlog:gc* AllocationDemo
```

## 练习

- 写一个不断创建短命对象的程序，观察 Young GC。
- 写一个静态集合保存对象的程序，观察老年代增长。
- 用 heap dump 查看对象的 GC Roots 保留链。

## 验收

- 能解释为什么引用计数不是 HotSpot Java 堆主流方案。
- 能列出常见 GC Roots。
- 能说明为什么需要写屏障和记忆集。

## 重点

- 可达不等于业务上有用。
- 分代优化来自对象生命周期分布。
- STW 时间和堆大小、存活对象数量、GC 算法、系统负载都有关。

## 难点

- 并发 GC 需要处理应用线程同时修改引用关系的问题。
- 写屏障、读屏障、三色标记、SATB、增量更新是理解现代 GC 的关键。

## 易错

> **易错：** 认为 GC 后内存没有降到很低就是泄漏。
>
> 正确做法：看老年代趋势、存活对象、保留链、GC 后基线，而不是单次内存曲线。
