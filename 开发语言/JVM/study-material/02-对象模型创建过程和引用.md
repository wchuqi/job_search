# JVM 学习资料：对象模型创建过程和引用

[返回索引](../JVM学习资料.md)

## 学习目标

- 理解对象从 `new` 到可用的过程。
- 掌握对象头、引用类型、对象布局和分配路径。
- 明白对象可达性和内存泄漏的关系。

## 理论导读

对象不是一行 `new` 语句凭空出现的。JVM 先检查类是否已加载，再在堆上分配空间，把对象内存初始化为零值，设置对象头，然后执行构造器。对象头保存运行时元信息，例如哈希码、锁状态、GC 年龄、类元数据指针。数组还会保存数组长度。

引用像门牌号，不是对象本身。多个引用可以指向同一个对象；引用断开不代表对象立即回收；只要对象能从 GC Roots 走到，它就仍然活着。很多内存泄漏本质上不是对象无法回收，而是业务已经不需要对象，但仍有集合、线程局部变量、静态字段或回调链保留引用。

## 核心心智模型

```text
new Object()
  -> 类加载检查
  -> 分配内存：TLAB / CAS
  -> 零值初始化
  -> 设置对象头
  -> 执行 <init>
  -> 引用指向对象
```

## 知识点详解

### 对象分配路径

- TLAB：线程本地分配缓冲，减少多线程分配竞争。
- 指针碰撞：堆空间规整时，只移动分配指针。
- 空闲列表：堆空间不规整时，从可用块列表中选择空间。
- CAS：多个线程竞争分配时保证原子性。

### 对象头

| 组成 | 说明 |
| --- | --- |
| Mark Word | 哈希码、锁状态、GC 年龄等 |
| Klass Pointer | 指向类元数据 |
| Array Length | 数组对象额外保存长度 |

### 引用类型

| 引用 | 回收语义 | 典型场景 |
| --- | --- | --- |
| 强引用 | 只要可达就不回收 | 普通对象引用 |
| 软引用 | 内存不足前可回收 | 缓存，但现代业务慎用 |
| 弱引用 | 下一次 GC 可回收 | `WeakHashMap`、ThreadLocal key |
| 虚引用 | 不影响生命周期 | Cleaner、回收通知 |

### 对象布局

对象大小受对象头、字段类型、字段重排、对齐填充、压缩指针影响。不能只按字段大小相加估算对象内存。

## 例子

ThreadLocal 泄漏的简化模型：

```java
public class ThreadLocalLeakDemo {
    private static final ThreadLocal<byte[]> LOCAL = new ThreadLocal<>();

    public static void handleRequest() {
        LOCAL.set(new byte[10 * 1024 * 1024]);
        // 线程池线程不会结束，如果不 remove，value 可能长时间保留。
        LOCAL.remove();
    }
}
```

## 练习

- 用一个静态 `List` 持续保存大对象，生成 heap dump 后查找 GC Roots 保留链。
- 对比普通对象、数组对象、含 `long` 字段对象的大致大小。
- 写一个 `WeakHashMap` 示例，观察 key 失去强引用后的回收行为。

## 验收

- 能说明对象创建每一步的目的。
- 能解释强、软、弱、虚引用的回收语义。
- 能解释为什么 ThreadLocal 使用后通常要 `remove()`。

## 重点

- 内存泄漏常见根因是引用链没有断，而不是 GC 不工作。
- 对象头和对齐会影响对象实际大小。
- `finalize()` 已不适合作为资源释放机制，应使用显式关闭、try-with-resources 或 Cleaner。

## 难点

- 逃逸分析可能让源代码中的对象不真实分配到堆。
- heap dump 里要看 retained size 和 GC Roots，不只看 shallow size。

## 易错

> **易错：** 软引用适合所有缓存。
>
> 正确做法：业务缓存应有容量、过期、淘汰和监控策略，软引用回收时机不适合作为主要缓存控制。

> **易错：** `System.gc()` 能可靠释放内存。
>
> 正确做法：它只是建议 JVM 执行 GC，生产中通常不依赖它，甚至会禁用显式 GC。
