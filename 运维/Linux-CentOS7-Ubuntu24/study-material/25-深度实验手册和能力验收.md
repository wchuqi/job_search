# Linux 学习资料：深度实验手册和能力验收

[返回索引](../Linux学习资料.md)

## 学习目标

- 通过可观察实验理解 Linux 内核、启动、I/O、网络、内存、安全和包管理机制。
- 建立从现象到证据再到结论的实验记录习惯。
- 验证是否具备 CentOS 7 存量维护和 Ubuntu 24.04 新环境交付能力。

## 实验原则

- 所有破坏性实验必须在虚拟机或测试环境执行。
- 每个实验记录：系统版本、命令、关键输出、判断、风险、回滚。
- 不在生产执行压测、删文件、改 GRUB、改 fstab、改 SSH、改 PAM 实验。

## 实验 1：系统调用观察

目标：理解用户态程序如何进入内核。

```bash
strace -f -o /tmp/cat.strace cat /etc/hostname
grep -E 'openat|read|write|close' /tmp/cat.strace | head -30
```

验收：

- 能解释 `openat`、`read`、`write`。
- 能说明 errno 如何用于排障。

## 实验 2：procfs 进程观察

```bash
sleep 300 &
pid=$!
cat /proc/$pid/status
ls -l /proc/$pid/fd
cat /proc/$pid/limits
cat /proc/$pid/cgroup
kill $pid
```

验收：

- 能解释进程状态、fd、limits、cgroup。

## 实验 3：启动参数和 initramfs

```bash
cat /proc/cmdline
test -d /sys/firmware/efi && echo UEFI || echo BIOS
```

CentOS 7：

```bash
lsinitrd /boot/initramfs-$(uname -r).img | head
```

Ubuntu 24.04：

```bash
lsinitramfs /boot/initrd.img-$(uname -r) | head
```

验收：

- 能解释 initramfs 的作用。
- 能说明 CentOS 7 与 Ubuntu 24.04 生成工具差异。

## 实验 4：Page Cache

```bash
dd if=/dev/zero of=/tmp/cache-lab bs=1M count=512 status=progress
sync
time cat /tmp/cache-lab >/dev/null
time cat /tmp/cache-lab >/dev/null
grep -E 'Cached|Dirty|Writeback' /proc/meminfo
rm -f /tmp/cache-lab
```

验收：

- 第二次读取通常更快。
- 能解释 page cache 与 free 低的关系。

## 实验 5：删除打开文件

```bash
python3 - <<'PY'
import time
f=open('/tmp/open-deleted.log','w')
f.write('x'*100*1024*1024)
f.flush()
print('pid ready')
time.sleep(300)
PY
```

另一个终端：

```bash
rm /tmp/open-deleted.log
sudo lsof +L1 | grep open-deleted
```

验收：

- 能解释为什么空间未释放。
- 能说明如何安全释放。

## 实验 6：路由决策

```bash
ip route
ip route get 8.8.8.8
ip route get 1.1.1.1
```

验收：

- 能解释出口网卡、下一跳、源地址。

## 实验 7：监听地址

```bash
python3 -m http.server 8080 --bind 127.0.0.1
ss -lntp | grep 8080
```

换成：

```bash
python3 -m http.server 8080 --bind 0.0.0.0
```

验收：

- 能解释外部访问差异。

## 实验 8：conntrack 观察

```bash
cat /proc/sys/net/netfilter/nf_conntrack_count 2>/dev/null || true
cat /proc/sys/net/netfilter/nf_conntrack_max 2>/dev/null || true
```

如果有工具：

```bash
sudo conntrack -S 2>/dev/null || true
```

验收：

- 能解释 conntrack 表满的现象和风险。

## 实验 9：cgroup 版本和 systemd 限制

```bash
stat -fc %T /sys/fs/cgroup
cat /proc/self/cgroup
systemctl show user.slice -p MemoryCurrent -p TasksCurrent 2>/dev/null || true
```

验收：

- 能判断 cgroup v1/v2。
- 能说明 CentOS 7 与 Ubuntu 24.04 常见差异。

## 实验 10：OOM 证据识别

不建议强行制造系统 OOM。用日志查询方式：

```bash
journalctl -k | grep -i -E 'oom|out of memory|killed process' | tail -20
dmesg -T | grep -i -E 'oom|out of memory|killed process' | tail -20
```

验收：

- 能识别系统 OOM 与 cgroup OOM 关键字。

## 实验 11：SELinux/AppArmor 判定

CentOS 7：

```bash
getenforce
ps -eZ | head
ls -Z /etc/passwd
sudo ausearch -m avc -ts recent
```

Ubuntu 24.04：

```bash
sudo aa-status
sudo journalctl -k | grep -i apparmor | tail -20
```

验收：

- 能说明当前系统使用哪种 MAC。
- 能找到拒绝日志入口。

## 实验 12：包文件验证

CentOS 7：

```bash
rpm -V bash
rpm -q --scripts bash | head -80
```

Ubuntu 24.04：

```bash
dpkg -V bash
ls /var/lib/dpkg/info/bash.*
```

验收：

- 能说明包数据库如何帮助发现文件被改。

## 实验 13：systemd 依赖图

```bash
systemd-analyze
systemd-analyze critical-chain
systemctl list-dependencies multi-user.target | head -80
```

验收：

- 能解释启动耗时和依赖链。

## 实验 14：网络防火墙证据链

在测试机放行/拒绝 8080 端口，同时抓包：

```bash
sudo tcpdump -ni any port 8080
```

验收：

- 能区分包没到、到达但无回包、应用无响应。

## 实验 15：迁移评估报告

在 CentOS 7 上采集：

```bash
rpm -qa | sort > packages.txt
systemctl list-unit-files > units.txt
ss -lntup > ports.txt
firewall-cmd --list-all > firewall.txt 2>/dev/null || true
getenforce > selinux.txt 2>/dev/null || true
```

输出 Ubuntu 24.04 迁移检查表：

- 包名映射。
- 服务名和 unit。
- 网络配置。
- 防火墙。
- 日志路径。
- 安全模块。
- Python 和脚本。
- cgroup 和容器。

## 最终能力验收

你需要能独立完成：

1. 解释一条命令从 Shell 到系统调用再到内核对象的路径。
2. 排查服务启动失败，并区分配置、权限、端口、安全模块和依赖问题。
3. 排查端口不通，并用 tcpdump 证明包停在哪一层。
4. 排查磁盘满但 du 对不上的问题。
5. 排查内存不足，并区分 page cache、系统 OOM 和 cgroup OOM。
6. 排查 SELinux/AppArmor 拒绝。
7. 审查 sudoers 是否存在命令逃逸。
8. 验证包文件完整性和仓库信任。
9. 制定 CentOS 7 到 Ubuntu 24.04 迁移方案。

## 重点

- 实验不是跑命令，而是建立“现象 -> 证据 -> 机制 -> 结论”。
- 深度能力体现在解释边界条件和生产风险。
- CentOS 7 与 Ubuntu 24.04 的差异要落实到命令、路径、服务和策略。

## 易错

> **易错：** 把实验输出当成所有系统的固定答案。
>
> 正确做法：不同内核、发行版、systemd、容器和云环境输出会不同。重点是解释机制和验证路径。
