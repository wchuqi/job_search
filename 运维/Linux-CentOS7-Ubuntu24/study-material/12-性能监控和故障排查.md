# Linux 学习资料：性能监控和故障排查

[返回索引](../Linux学习资料.md)

## 学习目标

- 掌握 CPU、内存、磁盘 I/O、网络、负载、进程和内核日志的排查方法。
- 能区分 CPU 忙、I/O 等待、内存不足、swap、连接耗尽、DNS 慢和磁盘满。
- 能写出标准故障排查剧本。

## 理论导读

Linux 性能排查要从现象到资源再到进程。`load average` 高不一定是 CPU 高，也可能是不可中断 I/O 等待。内存看起来快满不一定有问题，因为 Linux 会利用空闲内存做 page cache。真正的问题要看可用内存、swap、OOM、I/O 延迟、进程状态和应用指标。

排障时要先记录现场，再做变更。重启可能止血，但也会清掉关键证据。

## 核心心智模型

```text
用户感知慢
  -> CPU?
  -> 内存?
  -> 磁盘 I/O?
  -> 网络?
  -> 锁或外部依赖?
  -> 应用自身?
```

每个方向都要用证据排除或确认。

## 知识点详解

### 1. 快速现场快照

```bash
date
uptime
hostnamectl 2>/dev/null || hostname
free -h
df -hT
df -i
top -b -n1 | head -40
ss -s
systemctl --failed
journalctl -p err -b --no-pager | tail -100
```

### 2. CPU 和 load average

```bash
uptime
top
mpstat -P ALL 1 5 2>/dev/null || true
vmstat 1 5
```

`vmstat` 关键字段：

- `r`：等待 CPU 的 runnable 队列。
- `b`：不可中断睡眠，常见 I/O 等待。
- `us/sy/id/wa`：用户态、内核态、空闲、I/O wait。
- `cs/in`：上下文切换和中断。

load 高时判断：

- CPU busy：`us` 或 `sy` 高，`r` 高。
- I/O wait：`wa` 高，`b` 高。
- 进程太多或线程风暴：上下文切换高。

### 3. 内存

```bash
free -h
cat /proc/meminfo | head
vmstat 1 5
```

重点看 available，而不是只看 free。Linux 会把空闲内存用于缓存。

OOM：

```bash
dmesg -T | grep -i -E 'out of memory|killed process'
journalctl -k | grep -i -E 'out of memory|killed process'
```

swap：

```bash
swapon --show
vmstat 1 5
```

`si/so` 持续非 0 表示 swap in/out，性能会明显受影响。

### 4. 磁盘 I/O

```bash
iostat -xz 1 5 2>/dev/null || true
lsblk
df -h
```

`iostat -x` 关注：

- `%util`：设备忙碌程度。
- `await`：平均等待时间。
- `r/s`、`w/s`、`rkB/s`、`wkB/s`。

`%util` 高不一定说明饱和，要结合设备类型和延迟。云盘还要看云厂商 IOPS/吞吐限制。

### 5. 网络

```bash
ss -s
ss -ant state established | wc -l
ip -s link
sar -n DEV 1 5 2>/dev/null || true
```

连接状态：

```bash
ss -ant | awk '{print $1}' | sort | uniq -c | sort -nr
```

TIME_WAIT 多不一定是问题，要看端口耗尽、连接速率和应用模型。

### 6. 磁盘满

```bash
df -h
du -xh --max-depth=1 /var | sort -h
sudo lsof +L1
```

常见根因：

- 日志未轮转。
- core dump。
- 临时文件。
- Docker/container overlay。
- 删除文件仍被进程占用。
- journal 日志过大。

### 7. 服务启动失败

```bash
systemctl status service
journalctl -u service -b
systemctl cat service
ss -lntp
```

结合：

- 配置语法检查。
- 权限和路径。
- 端口冲突。
- 环境变量。
- SELinux/AppArmor。
- 磁盘和 inode。

## 故障剧本

### 剧本 1：CPU 飙高

1. `top` 找进程。
2. `top -H -p <pid>` 看线程。
3. `pidstat -p <pid> 1` 或 `perf top`，如工具可用。
4. 查看应用日志和最近发布。
5. 临时限流、重启实例或回滚发布。
6. 长期修复代码热点或配置。

### 剧本 2：内存持续上涨

1. `free -h` 看 available。
2. `ps aux --sort=-%mem | head` 找进程。
3. 查 OOM 日志。
4. 看 cgroup/systemd 限制。
5. 临时扩容或重启泄漏进程。
6. 长期修复泄漏、加限制和监控。

### 剧本 3：I/O wait 高

1. `vmstat` 确认 `wa` 和 `b`。
2. `iostat -xz` 找设备。
3. `iotop` 找进程，如可用。
4. 查日志、备份、压缩、数据库、容器写入。
5. 限速或错峰任务。
6. 长期优化写入模式或升级磁盘。

### 剧本 4：DNS 慢或失败

1. `getent hosts domain`。
2. `dig domain`。
3. 查看 `/etc/resolv.conf` 和 `resolvectl status`。
4. 抓包看 DNS 请求。
5. 检查 systemd-resolved、NetworkManager、netplan。
6. 修复 DNS 配置和缓存策略。

## 练习

1. 用 `stress` 或自写脚本在测试环境制造 CPU 忙，观察 `top` 和 `vmstat`。
2. 制造磁盘写入，观察 `iostat`。
3. 创建大量小文件，观察 inode。
4. 删除一个被进程打开的大文件，使用 `lsof +L1` 定位。
5. 故意让服务端口冲突，写排障记录。

## 验收

- 能解释 load average 高的不同原因。
- 能正确理解 Linux memory available。
- 能定位 OOM 证据。
- 能区分磁盘空间满和 inode 满。
- 能写出 CPU、内存、I/O、网络、服务失败的排查剧本。

## 重点

- load 高不等于 CPU 高。
- free 低不等于内存不足，要看 available 和 swap。
- 重启前先保存现场。
- 资源指标要和进程、日志、变更记录结合。

## 难点

- 性能问题常是资源链路问题。例如日志暴涨导致磁盘满，磁盘满导致服务写失败，服务重试导致 CPU 高。

## 易错

> **易错：** 看到 load 高就扩 CPU。
>
> 正确做法：先确认是 CPU runnable 队列高，还是 I/O wait 或 D 状态进程导致。

> **易错：** 看到 free 很低就判断内存不足。
>
> 正确做法：看 available、swap、OOM 和应用实际内存。
