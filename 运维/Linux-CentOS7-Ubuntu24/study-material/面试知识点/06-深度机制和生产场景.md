## 一、深度机制和生产场景

### 1. 一个 `cat /etc/hostname` 命令在内核层面大致发生什么？

**参考答案：**

Shell 解析命令后执行 `cat` 程序，`cat` 通过 libc 发起系统调用，例如 `openat` 打开文件，`read` 从文件描述符读取内容，`write` 写到 stdout，最后 `close`。这些系统调用进入内核，经过 VFS 找到对应 inode 和文件系统，命中 page cache 则直接返回，否则触发磁盘 I/O。

可以用：

```bash
strace -e openat,read,write,close cat /etc/hostname
```

> **重点：** 命令不是直接操作磁盘，而是通过系统调用和内核子系统完成。

### 2. `/proc` 和 `/sys` 为什么不是普通目录？

**参考答案：**

`/proc` 和 `/sys` 多数是虚拟文件系统。它们不是读取磁盘普通文件，而是内核动态暴露进程、内存、网络、设备和内核对象状态。写入某些文件还会改变内核运行时参数。

例如：

```bash
cat /proc/meminfo
cat /proc/<pid>/limits
cat /sys/class/net/eth0/operstate
```

> **易错：** 把 `/proc/sys` 修改当成持久配置。持久化应写 `/etc/sysctl.d/*.conf`。

### 3. `write()` 成功是否表示数据已经落盘？

**参考答案：**

通常不表示。普通 buffered I/O 的 `write()` 成功表示数据已交给内核并进入 page cache，可能只是 dirty page。真正持久化通常需要 `fsync()`、`fdatasync()` 或应用自身的刷盘策略。后台 writeback 稍后把脏页写入磁盘。

> **重点：** 数据库、日志系统和普通文件复制的持久性语义不同，不能混为一谈。

### 4. 删除大文件后空间不释放的底层原因是什么？

**参考答案：**

删除文件只是删除目录项并减少链接计数。如果某个进程仍持有该文件的打开文件描述符，内核 file 对象和 inode 引用仍存在，数据块不会释放。直到进程关闭 fd 或退出，空间才释放。

```bash
sudo lsof +L1
ls -l /proc/<pid>/fd | grep deleted
```

> **重点：** `du` 看不到已删除目录项，`df` 仍能看到空间占用。

### 5. 一次外部请求到 Linux 服务端口，经过哪些关键层？

**参考答案：**

大致经过：客户端 DNS、客户端路由、中间网络、云安全组、服务器网卡、驱动、netfilter PREROUTING、路由判断、INPUT 链、conntrack、socket lookup、应用 accept/read。任一层都可能丢包或拒绝。

排查：

```bash
ss -lntp
ip route get <client-ip>
sudo nft list ruleset 2>/dev/null || sudo iptables -S
sudo tcpdump -ni any host <client-ip> and port <port>
```

> **重点：** firewalld/ufw 是前端，netfilter 是内核框架。

### 6. `Connection refused` 和 `timeout` 的区别是什么？

**参考答案：**

`Connection refused` 通常表示目标主机可达，但目标端口无监听或主动返回 RST。`timeout` 通常表示没有收到回应，可能是防火墙丢包、路由不通、上游安全组、目标主机无回包或回程路径问题。

用 tcpdump 验证最可靠：

```bash
sudo tcpdump -ni any host <peer> and port <port>
```

### 7. free 很低但系统未必缺内存，为什么？

**参考答案：**

Linux 会把空闲内存用于 page cache 和 slab 缓存，以提高性能。这些缓存很多可以在内存压力下回收。因此应看 `MemAvailable`、swap in/out、OOM 日志和业务延迟，而不是只看 `MemFree`。

```bash
free -h
vmstat 1 5
journalctl -k | grep -i oom
```

> **易错：** 看到 cached 大就清缓存。缓存通常是性能收益。

### 8. 宿主机还有内存，为什么容器或服务仍然 OOM？

**参考答案：**

因为服务或容器可能受 cgroup 限制。超过 cgroup memory limit 会触发 memcg OOM，即使宿主机整体仍有可用内存。要查看进程 cgroup、systemd MemoryMax、容器限制和内核 OOM 日志。

```bash
cat /proc/<pid>/cgroup
systemctl show service -p MemoryCurrent -p MemoryMax
journalctl -k | grep -i oom
```

### 9. SELinux 和 AppArmor 如何进入权限决策？

**参考答案：**

传统权限和 ACL 允许之后，内核还会通过 Linux Security Module 调用 SELinux 或 AppArmor 做强制访问控制。SELinux 主要基于标签和类型策略；AppArmor 主要基于程序 profile 和路径规则。因此文件权限正确仍可能被拒绝。

CentOS 7：

```bash
getenforce
sudo ausearch -m avc -ts recent
```

Ubuntu 24.04：

```bash
sudo aa-status
sudo journalctl -k | grep -i apparmor
```

### 10. sudoers 授权一个命令时，为什么还要考虑“命令逃逸”？

**参考答案：**

某些命令本身能打开 shell、执行外部命令或写文件，例如 `vim`、`less`、`find -exec`、`tar --checkpoint-action`。即使 sudoers 只授权了这些命令，用户也可能借它获得 root shell。应授权固定参数、不可逃逸命令，必要时使用 root-owned wrapper 脚本。

> **重点：** sudo 风险不只在 `NOPASSWD`，更在授权命令是否可执行任意操作。

### 11. CentOS 7 到 Ubuntu 24.04 迁移为什么不能只替换 yum 为 apt？

**参考答案：**

迁移涉及发行版生态变化：包名、版本、systemd 版本、服务名、网络配置 ifcfg 到 netplan、防火墙 firewalld 到 ufw/nftables、安全模块 SELinux 到 AppArmor、日志路径、Python 2 到 Python 3、cgroup v1 到 v2、iptables 到 nftables、内核行为和第三方依赖都可能变化。

迁移必须做资产盘点、配置映射、回归测试和回滚方案。

### 12. 生产机器 load 很高，你如何判断是不是 CPU 问题？

**参考答案：**

先看 `uptime`、`vmstat`、`top`。如果 runnable 队列 `r` 高、CPU `us/sy` 高，可能是 CPU 忙；如果 `wa` 高、`b` 高，可能是 I/O wait；如果内存 swap in/out 高，可能是内存压力；如果 D 状态进程多，可能是磁盘或网络文件系统卡住。

```bash
uptime
vmstat 1 5
top
iostat -xz 1 5
```

> **重点：** load 高不等于 CPU 不够。

### 13. systemd 服务开机失败但手动启动成功，可能是什么原因？

**参考答案：**

常见原因包括：依赖顺序错误、网络或挂载未就绪、环境变量只存在于登录 shell、工作目录不同、服务用户权限不同、安全模块拒绝、`Type=` 配错、启动太早。

排查：

```bash
journalctl -u service -b
systemctl cat service
systemd-analyze critical-chain service
systemctl show service -p User -p WorkingDirectory -p Environment
```

### 14. 包管理器如何帮助发现系统文件被篡改？

**参考答案：**

包管理器记录包安装的文件、权限、大小、校验和等元数据。可以用验证命令检查文件是否与包数据库不一致。

CentOS：

```bash
rpm -V package
```

Ubuntu：

```bash
dpkg -V package
```

二进制或关键库异常变更需要按安全事件处理。

## 重点

- 深度面试题要答机制、证据和风险，不只答命令。
- CentOS 7 和 Ubuntu 24.04 差异要落实到具体路径、工具和系统行为。
- 生产场景回答必须包含止血和长期修复。
