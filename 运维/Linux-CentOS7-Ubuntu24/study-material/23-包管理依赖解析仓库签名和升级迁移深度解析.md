# Linux 学习资料：包管理依赖解析、仓库签名和升级迁移深度解析

[返回索引](../Linux学习资料.md)

## 学习目标

- 深入理解 RPM/YUM 与 DEB/APT 的包数据库、依赖解析、脚本钩子、仓库元数据和签名验证。
- 掌握 CentOS 7 EOL 后的仓库风险、Ubuntu 24.04 apt key 管理和第三方源治理。
- 能制定从 CentOS 7 到 Ubuntu 24.04 的迁移检查方案。
- 能处理依赖冲突、包文件被篡改、配置文件合并和升级回滚问题。

## 理论导读

包管理不是“下载并解压”。每个包包含文件列表、元数据、依赖、冲突、提供能力、pre/post install 脚本、配置文件标记和签名信息。包管理器维护本地数据库，记录系统由哪些包构成。绕过包管理器手工覆盖文件，会让审计、升级和回滚变得困难。

CentOS 7 和 Ubuntu 24.04 的包生态差异很大。CentOS 7 的问题不仅是 `yum` 命令旧，而是系统生命周期结束；Ubuntu 24.04 的问题不是“apt 新一些”，而是现代 apt 源、keyring、systemd、netplan、AppArmor、cgroup v2 体系都改变了交付方式。

## 核心心智模型

```text
仓库 metadata
  -> GPG trust
  -> dependency solver
  -> download package
  -> unpack files
  -> run maintainer scripts
  -> update package database
  -> merge or preserve config
```

## 知识点详解

## 一、包元数据和本地数据库

CentOS/RPM：

```bash
rpm -qi bash
rpm -ql bash
rpm -q --scripts bash
rpm -V bash
```

Ubuntu/DEB：

```bash
dpkg -s bash
dpkg -L bash
dpkg-query -W -f='${Package} ${Version}\n' bash
dpkg -V bash
```

`rpm -V` / `dpkg -V` 用于验证包管理器记录的文件是否被修改。配置文件变更可能是正常的，二进制被改则高风险。

## 二、依赖、Provides、Conflicts

包依赖不一定是具体包名，也可能是 capability。

RPM 中：

```bash
rpm -q --requires nginx
rpm -q --provides nginx
```

DEB 中：

```bash
apt-cache depends nginx
apt-cache show nginx | grep -E 'Depends|Provides|Conflicts'
```

依赖冲突常见来源：

- 混用第三方仓库。
- 锁定版本。
- 手工安装本地包。
- EOL 仓库和新仓库混用。
- ABI 版本不兼容。

## 三、仓库元数据

CentOS 7：

```bash
yum repolist -v
ls /etc/yum.repos.d/
yum clean all
yum makecache
```

Ubuntu 24.04：

```bash
apt-cache policy
ls /etc/apt/sources.list.d/
sudo apt update
```

Ubuntu 24.04 常见 deb822 `.sources` 格式：

```text
Types: deb
URIs: http://archive.ubuntu.com/ubuntu
Suites: noble noble-updates
Components: main universe
Signed-By: /usr/share/keyrings/ubuntu-archive-keyring.gpg
```

## 四、GPG 和信任边界

CentOS：

```bash
rpm -q gpg-pubkey
grep -R gpgcheck /etc/yum.repos.d/
```

Ubuntu：

```bash
ls /etc/apt/keyrings/
apt-key list 2>/dev/null || true
```

`apt-key` 是旧式全局信任思路。现代配置应使用 scoped keyring，例如 `signed-by=` 限定某个源只能使用指定 key。

不要为了安装方便关闭：

```text
gpgcheck=0
```

或使用不可信 key。供应链风险会变成系统级风险。

## 五、配置文件合并和升级

包升级时，配置文件可能被保留、生成 `.rpmnew`、`.rpmsave`，或在 dpkg 中提示合并。

CentOS/RPM：

```bash
find /etc -name '*.rpmnew' -o -name '*.rpmsave'
```

Ubuntu/DEB：

```bash
find /etc -name '*.dpkg-new' -o -name '*.dpkg-old' -o -name '*.ucf-*'
```

生产升级后要检查这些文件，否则新版本默认配置可能未生效，或旧配置不兼容。

## 六、脚本钩子风险

包安装/升级会执行维护脚本：

RPM：

```bash
rpm -q --scripts package
```

DEB：

```bash
ls /var/lib/dpkg/info/package.*
```

这些脚本可能创建用户、reload systemd、迁移数据、修改权限。第三方包脚本必须审查。

## 七、CentOS 7 EOL 仓库策略

CentOS 7 EOL 后：

- 默认 mirrorlist 可能不可用。
- vault 源能提供历史包，但不提供持续安全更新。
- 企业可用内部镜像或延长支持。
- 新漏洞和合规问题仍存在。

正确定位：

```text
vault/内部镜像 = 短期维持可安装
迁移/延长支持 = 安全风险治理
```

不要把换 vault 当成完成修复。

## 八、Ubuntu 24.04 第三方源治理

生产加入第三方 apt 源前检查：

- 维护者是否可信。
- 是否支持 noble。
- 是否有 scoped GPG key。
- 是否会覆盖系统核心包。
- 是否有 pinning 限制优先级。

查看优先级：

```bash
apt-cache policy package
```

APT pinning 可用于控制来源，但错误配置会导致意外降级或混装。

## 九、从 CentOS 7 迁移到 Ubuntu 24.04

迁移不是原地升级，而是重新适配发行版生态。建议清单：

### 1. 资产盘点

```bash
rpm -qa --qf '%{NAME} %{VERSION}-%{RELEASE}\n' | sort > centos7-packages.txt
systemctl list-unit-files > centos7-units.txt
ss -lntup > centos7-ports.txt
```

### 2. 配置盘点

- `/etc/sysconfig/`
- `/etc/systemd/system/`
- `/etc/nginx/`
- `/etc/sudoers.d/`
- `/etc/cron*`
- `/etc/logrotate.d/`
- SELinux 自定义策略
- firewalld 规则

### 3. 映射 Ubuntu 方案

| CentOS 7 项 | Ubuntu 24.04 替代或检查 |
| --- | --- |
| yum/rpm 包 | apt/dpkg 包名映射 |
| ifcfg | netplan |
| firewalld | ufw/nftables |
| SELinux | AppArmor 或保留应用级限制 |
| `/var/log/secure` | `/var/log/auth.log` |
| Python 2 | Python 3 兼容 |
| cgroup v1 | cgroup v2 |
| iptables legacy | nftables/iptables-nft |

### 4. 回归测试

- 服务启动。
- 配置语法。
- 网络监听。
- 防火墙规则。
- 日志路径。
- 权限和安全模块。
- 定时任务。
- 性能基线。
- 备份恢复。

## 例子：包文件被手工覆盖

现象：系统命令行为异常。

检查：

CentOS：

```bash
rpm -V coreutils
```

Ubuntu：

```bash
dpkg -V coreutils
```

如果二进制校验失败，要确认是否被手工替换、被恶意篡改或来自非官方包。处理通常是从可信仓库重装，并进行安全审计。

## 练习

1. 查询一个包的文件列表、依赖、脚本钩子。
2. 验证一个包是否有文件被修改。
3. 在 Ubuntu 24.04 上查看 apt 源优先级。
4. 在 CentOS 7 上检查 repo 是否开启 gpgcheck。
5. 写一份 CentOS 7 迁移 Ubuntu 24.04 的包和服务映射表。

## 验收

- 能解释包管理器如何解析依赖并维护数据库。
- 能说明 GPG 签名和 scoped key 的意义。
- 能定位包文件被修改。
- 能处理 rpmnew/rpmsave/dpkg-new 类配置合并问题。
- 能制定 CentOS 7 到 Ubuntu 24.04 的迁移检查清单。

## 重点

- 包管理是系统状态数据库，不只是安装工具。
- 第三方源和脚本钩子是供应链风险入口。
- CentOS 7 vault 不是安全更新方案。
- Ubuntu 24.04 apt key 应使用 scoped trust。

## 难点

- 迁移失败常不是应用代码问题，而是系统包版本、服务名、配置生成机制、安全模块和日志路径变化共同作用。

## 易错

> **易错：** 用 `--force` 或 `--nodeps` 强行安装 RPM。
>
> 正确做法：先理解依赖冲突来源。强行安装会破坏包数据库一致性。

> **易错：** 迁移只做包名替换。
>
> 正确做法：同时验证 systemd、网络、防火墙、安全模块、日志、定时任务和应用兼容。
