# Linux 学习资料：进程、信号、资源限制和调度

[返回索引](../Linux学习资料.md)

## 学习目标

- 理解进程、线程、PID、父子进程、文件描述符、信号和退出状态。
- 掌握 `ps`、`top`、`pstree`、`pgrep`、`kill`、`nice`、`ulimit`、`lsof`、`strace` 的使用场景。
- 能定位进程假死、端口占用、文件句柄耗尽、僵尸进程、OOM 和资源限制问题。

## 理论导读

Linux 运行中的程序以进程为基本管理对象。进程拥有虚拟地址空间、打开的文件描述符、环境变量、权限身份、信号处理方式和调度状态。线程在 Linux 内核中也是可调度任务，与进程共享同一地址空间和部分资源。

排查进程问题时，不能只看 CPU 占用。要同时看状态、父子关系、打开文件、网络连接、资源限制、等待事件、系统调用和内核日志。

## 核心心智模型

```text
可执行文件 + 参数 + 环境变量
  -> 进程
  -> 线程
  -> 文件描述符
  -> CPU 调度 / 内存映射 / I/O 等待
  -> 信号和退出码
```

## 知识点详解

### 1. 查看进程

```bash
ps aux
ps -ef
pstree -ap
pgrep -a nginx
top
```

常见进程状态：

| 状态 | 含义 |
| --- | --- |
| `R` | running 或 runnable |
| `S` | interruptible sleep |
| `D` | uninterruptible sleep，常见于 I/O 等待 |
| `Z` | zombie，进程已退出但父进程未回收 |
| `T` | stopped 或 traced |

`D` 状态进程通常不能被普通 kill 立即终止，因为它在等待内核不可中断操作完成。

### 2. 信号

```bash
kill -TERM <pid>
kill -KILL <pid>
kill -HUP <pid>
kill -USR1 <pid>
```

常见信号：

| 信号 | 含义 | 使用建议 |
| --- | --- | --- |
| SIGTERM | 请求正常退出 | 首选 |
| SIGKILL | 强制杀死，不能捕获 | 最后手段 |
| SIGHUP | 常用于重新加载配置 | 取决于程序 |
| SIGINT | 中断，类似 Ctrl-C | 交互命令 |
| SIGSTOP/SIGCONT | 暂停/继续 | 调试场景 |

生产中优先 TERM，再等待，再 KILL。直接 KILL 可能导致临时文件、锁文件、事务和状态没有清理。

### 3. 文件描述符

每个进程通过文件描述符访问文件、socket、pipe 等。

```bash
ls -l /proc/<pid>/fd
lsof -p <pid>
lsof -i :80
```

文件句柄耗尽常见现象：

```text
Too many open files
```

查看限制：

```bash
ulimit -n
cat /proc/<pid>/limits
```

systemd 服务的限制要在 unit 中设置：

```ini
[Service]
LimitNOFILE=65535
```

修改后：

```bash
sudo systemctl daemon-reload
sudo systemctl restart service
```

### 4. nice 和调度优先级

```bash
nice -n 10 command
renice 5 -p <pid>
```

nice 值越低优先级越高，普通用户通常只能调低自己的优先级。不要把业务进程都设成高优先级，这会让系统调度更难控制。

### 5. cgroup 和 systemd 资源控制

systemd 使用 cgroup 管理服务资源。查看：

```bash
systemctl status nginx
systemd-cgls
```

常见 unit 限制：

```ini
[Service]
MemoryMax=1G
CPUQuota=200%
TasksMax=1024
```

CentOS 7 默认 cgroup v1；Ubuntu 24.04 更常见 cgroup v2。容器、systemd 和资源隔离行为可能不同。

### 6. strace

`strace` 用于观察进程系统调用。

```bash
strace -p <pid>
strace -f -o trace.log command
```

适合排查：

- 打开哪个文件失败。
- 连接哪个地址失败。
- 卡在哪个系统调用。
- 权限错误来自哪个路径。

生产使用要谨慎，strace 会有开销，可能暴露敏感参数。

### 7. OOM

内存不足时，内核 OOM killer 可能杀死进程。

```bash
dmesg -T | grep -i -E 'out of memory|killed process'
journalctl -k | grep -i -E 'out of memory|killed process'
```

systemd 也可能根据 cgroup 限制杀进程。要区分系统级 OOM 和服务 cgroup 限制。

## 例子：排查端口被占用

```bash
sudo ss -lntp '( sport = :80 )'
sudo lsof -i :80
```

如果是 systemd 管理的服务：

```bash
systemctl status <service>
journalctl -u <service> -b
```

不要直接 kill 后就结束。应确认为什么有进程占用：旧服务未停、重复部署、配置端口冲突还是 socket activation。

## 练习

1. 启动一个 `sleep 300`，用 `ps`、`pstree`、`kill -TERM` 观察。
2. 用 `python3 -m http.server 8080` 占用端口，再用 `ss` 和 `lsof` 找出来。
3. 查看当前 shell 的 `ulimit -n`，再查看某个 systemd 服务的 `/proc/<pid>/limits`。
4. 对一个卡住的命令使用 `strace` 观察它在等待什么。
5. 用 `systemctl show service -p MainPID -p MemoryCurrent -p TasksCurrent` 查看服务资源。

## 验收

- 能解释进程状态 R/S/D/Z/T。
- 能安全选择 TERM、HUP、KILL。
- 能定位端口占用和文件句柄耗尽。
- 能区分 shell 的 ulimit 和 systemd 服务限制。
- 能用 dmesg/journalctl 查 OOM 证据。

## 重点

- 进程问题要看状态、资源、文件描述符、信号和日志。
- systemd 服务的资源限制不等同于交互 shell 的 ulimit。
- D 状态通常是内核 I/O 等待，kill 不一定立即生效。

## 难点

- OOM、cgroup 限制和应用自身退出都可能表现为进程消失，需要结合内核日志、服务日志和 systemd 状态判断。

## 易错

> **易错：** 进程卡住就直接 `kill -9`。
>
> 正确做法：先用 `strace`、日志、状态和资源指标判断原因，优先正常终止。

> **易错：** 在 shell 中 `ulimit -n 65535` 后以为 systemd 服务也生效。
>
> 正确做法：systemd 服务用 `LimitNOFILE=` 或 drop-in 配置。
