# Linux 学习资料：日志、时间同步、计划任务和审计

[返回索引](../Linux学习资料.md)

## 学习目标

- 掌握 journald、rsyslog、应用日志、内核日志和认证日志。
- 理解时间同步对日志、证书、分布式系统和审计的影响。
- 掌握 cron、systemd timer、logrotate 和 auditd。

## 理论导读

日志是生产排障的证据链。没有准确时间和完整日志，就很难还原事故。Linux 日志通常来自多个入口：systemd-journald 收集服务 stdout/stderr 和内核日志，rsyslog 写入传统 `/var/log/*` 文件，应用也可能自己写日志。CentOS 7 和 Ubuntu 24.04 都有 journald 和 rsyslog，但日志文件名和服务名不同。

时间同步是日志可信性的前提。时间错误会导致 TLS 证书校验失败、Kerberos 或 token 失效、分布式追踪混乱、备份和审计时间线不可信。

## 核心心智模型

```text
应用 stdout/stderr
  -> systemd-journald
  -> journalctl 查询
  -> 可转发给 rsyslog
  -> /var/log/* 文件
```

并非所有应用都只写 journal，也并非所有 journal 都持久化到磁盘。

## 知识点详解

### 1. journalctl

```bash
journalctl -b
journalctl -b -1
journalctl -u nginx
journalctl -u nginx --since "2026-06-06 10:00"
journalctl -p err -b
journalctl -k
journalctl -f
```

查看 journald 磁盘占用：

```bash
journalctl --disk-usage
```

清理旧日志：

```bash
sudo journalctl --vacuum-time=14d
sudo journalctl --vacuum-size=1G
```

生产应通过配置控制，而不是事故时临时大删。

### 2. 传统日志路径差异

| 日志 | CentOS 7 | Ubuntu 24.04 |
| --- | --- | --- |
| 系统消息 | `/var/log/messages` | `/var/log/syslog` |
| 认证日志 | `/var/log/secure` | `/var/log/auth.log` |
| yum/apt | `/var/log/yum.log` | `/var/log/apt/history.log`、`/var/log/dpkg.log` |
| 内核 | `dmesg`、journal | `dmesg`、journal |

### 3. logrotate

配置位置：

```bash
ls /etc/logrotate.conf
ls /etc/logrotate.d/
```

测试：

```bash
sudo logrotate -d /etc/logrotate.conf
```

强制执行：

```bash
sudo logrotate -f /etc/logrotate.conf
```

常见配置：

```text
/var/log/app/*.log {
    daily
    rotate 14
    compress
    missingok
    notifempty
    copytruncate
}
```

`copytruncate` 可用于不能重新打开日志的应用，但可能丢少量日志。更好的方式是让应用支持 log reopen。

### 4. 时间同步

查看时间：

```bash
timedatectl
date
hwclock --show 2>/dev/null || true
```

CentOS 7 常见 chrony：

```bash
systemctl status chronyd
chronyc sources -v
```

Ubuntu 24.04 可能使用 systemd-timesyncd 或 chrony：

```bash
timedatectl timesync-status 2>/dev/null || true
systemctl status systemd-timesyncd 2>/dev/null || true
systemctl status chrony 2>/dev/null || true
```

生产中建议统一 NTP 源，尤其是数据库、认证、日志、分布式系统节点。

### 5. cron

查看：

```bash
crontab -l
sudo crontab -l
ls /etc/cron.d/
ls /etc/cron.daily/
```

编辑：

```bash
crontab -e
```

格式：

```text
* * * * * command
```

cron 环境变量很少，脚本里要写绝对路径或设置 PATH。

### 6. systemd timer

systemd timer 比 cron 更适合 systemd 管理环境，可记录日志、依赖和状态。

查看：

```bash
systemctl list-timers
```

timer 示例：

```ini
[Unit]
Description=Run backup daily

[Timer]
OnCalendar=*-*-* 02:00:00
Persistent=true

[Install]
WantedBy=timers.target
```

配套 `.service` 执行实际命令。`Persistent=true` 表示错过时间后下次启动补跑。

### 7. auditd

auditd 用于审计敏感操作。

CentOS 7 常见默认安装，Ubuntu 可安装：

```bash
sudo systemctl status auditd
sudo auditctl -s
sudo ausearch -m avc -ts recent 2>/dev/null || true
```

监控文件访问示例：

```bash
sudo auditctl -w /etc/passwd -p wa -k passwd_changes
sudo ausearch -k passwd_changes
```

audit 规则过多会带来性能和日志压力。

## 例子：排查定时任务没有执行

检查：

```bash
crontab -l
systemctl status crond 2>/dev/null || systemctl status cron
grep CRON /var/log/cron 2>/dev/null || grep CRON /var/log/syslog 2>/dev/null
journalctl -u crond 2>/dev/null || journalctl -u cron
```

常见原因：

- cron 服务未运行。
- 命令依赖 PATH，但 cron 环境没有。
- 脚本没有执行权限。
- 脚本使用相对路径。
- 输出没有重定向，邮件未配置。
- 时间时区理解错误。

## 练习

1. 查看本次启动的错误日志。
2. 找到 SSH 登录失败日志。
3. 配置一个每分钟运行的 cron，记录时间到文件。
4. 用 systemd timer 实现同样任务。
5. 配置 logrotate 测试一个应用日志目录。
6. 检查 NTP 同步状态。

## 验收

- 能区分 journal 和传统日志文件。
- 能知道 CentOS 7 与 Ubuntu 24.04 的认证日志路径差异。
- 能使用 journalctl 按服务、时间、级别查询。
- 能排查 cron 环境差异导致的失败。
- 能说明时间同步对证书、日志和分布式系统的重要性。

## 重点

- 日志时间线必须可信，先保证时间同步。
- cron 的环境与交互 shell 不同。
- logrotate 的 copytruncate 有丢日志风险。
- auditd 能提供审计证据，但规则要控制范围。

## 难点

- 同一条日志可能存在 journal、rsyslog 和应用文件中，事故复盘要确认哪个是权威来源，避免漏掉 stdout/stderr。

## 易错

> **易错：** 定时任务手动执行成功，cron 中失败，就认为 cron 坏了。
>
> 正确做法：检查 PATH、工作目录、权限、环境变量、解释器路径和日志。

> **易错：** 日志时间混乱时继续按时间排序复盘。
>
> 正确做法：先确认各节点时间同步状态，再结合单调递增日志、请求 ID 或外部监控重建时间线。
