# Linux 学习资料：软件包、仓库和版本管理

[返回索引](../Linux学习资料.md)

## 学习目标

- 掌握 RPM/YUM 与 DEB/APT 的模型差异。
- 理解仓库、依赖解析、GPG 签名、版本锁定、升级、回滚和源编译边界。
- 能定位软件包冲突、依赖失败、仓库不可用和版本过旧问题。

## 理论导读

软件包管理器解决的是“哪些文件安装到哪里、由哪个包拥有、依赖哪些包、如何升级和删除”的问题。CentOS 7 使用 RPM 包格式和 yum 依赖解析；Ubuntu 24.04 使用 DEB 包格式和 apt 依赖解析。两者都不是简单下载二进制，而是围绕仓库元数据、GPG 签名、依赖关系和脚本钩子运作。

生产系统上，包管理比手工复制二进制更可审计、更可回滚、更可自动化。但包管理也带来仓库可用性、版本滞后、依赖冲突和生命周期风险。

## 核心心智模型

```text
仓库元数据
  -> 依赖解析
  -> 下载包
  -> 验证签名
  -> 执行 pre/post 脚本
  -> 安装文件和记录数据库
```

包管理数据库是系统状态的重要来源。不要绕过包管理器随意覆盖系统文件。

## 知识点详解

### 1. CentOS 7：RPM 和 YUM

常用命令：

```bash
sudo yum install -y nginx
sudo yum update -y nginx
sudo yum remove -y nginx
yum list installed
yum info nginx
yum provides '*/nginx'
rpm -qa | grep nginx
rpm -ql nginx
rpm -qf /usr/sbin/nginx
rpm -qi nginx
rpm -V nginx
```

仓库配置：

```bash
ls /etc/yum.repos.d/
yum repolist
yum clean all
yum makecache
```

CentOS 7 已 EOL，默认镜像可能不可用，需要迁移到 vault 或企业内部镜像。但这只是维持存量系统可安装包，不等于解决安全更新问题。

### 2. Ubuntu 24.04：DPKG 和 APT

常用命令：

```bash
sudo apt update
sudo apt install -y nginx
sudo apt upgrade
sudo apt remove nginx
apt list --installed
apt show nginx
apt-cache policy nginx
dpkg -l nginx
dpkg -L nginx
dpkg -S /usr/sbin/nginx
dpkg -V nginx
```

仓库配置：

```bash
ls /etc/apt/sources.list
ls /etc/apt/sources.list.d/
sudo apt update
```

Ubuntu 24.04 新环境常见 deb822 格式 `.sources` 文件，也可能继续使用传统 `.list` 文件。不要只假设所有仓库都在一个 `sources.list` 中。

### 3. apt update 与 apt upgrade 的区别

```bash
sudo apt update
sudo apt upgrade
```

- `apt update`：更新本地仓库索引，不安装新版本。
- `apt upgrade`：根据索引升级已安装包。

CentOS 对应：

```bash
sudo yum makecache
sudo yum update
```

### 4. 查询文件属于哪个包

CentOS 7：

```bash
rpm -qf /usr/bin/curl
yum provides '*/dig'
```

Ubuntu 24.04：

```bash
dpkg -S /usr/bin/curl
sudo apt install -y apt-file
sudo apt-file update
apt-file search /usr/bin/dig
```

### 5. 版本锁定

CentOS 7：

```bash
sudo yum install -y yum-plugin-versionlock
sudo yum versionlock add nginx
sudo yum versionlock list
```

Ubuntu 24.04：

```bash
sudo apt-mark hold nginx
apt-mark showhold
sudo apt-mark unhold nginx
```

版本锁定用于避免关键服务被无意升级，但长期锁定会积累安全风险。

### 6. GPG 签名和仓库信任

仓库签名用于确认包来源可信。生产不要随便执行网上的：

```bash
curl URL | sudo bash
```

更稳妥的做法：

- 使用官方仓库或企业镜像。
- 验证 GPG key 来源。
- 固定仓库 URL 和版本。
- 记录变更。

Ubuntu 新版更推荐把 key 放在 `/etc/apt/keyrings/`，并在源文件中用 `signed-by=` 指定，而不是把 key 全局加入信任。

### 7. 源码安装的边界

源码安装适合：

- 发行版仓库没有需要的版本。
- 需要自定义编译选项。
- 测试或隔离环境。

生产风险：

- 文件不归包管理器管理。
- 升级和卸载困难。
- 安全更新难追踪。
- 可能覆盖系统库或命令。

更好的方式是构建内部 RPM/DEB 包，或使用容器隔离。

### 8. Snap、PPA、EPEL

CentOS 7 常见扩展仓库：

```bash
sudo yum install -y epel-release
```

Ubuntu 常见 PPA 和 snap：

```bash
snap list
apt-cache policy package
```

PPA 和第三方仓库会引入供应链风险。生产使用前要确认维护者、更新策略和回滚方案。

## 例子：排查安装 Nginx 失败

CentOS 7：

```bash
yum repolist
yum info nginx
yum provides '*/nginx'
sudo yum clean all
sudo yum makecache
```

可能原因：

- EOL 后默认仓库不可达。
- 未启用 EPEL 或企业仓库。
- DNS 或代理问题。
- GPG key 问题。

Ubuntu 24.04：

```bash
sudo apt update
apt-cache policy nginx
sudo apt install -y nginx
```

可能原因：

- 源配置错误。
- `apt update` 未执行。
- dpkg 锁被另一个 apt 进程占用。
- 代理或 DNS 问题。

## 练习

1. 查询 `bash` 包安装了哪些文件。
2. 找出 `/usr/bin/systemctl` 属于哪个包。
3. 安装、启动、卸载 nginx，并观察配置文件是否被保留。
4. 在 Ubuntu 24.04 上用 `apt-mark hold` 锁定一个测试包。
5. 在 CentOS 7 上查看 yum repo 列表，并确认是否使用 EOL vault 或内部镜像。

## 验收

- 能区分包格式、包数据库和依赖解析工具。
- 能查询文件属于哪个包。
- 能解释 `apt update` 与 `apt upgrade` 的区别。
- 能处理仓库不可用、GPG 失败和依赖冲突。
- 能说明第三方仓库和源码安装的生产风险。

## 重点

- 包管理器维护系统状态，手工覆盖系统文件会破坏可审计性。
- CentOS 7 EOL 后仓库和安全更新是核心风险。
- Ubuntu 24.04 的 apt 源和 key 管理比老版本更重视 scoped trust。

## 难点

- 依赖冲突不只是“缺包”，可能来自仓库混用、版本锁定、第三方源优先级和 ABI 不兼容。

## 易错

> **易错：** 在生产执行 `curl URL | sudo bash` 安装软件。
>
> 正确做法：使用可信仓库、验证签名、固定版本，并保留回滚方案。

> **易错：** CentOS 7 改成 vault 源后认为安全风险解决。
>
> 正确做法：vault 只解决旧包可下载，不提供持续安全修复。应规划迁移。
