# Redis学习资料：字符串、哈希、列表、集合、有序集合底层编码

[返回索引](../Redis学习资料.md)

## 学习目标

- 理解 Redis value 类型和底层编码不是一回事。
- 掌握 String、Hash、List、Set、ZSet 的编码选择和性能影响。
- 能解释为什么小对象和大对象内存表现差异巨大。

## 理论导读

Redis 对外暴露的是数据类型，对内会根据元素数量、元素大小和配置选择不同编码。例如小 Hash 可能用紧凑编码，大 Hash 可能转为哈希表；小整数集合可能用 intset，大集合转为 hashtable；ZSet 可能从 listpack 转为 skiplist+dict。编码转换通常不可逆，数据增长会改变内存和性能特征。

## 类型和编码

可以用：

```bash
OBJECT ENCODING key
MEMORY USAGE key
TYPE key
```

常见概念：

- String：int、embstr、raw。
- Hash：listpack、hashtable。
- List：quicklist。
- Set：intset、hashtable。
- ZSet：listpack、skiplist。

具体名称和阈值受版本和配置影响，学习时重点理解“紧凑编码到通用结构”的转换思路。

## String 编码

- 小整数可能使用 int 编码。
- 短字符串可能使用 embstr，内存分配更紧凑。
- 长字符串使用 raw。

影响：大量小 String 的元数据开销可能很明显，key 名也占内存。

## Hash 编码

小 Hash 使用紧凑编码，有利于节省内存。字段数量或字段长度超过阈值后转为 hashtable，查询仍快，但内存开销上升。

适用：对象字段较多且需要局部更新时优于大 JSON String。

## List 编码

现代 Redis 常用 quicklist，将链表和紧凑块结合，平衡插入删除和内存效率。

风险：大 List 做 `LRANGE 0 -1` 或大批量删除会阻塞。

## Set 编码

整数小集合可用 intset。出现非整数或数量超过阈值后转为 hashtable。

风险：`SMEMBERS` 返回全量集合，大集合危险。

## ZSet 编码

小 ZSet 可用紧凑编码。大 ZSet 使用 dict + skiplist：dict 提供 member 到 score 查找，skiplist 支持按 score 排序范围查询。

## 练习

1. 创建小 hash，查看 `OBJECT ENCODING`。
2. 不断增加字段，观察编码变化。
3. 比较 String JSON 和 Hash 的 `MEMORY USAGE`。

## 验收

- 能区分 Redis 数据类型和底层编码。
- 能解释编码转换为何影响内存。
- 能说出 ZSet 为什么同时需要查找和排序结构。

## 重点

- 数据小和数据大时，同一类型的底层结构可能不同。
- big key 风险和底层编码、命令返回大小都相关。
- `OBJECT ENCODING`、`MEMORY USAGE` 是理解内存的入口。

## 易错

> **易错：** 只按数据类型评估性能，不看元素数量和编码。
>
> 正确做法：结合 `TYPE`、`OBJECT ENCODING`、`MEMORY USAGE`、命令复杂度评估。
