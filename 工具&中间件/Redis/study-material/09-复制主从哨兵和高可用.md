# Redis学习资料：复制、主从、哨兵和高可用

[返回索引](../Redis学习资料.md)

## 学习目标

- 理解主从复制、读写分离、复制延迟和故障切换。
- 掌握 Sentinel 的监控、选主和通知机制。
- 能说明 Redis 高可用的一致性边界。

## 理论导读

Redis 复制通常是主节点接收写入，从节点复制主节点数据。复制提高读扩展和可用性，但主要是异步复制，故障时可能丢失尚未复制到从节点的写入。Sentinel 负责监控主从、判断主观/客观下线、选举并执行故障转移。

## 主从复制

```bash
REPLICAOF 127.0.0.1 6379
INFO replication
```

复制用于：

- 读扩展。
- 高可用切换。
- 备份节点。
- 降低主节点持久化压力。

风险：

- 复制延迟。
- 主从不一致。
- 从节点读到旧数据。
- 大 key 或网络抖动导致延迟扩大。

## Sentinel

Sentinel 职责：

- 监控主从实例。
- 主观下线和客观下线判定。
- Sentinel 之间选举 leader。
- 选择从节点提升为主。
- 让其他从节点复制新主。
- 通知客户端或运维系统。

## 选主考虑

Sentinel 会综合从节点优先级、复制偏移量、run id 等因素选择更合适的从节点。实际运维中要配置合理的优先级，避免低规格节点被提升。

## 一致性边界

Redis 主从高可用不能保证零丢失。主节点写成功但还没复制给从节点时宕机，切换后这部分数据可能丢失。

缓解：

- 使用 `WAIT` 等待复制到一定从节点数量，但它不是完整强一致。
- 业务允许短暂不一致。
- 关键数据落主数据库。
- 监控复制延迟。

## 练习

1. 搭建一主一从，观察 `master_repl_offset` 和 `slave_repl_offset`。
2. 停止主节点，观察 Sentinel failover。
3. 在故障前高频写入，讨论可能丢失窗口。

## 验收

- 能解释主从复制用途和风险。
- 能说明 Sentinel 的主观下线、客观下线。
- 能解释高可用不等于强一致。

## 重点

- Redis 复制以异步为主，读从库要接受延迟。
- Sentinel 管故障切换，不做数据强一致。
- 客户端必须支持拓扑变更。

## 易错

> **易错：** 使用从节点读关键一致性数据。
>
> 正确做法：读从库只用于可接受延迟的场景；强一致读应读主库或使用更合适存储。
