# Redis学习资料：生产故障案例和排障剧本

[返回索引](../Redis学习资料.md)

## 学习目标

- 掌握 Redis 常见生产故障排查路径。
- 能从指标定位根因。
- 建立事故预防和复盘模板。

## 通用排障命令

```bash
INFO
INFO memory
INFO clients
INFO stats
INFO persistence
INFO replication
SLOWLOG GET 20
LATENCY LATEST
CLIENT LIST
CONFIG GET maxmemory
```

生产谨慎：

```bash
MONITOR
KEYS *
SMEMBERS huge:set
LRANGE huge:list 0 -1
```

## 剧本 1：Redis 突然变慢

检查：

1. SLOWLOG 是否有慢命令。
2. 是否有 big key 返回大结果。
3. 是否正在 BGSAVE 或 AOF rewrite。
4. 是否内存接近 maxmemory。
5. 客户端输出缓冲是否堆积。
6. CPU、网络、磁盘是否异常。

## 剧本 2：内存打满

检查：

```bash
INFO memory
INFO stats
redis-cli --bigkeys
```

判断：

- 新 key 增长。
- TTL 缺失。
- big key。
- 内存碎片。
- 客户端缓冲。
- fork COW 峰值。

修复：

- 分批删除或 UNLINK。
- 增加 TTL。
- 拆分 big key。
- 调整 maxmemory 和淘汰策略。
- 扩容或拆实例。

## 剧本 3：缓存命中率下降

检查：

- keyspace_hits/misses。
- 是否大批 key 过期。
- 是否淘汰增加。
- 是否发布导致 key 命名变化。
- 是否 Redis 重启冷启动。

修复：

- 预热。
- TTL 随机化。
- 防穿透。
- 扩容。

## 剧本 4：主从复制延迟

检查：

```bash
INFO replication
CLIENT LIST
```

原因：

- 网络慢。
- 从节点负载高。
- 主节点写入量大。
- big key 复制。
- backlog 不足导致全量同步。

## 剧本 5：Cluster CROSSSLOT

原因：多 key 不在同一 slot。修复：使用 hash tag 设计同槽 key，或改业务流程避免跨 slot 多 key 原子操作。

## 验收

- 能按故障类型选择命令。
- 能避免排障命令造成二次故障。
- 能写出 Redis 事故复盘。

## 重点

- Redis 故障常由业务 key 模型触发。
- 排障先看指标和慢命令，再操作。
- 删除和扫描要分批、低峰、可回滚。
