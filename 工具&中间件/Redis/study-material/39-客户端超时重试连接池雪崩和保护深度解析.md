# Redis学习资料：客户端超时、重试、连接池雪崩和保护深度解析

[返回索引](../Redis学习资料.md)

## 学习目标

- 理解客户端连接池、超时、重试如何放大 Redis 故障。
- 设计限流、熔断、降级、退避和隔离。
- 避免 Redis 故障拖垮应用和数据库。

## 理论导读

Redis 服务器短暂变慢时，客户端如果没有保护，会出现请求堆积、连接池耗尽、线程阻塞、超时重试、流量翻倍、数据库回源暴增。很多“Redis 故障”最终演变为应用和数据库雪崩，是客户端保护缺失。

## 连接池参数

关注：

- max connections。
- max wait time。
- command timeout。
- connect timeout。
- idle timeout。
- validation。
- per-node pool for Cluster。

连接池不是越大越好。过大连接池会让 Redis 承受更高并发，增加排队和内存。

## 超时设置

原则：

- 连接超时短。
- 命令超时基于 SLA。
- 区分读写和不同命令。
- 超时后不要无脑重试。

## 重试策略

只对幂等操作谨慎重试：

- GET 可重试。
- SET 视业务幂等性。
- INCR、LPUSH、XADD 重试可能重复写。
- Lua 可能部分执行成功但客户端超时。

要使用指数退避、最大次数、熔断和重试预算。

## 降级策略

- 本地缓存返回旧值。
- 非关键功能跳过。
- 限流保护数据库回源。
- 热点 key 请求合并。
- 只读模式。
- 快速失败。

## 练习

1. 模拟 Redis 延迟，观察连接池等待。
2. 设置不同重试次数，观察请求放大。
3. 为缓存读取设计降级路径。

## 验收

- 能画出 Redis 慢到应用雪崩链路。
- 能设计连接池和超时参数。
- 能说明哪些命令不能随意重试。

## 易错

> **易错：** Redis 超时后增加连接池和重试次数。
>
> 正确做法：先限流、熔断、降级和排查慢因；连接池过大可能放大故障。
