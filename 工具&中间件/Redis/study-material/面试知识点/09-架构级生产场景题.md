## 一、架构和生产

### 1. Redis 延迟升高后，为什么应用和数据库也可能雪崩？

**参考答案：**

Redis 变慢后，客户端请求在连接池排队，线程阻塞，命令超时。如果客户端无脑重试，会放大请求量；缓存未命中或超时后大量回源数据库，数据库承压，最终应用线程池、连接池、数据库都可能被拖垮。应限流、熔断、降级、控制重试和保护回源。

### 2. 如何设计 Redis 容量规划？

**参考答案：**

按 key pattern 统计数量、key 长度、value 大小、TTL、读写 QPS、热点、是否允许淘汰，再加入对象元数据、碎片、client buffer、replication backlog、fork COW、AOF rewrite 和峰值预留。用真实命令比例压测验证。

### 3. Redis 分布式锁如何增强安全性？

**参考答案：**

基础锁使用 `SET key value NX PX ttl`，释放用 Lua 校验 value。关键场景加入 fencing token，由下游资源拒绝旧 token；同时使用业务幂等、状态机和数据库约束。Redlock 可降低单节点风险，但不能替代下游一致性保护。

### 4. big key 在线拆分怎么做？

**参考答案：**

先止血限流和避免全量读取；设计新 key 模型；新写双写；后台分批迁移旧数据；读路径优先新结构，缺失回源旧结构；校验一致性；切换读写；低峰分批删除旧 key。整个过程要监控延迟和内存。

### 5. Redis 备份如何证明可靠？

**参考答案：**

备份必须定期在隔离环境恢复验证，记录恢复耗时、数据校验、版本兼容、RTO/RPO。只复制 RDB/AOF 文件但从未恢复，不算可靠备份。

### 6. Sentinel 切主期间客户端要注意什么？

**参考答案：**

客户端必须支持 Sentinel 发现新主，旧连接失败后刷新拓扑；写请求要有超时和有限重试；非幂等写不能盲目重试；切换期间需要降级和保护数据库回源。

### 7. Cluster 网络分区时如何降低脑裂风险？

**参考答案：**

合理设置 cluster-node-timeout，使用多数派机制，配置 `min-replicas-to-write` 和 `min-replicas-max-lag`，客户端及时刷新拓扑，业务侧设计幂等和补偿。Cluster failover 不能消除所有脑裂风险。

### 8. 如何做 Redis 压测？

**参考答案：**

压测要模拟真实命令比例、value 大小、key 热点分布、连接数、pipeline 批量、读写比例、Cluster 路由、持久化开启状态和故障场景。不能只用默认小 value redis-benchmark 代表生产能力。

### 9. Redis 内存 RSS 高但 used_memory 不高，怎么分析？

**参考答案：**

可能是内存碎片、allocator 保留、fork COW、client buffer 或 OS 视角差异。查看 `INFO memory`、allocator 指标、client list、持久化状态。处理方式可能是 active defrag、迁移重启、减少大 key、调整写入峰值。

### 10. 缓存一致性如何做到可观测？

**参考答案：**

监控命中率、回源 QPS、缓存删除失败、补偿队列堆积、binlog/CDC 延迟、热点重建次数、旧值命中比例和业务不一致告警。没有补偿和观测的一致性方案不可控。
