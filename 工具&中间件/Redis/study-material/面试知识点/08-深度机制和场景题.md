## 一、深度机制

### 1. Redis 单线程为什么还会有 fork、后台线程和 IO 线程？

**参考答案：**

“单线程”主要描述核心命令执行路径。持久化 RDB/AOF rewrite 可能 fork 子进程，异步释放和网络 IO 可能使用线程。命令执行仍需避免慢命令阻塞事件循环。

### 2. RDB fork 时为什么会有 COW 内存峰值？

**参考答案：**

fork 后父子进程共享物理页。父进程继续写入时，修改页会被复制，这就是 copy-on-write。写入越多、热数据越多，额外内存越高。

### 3. Redis 过期 key 会准时删除吗？

**参考答案：**

不会保证准时物理删除。Redis 通过惰性删除和定期抽样删除处理过期 key。访问过期 key 时会删除；未访问的过期 key 可能短暂占用内存。

### 4. LRU 淘汰是精确的吗？

**参考答案：**

不是全局精确 LRU，而是近似 LRU。Redis 从候选样本中选择较合适的 key 淘汰，样本越大越接近真实，但成本更高。

### 5. 为什么 big key 会影响复制和 Cluster 迁移？

**参考答案：**

big key 序列化、网络传输、加载和删除都耗时。复制时会占带宽和缓冲，Cluster 迁移时单个大 key 迁移不可细分，容易造成延迟尖刺。

### 6. Redis 分布式锁在主从切换时有什么风险？

**参考答案：**

如果锁写入主节点后尚未复制到从节点，主节点宕机并切换，从节点可能没有锁记录，其他客户端可以再次获得锁，破坏互斥。因此 Redis 锁不适合要求绝对互斥的关键一致性场景。

## 二、场景题

### 7. 商品详情缓存如何设计？

**参考答案：**

使用 cache-aside，key 设置随机 TTL；不存在商品做空值缓存；热点商品使用逻辑过期或互斥重建；更新数据库后删除缓存并做失败补偿；监控命中率、重建耗时和热点 key。

### 8. Redis 内存持续上涨怎么排查？

**参考答案：**

看 `INFO memory`、keyspace、TTL 分布、evicted_keys、big key、客户端缓冲、碎片、AOF/RDB rewrite、业务新增 key。确认是业务数据增长、TTL 缺失、碎片还是缓冲。

### 9. Cluster 迁移期间客户端大量报 ASK/MOVED 怎么办？

**参考答案：**

确认客户端是否支持 Cluster 重定向和路由缓存更新；检查迁移是否在高峰执行；查看是否有 big key；控制迁移速率；必要时暂停迁移或升级客户端。

### 10. Redis 做消息队列时如何处理重复消费？

**参考答案：**

使用 Stream 消费组，消费成功后 XACK；对 pending 消息做重领；业务处理必须幂等，可用业务唯一 ID、防重表或状态机；失败多次进入死信逻辑。
