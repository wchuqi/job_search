# Redis学习资料：缓存一致性、分布式锁和高并发场景深度解析

## 学习目标

- 深入分析缓存数据库双写失败路径。
- 理解分布式锁、逻辑过期、请求合并和热点保护。
- 能为高并发业务写出可落地方案。

## 理论导读

Redis 在高并发系统中常处于数据库前面。缓存一致性问题不是一个命令能解决，而是读写链路、失败补偿、消息可靠性、TTL、锁和业务容忍度共同决定。分布式锁也不是“加锁就安全”，锁过期、客户端暂停、主从切换都会破坏理想互斥。

## 数据库更新和缓存删除失败矩阵

| 步骤 | 失败 | 结果 |
| --- | --- | --- |
| 先删缓存再写库 | 写库失败 | 缓存缺失，后续可能重建旧值 |
| 先写库再删缓存 | 删缓存失败 | 缓存保留旧值 |
| 写库成功后发消息删缓存 | 消息失败 | 旧缓存存在 |
| 延迟双删 | 第二次删除失败 | 仍可能旧值 |

核心是补偿和可观测，而不是寻找绝对无失败顺序。

## 热点重建

互斥锁：

```bash
SET rebuild:product:1 token NX EX 10
```

只有拿到锁的请求查库重建，其他请求等待、返回旧值或降级。

逻辑过期：

- value 中保存 expireAt。
- 物理 TTL 可较长。
- 发现逻辑过期后异步重建。
- 请求可先返回旧值。

适合热点数据，牺牲短暂新鲜度换稳定。

## 分布式锁深层风险

- 锁过期后业务还没执行完。
- 客户端 GC 或 STW 暂停。
- Redis 主从故障切换导致锁丢失。
- 释放锁未校验 value。
- 锁粒度过大造成吞吐下降。

因此 Redis 锁适合“降低并发冲突”，关键一致性业务应结合数据库约束、幂等表、状态机。

## 限流设计

- 固定窗口：简单，有边界突刺。
- 滑动窗口：准确，内存和计算更多。
- 令牌桶：允许突发，平滑平均速率。
- 漏桶：输出平滑。

Redis 实现时要控制 key 数量和过期。

## 验收

- 能说出缓存一致性所有主要失败路径。
- 能为热点 key 设计逻辑过期方案。
- 能说明 Redis 锁的适用边界。

## 重点

- 一致性方案要有补偿、监控和业务容忍度。
- 热点保护要避免所有请求同时查库。
- 锁只能保护临界区，不能替代业务幂等。

## 易错

> **易错：** 认为加 Redis 分布式锁后业务就绝对串行。
>
> 正确做法：考虑锁过期、客户端暂停、故障切换和幂等兜底。
