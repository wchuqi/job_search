# Redis学习资料：命令执行路径、复杂度和阻塞点深度解析

[返回索引](../Redis学习资料.md)

## 学习目标

- 从命令路径理解 Redis 延迟来源。
- 掌握复杂度、返回大小、对象编码和阻塞点的关系。
- 能评估一个命令是否适合生产高峰执行。

## 理论导读

Redis 命令延迟不是只由时间复杂度决定。完整路径包括网络读取、协议解析、命令查找、权限检查、key 查找、过期判断、数据结构操作、生成响应、写回网络、传播给 AOF/复制。某些步骤不进入 SLOWLOG，但会影响客户端感知延迟。

## 命令执行路径

简化：

```text
socket readable
  -> read query buffer
  -> parse RESP
  -> lookup command
  -> ACL/auth check
  -> lookup key
  -> expire check
  -> execute data structure operation
  -> create reply
  -> append AOF / replication propagation
  -> write response
```

## 复杂度之外的因素

- 返回结果数量。
- value 大小。
- 编码转换。
- lazy expire。
- key miss 和 key hit 不同路径。
- AOF/replication 传播。
- 客户端读取速度。

## 阻塞型风险命令

风险不是命令名本身，而是数据规模：

- `SMEMBERS` 大 Set。
- `HGETALL` 大 Hash。
- `LRANGE 0 -1` 大 List。
- `ZRANGE` 大范围。
- `KEYS` 全库扫描。
- 大 Lua。
- 删除大 key。

## 练习

1. 对同一 Set 比较 `SCARD` 和 `SMEMBERS`。
2. 用不同范围执行 `ZRANGE`，观察响应大小和延迟。
3. 删除大 key，比较 `DEL` 和 `UNLINK`。

## 验收

- 能解释 SLOWLOG 不覆盖完整客户端延迟。
- 能用评审模板判断命令风险。
- 能识别返回大结果的命令。

## 重点

- O(1) 命令也可能因 value 大而慢。
- 返回大小是生产风险核心因素。
- 命令评审要结合数据规模。

## 易错

> **易错：** 看到命令复杂度 O(1) 就认为一定安全。
>
> 正确做法：同时评估 value 大小、网络返回、客户端读取和复制传播。
