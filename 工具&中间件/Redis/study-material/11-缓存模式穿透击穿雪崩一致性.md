# Redis学习资料：缓存模式、穿透、击穿、雪崩和一致性

[返回索引](../Redis学习资料.md)

## 学习目标

- 掌握 cache-aside、read-through、write-through、write-behind 的差异。
- 识别缓存穿透、击穿、雪崩。
- 设计缓存和数据库的一致性方案。

## 理论导读

Redis 最常见用途是缓存，但缓存的本质是牺牲部分一致性换取性能和抗压能力。缓存设计必须回答：数据从哪里来、何时写入、何时失效、失败怎么补偿、并发重建怎么控制、缓存和数据库短暂不一致是否可接受。

## Cache Aside

典型流程：

1. 读缓存。
2. 缓存命中返回。
3. 未命中读数据库。
4. 写入缓存并设置 TTL。

更新常见策略：先更新数据库，再删除缓存。删除失败需要重试、消息补偿或延迟双删等策略。

## 缓存穿透

查询不存在数据，缓存没有，数据库反复被打。

方案：

- 空值缓存，设置较短 TTL。
- Bloom Filter 过滤明显不存在 key。
- 参数校验。
- 限流和熔断。

## 缓存击穿

热点 key 过期，大量请求同时打到数据库。

方案：

- 互斥锁重建。
- 逻辑过期，后台刷新。
- 热点 key 不设置普通短 TTL，配合主动刷新。

## 缓存雪崩

大量 key 同时过期或 Redis 故障，流量压到数据库。

方案：

- TTL 加随机抖动。
- 多级缓存。
- 限流熔断。
- Redis 高可用。
- 预热和分批失效。

## 一致性边界

常见“更新数据库再删除缓存”仍可能失败：

- 数据库更新成功，删除缓存失败。
- 删除缓存成功，读请求重建旧值。
- 主从数据库延迟导致缓存旧值。
- 消息补偿延迟。

设计目标通常是最终一致，而不是强一致。

## 练习

1. 设计商品详情缓存，包含空值缓存和 TTL 随机化。
2. 用 SET NX EX 实现互斥重建。
3. 画出数据库更新和缓存删除失败的补偿流程。

## 验收

- 能解释穿透、击穿、雪崩区别。
- 能说明更新数据库后为什么通常删除缓存而不是更新缓存。
- 能设计最终一致方案和失败补偿。

## 重点

- 缓存一致性要明确业务可接受的不一致窗口。
- 热点 key 不能只靠普通 TTL。
- 删除缓存失败必须有补偿。

## 易错

> **易错：** 更新数据库后同步更新缓存，认为一定更一致。
>
> 正确做法：并发更新下同步更新缓存容易写入旧值，常见方案是更新数据库后删除缓存，并设计失败补偿。
