# Redis学习资料：big key、hot key 在线治理和拆分迁移深度解析

[返回索引](../Redis学习资料.md)

## 学习目标

- 掌握 big key 和 hot key 的发现、评估、止血和长期治理。
- 设计在线拆分、迁移和降级方案。
- 避免治理动作造成二次故障。

## 理论导读

big key 是数据规模问题，hot key 是访问集中问题。二者可能重叠，也可能独立。big key 治理难点是不能一次性读取、删除或迁移；hot key 治理难点是 Redis 单分片无法无限承受单 key 访问。治理要先止血，再迁移，再修模型。

## 发现 big key

方法：

- `redis-cli --bigkeys` 抽样。
- `MEMORY USAGE key`。
- SCAN + 类型抽样。
- 客户端侧记录大响应。
- 慢查询和带宽异常。

生产注意：扫描要低峰、限速、分批。

## big key 在线拆分

例：大 Hash 拆为多个分片 Hash：

```text
user:attrs:{uid}:0
user:attrs:{uid}:1
...
```

迁移步骤：

1. 新写双写到新结构。
2. 后台分批迁移旧数据。
3. 读优先新结构，缺失回源旧结构。
4. 校验数量和抽样一致性。
5. 切换读路径。
6. 分批删除旧 key。

## hot key 治理

方案：

- 本地缓存。
- 请求合并。
- 多副本读。
- key 分片，读写聚合。
- 限流和降级。
- 提前预热。

对于读多写少热点，本地缓存效果通常最好；对于写热点，拆分和异步聚合更合适。

## 删除 big key

优先使用异步删除类能力或分批删除成员：

```bash
UNLINK key
SSCAN key 0 COUNT 1000
```

直接 `DEL bigkey` 可能阻塞。

## 练习

1. 构造大 Set，设计分批迁移。
2. 用本地缓存方案治理热点商品详情。
3. 写 big key 拆分迁移 checklist。

## 验收

- 能区分 big key 和 hot key。
- 能设计在线拆分流程。
- 能说明治理命令的二次风险。

## 重点

- 先止血，后迁移，最后修模型。
- 不能为了查 big key 执行大范围阻塞命令。
- hot key 常需要客户端和业务层共同治理。

## 易错

> **易错：** 发现 big key 后生产直接 `DEL`。
>
> 正确做法：评估大小，优先低峰异步删除或分批删除，必要时先扩容和限流。
