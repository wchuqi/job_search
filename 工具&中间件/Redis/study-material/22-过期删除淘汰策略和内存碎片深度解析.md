# Redis学习资料：过期删除、淘汰策略和内存碎片深度解析

[返回索引](../Redis学习资料.md)

## 学习目标

- 深入理解过期删除和内存淘汰的触发机制。
- 掌握近似 LRU/LFU 的边界。
- 能处理内存碎片和容量告警。

## 理论导读

Redis 不会为每个 key 建一个精准定时器。过期 key 通过惰性删除和定期抽样删除处理。内存达到限制时，写命令触发淘汰流程。LRU/LFU 是近似算法，通过采样选择候选 key。理解“近似”和“抽样”很重要，否则会误以为 Redis 会精确淘汰最久未使用 key。

## 过期字典

设置 TTL 时，key 的过期时间记录在过期字典中。keyspace 保存 value，expires 保存 key 到过期时间的映射。

## 删除策略

- 惰性删除：访问时发现过期。
- 定期删除：周期性抽样。
- 主从复制中，过期删除通常由主节点决定并传播删除。

## 淘汰触发

当 used_memory 超过 maxmemory，Redis 在执行写命令时尝试释放内存。如果策略是 noeviction，写命令报错。

## LRU/LFU 近似

Redis 为对象维护访问相关元数据，淘汰时从候选 key 中采样，选择较合适的淘汰对象。

含义：

- 不是全局精确排序。
- sample 越大越接近真实，但 CPU 成本更高。
- LFU 更适合长期热点识别，但计数也会衰减。

## 内存碎片

碎片可能来自：

- 大量创建删除不同大小对象。
- allocator 保留内存。
- fork COW。
- 大 key 删除。

指标：

```bash
INFO memory
MEMORY STATS
MEMORY PURGE
```

具体命令可用性与版本相关。

## 练习

1. 设置 maxmemory 和 allkeys-lru，观察 evicted_keys。
2. 创建大量带 TTL 的 key，观察 expired_keys。
3. 创建和删除不同大小 key，观察碎片指标。

## 验收

- 能解释过期删除为什么不精准。
- 能说明 LRU/LFU 是近似淘汰。
- 能列出内存碎片的常见来源。

## 重点

- 淘汰策略要和业务数据分层配合。
- volatile 策略只淘汰有 TTL 的 key。
- 内存告警要看 used_memory、RSS、碎片和 fork 峰值。

## 易错

> **易错：** 选择 volatile-lru 后，所有 key 都会被淘汰。
>
> 正确做法：volatile 只在设置 TTL 的 key 中淘汰；无 TTL key 不参与。
