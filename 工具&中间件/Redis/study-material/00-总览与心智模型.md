# Redis学习资料：总览与心智模型

[返回索引](../Redis学习资料.md)

## 学习目标

- 建立 Redis 的整体模型：内存数据结构服务器、事件循环、命令处理、持久化和高可用。
- 区分 Redis 用作缓存、内存数据库、消息系统、分布式协调组件时的边界。
- 明确 Redis 的优势、风险和生产使用前提。

## 理论导读

Redis 的核心不是“把数据放内存所以快”这么简单。它提供多种内存数据结构，每条命令在服务端按照协议解析后执行，结果返回给客户端。常规命令执行路径由事件循环驱动，避免了复杂锁竞争；数据结构经过专门编码优化，常见操作复杂度低；持久化、复制和 Cluster 则解决恢复、高可用和扩展问题。

Redis 的危险也来自这些特性：单线程命令执行意味着单个慢命令会阻塞其他请求；内存数据库意味着容量和淘汰策略必须设计；异步复制意味着故障时可能丢最近写入；缓存意味着一致性不是天然保证。

## 核心心智模型

可以把 Redis 想成一台“内存中的数据结构机器”：

- keyspace：所有 key 的名字空间。
- object：每个 value 都有类型、编码、引用信息和 LRU/LFU 相关元数据。
- event loop：接收网络事件、解析命令、执行命令、返回结果。
- persistence：把内存状态用 RDB/AOF 方式落盘。
- replication：把主节点写入传播到从节点。
- cluster：把 key 按 hash slot 分布到多个主节点。

## Redis 适合什么

- 热点数据缓存。
- 会话、验证码、短期 token。
- 计数器、排行榜、去重、签到。
- 限流、分布式锁、幂等标记。
- 轻量消息队列、Stream 消费组。
- 实时状态和短生命周期数据。

## Redis 不适合什么

- 作为唯一强一致持久化数据库保存核心账务数据。
- 大 value、大集合、无限增长队列不做治理。
- 跨 key 复杂事务。
- 大范围扫描和复杂查询。
- 需要强多行事务、复杂索引和 SQL 语义的场景。

## 例子

```bash
redis-cli
SET user:1:name alice EX 3600
GET user:1:name
INCR article:100:views
ZADD rank:daily 100 user:1
ZRANGE rank:daily 0 -1 WITHSCORES
```

## 练习

1. 用 String 实现验证码，设置 TTL。
2. 用 Hash 保存用户资料。
3. 用 ZSet 实现排行榜。
4. 用 Set 实现去重。

## 验收

- 能说清 Redis 和关系型数据库的职责差异。
- 能解释“单条命令原子”和“事务强一致”不是一回事。
- 能说明 Redis 作为缓存时为什么需要 TTL 和容量治理。

## 重点

- Redis 是数据结构服务器，不只是 key-value 字符串缓存。
- 快的前提是命令复杂度和 value 大小可控。
- 高可用不等于强一致，缓存不等于永久存储。

## 难点

- Redis 的很多风险不是语法错误，而是数据模型和容量设计错误。
- “单线程”只描述核心命令执行模型的一部分，不能忽略网络 IO、后台线程、fork 和持久化。

## 易错

> **易错：** 所有数据都加到 Redis，没有 TTL、没有容量评估、没有淘汰策略。
>
> 正确做法：按业务重要性、生命周期、数据大小、访问频率和一致性要求设计 key。
