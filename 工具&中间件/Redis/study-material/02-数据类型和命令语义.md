# Redis学习资料：数据类型和命令语义

[返回索引](../Redis学习资料.md)

## 学习目标

- 掌握 String、Hash、List、Set、Sorted Set 的使用场景和命令复杂度。
- 能根据业务建模选择正确类型。
- 理解 Redis 命令是服务端数据结构操作，不是简单字符串操作。

## 理论导读

Redis 的 value 有类型。不同类型支持不同命令和复杂度。选错类型会导致代码复杂、性能差、内存浪费或无法维护。数据类型设计要同时考虑访问模式、更新模式、数据规模、过期策略和是否需要原子操作。

## String

适合缓存对象序列化值、计数器、开关、token。

```bash
SET code:phone:13800138000 123456 EX 300 NX
GET code:phone:13800138000
INCR article:100:view
MGET key1 key2 key3
```

注意：String 最大不是无限，value 过大就是 big key，会阻塞网络传输、持久化和复制。

## Hash

适合保存对象字段，局部更新。

```bash
HSET user:1 name alice age 20 city shanghai
HGET user:1 name
HMGET user:1 name age
HINCRBY user:1 login_count 1
```

边界：单个 hash 字段过多或 value 过大仍是 big key。

## List

适合简单队列、最近列表。

```bash
LPUSH queue:email job1
BRPOP queue:email 5
LRANGE list 0 9
```

边界：大范围 `LRANGE 0 -1` 风险高，队列可靠性不如 Stream。

## Set

适合去重、集合运算。

```bash
SADD uv:2026-06-06 user1 user2
SISMEMBER uv:2026-06-06 user1
SCARD uv:2026-06-06
SINTER set1 set2
```

边界：大集合交并差可能阻塞。

## Sorted Set

适合排行榜、延迟队列、按分数范围查询。

```bash
ZADD rank:daily 100 user1 80 user2
ZREVRANGE rank:daily 0 9 WITHSCORES
ZRANGEBYSCORE delay 0 1710000000000
```

边界：score 是浮点数语义，复杂范围操作和大结果返回要控制。

## 命令复杂度意识

| 命令 | 常见复杂度风险 |
| --- | --- |
| `GET`、`SET` | O(1)，但 value 大会拖慢 |
| `HGET` | O(1)，但 hash 过大有内存和迁移风险 |
| `LRANGE 0 -1` | 返回越大越慢 |
| `SMEMBERS` | 大集合风险高 |
| `ZREVRANGE` | 返回范围大则慢 |
| `KEYS` | O(N)，生产禁用思维 |

## 练习

1. 为用户资料、验证码、排行榜、关注关系分别选择数据类型。
2. 写出每个场景的 key 命名、TTL、读写命令。
3. 估算单个 key 的最大元素数量。

## 验收

- 能根据访问模式选择数据类型。
- 能说明哪些命令会因返回数据量大而阻塞。
- 能解释 big key 不只是 key 名大，而是 value 大或元素多。

## 重点

- Redis 建模先看访问模式。
- 命令复杂度和返回结果大小都影响性能。
- 数据结构不是越复杂越好，越贴合业务越好。

## 易错

> **易错：** 用 JSON String 存所有对象，任何字段更新都重写整个 value。
>
> 正确做法：字段级频繁更新用 Hash；整体缓存且一次读写可用 String。
