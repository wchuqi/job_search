# Redis学习资料：持久化 RDB、AOF 和混合持久化

[返回索引](../Redis学习资料.md)

## 学习目标

- 理解 RDB、AOF、混合持久化的机制和取舍。
- 掌握 fsync 策略、AOF rewrite、恢复流程。
- 能评估持久化对性能和数据安全的影响。

## 理论导读

Redis 是内存数据库，持久化负责把内存状态保存到磁盘。RDB 是某个时刻的快照，恢复快、文件紧凑，但可能丢失最近写入。AOF 记录写命令，数据丢失窗口更小，但文件可能更大，需要重写。混合持久化用 RDB 内容加 AOF 增量，兼顾恢复速度和数据完整性。

## RDB

```bash
BGSAVE
LASTSAVE
```

RDB 通常通过 fork 子进程生成快照。fork 后父进程继续处理请求，子进程写快照。写入变更会触发 COW，可能导致内存峰值增加。

优点：

- 文件紧凑。
- 恢复快。
- 适合备份。

缺点：

- 快照间隔内数据可能丢失。
- fork 和写盘可能造成抖动。

## AOF

```conf
appendonly yes
appendfsync everysec
```

fsync 策略：

- always：每次写 fsync，安全高，性能差。
- everysec：常用，最多约秒级丢失窗口。
- no：交给 OS，性能好，风险大。

## AOF rewrite

AOF 会增长。重写不是逐条压缩旧 AOF，而是根据当前内存状态生成等价的最短命令序列。重写期间新写入要追加到缓冲，最终合并。

## 混合持久化

混合持久化在 AOF 文件中先放 RDB 格式快照，再追加 AOF 增量。恢复时先加载 RDB 部分，再回放增量。

## 选择建议

- 纯缓存：可关闭持久化或只 RDB，但要接受重启冷启动。
- 可丢少量数据：AOF everysec。
- 需要备份：保留 RDB 备份。
- 关键数据：不要只依赖 Redis 持久化，应有主数据库或更强存储。

## 练习

1. 开启 AOF，写入数据后重启 Redis，验证恢复。
2. 执行 BGSAVE，观察 `INFO persistence`。
3. 构造大量写入，观察 AOF 文件增长和 rewrite。

## 验收

- 能比较 RDB 和 AOF。
- 能解释 fork COW 对内存峰值的影响。
- 能根据业务选择持久化策略。

## 重点

- 持久化是恢复手段，不是强一致保证。
- AOF everysec 仍可能丢最近一秒左右数据。
- fork 需要额外内存余量。

## 易错

> **易错：** 开了 AOF 就认为绝不会丢数据。
>
> 正确做法：理解 fsync 策略和故障窗口，并结合主数据库、复制和备份设计。
