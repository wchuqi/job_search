# Nginx学习资料：499、502、504、TCP 关闭和超时矩阵深度解析

[返回索引](../Nginx学习资料.md)

## 学习目标

- 深入理解 499、502、504 的时序。
- 建立客户端、Nginx、upstream 三方超时矩阵。
- 能从 error_log 和 access_log 判断断开方。

## 499

499 是 Nginx 自定义日志状态，表示客户端在 Nginx 返回响应前断开连接。原因可能是客户端超时、用户取消、移动网络断开、上游太慢或 Nginx 排队。

## 502

常见：

- upstream refused connection。
- upstream prematurely closed connection。
- invalid header。
- DNS 或 TLS 上游问题。
- upstream 进程崩溃。

## 504

Nginx 等 upstream 响应超时，常与 `proxy_read_timeout` 相关。后端可能仍在处理，只是超过 Nginx 等待时间。

## 超时矩阵

```text
client timeout < nginx timeout < upstream business timeout
```

如果客户端超时时间短于 Nginx，Nginx 可能记录 499。若 Nginx 比应用更早超时，会记录 504，但应用可能继续执行。

## 练习

1. 客户端设置短超时，后端 sleep，观察 499。
2. Nginx proxy_read_timeout 短，观察 504。
3. 后端直接关闭连接，观察 502。

## 验收

- 能解释 499 不是 Nginx 返回给客户端的标准响应。
- 能区分 502 和 504。
- 能设计三方超时矩阵。

## 易错

> **易错：** 看到 499 就认为 Nginx 有问题。
>
> 正确做法：检查客户端超时、上游耗时、网络和请求体/响应体大小。
