# Nginx学习资料：总览与心智模型

[返回索引](../Nginx学习资料.md)

## 学习目标

- 理解 Nginx 的定位：Web server、reverse proxy、load balancer、TLS terminator、stream proxy。
- 建立请求处理心智模型：连接进入、选择 server、选择 location、执行阶段、返回响应。
- 明确 Nginx 适合什么、不适合什么。

## 理论导读

Nginx 的核心价值是高并发连接处理和灵活的请求路由。它通常运行一个 master 进程和多个 worker 进程。master 管理配置加载、信号和 worker 生命周期；worker 通过事件循环处理连接和请求。配置文件不是普通脚本，而是声明式地描述不同上下文下的指令，Nginx 在启动或 reload 时解析并生成运行时配置。

Nginx 难点不是“会写几行 proxy_pass”，而是理解匹配和阶段：请求先根据监听地址和 Host 选择 server，再根据 URI 选择 location，然后进入 rewrite、access、content、log 等阶段。每个模块在不同阶段工作，指令作用域和继承规则决定最终行为。

## 核心心智模型

```text
client socket
  -> listen socket
  -> server selection
  -> URI normalization
  -> location selection
  -> rewrite/access/content phases
  -> static file or upstream
  -> filter response
  -> log
```

## Nginx 适合什么

- 静态文件服务。
- HTTPS 入口和 TLS 终止。
- 反向代理和负载均衡。
- 基础限流、访问控制、缓存。
- 四层 TCP/UDP 转发。
- 边缘路由和流量治理。

## Nginx 不适合什么

- 复杂业务逻辑。
- 复杂身份认证和授权决策。
- 动态服务发现和复杂 API 网关能力，除非配合额外模块或平台。
- 替代应用层错误处理。

## 例子

```nginx
server {
    listen 80;
    server_name example.com;

    location /static/ {
        root /var/www;
        expires 7d;
    }

    location /api/ {
        proxy_pass http://backend;
    }
}
```

## 练习

1. 配置一个静态站点。
2. 配置 `/api/` 反向代理到本地服务。
3. 用 `curl -v` 观察请求头和响应头。

## 验收

- 能画出请求处理路径。
- 能区分 server 和 location。
- 能说明 Nginx 和应用服务的责任边界。

## 重点

- Nginx 配置是声明式配置，不是顺序脚本。
- server/location 匹配决定请求落点。
- 日志是排查入口。

## 易错

> **易错：** 只看配置顺序，不理解 location 匹配优先级。
>
> 正确做法：按匹配算法推导请求命中的 location。
