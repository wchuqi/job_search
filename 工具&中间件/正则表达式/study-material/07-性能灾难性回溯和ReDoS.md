# 正则表达式学习资料：性能、灾难性回溯和 ReDoS

[返回索引](../正则表达式学习资料.md)

## 学习目标

- 识别灾难性回溯的典型结构。
- 理解 ReDoS 的攻击方式和危害。
- 掌握降低正则性能风险的改写方法。
- 能为高风险正则设置测试和运行保护。

> 深入阅读：如果你需要做生产审计或准备中高级面试，继续读 [16-ReDoS审计和复杂度分析.md](16-ReDoS审计和复杂度分析.md)，重点看风险三条件、慢失败输入构造和修复优先级。

## 理论导读

正则性能问题最危险的地方在于：短输入可能很快，恶意构造的输入却能让匹配时间指数增长。ReDoS 是 Regular Expression Denial of Service，即攻击者用特定字符串让服务端正则匹配消耗大量 CPU，导致接口阻塞或服务不可用。

灾难性回溯常来自嵌套量词、重叠分支和模糊边界。风险最高的场景是：用户可控输入、服务端同步匹配、缺少超时、表达式来自配置或用户。

## 核心心智模型

危险表达式像“有太多拆分方案的拼图”。输入前半段看起来都能匹配，直到最后一个字符失败，引擎才开始尝试所有拆分组合。

## 知识点详解

### 典型危险结构

嵌套量词：

```regex
^(a+)+$
```

重叠分支：

```regex
^(a|aa)+$
```

模糊重复：

```regex
^.*(foo|bar).*$
```

多个可选片段叠加：

```regex
^(a?){30}a{30}$
```

### 攻击输入

对 `^(a+)+$`，输入：

```text
aaaaaaaaaaaaaaaaaaaa!
```

前面的 `a` 可以被内层和外层量词拆成许多组合，最后 `!` 让整体失败，引擎开始大量回溯。

### 改写策略

使用明确边界：

```regex
^[a]+$
```

让分支互斥：

```regex
(?:ab|a)
```

优先使用字符类：

```regex
"[^"]*"
```

替代：

```regex
".*?"
```

使用占有量词或原子组，若方言支持：

```regex
a++
(?>a+)
```

### 工程防护

- 用户输入长度限制。
- 正则执行超时。
- 避免在请求主线程执行高风险正则。
- 对复杂正则添加性能测试。
- 优先使用 RE2 类引擎处理不可信输入。
- 禁止用户提交任意正则，或放到隔离环境执行。

## 例子

不推荐：

```regex
^(.+)+@example\.com$
```

推荐：

```regex
^[^@\s]+@example\.com$
```

第二个表达式明确排除了 `@` 和空白，搜索空间更小，也更贴近邮箱局部形态。

## 练习

1. 判断 `^(.*a){5}$` 是否有回溯风险。
2. 把 `^<.*>$` 改写成匹配单个简单标签的更安全写法。
3. 为 `^(a+)+$` 构造一个慢失败输入。
4. 给你常用语言写一个正则性能基准测试。

## 验收

- 能识别嵌套量词和重叠分支。
- 能解释 ReDoS 的攻击路径。
- 能用明确字符类、互斥分支、原子组或占有量词改写。
- 能提出运行时防护措施。

## 重点

- 高风险正则通常慢在“失败路径”。
- 安全正则需要限制输入长度和执行时间。
- 对不可信输入，性能是安全问题。

## 难点

- 惰性量词也可能回溯，不是性能免疫。
- 有些表达式在测试样本上很快，在攻击样本上很慢。

## 易错

> **易错：** 只测试能成功匹配的样本。
>
> 正确做法：同时测试正例、反例、边界例和恶意长输入，尤其关注失败耗时。
