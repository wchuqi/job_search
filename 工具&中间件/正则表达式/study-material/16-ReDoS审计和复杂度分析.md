# 正则表达式学习资料：ReDoS 审计和复杂度分析

[返回索引](../正则表达式学习资料.md)

## 学习目标

- 能系统识别正则中的 ReDoS 风险。
- 能区分线性、近似二次、指数级回溯风险。
- 能构造攻击样本验证慢失败。
- 能给出可落地的修复方案和工程防护。

## 理论导读

ReDoS 审计不是看到 `.*` 就判死刑，也不是看到嵌套括号就一定危险。关键要看三个条件是否同时出现：

1. 是否存在大量可选路径。
2. 这些路径是否能匹配同一段输入。
3. 是否存在一个靠后的失败点迫使引擎穷举路径。

满足这三个条件时，风险就从“写得丑”变成“可能拖垮服务”。

## 一、复杂度直觉

| 风险级别 | 典型结构 | 失败表现 |
| --- | --- | --- |
| 线性 | 明确字符类、互斥分支、固定长度 | 输入翻倍，耗时约翻倍 |
| 近似二次 | 多个无界量词串联，失败点靠后 | 输入翻倍，耗时可能翻 4 倍左右 |
| 指数级 | 嵌套量词、重叠分支、等价拆分 | 输入稍长，耗时急剧增长 |

这不是严格数学证明，而是工程审查中的判断框架。

## 二、危险模式 1：嵌套量词

危险：

```regex
^(a+)+$
```

原因：内层 `a+` 和外层 `(...)+` 都在消费同一类字符 `a`，同一输入可以被拆成大量组。

修复：

```regex
^a+$
```

如果真实需求不是纯 `a`，就把内部单元定义清楚，避免重复层级竞争同一输入。

## 三、危险模式 2：重叠分支

危险：

```regex
^(a|aa)+$
```

原因：`a` 和 `aa` 可以覆盖同一串 `a` 的不同拆法。

修复思路：

```regex
^a+$
```

或让分支互斥：

```regex
^(?:aa|b)+$
```

这里 `aa` 和 `b` 不会竞争同一字符开头。

## 四、危险模式 3：模糊前缀加靠后失败

风险表达式：

```regex
^.*foo.*bar$
```

在很长文本中没有 `bar` 或末尾不满足时，两个 `.*` 都可能调整消费范围。它通常未必指数级，但在高频请求和长输入下仍可能造成明显性能问题。

改写思路：

- 如果只是判断包含顺序，用代码的 `indexOf`/`find` 可能更清晰。
- 如果有边界，使用明确字符类。
- 如果必须正则，限制输入长度并加入测试。

## 五、危险模式 4：可选项堆叠

危险：

```regex
^(a?){30}a{30}$
```

原因：前面的 30 个 `a?` 有大量“取或不取”的组合，后面的 `a{30}` 又需要消耗同类字符。引擎可能反复调整前面可选项。

修复思路：不要用大量可选 token 模拟长度范围，改用明确量词：

```regex
^a{30,60}$
```

## 六、构造攻击样本的方法

构造慢失败输入，通常按三步：

1. 找到表达式中能重复消费的字符集合。
2. 输入大量这种字符，让前半段看似能匹配。
3. 在末尾加一个让整体失败的字符。

例子：

| 表达式 | 攻击样本 |
| --- | --- |
| `^(a+)+$` | `aaaaaaaaaaaaaaaa!` |
| `^(a|aa)+$` | `aaaaaaaaaaaaaaaa!` |
| `^(\w+\s*)+$` | 很长的单词串后加一个不允许字符 |
| `^([A-Za-z]+)*$` | 很长字母后加 `!` |

## 七、审计清单

审计生产正则时逐项检查：

- 输入是否用户可控。
- 输入长度是否有限制。
- 表达式是否运行在请求线程、事件循环或关键任务中。
- 是否有嵌套量词。
- 是否有重叠分支。
- 是否有多个 `.*` 或 `.+`。
- 是否有复杂断言在每个位置重复执行。
- 是否有反向引用。
- 是否有超时机制。
- 是否有恶意输入性能测试。
- 是否可用 parser 或普通字符串 API 替代。

## 八、修复策略优先级

1. 限制输入长度。这是最直接的安全边界。
2. 改写表达式，让分支互斥，边界明确。
3. 去掉嵌套量词和重叠分支。
4. 使用 parser 或字符串 API。
5. 在支持的引擎中使用原子组或占有量词。
6. 对不可信输入使用 RE2 类线性引擎。
7. 加入超时、隔离执行和监控。

## 九、案例：用户名校验

低风险：

```regex
^[A-Za-z_][A-Za-z0-9_]{2,31}$
```

原因：

- 起止锚点固定整体范围。
- 字符类明确。
- 长度有上限。
- 没有分支和嵌套量词。

## 十、案例：路径白名单

危险：

```regex
^/(api|api/v1|api/v1/jobs)+.*$
```

问题：

- 分支有前缀重叠。
- `+` 重复整个分支没有必要。
- 末尾 `.*` 过宽。

改写：

```regex
^/api(?:/v1(?:/jobs)?)?(?:/[A-Za-z0-9_-]+)*$
```

仍需说明：如果路径解析、URL 解码、权限判断复杂，应使用路由或 URL parser，不应只靠正则。

## 练习

1. 审计 `^([a-z]+|\d+)*$`，判断风险并构造慢失败输入。
2. 改写 `^(.+)*@example\.com$`。
3. 判断 `^[^@\s]+@[^@\s]+\.[^@\s]+$` 的性能和语义边界。
4. 为一个登录接口的正则校验设计 ReDoS 防护策略。

## 验收

- 能用三条件判断 ReDoS 风险。
- 能构造慢失败样本。
- 能给出表达式改写和运行时防护。
- 能说明为什么某个正则是线性风险较低的。

## 重点

- ReDoS 风险来自可控输入、复杂路径和靠后失败的组合。
- 修复不是只把贪婪改惰性，而是减少可选路径。
- 输入长度限制和超时是工程安全底线。

## 易错

> **易错：** 看到 `.*?` 就认为性能安全。
>
> 正确做法：检查是否仍有重叠路径、重复断言和靠后失败点。
