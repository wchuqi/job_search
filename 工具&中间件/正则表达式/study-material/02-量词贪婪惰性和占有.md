# 正则表达式学习资料：量词、贪婪、惰性和占有

[返回索引](../正则表达式学习资料.md)

## 学习目标

- 掌握 `*`、`+`、`?`、`{m}`、`{m,n}` 的含义。
- 理解贪婪量词、惰性量词和占有量词的匹配策略。
- 能解释量词为什么会引发回溯。
- 能在提取文本时控制匹配范围，避免过度匹配。

## 理论导读

量词用于描述“前一个元素重复多少次”。这个“前一个元素”可以是一个字符、字符类、分组或断言前后的组合单元。量词不是简单地数次数，它还决定引擎尝试匹配时的路径。

回溯型引擎遇到量词时，通常会先按量词策略吃掉一部分字符，再继续匹配后面的规则。如果后面的规则失败，引擎会回到量词位置，调整已经吃掉的字符数量，再尝试后续规则。

## 核心心智模型

把量词想成“伸缩夹子”：

- 贪婪量词：夹子先尽量拉长，后面失败再缩短。
- 惰性量词：夹子先尽量短，后面不够再拉长。
- 占有量词：夹子拉长后不退让，后面失败就整体失败。

## 知识点详解

### 基本量词

| 写法 | 含义 |
| --- | --- |
| `a*` | `a` 出现 0 次或多次 |
| `a+` | `a` 出现 1 次或多次 |
| `a?` | `a` 出现 0 次或 1 次 |
| `a{3}` | `a` 正好出现 3 次 |
| `a{2,5}` | `a` 出现 2 到 5 次 |
| `a{2,}` | `a` 至少出现 2 次 |

### 贪婪量词

默认量词一般是贪婪的：

```regex
".+"
```

匹配 `"a" and "b"` 时，可能从第一个引号匹配到最后一个引号，因为 `.+` 会尽量多吃字符。

### 惰性量词

在量词后加 `?` 通常变为惰性：

```regex
".+?"
```

它会尽量少吃字符，适合提取最短片段，例如从 `"a" and "b"` 中分别找出 `"a"` 和 `"b"`。

### 占有量词

部分引擎支持占有量词，例如 Java、PCRE：

```regex
\d++
```

占有量词吃掉字符后不回溯。它可减少无意义尝试，但也可能导致本来能匹配的表达式失败。

```regex
\d++\d
```

匹配 `123` 会失败，因为 `\d++` 先吃掉全部数字，不会退回一个数字给后面的 `\d`。

### 量词作用范围

```regex
ab+
```

表示 `a` 后面跟一个或多个 `b`，不是 `ab` 重复。

```regex
(ab)+
```

才表示 `ab` 这个整体重复。

## 例子

提取 HTML 中简单标签内容：

```regex
<title>.*?</title>
```

注意：这只适合简单场景，不适合完整解析 HTML。

匹配逗号分隔的字段：

```regex
[^,]+
```

这里用“不是逗号”的字符类，比 `.+?` 更稳定，因为边界明确。

## 练习

1. 写出匹配 2 到 4 位数字的正则。
2. 从 `name="alice" age="20"` 中提取每个双引号包裹的值。
3. 解释 `a.*b` 匹配 `axxbxxb` 的过程。
4. 把一个使用 `.*?` 的表达式改写成更明确的字符类写法。

## 验收

- 能说明 `*`、`+`、`?`、`{m,n}` 的边界。
- 能区分贪婪和惰性。
- 能解释占有量词为什么不回溯。
- 能为量词加上清晰边界。

## 重点

- 量词修饰的是前一个正则单元。
- 贪婪和惰性都可能回溯，只是尝试顺序不同。
- 边界明确的字符类通常比无约束的 `.*?` 更可控。

## 难点

- 惰性量词不是“不会多匹配”，它只是先尝试少匹配。
- 多个量词叠加会形成大量回溯路径。

## 易错

> **易错：** 认为 `.*?` 永远安全。
>
> 正确做法：优先用明确边界，例如 `[^"]*`、`[^,]+`、`\d{4}-\d{2}-\d{2}`。
