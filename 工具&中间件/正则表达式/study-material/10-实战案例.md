# 正则表达式学习资料：实战案例

[返回索引](../正则表达式学习资料.md)

## 学习目标

- 能把正则用于真实日志、表单、路径和文本清洗场景。
- 能根据数据边界选择合适表达式。
- 能判断某个需求是否应交给 parser、业务代码或正则。
- 能为实战正则准备正例、反例和边界样本。

## 理论导读

实战中写正则的关键不是“写得短”，而是“边界清楚、失败可控、维护容易”。一个好的工程正则通常会说明输入来源、字段含义、允许字符、失败策略和性能风险。

## 核心心智模型

实战正则遵循四步：

```text
明确输入格式 -> 确定字段边界 -> 写最小表达式 -> 用样本验证和迭代
```

## 知识点详解

### 日志解析

样本：

```text
10.0.0.1 - - [06/Jun/2026:21:10:00 +0800] "GET /api/jobs HTTP/1.1" 200 532
```

表达式：

```regex
^(?<ip>\d{1,3}(?:\.\d{1,3}){3}) \S+ \S+ \[(?<time>[^\]]+)\] "(?<method>[A-Z]+) (?<path>[^ ]+) HTTP/(?<version>[^"]+)" (?<status>\d{3}) (?<bytes>\d+|-)$
```

说明：IP 这里只匹配形态，不校验 0-255 语义。状态码和字节数字段可交给代码转换。

### 表单校验

简单用户名：

```regex
^[A-Za-z_][A-Za-z0-9_]{2,31}$
```

它表达的是：3 到 32 位，字母或下划线开头，后续允许字母、数字、下划线。

### 手机号脱敏

```regex
^(\d{3})\d{4}(\d{4})$
```

替换：

```text
$1****$2
```

### URL 局部提取

```regex
^(?<scheme>https?)://(?<host>[^/:?#]+)(?<path>/[^?#]*)?
```

完整 URL 解析应使用 URL parser。正则适合在日志中快速定位 URL 片段。

### 文本清洗

去掉行尾空白：

```regex
[ \t]+$
```

需要多行模式。不要用过宽的 `\s+$` 盲删，因为它可能影响换行结构。

## 例子

JavaScript 日志解析：

```javascript
const re = /^(?<level>INFO|WARN|ERROR) (?<time>\d{2}:\d{2}:\d{2}) (?<message>.+)$/;
const line = "ERROR 21:30:01 job failed";
const match = re.exec(line);

if (match) {
  console.log(match.groups.level, match.groups.message);
}
```

## 练习

1. 解析 `level=INFO cost=12ms path=/api/jobs`。
2. 写一个用户名校验正则，并列出 5 个合法样本和 5 个非法样本。
3. 将 `2026/06/06` 替换成 `2026-06-06`。
4. 判断解析 HTML 表格是否适合用正则，并说明边界。

## 验收

- 能根据样本写出字段提取正则。
- 能说明哪些校验由正则负责，哪些由代码负责。
- 能准备正例、反例、边界例。
- 能识别 parser 更合适的场景。

## 重点

- 实战正则要服务于输入格式和下游字段。
- 复杂结构优先用 parser。
- 正则样本集应包含失败样本。

## 难点

- 真实数据常有脏样本、缺失字段和格式漂移。
- 过度宽松会误匹配，过度严格会漏掉合法数据。

## 易错

> **易错：** 用一个表达式同时完成格式识别、业务校验、数据转换和异常处理。
>
> 正确做法：正则负责定位和初步形态，业务规则拆到代码中明确处理。
