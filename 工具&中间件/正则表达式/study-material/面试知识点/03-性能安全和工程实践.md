## 一、性能、安全和工程实践

### 1. 什么是灾难性回溯？

**参考答案：**

灾难性回溯是指回溯型正则引擎在某些表达式和输入组合下产生指数级尝试，导致匹配耗时急剧增加。典型例子是 `^(a+)+$` 遇到 `aaaaaaaaaaaa!`。前面的 `a` 可以被嵌套量词拆成大量组合，最后的 `!` 导致失败，引擎开始尝试许多拆分路径。

> **重点：** 灾难性回溯往往慢在失败路径。
>
> **易错：** 只测试成功匹配样本，看不出风险。

### 2. 什么是 ReDoS？如何防护？

**参考答案：**

ReDoS 是 Regular Expression Denial of Service，攻击者通过构造输入让服务端正则消耗大量 CPU，从而造成接口阻塞或服务不可用。

防护方式包括：限制输入长度、设置正则执行超时、避免嵌套量词和重叠分支、使用明确字符类、使用占有量词或原子组、对不可信输入使用 RE2 类线性引擎、禁止用户提交任意正则或隔离执行。

> **重点：** 对不可信输入，正则性能就是安全问题。

### 3. 如何改写一个有性能风险的正则？

**参考答案：**

常见策略是缩小搜索空间：用明确字符类替代 `.*`，让分支互斥，去掉嵌套量词，增加锚点，使用占有量词或原子组。

例如提取双引号内容，不推荐：

```regex
".*?"
```

更推荐：

```regex
"[^"]*"
```

因为后者明确表示引号内不能出现双引号，失败路径更可控。

> **难点：** 惰性量词也可能回溯，关键是边界是否明确。

### 4. 如何为正则编写测试？

**参考答案：**

测试至少包含正例、反例、边界例、脏数据和性能例。正例证明应匹配的能匹配；反例证明不该匹配的不会误匹配；边界例覆盖最短、最长、空值、特殊字符；性能例覆盖长输入和慢失败输入。

生产正则还应测试捕获组字段、替换结果和目标语言 API 行为。

> **重点：** 正则也是代码，需要回归测试。

### 5. Java、JavaScript、Python 的正则 API 有哪些容易混淆的地方？

**参考答案：**

Java 中 `matches()` 要求整体匹配，`find()` 查找片段。Python 中 `match()` 从开头匹配，`search()` 查找任意位置，`fullmatch()` 整体匹配。JavaScript 的 `test()` 判断是否存在匹配，带 `g` 标志的正则对象会受 `lastIndex` 状态影响。

这些 API 差异会导致同一个表达式在不同代码中呈现不同结果。

> **易错：** 用 API 行为误判正则语法行为。

### 6. 什么时候不应该使用正则？

**参考答案：**

当输入是复杂嵌套语言、标准格式已有成熟 parser、规则依赖外部状态、语义校验复杂或表达式难以维护时，不应强行使用正则。例如完整 HTML、JSON、CSV、URL 解析应优先使用 parser；身份证、日期、业务编码校验应把语义部分交给代码。

> **重点：** 选择 parser 不是正则能力不足，而是工程边界更清晰。
