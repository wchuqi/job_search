# 正则表达式学习资料：正则引擎机制、NFA、DFA 和回溯

[返回索引](../正则表达式学习资料.md)

## 学习目标

- 理解正则引擎不是只有一种实现。
- 掌握回溯型 NFA 的匹配过程和选择点。
- 了解 DFA/RE2 类线性引擎的优势和限制。
- 能用“尝试顺序”解释匹配结果和性能问题。

> 深入阅读：如果你已经理解本章概念，继续读 [15-引擎执行轨迹深度解析.md](15-引擎执行轨迹深度解析.md)，那里会逐步推演 `a.*b`、分支顺序、捕获组回溯和 `^(a+)+$` 的拆分爆炸。

## 理论导读

正则表达式写出来后，需要由正则引擎执行。常见工程语言中的正则大多是回溯型 NFA，例如 Java、JavaScript、Python `re`、PCRE、.NET。它们表达能力强，支持捕获、反向引用、环视等特性，但复杂表达式可能产生大量回溯。

另一类引擎偏向 DFA 或自动机实现，例如 RE2。它们通常保证线性时间，避免灾难性回溯，但会放弃反向引用、部分环视等难以线性化的能力。

## 核心心智模型

回溯型引擎像“走迷宫”：

```text
遇到量词或分支 -> 记录岔路口
继续向前走     -> 后续匹配成功就结束
后续失败       -> 回到最近岔路口换一条路
岔路耗尽       -> 整体失败
```

## 知识点详解

### 匹配起点选择

如果没有 `^` 或 API 要求从头匹配，引擎可能从字符串每个位置尝试一次：

```regex
abc
```

在 `xxabc` 中会从 `x`、第二个 `x`、`a` 依次尝试，直到成功。

### 分支选择顺序

多数回溯型引擎按从左到右尝试：

```regex
cat|catalog
```

匹配 `catalog` 时，可能先得到 `cat`，因为左侧分支已经成功。若想匹配长词，应写：

```regex
catalog|cat
```

### 量词选择点

```regex
a.*b
```

匹配 `axxbxxb`：

1. `a` 匹配开头。
2. `.*` 贪婪吃到末尾。
3. 后面的 `b` 失败，因为已经没有字符。
4. `.*` 回退一个字符。
5. 后面的 `b` 成功。

### 回溯状态

回溯状态一般包含：当前正则 token、输入位置、捕获组状态、量词已重复次数、分支选择点。表达式越模糊，状态越多。

### DFA/RE2 类引擎

RE2 类引擎倾向于用自动机状态集合推进输入，避免指数级回溯。代价是不能支持某些需要“记住任意文本再比较”的特性，例如反向引用。

| 能力 | 回溯型 NFA | RE2 类线性引擎 |
| --- | --- | --- |
| 表达能力 | 强 | 较受限 |
| 性能上界 | 可能很差 | 通常线性 |
| 反向引用 | 常支持 | 通常不支持 |
| 环视 | 常支持 | 通常不支持或受限 |
| ReDoS 风险 | 高 | 低 |

## 例子

重叠分支：

```regex
(a|aa)+b
```

输入 `aaaaaaaaac` 时，分支可以把一串 `a` 拆成许多组合，最后 `b` 失败，引擎需要回溯大量路径。

## 练习

1. 手动写出 `a.*b` 匹配 `a123b456b` 的回溯过程。
2. 判断 `foo|foobar` 在 `foobar` 上可能得到什么结果。
3. 说明为什么 `^(a+)+$` 遇到末尾非 `a` 字符很危险。
4. 查出你常用语言的正则引擎是否是回溯型。

## 验收

- 能解释选择点和回溯。
- 能说明分支顺序对结果的影响。
- 能比较回溯型引擎和线性引擎。
- 能从引擎角度分析一个正则的风险。

## 重点

- 正则匹配结果受引擎尝试顺序影响。
- 回溯型引擎表达能力强，但需要性能意识。
- 锚点、明确字符类、互斥分支能减少搜索空间。

## 难点

- 捕获组状态也会随回溯改变。
- “能匹配”与“快速失败”是两回事。

## 易错

> **易错：** 只从语义角度看正则，不考虑引擎执行路径。
>
> 正确做法：对复杂正则画出量词、分支和失败时的回溯路径。
