# 正则表达式学习资料：调试、测试和可维护性

[返回索引](../正则表达式学习资料.md)

## 学习目标

- 掌握正则调试的拆分方法。
- 能为正则编写正例、反例、边界例和性能测试。
- 能用命名组、注释模式和分层表达式提高可维护性。
- 能在团队协作中管理正则变更风险。

> 深入阅读：生产环境中的正则评审、测试样本模板和案例改写，继续读 [18-正则设计评审清单和案例库.md](18-正则设计评审清单和案例库.md)。

## 理论导读

正则通常短而密，维护难度高。调试正则时不要只盯着最终表达式，应把它拆成 token、分组、边界、量词和断言逐段验证。团队项目中，正则应像普通代码一样有命名、有测试、有变更说明。

## 核心心智模型

维护正则要把它当成一段“小程序”：

```text
输入 -> 匹配路径 -> 捕获输出 -> 失败路径 -> 性能边界
```

只要它影响业务结果，就需要测试。

## 知识点详解

### 调试步骤

1. 明确 API 是查找、从头匹配还是整体匹配。
2. 给表达式加锚点或移除锚点，确认匹配范围。
3. 从左到右逐段运行。
4. 暂时去掉复杂断言，确认主体匹配。
5. 打印捕获组结果。
6. 用最短失败样本定位边界问题。

### 测试样本分类

| 类型 | 目的 |
| --- | --- |
| 正例 | 应该匹配 |
| 反例 | 不应该匹配 |
| 边界例 | 最短、最长、空值、特殊字符 |
| 脏数据 | 真实系统里的不规范输入 |
| 性能例 | 长输入、恶意输入、慢失败输入 |

### 可维护写法

使用命名组：

```regex
^(?<date>\d{4}-\d{2}-\d{2}) (?<level>INFO|WARN|ERROR) (?<message>.+)$
```

使用扩展模式，若方言支持：

```regex
(?x)
^
(?<date>\d{4}-\d{2}-\d{2}) \s+
(?<level>INFO|WARN|ERROR) \s+
(?<message>.+)
$
```

### 版本和变更

正则变更要关注：

- 新增合法样本是否通过。
- 旧合法样本是否仍通过。
- 旧非法样本是否仍失败。
- 捕获组编号或名称是否变化。
- 性能是否退化。

## 例子

Python 单元测试思路：

```python
import re

pattern = re.compile(r"^[A-Za-z_][A-Za-z0-9_]{2,31}$")

valid = ["abc", "_user1", "Alice_2026"]
invalid = ["1abc", "ab", "a-b", "x" * 33]

for value in valid:
    assert pattern.fullmatch(value)

for value in invalid:
    assert not pattern.fullmatch(value)
```

## 练习

1. 为用户名正则设计 10 个测试样本。
2. 把一个复杂日志正则拆成多个命名片段。
3. 给 `^(a+)+$` 写一个性能测试样本。
4. 解释为什么捕获组编号变化是兼容性风险。

## 验收

- 能系统定位正则不匹配的原因。
- 能写完整样本集。
- 能用命名组和注释模式提升可读性。
- 能说明正则变更的回归风险。

## 重点

- 正则也是代码，需要测试。
- 正例之外必须有反例和边界例。
- 命名组比编号组更适合长期维护。

## 难点

- 部分失败来自 API 语义，而不是表达式本身。
- 性能退化常发生在失败路径，不容易被普通正例发现。

## 易错

> **易错：** 在线测试器通过后就直接上线。
>
> 正确做法：在目标语言、目标运行时和真实 API 中测试，并覆盖失败和性能样本。
