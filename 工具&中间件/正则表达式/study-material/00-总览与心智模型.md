# 正则表达式学习资料：总览与心智模型

[返回索引](../正则表达式学习资料.md)

## 学习目标

- 理解正则表达式的定位、能力边界和常见使用场景。
- 建立“模式匹配”和“引擎尝试路径”的心智模型。
- 区分匹配、查找、提取、替换、分割和校验。
- 知道为什么正则会有性能风险和方言差异。

## 理论导读

正则表达式是一种描述字符串模式的语言。它擅长处理“局部、规则、线性”的文本结构，例如提取日志字段、校验简单格式、替换重复空白、识别 token。它不擅长完整解析嵌套语言、复杂语义校验和无限层级结构，例如完整 HTML、完整 JSON、任意嵌套括号。

正则的难点不在于符号多，而在于这些符号组合后会形成匹配路径。回溯型引擎会不断尝试、失败、回退、再尝试。理解这个过程，才能写出既正确又安全的正则。

## 核心心智模型

把正则想成一个“文本扫描器”：

```text
输入字符串
  -> 从某个位置开始尝试
  -> 按正则 token 逐步匹配
  -> 遇到分支或量词时保存选择点
  -> 后续失败时回溯到选择点
  -> 找到整体匹配或最终失败
```

例子：

```regex
a.*b
```

匹配 `axxxbxxb` 时，`.*` 会先尽可能吃到最后，再为了让后面的 `b` 匹配而回退。这就是贪婪量词和回溯。

## 知识点详解

### 正则常见用途

| 用途 | 例子 |
| --- | --- |
| 匹配 | 判断字符串是否像日期 |
| 查找 | 找出文本中所有邮箱 |
| 提取 | 从日志中提取 IP 和状态码 |
| 替换 | 把多个空格替换成一个空格 |
| 分割 | 按逗号和空白拆分字段 |
| 清洗 | 删除不可见字符 |

### 正则不是万能解析器

正则适合规则局部匹配，不适合完整语法树解析。比如：

- 简单提取 `<title>...</title>` 可以。
- 完整解析 HTML 不适合。
- 简单校验邮箱形态可以。
- 完整判断邮箱是否真实存在不可以。
- 简单匹配 JSON 中某个固定字段可以。
- 完整解析 JSON 应使用 JSON parser。

### 正则方言

不同语言正则不完全一样：

- JavaScript 支持 lookbehind 的现代环境才可用。
- Java 支持占有量词和原子组。
- Python 标准 `re` 不支持所有 PCRE 特性。
- PCRE 功能强，但也更容易写出复杂表达式。
- RE2 这类引擎避免灾难性回溯，但不支持某些回溯特性。

## 例子

日期形态：

```regex
^\d{4}-\d{2}-\d{2}$
```

它只能保证形态像 `2026-06-06`，不能保证 `2026-99-99` 是真实日期。真实日期校验应交给日期库。

## 练习

1. 写一个匹配日期形态的正则。
2. 写一个提取日志状态码的正则。
3. 找一个你写过的复杂正则，说明它是否适合用 parser 替代。
4. 找出一个可能有性能风险的正则。

## 验收

- 能说明正则适合和不适合的场景。
- 能解释回溯的基本含义。
- 能区分匹配、提取、替换和校验。
- 能说出至少 3 个正则方言差异。

## 重点

- 正则描述字符串模式，不等于完整语义校验。
- 回溯型引擎会尝试多条路径。
- 工程中必须关注可读性、测试和性能。

## 难点

- 同一个正则在不同语言中行为可能不同。
- 看起来短的正则也可能有严重性能风险。

## 易错

> **易错：** 用一个巨大正则校验所有业务规则。
>
> 正确做法：正则负责形态，复杂语义交给代码或专用 parser。
