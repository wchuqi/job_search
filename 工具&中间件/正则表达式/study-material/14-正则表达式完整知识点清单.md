## 必须掌握

- 正则表达式用于描述字符串模式，不等于完整业务语义校验。
- 字符类 `[]` 匹配单个字符，量词修饰前一个正则单元。
- 贪婪和惰性都可能回溯，占有量词通常不回溯。
- 捕获组会产生输出和编号，结构分组应优先用非捕获组。
- 断言只检查位置上下文，不消耗字符。
- 分支通常按从左到右尝试，分支顺序会影响结果。
- 回溯型引擎可能出现灾难性回溯。
- 正则方言和 API 行为必须按目标语言验证。
- 复杂正则要能画出成功路径和慢失败路径。
- ReDoS 审计要同时看可控输入、大量路径和靠后失败点。
- 生产正则需要写清输入来源、最大长度、目标引擎、测试样本和已知边界。

## 进阶和容易遗漏

- `^`、`$` 在多行模式下的语义变化。
- `$` 在部分引擎中可匹配结尾换行前。
- `.` 默认通常不匹配换行，dotAll 才改变。
- `\w`、`\b` 和大小写忽略在 Unicode 文本中有差异。
- JavaScript 带 `g` 的正则对象有 `lastIndex` 状态。
- Python `match` 不是任意位置查找，`fullmatch` 才是整体匹配。
- 替换中的 `$1`、`\1`、`\g<name>` 依语言不同。
- lookbehind 在不同引擎中支持差异明显。
- 反向引用会让表达能力增强，但也会带来性能和兼容性限制。
- 不可信输入应限制长度、设置超时或使用线性引擎。
- 安全白名单正则必须放在 URL 解码、路径标准化和权限模型上下文中评审。
- 迁移正则时要检查源码字符串、引擎能力和 API 语义三层，而不是只复制表达式。

## 常见错误清单

| 错误 | 后果 | 正确做法 |
| --- | --- | --- |
| 用 `\d{6}` 做全量验证码校验 | 包含 6 位数字的任意字符串都可能通过 | 使用 `^\d{6}$` 或 fullmatch API |
| 用 `.*` 提取引号内容 | 过度匹配 | 使用 `"[^"]*"` |
| 用复杂正则解析完整 HTML/JSON/CSV | 边界错误、维护困难 | 使用 parser |
| 随意添加捕获组 | 破坏组编号 | 不需要输出时用 `(?:...)` |
| 只测正例 | 漏掉失败路径和性能风险 | 增加反例、边界例、恶意输入 |
| 忽略方言差异 | 线上行为和测试器不同 | 在目标语言和目标 API 中测试 |
