# 正则表达式学习资料：正则方言深度对照

[返回索引](../正则表达式学习资料.md)

## 学习目标

- 掌握正则迁移时真正容易出错的差异层。
- 能比较 Java、JavaScript、Python、PCRE、.NET、RE2 的关键能力。
- 能解释字符串字面量、正则编译和 API 行为如何共同影响结果。
- 能制定跨语言迁移检查表。

## 理论导读

正则跨语言失败，常常不是因为核心语法差异，而是因为“看起来一样的代码走了不同路径”。例如 `\d+` 在 Java 字符串中要写 `"\\d+"`，Python `match` 不是任意位置查找，JavaScript 的全局正则对象可能携带 `lastIndex` 状态。

因此，方言差异要分三层看：

```text
源码层：字符串如何转义
引擎层：正则语法支持什么
API 层：调用方法如何定义匹配范围和状态
```

## 一、源码层：字符串字面量

| 目标 | 推荐写法 | 说明 |
| --- | --- | --- |
| Java | `"\\d+"` | Java 字符串先把 `\\` 变成 `\` |
| JavaScript 字面量 | `/\d+/` | 正则字面量中直接写正则转义 |
| JavaScript 构造器 | `new RegExp("\\d+")` | 字符串构造器需要双重转义 |
| Python | `r"\d+"` | 原始字符串减少转义混乱 |
| JSON 配置 | `"\\d+"` | JSON 字符串也需要转义反斜杠 |

> **易错：** 配置文件、数据库、源码三处传递同一个正则时，反斜杠可能被转义多次。

## 二、引擎层：能力差异

| 特性 | Java | JavaScript | Python `re` | PCRE | .NET | RE2 |
| --- | --- | --- | --- | --- | --- | --- |
| 捕获组 | 支持 | 支持 | 支持 | 支持 | 支持 | 支持 |
| 命名组 | `(?<n>...)` | `(?<n>...)` | `(?P<n>...)` | 多种 | `(?<n>...)` | 依封装而定 |
| 非捕获组 | 支持 | 支持 | 支持 | 支持 | 支持 | 支持 |
| 反向引用 | 支持 | 支持 | 支持 | 支持 | 支持 | 通常不支持 |
| lookahead | 支持 | 支持 | 支持 | 支持 | 支持 | 通常不支持 |
| lookbehind | 支持 | 现代环境支持 | 多数要求固定长度 | 支持较强 | 支持较强 | 通常不支持 |
| 占有量词 | 支持 | 通常不支持 | 标准 `re` 不支持 | 支持 | 不同写法/能力 | 不支持 |
| 原子组 | 支持 | 通常不支持 | 版本相关 | 支持 | 支持 | 不支持 |
| 条件分支 | 不常用/有限 | 通常不支持 | 支持部分 | 支持 | 支持 | 不支持 |

结论：如果表达式依赖反向引用、复杂环视或原子组，不要默认能迁移到 RE2、JavaScript 或 Python 标准 `re`。

## 三、API 层：匹配范围

### Java

- `matches()`：整个字符串必须匹配。
- `find()`：查找下一个匹配片段。
- `lookingAt()`：从开头匹配，但不要求整体。

### Python

- `re.match()`：从字符串开头尝试。
- `re.search()`：任意位置查找。
- `re.fullmatch()`：整体匹配。
- `findall()`：返回所有匹配；有捕获组时返回组内容，行为容易让初学者误解。

### JavaScript

- `test()`：判断是否能找到匹配。
- `exec()`：返回匹配详情。
- `match()`：行为受 `g` 标志影响。
- 带 `g` 或 `y` 的 RegExp 对象会维护 `lastIndex`。

示例：

```javascript
const re = /\d/g;
console.log(re.test("1")); // true
console.log(re.test("1")); // false，lastIndex 已到末尾
```

> **重点：** API 状态可能让同一个表达式连续调用结果不同。

## 四、Unicode 差异

Unicode 是正则中最容易被低估的部分：

- `\w` 是否只包含 ASCII 字母数字下划线，依引擎和标志而定。
- `\b` 依赖“单词字符”定义，不适合做中文分词。
- 忽略大小写涉及大小写折叠，不同语言可能处理不同。
- Unicode 属性类如 `\p{L}`、`\p{Script=Han}` 支持差异明显。

对于多语言文本，推荐策略：

1. 明确业务允许字符集合。
2. 优先使用语言/库提供的 Unicode 属性支持。
3. 不要用 `\w+` 直接代表“用户名”或“单词”。
4. 中文分词、自然语言处理交给专门工具。

## 五、替换语法差异

| 语言 | 编号引用 | 命名引用 |
| --- | --- | --- |
| JavaScript | `$1` | `$<name>` |
| Java | `$1` | `${name}` |
| Python | `\1` 或 `\g<1>` | `\g<name>` |
| .NET | `$1` | `${name}` |

注意：匹配表达式里的 `\1` 是反向引用；替换模板里的 `$1` 或 `\1` 是替换引用。它们处在不同语境中。

## 六、迁移检查表

迁移正则前逐项确认：

- 原表达式来自哪个引擎。
- 目标语言字符串是否需要双重转义。
- 目标引擎是否支持命名组、lookbehind、反向引用、原子组、占有量词。
- 目标 API 是查找、从头匹配还是整体匹配。
- 模式标志是否等价，例如 `m`、`s`、`u`、`x`。
- 替换引用语法是否变化。
- Unicode 行为是否变化。
- 是否有性能保护，尤其从 RE2 迁到回溯型引擎时。

## 七、案例：从 JavaScript 迁移到 Java

JavaScript：

```javascript
const re = /^(?<year>\d{4})-(?<month>\d{2})-(?<day>\d{2})$/;
```

Java：

```java
Pattern re = Pattern.compile("^(?<year>\\d{4})-(?<month>\\d{2})-(?<day>\\d{2})$");
```

变化：

- `\d` 在 Java 字符串中写成 `\\d`。
- 命名组语法相同，但读取 API 不同。
- Java `matcher.matches()` 是整体匹配；`find()` 是查找。

## 练习

1. 把 Python `r"(?P<name>\w+)"` 迁移到 JavaScript。
2. 解释 JavaScript `/\d/g.test("1")` 连续调用结果变化的原因。
3. 判断一个使用 lookbehind 的表达式能否迁移到 RE2。
4. 比较 Python `findall()` 在有无捕获组时的返回值差异。

## 验收

- 能从源码层、引擎层、API 层解释差异。
- 能列出迁移检查项。
- 能指出 Unicode、替换引用和全局状态风险。
- 能把一个简单表达式迁移到至少两种语言。

## 重点

- 正则迁移必须在目标运行时测试。
- API 行为经常比语法差异更隐蔽。
- Unicode 和替换引用是跨语言高频坑。
