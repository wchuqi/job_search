# 正则表达式学习资料：断言和环视

[返回索引](../正则表达式学习资料.md)

## 学习目标

- 理解断言“检查条件但不消耗字符”的机制。
- 掌握正向先行、负向先行、正向后行、负向后行。
- 能使用断言表达上下文约束。
- 了解 lookbehind 的方言限制和性能影响。

## 理论导读

断言用于判断当前位置前后是否满足某种模式。它像门卫，只检查条件，不把字符带走。断言适合“匹配 A，但要求 A 前后满足 B”的场景，例如匹配后面跟单位的数字、排除某些后缀、限制密码必须包含某类字符。

断言增强了表达能力，也增加了阅读和调试成本。复杂断言组合通常需要测试覆盖，否则容易出现看起来正确但边界失败的表达式。

## 核心心智模型

当前位置是一根光标：

```text
price: 128元
       ^
```

断言会从光标位置向前或向后偷看，确认条件成立后，光标仍停在原位。

## 知识点详解

### 正向先行断言

```regex
\d+(?=元)
```

匹配后面紧跟 `元` 的数字，但结果不包含 `元`。

### 负向先行断言

```regex
foo(?!bar)
```

匹配后面不是 `bar` 的 `foo`。

### 正向后行断言

```regex
(?<=￥)\d+
```

匹配前面是 `￥` 的数字，但结果不包含 `￥`。

### 负向后行断言

```regex
(?<!-)\d+
```

匹配前面不是 `-` 的数字。实际使用时要注意数字内部位置也可能被匹配，通常需要配合边界。

### 密码校验中的断言

```regex
^(?=.*[A-Z])(?=.*\d)[A-Za-z\d]{8,20}$
```

含义：

- `^` 从开头开始。
- `(?=.*[A-Z])` 要求某处有大写字母。
- `(?=.*\d)` 要求某处有数字。
- `[A-Za-z\d]{8,20}` 限制整体字符集和长度。

这种写法的核心是：前面的断言负责“必须包含”，最后的主体负责“只能包含”和长度。

### 方言限制

lookahead 通常支持较广；lookbehind 支持差异更大。部分引擎要求后行断言长度固定，部分现代引擎支持有限可变长度，RE2 不支持回溯型 lookaround 特性。

## 例子

匹配不是 `.tmp` 结尾的文件名：

```regex
^(?!.*\.tmp$).+
```

匹配被括号包裹的内容：

```regex
(?<=\()[^)]+(?=\))
```

## 练习

1. 匹配后面跟 `%` 的数字，但不包含 `%`。
2. 匹配不以 `test_` 开头的标识符。
3. 写一个密码正则：8 到 16 位，至少一个字母和一个数字。
4. 说明为什么复杂密码规则不一定适合全部写进正则。

## 验收

- 能解释断言不消耗字符。
- 能区分先行和后行、正向和负向。
- 能用断言表达包含条件和排除条件。
- 能说出 lookbehind 的兼容性风险。

## 重点

- 断言匹配的是条件，不是结果文本。
- 多个先行断言常用于叠加全局约束。
- 断言要配合主体表达式，否则可能只检查条件但没有限制整体。

## 难点

- 负向断言的作用位置非常关键，位置错了就会漏匹配或误匹配。
- lookbehind 的支持和长度限制在不同引擎中差异明显。

## 易错

> **易错：** 用 `(?=.*\d)` 后就以为整体只能包含数字。
>
> 正确做法：断言只检查“存在数字”，还要用主体表达式限制允许字符和长度。
