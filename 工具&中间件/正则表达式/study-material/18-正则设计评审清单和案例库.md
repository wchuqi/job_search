# 正则表达式学习资料：正则设计评审清单和案例库

[返回索引](../正则表达式学习资料.md)

## 学习目标

- 能像代码评审一样评审正则表达式。
- 能从正确性、性能、安全、可维护性和跨语言兼容性五个维度分析。
- 能通过案例学习如何从粗糙正则演进到工程可用正则。
- 能为生产正则编写说明、测试和变更策略。

## 理论导读

工程里的正则很少是孤立存在的。它通常嵌在表单校验、日志解析、网关规则、爬虫清洗、风控规则或配置系统里。评审一个正则，不能只问“能不能匹配样本”，还要问：

- 它会不会误伤合法输入？
- 它会不会放过非法输入？
- 它在恶意输入下是否安全？
- 下一个维护者能不能看懂？
- 换语言或运行时后是否还能工作？

## 一、正则设计文档模板

生产正则建议至少写清楚：

```text
名称：
用途：
输入来源：
是否用户可控：
最大输入长度：
目标语言和引擎：
匹配 API：
表达式：
捕获字段：
正例：
反例：
边界例：
性能样本：
已知不处理的场景：
```

这不是形式主义。它能把隐藏假设显式化，避免后续维护时把正则当成黑盒。

## 二、评审清单

### 正确性

- 是否使用了正确 API：查找、从头匹配、整体匹配。
- 是否需要 `^`、`$` 或 `fullmatch`。
- 字符类是否过宽，例如 `.`、`\w`、`\s`。
- 是否明确处理空字符串、换行、首尾空白。
- 捕获组字段是否稳定。
- 是否遗漏 Unicode、大小写、全角半角等输入。

### 性能

- 是否有嵌套量词。
- 是否有重叠分支。
- 是否有多个 `.*`。
- 是否有无界 lookaround。
- 是否有靠后失败点。
- 是否限制输入长度。

### 安全

- 输入是否用户可控。
- 是否运行在服务端请求路径。
- 是否可能 ReDoS。
- 是否用于权限、路由、白名单或安全过滤。
- 是否在 URL 解码、大小写归一、路径标准化之前使用。

### 可维护性

- 是否可以拆分成命名片段。
- 是否应使用命名组。
- 是否需要 verbose/comment 模式。
- 是否有测试用例。
- 是否说明“不处理哪些情况”。

### 兼容性

- 是否依赖特定方言。
- 是否依赖 lookbehind、原子组、占有量词、反向引用。
- 替换引用语法是否确认。
- Unicode 标志和模式修饰符是否确认。

## 三、案例 1：邮箱形态校验

过度复杂版本不一定更好。完整 RFC 邮箱正则通常难读难维护，也不能证明邮箱真实存在。

工程常用形态校验：

```regex
^[^@\s]+@[^@\s]+\.[^@\s]+$
```

它的边界：

- 能排除空白和缺少 `@`、`.` 的明显错误。
- 不能覆盖所有合法邮箱。
- 不能判断域名是否存在。
- 不能判断邮箱是否可收信。

更稳妥的工程流程：

1. 前端用简单形态校验提供即时反馈。
2. 后端重复校验并限制长度。
3. 发送验证邮件确认真实可用。
4. 存储前做规范化策略，例如是否保留大小写。

> **重点：** 邮箱正则的目标通常是“低成本排除明显错误”，不是“证明邮箱有效”。

## 四、案例 2：URL 判断

危险倾向：

```regex
^https?://.+$
```

问题：

- `.+` 允许空格、控制字符和危险结构。
- 无法处理 URL 编码、用户名密码、端口、IPv6、国际化域名。
- 如果用于安全白名单，容易绕过。

建议：

- 完整解析使用语言 URL parser。
- 正则只做非常粗的前置判断。
- 安全判断必须在解析、规范化之后做。

粗校验：

```regex
^https?://[^\s/$.?#].[^\s]*$
```

仍需说明：这不是安全 URL 白名单。

## 五、案例 3：日志字段提取

样本：

```text
level=INFO user=alice cost=20ms path=/api/jobs
```

脆弱写法：

```regex
level=(.*) user=(.*) cost=(.*) path=(.*)
```

问题：

- `.*` 过宽。
- 字段边界不明确。
- 捕获结果容易吞掉后续字段。

改进：

```regex
^level=(?<level>INFO|WARN|ERROR) user=(?<user>\S+) cost=(?<cost>\d+)ms path=(?<path>/\S*)$
```

继续改进的方向：

- 如果字段顺序会变化，正则应改为逐个 key-value 提取，再由代码组装。
- 如果 value 允许引号和转义，应设计词法规则或使用 parser。

## 六、案例 4：密码规则

常见写法：

```regex
^(?=.*[A-Za-z])(?=.*\d)[A-Za-z\d@$!%*#?&]{8,20}$
```

拆解：

- `(?=.*[A-Za-z])`：至少一个字母。
- `(?=.*\d)`：至少一个数字。
- `[A-Za-z\d@$!%*#?&]{8,20}`：整体只能由这些字符组成，长度 8 到 20。

评审点：

- `.*` 在断言中会扫描字符串，长度必须有限制。
- 规则复杂后，代码校验可能更清晰。
- 密码策略还涉及泄露密码库、限流、多因素认证，不是正则能解决。

## 七、案例 5：路径白名单

不安全思路：

```regex
^/admin/.*
```

如果用于权限判断，可能被编码、路径归一化、大小写规则、反斜杠、重复斜杠绕过。

正确流程：

1. URL 解码。
2. 路径标准化。
3. 使用路由系统或权限模型判断。
4. 正则只作为辅助，而不是唯一安全边界。

## 八、正则测试样本模板

```text
正例：
- 最短合法值
- 最长合法值
- 常见合法值
- 包含边界字符的合法值

反例：
- 空字符串
- 超短
- 超长
- 非法字符
- 缺少必要字段
- 多余字段

性能例：
- 很长但合法
- 很长且末尾失败
- 重复结构输入
```

## 练习

1. 为手机号脱敏正则写设计文档。
2. 评审 `^(\w+\.)*\w+@\w+(\.\w+)+$`。
3. 把一个使用 `.*` 的日志提取正则改成字段边界明确的写法。
4. 为路径白名单设计“不依赖正则单点防护”的方案。

## 验收

- 能从五个维度评审正则。
- 能写出生产正则说明。
- 能指出一个正则的已知边界。
- 能用案例说明为什么 parser 或代码有时更合适。

## 重点

- 生产正则必须有上下文、测试和边界说明。
- 安全场景不能只靠正则。
- 可维护性来自命名、拆分、注释和测试。

## 易错

> **易错：** 评审正则时只看正例是否匹配。
>
> 正确做法：同时审查反例、边界例、性能例、运行 API 和安全上下文。
