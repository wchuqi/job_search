# 正则表达式学习资料：引擎执行轨迹深度解析

[返回索引](../正则表达式学习资料.md)

## 学习目标

- 能像调试程序一样跟踪正则引擎的执行过程。
- 理解匹配起点、token 推进、选择点、回溯栈和捕获组状态。
- 能解释为什么两个语义相近的表达式性能差异很大。
- 能通过执行轨迹定位误匹配、漏匹配和慢失败。

## 理论导读

正则表达式不是“从左到右扫一遍这么简单”。对回溯型引擎来说，正则更像一段带分支、循环和局部变量的小程序。字符类是条件判断，量词是循环，分支是 if/else，捕获组是临时变量，断言是只读检查，回溯栈记录尚未尝试的路径。

理解执行轨迹有两个价值：

- 正确性：知道为什么匹配到了不该匹配的内容。
- 性能：知道为什么失败时尝试次数爆炸。

## 一、回溯型引擎的状态模型

一次匹配至少包含这些状态：

| 状态 | 含义 |
| --- | --- |
| 输入位置 | 当前读到字符串哪个索引 |
| 正则位置 | 当前执行到哪个 token |
| 捕获组内容 | 每个捕获组当前保存的文本 |
| 量词计数 | 当前量词已经重复了几次 |
| 选择点 | 量词和分支留下的可回退位置 |
| 模式标志 | 是否多行、忽略大小写、dotAll 等 |

当后续 token 失败时，引擎会弹出最近的选择点，把输入位置、正则位置、捕获组状态恢复到那个选择点，再换一种方案继续。

## 二、逐步轨迹：`a.*b`

表达式：

```regex
a.*b
```

输入：

```text
axxbxxb
```

索引：

```text
0 1 2 3 4 5 6
a x x b x x b
```

执行过程：

| 步骤 | 正则 token | 输入位置 | 动作 | 选择点 |
| --- | --- | --- | --- | --- |
| 1 | `a` | 0 | 匹配 `a`，前进到 1 | 无 |
| 2 | `.*` | 1 | 贪婪匹配到末尾 7 | 保存可回退位置 1-7 |
| 3 | `b` | 7 | 末尾无字符，失败 | 弹出选择点 |
| 4 | `.*` 回退 | 6 | 让出索引 6 的 `b` | 剩余回退位置 1-6 |
| 5 | `b` | 6 | 匹配成功 | 完成 |

结论：贪婪不是“直接错”，而是“先吃多，再按需吐回”。如果后面边界明确，回溯量可控；如果后面还有模糊量词，回溯路径会叠加。

## 三、逐步轨迹：分支顺序影响结果

表达式：

```regex
cat|catalog
```

输入：

```text
catalog
```

多数回溯型引擎从左到右尝试分支：

1. 尝试 `cat`。
2. 前三个字符匹配成功。
3. 如果 API 允许返回首次匹配，结果就是 `cat`。
4. 右侧 `catalog` 不会再尝试。

如果业务想优先匹配长词，应写：

```regex
catalog|cat
```

或使用边界：

```regex
cat\b|catalog\b
```

但第二种写法对 `catalog` 仍会先试 `cat\b`，失败后再试 `catalog\b`。它结果正确，但多了一次尝试。

> **重点：** 分支不是集合，很多引擎中它是有顺序的决策列表。

## 四、逐步轨迹：捕获组会随回溯改变

表达式：

```regex
(a|ab)c
```

输入：

```text
abc
```

过程：

1. 第 1 组先尝试 `a`，捕获组 1 = `a`。
2. 后续 `c` 要匹配输入位置 1 的 `b`，失败。
3. 回溯到分支选择点。
4. 第 1 组改试 `ab`，捕获组 1 = `ab`。
5. 后续 `c` 匹配成功。

这说明捕获组不是固定写入一次。只要发生回溯，捕获结果也可能被覆盖。

## 五、逐步轨迹：嵌套量词为什么爆炸

表达式：

```regex
^(a+)+$
```

输入：

```text
aaaa!
```

`(a+)+` 的核心问题是：同一串 `a` 可以被拆成很多组。

对 `aaaa`，可能拆法包括：

```text
aaaa
aaa + a
aa + aa
aa + a + a
a + aaa
a + aa + a
a + a + aa
a + a + a + a
```

当最后遇到 `!`，`$` 失败，引擎会尝试这些拆法。输入越长，拆法越多。对 n 个 `a`，拆分点有 n-1 个，每个拆分点可以切或不切，因此组合数量接近 `2^(n-1)`。

这不是因为 `a+` 本身慢，而是“内层量词”和“外层量词”在同一字符集合上竞争同一段输入。

> **难点：** 灾难性回溯的本质是同一输入片段存在大量等价拆分方式。

## 六、执行轨迹审查方法

审查一个正则时，按这个顺序画轨迹：

1. 标出输入起点：是否从每个位置尝试，还是从开头固定。
2. 标出所有量词：尤其是 `.*`、`.+`、`.*?`、`(.+)+`。
3. 标出所有分支：检查分支是否重叠，例如 `a|aa`、`\w+|\d+`。
4. 标出失败点：哪个 token 最可能在长输入末尾失败。
5. 标出捕获依赖：是否使用反向引用或命名组。
6. 标出断言：断言是在每个位置重复检查，还是只检查一次。

## 七、例子：从执行轨迹优化表达式

需求：匹配双引号字符串，不允许字符串内部出现未转义双引号。

粗糙写法：

```regex
".*?"
```

问题：

- `.` 的范围过宽。
- 是否匹配换行受 dotAll 影响。
- 遇到转义引号时行为不符合需求。

更明确的写法：

```regex
"(?:\\.|[^"\\])*"
```

解释：

- `\\.` 表示一个反斜杠加任意被转义字符。
- `[^"\\]` 表示既不是双引号也不是反斜杠的普通字符。
- 两个分支互斥：一个以反斜杠开头，一个明确排除反斜杠。
- 外层 `*` 重复的是“一个合法字符串内部单元”。

这比 `.*?` 更接近字符串词法规则，失败路径也更清晰。

## 练习

1. 画出 `".*"` 匹配 `"a" "b"` 的执行轨迹。
2. 解释 `(ab|a)b` 匹配 `ab` 的分支尝试过程。
3. 说明 `(\w+\s*)+` 在长文本末尾失败时为什么可能慢。
4. 把 `.*end` 改写成有明确边界的表达式，并说明执行轨迹变化。

## 验收

- 能写出至少 5 步正则执行轨迹。
- 能指出选择点来自哪个量词或分支。
- 能解释捕获组为什么会随回溯改变。
- 能用输入拆分组合解释灾难性回溯。

## 重点

- 回溯型正则的核心是“保存选择点，然后失败时换路”。
- 分支顺序是执行策略，不是数学集合。
- 嵌套量词的危险来自同一输入片段的多种拆分方式。

## 易错

> **易错：** 只看表达式含义，不画失败路径。
>
> 正确做法：复杂正则至少画一条成功路径和一条慢失败路径。
