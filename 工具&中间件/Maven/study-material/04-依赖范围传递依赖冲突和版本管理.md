# Maven 学习资料：依赖范围、传递依赖、冲突和版本管理

[返回索引](../Maven学习资料.md)

## 学习目标

- 掌握 dependency scope。
- 理解传递依赖。
- 排查版本冲突。
- 使用 dependencyManagement 和 BOM 管理版本。

## 理论导读

依赖管理是 Maven 最核心也最容易出问题的部分。一个项目不会只依赖你在 POM 中声明的 jar，还会间接依赖这些 jar 自己声明的依赖。依赖树像一张供应链图，直接依赖是你主动采购的物料，传递依赖是物料供应商又带来的子物料。

版本冲突来自同一个构件被不同路径引入多个版本。Maven 必须选一个版本进入最终 classpath，于是使用“路径最近优先、同级声明靠前优先”等规则。这个选择不一定符合业务预期，所以大型项目必须主动治理依赖版本。

## Maven 依赖收集和版本仲裁算法

Maven 依赖解析可以分成“收集、过滤、仲裁、解析文件”四个阶段。收集阶段从当前项目 POM 出发，把直接依赖和传递依赖递归展开成依赖图；过滤阶段处理 scope、optional、exclusion；仲裁阶段在同一个 `groupId:artifactId` 出现多个版本时决定保留哪个版本；解析文件阶段再到本地仓库或远程仓库寻找实际 POM、jar 等构件。

### 1. 从根 POM 收集直接依赖

当前项目的 `dependencies` 是根节点的一层直接依赖。直接依赖的优先级最高，因为它们表达了当前项目显式选择。如果一个传递依赖版本不符合预期，通常可以在当前 POM 中显式声明该依赖或在 `dependencyManagement` 中管理版本。

### 2. 递归读取传递依赖

Maven 会读取每个依赖构件自己的 POM，继续收集它声明的依赖。依赖树越深，越容易出现重复构件和版本冲突。传递依赖不是“额外下载的随机 jar”，而是 Maven 根据每个依赖 POM 递归推导出来的结果。

### 3. 应用 scope 传播规则

scope 会影响依赖是否进入编译、测试、运行 classpath，也会影响它是否继续向下游传播。例如 test 依赖只服务测试，不会传递到生产依赖链路；provided 依赖编译可见，但运行时通常由容器提供。

### 4. 应用 optional 和 exclusion

`optional=true` 表示该依赖不应自动传给下游项目，适合“可选功能需要时再显式引入”的场景。`exclusion` 则是在某条依赖路径上排除指定传递依赖，适合去掉冲突、重复或有风险的依赖。

### 5. 版本仲裁：nearest wins

当同一个 `groupId:artifactId` 在依赖树中出现多个版本时，Maven 默认选择离根项目最近的版本。路径越短，说明它离当前项目的显式依赖越近，优先级越高。

```text
project
  |-- A -> X:1.0
  |-- B -> C -> X:2.0
```

这里 `X:1.0` 路径更近，所以默认选择 `X:1.0`。

### 6. 同深度冲突：first declaration wins

如果两个版本到根项目的路径深度相同，通常按当前 POM 中声明顺序，先出现的路径优先。

```text
project
  |-- A -> X:1.0
  |-- B -> X:2.0
```

如果 POM 中 A 声明在 B 前面，通常选择 `X:1.0`；如果 B 在 A 前面，结果可能反过来。

### 7. dependencyManagement 覆盖版本选择

`dependencyManagement` 可以为依赖树中的构件指定版本，相当于给版本仲裁增加一条显式管理规则。它不会自动引入依赖，但一旦依赖出现，就可以用受管理的版本替代传递依赖中声明的版本。

### 8. 仓库解析：local first, remote later

版本确定后，Maven 才去解析具体文件。它会先查本地仓库；本地没有，再根据 mirror、repository 和 settings 配置访问远程仓库。对于插件依赖，还会涉及 pluginRepositories。构件下载成功后缓存到本地仓库。

> **重点：** “最近优先”只解决版本仲裁，不等于完整依赖寻找算法。完整流程还包括传递依赖收集、scope 传播、optional/exclusion、dependencyManagement、BOM 和仓库解析。

> **难点：** Maven 最终 classpath 是算法结果，不是 POM 中肉眼看到的直接依赖列表。

## 依赖范围 scope

| scope | 编译 | 测试 | 运行 | 传递给下游 | 常见用途 |
| --- | --- | --- | --- | --- | --- |
| compile | 是 | 是 | 是 | 是 | 默认 |
| provided | 是 | 是 | 否 | 否 | Servlet API |
| runtime | 否 | 是 | 是 | 是 | JDBC 驱动 |
| test | 否 | 是 | 否 | 否 | JUnit |
| system | 是 | 是 | 是 | 否 | 不推荐 |
| import | 仅 dependencyManagement | 仅管理版本 | 仅管理版本 | 否 | BOM |

## 查看依赖树

```bash
mvn dependency:tree
mvn dependency:tree -Dincludes=org.slf4j
mvn dependency:tree -Dverbose
```

看依赖树时重点关注：

- 某个依赖从哪条路径进来。
- 同一个构件是否出现多个版本。
- 哪个版本最终被保留。
- scope 是否符合预期。
- 是否需要 dependencyManagement 或 exclusion。

## 排除传递依赖

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>legacy-lib</artifactId>
    <version>1.0.0</version>
    <exclusions>
        <exclusion>
            <groupId>commons-logging</groupId>
            <artifactId>commons-logging</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

## dependencyManagement

`dependencyManagement` 不会自动引入依赖，只管理版本。

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>com.fasterxml.jackson.core</groupId>
            <artifactId>jackson-databind</artifactId>
            <version>2.17.2</version>
        </dependency>
    </dependencies>
</dependencyManagement>
```

子模块可以省略版本：

```xml
<dependency>
    <groupId>com.fasterxml.jackson.core</groupId>
    <artifactId>jackson-databind</artifactId>
</dependency>
```

## BOM

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>3.3.2</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

## 重点

> **重点：** `dependencyManagement` 管版本，不等于引入依赖。真正引入依赖仍要写在 `dependencies` 中。

> **重点：** Maven 依赖版本仲裁的默认核心规则是 nearest wins；同深度时 first declaration wins。大型项目不要完全依赖默认仲裁，应主动用 dependencyManagement 和 BOM 管理版本。

## 难点

> **难点：** 版本冲突不一定导致构建失败，更危险的是构建成功但运行时出现 `NoSuchMethodError`、`ClassNotFoundException` 或行为异常。

> **难点：** 依赖寻找是先决定有效依赖集合，再解析构件文件。依赖树问题和仓库下载问题要分开排查。

## 易错

> **易错：** 看到依赖冲突只排除某个 jar，却不分析谁引入它、为什么引入、排除后是否破坏上游库。

## 练习

执行：

```bash
mvn dependency:tree
mvn dependency:analyze
```

回答：

- 哪些依赖是直接依赖？
- 哪些依赖是传递依赖？
- 是否存在声明了但未使用的依赖？
- 某个重复依赖最终保留了哪个版本？为什么？
