# Maven 学习资料：总览与心智模型

[返回索引](../Maven学习资料.md)

## 学习目标

完成本章后，你应该能：

- 说清 Maven 解决什么问题。
- 理解 Maven 的四个核心模型：POM、坐标、仓库、生命周期。
- 区分 Maven 和 IDE、JDK、Gradle、Ant 的关系。
- 知道后续学习资料的覆盖范围。

## 理论导读

Maven 可以想象成一条标准化构建流水线。`pom.xml` 是流水线说明书，告诉 Maven 项目叫什么、依赖什么、怎么编译、怎么测试、怎么打包；坐标像仓库货架编号，帮助 Maven 找到某个 jar；本地仓库和远程仓库像物料仓库，保存项目依赖和构建产物；生命周期像流水线步骤，规定从清理、编译、测试到打包、安装、部署的顺序。

Maven 的价值不只是“自动下载 jar”。在没有 Maven 之前，项目依赖可能靠手工复制 jar，编译和打包靠 IDE 或脚本，团队成员环境稍有不同就容易构建失败。Maven 用统一目录结构、统一依赖坐标、统一生命周期和插件机制，把“这个项目怎么构建”写进 POM，让团队和 CI 都能重复执行同一套流程。

学习 Maven 的关键是把命令放回模型里。`mvn package` 不是简单打包，它会从 default 生命周期开头一路执行到 package 阶段，中间可能执行资源处理、编译、测试、打包等插件目标。构建失败时，也要先判断失败发生在依赖解析、编译、测试、插件执行还是部署阶段。

## 核心心智模型

```text
pom.xml
  |-- 项目坐标 groupId:artifactId:version
  |-- 依赖 dependencies
  |-- 插件 build/plugins
  |-- 模块 modules
  |-- 配置 properties/profiles

mvn package
  -> 读取 POM
  -> 解析依赖和插件
  -> 按生命周期执行阶段
  -> 插件目标完成具体任务
  -> 生成 target/ 下的构建产物
```

## Maven 解决的问题

- 项目结构标准化。
- 依赖声明和版本管理。
- 构建生命周期统一。
- 插件化扩展编译、测试、打包、发布。
- 多模块项目构建顺序管理。
- 与私服、CI、制品发布流程集成。

## Maven 不解决的问题

- Maven 不替代 JDK，它需要 JDK 编译 Java。
- Maven 不替代 Git，它不管理源码版本。
- Maven 不替代业务框架，它只是构建和依赖工具。
- Maven 不保证依赖绝对安全，仍需漏洞扫描和版本治理。

## 重点

> **重点：** Maven = POM + 坐标 + 仓库 + 生命周期 + 插件。只背命令无法真正掌握 Maven。

## 难点

> **难点：** 生命周期阶段本身不做事，真正做事的是绑定到阶段上的插件目标。

## 易错

> **易错：** 把 Maven 理解成“下载依赖的工具”。依赖解析只是 Maven 的一部分，构建编排和项目描述同样关键。

## 练习

找一个已有 Maven 项目，回答：

- `pom.xml` 中项目坐标是什么？
- 依赖有哪些？
- 打包类型是什么？
- 执行 `mvn package` 后产物在哪里？
- `target/` 是否应该提交到 Git？

## 验收

- 能解释 Maven 的四个核心模型。
- 能说明 `pom.xml` 的作用。
- 能解释本地仓库和远程仓库。
- 能区分生命周期和插件。
