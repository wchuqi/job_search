# Git 学习资料：仓库维护、性能和底层命令

[返回索引](../Git学习资料.md)

## 理论导读：仓库维护是在整理对象库、引用和索引的健康状态

Git 仓库长期使用后，对象会增多，松散对象会积累，packfile 可能需要重写，引用可能打包，工作区和索引状态也可能变得复杂。维护的目标是让对象存储更紧凑、查询更快、不可达对象按规则清理，并降低大仓库操作成本。

`git gc`、`git maintenance`、`git fsck`、`git count-objects`、`git repack` 等命令不是日常随便执行的魔法按钮，而是观察和整理仓库内部结构的工具。它们处理的是对象库和引用层面的健康问题，和普通文件修改不是一个层级。

性能问题也要分类：clone 慢可能是历史和对象太大，status 慢可能是工作区文件太多，fetch 慢可能是远程对象和网络问题，checkout 慢可能是大文件或过滤规则。先定位瓶颈，再选择维护策略。

## 学习目标

完成本章后，你应该能：

- 理解 Git 垃圾回收和对象清理。
- 知道仓库变慢的常见原因。
- 使用 `fsck`、`gc`、`repack` 等维护命令。
- 了解常见 plumbing commands。
- 识别维护命令的风险边界。

## 一、Git 命令分类：porcelain 和 plumbing

Git 命令可粗略分为：

- porcelain：面向用户的高层命令。
- plumbing：面向脚本和内部机制的底层命令。

porcelain 示例：

```bash
git status
git add
git commit
git switch
git merge
```

plumbing 示例：

```bash
git cat-file
git hash-object
git ls-tree
git rev-parse
git update-index
```

> **重点：** 日常开发主要用 porcelain，理解 plumbing 有助于排障和理解原理。

## 二、对象完整性检查：git fsck

检查对象数据库完整性：

```bash
git fsck
```

查看不可达对象：

```bash
git fsck --unreachable
```

查看悬空对象：

```bash
git fsck --lost-found
```

> **易错：** 不可达对象不一定是错误，可能是 reset、rebase、amend 后暂时没有引用指向的旧对象。

## 三、垃圾回收：git gc

执行垃圾回收和压缩：

```bash
git gc
```

更积极：

```bash
git gc --aggressive
```

常见作用：

- 压缩 loose objects。
- 生成或优化 packfile。
- 清理过期 reflog。
- 移除过期不可达对象。

> **重点：** `git gc` 通常安全，但它可能让过期不可达对象被清理，降低误操作找回机会。

## 四、prune 和 reflog 过期

清理不可达对象：

```bash
git prune
```

查看 reflog 过期配置：

```bash
git config --get gc.reflogExpire
git config --get gc.reflogExpireUnreachable
```

手动过期 reflog：

```bash
git reflog expire --expire=now --all
git gc --prune=now
```

> **易错：** 不要在刚发生误 reset、误 rebase 后立刻 aggressive gc 或 prune，否则可能增加恢复难度。

## 五、repack

重新打包对象：

```bash
git repack -ad
```

常见用途：

- 优化对象存储。
- 减少 loose objects。
- 改善大型仓库性能。

通常不需要日常手动执行，Git 会自动维护。

## 六、仓库变慢的常见原因

排查方向：

- 历史中有大文件。
- loose objects 太多。
- packfile 过大或过多。
- 工作区文件极多。
- 未跟踪文件极多。
- hooks 或状态检查太慢。
- 文件系统或杀毒软件影响。
- submodule / LFS 拉取慢。

查看对象统计：

```bash
git count-objects -vH
```

查看最大对象可用脚本或专门工具，例如：

```bash
git verify-pack -v .git/objects/pack/*.idx
```

> **重点：** 仓库性能问题通常不是靠一个命令解决，要先判断是历史对象、工作区文件、网络还是工具链问题。

## 七、常见底层命令

### cat-file

查看对象类型和内容：

```bash
git cat-file -t HEAD
git cat-file -p HEAD
```

### hash-object

计算对象哈希：

```bash
git hash-object README.md
```

写入对象数据库：

```bash
git hash-object -w README.md
```

### ls-tree

查看 tree：

```bash
git ls-tree HEAD
git ls-tree -r HEAD
```

### rev-parse

解析引用：

```bash
git rev-parse HEAD
git rev-parse --show-toplevel
git rev-parse --abbrev-ref HEAD
```

### update-index

直接操作索引：

```bash
git update-index --chmod=+x script.sh
git update-index --assume-unchanged file
git update-index --no-assume-unchanged file
```

> **易错：** `update-index` 是偏底层命令，不应随意作为常规忽略文件方案。

## 八、恢复和维护的关系

误操作后常用：

```bash
git reflog
git branch recover/old HEAD@{1}
```

但如果对象已经被清理，恢复会变难。

所以误操作后：

1. 停止继续破坏性操作。
2. 查看 reflog。
3. 创建恢复分支。
4. 确认恢复后再清理。

> **重点：** 维护命令应在仓库状态明确、没有恢复需求时执行。

## 九、备份和镜像

创建镜像仓库：

```bash
git clone --mirror <repo-url> repo.git
```

推送镜像：

```bash
git push --mirror <new-url>
```

适合：

- 仓库迁移。
- 完整备份 refs。
- 搭建镜像。

> **易错：** `push --mirror` 会同步删除目标端多余 refs，使用前必须确认目标仓库用途。

## 十、维护建议

个人仓库：

- 定期 `git gc` 通常足够。
- 不提交大文件和构建产物。
- 不滥用 submodule。
- 出问题先 `git status`、`git log`、`git reflog`。

团队仓库：

- 对大文件使用 LFS 或制品仓库。
- 开启 protected branch。
- 控制分支和 tag 清理策略。
- 对 monorepo 使用 sparse checkout 或 partial clone。
- 使用 CI 做安全和质量检查。

- 某个文件在 HEAD 中对应哪个 blob？
- 当前仓库根目录路径是什么？
