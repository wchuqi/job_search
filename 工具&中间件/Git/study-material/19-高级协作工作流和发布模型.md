# Git 学习资料：高级协作工作流和发布模型

[返回索引](../Git学习资料.md)

## 理论导读：工作流是在约束分支指针如何被团队共同移动

团队 Git 工作流的核心不是分支名字，而是规定谁能移动哪些共享引用、什么时候合并、怎样评审、如何发布、出错后如何回滚。feature branch、Git Flow、trunk-based development 都是在不同团队规模、发布频率和质量控制要求下，对分支和提交流动方式做出的取舍。

受保护分支、PR/MR、CI、代码评审共同构成协作边界。受保护分支限制直接 push，PR/MR 提供审查入口，CI 验证构建和测试，代码评审检查设计和风险。它们的目标不是增加流程，而是让共享历史更可靠。

发布模型决定标签、release 分支、hotfix 分支和版本号如何配合。频繁发布的团队更看重主干稳定和自动化验证，长周期发布的团队更需要 release 分支和补丁流程。

## 学习目标

完成本章后，你应该能：

- 区分 feature branch、Git Flow、trunk-based development。
- 理解 protected branch、PR/MR、CI 和代码评审的关系。
- 掌握 fork 工作流中的 `origin` 和 `upstream`。
- 知道 release、hotfix、tag 和回滚策略。

## 一、常见协作模型

### 1. Feature Branch Workflow

流程：

```text
main
  \
   feature/task -> PR/MR -> main
```

常用命令：

```bash
git switch main
git pull --rebase
git switch -c feature/task-name

# 开发提交
git push -u origin feature/task-name
```

适合：

- 多数中小团队。
- 需要代码评审。
- 每个需求独立开发。

### 2. Git Flow

典型分支：

- `main`：生产版本。
- `develop`：集成开发分支。
- `feature/*`：功能。
- `release/*`：发布准备。
- `hotfix/*`：生产紧急修复。

优点：

- 发布节奏清晰。
- 适合传统版本发布。

缺点：

- 分支较多。
- 合并链路较长。
- 对持续交付不够轻量。

### 3. Trunk-Based Development

核心思想：

- 大多数开发尽快合入主干。
- 功能开关控制未完成能力。
- 分支生命周期很短。

适合：

- 自动化测试完善。
- CI/CD 成熟。
- 团队能频繁小步提交。

> **难点：** trunk-based 不是“不用分支”，而是避免长期分支。

## 二、fork 工作流

常见于开源项目：

```text
upstream/project  原项目
origin/project    你的 fork
```

配置：

```bash
git remote -v
git remote add upstream <original-repo-url>
git fetch upstream
```

同步原项目：

```bash
git switch main
git fetch upstream
git rebase upstream/main
git push origin main
```

开发：

```bash
git switch -c feature/fix-docs
git push -u origin feature/fix-docs
```

> **易错：** fork 工作流中 `origin` 通常是自己的 fork，`upstream` 才是原项目。

## 三、PR / MR 工作流

一个好的 PR / MR 应包含：

- 背景和目的。
- 改动摘要。
- 测试方式。
- 风险点。
- 关联 issue / 需求。
- 截图或日志，若相关。

模板：

```markdown
## 变更内容

- ...

## 测试方式

- ...

## 风险点

- ...
```

> **重点：** PR 不是单纯合代码的按钮，而是团队同步上下文、评审质量和触发 CI 的协作单元。

## 四、protected branch

保护分支通常限制：

- 禁止直接 push。
- 必须通过 PR/MR。
- 必须通过 CI。
- 必须至少一人 review。
- 禁止 force push。
- 要求线性历史。

> **易错：** 本地 Git 命令能做的事，不代表远程平台策略允许。

## 五、发布分支和 tag

发布分支：

```bash
git switch -c release/v1.2.0
```

修复发布问题后：

```bash
git commit -m "Fix release blocker"
```

打 tag：

```bash
git tag -a v1.2.0 -m "Release v1.2.0"
git push origin v1.2.0
```

> **重点：** tag 标记发布点，release 分支用于发布前稳定和修复。

## 六、hotfix 流程

生产紧急修复：

```bash
git switch main
git pull --rebase
git switch -c hotfix/login-crash

# 修复并测试
git add .
git commit -m "Fix login crash"
git push -u origin hotfix/login-crash
```

合入生产后，还要同步到开发分支：

```bash
git switch develop
git merge main
```

具体是否 merge / cherry-pick 取决于团队流程。

## 七、回滚策略

### 1. 代码级回滚

已合入共享分支的错误提交：

```bash
git revert <commit>
```

### 2. 发布级回滚

部署系统回滚到上一个 tag 或构建产物：

```bash
git checkout v1.1.9
```

> **重点：** Git 回滚和部署回滚不是同一件事。生产故障时通常先恢复服务，再整理代码历史。

## 八、提交规范

Conventional Commits：

```text
feat(auth): add login timeout handling
fix(payment): handle empty callback payload
docs(git): add study material
refactor(api): simplify response parser
test(user): add profile validation tests
```

常见类型：

| 类型 | 含义 |
| --- | --- |
| `feat` | 新功能 |
| `fix` | 修复 |
| `docs` | 文档 |
| `refactor` | 重构 |
| `test` | 测试 |
| `chore` | 杂项 |

## 九、合并策略选择

平台常见合并按钮：

| 方式 | 结果 | 适合 |
| --- | --- | --- |
| Merge commit | 保留分支结构 | 需要完整上下文 |
| Squash merge | 多个提交压成一个 | 功能分支提交较碎 |
| Rebase merge | 线性历史 | 团队要求线性 |

> **易错：** squash merge 后，原分支提交不会原样出现在主分支历史中。

## 十、团队约定建议

建议明确：

- 分支命名规则。
- 提交信息规范。
- 是否允许 merge commit。
- 是否要求 rebase 主分支。
- 是否允许 force push。
- tag 命名规则。
- 发布分支和 hotfix 流程。
- 大文件和敏感信息处理规则。

## 练习

模拟一个团队流程：

1. 从 `main` 创建 `feature/demo`。
2. 提交两次。
3. rebase 最新 `origin/main`。
4. 创建 PR 描述。
5. 选择 squash merge 或 merge commit，并解释原因。
6. 打一个 release tag。

## 验收

- 能解释 Git Flow 和 trunk-based 的差异。
- 能说明 fork 工作流中 `origin` 和 `upstream`。
- 能描述一个完整 PR/MR 流程。
- 能说明 tag、release branch、hotfix 的关系。
