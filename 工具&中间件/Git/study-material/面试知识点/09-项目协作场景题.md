## 九、项目协作场景题

### 41. 你平时开发一个新功能的 Git 流程是什么？

**参考答案：**

```bash
git switch main
git pull --rebase
git switch -c feature/task-name

# 开发并提交
git status
git add .
git commit -m "Implement task"

# 同步主分支
git fetch origin
git rebase origin/main

# 推送并创建 PR/MR
git push -u origin feature/task-name
```

说明重点：

- 从最新主分支创建功能分支。
- 提交前检查状态和差异。
- 推送前同步主分支。
- 通过 PR/MR 做代码评审。

### 42. 你如何保证提交历史清晰？

**参考答案：**

我会控制每次提交的范围，让一次提交只表达一个明确目标；提交前用 `git diff --staged` 检查内容；本地未推送前可以用 `commit --amend` 或 `rebase -i` 整理提交；已经推送到共享分支后避免随意改写历史。

> **重点：** 清晰历史来自“小而完整的提交”和“明确提交信息”。

### 43. 线上分支误合入错误代码怎么办？

**参考答案：**

如果错误提交已经进入共享分支，优先使用 `git revert` 生成反向提交，再走正常发布流程。

```bash
git revert <bad-commit>
git push
```

如果是尚未推送的本地错误，可以根据情况使用 `reset` 或 `restore`。

> **易错：** 线上共享分支不要直接 hard reset 后强推，除非团队明确制定了这种应急流程。

### 44. 有未提交修改时突然要切分支修 bug，怎么办？

**参考答案：**

如果当前修改还没完成，可以先 stash：

```bash
git stash push -u -m "WIP current task"
git switch main
git pull --rebase
git switch -c hotfix/bug-name
```

修完后回到原分支恢复：

```bash
git switch feature/current-task
git stash pop
```

也可以在当前分支先提交一个 WIP commit，之后再整理历史。

### 45. 多人同时改了同一文件，你怎么降低冲突成本？

**参考答案：**

- 任务拆分时减少多人同时修改同一文件同一位置。
- 小步提交和小范围 PR。
- 经常同步主分支最新变化。
- 冲突发生后先理解两边业务意图，再手动合并。
- 解决后运行相关测试。

> **重点：** 冲突处理不是简单选一边，而是合并业务意图。
