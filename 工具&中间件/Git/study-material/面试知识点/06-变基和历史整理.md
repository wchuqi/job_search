## 六、rebase 和历史整理

### 30. merge 和 rebase 有什么区别？

**参考答案：**

`merge` 会保留分支真实分叉和合并历史，通常产生 merge commit。`rebase` 会把当前分支提交复制到新的基底上，使历史更线性，但会改写提交哈希。

```bash
git merge main
git rebase main
```

> **难点：** rebase 不是移动原提交，而是创建新提交。

### 31. rebase 适合什么场景？

**参考答案：**

适合在自己的功能分支上同步主分支最新变化，或者在提交 PR 前整理本地提交历史。

```bash
git fetch origin
git rebase origin/main
```

> **易错：** 不要对多人共享分支随意 rebase 后强推。

### 32. 交互式 rebase 能做什么？

**参考答案：**

交互式 rebase 可以整理提交历史，例如修改提交信息、合并提交、删除提交、暂停修改某个历史提交。

```bash
git rebase -i HEAD~3
```

常见动作：

| 动作 | 含义 |
| --- | --- |
| `pick` | 保留提交 |
| `reword` | 修改提交信息 |
| `squash` | 合并到上一个提交并编辑信息 |
| `fixup` | 合并到上一个提交并丢弃当前信息 |
| `drop` | 删除提交 |

### 33. `git push --force-with-lease` 和 `git push --force` 有什么区别？

**参考答案：**

`--force` 会直接强推覆盖远程分支。`--force-with-lease` 会先检查远程分支是否有别人新推送的提交，如果有就拒绝覆盖，因此更安全。

```bash
git push --force-with-lease
```

> **重点：** 功能分支 rebase 后需要强推时，优先使用 `--force-with-lease`。
