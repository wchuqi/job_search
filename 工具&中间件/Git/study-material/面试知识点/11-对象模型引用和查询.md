## 一、对象模型

### 46. Git 的四类核心对象是什么？

**参考答案：**

Git 的核心对象包括 blob、tree、commit、tag。

- blob 保存文件内容，不保存文件名。
- tree 保存目录结构、文件名、文件模式和对象引用。
- commit 指向一个 tree，并记录父提交、作者、提交者、时间和提交信息。
- tag 可以给某个对象做标记，附注 tag 还会生成 tag 对象。

> **重点：** commit 不是直接保存所有文件名和文件内容，而是指向一棵 tree。

### 47. Git 为什么说提交是快照？

**参考答案：**

因为每个 commit 指向一个 tree，这个 tree 表示某一时刻完整目录结构和文件内容。`git show` 展示的 diff 是 Git 根据两个快照计算出来的，不代表 commit 本身只是一个补丁。

> **易错：** 不要把 Git commit 理解成 SVN 风格的简单差异包。

### 48. 分支为什么很轻量？

**参考答案：**

因为分支本质是一个引用，指向某个提交。创建分支只是创建或更新引用，不会复制项目文件，也不会复制完整历史。

```bash
git branch feature/login
```

> **重点：** Git 鼓励频繁创建分支，根本原因是分支只是指针。

## 二、引用和版本选择器

### 49. HEAD、branch、tag 有什么区别？

**参考答案：**

HEAD 表示当前检出位置，通常指向当前分支。branch 是可移动引用，随着提交向前移动。tag 通常用于固定标记某个提交，发布版本中常用。

| 名称 | 是否常移动 | 常见用途 |
| --- | --- | --- |
| HEAD | 会变 | 当前所在位置 |
| branch | 会变 | 开发线 |
| tag | 通常不变 | 发布点 |

### 50. 什么是 detached HEAD？

**参考答案：**

detached HEAD 是 HEAD 直接指向某个提交，而不是指向分支。此时提交新内容不会被普通分支保存，后续切走后可能难以找到。

解决方式：

```bash
git switch -c experiment/save-work
```

> **易错：** detached HEAD 下不是不能提交，而是提交没有分支名保护。

### 51. `HEAD~1` 和 `HEAD^` 有什么区别？

**参考答案：**

普通线性历史中二者通常相同，都表示当前提交的第一个父提交。但在 merge commit 中，`HEAD^2` 可以表示第二个父提交，而 `HEAD~2` 表示沿第一个父提交连续向前两代。

> **难点：** `~` 是沿第一父链走代数，`^` 是选择第几个父提交。

### 52. `A..B` 和 `A...B` 有什么区别？

**参考答案：**

在 `git log` 中，`A..B` 表示 B 有但 A 没有的提交；`A...B` 表示 A 和 B 的对称差集。在 `git diff` 中，`A...B` 通常表示从 merge base 到 B 的差异。

```bash
git log main..feature
git diff main...feature
```

> **易错：** 三点语法在 `log` 和 `diff` 中语义不同，要结合命令说明。

## 三、查询和排障

### 53. 如何查某个文件的历史？

**参考答案：**

```bash
git log -- path/to/file
git log --follow -- path/to/file
```

`--follow` 可以在文件重命名后继续追踪历史，但它有局限，复杂重命名不一定完全准确。

### 54. 如何查某段代码是谁引入的？

**参考答案：**

可以用 pickaxe：

```bash
git log -S "关键字符串" --patch
git log -G "正则表达式" --patch
```

`-S` 关注字符串出现次数变化，`-G` 关注 patch 中匹配正则的行。

### 55. `git blame` 的正确使用方式是什么？

**参考答案：**

`git blame` 用来查看某行代码最后由哪个提交修改，常用于理解历史和需求背景。

```bash
git blame -L 20,40 src/app.js
git show <commit>
```

> **易错：** blame 不是追责工具，面试中应强调它用于理解上下文。
