## 三、分支、合并和冲突

### 11. Git 分支的本质是什么？

**参考答案：**

分支本质是指向某个提交的轻量指针。创建分支不会复制整个项目目录，而是创建一个新的指针。

```bash
git switch -c feature/login
```

> **重点：** 分支很轻量，所以 Git 鼓励频繁创建功能分支。

### 12. `git switch` 和 `git checkout` 有什么区别？

**参考答案：**

`git checkout` 是老命令，既能切分支，也能恢复文件，职责较多。`git switch` 专注于切换分支，语义更清晰。

```bash
git switch main
git switch -c feature/login
```

> **易错：** 新项目中建议优先使用 `git switch` 切分支，用 `git restore` 恢复文件。

### 13. 什么是 fast-forward merge？

**参考答案：**

如果主分支在创建功能分支后没有新提交，合并时 Git 只需要把主分支指针向前移动，这就是快进合并。

```text
合并前：
A --- B  main
       \
        C --- D  feature

合并后：
A --- B --- C --- D  main
```

> **重点：** fast-forward 不会产生新的 merge commit。

### 14. 什么是三方合并？

**参考答案：**

当两个分支从共同祖先开始各自有新提交时，Git 会基于共同祖先、当前分支、被合并分支进行三方合并，并生成一个新的 merge commit。

```text
A --- B --- C ------- M
       \             /
        D --- E -----
```

> **难点：** 三方合并不是简单覆盖文件，而是比较共同祖先后的两边变化。

### 15. 合并冲突为什么会发生？

**参考答案：**

当两个分支修改了同一文件的同一位置，Git 无法自动判断应该保留哪一边，就会产生冲突。冲突不是 Git 出错，而是需要人工判断最终业务结果。

冲突标记示例：

```text
<<<<<<< HEAD
当前分支内容
=======
被合并分支内容
>>>>>>> feature/login
```

> **易错：** 解决冲突后必须删除冲突标记，并确认最终内容符合业务预期。

### 16. 解决冲突的一般步骤是什么？

**参考答案：**

```bash
git status
# 打开冲突文件，手动修改成最终结果
git add <file>
git commit
```

如果是在 rebase 中解决冲突：

```bash
git add <file>
git rebase --continue
```

> **易错：** merge 冲突解决后通常 `git commit`，rebase 冲突解决后是 `git rebase --continue`。

### 17. 如何放弃一次未完成的 merge？

**参考答案：**

```bash
git merge --abort
```

它会回到合并开始前的状态。

> **易错：** `git merge --abort` 只适用于合并进行中。合并提交完成后不能用它撤销。
