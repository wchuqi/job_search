## 一、大型仓库

### 64. Git LFS 解决什么问题？

**参考答案：**

Git LFS 用指针文件替代 Git 历史中的大文件内容，大文件本体存储在 LFS 服务中。它适合管理设计稿、模型文件、压缩包等大二进制文件。

```bash
git lfs track "*.psd"
git add .gitattributes
```

> **易错：** 已经进入普通 Git 历史的大文件，不会因为启用 LFS 自动消失，需要迁移历史。

### 65. submodule 和 subtree 有什么区别？

**参考答案：**

submodule 在主仓库中记录另一个仓库的某个提交指针，依赖边界清晰，但使用复杂。subtree 把外部仓库内容合入当前仓库目录，克隆后直接可用，但同步和历史更重。

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| submodule | 边界清晰 | 初始化、更新复杂 |
| subtree | 使用简单 | 同步和历史更重 |

### 66. worktree 解决什么问题？

**参考答案：**

worktree 允许同一个仓库同时检出多个工作区，适合当前分支未完成但需要临时处理 hotfix，或者同时对比多个分支。

```bash
git worktree add -b hotfix/login ../project-hotfix main
git worktree list
```

### 67. sparse checkout 适合什么场景？

**参考答案：**

sparse checkout 适合大型 monorepo，只检出自己需要的部分目录，减少工作区文件数量。

```bash
git sparse-checkout init --cone
git sparse-checkout set apps/web packages/ui
```

### 68. shallow clone 有什么限制？

**参考答案：**

shallow clone 只拉取部分历史，适合 CI 或临时环境。但深层历史、bisect、旧 tag、某些 merge base 计算可能受限。

```bash
git clone --depth 1 <repo-url>
```

## 二、维护和底层命令

### 69. `git gc` 做什么？

**参考答案：**

`git gc` 会执行垃圾回收和对象压缩，清理过期不可达对象，优化 packfile。它通常安全，但可能减少误操作后找回旧对象的机会。

```bash
git gc
```

### 70. `git fsck` 有什么作用？

**参考答案：**

`git fsck` 用于检查对象数据库完整性，发现损坏对象、不可达对象或悬空对象。

```bash
git fsck
```

> **重点：** 不可达对象不一定是错误，可能是 amend、rebase、reset 后遗留的旧对象。

### 71. `git cat-file`、`git ls-tree`、`git rev-parse` 分别做什么？

**参考答案：**

- `git cat-file` 查看对象类型和内容。
- `git ls-tree` 查看 tree 对象。
- `git rev-parse` 解析引用或显示仓库路径。

```bash
git cat-file -p HEAD
git ls-tree -r HEAD
git rev-parse --show-toplevel
```

### 72. 仓库变慢常见原因有哪些？

**参考答案：**

常见原因包括历史中有大文件、未跟踪文件过多、loose objects 太多、packfile 过大、工作区文件极多、hooks 慢、LFS 或 submodule 拉取慢、杀毒软件或文件系统影响。

排查入口：

```bash
git count-objects -vH
git status
```

### 73. `git push --mirror` 有什么风险？

**参考答案：**

`git push --mirror` 会把本地所有 refs 镜像到远程，包括删除远程多余 refs。它适合仓库迁移或镜像，但误用可能删除目标仓库分支和 tag。

> **易错：** 不要把 `push --mirror` 当成普通 push 使用。
