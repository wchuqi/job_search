# Git 学习资料：文件跟踪、忽略规则、属性和换行

[返回索引](../Git学习资料.md)

## 理论导读：文件规则决定 Git 看见什么、如何保存什么

Git 只管理被跟踪的文件。未跟踪文件即使在工作区里，也不会进入提交；被跟踪文件即使写进 `.gitignore`，仍会继续被 Git 关注。忽略规则解决的是“哪些新文件不该被 Git 看见”，不是“从历史中删除已跟踪文件”。

`.gitattributes` 比 `.gitignore` 更像文件处理规则表。它可以指定换行归一化、diff 驱动、merge 策略、LFS 跟踪等行为。跨平台团队常见的换行问题，本质上是 Windows、macOS、Linux 对行尾约定不同，Git 需要在工作区和仓库对象之间做一致转换。

文件跟踪、忽略、属性和换行看似琐碎，但它们决定了仓库是否干净、构建产物是否污染历史、二进制文件是否被错误合并、跨平台协作是否反复出现无意义 diff。

## 学习目标

完成本章后，你应该能：

- 判断文件是否被 Git 跟踪。
- 正确编写 `.gitignore`。
- 处理已经被提交过但后来需要忽略的文件。
- 理解 `.gitattributes`、换行符和二进制文件处理。
- 知道 `assume-unchanged` 和 `skip-worktree` 的风险。

## 一、文件跟踪状态

Git 文件大致有这些状态：

| 状态 | 含义 |
| --- | --- |
| untracked | Git 还没有跟踪 |
| tracked | 已被 Git 跟踪 |
| modified | 工作区有修改 |
| staged | 修改已进入暂存区 |
| deleted | 跟踪文件被删除 |
| conflicted | 合并或变基冲突中 |

查看状态：

```bash
git status
git status --short
```

> **重点：** `.gitignore` 只影响 untracked 文件，不影响 tracked 文件。

## 二、`.gitignore` 规则

常见写法：

```gitignore
# 目录
node_modules/
dist/

# 文件扩展名
*.log
*.tmp

# 本地环境
.env
.env.local
!.env.example

# 只忽略根目录 dist
/dist/
```

规则说明：

| 写法 | 含义 |
| --- | --- |
| `node_modules/` | 忽略所有同名目录 |
| `*.log` | 忽略所有 log 文件 |
| `/dist/` | 只忽略仓库根目录 dist |
| `!file` | 取消忽略 |
| `**/temp` | 匹配任意层级 temp |

检查某文件为什么被忽略：

```bash
git check-ignore -v .env
```

> **易错：** 忽略规则写对了但文件仍被提交，通常是因为它之前已经被 Git 跟踪。

## 三、停止跟踪已提交文件

例如 `.env` 已经被提交，现在想保留本地文件但从仓库移除：

```bash
git rm --cached .env
echo ".env" >> .gitignore
git add .gitignore
git commit -m "Stop tracking env file"
```

> **重点：** `git rm --cached` 只从索引中移除，不删除本地文件。

> **易错：** 如果敏感信息已经进入历史，仅删除当前版本不够，仍需要清理历史并轮换密钥。

## 四、全局忽略和仓库本地忽略

全局忽略适合操作系统或个人编辑器文件：

```bash
git config --global core.excludesfile ~/.gitignore_global
```

仓库本地忽略但不提交给团队：

```text
.git/info/exclude
```

使用建议：

- 项目通用规则放 `.gitignore`。
- 个人工具文件放全局 ignore。
- 临时本地规则放 `.git/info/exclude`。

## 五、`.gitattributes`

`.gitattributes` 用于定义文件属性，例如换行、diff 方式、合并策略、LFS 规则。

常见例子：

```gitattributes
* text=auto
*.sh text eol=lf
*.bat text eol=crlf
*.png binary
*.jpg binary
```

> **重点：** `.gitattributes` 是仓库级规则，适合团队统一文本和二进制处理。

## 六、换行符 LF 和 CRLF

不同系统默认换行：

- Linux / macOS：LF
- Windows：CRLF

相关配置：

```bash
git config --global core.autocrlf true
git config --global core.autocrlf input
git config --global core.eol lf
```

常见策略：

- Windows 团队：可以使用 `core.autocrlf true`。
- 跨平台项目：更推荐 `.gitattributes` 明确规则。

> **易错：** 大量“整文件变化”可能只是换行符变化。提交前要检查 diff。

## 七、文件权限和可执行位

Git 会跟踪可执行位变化：

```bash
git update-index --chmod=+x scripts/deploy.sh
```

如果 Windows 上经常出现权限误变，可以检查：

```bash
git config core.filemode
```

## 八、重命名和大小写问题

Git 不直接存储“重命名”操作，而是通过内容相似度在 diff/log 中检测。

```bash
git mv old-name.js new-name.js
git status
```

大小写敏感问题：

```bash
git config core.ignorecase
```

跨平台项目中，`User.js` 和 `user.js` 这种命名容易造成问题。

## 九、assume-unchanged 和 skip-worktree

### assume-unchanged

告诉 Git 暂时假设文件没变化，主要用于性能优化：

```bash
git update-index --assume-unchanged config.local
git update-index --no-assume-unchanged config.local
```

### skip-worktree

告诉 Git 工作区文件可与索引不同，常用于稀疏检出内部机制：

```bash
git update-index --skip-worktree config.local
git update-index --no-skip-worktree config.local
```

> **易错：** 不要把这两个命令当作“本地忽略已跟踪配置文件”的常规方案。团队项目更应使用模板配置，例如 `.env.example`。

## 十、二进制文件和大文件

普通 Git 不适合频繁变更的大二进制文件，例如：

- 视频
- 大模型权重
- 设计源文件
- 压缩包
- 数据库 dump

更适合：

- Git LFS
- 对象存储
- 制品仓库
- 发布系统

- [21-大型仓库子模块LFS和工作树.md](21-大型仓库子模块LFS和工作树.md)

## 练习

1. 创建 `.gitignore`，忽略 `.env`、`dist/`、`*.log`。
2. 创建 `.env.example` 并确保它不会被忽略。
3. 使用 `git check-ignore -v` 检查 `.env`。
4. 用 `.gitattributes` 规定 `*.sh` 使用 LF。
5. 用 `git diff --check` 检查空白和换行问题。

```bash
git check-ignore -v .env
git diff --check
```

## 验收

- 能解释 `.gitignore` 为什么不影响已跟踪文件。
- 能安全移除已跟踪的本地配置文件。
- 能说明 `.gitattributes` 和 `.gitignore` 的区别。
- 能识别换行符造成的大量 diff。
