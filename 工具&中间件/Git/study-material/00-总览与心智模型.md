## 理论导读：Git 是对象数据库加一组可移动指针

Git 表面上是在管理文件版本，内部更像一个内容寻址的对象数据库。文件内容会变成 blob，对目录结构的描述会变成 tree，一次提交会变成 commit，分支和标签则是给这些对象起的名字或指针。你看到的 `main`、`feature/login`、`HEAD`，本质上都在帮助 Git 找到某个提交对象。

学习 Git 最重要的心智模型，是同时看见三个区域：工作区是你正在编辑的文件，暂存区是下一次提交的草稿，本地仓库保存已经提交的历史。`git add` 不是“保存”，而是把工作区内容放进暂存区；`git commit` 不是“上传”，而是把暂存区做成一个新的提交对象；`git push` 才是把本地对象和引用同步到远程。

所以 Git 出问题时，不要先背命令，而要先定位状态：文件改动在哪里，HEAD 指向哪里，当前分支指向哪里，远程跟踪分支停在哪里，操作是否会移动分支指针或改写历史。

## 使用方式

- 每学完一个阶段，至少做一次完整练习。
- 每次操作前先执行 `git status`，确认当前状态。
- 每次复杂操作前创建备份分支。
- 遇到错误时先停下来，不要连续尝试多个撤销命令。
- 先用 [14-Git完整知识点清单.md](14-Git完整知识点清单.md) 检查覆盖范围，再按模块深入学习。

```bash
git status
git branch backup/before-risky-operation
```

> **重点：** Git 学习的核心不是背命令，而是理解“当前修改在哪里”：工作区、暂存区、本地仓库、远程仓库。

> **易错：** 不清楚当前分支、不看 `git status`、在共享分支上随意 `reset` 或 `rebase`，是最常见的事故来源。

## Git 总体心智模型

Git 可以理解成一个“快照系统”。每次提交都会记录项目在某一时刻的完整状态，并通过父提交把历史串起来。

### 四个重要区域

| 区域 | 英文 | 作用 | 常见命令 |
| --- | --- | --- | --- |
| 工作区 | Working Tree | 你正在编辑的真实文件 | `git status`, `git diff`, `git restore` |
| 暂存区 | Staging Area / Index | 下一次提交要包含的内容 | `git add`, `git restore --staged` |
| 本地仓库 | Local Repository | 已经提交的历史 | `git commit`, `git log`, `git reset` |
| 远程仓库 | Remote Repository | 团队共享的仓库 | `git fetch`, `git pull`, `git push` |

### 文件状态

```text
未跟踪 untracked
  -> git add

已暂存 staged
  -> git commit

已提交 committed
  -> 修改文件

已修改 modified
  -> git add 或 git restore
```

> **重点：** `git add` 不是“添加文件”这么简单，它的本质是“把当前文件内容放入暂存区，准备进入下一次提交”。

> **易错：** 修改文件后直接 `git commit`，如果没有 `git add`，修改不会进入提交，除非使用 `git commit -a` 且文件已经被 Git 跟踪。

### 提交、分支和 HEAD

一个提交包含：

- 文件快照
- 提交信息
- 作者和时间
- 父提交
- 提交哈希

分支是指向某个提交的指针，`HEAD` 表示你当前所在的位置，通常指向当前分支。

```text
A --- B --- C  main
              ^
             HEAD
```

如果创建新分支：

```text
A --- B --- C  main
              \
               feature
```

> **难点：** 分支不是复制了一份代码目录，而是一个轻量指针。切换分支时，Git 会根据目标分支对应的提交更新工作区文件。
