# Git 学习资料：阶段 2：分支开发、合并和冲突

[返回索引](../Git学习资料.md)

## 理论导读：分支是指针，合并是在重组提交图

Git 分支不是复制一份项目文件，也不是复制一份完整历史。分支只是一个指向某个提交的可移动引用。你在分支上提交时，Git 创建新 commit，然后把当前分支指针向前移动。正因为分支只是指针，所以创建、切换和删除分支都很轻量。

HEAD 表示“当前所在位置”。正常情况下 HEAD 指向当前分支，当前分支再指向某个提交；提交时移动的是分支指针。理解这层关系后，`git switch` 就不只是切换目录文件，而是在让 HEAD 指向另一个分支，并把工作区更新到那个分支对应的快照。

合并的本质是把两条提交路径重新接到一起。快进合并只是移动指针，三方合并会参考共同祖先、当前分支和被合并分支三个快照生成新的合并提交。冲突不是 Git 出错，而是 Git 无法根据文本和历史自动判断哪个业务结果正确，需要人来决定最终内容。

## 阶段 2：分支开发、合并和冲突

### 学习目标

完成本阶段后，你应该能：

- 创建、切换和删除分支。
- 理解分支指针和 HEAD。
- 区分快进合并和三方合并。
- 解决常见合并冲突。

### 创建和切换分支

创建分支的成本很低，因为 Git 只是创建一个指向当前提交的新引用。切换分支时，Git 会让 HEAD 指向目标分支，并把工作区文件更新成目标分支对应提交的快照。如果工作区有未提交修改，切换可能被拒绝，因为 Git 不确定这些修改是否会被覆盖。

删除分支删除的是引用名，不是立刻删除提交对象。已经被其他分支或标签引用的提交仍然存在；未被引用的提交也通常能在一段时间内通过 reflog 找回。但这不代表可以随意强删分支，因为找回需要你知道丢失前的位置。

查看分支：

```bash
git branch
```

创建并切换到新分支：

```bash
git switch -c feature/add-multiply
```

只切换分支：

```bash
git switch main
```

删除已经合并的分支：

```bash
git branch -d feature/add-multiply
```

强制删除未合并分支：

```bash
git branch -D feature/add-multiply
```

> **易错：** `git branch -D` 会删除未合并分支指针。如果分支上有没合并的重要提交，虽然通常还能通过 `reflog` 找回，但不要把它当成常规操作。

### 分支开发例子

功能分支的作用是隔离任务。主分支可以保持稳定，功能分支可以承载某个需求的多次提交。等功能完成后，再把这条提交路径合回主分支。这样团队可以同时推进多个任务，而不是所有人都直接在主分支上交错提交。

一个分支最好服务于一个明确目标。分支里混入多个无关任务，会让合并、回滚和代码评审变复杂。分支名不只是标签，它应该帮助团队快速判断这条历史的用途。

当前 `src/calculator.js`：

```javascript
function add(a, b) {
  return a + b;
}

function subtract(a, b) {
  return a - b;
}

module.exports = {
  add,
  subtract,
};
```

创建功能分支：

```bash
git switch -c feature/multiply
```

修改代码：

```javascript
function add(a, b) {
  return a + b;
}

function subtract(a, b) {
  return a - b;
}

function multiply(a, b) {
  return a * b;
}

module.exports = {
  add,
  subtract,
  multiply,
};
```

提交：

```bash
git add src/calculator.js
git commit -m "Add multiply function"
```

切回主分支并合并：

```bash
git switch main
git merge feature/multiply
```

### 快进合并

快进合并发生在目标分支没有分叉时。Git 不需要创建新提交，只要把目标分支指针沿着已有提交链向前移动即可。它像把书签从旧章节挪到新章节，中间没有产生额外合并记录。

快进合并的历史很直观，但会隐藏“这个功能分支曾经作为独立任务存在”的痕迹。团队是否允许快进，取决于更重视线性历史，还是更重视保留分支合并节点。

如果 `main` 在创建分支后没有新提交，合并时只是把 `main` 指针向前移动：

```text
合并前：

A --- B  main
       \
        C --- D  feature

合并后：

A --- B --- C --- D  main, feature
```

这种合并叫 fast-forward。

### 三方合并

三方合并发生在两条分支都从共同祖先之后产生了新提交。Git 会拿共同祖先、当前分支最新提交、被合并分支最新提交进行比较，自动合并不冲突的部分，并创建一个有两个父提交的 merge commit。

merge commit 的意义是明确记录“两条历史在这里汇合”。它让历史图保留协作结构，也让以后排查时能看出某个功能是在何时整体合入的。代价是历史图会比线性历史更复杂。

如果 `main` 和 `feature` 都有新提交，Git 会创建一个新的合并提交：

```text
合并前：

A --- B --- C  main
       \
        D --- E  feature

合并后：

A --- B --- C ------- M  main
       \             /
        D --- E -----
```

> **难点：** 三方合并会参考三个点：共同祖先、当前分支最新提交、被合并分支最新提交。

### 制造并解决冲突

冲突通常发生在两个分支修改了同一文件的同一区域，Git 无法仅凭文本判断最终业务结果。冲突标记展示了当前分支版本和被合并分支版本，你要做的不是机械删除某一边，而是根据业务含义写出最终正确内容。

解决冲突分三步：理解双方修改目的，编辑文件得到最终结果，`git add` 告诉 Git 冲突已解决。`git add` 在这里不是简单暂存，而是把“人工解决后的版本”放入暂存区，供最终合并提交使用。

在 `main` 分支修改 `README.md`：

```markdown
# Git Demo

这是主分支上的说明。
```

提交：

```bash
git add README.md
git commit -m "Update README on main"
```

创建另一个分支并修改同一行：

```bash
git switch -c feature/readme-wording
```

`README.md`：

```markdown
# Git Demo

这是功能分支上的说明。
```

提交：

```bash
git add README.md
git commit -m "Update README wording"
```

切回 `main` 并合并：

```bash
git switch main
git merge feature/readme-wording
```

可能出现冲突：

```text
<<<<<<< HEAD
这是主分支上的说明。
=======
这是功能分支上的说明。
>>>>>>> feature/readme-wording
```

解决冲突后的文件：

```markdown
# Git Demo

这是 Git 练习仓库的说明，包含主分支和功能分支的更新。
```

标记冲突已解决并提交：

```bash
git add README.md
git commit
```

> **重点：** 冲突文件里的 `<<<<<<<`、`=======`、`>>>>>>>` 必须全部删除，保留最终正确内容。

> **易错：** 不要一看到冲突就直接 `git add . && git commit`。先打开文件确认最终内容是否正确。

### 放弃一次合并

`git merge --abort` 的作用是把仓库状态退回合并开始前。它适合在冲突太复杂、合并方向选错、或需要先整理工作区时使用。它不是撤销已经完成的合并提交，只能用于合并进行中。

如果合并已经提交完成，历史里已经出现 merge commit，此时要撤销就需要根据是否共享选择 reset 或 revert。区分“合并进行中”和“合并已完成”，是选择正确撤销方式的关键。

如果合并后发现冲突很多，想回到合并前状态：

```bash
git merge --abort
```

> **易错：** `git merge --abort` 只能在正在合并且未完成时使用。合并提交已经完成后，需要用其他方式撤销。

### 分支命名建议

常见命名：

```text
feature/login
feature/payment-callback
bugfix/fix-login-timeout
hotfix/production-crash
release/v1.2.0
docs/git-study-notes
```

### 阶段 2 要点

> **重点：** 分支用来隔离任务，一个分支最好只做一个明确目标。

> **难点：** 合并冲突不是 Git 出错，而是 Git 无法判断哪一份修改才是正确业务结果。

> **易错：** 合并前工作区必须干净。切分支或合并前先执行 `git status`。
