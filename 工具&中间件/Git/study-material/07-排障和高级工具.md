# Git 学习资料：阶段 7：排障和高级工具

[返回索引](../Git学习资料.md)

## 理论导读：排障是在恢复对象、引用和工作区的一致性

Git 排障不是盲目试命令，而是确认对象还在不在、引用指向哪里、工作区是否干净、暂存区是否和预期一致。很多“丢失”的提交其实没有被立即删除，只是没有分支名指向它；`reflog` 记录了 HEAD 和分支指针移动历史，因此常常能找回 amend、reset、rebase 后看似消失的提交。

高级工具的价值是让隐藏状态可见。`git bisect` 用二分法在历史中定位引入 bug 的提交；`git stash` 把当前未提交修改临时打包保存；`git worktree` 允许同一个仓库同时检出多个工作区；`git blame` 和 `git log -S` 帮助追踪某行或某段代码的变化来源。

排障时要先停止扩大损失：不要在不理解状态时连续 reset、clean、force push。先看 `status`、`log`、`reflog`，必要时创建备份分支，再做恢复动作。

## 阶段 7：排障和高级工具

### 学习目标

完成本阶段后，你应该能：

- 用 `reflog` 找回误操作前的位置。
- 用 `stash` 临时保存现场。
- 用 `cherry-pick` 挑选提交。
- 用 `bisect` 定位引入 bug 的提交。
- 用 `blame` 追踪代码变更来源。

### reflog：找回历史位置

`git reflog` 记录 HEAD 和分支指针的移动历史。

```bash
git reflog
```

示例输出：

```text
a1b2c3d HEAD@{0}: reset: moving to HEAD~1
f8e9d7c HEAD@{1}: commit: Add payment validation
3d9a4aa HEAD@{2}: commit: Add calculator module
```

如果误执行了：

```bash
git reset --hard HEAD~1
```

可以找回：

```bash
git reflog
git reset --hard HEAD@{1}
```

更保守的做法是创建分支保存：

```bash
git branch recover/payment-validation HEAD@{1}
git switch recover/payment-validation
```

> **重点：** `reflog` 是 Git 排障神器，能找回很多“看起来消失”的提交。

> **易错：** reflog 不是永久备份，会被 Git 定期清理。重要提交要及时创建分支或 tag。

### stash：临时保存工作现场

你正在开发功能，但突然要切分支修线上问题：

```bash
git status
git stash push -m "WIP user profile"
git switch main
git switch -c hotfix/login-timeout
```

查看 stash：

```bash
git stash list
```

恢复最近一次 stash 并删除记录：

```bash
git stash pop
```

恢复但保留 stash 记录：

```bash
git stash apply stash@{0}
```

删除 stash：

```bash
git stash drop stash@{0}
```

保存未跟踪文件：

```bash
git stash push -u -m "WIP including new files"
```

> **重点：** `stash` 适合短期临时保存，不适合长期保存工作成果。

> **易错：** 默认 `git stash` 不保存未跟踪文件。新建文件要加 `-u`。

### cherry-pick：挑选提交

场景：`feature/payment` 分支里有一个修复提交，需要单独放到 `main`。

查看提交：

```bash
git log --oneline feature/payment
```

切换目标分支：

```bash
git switch main
git cherry-pick <commit-hash>
```

如果冲突：

```bash
# 手动解决冲突
git add <file>
git cherry-pick --continue
```

放弃：

```bash
git cherry-pick --abort
```

> **重点：** cherry-pick 是“复制某个提交的改动到当前分支”，不是移动提交。

> **易错：** cherry-pick 后会生成新提交，新提交哈希和原提交不同。

### bisect：定位引入 bug 的提交

假设当前版本有 bug，但 `v1.0.0` 是好的。

开始二分：

```bash
git bisect start
git bisect bad
git bisect good v1.0.0
```

Git 会切到中间某个提交，你运行测试：

```bash
npm test
```

如果测试通过：

```bash
git bisect good
```

如果测试失败：

```bash
git bisect bad
```

重复直到 Git 找到第一个坏提交。

结束：

```bash
git bisect reset
```

自动运行测试：

```bash
git bisect start
git bisect bad
git bisect good v1.0.0
git bisect run npm test
git bisect reset
```

> **难点：** `bisect` 要求你能判断某个提交是 good 还是 bad。最好有自动化测试。

> **易错：** `bisect` 结束后记得 `git bisect reset`，否则你可能停留在某个历史提交上。

### blame：追踪某行代码来源

查看文件每一行最后由哪个提交修改：

```bash
git blame src/calculator.js
```

只看部分行：

```bash
git blame -L 1,40 src/calculator.js
```

查看某行对应提交：

```bash
git show <commit-hash>
```

> **重点：** `blame` 用于理解历史，不应用来指责个人。重点是找上下文、找需求背景、找相关改动。

### clean：清理未跟踪文件

查看将要删除什么：

```bash
git clean -nd
```

真正删除：

```bash
git clean -fd
```

> **易错：** `git clean -fd` 会删除未跟踪文件。执行前一定先用 `git clean -nd` 预览。

### 阶段 7 要点

> **重点：** 出问题先看 `git status`、`git log --oneline --graph --decorate --all`、`git reflog`。

> **难点：** 高级命令通常会改变历史或工作区状态，操作前先创建备份分支。

> **易错：** 不要把 stash 当成长期存储。重要工作必须提交到分支。
