# Git 学习资料：阶段 5：变基（rebase）和整理提交历史

[返回索引](../Git学习资料.md)

## 理论导读：rebase 是复制提交，不是移动原提交

变基的核心不是“让历史变直”这么简单，而是把一串提交拿出来，在新的基底上重新应用一遍。重新应用会生成新的 commit 对象，所以提交哈希会变化。原来的提交通常还会在一段时间内留在对象库和 reflog 中，但分支指针已经指向新生成的那串提交。

可以把 rebase 想成把一叠便签从旧白板上揭下来，按顺序贴到另一块白板后面。便签内容可能相同，但它们所在位置、父提交和哈希都变了。交互式 rebase 则是在重贴过程中允许你压缩、改名、拆分或删除提交，让提交历史更像一份清晰的开发叙事。

rebase 的边界是共享历史。整理自己尚未推送的本地提交很合适；改写别人已经基于其开发的公共提交，会迫使团队处理不必要的历史分叉。因此学习 rebase 必须同时学习何时不用它。

## 阶段 5：变基（rebase）和整理提交历史

### 学习目标

完成本阶段后，你应该能：

- 区分 merge 和 rebase。
- 在功能分支上同步主分支最新代码。
- 使用交互式 rebase 整理提交。
- 理解改写历史的风险。

### merge 和 rebase 的区别

起始历史：

```text
A --- B --- C  main
       \
        D --- E  feature
```

使用 merge：

```bash
git switch feature
git merge main
```

结果：

```text
A --- B --- C -------- M  feature
       \              /
        D --- E ------
```

使用 rebase：

```bash
git switch feature
git rebase main
```

结果：

```text
A --- B --- C  main
             \
              D' --- E'  feature
```

> **重点：** rebase 会把你的提交“复制”到新的基底上，因此提交哈希会变化。

> **难点：** rebase 不是移动原提交，而是创建新提交。旧提交通常还可以通过 `reflog` 找到。

### 日常功能分支 rebase

推荐流程：

```bash
git switch feature/user-profile
git fetch origin
git rebase origin/main
```

如果有冲突：

```bash
# 1. 打开冲突文件并解决
git status

# 2. 标记解决
git add <file>

# 3. 继续 rebase
git rebase --continue
```

跳过当前提交：

```bash
git rebase --skip
```

放弃整个 rebase：

```bash
git rebase --abort
```

> **易错：** rebase 冲突解决后使用 `git rebase --continue`，不是普通的 `git commit`。

### 交互式 rebase

整理最近 3 个提交：

```bash
git rebase -i HEAD~3
```

编辑器中可能出现：

```text
pick 1111111 Add user profile page
pick 2222222 Fix typo
pick 3333333 Add profile tests
```

可以改为：

```text
pick 1111111 Add user profile page
fixup 2222222 Fix typo
pick 3333333 Add profile tests
```

常用动作：

| 动作 | 含义 |
| --- | --- |
| `pick` | 保留提交 |
| `reword` | 修改提交信息 |
| `edit` | 停下来修改这个提交 |
| `squash` | 合并到上一个提交，并编辑提交信息 |
| `fixup` | 合并到上一个提交，丢弃当前提交信息 |
| `drop` | 删除这个提交 |

> **重点：** `squash` 和 `fixup` 都是合并提交。区别是 `squash` 会保留并编辑提交信息，`fixup` 直接丢弃当前提交信息。

### 修改历史提交内容

假设最近 3 个提交里第二个漏了文件：

```bash
git rebase -i HEAD~3
```

把对应提交从 `pick` 改为 `edit`：

```text
pick 1111111 Add API client
edit 2222222 Add user profile page
pick 3333333 Add tests
```

Git 停在第二个提交时：

```bash
# 修改或补充文件
git add <file>
git commit --amend
git rebase --continue
```

### 推送 rebase 后的分支

如果功能分支已经推送过，rebase 后提交哈希变化，普通 push 可能被拒绝：

```bash
git push --force-with-lease
```

> **重点：** `--force-with-lease` 会在覆盖远程分支前检查远程是否有别人新推送的提交，比 `--force` 更安全。

> **易错：** 不要对 `main`、`master`、`develop` 等公共分支做 rebase 后强推，除非团队明确允许。

### 阶段 5 要点

> **重点：** 在自己的功能分支上 rebase 很常见，在共享分支上 rebase 要非常谨慎。

> **难点：** rebase 期间可能连续出现多次冲突，因为 Git 在逐个重放提交。

> **易错：** 交互式 rebase 的顺序是从旧到新，不是从新到旧。
