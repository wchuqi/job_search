# Git 学习资料：阶段 4：回退、撤销和恢复

[返回索引](../Git学习资料.md)

## 理论导读：撤销前先判断“要撤的是哪个区域”

Git 撤销类命令最容易混乱，是因为它们名字相近但作用区域不同。工作区改坏了，要恢复文件内容；暂存区放错了，要把文件从下一次提交草稿里拿出来；本地提交做错了，可能要移动分支指针或补一个新提交；已经推送的错误，则要优先保留历史并用反向提交撤销。先定位区域，再选命令。

`restore` 主要处理文件内容和暂存状态，像把某个文件从指定快照里重新拿一份出来；`reset` 主要移动当前分支指针，并根据模式决定是否同时改暂存区和工作区；`revert` 不删除旧提交，而是创建一个新提交来抵消旧提交的修改。它们的差异不是语法细节，而是对历史的态度不同。

安全原则很直接：未共享历史可以谨慎 amend 或 reset，已共享历史优先 revert。`reset --hard` 和 `clean` 会丢弃本地未提交内容，执行前必须确认这些内容不需要，或者先建备份分支、stash、提交临时保存。

## 阶段 4：回退、撤销和恢复

### 学习目标

完成本阶段后，你应该能：

- 撤销工作区修改。
- 从暂存区撤回文件。
- 修改最近一次提交。
- 安全撤销已经推送的提交。
- 理解 `restore`、`reset`、`revert` 的区别。

### 撤销决策表

撤销决策表的核心是先定位修改所在层级。工作区修改还没有进入暂存区，风险范围最小；暂存区内容已经准备进入下一次提交，但还没有形成历史；本地提交已经生成 commit 对象并移动分支指针；远程共享提交则可能已经被别人拉取。层级越靠后，越要谨慎。

同样是“撤销”，有些命令只是恢复文件，有些命令会移动分支指针，有些命令会新增反向提交。先判断是否改写历史，再判断是否影响别人，才能避免把本地修正变成团队事故。

| 场景 | 推荐命令 | 是否改写历史 |
| --- | --- | --- |
| 文件改坏了，未暂存 | `git restore <file>` | 否 |
| 文件已经 `git add`，想取消暂存 | `git restore --staged <file>` | 否 |
| 最近一次提交信息写错，未推送 | `git commit --amend` | 是 |
| 最近一次提交内容漏了，未推送 | `git add <file>` + `git commit --amend` | 是 |
| 已推送提交需要撤销 | `git revert <commit>` | 否 |
| 想撤回最近提交但保留修改 | `git reset --soft HEAD~1` | 是 |
| 想撤回最近提交和暂存，但保留文件修改 | `git reset --mixed HEAD~1` | 是 |
| 想彻底丢弃最近提交和修改 | `git reset --hard HEAD~1` | 是，危险 |

> **重点：** 已经推送给别人使用的提交，优先用 `git revert`，不要随意 `reset`。

### 撤销工作区修改

工作区修改是还没进入暂存区的文件内容。`git restore <file>` 会用暂存区或 HEAD 中的版本覆盖工作区文件，相当于丢掉你手上这份未保存进 Git 历史的草稿。这个操作简单，但丢弃后如果没有其他备份，很难恢复。

执行前要确认这份草稿确实不需要。如果只是暂时不想处理，优先考虑 stash 或临时提交；如果只是想对比，可以先用 `git diff` 看清将被丢弃的内容。

修改 `src/calculator.js` 后发现写错了：

```javascript
function add(a, b) {
  return a - b; // 写错了
}
```

如果还没有暂存：

```bash
git restore src/calculator.js
```

恢复后文件回到最近一次提交或暂存区中的版本。

> **易错：** `git restore <file>` 会丢弃工作区修改。执行前确认这些修改不需要保留。

### 从暂存区撤回

取消暂存不是删除修改，而是把文件从“下一次提交草稿”里拿出来，工作区内容仍然保留。它适合发现暂存范围太大、想拆分提交、或者误把某个文件加入暂存区的场景。

理解这一点后，`restore --staged` 就不再可怕：它改变的是暂存区，不是工作区文件内容。撤回后可以继续编辑，也可以按更小范围重新 add。

如果已经执行：

```bash
git add src/calculator.js
```

想取消暂存：

```bash
git restore --staged src/calculator.js
```

这不会删除文件修改，只是把它从暂存区拿出来。

### 修改最近一次提交

`commit --amend` 不是在原提交上直接涂改，而是用新的提交替换当前分支末端的旧提交。新提交可能内容相似，但父提交、元数据或 tree 变化后，哈希会重新计算。旧提交不再被当前分支引用，但短期内通常能从 reflog 找回。

amend 适合修正尚未推送的最后一次提交，例如补漏文件、改提交信息、调整小错误。一旦旧提交已经被别人拉取，amend 会制造一条不同哈希的新历史，容易导致协作分叉。

提交信息写错：

```bash
git commit --amend -m "Add calculator multiply function"
```

漏提交一个文件：

```bash
git add docs/test-plan.md
git commit --amend
```

> **易错：** `commit --amend` 会生成一个新的提交替换旧提交。旧提交哈希会变化。如果旧提交已经推送并被别人基于它开发，不要随意 amend。

### reset 的三种常用模式

`reset` 的共同点是移动当前分支指针，区别在于是否同步改暂存区和工作区。`--soft` 只移动分支，保留暂存区和工作区；`--mixed` 移动分支并重置暂存区，保留工作区；`--hard` 同时重置分支、暂存区和工作区。

可以把 reset 想成把书签往回挪。soft 挪书签但保留准备提交的草稿，mixed 挪书签并把草稿拆回桌面，hard 挪书签并把桌面也清空到目标版本。hard 的风险最高，因为它会丢弃未提交文件修改。

假设历史如下：

```text
A --- B --- C  main
              ^
             HEAD
```

执行：

```bash
git reset --soft HEAD~1
```

结果：

- 分支回到 `B`。
- `C` 中的修改保留在暂存区。
- 适合“提交太早了，想重新提交”。

执行：

```bash
git reset --mixed HEAD~1
```

结果：

- 分支回到 `B`。
- `C` 中的修改保留在工作区。
- 暂存区清空。
- `--mixed` 是默认模式。

执行：

```bash
git reset --hard HEAD~1
```

结果：

- 分支回到 `B`。
- `C` 中的修改从工作区和暂存区消失。

> **易错：** `reset --hard` 是高风险命令。执行前至少先用 `git branch backup/...` 或确认 `git reflog` 能找回。

### revert：用新提交撤销旧提交

`revert` 的思路是保留历史事实，再新增一个反向修改。它不会让错误提交从历史里消失，而是明确记录“我们在这里撤销了它”。这对共享分支很重要，因为所有人的历史仍然能对齐，不需要强制改写远程分支。

revert 也可能冲突，尤其是被撤销提交后面又有其他提交修改了相关文件时。此时仍然需要人工判断最终业务结果。revert 不是低级命令，它是团队协作中最安全、最可审计的撤销方式之一。

假设某个已经推送的提交引入了错误：

```bash
git log --oneline
```

输出：

```text
f8a1c31 Add experimental payment flow
3d9a4aa Add calculator module
1c2b111 Add README
```

撤销第一个提交：

```bash
git revert f8a1c31
```

Git 会创建一个新的提交，内容是 `f8a1c31` 的反向修改。

```text
1c2b111 --- 3d9a4aa --- f8a1c31 --- a7e2d90
                         错误提交     revert 提交
```

> **重点：** `revert` 不会删除历史，而是新增历史，因此适合共享分支。

### 路径和版本写法

版本写法是恢复操作的定位语言。`HEAD~1`、`main`、`origin/main`、提交哈希、标签都可以作为“从哪里取内容”的来源。路径写法则决定“只恢复哪个文件”。把来源和路径组合起来，就能做到只取某个版本的某个文件，而不是大范围移动整个分支。

这类操作适合精细恢复。例如只想拿回上一版配置文件，不需要 reset 整个仓库；只想查看旧版本文件内容，可以用 `git show`，不必切换分支。精确定位能显著降低误操作风险。

常用引用：

```bash
HEAD        当前提交
HEAD~1      当前提交的第 1 个父提交
HEAD~2      当前提交往前数 2 个提交
main        main 分支指向的提交
origin/main origin/main 指向的提交
<hash>      指定提交哈希
```

查看某个版本的文件：

```bash
git show HEAD~1:src/calculator.js
```

把某个文件恢复到指定提交：

```bash
git restore --source=HEAD~1 src/calculator.js
```

### 阶段 4 要点

> **重点：** 撤销前先判断修改位于哪里：工作区、暂存区、本地提交、远程共享历史。

> **难点：** `reset` 移动分支指针，`revert` 新增反向提交，`restore` 主要恢复文件内容。

> **易错：** `git reset --hard` 和 `git clean -fd` 组合非常危险，会丢弃未提交修改和未跟踪文件。
