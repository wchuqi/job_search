# Git 学习资料：常见错误和处理方式

[返回索引](../Git学习资料.md)

## 理论导读：错误处理要先保留现场，再选择恢复路径

Git 常见错误大多不是因为命令罕见，而是因为操作者没有先判断当前状态。提交错分支、误删文件、冲突处理错、push 被拒绝、reset 后提交消失，本质上都和工作区、暂存区、引用指针或远程状态有关。恢复前先看现场，才能避免把小问题扩大成历史事故。

处理错误时优先使用可逆、可解释的动作。能用 `status` 看清就先看状态，能用 `reflog` 找回就不要重复 reset，能用新提交修复共享历史就不要强推改写。必要时先创建备份分支，让当前对象有一个名字指向它，再做下一步操作。

把错误分类后，命令会更清晰：文件内容错了用 restore 或 checkout 类操作，提交位置错了移动分支或 cherry-pick，历史公开后用 revert，远程不同步先 fetch 再处理分叉。

## 常见错误和处理方式

### 错误 1：提交到了错误分支

如果还没推送：

```bash
git branch feature/right-branch
git reset --hard HEAD~1
git switch feature/right-branch
```

更保守：

```bash
git branch backup/wrong-branch-commit
```

如果已经推送，先和团队确认处理方式，通常使用 `revert` 或新建 PR 修正。

> **易错：** 不要在公共分支上直接 `reset --hard` 后强推。

### 错误 2：忘记切分支，在 main 上开发了很多

如果还没提交：

```bash
git switch -c feature/task-name
git add .
git commit -m "Implement task"
```

如果已经在 `main` 提交，但没推送：

```bash
git branch feature/task-name
git reset --hard origin/main
git switch feature/task-name
```

> **重点：** 创建分支只是创建指针，所以可以在当前提交处补建分支。

### 错误 3：commit 后发现漏文件

未推送：

```bash
git add <missing-file>
git commit --amend
```

已推送：

```bash
git add <missing-file>
git commit -m "Add missing file"
git push
```

> **易错：** 已推送后不要随意 amend，除非确定没有别人基于该提交工作。

### 错误 4：误删文件

未提交删除：

```bash
git restore <file>
```

已经提交删除但未推送：

```bash
git reset --mixed HEAD~1
git restore <file>
```

已经推送删除：

```bash
git revert <delete-commit>
```

### 错误 5：误执行 reset

查看 reflog：

```bash
git reflog
```

恢复到误操作前：

```bash
git reset --hard HEAD@{1}
```

或者创建恢复分支：

```bash
git branch recover/reset-before HEAD@{1}
```

> **易错：** 连续执行多次 reset 会让判断更复杂。误操作后先看 `git reflog`，不要继续乱试。
