# Git 学习资料：日常工作完整流程

[返回索引](../Git学习资料.md)

## 理论导读：日常流程是在降低协作冲突和历史混乱

Git 日常流程不是机械执行命令列表，而是在每个时间点保持三个目标：开始工作前基于最新可信历史，开发过程中把改动拆成清晰提交，交付前让本地历史和远程协作规则一致。好的流程能减少冲突、减少误提交，也让代码评审更容易理解。

一个稳定的工作循环通常是：先 fetch 获取远程状态，再创建任务分支隔离改动；开发中频繁用 status 和 diff 检查状态，用小而明确的 commit 记录进展；推送前再同步目标分支，解决冲突或整理历史；最后通过 PR/MR 让团队审查并合并。

流程的核心不是“必须按某个团队模板”，而是让每次提交都能回答：为什么改、改了什么、是否包含无关内容、是否基于正确分支、是否会影响共享历史。

## 日常工作完整流程

### 开发新功能

```bash
git switch main
git pull --rebase
git switch -c feature/task-name

# 开发
git status
git diff
git add .
git diff --staged
git commit -m "Implement task"

# 同步主分支最新变化
git fetch origin
git rebase origin/main

# 推送并创建 PR/MR
git push -u origin feature/task-name
```

### 修复线上问题

```bash
git switch main
git pull --rebase
git switch -c hotfix/login-timeout

# 修复问题并测试
git add .
git commit -m "Fix login timeout"

git push -u origin hotfix/login-timeout
```

### 提交前检查清单

- `git status --short`：确认只提交本次任务相关文件。
- `git diff`：确认工作区剩余修改。
- `git diff --staged`：确认暂存区内容。
- 本地测试通过。
- 提交信息表达清楚。

### 推荐提交信息格式

简单格式：

```text
动词 + 做了什么
```

例子：

```text
Add user profile page
Fix login timeout
Refactor payment validation
Document Git workflow
```

团队常用 Conventional Commits：

```text
type(scope): summary
```

例子：

```text
feat(auth): add login timeout handling
fix(payment): handle empty callback payload
docs(git): add study material
refactor(api): simplify response parser
test(user): add profile validation tests
```

常见类型：

| 类型 | 含义 |
| --- | --- |
| `feat` | 新功能 |
| `fix` | 修复问题 |
| `docs` | 文档 |
| `style` | 格式调整，不影响逻辑 |
| `refactor` | 重构 |
| `test` | 测试 |
| `chore` | 构建、脚手架、依赖等杂项 |

> **重点：** 好提交信息回答“为什么这个提交存在”和“它做了什么”。
