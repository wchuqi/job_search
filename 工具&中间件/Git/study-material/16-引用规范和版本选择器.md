# Git 学习资料：引用规范和版本选择器

[返回索引](../Git学习资料.md)

## 理论导读：引用是给对象起名字，版本选择器是在对象图里导航

提交哈希准确但不方便，人更适合使用名字。分支、标签、远程跟踪分支都是引用，它们把可读名称映射到某个对象。`HEAD` 则表示当前检出位置，通常指向分支，也可能直接指向某个提交形成 detached HEAD。

版本选择器像对象图里的导航语言。`HEAD~1` 表示沿第一个父提交往前走一步，`HEAD^2` 表示选择合并提交的第二个父提交，`main..feature` 表示比较两个引用可达提交集合。掌握这些写法，才能准确地查看、恢复、比较和挑选历史。

引用的风险在于它们会移动。分支指针移动后，某些提交可能看似消失；但只要对象未被清理，reflog 或其他引用仍可能找回它们。

## 学习目标

完成本章后，你应该能：

- 区分 HEAD、分支、tag、远程跟踪分支。
- 理解 detached HEAD。
- 掌握常见版本选择器和提交范围表达式。
- 看懂 `HEAD~1`、`HEAD^2`、`A..B`、`A...B`。
- 知道 upstream / tracking branch 的作用。

## 一、引用是什么

引用是 Git 给对象起的名字。最常见的引用包括：

| 引用 | 示例 | 含义 |
| --- | --- | --- |
| 本地分支 | `refs/heads/main` | 本地分支指向的提交 |
| 远程跟踪分支 | `refs/remotes/origin/main` | 上次 fetch 后远程分支位置 |
| 标签 | `refs/tags/v1.0.0` | 某个发布点 |
| HEAD | `.git/HEAD` | 当前检出位置 |

查看所有引用：

```bash
git show-ref
```

查看某个引用对应哈希：

```bash
git rev-parse HEAD
git rev-parse main
git rev-parse origin/main
```

> **重点：** 分支名、tag 名、`origin/main` 最终都会解析到某个对象哈希。

## 二、HEAD 的常见状态

### 1. HEAD 指向分支

正常开发时：

```text
HEAD -> refs/heads/main -> commit
```

查看：

```bash
cat .git/HEAD
```

输出：

```text
ref: refs/heads/main
```

### 2. detached HEAD

当你直接检出某个提交：

```bash
git switch --detach <commit>
```

此时 HEAD 直接指向提交，不在任何分支上。

> **易错：** detached HEAD 状态下提交不会被分支保存。要保留工作，应创建分支：

```bash
git switch -c experiment/save-work
```

## 三、父提交选择器

假设历史：

```text
A --- B --- C  main
```

常见写法：

| 写法 | 含义 |
| --- | --- |
| `HEAD` | 当前提交 |
| `HEAD~1` | 当前提交往前第 1 代 |
| `HEAD~2` | 当前提交往前第 2 代 |
| `HEAD^` | 当前提交的第 1 个父提交 |
| `HEAD^1` | 当前提交的第 1 个父提交 |
| `HEAD^2` | 当前提交的第 2 个父提交，常用于 merge commit |

merge commit 示例：

```text
A --- B ------ M
     \        /
      C -----
```

`M^1` 是当前分支父提交，`M^2` 是被合并分支父提交。

> **难点：** `~` 表示沿第一个父提交连续向前走，`^2` 表示选择第二个父提交。

## 四、提交范围：双点和三点

### 1. `A..B`

表示在 B 中有、但 A 中没有的提交。

```bash
git log main..feature
```

常用于查看 feature 比 main 多了哪些提交。

### 2. `A...B`

在 `git log` 中，表示 A 和 B 对称差集：A 或 B 独有、但不是双方共有的提交。

```bash
git log main...feature
```

在 `git diff` 中，三点通常表示比较 merge base 到 B 的差异：

```bash
git diff main...feature
```

> **易错：** `git log A...B` 和 `git diff A...B` 的语义不完全一样。面试和排障时要说清楚具体命令上下文。

## 五、路径限制 pathspec

很多命令可以只针对某些文件：

```bash
git log -- src/app.js
git diff -- README.md
git restore -- src/app.js
```

`--` 用来分隔 revision 和 path，避免歧义。

> **重点：** 当分支名和文件名可能重名时，使用 `--` 可以让命令更明确。

## 六、upstream 和 tracking branch

本地分支可以追踪远程分支：

```bash
git push -u origin feature/login
```

查看：

```bash
git branch -vv
```

设置 upstream：

```bash
git branch --set-upstream-to=origin/main main
```

有 upstream 后可以直接：

```bash
git pull
git push
```

> **易错：** upstream 不是远程仓库本身，而是“当前本地分支默认拉取/推送对应的远程分支”。

## 七、引用日志 reflog

reflog 记录 HEAD 或分支引用移动历史：

```bash
git reflog
git reflog show main
```

常见用途：

- 找回误 reset 前的提交。
- 找回误删分支。
- 找到 rebase 前的位置。

```bash
git branch recover/before-rebase HEAD@{3}
```

> **重点：** reflog 是本地记录，不会随 push 同步给别人。

## 八、常见引用写法速查

| 写法 | 含义 |
| --- | --- |
| `@` | `HEAD` 的简写 |
| `@~2` | `HEAD~2` |
| `main@{yesterday}` | main 昨天的位置 |
| `HEAD@{1}` | HEAD 上一次位置 |
| `v1.0.0^{}` | 解引用 tag 到其指向对象 |
| `HEAD^{tree}` | 当前提交对应 tree |
| `:/message` | 最近提交信息匹配 message 的提交 |

例子：

```bash
git show HEAD@{1}
git diff HEAD~2..HEAD
git show :/fix-login
```

## 练习

```bash
git log --oneline --graph --decorate --all
git rev-parse HEAD
git rev-parse HEAD~1
git rev-parse origin/main
git log origin/main..HEAD
git diff origin/main...HEAD
```

回答：

- 当前分支比远程多了哪些提交？
- 当前分支和远程的共同祖先是什么？
- `HEAD~1` 和 `HEAD^` 在当前提交上是否一样？

## 验收

- 能解释 `main`、`origin/main`、远程服务器上的 `main` 的区别。
- 能正确使用 `A..B` 和 `A...B`。
- 能解释 detached HEAD 的风险。
- 能通过 reflog 找回历史位置。
