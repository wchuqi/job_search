# Git 学习资料：最终学习验收

[返回索引](../Git学习资料.md)

## 理论导读：验收不是看会不会背命令，而是看能否解释状态和风险

Git 学习合格的标志，是你能在真实仓库里判断当前状态、选择低风险操作、解释命令后果，并在出错时恢复。命令记忆会随着场景变化失效，但对象、引用、工作区、暂存区和远程状态这些模型稳定不变。

最终验收应覆盖三类能力：日常开发能顺畅提交和协作；复杂操作前能判断是否改写历史、是否影响团队；出现错误时能先保留现场，再用 reflog、restore、revert、reset 等工具恢复。

## 使用方式

这份验收清单用于判断你是否已经系统掌握 Git。不要只回答“用什么命令”，还要能解释原因、风险和适用场景。

## 一、基础和心智模型

- 能解释 Git、GitHub、GitLab、Gitee 的区别。
- 能解释工作区、暂存区、本地仓库、远程仓库。
- 能解释 `git add` 和 `git commit` 的区别。
- 能说明 Git 为什么是分布式版本控制。
- 能解释提交是快照而不是简单补丁。

## 二、对象模型和引用

- 能解释 blob、tree、commit、tag 四类对象。
- 能解释分支为什么很轻量。
- 能解释 HEAD、branch、tag、`origin/main` 的区别。
- 能解释 detached HEAD 的风险。
- 能正确使用 `HEAD~1`、`HEAD^2`、`A..B`、`A...B`。

## 三、提交组织

- 能用 `git status` 判断文件状态。
- 能解释 `git diff` 和 `git diff --staged`。
- 能用 `git add -p` 拆分提交。
- 能写清晰提交信息。
- 能说明 `commit --amend` 为什么会改变哈希。

## 四、分支、合并和冲突

- 能创建、切换、删除分支。
- 能解释 fast-forward merge 和 three-way merge。
- 能解决一次真实冲突。
- 能说明 merge 冲突和 rebase 冲突处理命令差异。
- 能解释 ours / theirs 在不同上下文中的风险。

## 五、远程协作

- 能解释 clone、fetch、pull、push 的区别。
- 能解释本地分支、远程跟踪分支、远程服务器分支。
- 能处理 push 被拒绝。
- 能说明 `push -u` 的作用。
- 能解释 fork 工作流中的 `origin` 和 `upstream`。
- 能描述 PR/MR、review、CI、protected branch 的关系。

## 六、撤销和恢复

- 能说清楚 `restore`、`reset`、`revert` 的区别。
- 能解释 `reset --soft`、`--mixed`、`--hard`。
- 能说明共享分支为什么优先 `revert`。
- 能用 `reflog` 找回误 reset 或误删分支。
- 能说明 `git clean -fd` 的风险。

## 七、变基和历史整理

- 能说明 merge 和 rebase 的差异。
- 能解释 rebase 为什么改写提交哈希。
- 能用 `rebase -i` 整理提交。
- 能说明 `squash` 和 `fixup` 的区别。
- 能解释 `--force-with-lease` 为什么比 `--force` 安全。
- 能说明什么时候不应该 rebase。

## 八、查询和排障

- 能用 `git log -- <file>` 查文件历史。
- 能用 `git diff origin/main...HEAD` 查看 PR 差异。
- 能用 `git log -S` 或 `git log -G` 查代码引入历史。
- 能用 `git blame` 找上下文。
- 能用 `git bisect` 定位问题提交。

## 九、忽略、属性和安全

- 能解释 `.gitignore` 为什么不影响已跟踪文件。
- 能用 `git check-ignore -v` 排查忽略规则。
- 能说明 `.gitattributes` 的作用。
- 能识别换行符造成的大量 diff。
- 能说明敏感信息误提交后的处理顺序。
- 能解释 hooks、签名和 CI 的边界。

## 十、大型仓库和维护

- 能解释 Git LFS 解决什么问题。
- 能说明 submodule 和 subtree 的区别。
- 能使用 worktree 处理并行任务。
- 能说明 sparse checkout 和 shallow clone 的适用场景。
- 能解释 `git gc`、`git fsck`、`git count-objects` 的作用。
- 能说明 `push --mirror` 的风险。

## 十一、实践验收任务

完成一个练习仓库，至少包含：

- 15 次以上提交。
- 3 个功能分支。
- 1 次 merge 冲突并解决。
- 1 次 rebase 冲突并解决。
- 1 次误操作后用 reflog 恢复。
- 1 次 tag 发布。
- 1 个 `.gitignore` 和 `.gitattributes`。
- 1 次 `git bisect` 定位问题。
- 1 次 worktree 或 submodule / LFS 练习。

## 最终标准

如果你能解释上述问题，并在真实仓库中完成实践任务，说明你已经从“会用 Git 命令”进入“理解 Git 工作机制并能处理团队协作问题”的阶段。
