# Git 学习资料：阶段 3：远程仓库协作

[返回索引](../Git学习资料.md)

## 理论导读：远程协作是在同步两份对象数据库和引用

远程仓库不是神秘的“云端 Git”，它本质上也是一份 Git 对象数据库，只是放在 GitHub、GitLab、Gitee 或公司服务器上。`origin` 是远程地址的默认别名，`origin/main` 是本地记录的远程分支位置，`main` 则是你的本地分支。协作时要分清这三者，否则很容易误解 pull、push 和 fetch。

`git fetch` 只是把远程新增对象和远程引用状态拿回来，不会直接改你的本地工作分支；`git pull` 通常等于 fetch 后再 merge 或 rebase；`git push` 则把本地提交和分支指针更新发送给远程。可以把 fetch 想成“拿回最新情报”，pull 是“拿回并合入当前工作”，push 是“把自己的进展发布出去”。

团队协作的风险在于共享历史。你本地分支上随意 reset 或 rebase 影响的是自己；一旦推送到共享分支，别人可能已经基于这些提交继续工作。远程协作章节的重点不是会敲命令，而是知道哪些操作会改变公共引用，什么时候需要先 fetch，什么时候需要通过 Pull Request 让团队审查。

## 阶段 3：远程仓库协作

### 学习目标

完成本阶段后，你应该能：

- 连接远程仓库。
- 理解 clone、fetch、pull、push 的区别。
- 处理推送被拒绝。
- 参与 Pull Request / Merge Request 流程。

### 远程仓库基本概念

远程仓库是团队共享的 Git 仓库。常见平台包括 GitHub、GitLab、Gitee。

`origin` 是默认远程仓库名，不是固定必须叫 `origin`，只是约定俗成。

查看远程仓库：

```bash
git remote -v
```

添加远程仓库：

```bash
git remote add origin <repo-url>
```

修改远程地址：

```bash
git remote set-url origin <new-repo-url>
```

删除远程：

```bash
git remote remove origin
```

### clone、fetch、pull、push

| 命令 | 作用 | 是否修改工作区 |
| --- | --- | --- |
| `git clone` | 下载整个远程仓库 | 是，会创建工作目录 |
| `git fetch` | 拉取远程提交和分支信息 | 否，不自动合并 |
| `git pull` | 拉取并合并或变基 | 是 |
| `git push` | 推送本地提交到远程 | 否，对远程有影响 |

> **重点：** `git pull` 大致等价于 `git fetch` + `git merge`，如果配置或加参数也可以是 `git fetch` + `git rebase`。

> **易错：** 不要把 `fetch` 当作“没用的 pull”。`fetch` 是更安全的同步方式，因为它不会自动改你的工作区。

### 第一次推送本地仓库

```bash
git remote add origin <repo-url>
git branch -M main
git push -u origin main
```

`-u` 表示建立本地分支和远程分支的追踪关系。之后可以直接：

```bash
git push
git pull
```

查看追踪关系：

```bash
git branch -vv
```

### 模拟两个人协作

克隆两份目录：

```bash
git clone <repo-url> alice-repo
git clone <repo-url> bob-repo
```

Alice 修改并推送：

```bash
cd alice-repo
git switch -c feature/add-title

# 修改 README.md
git add README.md
git commit -m "Add README title"
git push -u origin feature/add-title
```

Bob 修改主分支并推送：

```bash
cd bob-repo
git switch main

# 修改 README.md
git add README.md
git commit -m "Update README intro"
git push
```

Alice 同步主分支：

```bash
git fetch origin
git rebase origin/main
```

如果产生冲突：

```bash
# 手动解决冲突
git add README.md
git rebase --continue
```

如果想放弃本次 rebase：

```bash
git rebase --abort
```

### push 被拒绝怎么办

常见错误：

```text
! [rejected]        main -> main (fetch first)
error: failed to push some refs
```

原因：远程分支有你本地没有的新提交，Git 不允许你直接覆盖它。

推荐处理：

```bash
git fetch origin
git rebase origin/main
git push
```

如果团队使用 merge 风格：

```bash
git pull
git push
```

> **重点：** push 被拒绝不是异常情况，而是 Git 在保护远程历史。

> **易错：** push 被拒绝后不要马上 `git push --force`。这可能覆盖别人的提交。

### Pull Request / Merge Request 流程

推荐流程：

```bash
git switch main
git pull --rebase
git switch -c feature/user-profile

# 开发
git add .
git commit -m "Add user profile page"

git fetch origin
git rebase origin/main
git push -u origin feature/user-profile
```

然后在平台上创建 PR/MR，通常需要：

- 描述改动目的。
- 说明测试方式。
- 关联需求或 issue。
- 等待代码评审。
- 通过 CI 后合并。

PR 描述模板：

```markdown
## 变更内容

- 添加用户资料页
- 支持展示昵称和邮箱

## 测试方式

- 本地运行单元测试
- 手动验证资料页展示

## 风险点

- 依赖接口返回字段稳定
```

### 阶段 3 要点

> **重点：** 日常协作推荐先 `fetch`，看清楚远程变化后再 `rebase` 或 `merge`。

> **难点：** 本地分支、远程分支、远程追踪分支是三个概念。例如 `main`、`origin/main`、远程服务器上的 `main` 并不是同一个东西。

> **易错：** `origin/main` 是本地保存的远程分支快照，只有执行 `git fetch origin` 后才会更新。
