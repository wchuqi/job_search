# Docker学习资料：故障案例和排障剧本深度版

[返回索引](../Docker学习资料.md)

## 学习目标

- 掌握 Docker 故障的标准分层排查方法。
- 能根据现象选择最短诊断路径。
- 能保留现场并避免高风险清理。

## 总体排障框架

先问四个问题：

1. 对象是什么：image、container、network、volume、daemon、registry、compose project。
2. 状态是什么：created、running、exited、restarting、healthy、unhealthy。
3. 最近变化是什么：镜像、配置、端口、volume、权限、宿主机、网络、registry。
4. 数据风险是什么：是否涉及 volume、数据库、日志和未备份文件。

通用命令：

```bash
docker ps -a
docker logs --tail 200 <container>
docker inspect <container>
docker events --since 1h
docker system df -v
docker compose config
docker compose ps
```

## 剧本 1：容器启动后立即退出

### 现象

`docker ps` 看不到容器，`docker ps -a` 显示 `Exited`。

### 排查

```bash
docker logs app
docker inspect app --format '{{.State.ExitCode}} {{.State.Error}} {{.Config.Entrypoint}} {{.Config.Cmd}}'
```

### 判断

- 退出码 0：命令正常执行完，可能主进程不是前台服务。
- 退出码 1/2：应用配置或命令错误。
- 退出码 126/127：命令无权限或不存在。
- 退出码 137：可能被 kill 或 OOM。

### 修复

- 修正 ENTRYPOINT/CMD。
- 补齐环境变量或配置文件。
- 确认应用以前台运行。
- OOM 时调大限制或调整应用内存参数。

## 剧本 2：端口映射了但访问不到

### 排查顺序

```bash
docker ps
docker port app
docker exec app ss -lntp
docker logs app
ss -lntp | grep 8080
```

### 常见原因

- 应用只监听 `127.0.0.1`。
- 容器端口写错。
- 宿主机端口被占用。
- 防火墙或安全组阻断。
- Docker Desktop 端口转发异常。

### 修复原则

让应用监听 `0.0.0.0`，确认 `HOST:CONTAINER` 端口方向正确，再检查宿主机防火墙。

## 剧本 3：Compose 中服务名连接失败

### 排查

```bash
docker compose ps
docker compose exec app getent hosts db
docker compose exec app nc -vz db 5432
docker network inspect <project>_default
```

### 常见原因

- 服务不在同一网络。
- 服务名写错。
- 数据库还没 ready。
- 应用使用 `localhost`。
- Compose 项目名不同，网络不同。

### 修复

- 使用 service name。
- 增加 healthcheck 和应用重试。
- 确认 `docker compose config` 输出。

## 剧本 4：volume 权限错误

### 现象

应用报 `permission denied`，数据库无法初始化，非 root 容器写不了目录。

### 排查

```bash
docker exec app id
docker inspect app --format '{{json .Mounts}}'
ls -ln /host/path
```

### 修复

- 统一应用 UID/GID。
- 修改宿主机目录所有者。
- 改用 named volume。
- 避免粗暴 `chmod 777`。

## 剧本 5：磁盘空间被 Docker 占满

### 排查

```bash
docker system df -v
docker ps -a --size
docker builder du
docker inspect app --format '{{.LogPath}}'
```

### 判断

- 镜像多：清理未使用镜像。
- 构建缓存大：清理 builder cache。
- 日志大：配置日志轮转。
- volume 大：确认业务数据，不能盲删。

### 风险提示

`docker system prune -a --volumes` 是高风险命令，会删除未使用镜像、容器、网络、构建缓存和 volume。生产中执行前必须确认 volume 不含业务数据。

## 剧本 6：镜像构建很慢或缓存不命中

### 排查

```bash
docker build --progress=plain -t app:local .
docker image history app:local
```

### 常见原因

- `COPY . .` 放在依赖安装前。
- `.dockerignore` 缺失导致上下文巨大。
- lock 文件变化频繁。
- CI 没有导入导出构建缓存。
- 构建命令中使用了总是变化的参数。

### 修复

- 依赖文件前置。
- 加 `.dockerignore`。
- 使用 BuildKit cache mount。
- CI 配置 registry cache 或本地缓存。

## 剧本 7：容器 OOM 或性能异常

### 排查

```bash
docker stats
docker inspect app --format '{{.State.OOMKilled}} {{.HostConfig.Memory}} {{.HostConfig.NanoCpus}}'
dmesg | grep -i oom
```

### 常见原因

- 容器 memory limit 过小。
- Java、Node、Python 运行时没有感知限制或配置不当。
- 日志、缓存、连接池过大。
- 数据库放在资源过小的容器中。

### 修复

- 调整容器限制和应用运行时参数。
- 限制连接池、缓存和并发。
- 加监控和告警。

## 剧本 8：registry 拉取失败

### 排查

```bash
docker login registry.example.com
docker pull registry.example.com/team/app:tag
docker inspect registry.example.com/team/app:tag
```

### 常见原因

- 未登录或 token 过期。
- tag 不存在或被覆盖。
- TLS 证书错误。
- 代理或 DNS 问题。
- 多架构镜像不包含当前平台。

## 最终验收

- 能对每类故障写出“现象、命令、判断、修复、风险”。
- 遇到故障先保留日志、inspect 和状态。
- 涉及 volume 和 prune 时先评估数据风险。

## 重点

- Docker 排障要分层，不能只靠重启。
- 先证据，后修改。
- 所有删除类操作都要先识别数据对象。
