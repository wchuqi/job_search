# Docker学习资料：生产发布、供应链和可观测性深度解析

[返回索引](../Docker学习资料.md)

## 学习目标

- 理解镜像作为生产制品的治理要求。
- 掌握 tag、digest、SBOM、扫描、签名、回滚和保留策略。
- 能设计 Docker 化服务的可观测性和发布验收。

## 理论导读

生产 Docker 化的关键不是把应用放进容器，而是建立可追溯、可审计、可回滚的制品链。镜像一旦进入生产，就代表源码、依赖、基础镜像、构建参数和运行配置共同形成的结果。发布失败也不一定能简单回滚镜像，因为数据库迁移、外部配置和协议兼容可能已经变化。

## 制品链路

```text
source commit
  -> test
  -> docker build
  -> image scan
  -> SBOM/provenance
  -> registry push
  -> deployment by tag/digest
  -> runtime logs/metrics/traces
```

每个环节都应留下可查询记录。

## tag 和 digest 策略

推荐同时使用多种 tag：

- `1.8.3`：人类可读版本。
- `sha-abc1234`：源码提交追溯。
- `2026.06.06.1`：构建批次。
- `staging`、`prod`：环境指针，只作为辅助，不作为唯一依据。

生产部署记录应包含 digest：

```text
image: registry.example.com/team/app@sha256:...
source: git commit abc1234
build: 2026.06.06.1
config: config version 42
database migration: 202606061200_add_index
```

## SBOM 和漏洞扫描

SBOM 是软件物料清单，记录镜像里有哪些包和版本。漏洞扫描根据包版本匹配已知漏洞。

实践边界：

- 扫描结果不是绝对风险结论，需要结合是否可触达、是否运行、是否有修复版本。
- 基础镜像要定期重建，不要只在业务代码变更时构建。
- 高危漏洞应有处理 SLA。
- 扫描不替代密钥扫描和运行时安全。

## Secret 泄漏路径

常见泄漏位置：

- Dockerfile 中 `ENV PASSWORD=...`。
- `ARG TOKEN=...` 出现在构建历史或日志。
- `COPY . .` 把 `.env`、私钥、云凭据复制进镜像。
- CI 日志打印 secret。
- 构建缓存保存私有依赖配置。

正确做法：

- `.dockerignore` 排除敏感文件。
- 使用 BuildKit secret mount。
- 使用 CI secret 注入，不打印。
- 生产使用平台 secret 管理。

## 可观测性

Docker 化服务至少需要：

- 日志：stdout/stderr，结构化，带 request id。
- 指标：CPU、内存、重启次数、请求量、错误率、延迟。
- 健康检查：liveness 和 readiness 思路分离。
- 事件：部署版本、配置版本、数据库迁移。
- 资源：磁盘、volume、日志文件、构建缓存。

> **重点：** 容器健康不等于业务健康。进程活着、端口开着、依赖可用、业务查询正常是不同层级。

## 发布和回滚风险

回滚前要确认：

- 数据库 migration 是否可逆。
- 新版本写入的数据旧版本是否能读。
- 配置项是否兼容。
- 外部 API 或消息格式是否兼容。
- 镜像 digest 是否仍在 registry 保留。
- 旧版本依赖的 secret 和证书是否仍有效。

## 练习

1. 为一个服务设计镜像标签和发布记录模板。
2. 写一份 `.dockerignore`，明确排除 secret、日志、构建产物和 VCS 目录。
3. 生成镜像 SBOM 或至少列出基础镜像、运行时包、应用依赖。
4. 设计一次失败发布的回滚检查单。

## 验收

- 能解释为什么生产发布要记录 digest。
- 能指出至少 5 种 secret 泄漏路径。
- 能设计 Docker 服务上线前的日志、指标、健康检查和回滚要求。

## 难点

- 漏洞扫描和生产风险之间不是一一对应，需要治理流程。
- 回滚常被数据库和数据格式阻塞，不能只看镜像。

## 易错

> **易错：** 认为镜像可回滚就等于系统可回滚。
>
> 正确做法：把镜像、配置、数据库迁移、外部依赖和数据兼容一起纳入发布设计。
