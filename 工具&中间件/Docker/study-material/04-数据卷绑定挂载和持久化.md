# Docker学习资料：数据卷、绑定挂载和持久化

[返回索引](../Docker学习资料.md)

## 学习目标

- 区分容器可写层、volume、bind mount、tmpfs。
- 能为数据库、上传目录、配置文件选择正确挂载方式。
- 掌握数据备份、迁移和权限问题排查。

## 理论导读

容器可写层适合临时状态，不适合保存重要数据。持久化数据应放在 Docker volume、宿主机绑定挂载或外部存储。volume 由 Docker 管理，适合数据库和服务数据；bind mount 直接映射宿主机路径，适合开发调试和挂载配置。

## 核心心智模型

容器文件系统分三类：

- 镜像层：只读，来自镜像。
- 容器可写层：随容器生命周期存在，删除容器通常删除它。
- 外部挂载：volume 或宿主机路径，独立于容器生命周期。

## 知识点详解

### volume

```powershell
docker volume create pgdata
docker run -d --name pg -e POSTGRES_PASSWORD=demo -v pgdata:/var/lib/postgresql/data postgres:16
docker volume inspect pgdata
```

volume 的宿主机实际位置由 Docker 管理。优点是生命周期清晰、跨容器复用方便、适合备份。

### bind mount

```powershell
docker run --rm -v ${PWD}:/work -w /work alpine:3.20 ls
```

bind mount 依赖宿主机路径，适合开发时把源码挂进容器。生产中使用时要注意路径存在性、权限和误覆盖。

### tmpfs

tmpfs 把数据放在内存中，容器停止后消失，适合临时敏感文件或缓存。

### 权限问题

容器内用户 ID 和宿主机文件所有者是数字 ID 对应，不是用户名对应。非 root 容器访问 bind mount 时，常见问题是 UID/GID 不匹配。

## 例子

备份 volume：

```powershell
docker run --rm -v pgdata:/data -v ${PWD}:/backup alpine:3.20 tar czf /backup/pgdata.tar.gz -C /data .
```

恢复 volume：

```powershell
docker volume create pgdata2
docker run --rm -v pgdata2:/data -v ${PWD}:/backup alpine:3.20 tar xzf /backup/pgdata.tar.gz -C /data
```

## 练习

1. 用 volume 启动 PostgreSQL，删除容器后重新挂载同一个 volume。
2. 用 bind mount 挂载本地配置文件到 Nginx。
3. 故意用非 root 用户访问宿主机只读目录，观察权限错误。

## 验收

- 能解释容器删除后哪些数据会保留。
- 能选择 volume 或 bind mount。
- 能备份和恢复 volume。

## 重点

- 数据库数据不要放在容器可写层。
- bind mount 会让宿主机路径直接影响容器，误操作风险更高。
- 权限排障要看 UID/GID。

## 难点

- Windows/macOS Docker Desktop 的文件挂载经过虚拟化层，性能和权限语义与 Linux 不完全一致。
- 挂载到容器已有目录会遮挡镜像内原目录内容。

## 易错

> **易错：** 挂载空目录到应用配置目录，导致镜像内默认配置被遮住。
>
> 正确做法：明确挂载单个文件或先初始化目录内容。
