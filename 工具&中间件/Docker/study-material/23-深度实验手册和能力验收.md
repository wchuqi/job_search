# Docker学习资料：深度实验手册和能力验收

[返回索引](../Docker学习资料.md)

## 使用方式

这份实验手册用于把 Docker 从“会用命令”推进到“理解机制”。建议在 Linux 虚拟机或 WSL2 中完成；Docker Desktop 可以完成大部分实验，但 runtime、iptables、overlay2 路径观察可能和原生 Linux 不同。

## 实验 1：追踪 `docker run` 对象创建

### 目标

理解容器启动前后镜像、容器、网络、挂载和进程状态变化。

### 步骤

```bash
docker pull nginx:alpine
docker image inspect nginx:alpine
docker run -d --name lab-web -p 8080:80 nginx:alpine
docker inspect lab-web
docker logs lab-web
docker exec lab-web ps -ef
docker rm -f lab-web
```

### 验收

- 能指出 Entrypoint、Cmd、Mounts、NetworkSettings、State.Pid。
- 能说明 `-p 8080:80` 的方向。

## 实验 2：构建缓存和上下文

### 目标

观察 Dockerfile 顺序如何影响缓存。

### 步骤

1. 创建一个带依赖文件和源码文件的项目。
2. Dockerfile A：先 `COPY . .`，再安装依赖。
3. Dockerfile B：先复制依赖文件并安装，再复制源码。
4. 修改源码，比较两次构建时间和 `--progress=plain` 输出。

### 验收

- 能解释哪个步骤缓存命中，哪个步骤失效。
- 能写出合理 `.dockerignore`。

## 实验 3：overlay2 和 copy-on-write

### 目标

理解容器修改文件如何进入可写层。

### 步骤

```bash
docker run -d --name cow alpine:3.20 sleep 3600
docker exec cow sh -c 'echo hello > /tmp/a.txt'
docker diff cow
docker rm -f cow
```

Linux 原生环境可进一步查看 Docker root dir：

```bash
docker info | grep "Docker Root Dir"
```

### 验收

- 能解释 `docker diff` 中 A、C、D 的含义。
- 能说明容器可写层为什么不适合业务数据。

## 实验 4：网络和 DNS

### 目标

掌握用户自定义网络和服务名解析。

### 步骤

```bash
docker network create labnet
docker run -d --name lab-redis --network labnet redis:7-alpine
docker run --rm --network labnet redis:7-alpine redis-cli -h lab-redis ping
docker network inspect labnet
docker rm -f lab-redis
docker network rm labnet
```

### 验收

- 能解释为什么容器 IP 不适合写进配置。
- 能说明同网络容器为什么不需要宿主机端口映射。

## 实验 5：volume 持久化和挂载遮挡

### 目标

理解 volume 生命周期和挂载遮挡。

### 步骤

```bash
docker volume create labdata
docker run --rm -v labdata:/data alpine:3.20 sh -c 'echo v1 > /data/value.txt'
docker run --rm -v labdata:/data alpine:3.20 cat /data/value.txt
docker volume rm labdata
```

再准备一个镜像内有 `/data/default.txt` 的 Dockerfile，把空 volume 挂到 `/data`，观察初始化和遮挡行为。

### 验收

- 能解释删除容器与删除 volume 的不同影响。
- 能说明为什么挂载可能遮住镜像内文件。

## 实验 6：非 root、capability 和只读文件系统

### 目标

验证最小权限运行对应用的影响。

### 步骤

```bash
docker run --rm --read-only --tmpfs /tmp --cap-drop ALL nginx:alpine
```

如果启动失败，观察日志，判断需要哪些可写路径或 capability。

### 验收

- 能区分文件系统只读、UID 权限、capability 缺失三类问题。
- 能给应用列出最小可写目录。

## 实验 7：Compose 多环境解析

### 目标

理解 Compose 多文件、变量和最终配置。

### 步骤

```bash
docker compose -f compose.yml -f compose.dev.yml config
docker compose --profile debug config
docker compose up -d
docker compose down
```

### 验收

- 能说明 `.env`、`env_file`、`environment` 的区别。
- 能通过 `config` 输出定位变量替换问题。

## 实验 8：故障注入

### 目标

形成真实排障能力。

### 故障清单

1. ENTRYPOINT 写错导致容器退出。
2. 应用只监听 `127.0.0.1` 导致端口发布访问失败。
3. Web 使用 `localhost` 连接数据库失败。
4. volume 权限不匹配导致写入失败。
5. 日志无限增长导致磁盘告警。
6. memory limit 过小导致 OOM。

### 验收

每个故障写一份记录：

```text
现象：
影响：
关键命令：
证据：
根因：
修复：
预防：
```

## 最终能力验收

- 能画出 Docker 架构链路图。
- 能写生产可用 Dockerfile，包含多阶段构建、非 root、健康检查和 `.dockerignore`。
- 能写 Compose 文件，包含 Web、DB、Redis、网络、volume、healthcheck 和 profile。
- 能解释 overlay2、copy-on-write、volume、bind mount 的区别。
- 能解释 bridge 网络、DNS、端口发布和容器访问宿主机的路径。
- 能设计镜像 tag、digest、SBOM、扫描和回滚策略。
- 能完成至少 6 类故障注入和排障报告。

## 重点

- 实验必须写记录，不只跑命令。
- 每次删除 volume 或 prune 前都要写明数据风险。
- Docker 深度能力来自机制理解和故障复盘。
