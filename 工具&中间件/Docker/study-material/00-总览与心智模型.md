# Docker学习资料：总览与心智模型

[返回索引](../Docker学习资料.md)

## 学习目标

- 理解 Docker 的核心价值：把应用及其运行依赖打包成可复制、可分发、可运行的单元。
- 建立镜像、容器、仓库、Dockerfile、网络、卷之间的整体模型。
- 明确 Docker 与虚拟机、Kubernetes、传统部署脚本的边界。

## 理论导读

Docker 解决的是“我的机器可以运行，换个环境就不行”的一致性问题。它不把整个操作系统都复制一份，而是把应用运行所需的文件系统、启动命令、环境变量、依赖库和元数据打包成镜像。容器启动时，Docker 在宿主机内核上为进程提供隔离的视图，包括文件系统、网络、进程号、用户、资源限制等。

Docker 的关键不是“省事地启动软件”，而是把运行环境变成可版本化的工程产物。一个团队只要约定镜像构建方式、镜像标签、配置注入和数据持久化方式，就能让开发、测试和部署拥有相同的运行基础。

## 核心心智模型

可以把 Docker 想成三层：

1. 镜像：只读模板，像一个应用的“压缩运行包”，包含文件系统层、默认命令、环境变量和暴露端口等元数据。
2. 容器：镜像运行后的实例，本质上是一组被 namespace 和 cgroup 约束的进程，加上一层可写层。
3. 仓库：镜像的分发中心，用标签或 digest 找到某个镜像版本。

容器不是小型虚拟机。虚拟机虚拟硬件并运行独立内核，容器共享宿主机内核，只隔离进程看到的系统资源。因此 Linux 容器依赖 Linux 内核能力；在 Windows 或 macOS 上使用 Docker Desktop 时，中间通常有一层 Linux 虚拟机。

## 知识点详解

### 镜像和容器的关系

镜像是静态产物，容器是运行态实例。同一个镜像可以启动多个容器，每个容器都有自己的进程、网络命名空间和可写层。删除容器不会删除镜像，删除镜像也不能影响已经运行的容器文件系统视图，除非镜像层还被其他对象引用。

### Docker Engine 的组成

- Docker CLI：用户输入命令的客户端，如 `docker run`。
- Docker daemon：后台服务，负责创建容器、拉取镜像、管理网络和卷。
- containerd/runc：更底层的容器运行时组件，真正负责容器生命周期和 OCI 运行规范。

### Docker 适用场景

- 本地开发环境统一。
- CI 测试环境隔离。
- 微服务或后台服务打包发布。
- 快速启动中间件，如 Redis、PostgreSQL、Nginx。
- 作为 Kubernetes 工作负载镜像构建和调试基础。

### Docker 不适合什么

- 需要强虚拟化隔离的高风险多租户环境，通常还要额外沙箱或虚拟机边界。
- 把有状态数据库随意放在临时容器可写层里。
- 在容器里长期手工运维，把容器当传统服务器。

## 例子

```powershell
docker run --name web-demo -p 8080:80 nginx:alpine
docker ps
docker logs web-demo
docker exec -it web-demo sh
docker rm -f web-demo
```

这个例子里，`nginx:alpine` 是镜像，`web-demo` 是容器，`-p 8080:80` 表示宿主机 8080 端口转发到容器 80 端口。

## 练习

1. 运行 `hello-world`，观察输出中 Docker client、daemon、镜像拉取和容器执行的过程。
2. 分别运行两个 `nginx:alpine` 容器，映射到宿主机不同端口。
3. 删除一个容器，再确认镜像是否还存在。

## 验收

- 能用自己的话区分镜像、容器、仓库。
- 能说明 Docker 为什么不是虚拟机。
- 能解释删除容器、删除镜像、删除 volume 的影响不同。

## 重点

- 容器是进程隔离，不是完整机器。
- 镜像是不可变构建产物，容器可写层是运行时临时状态。
- Docker 的工程价值来自可复现构建和一致运行环境。

## 难点

- 容器内部看到的是独立文件系统和网络，但底层仍使用宿主机内核。
- Docker CLI 命令简单，背后涉及镜像层、运行时、网络驱动、存储驱动。

## 易错

> **易错：** 在容器中手工安装软件并把它当最终环境。
>
> 正确做法：把环境变更写进 Dockerfile，重新构建镜像。

> **易错：** 认为容器删除后所有数据都会保留。
>
> 正确做法：需要保留的数据必须放到 volume、bind mount 或外部存储。
