# Docker学习资料：容器生命周期和资源隔离

[返回索引](../Docker学习资料.md)

## 学习目标

- 理解容器从创建到删除的状态变化。
- 掌握 restart policy、健康检查、退出码、资源限制。
- 理解 namespace 和 cgroup 的作用。

## 理论导读

容器的生命周期由主进程决定。容器里 PID 1 进程退出，容器就退出。Docker 不关心你的应用是不是“服务”，只关心启动命令对应的主进程是否还在运行。资源隔离依赖 Linux namespace 和 cgroup：namespace 改变进程看到的系统视图，cgroup 限制和统计资源使用。

## 核心心智模型

容器像一个被限制视野和额度的进程组：

- namespace 决定它“看见什么”。
- cgroup 决定它“最多用多少”。
- 镜像文件系统决定它“从哪里启动”。
- 启动命令决定它“活多久”。

## 知识点详解

### 生命周期状态

- created：已创建，未启动。
- running：主进程运行中。
- paused：进程被冻结。
- exited：主进程退出。
- restarting：按策略重启中。
- removing/dead：删除或异常状态。

### 退出码

- `0`：正常退出。
- 非 `0`：应用错误、命令错误或信号终止。
- `137`：常见于 SIGKILL 或内存 OOM。

### 重启策略

```powershell
docker run -d --restart unless-stopped --name app demo/app:1.0
```

- `no`：默认，不自动重启。
- `on-failure`：非 0 退出时重启。
- `always`：总是重启。
- `unless-stopped`：除非手工停止，否则 daemon 重启后继续拉起。

### 健康检查

健康检查不等于进程是否存在。进程可能还活着，但 HTTP 服务已经卡死。

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --retries=3 CMD wget -qO- http://localhost:8080/health || exit 1
```

### 资源限制

```powershell
docker run -d --memory 512m --cpus 1.5 --name app demo/app:1.0
docker stats app
```

资源限制能避免单个容器耗尽宿主机资源，但限制过小会导致应用频繁 OOM 或延迟升高。

## 例子

```powershell
docker run --name exit-demo alpine:3.20 sh -c "echo start && exit 2"
docker ps -a
docker inspect exit-demo --format "{{.State.ExitCode}}"
```

## 练习

1. 创建一个会立即退出的容器，查看退出码。
2. 给容器设置 `--memory`，观察内存压力下的行为。
3. 为一个 HTTP 服务添加健康检查。

## 验收

- 能根据容器状态判断下一步排查方向。
- 能解释 PID 1 退出为什么导致容器退出。
- 能使用资源限制和健康检查。

## 重点

- 容器生命周期由主进程控制。
- namespace 隔离视图，cgroup 控制资源。
- 健康检查是服务可用性信号，不是进程存在信号。

## 难点

- PID 1 对信号和僵尸进程处理有特殊性，复杂启动脚本可能导致信号不能正确传给业务进程。
- OOM、CPU 限制、文件描述符限制可能表现为应用层错误。

## 易错

> **易错：** 在 Dockerfile 里用后台启动命令，如 `service nginx start`，主命令立刻退出。
>
> 正确做法：让业务进程以前台方式作为容器主进程运行。
