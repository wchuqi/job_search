# Docker学习资料：生产实践、CI 和发布

[返回索引](../Docker学习资料.md)

## 学习目标

- 把 Docker 纳入团队开发、CI、测试和发布流程。
- 设计可回滚、可追溯、可扫描的镜像交付方式。
- 理解 Docker 与 Kubernetes、制品库、配置中心的配合。

## 理论导读

生产中的 Docker 不是“在服务器上手动 run 一个容器”，而是一个制品交付流程：源码提交触发 CI，构建镜像，运行测试和扫描，推送到 registry，部署系统按明确版本或 digest 拉取并运行。配置和密钥由环境注入，数据由外部存储承载，日志和指标由平台采集。

## 核心心智模型

把镜像看成和 jar、war、二进制同级的发布制品，但它额外包含运行环境。制品必须可追溯，运行必须可观测，失败必须可回滚。

## 知识点详解

### CI 流程

典型步骤：

1. 检出源码。
2. 运行单元测试和静态检查。
3. 构建镜像。
4. 运行镜像级集成测试。
5. 扫描漏洞和敏感信息。
6. 推送镜像。
7. 记录 tag、digest、commit、构建日志。

### 环境配置

镜像不应为每个环境重新构建一份。推荐同一镜像通过环境变量、配置文件挂载、Secret、服务发现连接不同环境。

### 发布策略

- 蓝绿发布：两套环境切换流量。
- 滚动发布：逐批替换实例。
- 金丝雀发布：小比例流量验证。
- 回滚：重新部署上一版本镜像 digest。

### Docker 和 Kubernetes

Docker 负责构建和本地运行体验，Kubernetes 负责集群调度、服务发现、声明式部署和自愈。学习 Kubernetes 前必须理解镜像、容器、端口、环境变量、volume 和日志。

## 例子

```yaml
name: build-image
on: [push]
jobs:
  docker:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: docker build -t registry.example.com/app:${{ github.sha }} .
      - run: docker push registry.example.com/app:${{ github.sha }}
```

示例仅展示思路，真实项目需要登录 registry、缓存、测试、扫描和权限控制。

## 练习

1. 为一个项目设计镜像标签：版本、commit、环境指针。
2. 写一份发布记录模板，包含 digest、配置版本、数据库迁移版本。
3. 在本地模拟上一版本和下一版本回滚。

## 验收

- 能说明镜像从源码到运行的完整链路。
- 能设计不把配置和密钥写进镜像的方案。
- 能说清楚 Docker 和 Kubernetes 的职责边界。

## 重点

- 生产发布依赖可追溯制品，不依赖手工容器修改。
- 配置外置，镜像不可变。
- 日志、指标、健康检查和资源限制是上线基础。

## 难点

- 回滚不只是换镜像，还要考虑数据库迁移、配置兼容和外部依赖。
- 镜像扫描发现漏洞后，需要平衡修复成本、可利用性、基础镜像升级风险。

## 易错

> **易错：** 每个环境构建一份不同镜像。
>
> 正确做法：同一镜像多环境运行，差异通过配置和 Secret 注入。
