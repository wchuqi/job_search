# Spring Framework 7 学习资料：事务拦截器、传播、回滚和资源同步深度解析

[返回索引](../SpringFramework7学习资料.md)

## 学习目标

- 从调用链角度理解声明式事务。
- 掌握传播行为的资源语义，不只背定义。
- 理解事务同步、连接绑定、回滚规则和常见一致性风险。

## 声明式事务调用链

```text
client
  -> transaction proxy
      -> TransactionInterceptor
          -> 获取 TransactionAttribute
          -> 选择 PlatformTransactionManager
          -> getTransaction()
              -> 根据传播行为决定加入、创建、挂起、报错
          -> 调用目标方法
          -> 正常返回：commit()
          -> 异常：按 rollback rules 决定 rollback 或 commit
```

> **重点：** `@Transactional` 的关键不是注解本身，而是 TransactionInterceptor 是否被代理链执行。

## 连接和线程绑定

在典型 JDBC 事务中，事务管理器会把数据库连接绑定到当前线程。事务期间，同一线程内通过 Spring 获取连接的 JDBC 操作会复用这个连接。

```text
TransactionSynchronizationManager
  ThreadLocal resources:
    DataSource -> ConnectionHolder
```

> **难点：** 如果绕过 Spring 直接从 DataSource 获取连接，或切换线程执行数据库操作，就可能脱离当前事务。

## 传播行为资源语义

| 传播 | 资源语义 |
| --- | --- |
| REQUIRED | 当前线程有事务资源就复用，没有就绑定新资源 |
| REQUIRES_NEW | 挂起当前资源，绑定一套新事务资源 |
| NESTED | 在当前事务中创建保存点，依赖数据库和管理器支持 |
| NOT_SUPPORTED | 挂起当前事务资源，非事务执行 |
| MANDATORY | 没有当前事务资源就报错 |
| NEVER | 有当前事务资源就报错 |
| SUPPORTS | 有就用，没有也能执行 |

## REQUIRES_NEW 风险示例

```java
@Transactional
public void createOrder() {
    orderRepository.insert(order);
    auditService.record("created"); // REQUIRES_NEW
    throw new RuntimeException("fail");
}
```

如果 `record()` 使用 `REQUIRES_NEW`，审计日志可能已经提交，而订单回滚。这可能是想要的，也可能造成业务不一致。

> **易错：** 把 `REQUIRES_NEW` 当成“更强事务”。它是“独立事务”，不是“更安全事务”。

## 回滚规则深入

默认规则：

- RuntimeException 回滚。
- Error 回滚。
- checked exception 默认不回滚。

规则匹配时，具体规则优先。配置过宽可能误伤：

```java
@Transactional(rollbackFor = Exception.class)
```

这会让所有受检异常都回滚，适合部分导入场景，但不应无脑全局使用。

## 只读事务

`readOnly = true` 是一个优化和语义提示，不是绝对安全限制。数据库、驱动、事务管理器的支持程度不同。

> **易错：** 以为 readOnly 能阻止所有写入。应该把它当作优化提示和代码语义，而不是权限控制。

## 异步和事务

事务通常绑定当前线程。以下场景要谨慎：

- `@Async` 方法在另一个线程执行，不继承调用线程事务。
- Reactor/WebFlux 不是传统 ThreadLocal 模型，需要响应式事务管理。
- 手动创建线程会丢失事务上下文。

## 多数据源

如果一个方法操作多个数据库，普通本地事务管理器只能管理一个资源。要么使用分布式事务方案，要么采用业务补偿、消息最终一致性、事务性 outbox 等设计。

## 练习

1. 打印事务内外当前连接是否相同。
2. 用 `REQUIRES_NEW` 记录日志，观察外层回滚后的数据状态。
3. 在事务中开启新线程写库，验证是否参与事务。
4. 用 checked exception 测试默认是否回滚。

## 验收

- 能从线程资源绑定角度解释事务。
- 能说明每种传播行为的资源语义。
- 能设计避免事务边界误用的 Service 分层。
