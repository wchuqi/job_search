# Spring Framework 7 学习资料：依赖注入和自动装配解析规则

[返回索引](../SpringFramework7学习资料.md)

## 学习目标

- 掌握构造器注入、setter 注入、字段注入的差异。
- 理解自动装配多候选 Bean 的选择规则。
- 会定位 `NoSuchBeanDefinitionException` 和 `NoUniqueBeanDefinitionException`。

## 理论导读

依赖注入的目标是让对象声明依赖，而不是自己寻找依赖。Spring 负责在容器里查找候选 Bean，按类型、名称、限定符、优先级、泛型信息等规则选择最终对象。

自动装配不是“随便找一个”。当候选 Bean 多于一个时，Spring 会按一组明确规则缩小范围。理解这些规则，比记住注解名字更重要。

## 推荐注入方式

| 方式 | 优点 | 风险 |
| --- | --- | --- |
| 构造器注入 | 依赖不可变、便于测试、失败更早 | 构造参数过多暴露类职责过重 |
| setter 注入 | 适合可选依赖和后置配置 | 对象可能处于半初始化状态 |
| 字段注入 | 写法短 | 不利于测试、隐藏依赖、无法 final |

> **重点：** 业务服务优先使用构造器注入。

## 多候选解析规则

常见决策顺序可以这样理解：

```text
按类型查找候选 Bean
  -> 使用 @Qualifier 精确限定
  -> 使用 @Primary 选择主候选
  -> 根据注入点名称匹配 Bean 名称
  -> 根据 @Priority/@Order 处理集合或优先级
  -> 仍无法唯一确定则报错
```

实际实现还会考虑泛型、父子容器、可选依赖、延迟代理等因素。

## 例子：多实现注入

```java
interface PaymentClient {
    void pay(long amount);
}

@Component("alipayClient")
class AlipayClient implements PaymentClient {
    public void pay(long amount) {
        System.out.println("alipay: " + amount);
    }
}

@Primary
@Component
class WechatPayClient implements PaymentClient {
    public void pay(long amount) {
        System.out.println("wechat: " + amount);
    }
}

@Service
class PayService {
    private final PaymentClient paymentClient;

    PayService(@Qualifier("alipayClient") PaymentClient paymentClient) {
        this.paymentClient = paymentClient;
    }
}
```

## 集合注入

```java
@Service
class ExportService {
    private final List<Exporter> exporters;
    private final Map<String, Exporter> exporterMap;

    ExportService(List<Exporter> exporters, Map<String, Exporter> exporterMap) {
        this.exporters = exporters;
        this.exporterMap = exporterMap;
    }
}
```

集合注入适合策略模式、插件式处理链、多个校验器、多个事件处理器。列表顺序可通过 `@Order` 或 `Ordered` 控制。

## 可选和懒加载依赖

```java
@Service
class ReportService {
    private final ObjectProvider<AuditClient> auditClientProvider;

    ReportService(ObjectProvider<AuditClient> auditClientProvider) {
        this.auditClientProvider = auditClientProvider;
    }

    void report() {
        auditClientProvider.ifAvailable(AuditClient::send);
    }
}
```

`ObjectProvider` 适合可选依赖、延迟获取、多实例 prototype 获取。

## 常见错误

> **易错：** 两个实现类都注册为 Bean，却没有 `@Qualifier` 或 `@Primary`。
>
> 正确做法：业务语义明确时用 `@Qualifier`，默认实现明确时用 `@Primary`。

> **易错：** 为了省事使用字段注入，导致单元测试必须启动 Spring 容器。
>
> 正确做法：构造器注入让测试可以直接 `new` 目标对象。

> **易错：** 依赖解析失败时只看最后一行异常。
>
> 正确做法：从 `UnsatisfiedDependencyException` 的 cause 链向下找第一个真正失败的 Bean。

## 练习

1. 创建 3 个 `NotificationSender` 实现，分别用 `@Primary`、`@Qualifier`、`@Order` 控制注入结果。
2. 写一个策略分发器，通过 `Map<String, Handler>` 根据业务类型选择处理器。
3. 故意制造多候选冲突，记录异常和解决方式。

## 验收

- 能说明构造器注入为什么优先。
- 能解释多候选 Bean 解析顺序。
- 能定位自动装配失败的根因。
