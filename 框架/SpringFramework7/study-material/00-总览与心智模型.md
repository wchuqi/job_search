# Spring Framework 7 学习资料：总览与心智模型

[返回索引](../SpringFramework7学习资料.md)

## 学习目标

- 理解 Spring Framework 是什么、不是什么。
- 建立“容器、代理、请求管线、事务边界”四个核心心智模型。
- 明确 Spring Framework 7.x、Spring Boot、Jakarta EE、Servlet、Reactor 的关系。

## 理论导读

Spring Framework 是 Java 应用的基础设施框架。它的核心价值不是“提供很多注解”，而是把对象创建、依赖关系、生命周期、横切逻辑、Web 请求处理、事务边界、数据访问异常转换、测试集成等通用问题收进统一模型。

如果没有 Spring，大型 Java 项目会在每个模块里手写对象创建、配置读取、单例管理、事务开启提交、异常翻译、请求参数转换、测试替身装配。Spring 的思路是：业务代码只表达业务职责，基础设施由容器和框架扩展点统一接管。

## 核心心智模型

### 1. 容器像“对象工厂和调度中心”

你交给 Spring 的不是对象本身，而是对象定义。Spring 先收集 `BeanDefinition`，再按依赖关系、作用域、生命周期回调和后置处理器规则创建对象。

> **重点：** `ApplicationContext` 是高级容器，底层核心能力来自 `BeanFactory`。

### 2. 代理像“业务对象外面的一层壳”

AOP、事务、缓存、权限这类逻辑不应该散落在每个业务方法里。Spring 常通过代理对象包住目标对象，调用先进入代理，再决定是否开启事务、执行通知、捕获异常。

> **易错：** 同一个类内部 `this.method()` 调用不会经过代理，因此可能绕过 `@Transactional` 或切面。

### 3. MVC 像“请求处理流水线”

HTTP 请求进入 `DispatcherServlet` 后，不是直接调用 Controller。Spring MVC 会先找处理器、选择适配器、解析参数、调用方法、处理返回值、转换 JSON、处理异常。

### 4. 事务像“业务用例边界”

事务不应该按单条 SQL 理解，而应该按业务动作理解。一次下单、一次转账、一次库存扣减，才是事务边界的候选。

## 模块地图

| 模块 | 解决的问题 | 典型能力 |
| --- | --- | --- |
| Core Container | 对象创建和依赖管理 | IoC、DI、Bean 生命周期 |
| AOP | 横切逻辑 | 日志、事务、权限、监控 |
| Data Access | 数据访问基础设施 | JDBC、事务、异常转换 |
| Web MVC | Servlet Web 应用 | REST、参数绑定、JSON、异常处理 |
| WebFlux | 响应式 Web 应用 | Reactor、非阻塞 IO、流式响应 |
| Test | 测试集成 | 上下文缓存、事务测试、MockMvc |

## Spring Framework 与 Spring Boot

| 对比项 | Spring Framework | Spring Boot |
| --- | --- | --- |
| 定位 | 基础框架 | 应用快速开发和自动配置框架 |
| 核心能力 | IoC、AOP、MVC、事务、测试 | 自动配置、Starter、Actuator、外部化配置 |
| 是否必须一起用 | 不必须 | Boot 依赖 Spring Framework |
| 学习重点 | 机制和边界 | 约定和生产集成 |

> **易错：** `@SpringBootApplication` 不是 Spring Framework 的核心入口；它是 Boot 对配置类、组件扫描、自动配置的组合封装。

## 例子

最小 Spring 容器示例：

```java
import org.springframework.context.annotation.AnnotationConfigApplicationContext;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

public class Main {
    public static void main(String[] args) {
        var context = new AnnotationConfigApplicationContext(AppConfig.class);
        var service = context.getBean(OrderService.class);
        service.createOrder("A1001");
        context.close();
    }
}

@Configuration
class AppConfig {
    @Bean
    OrderRepository orderRepository() {
        return new OrderRepository();
    }

    @Bean
    OrderService orderService(OrderRepository repository) {
        return new OrderService(repository);
    }
}

class OrderService {
    private final OrderRepository repository;

    OrderService(OrderRepository repository) {
        this.repository = repository;
    }

    void createOrder(String orderNo) {
        repository.save(orderNo);
    }
}

class OrderRepository {
    void save(String orderNo) {
        System.out.println("save order: " + orderNo);
    }
}
```

## 练习

1. 用 `AnnotationConfigApplicationContext` 创建 3 个 Bean：Controller、Service、Repository。
2. 分别用构造器注入和 `@Bean` 方法参数注入实现依赖。
3. 打印 Bean 的真实 class，观察是否为代理。

## 验收

- 能解释 IoC、DI、Bean、ApplicationContext 的区别。
- 能说清 Spring Boot 和 Spring Framework 的边界。
- 能用不依赖 Boot 的方式启动一个最小 Spring 容器。

## 难点

Spring 的难点不在注解本身，而在注解背后的解析顺序、扩展点调用时机和代理边界。面试和生产排障常考的也正是这些机制。
