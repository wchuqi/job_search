# Spring Framework 7 学习资料：生产故障案例和排障剧本深度版

[返回索引](../SpringFramework7学习资料.md)

## 学习目标

- 把 Spring 故障按容器、代理、事务、Web、数据访问、线程和配置分层定位。
- 掌握高频生产问题的复现、验证和回归测试方法。
- 形成可执行排障剧本，而不是只背异常名称。

## 排障总原则

```text
现象
  -> 确定层级
  -> 找第一处异常
  -> 复现最小案例
  -> 打开局部 DEBUG 日志
  -> 验证配置和运行对象
  -> 修复
  -> 加回归测试
```

> **重点：** Spring 异常经常层层包装，最后一行未必是根因。要沿 cause 链找第一个业务或基础设施失败点。

## 案例 1：启动失败，NoSuchBeanDefinitionException

### 现象

`OrderService` 构造器需要 `PaymentClient`，启动时报找不到 Bean。

### 排查路径

1. `PaymentClient` 实现类是否有 `@Component` 或 `@Bean`。
2. 组件扫描包是否覆盖实现类所在包。
3. 实现类是否被 `@Profile` 或 `@Conditional` 过滤。
4. 是否在父容器注册，子容器查找方向是否正确。
5. 类型是否被代理改变，注入点是否过于具体。

### 修复方向

- 扫描范围显式写到应用根包。
- 条件装配增加日志。
- 注入接口而不是具体实现。

### 回归测试

写一个只启动配置类的容器测试：

```java
@ExtendWith(SpringExtension.class)
@ContextConfiguration(classes = AppConfig.class)
class WiringTest {
    @Autowired
    PaymentClient paymentClient;

    @Test
    void paymentClientExists() {
        assertNotNull(paymentClient);
    }
}
```

## 案例 2：事务没有回滚

### 现象

业务方法抛异常后，数据库仍写入成功。

### 排查路径

1. 方法是否通过 Spring 代理调用。
2. 方法是否 `public`，所在类是否是 Bean。
3. 是否同类内部调用。
4. 异常是否被 catch 后吞掉。
5. 异常类型是否触发回滚。
6. `PlatformTransactionManager` 是否管理了实际使用的 `DataSource`。
7. 是否存在 `REQUIRES_NEW` 独立提交。

### 验证代码

```java
System.out.println(AopUtils.isAopProxy(orderService));
System.out.println(TransactionSynchronizationManager.isActualTransactionActive());
```

> **易错：** 只看到方法上有 `@Transactional` 就认为事务一定生效。真正要看代理链是否执行。

## 案例 3：接口 404，但代码存在

### 排查路径

1. Controller 是否被扫描。
2. `@RequestMapping` 类路径和方法路径组合是否正确。
3. 请求 HTTP 方法是否匹配。
4. context path、网关前缀、反向代理 rewrite 是否改变路径。
5. 是否注册到另一个 DispatcherServlet。

### 验证方式

启动时打印所有映射：

```java
@Component
class MappingPrinter {
    MappingPrinter(RequestMappingHandlerMapping mapping) {
        mapping.getHandlerMethods().forEach((info, method) ->
            System.out.println(info + " -> " + method));
    }
}
```

## 案例 4：偶发慢请求

### 分段定位

| 分段 | 观察指标 |
| --- | --- |
| 网关到应用 | access log、上游耗时 |
| MVC 入参解析 | 请求体大小、JSON 反序列化 |
| Service | 业务锁、远程调用、重试 |
| DB | 慢 SQL、连接池等待、锁等待 |
| 响应写出 | 响应体大小、客户端断开 |

### 排查动作

- 给入口、Service、Repository 分段打耗时。
- 采集线程栈，看线程卡在 DB、锁、HTTP 客户端还是 JSON。
- 检查连接池 active、idle、pending。
- 检查是否无限重试或重试风暴。

## 案例 5：参数校验失败却返回 500

### 常见原因

- 没有全局处理 `MethodArgumentNotValidException`。
- 全局异常处理器顺序不当。
- 校验错误对象序列化失败。
- 把校验异常重新包装成系统异常。

### 修复标准

- 参数错误返回 400。
- 业务不存在返回 404。
- 冲突返回 409。
- 系统异常返回 500，且不泄漏内部堆栈。

## 练习

1. 构造 5 类故障并写复盘：Bean 缺失、多候选、事务不回滚、MVC 404、JSON 415。
2. 每类故障都写一个自动化测试防回归。
3. 为订单项目增加 requestId，串联 Controller、Service、Repository 日志。

## 验收

- 能在 10 分钟内判断故障属于哪条链路。
- 能给出复现步骤、根因、修复、回归测试。
- 能解释为什么修复有效，而不只是改到不报错。
