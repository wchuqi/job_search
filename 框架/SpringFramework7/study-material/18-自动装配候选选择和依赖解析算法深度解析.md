# Spring Framework 7 学习资料：自动装配候选选择和依赖解析算法深度解析

[返回索引](../SpringFramework7学习资料.md)

## 学习目标

- 能讲清依赖解析不是“按类型随便注入”。
- 掌握候选 Bean 过滤、主候选、名称匹配、泛型、集合注入的决策规则。
- 能系统排查 `NoSuchBeanDefinitionException` 和 `NoUniqueBeanDefinitionException`。

## 依赖解析主线

依赖注入入口可以简化理解为：

```text
DependencyDescriptor 描述注入点
  -> 判断是否懒加载/Optional/ObjectProvider
  -> 解析快捷依赖，如 @Value、内建依赖
  -> 按类型查找候选 Bean name
  -> 过滤 autowireCandidate=false 的 Bean
  -> 应用 @Qualifier、泛型、条件限定
  -> 判断 @Primary
  -> 按注入点名称匹配
  -> 判断 @Priority
  -> 最终唯一则注入，否则报错
```

`DependencyDescriptor` 很关键，它记录注入点类型、泛型、字段名或参数名、注解、是否必须、嵌套级别等信息。

## 单值注入决策表

| 步骤 | 目的 | 示例 |
| --- | --- | --- |
| 类型匹配 | 找到所有 `PaymentClient` | alipay、wechat |
| 候选过滤 | 排除不允许自动装配的 Bean | `autowireCandidate=false` |
| Qualifier | 语义精确匹配 | `@Qualifier("alipayClient")` |
| Primary | 默认主实现 | `@Primary` |
| 名称匹配 | 参数名/字段名匹配 Bean 名 | `PaymentClient alipayClient` |
| Priority | 优先级选择 | `@Priority(1)` |
| 报错 | 无法唯一或没有候选 | NoUnique / NoSuch |

> **重点：** `@Primary` 是默认实现语义，`@Qualifier` 是明确指定语义。明确指定通常优先。

## 泛型匹配

Spring 会利用泛型信息缩小候选范围：

```java
interface Repository<T> {
}

@Component
class UserRepository implements Repository<User> {
}

@Component
class OrderRepository implements Repository<Order> {
}

@Service
class UserService {
    UserService(Repository<User> repository) {
        // 通常会匹配 UserRepository
    }
}
```

> **难点：** 泛型擦除不代表 Spring 完全拿不到泛型。Spring 会从类声明、父类、接口、方法返回值中解析 `ResolvableType`。

## 集合注入规则

```java
class Pipeline {
    Pipeline(List<Handler> handlers, Map<String, Handler> handlerMap) {
    }
}
```

- `List<Handler>` 注入所有匹配 Handler。
- 顺序受 `@Order`、`Ordered`、`@Priority` 影响。
- `Map<String, Handler>` 的 key 通常是 Bean 名。
- 集合注入通常不要求唯一，但要求元素类型可解析。

## ObjectProvider 延迟解析

```java
class JobRunner {
    private final ObjectProvider<JobHandler> handlers;

    JobRunner(ObjectProvider<JobHandler> handlers) {
        this.handlers = handlers;
    }

    void run() {
        handlers.orderedStream().forEach(JobHandler::handle);
    }
}
```

`ObjectProvider` 把“启动时必须立刻解析”变成“使用时再解析”，适合可选依赖、延迟依赖、prototype、多处理器流式获取。

## 参数名匹配的前提

构造器参数名匹配依赖编译器保留参数名信息，例如 Maven 配置：

```xml
<compilerArgs>
    <arg>-parameters</arg>
</compilerArgs>
```

否则 Spring 未必能知道参数源码名。

## 排障流程

### 没有候选

1. 类是否注册成 Bean。
2. 扫描包是否覆盖。
3. Profile 是否启用。
4. 条件装配是否满足。
5. Bean 类型是否是代理后暴露类型。
6. 父子容器边界是否正确。

### 多候选

1. 是否应该用集合注入。
2. 是否缺 `@Qualifier`。
3. 是否应设置一个 `@Primary`。
4. Bean 名是否和注入点名称一致。
5. 泛型是否能区分候选。

## 练习

1. 写 3 个支付客户端，分别用 Qualifier、Primary、参数名匹配控制注入。
2. 写 `List<Handler>` 处理链，用 `@Order` 控制顺序。
3. 删除 `-parameters`，观察构造器参数名匹配是否受影响。
4. 写 `ObjectProvider<Handler>`，运行时按需获取所有处理器。

## 验收

- 能口述单值 Bean 自动装配完整决策流程。
- 能解释泛型和集合注入的处理方式。
- 能独立解决多候选和无候选装配失败。
