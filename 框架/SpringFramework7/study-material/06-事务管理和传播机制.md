# Spring Framework 7 学习资料：事务管理和传播机制

[返回索引](../SpringFramework7学习资料.md)

## 学习目标

- 理解声明式事务的代理机制。
- 掌握传播行为、隔离级别、回滚规则。
- 能定位 `@Transactional` 不生效和事务边界错误。

## 理论导读

Spring 事务抽象把不同事务技术统一到 `PlatformTransactionManager` 之下。业务代码通过 `@Transactional` 声明事务边界，Spring 通过 AOP 代理在方法调用前开启事务，方法成功后提交，异常时按规则回滚。

事务的本质不是“给每个 DAO 方法加注解”，而是保护一个业务用例的一致性。事务边界通常应放在 Service 层。

## 核心组件

| 组件 | 作用 |
| --- | --- |
| `PlatformTransactionManager` | 事务管理器抽象 |
| `TransactionDefinition` | 传播、隔离、超时、只读等定义 |
| `TransactionStatus` | 当前事务状态 |
| `TransactionInterceptor` | AOP 拦截事务方法 |
| `@Transactional` | 声明事务属性 |

## 传播行为

| 传播行为 | 含义 | 常见场景 |
| --- | --- | --- |
| `REQUIRED` | 有事务加入，没有则新建 | 默认，最常用 |
| `REQUIRES_NEW` | 挂起当前事务，新建事务 | 审计日志、独立记录 |
| `SUPPORTS` | 有事务加入，没有就非事务执行 | 只读查询 |
| `MANDATORY` | 必须已有事务 | 强制由上层控制边界 |
| `NOT_SUPPORTED` | 挂起事务，非事务执行 | 不希望占用事务 |
| `NEVER` | 存在事务则报错 | 禁止事务场景 |
| `NESTED` | 嵌套事务，依赖保存点 | 局部回滚 |

> **难点：** `REQUIRES_NEW` 会独立提交。外层事务回滚时，内层已提交事务不会自动撤销。

## 回滚规则

默认情况下，运行时异常和 Error 会触发回滚，受检异常默认不回滚。可以用 `rollbackFor` 指定。

```java
@Transactional(rollbackFor = IOException.class)
public void importOrders(Path file) throws IOException {
    // parse file and save rows
}
```

> **易错：** 捕获异常后不再抛出，事务拦截器看不到异常，就会提交。

## 例子：业务事务边界

```java
@Service
class TransferService {
    private final AccountRepository accountRepository;

    TransferService(AccountRepository accountRepository) {
        this.accountRepository = accountRepository;
    }

    @Transactional
    public void transfer(long fromId, long toId, BigDecimal amount) {
        accountRepository.debit(fromId, amount);
        accountRepository.credit(toId, amount);
    }
}
```

## 常见不生效场景

| 场景 | 原因 | 修复 |
| --- | --- | --- |
| 同类内部调用 | 未经过代理 | 拆到另一个 Bean 或调整调用边界 |
| private 方法 | 代理无法拦截 | 放到 public service 方法 |
| 手动 new 对象 | 未受容器管理 | 交给 Spring 创建 |
| 异常被吞 | 拦截器看不到失败 | 重新抛出或显式标记回滚 |
| 数据源事务管理器不匹配 | 事务管的不是实际连接 | 检查 DataSource 和 TransactionManager |

## 隔离级别

隔离级别最终由数据库支持决定。Spring 只是传递事务定义，不会魔法般消除数据库并发问题。

| 隔离级别 | 典型现象控制 |
| --- | --- |
| READ_UNCOMMITTED | 可能脏读，少用 |
| READ_COMMITTED | 避免脏读，常见默认 |
| REPEATABLE_READ | 避免不可重复读，MySQL 常见默认 |
| SERIALIZABLE | 最严格，并发成本高 |

## 练习

1. 写转账服务，故意在扣款后抛异常，验证回滚。
2. 写 `REQUIRES_NEW` 审计日志，观察外层回滚后日志是否存在。
3. 捕获异常不抛出，观察事务是否提交。

## 验收

- 能解释声明式事务和 AOP 的关系。
- 能说清 7 种传播行为的语义和风险。
- 能定位事务不生效的根因。

## 重点

事务边界是业务设计问题，不只是注解位置问题。边界放错，技术配置正确也会产生数据一致性风险。
