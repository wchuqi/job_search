## 一、源码机制和深度场景题

### 1. 请按源码主线讲一下 Spring 容器启动过程。

**参考答案：**

从 `ApplicationContext#refresh()` 讲起：先准备环境，再获取 BeanFactory 和加载 BeanDefinition，然后准备 BeanFactory，执行 BeanFactoryPostProcessor 修改定义，注册 BeanPostProcessor，初始化消息源和事件广播器，注册监听器，最后实例化非懒加载单例 Bean 并发布刷新完成事件。

> **重点：** 普通 Bean 大规模创建发生在 `finishBeanFactoryInitialization()`，不是扫描阶段。

### 2. BeanDefinition 和 Bean 实例有什么区别？

**参考答案：**

BeanDefinition 是 Bean 的元数据，描述 class、scope、构造参数、属性、初始化方法等；Bean 实例是根据这些元数据创建出来的对象。Spring 很多扩展点先修改 BeanDefinition，再影响后续对象创建。

> **易错：** 把“注册 BeanDefinition”说成“创建了 Bean”。

### 3. Spring 三级缓存为什么需要第三级？

**参考答案：**

第三级缓存 `singletonFactories` 保存的是能生成早期引用的 ObjectFactory。它让 Spring 在循环依赖场景下延迟决定暴露原始对象还是早期代理对象。没有第三级，AOP 参与循环依赖时很难保证早期注入对象和最终容器对象一致。

> **难点：** 三级缓存核心是早期代理时机，不是单纯多放一层 Map。

### 4. 自动装配多个候选 Bean 的完整决策怎么说？

**参考答案：**

先按类型查找候选，再排除不允许自动装配的 Bean，结合 `@Qualifier`、泛型、`@Primary`、注入点名称、`@Priority` 等规则决定唯一候选。集合注入会注入所有匹配候选并按顺序规则排序；单值注入无法唯一时抛异常。

> **重点：** 单值注入要求唯一，集合注入表达的是策略集合。

### 5. 为什么事务和 AOP 的顺序会影响结果？

**参考答案：**

事务本身是一个方法拦截器。如果还有重试、缓存、日志等切面，它们会组成拦截器链。谁在外层，谁就先进入后退出。比如重试包住事务时，每次重试都可能开启独立事务；事务包住重试时，多次尝试可能处于同一事务边界，语义完全不同。

> **易错：** 只关心切面是否执行，不关心执行顺序。

### 6. `REQUIRES_NEW` 为什么可能造成业务不一致？

**参考答案：**

`REQUIRES_NEW` 会挂起外层事务并创建新事务。内层事务提交后，即使外层事务回滚，内层提交的数据也不会自动撤销。因此它适合独立审计等场景，但不适合必须和主业务原子一致的数据。

> **重点：** `REQUIRES_NEW` 是独立提交，不是更强一致性。

### 7. 一个请求返回 415，你如何从 Spring MVC 链路定位？

**参考答案：**

先检查请求 `Content-Type`，再看 Controller 的 `consumes` 条件是否匹配，然后检查是否存在能读取该媒体类型和目标 Java 类型的 `HttpMessageConverter`。如果是 JSON，还要检查 JSON 转换依赖和 DTO 可反序列化性。

> **重点：** 415 通常发生在请求体媒体类型和消息转换阶段。

### 8. WebFlux 中为什么不能随便调用 `block()`？

**参考答案：**

`block()` 会阻塞当前线程，破坏响应式非阻塞模型。如果发生在事件循环线程上，会让少量阻塞拖慢大量连接，甚至造成死锁或吞吐崩溃。必须调用阻塞 API 时，应隔离到有界弹性线程池，但大量阻塞业务应优先考虑 MVC。

> **难点：** WebFlux 的关键是整条链路非阻塞，不是返回 `Mono` 就够。

### 9. 如果线上 BeanPostProcessor 没有处理某个 Bean，可能是什么原因？

**参考答案：**

可能该 Bean 在 BeanPostProcessor 完成注册前被过早创建，例如它是某个后置处理器或基础设施 Bean 的依赖。也可能 Bean 类型不匹配、被条件过滤、代理后类型变化，或后置处理器排序不符合预期。

> **重点：** 先确认 Bean 创建时机，再确认匹配逻辑。

### 10. 你如何证明一个 Service 方法真的运行在事务中？

**参考答案：**

可以检查对象是否是 AOP 代理，方法调用时打印 `TransactionSynchronizationManager.isActualTransactionActive()`，检查当前线程是否绑定连接资源，并通过写库后抛异常验证是否回滚。最终用集成测试证明数据状态。

> **易错：** 只看注解不做运行时验证。
