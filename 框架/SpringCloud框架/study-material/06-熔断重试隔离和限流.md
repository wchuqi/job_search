# Spring Cloud 学习资料：熔断、重试、隔离和限流

[返回索引](../SpringCloud学习资料.md)

## 学习目标

- 理解分布式调用中的失败传播：一个慢下游如何拖垮整个系统。
- 掌握 CircuitBreaker、Retry、Bulkhead、RateLimiter、TimeLimiter 的区别和组合方式。
- 能用 Spring Cloud Circuit Breaker 与 Resilience4j 思路设计故障保护。
- 能排查熔断误触发、重试风暴、线程池隔离失效、限流误伤和 fallback 掩盖问题。

## 理论导读

分布式系统的下游服务总会失败：超时、错误、慢响应、连接耗尽、限流、部署中断。如果上游没有保护，线程会等待、连接会堆积、请求会排队，最终从局部故障变成级联故障。熔断、重试、隔离、限流不是为了“让请求一定成功”，而是为了控制失败范围。

Spring Cloud Circuit Breaker 提供统一抽象，具体实现常接 Resilience4j。它把故障保护从业务代码中抽出来，但策略仍然要由业务决定：哪些能重试，哪些能降级，哪些必须快速失败。

## 核心心智模型

```text
Timeout      -> 不无限等待
Retry        -> 短暂故障再试一次
CircuitBreaker -> 下游持续失败时快速失败
Bulkhead     -> 限制并发，隔离资源池
RateLimiter  -> 控制进入速率
Fallback     -> 失败后的可控替代路径
```

这些机制像电路保护：不是让电器永远不断电，而是在故障时避免烧毁整栋楼。

## 知识点详解

### 1. CircuitBreaker 状态

典型熔断器有三种状态：

| 状态 | 含义 |
| --- | --- |
| CLOSED | 正常放行，统计失败率和慢调用 |
| OPEN | 熔断打开，直接拒绝或走 fallback |
| HALF_OPEN | 放少量探测请求，成功则关闭，失败则重新打开 |

关键参数：

- 统计窗口大小。
- 失败率阈值。
- 慢调用阈值。
- 最小调用数。
- 打开后等待时间。
- 半开探测数量。

### 2. 熔断参数如何影响行为

熔断器是否打开，通常不是看单次失败，而是看统计窗口内的失败率或慢调用率。

| 参数 | 含义 | 设置过小的风险 | 设置过大的风险 |
| --- | --- | --- | --- |
| slidingWindowSize | 统计窗口大小 | 少量异常就误触发 | 故障感知太慢 |
| minimumNumberOfCalls | 最小调用数 | 低流量接口误判 | 低流量接口长期不触发 |
| failureRateThreshold | 失败率阈值 | 过度保护 | 保护太晚 |
| slowCallDurationThreshold | 慢调用阈值 | 正常波动被判慢 | 慢调用拖垮线程 |
| waitDurationInOpenState | OPEN 等待时间 | 频繁探测抖动 | 恢复后长时间不可用 |
| permittedNumberOfCallsInHalfOpenState | 半开探测数 | 样本不足 | 探测流量过大 |

> **难点：** 熔断参数必须结合接口 QPS、下游 SLA、业务重要性和超时预算。低流量接口和高流量接口不能套同一组参数。

### 3. Retry

重试适合短暂网络抖动，不适合稳定失败或非幂等写操作。

风险：

- 下游慢时重试会放大流量。
- 多层重试叠加造成请求风暴。
- POST 没有幂等键时重试会造成重复副作用。

设计：

- 限制最大次数。
- 使用指数退避和抖动。
- 只对可恢复错误重试。
- 与总体超时预算绑定。

### 4. Bulkhead

Bulkhead 隔离不同调用资源，避免一个下游占满所有线程或并发。

例子：

- 库存服务最多 50 并发。
- 支付服务最多 20 并发。
- 推荐服务失败不能影响下单主链路。

隔离策略可以是信号量，也可以是独立线程池。线程池隔离成本更高，但隔离更彻底。

### 5. RateLimiter

限流控制进入系统或某资源的请求速率。常见维度：

- 用户。
- IP。
- 租户。
- API key。
- 路径。
- 服务实例。

限流返回 429 时，客户端应知道是否可以稍后重试。

### 6. Fallback 的边界

fallback 不是万能兜底。它适合：

- 返回缓存。
- 返回默认推荐。
- 降级非核心功能。
- 快速失败并给明确错误。

不适合：

- 假装创建订单成功。
- 吞掉支付失败。
- 返回空数据掩盖严重故障。

### 7. 策略组合

订单查库存：

```text
TimeLimiter 1.5s
  -> Retry 1 次，仅 GET 查询
  -> CircuitBreaker 失败率超过阈值打开
  -> Bulkhead 限制库存调用并发
  -> Fallback 返回“库存暂不可用”
```

订单扣库存：

```text
Timeout
  -> 不自动重试，除非有幂等键
  -> CircuitBreaker 快速失败
  -> 业务补偿或人工处理
```

### 8. 级联故障的真实路径

一个典型故障传播：

```text
inventory-service 慢
  -> order-service Feign 线程等待
  -> order-service 数据库事务变长
  -> 连接池 pending 上升
  -> Gateway 请求堆积
  -> 用户重试
  -> 流量进一步放大
```

治理动作应按层次执行：

1. 设置 Feign 超时，避免无限等待。
2. 对库存调用设置 Bulkhead，限制并发。
3. CircuitBreaker 快速失败，保护订单服务。
4. Gateway 限流，保护入口。
5. 对写请求使用幂等键，避免用户重试造成重复副作用。
6. trace 和指标记录被拒绝、熔断、超时的原因。

## 例子：声明式思路

```java
@Service
class InventoryFacade {
    private final InventoryClient inventoryClient;

    InventoryFacade(InventoryClient inventoryClient) {
        this.inventoryClient = inventoryClient;
    }

    public InventoryResponse findInventory(String sku) {
        return inventoryClient.findBySku(sku);
    }
}
```

策略通常通过 Resilience4j/Spring Cloud Circuit Breaker 配置和装饰器应用到调用上，重点是先定义业务语义：是否可重试、是否可降级、超时预算是多少。

## 练习

1. 给库存查询设置 1.5 秒超时。
2. 设置失败率超过 50% 后熔断。
3. 模拟库存服务持续 500 错误，观察 CLOSED -> OPEN -> HALF_OPEN。
4. 给库存扣减接口加幂等键，再评估是否允许重试。
5. 设计 Gateway 和服务内部限流的分工。

## 验收

- 能解释熔断器三种状态。
- 能说明重试为什么可能放大故障。
- 能区分熔断、限流、隔离、超时。
- 能判断 fallback 是否会掩盖业务错误。
- 能为读接口和写接口设计不同保护策略。

## 重点

- 故障保护的目标是限制损害，不是保证所有请求成功。
- 重试必须结合幂等、退避、超时和熔断。
- fallback 要保守，不能制造虚假的业务成功。

## 难点

- 多层重试和多层限流叠加后系统行为很难直觉判断。
- 熔断参数过敏会误伤，过钝又保护不了系统。

## 易错

> **易错：** 给所有 Feign 调用统一配置三次重试。
>
> 正确做法：按接口语义区分读写、幂等性、下游 SLA 和总体超时预算。
