# Spring Cloud 学习资料：安全认证、授权和零信任

[返回索引](../SpringCloud学习资料.md)

## 学习目标

- 理解微服务安全不能只依赖网关，内部服务也要保护。
- 掌握 Gateway 鉴权、资源服务器、JWT 传播、Token Relay、服务间身份、mTLS、Actuator 安全。
- 能设计用户身份、服务身份、租户上下文和 trace 上下文的传播策略。
- 能排查 401/403、Token 过期、scope 映射错误、内部接口裸露和敏感端点泄漏。

## 理论导读

单体应用的安全边界通常在应用入口。微服务系统有多个入口：外部网关、内部服务、消息消费者、Actuator、管理端口、Kubernetes 网络、CI/CD 和配置中心。如果只在 Gateway 做鉴权，内部服务一旦被绕过或网络策略失效，就可能裸露关键能力。

零信任思路是：不要因为请求来自内网就默认可信。每个服务都应知道调用方是谁、用户是谁、能做什么、请求是否可追踪。

## 核心心智模型

```text
外部用户身份 -> Gateway 验证
  -> 用户 claims/scope 传递
  -> 内部服务再次授权
服务身份 -> mTLS / service token / workload identity
  -> 服务间调用校验
```

用户身份和服务身份是两件事。用户能下单，不代表 order-service 可以无限调用 payment-service 的所有接口。

## 知识点详解

### 1. Gateway 安全职责

Gateway 适合：

- OAuth2/OIDC 登录入口。
- JWT 验证。
- 粗粒度路径授权。
- CORS。
- 限流。
- Header 清洗。

Gateway 不适合：

- 所有细粒度业务授权。
- 长时间阻塞查询权限。
- 替代内部服务安全。

### 2. Token Relay 和身份传播

如果下游需要用户身份，Gateway 可以把用户 Token 或转换后的 Token 传给下游。风险：

- Token 泄漏影响面扩大。
- 下游服务过度依赖外部 Token 格式。
- scope/audience 不匹配。

更成熟方案会使用 Token Exchange，把外部用户 Token 换成面向内部服务的短期 Token。

### 3. 服务间认证

常见方式：

- 服务账号 Token。
- OAuth2 client credentials。
- mTLS。
- Service Mesh workload identity。
- 云平台 IAM。

选择取决于平台能力。应用侧至少要避免内部接口完全裸奔。

### 4. 多租户上下文

租户 ID 不能只靠前端传 Header。应从可信 Token claim 或网关校验后的上下文得到，并在下游服务重新验证租户权限。

### 5. Actuator 和管理端点

敏感端点必须限制：

- `/actuator/env`
- `/actuator/configprops`
- `/actuator/heapdump`
- `/actuator/threaddump`
- `/actuator/gateway`
- `/actuator/refresh`
- `/actuator/busrefresh`

Config refresh 和 Gateway route 管理端点一旦暴露，攻击者可能改变系统行为。

## 例子：资源服务器配置

```yaml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://issuer.example.com
```

授权：

```java
@PreAuthorize("hasAuthority('SCOPE_order:write')")
public OrderResponse create(CreateOrderCommand command) {
    return orderService.create(command);
}
```

## 练习

1. Gateway 验证 JWT。
2. order-service 也配置 Resource Server。
3. Gateway 放行 `/actuator/health/readiness`，保护其他管理端点。
4. 传递 traceId、tenantId，但 tenantId 必须从 Token 中确认。
5. 模拟 scope 缺失，观察 403。

## 验收

- 能区分用户身份和服务身份。
- 能说明为什么不能只依赖 Gateway 安全。
- 能保护 refresh、gateway、env 等敏感端点。
- 能排查 401 和 403。

## 重点

- 微服务内部调用也需要身份和授权边界。
- Token 传播要考虑 audience、scope、过期和泄漏范围。
- 管理端点是高风险入口。

## 难点

- 用户身份、服务身份、租户上下文、链路追踪上下文容易混在一起。
- Gateway、Service Mesh、业务服务三层安全职责需要明确。

## 易错

> **易错：** 网关鉴权后，内部服务完全不校验权限。
>
> 正确做法：网关做入口保护，业务服务做最终业务授权，网络层限制绕过路径。
