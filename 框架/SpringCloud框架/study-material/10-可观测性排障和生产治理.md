# Spring Cloud 学习资料：可观测性、排障和生产治理

[返回索引](../SpringCloud学习资料.md)

## 学习目标

- 建立跨服务排障证据链：Gateway、Feign、LoadBalancer、CircuitBreaker、Stream、数据库和下游依赖。
- 掌握日志、指标、trace、Actuator、Gateway metrics、Feign 日志、熔断指标和消息堆积指标。
- 能处理服务间调用慢、路由失败、配置错误、熔断打开、消息积压和级联故障。
- 能设计生产治理基线：超时、限流、熔断、回滚、灰度、告警和 SLO。

## 理论导读

Spring Cloud 系统的故障经常跨服务传播。用户看到一个 500，根因可能是 Gateway 路由错误、订单服务 Feign 超时、库存服务连接池耗尽、配置中心刷新错误、消息消费者堆积或下游支付 API 慢。没有统一 traceId、指标和日志，排障会变成猜测。

可观测性不是安装一个监控系统就结束，而是让每一次请求、每一次远程调用、每一次消息消费、每一次熔断和限流都有证据。

## 核心心智模型

```text
用户请求失败
  -> Gateway routeId/status/latency
  -> traceId 串联 order-service
  -> Feign span 定位 inventory-service
  -> CircuitBreaker 指标显示 open
  -> inventory-service 日志显示连接池耗尽
  -> 数据库慢查询定位根因
```

排障要从入口到下游逐层收缩范围。

## 知识点详解

### 1. 日志

要求：

- 所有服务输出 traceId。
- 关键业务日志包含业务主键。
- 不记录密码、Token、身份证等敏感数据。
- 网关日志记录 routeId、path、status、latency。
- Feign 错误日志记录下游服务名、接口、状态码、耗时。

### 2. 指标

关键指标：

- Gateway：路由维度 QPS、状态码、延迟。
- HTTP Client/Feign：下游服务维度耗时、错误率。
- LoadBalancer：实例选择和失败。
- CircuitBreaker：状态、失败率、慢调用率、拒绝数。
- RateLimiter：放行数、拒绝数。
- Stream：消费延迟、堆积、DLQ 数。
- JVM：CPU、GC、线程、内存。

### 3. Trace

Trace 能看到跨服务调用路径。要确保：

- Gateway 创建或传递 traceId。
- Feign/HTTP 客户端传播上下文。
- 消息生产和消费也关联 trace。
- 采样率与成本平衡。

### 4. 跨服务证据链怎么建

一次线上问题不能只靠一条日志。建议按固定字段串联：

```text
traceId       -> 串联一次请求
routeId       -> Gateway 命中的路由
serviceId     -> 被调用服务名
instanceId    -> 实际实例
spanId        -> 调用片段
errorCode     -> 业务错误码
status        -> HTTP 或消息处理状态
duration      -> 耗时
```

排障时先回答四个问题：

1. 流量从哪个入口进来。
2. Gateway 是否正确路由。
3. 慢或错发生在哪个 span。
4. 该 span 对应的服务实例当时资源和日志如何。

如果缺少 instanceId，只知道库存服务慢，不知道是哪一个实例慢；如果缺少 routeId，Gateway 多路由时很难判断匹配了哪条规则。

### 5. 常见故障剧本

#### 服务间调用超时

1. 看 Gateway 是否慢。
2. Trace 定位慢 span。
3. 看 Feign 超时配置。
4. 看下游服务指标和日志。
5. 看 CircuitBreaker 是否打开。
6. 临时降级或回滚。

证据链示例：

```text
gateway route=order status=504 duration=3000ms
  -> trace span order-service POST /orders duration=2950ms
  -> child span feign inventory-service POST /reserve timeout=1500ms
  -> circuitbreaker inventoryReserve state=OPEN rejected=120
  -> inventory-service hikari pending=80, slow sql detected
```

这条链说明根因更接近库存服务或数据库，而不是 Gateway 本身。

#### Gateway 404

1. route 是否加载。
2. Predicate 是否匹配。
3. StripPrefix/RewritePath 是否正确。
4. 下游 Controller 路径是否一致。

#### 熔断频繁打开

1. 看失败率和慢调用率。
2. 看下游真实错误。
3. 判断阈值是否过敏。
4. 检查是否重试放大。
5. 调整保护策略或修复下游。

#### 消息积压

1. 看生产速率和消费速率。
2. 看消费者错误和重试。
3. 看分区和消费组实例数。
4. 看下游依赖是否慢。
5. 处理毒消息和 DLQ。

### 6. 生产治理参数基线

每条服务间调用至少记录一张表：

| 项 | 示例 |
| --- | --- |
| 调用方 | order-service |
| 被调用方 | inventory-service |
| 接口 | POST /inventory/reserve |
| 业务语义 | 预占库存，非幂等但有幂等键 |
| connect timeout | 500ms |
| read timeout | 1500ms |
| retry | 默认不重试 |
| circuit breaker | 失败率 50%，慢调用 2s |
| bulkhead | 50 并发 |
| fallback | 返回库存暂不可用，订单不创建 |
| 指标 | latency、error、timeout、rejected |

没有这张表，团队很容易在故障时临时改参数，导致系统行为不可预测。

### 5. 生产治理基线

每个服务必须有：

- 超时配置。
- 熔断和限流策略。
- 健康检查。
- 指标和告警。
- 结构化日志。
- 灰度和回滚方案。
- 容量基线。
- 配置变更审计。

## 练习

1. 在 Gateway、order-service、inventory-service 中打通 traceId。
2. 制造 inventory-service 延迟，观察 trace 和熔断指标。
3. 制造 Gateway 路径改写错误，定位 404。
4. 制造消息消费者异常，观察 DLQ。
5. 写一次故障复盘。

## 验收

- 能从 Gateway 指标定位到具体下游服务。
- 能用 trace 解释一次跨服务请求。
- 能判断熔断打开是保护还是误配置。
- 能处理消息堆积和毒消息。
- 能写生产故障复盘。

## 重点

- Spring Cloud 排障必须跨服务看证据链。
- trace、metrics、logs 三者缺一不可。
- 保护策略必须可观测，否则不知道是在保护还是误伤。

## 难点

- 多层网关、Mesh、服务内部调用会让 trace 变复杂。
- 故障期间指标可能互相影响，需要先找触发点。

## 易错

> **易错：** 只看报错服务日志，不看完整调用链。
>
> 正确做法：从入口 Gateway 开始，用 traceId 串起所有服务和下游依赖。
