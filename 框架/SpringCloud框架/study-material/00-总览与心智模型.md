# Spring Cloud 学习资料：总览与心智模型

[返回索引](../SpringCloud学习资料.md)

## 学习目标

- 理解 Spring Cloud 的定位：为 Spring Boot 应用提供分布式系统通用模式。
- 建立微服务运行时心智模型：服务实例动态变化、网络调用不可靠、局部失败常态化。
- 区分 Spring Cloud、Spring Boot、Kubernetes、Service Mesh、消息中间件的责任边界。
- 明确哪些能力是新项目常用核心，哪些是历史组件或特定场景组件。

## 理论导读

Spring Boot 让一个服务更容易开发和运行；Spring Cloud 让多个服务在分布式环境中更容易协作。单体应用里一次方法调用通常在一个进程内完成，失败模式相对简单；微服务里一次业务请求可能经过网关、多个服务、数据库、缓存、消息队列和第三方 API。网络抖动、实例扩缩容、部分节点故障、配置不一致、消息重复、链路追踪缺失都会成为日常问题。

Spring Cloud 提供的不是“业务架构答案”，而是一组工程模式：配置中心、服务注册发现、客户端负载均衡、声明式 HTTP 调用、API 网关、熔断限流、消息驱动、分布式事件、Kubernetes 集成、可观测性集成。它的价值在于把分布式系统中反复出现的基础设施代码标准化。

## 核心心智模型

把 Spring Cloud 系统想成城市交通：

```text
Gateway          -> 城市入口和交通规则
Discovery        -> 地址簿，知道服务实例在哪
LoadBalancer     -> 选择哪辆车/哪条路
Feign/RestClient -> 服务间通信方式
Config           -> 集中配置仓库
CircuitBreaker   -> 道路故障时的保护闸
Stream           -> 物流系统，异步投递消息
Observability    -> 摄像头、路况图和事故报告
```

每个组件都在解决一个分布式系统基本问题：

| 问题 | Spring Cloud 能力 |
| --- | --- |
| 服务实例 IP 动态变化 | DiscoveryClient、Consul、Eureka、Kubernetes Discovery |
| 调用方如何选实例 | Spring Cloud LoadBalancer |
| HTTP 调用重复样板多 | Spring Cloud OpenFeign |
| 外部流量统一入口 | Spring Cloud Gateway |
| 配置集中管理 | Spring Cloud Config、Vault |
| 下游故障拖垮上游 | Circuit Breaker、Bulkhead、RateLimiter |
| 跨服务事件通知 | Spring Cloud Stream、Bus |
| 生产问题难定位 | Actuator、Micrometer、Tracing |

## 知识点详解

### Spring Cloud 不是微服务的充分条件

微服务不是“用了 Spring Cloud 就成立”。真正的微服务架构还要求：

- 服务边界清晰。
- 数据所有权明确。
- 跨服务调用可控。
- 发布和回滚独立。
- 观测、告警、容量和故障恢复到位。
- 团队能承担分布式复杂度。

如果业务规模小、团队人数少、模块边界不稳定，单体或模块化单体往往更合理。

### Spring Cloud 和 Kubernetes 的边界

Kubernetes 提供编排、调度、Service、Ingress、ConfigMap、Secret、探针、扩缩容等平台能力。Spring Cloud 提供应用内的调用、配置、熔断、路由、消息、观测集成能力。

两者会重叠：

- 服务发现：Kubernetes Service DNS vs Spring Cloud DiscoveryClient。
- 配置：ConfigMap/Secret vs Spring Cloud Config。
- 网关：Ingress/Gateway API vs Spring Cloud Gateway。
- 流量治理：Service Mesh vs 应用内 LoadBalancer/CircuitBreaker。

选择时看团队平台能力。如果平台已经有成熟 Service Mesh 和配置系统，Spring Cloud 可以只使用 Feign、Gateway、CircuitBreaker 等应用侧能力。

### 新项目常用核心组件

- Spring Cloud Gateway。
- Spring Cloud OpenFeign。
- Spring Cloud LoadBalancer。
- Spring Cloud Circuit Breaker。
- Spring Cloud Config 或平台配置系统。
- Spring Cloud Stream。
- Spring Cloud Kubernetes。
- Micrometer/Actuator/Tracing 集成。

历史和谨慎使用：

- Ribbon：旧客户端负载均衡方案，新项目使用 Spring Cloud LoadBalancer。
- Hystrix：旧熔断方案，新项目通常用 Resilience4j 体系。
- Zuul 1：旧网关方案，新项目用 Spring Cloud Gateway。

## 例子：订单链路

```text
用户
  -> Gateway: 鉴权、路由、限流
  -> order-service: 创建订单
  -> inventory-service: 扣库存
  -> payment-service: 预支付
  -> Stream: 发布 OrderCreated 事件
  -> notification-service: 异步发送通知
```

这个链路需要考虑：

- Gateway 是否正确路由。
- order-service 如何发现 inventory-service。
- 调用超时和重试如何设置。
- 库存服务慢时是否熔断。
- 创建订单和发消息如何保证一致性。
- traceId 如何贯穿所有服务。

## 练习

1. 画出一个电商微服务系统，包括 Gateway、订单、库存、支付、通知、配置中心、消息队列。
2. 标出同步调用和异步消息。
3. 给每条同步调用标注超时、重试、熔断策略。
4. 给每个服务标注它拥有的数据。

## 验收

- 能说清 Spring Cloud 解决的 6 个以上分布式问题。
- 能区分 Boot 单服务能力和 Cloud 多服务协作能力。
- 能指出微服务拆分的成本，而不是只说优点。
- 能解释为什么 Ribbon、Hystrix、Zuul 不应作为新项目默认选择。

## 重点

- Spring Cloud 是分布式模式工具集，不是业务拆分方法论。
- 网络调用默认不可靠，所有服务间调用都要有超时、降级和观测。
- 组件选型要结合 Kubernetes、云平台和团队运维能力。

## 难点

- Spring Cloud 与平台能力重叠，需要按组织环境取舍。
- 微服务失败常常是链式传播，不是单个组件配置错误。

## 易错

> **易错：** 项目还没有清晰边界，就先引入一整套微服务组件。
>
> 正确做法：先明确业务边界、数据边界和团队维护能力，再决定是否拆分。
