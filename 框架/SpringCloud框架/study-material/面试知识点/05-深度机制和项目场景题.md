## 一、服务发现和负载均衡深度题

### 1. 一个 `lb://inventory-service` 请求从服务名到实例的完整过程是什么？

**参考答案：**

先从 `lb://` 中解析出 serviceId，也就是 `inventory-service`。然后通过 DiscoveryClient 或 ServiceInstanceListSupplier 获取候选实例列表。实例列表可能经过健康、zone、metadata、灰度版本、权重等过滤。接着 LoadBalancer 用轮询、随机或自定义策略选出一个实例，把逻辑地址替换成真实 host/port，再由 HTTP 客户端发送请求。

排查时要分别看注册中心实例、调用方拿到的实例列表、筛选后的列表和最终选中实例。注册中心页面有实例，不代表调用方实际可用。

> **重点：** 服务发现负责“拿列表”，负载均衡负责“选实例”，调用保护负责“失败后怎么办”。

### 2. 服务下线后调用方还打到旧实例，原因可能是什么？

**参考答案：**

可能是注册中心剔除延迟、客户端实例列表缓存未刷新、Kubernetes readiness 还没失败、Gateway 或 Mesh 连接未排空，也可能是应用没有优雅停机。正确下线应先停止接新流量，再等待注册表和客户端缓存更新，处理完存量请求后关闭应用。

> **难点：** 下线窗口由注册中心、客户端缓存、网关、Kubernetes 和连接复用共同决定。

## 二、Feign 深度题

### 3. Feign 为什么容易让人低估远程调用风险？

**参考答案：**

Feign 把 HTTP 调用包装成 Java 接口方法，调用形式像本地方法，但底层仍然是网络请求。它会经历注解契约解析、参数展开、编码、拦截器、负载均衡、HTTP 调用、解码和错误处理。任何一步都可能失败，包括超时、连接失败、服务实例不可用、JSON 不兼容、Header 丢失、权限不足。

> **重点：** Feign 简化样板代码，不消除网络不可靠性。

### 4. Feign 调用放在数据库事务里有什么风险？

**参考答案：**

远程调用可能慢、超时或重试。如果放在数据库事务内，会延长事务时间，长时间占用数据库连接和锁。下游成功但本地事务回滚，还会产生一致性问题。更好的做法是缩短本地事务，把跨服务一致性通过幂等、outbox、Saga 或补偿处理。

> **易错：** 认为一个 `@Transactional` 能覆盖本地数据库和远程服务。

### 5. Feign 404、解码失败、超时分别怎么排查？

**参考答案：**

404 看服务名、path、Controller 路径、Gateway 路径改写和下游是否真的有接口。解码失败看响应 Content-Type、错误页是否是 HTML、DTO 字段类型、枚举和时间格式兼容性。超时看下游耗时、HTTP 客户端连接池、LoadBalancer 是否选到异常实例、是否重试放大、上游线程是否堆积。

## 三、Gateway 深度题

### 6. Gateway 的过滤链如何执行？

**参考答案：**

请求进入 Netty 后，经 WebFlux HandlerMapping 找到 RoutePredicateHandlerMapping。Predicate 匹配成功后，Gateway 构建 GlobalFilter 和当前 Route 的 GatewayFilter 链。过滤器按 order 执行，`lb://` 会由负载均衡相关过滤器解析，最后 NettyRoutingFilter 转发请求，NettyWriteResponseFilter 写回响应。

> **重点：** Predicate 没匹配就不会进入该 route 的 filters；路径改写发生在转发前。

### 7. Gateway 适合做哪些事，不适合做哪些事？

**参考答案：**

适合做统一入口、认证前置、粗粒度授权、路由、Header 清洗、限流、灰度标记、日志和指标。不适合做复杂业务逻辑、阻塞数据库查询、长时间同步调用、细粒度业务授权。Gateway 是高流量入口，阻塞 Filter 会影响整体吞吐。

## 四、配置中心深度题

### 8. Config Client 为什么要关注加载时机？

**参考答案：**

因为很多配置会影响自动配置，例如数据库地址、注册中心地址、日志级别、加密参数。如果 Config Server 配置加载太晚，相关 Bean 已经创建，配置就无法影响启动行为。现代方式通过 `spring.config.import` 在配置数据加载阶段导入远程配置，使后续自动配置能读取最终 Environment。

### 9. `/actuator/refresh` 后是不是所有配置都会生效？

**参考答案：**

不是。`@ConfigurationProperties` 这类绑定对象更适合刷新，但普通单例 Bean 构造时读入的值不会自动重建。连接池、线程池、HTTP 客户端、Kafka Consumer 等资源型 Bean 动态刷新要谨慎。多实例刷新还会短暂不一致。

> **重点：** 动态刷新是有限能力，不是重新启动应用。

## 五、熔断和级联故障深度题

### 10. 熔断器参数怎么影响系统行为？

**参考答案：**

熔断器通常按统计窗口内失败率或慢调用率判断是否打开。窗口太小会被少量异常误触发，窗口太大感知故障太慢；最小调用数太小低流量接口会误判，太大则低流量接口长期不触发；OPEN 等待时间太短会抖动，太长会恢复慢。参数必须结合接口 QPS、下游 SLA、业务重要性和超时预算。

### 11. 库存服务慢如何拖垮订单服务？

**参考答案：**

库存服务慢会让订单服务 Feign 线程等待，如果 Feign 调用在事务中，还会拉长数据库事务，占用连接和锁。连接池 pending 上升后订单接口变慢，Gateway 请求堆积，用户重试进一步放大流量，最终形成级联故障。治理要组合超时、Bulkhead、CircuitBreaker、Gateway 限流、幂等键和可观测性。

> **重点：** 慢下游比直接失败更危险，因为它会占用资源。

## 六、消息可靠性深度题

### 12. 消费者处理成功但消息又被消费一次，可能是什么原因？

**参考答案：**

可能是业务处理成功后 ack 前宕机，或者处理耗时超过消费者超时被重新投递，也可能是消费组再均衡、人工重放 offset、DLQ 修复后重投、生产者重试导致重复事件。因此消费者必须按至少一次投递设计幂等。

### 13. outbox 为什么比“业务成功后直接发消息”可靠？

**参考答案：**

直接发消息无法和本地数据库事务原子提交，可能出现订单已创建但消息没发，或消息发了但订单回滚。outbox 把业务数据和待发送事件写入同一个本地事务，再由发布器可靠发送，消费者幂等处理，形成最终一致性。

### 14. 事件版本如何演进？

**参考答案：**

新增字段通常兼容，删除字段、改字段类型、改枚举语义都高风险。事件应包含 eventId、eventType、eventVersion、occurredAt、producer、traceId。消费者应忽略未知字段，对不支持版本明确拒绝或进入 DLQ，而不是静默错误处理。

## 七、生产排障场景题

### 15. 一个跨服务请求 P99 升高，你如何给出证据链？

**参考答案：**

先从 Gateway 看 routeId、状态码和延迟，用 traceId 串联 order-service、Feign span、inventory-service 和数据库。再看 LoadBalancer 选中的 instanceId、CircuitBreaker 状态、Feign 超时指标、下游线程/连接池/GC、数据库慢查询和最近变更。证据链要能回答：入口是谁、路由是否正确、慢在哪个 span、哪个实例慢、当时资源是否饱和。

> **重点：** 没有 traceId、routeId、serviceId、instanceId，就很难做跨服务排障。
