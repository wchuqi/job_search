## 一、配置中心

### 1. Spring Cloud Config 解决什么问题？

**参考答案：**

它解决多服务、多环境、多实例配置集中管理、版本化、审计和回滚的问题。Config Server 从 Git、文件系统或其他后端读取配置，客户端通过 `spring.config.import` 在启动时导入配置，最终进入 Spring Environment。

> **重点：** 配置中心是配置供应链，不是线上随手改参数的后台。

### 2. 动态刷新有什么边界？

**参考答案：**

动态刷新可以让部分配置在不重启的情况下生效，但不是所有 Bean 都会自动重建。`@ConfigurationProperties` 更适合刷新，连接池、线程池、认证策略、复杂客户端配置要谨慎。多实例刷新还会存在短暂不一致。

> **易错：** 以为 `/refresh` 后所有配置和 Bean 都会完全重建。

## 二、熔断限流

### 3. CircuitBreaker、Retry、Bulkhead、RateLimiter 的区别？

**参考答案：**

CircuitBreaker 在下游持续失败或慢调用时打开，快速失败，避免拖垮上游。Retry 用于短暂错误重试，但必须限制次数和退避。Bulkhead 限制并发或隔离资源池，避免一个下游耗尽全部资源。RateLimiter 控制进入速率，保护系统容量。

> **重点：** 它们解决的问题不同，不能互相替代。

### 4. 熔断器有哪些状态？

**参考答案：**

通常有 CLOSED、OPEN、HALF_OPEN。CLOSED 正常放行并统计失败率；OPEN 快速拒绝或走 fallback；HALF_OPEN 放少量探测请求，成功则关闭，失败则再次打开。

### 5. fallback 应该怎么设计？

**参考答案：**

fallback 应保守。适合返回缓存、默认推荐、非核心功能降级或明确失败。不适合假装创建订单成功、吞掉支付失败或返回空数据掩盖严重故障。fallback 必须有日志和指标。

## 三、消息驱动

### 6. Spring Cloud Stream 中 destination、binding、binder、group 是什么？

**参考答案：**

destination 是消息目的地，如 Kafka topic 或 Rabbit exchange。binding 是应用函数输入输出与目的地的绑定关系。binder 是连接具体消息中间件的适配层。group 是消费组，同组实例竞争消费，不同组各自收到消息。

### 7. 为什么消息消费者必须幂等？

**参考答案：**

消息系统通常按至少一次投递设计，消费者可能因为重试、再均衡、超时、手动重放收到重复消息。消费者应使用事件 ID 去重表、唯一约束、业务状态机或幂等键保证重复消费不产生重复副作用。

### 8. outbox 模式解决什么问题？

**参考答案：**

它解决本地数据库事务和消息发送无法原子提交的问题。业务数据和 outbox 事件在同一数据库事务中提交，后台发布器可靠发送消息并标记状态，消费者幂等处理。这样避免“订单已创建但消息丢失”。

> **重点：** outbox 是最终一致性的常见基础模式。
