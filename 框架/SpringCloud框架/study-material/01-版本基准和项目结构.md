# Spring Cloud 学习资料：版本基准和项目结构

[返回索引](../SpringCloud学习资料.md)

## 学习目标

- 明确 Spring Cloud `2025.1.1` 的官方版本基准和 Boot 兼容关系。
- 理解 Release Train、BOM、starter、子项目版本之间的关系。
- 能创建一个多服务项目结构，并避免 Boot/Cloud 版本混用导致的依赖冲突。
- 能识别迁移中的历史组件替代关系。

## 理论导读

Spring Cloud 不是一个单独 Jar，而是一组项目的发布列车。Release Train 负责把 Spring Cloud Gateway、Config、OpenFeign、Stream、CircuitBreaker、Kubernetes 等子项目组合成一组经过验证的版本。业务项目通常不逐个指定这些子项目版本，而是导入 Spring Cloud BOM，让 Maven/Gradle 统一管理。

Spring Boot 与 Spring Cloud 的兼容关系非常关键。Cloud 的自动配置、HTTP 客户端、Actuator、Micrometer、Spring Framework API 都依赖 Boot 生态版本。混用不兼容版本时，常见表现是启动失败、自动配置条件不匹配、运行时类找不到或方法找不到。

## 版本基准

截至 2026-06-06，官方 Spring Cloud Release 文档显示：

| 项 | 当前基准 |
| --- | --- |
| 最新稳定 Release Train | `2025.1.1` |
| 代号 | Oakwood |
| 支持 Spring Boot | `4.0.2` |
| 其他稳定维护线 | `2025.0.2`、`2024.0.3`、`2023.0.6` |
| 快照线 | `2025.1.2-SNAPSHOT`、`2025.0.3-SNAPSHOT` |
| 主体子项目版本 | 多数为 `5.0.1`，Spring Cloud Contract 为 `5.0.2` |

> **重点：** 新项目优先评估最新稳定线；生产不要使用 SNAPSHOT。存量项目升级前先确认 Spring Boot、Spring Cloud、Spring Cloud Alibaba、公司内部 starter、网关插件和消息中间件 binder 的兼容性。

## Maven 依赖管理

```xml
<properties>
    <java.version>21</java.version>
    <spring-boot.version>4.0.2</spring-boot.version>
    <spring-cloud.version>2025.1.1</spring-cloud.version>
</properties>

<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>${spring-boot.version}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
        <dependency>
            <groupId>org.springframework.cloud</groupId>
            <artifactId>spring-cloud-dependencies</artifactId>
            <version>${spring-cloud.version}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

实际项目如果使用 Spring Boot parent，也可以只额外导入 Cloud BOM。

## 多模块项目结构

```text
spring-cloud-shop/
  pom.xml
  gateway-service/
  config-server/
  order-service/
  inventory-service/
  payment-service/
  notification-service/
  common-api/
  docker-compose.yml
```

结构原则：

- 每个业务服务独立 Spring Boot 应用。
- 公共模块只放稳定 API、DTO、错误码或工具，不放业务流程。
- 不要把所有服务强行依赖一个巨大的 common 包，否则服务边界会重新耦合。
- 配置、网关、业务服务、消息消费者要分开部署和观测。

## 子项目速览

| 子项目 | 作用 |
| --- | --- |
| Spring Cloud Commons | Discovery、LoadBalancer、通用抽象 |
| Spring Cloud Gateway | API 网关、路由、过滤、限流 |
| Spring Cloud OpenFeign | 声明式 HTTP 客户端 |
| Spring Cloud Config | 配置中心 |
| Spring Cloud Bus | 配置刷新和事件广播 |
| Spring Cloud Circuit Breaker | 熔断抽象，常接 Resilience4j |
| Spring Cloud Stream | 消息驱动应用抽象 |
| Spring Cloud Kubernetes | Kubernetes 服务发现、配置、负载均衡 |
| Spring Cloud Vault | Secret 管理集成 |
| Spring Cloud Contract | 消费者驱动契约测试 |

## 历史组件替代关系

| 老组件 | 新项目建议 |
| --- | --- |
| Ribbon | Spring Cloud LoadBalancer |
| Hystrix | Spring Cloud Circuit Breaker + Resilience4j |
| Zuul 1 | Spring Cloud Gateway |
| 手写 RestTemplate 调服务名 | OpenFeign 或 RestClient + LoadBalancer |

## 练习

1. 创建多模块 Maven 项目。
2. 引入 Spring Boot `4.0.2` 和 Spring Cloud `2025.1.1` BOM。
3. 创建 gateway、order-service、inventory-service 三个 Boot 应用。
4. 用 `mvn dependency:tree` 检查是否手工覆盖了 Cloud 管理版本。

## 验收

- 能解释 Release Train 和子项目版本的关系。
- 能说明为什么不要随意混用 Boot 和 Cloud 版本。
- 能创建清晰的多服务目录结构。
- 能列出 Ribbon、Hystrix、Zuul 的替代方案。

## 重点

- Spring Cloud 版本管理以 Release Train 为核心。
- BOM 是依赖版本一致性的底座。
- 公共模块越大，服务边界越容易失效。

## 难点

- 第三方生态未必同步支持最新 Cloud 版本。
- 公司内部 starter 可能绑定旧 Boot/Cloud API，迁移时风险高。

## 易错

> **易错：** 为解决一个依赖冲突，手动覆盖 Cloud 管理的核心库版本。
>
> 正确做法：先查兼容矩阵和依赖树，优先升级到匹配的 Release Train，而不是局部硬改版本。
