# Spring Cloud 学习资料：服务注册发现和负载均衡

[返回索引](../SpringCloud学习资料.md)

## 学习目标

- 理解服务注册发现解决的问题：服务实例动态变化时，调用方如何找到可用实例。
- 掌握 DiscoveryClient、ServiceInstance、服务注册表、健康检查和实例元数据。
- 理解 Spring Cloud LoadBalancer 的实例选择过程、缓存、重试和故障场景。
- 能排查“服务注册了但调用不到”“调用到下线实例”“负载不均”“Kubernetes 和注册中心重复发现”等问题。

## 理论导读

在单体或固定部署中，调用方可以写死 `http://10.0.0.12:8080`。但微服务实例会扩容、缩容、重启、漂移到不同节点，IP 和端口都可能变化。服务发现把“具体地址”抽象成“服务名”，调用方请求 `inventory-service`，再由发现组件返回当前可用实例列表，负载均衡器从中选择一个实例。

Spring Cloud Commons 提供通用发现抽象。底层可以是 Eureka、Consul、Zookeeper，也可以是 Kubernetes Service/Endpoints。新项目在 Kubernetes 中不一定需要独立注册中心，但仍需要理解 Spring Cloud 的 DiscoveryClient 和 LoadBalancer，因为 Feign、Gateway 的 `lb://` 调用都依赖它们。

## 核心心智模型

```text
服务启动
  -> 注册自身实例信息：serviceId、host、port、metadata、health
调用方发起请求
  -> 根据 serviceId 查询实例列表
  -> LoadBalancer 选择一个实例
  -> 组装真实 URL
  -> 发送 HTTP 请求
  -> 失败时按策略重试或降级
```

服务名是逻辑地址，实例是物理地址，负载均衡是从逻辑地址到物理地址的选择过程。

## 知识点详解

### 1. DiscoveryClient 抽象

`DiscoveryClient` 提供两个核心问题的答案：

- 当前有哪些服务名。
- 某个服务名下有哪些实例。

伪代码：

```java
List<ServiceInstance> instances = discoveryClient.getInstances("inventory-service");
```

`ServiceInstance` 通常包含：

- serviceId。
- host。
- port。
- scheme。
- metadata。
- instanceId。

metadata 可以存机房、版本、权重、zone、灰度标记等，但业务代码不应过度依赖注册中心元数据，否则服务治理逻辑会散落到业务里。

### 2. 服务健康和实例剔除

注册中心通常会结合心跳或健康检查判断实例是否可用。常见故障：

- 实例已经不可用，但注册表还没剔除。
- 实例刚启动，但依赖未准备好就被注册。
- readiness/liveness 配置错误，Kubernetes 认为可用但应用还不能处理请求。
- 网络分区导致调用方拿到过期实例列表。

正确做法：

- 服务只有 ready 后才接流量。
- 下线要优雅停机，先停止接新流量，再处理存量请求。
- 客户端必须设置超时、熔断和重试，不要完全信任注册表。

### 3. Spring Cloud LoadBalancer 决策过程

一次 `lb://inventory-service` 调用大致流程：

```text
解析 serviceId = inventory-service
  -> 查询 ServiceInstanceListSupplier
  -> 过滤实例：健康、zone、metadata、权重
  -> 选择实例：轮询、随机或自定义策略
  -> 替换为真实 host:port
  -> 执行请求
```

常见扩展点：

- `ServiceInstanceListSupplier`：实例列表来源和过滤。
- `ReactorServiceInstanceLoadBalancer`：响应式选择器。
- 缓存配置：减少频繁查询注册中心。

### 4. 实例选择的深层规则

负载均衡不是“随便挑一个实例”。一个成熟选择过程通常包含多层筛选：

```text
原始实例列表
  -> 按 serviceId 获取候选实例
  -> 过滤不可用实例
  -> 按 zone/region 过滤或优先
  -> 按 metadata 过滤灰度、版本、租户
  -> 按权重或轮询策略选择
  -> 记录选择结果和调用指标
```

常见问题是“注册中心里有实例，但调用方拿到的候选列表为空”，这时要区分：

- 注册中心没有该 serviceId。
- serviceId 大小写或命名不一致。
- 客户端缓存还没刷新。
- zone/metadata 过滤把所有实例过滤掉。
- Kubernetes Service selector 没有匹配 Pod。
- NetworkPolicy 或安全组阻止访问。

> **难点：** 服务发现问题不能只看注册中心页面。调用方实际拿到的实例列表、负载均衡筛选后的列表、最终选中的实例，三者都要看。

### 5. 灰度、区域亲和和优雅上下线

服务实例 metadata 常用于灰度和区域亲和：

```text
inventory-service A: version=v1, zone=shanghai-a
inventory-service B: version=v2, zone=shanghai-a, canary=true
```

灰度调用可以按 Header、用户 ID、租户或百分比选择版本。但灰度规则应尽量集中在网关、Mesh 或治理层，不要散落在业务 Service。trace 和日志要记录命中的版本和实例，否则故障时无法判断请求走了哪条路径。

服务下线时的正确顺序：

```text
停止接收新流量
  -> readiness 失败或从注册中心摘除
  -> 等待客户端缓存和连接排空
  -> 处理完存量请求
  -> 关闭应用
```

如果直接 kill 进程，调用方可能仍然拿到旧实例，导致短时间 502、connection refused 或 timeout。Kubernetes、注册中心、Gateway 和客户端 LoadBalancer 的缓存时间都会影响下线窗口。

### 6. 负载均衡和重试的风险

重试不是免费能力：

- 非幂等请求重试可能重复下单、重复扣款。
- 下游已经慢了，上游重试会放大流量。
- 网关、Feign、LoadBalancer、HTTP 客户端都可能有重试，叠加后请求数爆炸。

规则：

- GET 类幂等请求可以谨慎重试。
- POST/支付/扣库存默认不自动重试，除非有幂等键。
- 重试次数、退避、超时必须和熔断一起设计。

### 7. Kubernetes 场景

Kubernetes 已有 Service DNS 和 Endpoints。Spring Cloud Kubernetes 可以把 Kubernetes API 映射为 DiscoveryClient，也可以使用平台原生 DNS。

选择思路：

- 简单服务调用：Kubernetes DNS 足够。
- 需要 DiscoveryClient 抽象、Feign/Gateway `lb://`、metadata 过滤：使用 Spring Cloud Kubernetes。
- 已有 Service Mesh：很多负载均衡和流量治理可能交给 Mesh。

## 例子

Feign 通过服务名调用：

```java
@FeignClient(name = "inventory-service")
public interface InventoryClient {
    @GetMapping("/inventory/{sku}")
    InventoryResponse findBySku(@PathVariable String sku);
}
```

Gateway 使用服务名：

```yaml
spring:
  cloud:
    gateway:
      server:
        webflux:
          routes:
            - id: inventory
              uri: lb://inventory-service
              predicates:
                - Path=/api/inventory/**
              filters:
                - StripPrefix=1
```

## 练习

1. 启动两个 inventory-service 实例，端口不同，serviceId 相同。
2. 用 order-service 通过 Feign 调用 inventory-service。
3. 观察请求是否分散到两个实例。
4. 停掉一个实例，观察调用方多久感知。
5. 增加超时和熔断，避免调用挂死。
6. 给一个实例加 `version=v2` metadata，设计灰度筛选规则。

## 验收

- 能解释 DiscoveryClient 和 LoadBalancer 的职责边界。
- 能说明 `lb://service-name` 如何变成真实地址。
- 能列出注册表不准确时的保护手段。
- 能判断什么时候使用 Kubernetes DNS，什么时候使用 Spring Cloud Kubernetes Discovery。
- 能说清实例从注册表到最终选中之间的过滤和选择过程。

## 重点

- 服务发现只解决“找到实例”，不保证实例一定健康。
- 负载均衡策略、重试、超时和熔断必须一起考虑。
- 服务名是稳定逻辑地址，实例地址是动态物理地址。

## 难点

- 注册中心、客户端缓存、Kubernetes readiness 三者状态可能短暂不一致。
- 重试和负载均衡叠加后可能放大故障。

## 易错

> **易错：** 以为注册中心返回实例就代表实例一定可用。
>
> 正确做法：调用方仍必须有超时、熔断、重试边界和可观测性。
