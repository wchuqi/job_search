# Spring Cloud 学习资料：迁移、兼容和版本管理

[返回索引](../SpringCloud学习资料.md)

## 学习目标

- 掌握 Spring Cloud Release Train 与 Spring Boot 的兼容检查方法。
- 能制定从旧版本迁移到 `2025.1.1` 的步骤。
- 能识别 Ribbon、Hystrix、Zuul、旧 bootstrap 配置方式等历史包袱。
- 能处理依赖冲突、自动配置变化、配置属性变化和生产回滚。

## 理论导读

Spring Cloud 主版本迁移比单个库升级更复杂，因为它连接 Spring Boot、Spring Framework、HTTP 客户端、网关、配置中心、服务发现、熔断库、消息 binder、Kubernetes API 和公司内部 starter。升级失败常表现为运行时方法找不到、自动配置不生效、配置不加载、Gateway 路由变化、Feign 编码解码变化、熔断策略失效。

迁移不是一次改版本号，而是一条可验证、可回滚的路径。

## 知识点详解

### 1. 版本检查清单

- Spring Boot 是否匹配 Cloud Release Train。
- Java 版本。
- Spring Cloud Alibaba 或其他第三方组件兼容性。
- Gateway、Feign、Config、Stream、Kubernetes 子项目版本。
- 内部 starter 是否使用旧 API。
- 构建插件和容器镜像。
- Actuator 端点和安全规则。
- 配置属性迁移。

### 2. 迁移路径

1. 升级到当前旧主线的最新 patch。
2. 修复所有弃用警告。
3. 增加集成测试和契约测试。
4. 替换历史组件：Ribbon -> LoadBalancer，Hystrix -> CircuitBreaker/Resilience4j，Zuul -> Gateway。
5. 切换 Boot 和 Cloud BOM。
6. 跑完整测试和压测。
7. 灰度发布。
8. 保留回滚版本和配置。

### 3. 配置迁移

旧项目可能依赖 bootstrap 阶段配置加载。现代 Boot/Cloud 配置通常使用 `spring.config.import`。迁移时要确认 Config Client 是否在足够早的阶段加载配置。

### 4. 依赖冲突排查

命令：

```powershell
mvn dependency:tree
```

或：

```powershell
gradle dependencies
```

重点查：

- Spring Framework 版本。
- Spring Boot 版本。
- Spring Cloud 子项目版本。
- Reactor/Netty。
- Jackson。
- Resilience4j。
- Kafka/Rabbit binder。

## 练习

1. 找一个旧 Cloud 项目，列出 Ribbon/Hystrix/Zuul 使用点。
2. 生成迁移清单。
3. 替换为 LoadBalancer、CircuitBreaker、Gateway。
4. 用集成测试验证路由、Feign、熔断、配置加载。
5. 制定回滚方案。

## 验收

- 能说明 `2025.1.1` 与 Boot `4.0.2` 的关系。
- 能列出历史组件替代方案。
- 能用依赖树定位版本冲突。
- 能写出可灰度、可回滚的迁移计划。

## 重点

- Cloud 迁移要按 release train 迁移。
- 替换旧组件比硬兼容更可靠。
- 配置加载机制变化是高风险点。

## 难点

- 第三方生态和内部 starter 常滞后于最新主线。
- 网关和熔断迁移可能改变运行时行为。

## 易错

> **易错：** 直接把 Spring Cloud BOM 改成最新版本并上线。
>
> 正确做法：先兼容性评估、替换旧组件、补测试、灰度发布和准备回滚。
