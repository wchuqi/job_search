# Spring Boot 学习资料：自动配置和启动流程

[返回索引](../SpringBoot学习资料.md)

## 学习目标

- 能从 `main` 方法说清楚应用启动到可接收请求之间的关键阶段。
- 理解自动配置不是“魔法”，而是一套基于类路径、Bean、属性、资源、应用类型的条件化装配机制。
- 掌握自动配置导入、条件评估、Bean 覆盖、配置顺序、诊断报告和自定义 starter 的核心规则。
- 能排查“starter 引了但没有生效”“自定义 Bean 没覆盖默认 Bean”“本地能启动线上启动失败”等问题。

## 理论导读

Spring Boot 的核心价值不是替代 Spring Framework，而是把常见工程约定固化成可预测的默认配置。你引入 `spring-boot-starter-web` 后，不需要手写 `DispatcherServlet`、内嵌 Tomcat、Jackson 消息转换器、错误处理、静态资源等配置，是因为 Boot 根据当前类路径和配置属性推断出你在构建一个 Servlet Web 应用，然后导入一批候选自动配置类。每个候选配置类都带条件，条件满足才注册 Bean。

这套机制的重点是“证据驱动”：类路径中有什么类、`Environment` 中有什么属性、容器里已有何种 Bean、当前应用类型是什么、资源文件是否存在。Boot 并不会无条件创建所有 Bean，而是尽量“默认配置先铺底，用户配置优先覆盖”。因此学习自动配置必须同时理解启动流程、BeanDefinition 注册时机、条件判断时机和诊断入口。

## 核心心智模型

把 Spring Boot 启动想成一次“装配流水线”：

```text
main 方法
  -> 创建 SpringApplication
  -> 推断应用类型和主配置类
  -> 准备 Environment 和配置数据
  -> 创建 ApplicationContext
  -> 读取用户配置类和自动配置类
  -> 注册 BeanDefinition
  -> 条件评估决定哪些配置有效
  -> refresh 容器，实例化单例 Bean
  -> 启动内嵌 Web 服务器
  -> 执行 Runner
  -> 应用进入可服务状态
```

自动配置像“候选零件清单”。每个零件都写着安装条件：某个类必须存在、某个属性必须开启、用户没有提供同类零件、当前必须是 Web 应用。条件满足就安装；用户已经安装了自己的零件，默认零件通常退让。

## 知识点详解

### 1. `SpringApplication.run` 的关键阶段

`SpringApplication.run` 不是简单创建一个 IoC 容器，它会组织完整启动生命周期：

1. 创建并初始化 `SpringApplication`：推断主类、应用类型、初始化器、监听器。
2. 准备启动监听：发布启动事件，给日志、配置加载、失败分析等扩展点机会。
3. 准备 `Environment`：加载命令行参数、系统属性、环境变量、配置文件、Profile 配置和导入配置。
4. 创建 `ApplicationContext`：Servlet 应用通常创建 Web 相关上下文，Reactive 和非 Web 应用不同。
5. 加载 BeanDefinition：用户的 `@Configuration`、`@Component`、自动配置类都会在这个阶段变成 Bean 定义。
6. 刷新容器：执行 BeanFactory 后处理器，实例化非懒加载单例 Bean，创建代理，完成依赖注入。
7. 启动 WebServer：Servlet 应用会创建并启动 Tomcat/Jetty 等内嵌容器，注册 Servlet、Filter、Listener。
8. 执行 `ApplicationRunner`、`CommandLineRunner`：适合做启动后的轻量初始化，不适合做不可控耗时任务。

> **重点：** 自动配置条件大多在 BeanDefinition 注册阶段判断；Bean 初始化顺序是另一件事。不要把“自动配置类排序”和“Bean 创建顺序”混为一谈。

### 2. `@SpringBootApplication` 到底做了什么

`@SpringBootApplication` 组合了三类能力：

| 能力 | 注解 | 作用 |
| --- | --- | --- |
| Boot 配置类标记 | `@SpringBootConfiguration` | 本质是特殊的 `@Configuration` |
| 自动配置开关 | `@EnableAutoConfiguration` | 触发自动配置类导入 |
| 组件扫描 | `@ComponentScan` | 从当前包向下扫描 `@Component`、`@Service`、`@Controller` 等 |

主类包名非常重要。如果主类放在 `com.example.app`，组件扫描默认覆盖 `com.example.app` 及其子包；如果 Controller 放在 `com.example.web` 而主类放在 `com.example.app`，就可能扫描不到。

```java
@SpringBootApplication
public class OrderApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderApplication.class, args);
    }
}
```

> **易错：** 把启动类放在很深的包里，导致同级或上级业务包扫描不到。
>
> 正确做法：启动类放在业务根包，或明确配置 `scanBasePackages`，但后者要谨慎，避免扫描范围过大。

### 3. 自动配置类如何被导入

Boot 会从依赖中的自动配置元数据读取候选自动配置类。你引入 starter 后，starter 通常会带来一个或多个 auto-configuration 模块。Boot 在启动时读取这些候选类，然后通过条件过滤。

自动配置导入不是组件扫描。自动配置类即使不在你的主类包下面，也能被导入，这是 starter 能跨项目生效的关键。

常见判断证据：

| 条件 | 判断内容 | 典型用途 |
| --- | --- | --- |
| `@ConditionalOnClass` | 类路径上是否存在某类 | 有 Jackson 才配置 JSON |
| `@ConditionalOnMissingClass` | 类路径上是否不存在某类 | 避免与某技术栈冲突 |
| `@ConditionalOnBean` | 容器中是否已有某 Bean | 依赖用户提供的核心组件 |
| `@ConditionalOnMissingBean` | 容器中是否没有某 Bean | 默认 Bean 让位给用户 Bean |
| `@ConditionalOnProperty` | 配置属性是否满足 | 功能开关 |
| `@ConditionalOnResource` | 某资源是否存在 | SQL 初始化、模板资源 |
| `@ConditionalOnWebApplication` | 是否 Web 应用 | MVC、错误页、WebServer |
| `@ConditionalOnNotWebApplication` | 是否非 Web 应用 | CLI、批处理 |

### 4. 条件评估的决策顺序

排查自动配置时按这个顺序最有效：

1. 依赖是否存在：starter 是否引入，类路径上是否有目标类。
2. 应用类型是否正确：Servlet、Reactive、非 Web 是否被错误推断。
3. 配置属性是否正确：属性名、Profile、环境变量映射、默认值是否符合条件。
4. 用户 Bean 是否已经存在：是否触发了 `@ConditionalOnMissingBean` 退让。
5. 自动配置顺序是否影响判断：某些条件依赖先注册的 BeanDefinition。
6. Bean 创建是否失败：条件匹配只代表会尝试创建，不代表实例化一定成功。

> **难点：** `@ConditionalOnMissingBean` 判断的是当前 BeanFactory 中已经注册的 BeanDefinition 或 Bean，而不是未来一定会注册的 Bean。自定义自动配置时如果顺序设计不当，可能出现默认 Bean 抢先注册，导致用户扩展不生效。

### 5. 自动配置排序和 Bean 初始化排序

自动配置类之间可以用 `@AutoConfigureBefore`、`@AutoConfigureAfter`、`@AutoConfigureOrder` 表达顺序。这个顺序主要影响配置类处理和 BeanDefinition 注册时机。

Bean 实例化顺序则由依赖关系、生命周期回调、懒加载、`@DependsOn` 等因素决定。不要用自动配置排序去解决业务 Bean 初始化先后问题。

### 6. 用户 Bean 如何覆盖默认行为

Boot 常见设计是：

```java
@Bean
@ConditionalOnMissingBean
ObjectMapper objectMapper() {
    return new ObjectMapper();
}
```

如果用户自己声明一个 `ObjectMapper`，默认 Bean 不创建。但这不等于“所有自动配置都能被同名 Bean 覆盖”。有些扩展点推荐提供 `Customizer`，例如配置 Jackson、WebMvc、RestClient、线程池等。

选择方式：

| 需求 | 推荐方式 |
| --- | --- |
| 完全替换默认组件 | 声明同类型核心 Bean |
| 局部调整默认组件 | 使用 `Customizer` 或属性 |
| 禁用整段自动配置 | `spring.autoconfigure.exclude` 或注解排除 |
| 创建可复用集成 | 写 starter 和 auto-configuration |

### 7. 自动配置诊断入口

| 工具 | 回答的问题 |
| --- | --- |
| `--debug` | 哪些自动配置匹配、哪些没匹配 |
| `/actuator/conditions` | 运行时查看条件评估报告 |
| `/actuator/beans` | 容器最终有哪些 Bean |
| `/actuator/configprops` | 配置属性最终绑定成什么 |
| `/actuator/env` | 某配置来自哪个属性源 |
| 启动失败分析器 | 某些典型错误的友好说明 |

排查模板：

```text
问题：Redis 自动配置没有生效
1. pom/gradle 是否有 spring-boot-starter-data-redis
2. conditions 中 RedisAutoConfiguration 是否 matched
3. spring.data.redis.* 属性是否被正确加载
4. 是否已有 RedisConnectionFactory Bean
5. Profile 是否导致 redis 配置没加载
6. Redis 服务是否可达，Bean 实例化是否失败
```

### 8. 自定义 starter 的基本结构

一个成熟 starter 通常分成两层：

```text
audit-spring-boot-starter
  -> 只放依赖聚合，方便业务方引入

audit-spring-boot-autoconfigure
  -> AuditAutoConfiguration
  -> AuditProperties
  -> 默认实现类
  -> 自动配置元数据
```

最小自动配置示例：

```java
@AutoConfiguration
@EnableConfigurationProperties(AuditProperties.class)
@ConditionalOnClass(AuditLogger.class)
@ConditionalOnProperty(prefix = "audit", name = "enabled", havingValue = "true", matchIfMissing = true)
public class AuditAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    AuditLogger auditLogger(AuditProperties properties) {
        return new DefaultAuditLogger(properties.topic(), properties.includeBody());
    }
}
```

```java
@ConfigurationProperties(prefix = "audit")
public record AuditProperties(
        String topic,
        boolean includeBody) {
}
```

自动配置注册文件通常位于：

```text
META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
```

内容示例：

```text
com.example.audit.AuditAutoConfiguration
```

## 例子：从条件报告定位问题

现象：引入 Web starter 后访问接口 404。

排查：

1. `/actuator/beans` 中是否有目标 Controller。
2. 如果没有，检查启动类包名和组件扫描范围。
3. 如果有，打开 MVC 映射日志或 Actuator mappings。
4. 检查路径、HTTP 方法、`consumes`、`produces`、上下文路径。
5. 检查是否被 Security 拦截，401/403 被误认为 404。

## 练习

实现一个 `audit.enabled=true` 才启用的审计 starter：

1. 定义 `AuditProperties`，包含 `topic`、`includeBody`、`timeout`。
2. 定义 `AuditLogger` 接口和默认实现。
3. 编写 `AuditAutoConfiguration`。
4. 支持用户声明自己的 `AuditLogger` 覆盖默认实现。
5. 写测试覆盖：类路径存在、属性关闭、用户 Bean 覆盖、默认属性绑定。

## 验收

- 能从 conditions 报告解释一个自动配置为什么匹配或不匹配。
- 能说明 `@ConditionalOnMissingBean` 的退让规则和时机风险。
- 能区分自动配置排序、Bean 依赖顺序、组件扫描范围。
- 能写出最小可用的自定义 auto-configuration。
- 能定位 starter 引入后不生效的具体原因，而不是靠猜。

## 重点

- 自动配置是条件化 BeanDefinition 注册，不是无条件创建 Bean。
- 用户 Bean、属性、类路径、Profile、应用类型共同决定最终装配结果。
- 诊断自动配置问题优先看 conditions、beans、env、configprops。

## 难点

- 条件评估发生在容器启动过程的特定阶段，复杂 starter 的顺序和退让策略容易写错。
- 自动配置类顺序不等于运行时 Bean 初始化顺序。
- 多个 starter 组合时，依赖版本冲突可能表现为条件不匹配、Bean 创建失败或运行时方法找不到。

## 易错

> **易错：** Bean 没注入就立刻手写配置覆盖。
>
> 正确做法：先查依赖、条件报告、配置属性、Profile、已有 Bean，再决定是补 starter、改属性、声明 Bean 还是排除自动配置。

> **易错：** 在业务代码中大量使用 `@ComponentScan` 扫全公司包。
>
> 正确做法：保持清晰包结构；真正跨项目复用的能力用 starter 和自动配置，而不是扩大扫描范围。
