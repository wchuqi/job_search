# Spring Boot 学习资料：数据访问、事务和数据库迁移

[返回索引](../SpringBoot学习资料.md)

## 学习目标

- 理解 Boot 如何自动配置 DataSource、连接池、事务管理器、Repository 和数据库迁移工具。
- 能根据业务复杂度选择 JDBC、JdbcClient、Spring Data JDBC、JPA/Hibernate 或 R2DBC。
- 掌握事务代理、传播行为、隔离级别、回滚规则、失效场景和一致性边界。
- 能用 Flyway 或 Liquibase 管理数据库结构演进，并避免生产迁移事故。

## 理论导读

数据访问是 Spring Boot 应用中最容易出现正确性和性能问题的层。框架可以帮你创建连接池、事务管理器和 Repository，但它不会替你决定事务边界、索引设计、锁策略、SQL 复杂度、迁移脚本发布顺序和失败恢复方案。

事务不是“给方法加一个注解”这么简单。`@Transactional` 通常通过 AOP 代理生效，代理在方法调用前开启事务，方法成功后提交，遇到符合规则的异常时回滚。如果调用没有经过代理、异常被吞掉、数据库引擎不支持事务、跨线程执行，事务就可能不按预期工作。

## 核心心智模型

一次写库业务像办理一组银行操作：开启窗口、执行多步操作、全部成功后提交，失败则撤销。`@Transactional` 定义的是窗口边界；连接池提供窗口资源；数据库隔离级别决定不同窗口之间能看到什么；迁移工具管理银行规则的版本变化。

```text
Controller
  -> Service 事务边界
  -> Repository / Mapper
  -> Connection from Pool
  -> Database
```

## 知识点详解

### 1. Boot 数据访问自动配置

引入数据访问 starter 后，Boot 会根据类路径和配置尝试自动配置：

- `DataSource`。
- 连接池，常见是 HikariCP。
- `JdbcTemplate`、`JdbcClient`。
- `PlatformTransactionManager`。
- Spring Data Repository。
- SQL 初始化。
- Flyway 或 Liquibase。

常见配置：

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/orderdb
    username: order
    password: ${DB_PASSWORD}
    hikari:
      maximum-pool-size: 20
      connection-timeout: 2s
  flyway:
    enabled: true
```

> **重点：** 连接池大小不是越大越好。它受数据库最大连接数、应用实例数、SQL 耗时、并发量共同约束。

### 2. 数据访问技术选择

| 技术 | 适合 | 风险 |
| --- | --- | --- |
| JDBC/JdbcClient | SQL 可控、查询明确、性能敏感 | 手写映射较多 |
| MyBatis | SQL 主导、复杂报表、团队熟悉 SQL | 动态 SQL 复杂后难维护 |
| Spring Data JDBC | 聚合简单、偏领域模型但不需要 ORM | 关联能力不如 JPA |
| JPA/Hibernate | 复杂对象关系、脏检查、Repository 抽象 | N+1、懒加载、缓存和 SQL 不透明 |
| R2DBC | 端到端响应式数据库访问 | 生态、事务和调试成本更高 |

选择原则：

- 强 SQL 控制和复杂查询：JDBC/MyBatis。
- 常规 CRUD 且团队理解 ORM：JPA。
- 响应式链路必须端到端非阻塞：R2DBC。
- 不要因为“高级”而选择 JPA，也不要因为“简单”忽略 JDBC 的重复成本。

### 3. 事务代理如何工作

`@Transactional` 通常通过代理实现：

```text
调用方 -> 代理对象 -> 开启事务 -> 目标方法 -> 提交/回滚 -> 返回
```

因此它有几个关键约束：

- 调用必须经过 Spring 管理的代理对象。
- 通常放在 Service 层 public 方法上。
- 同一个类内部 `this.method()` 调用不会经过代理。
- 异步线程不会继承原线程事务。
- 私有方法、构造方法上的事务不会按普通预期生效。

```java
@Service
class OrderService {

    @Transactional
    public void createOrder(CreateOrderCommand command) {
        saveOrder(command);
        publishOutboxEvent(command);
    }

    private void saveOrder(CreateOrderCommand command) {
        // private 方法本身不是事务入口，但在 createOrder 的事务范围内执行
    }
}
```

> **易错：** 在同类方法中用 `this.innerTransactionalMethod()` 期望开启新事务。
>
> 正确做法：把需要独立事务的方法放到另一个 Spring Bean，或使用明确的事务模板。

### 4. 传播行为

| 传播行为 | 语义 | 典型场景 |
| --- | --- | --- |
| `REQUIRED` | 有事务加入，没有就新建 | 默认业务写操作 |
| `REQUIRES_NEW` | 挂起外层事务，开启新事务 | 记录审计、outbox、独立流水 |
| `SUPPORTS` | 有事务加入，没有就非事务 | 可选事务读 |
| `MANDATORY` | 必须已有事务 | 强制由上层用例控制 |
| `NOT_SUPPORTED` | 挂起事务，以非事务执行 | 不应占用事务的慢操作 |
| `NEVER` | 有事务则报错 | 禁止事务上下文 |
| `NESTED` | 嵌套事务或保存点 | 局部回滚，依赖数据库和事务管理器支持 |

`REQUIRES_NEW` 不是免费能力：它会额外占用连接。如果外层事务持有一个连接，内层新事务再取一个连接，高并发下可能更快耗尽连接池。

### 5. 回滚规则

默认情况下，Spring 对运行时异常和 Error 回滚，对受检异常不一定回滚。需要时显式声明：

```java
@Transactional(rollbackFor = IOException.class)
public void importOrders(Path file) throws IOException {
    // ...
}
```

如果异常被捕获并吞掉，事务代理看到方法正常返回，就会提交。

```java
@Transactional
public void create() {
    try {
        repository.save(order);
        remoteCall();
    } catch (Exception ex) {
        log.error("failed", ex);
    }
    // 这里可能提交
}
```

正确做法是重新抛出异常，或明确设置回滚状态。

### 6. 隔离级别和并发现象

数据库隔离级别决定并发事务之间能看到什么。

| 现象 | 含义 |
| --- | --- |
| 脏读 | 读到其他事务未提交数据 |
| 不可重复读 | 同一事务两次读同一行结果不同 |
| 幻读 | 同一事务两次范围查询出现新行 |

常见隔离级别：

- `READ_COMMITTED`：很多数据库默认，避免脏读。
- `REPEATABLE_READ`：同一事务内重复读更稳定，但实现因数据库不同。
- `SERIALIZABLE`：最强，性能和锁冲突成本高。

业务并发控制不能只靠隔离级别。库存扣减、余额变更、状态机流转常需要：

- 唯一约束。
- 乐观锁版本号。
- 悲观锁。
- 条件更新。
- 幂等表。

### 7. JPA 的关键机制

JPA/Hibernate 的核心机制包括：

- 持久化上下文：事务内管理实体状态。
- 脏检查：事务提交时检测实体字段变化并生成 SQL。
- 懒加载：关联对象用到时才查。
- flush：把内存状态同步到数据库，不等于提交事务。

常见坑：

- N+1 查询：遍历列表时每条记录额外查关联。
- 懒加载异常：离开事务后访问懒加载属性。
- Entity 暴露给 JSON 序列化触发大量查询。
- 批量更新绕过持久化上下文，导致内存对象和数据库不一致。

### 8. 数据库迁移

Flyway 文件命名示例：

```text
db/migration/V1__create_order_table.sql
db/migration/V2__add_order_status.sql
db/migration/V3__create_idempotency_key.sql
```

原则：

- 已在共享环境执行过的版本脚本不要修改。
- 每个变更向前演进，必要时新增修复脚本。
- DDL 大表变更要评估锁表、回填、索引创建耗时。
- 应用代码和数据库变更要考虑发布顺序。

安全发布模式：

1. 先添加兼容字段或表，不删除旧字段。
2. 部署能同时兼容新旧结构的应用。
3. 回填数据。
4. 切流量或切读写路径。
5. 观察稳定后再删除旧字段。

## 例子：事务和幂等创建订单

```java
@Service
class OrderService {
    private final OrderRepository orderRepository;
    private final IdempotencyRepository idempotencyRepository;

    OrderService(OrderRepository orderRepository, IdempotencyRepository idempotencyRepository) {
        this.orderRepository = orderRepository;
        this.idempotencyRepository = idempotencyRepository;
    }

    @Transactional
    public OrderResponse create(String idempotencyKey, CreateOrderRequest request) {
        idempotencyRepository.insertIfAbsent(idempotencyKey);
        Order order = Order.create(request.productId(), request.quantity());
        Order saved = orderRepository.save(order);
        return OrderResponse.from(saved);
    }
}
```

数据库唯一约束防重复：

```sql
CREATE TABLE idempotency_key (
    key_value VARCHAR(128) PRIMARY KEY,
    created_at TIMESTAMP NOT NULL
);
```

## 练习

1. 用 PostgreSQL 和 Flyway 创建订单表、幂等键表。
2. 实现创建订单 Service，使用 `@Transactional`。
3. 制造运行时异常，验证订单和幂等键是否一起回滚。
4. 制造同类内部调用事务失效案例，并修复。
5. 写一个 JPA 查询，观察是否出现 N+1，并用 fetch join 或 EntityGraph 修复。

## 验收

- 能画出事务代理调用路径。
- 能解释 `REQUIRED` 和 `REQUIRES_NEW` 的连接占用差异。
- 能说明默认回滚规则。
- 能列出至少 5 种事务失效场景。
- 能用迁移脚本从空库初始化完整结构。
- 能解释为什么修改已执行的 Flyway 脚本是危险操作。

## 重点

- 事务边界应在 Service 用例层，而不是 Controller 或 Repository 零散方法。
- 事务正确性依赖代理、异常、数据库能力和调用路径。
- 数据库迁移必须版本化、可回滚、可观测。
- ORM 不能替代 SQL、索引和并发控制意识。

## 难点

- 事务传播和连接池之间存在资源耦合，高并发下尤其明显。
- JPA 的持久化上下文、flush、懒加载和事务边界连在一起，错误使用会产生隐蔽问题。
- 生产数据库 DDL 变更可能影响锁、复制延迟、回滚窗口和发布顺序。

## 易错

> **易错：** 捕获异常后只记录日志，方法正常返回，导致事务提交。
>
> 正确做法：需要回滚时重新抛出异常，或明确设置事务回滚状态。

> **易错：** 生产上修改已经执行过的 Flyway 版本脚本。
>
> 正确做法：新增修复版本脚本，保持迁移历史可追溯。
