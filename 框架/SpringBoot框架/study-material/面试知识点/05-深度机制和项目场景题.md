## 一、自动配置深度题

### 1. 你怎么解释 Spring Boot 自动配置的底层决策过程？

**参考答案：**

自动配置是条件化 BeanDefinition 注册。Boot 启动时读取依赖中的自动配置候选类，再根据类路径、配置属性、已有 Bean、资源、应用类型和 Profile 判断哪些配置类生效。生效的配置类会注册 Bean；很多默认 Bean 使用 `@ConditionalOnMissingBean`，所以用户自定义 Bean 通常优先。

排查时我会看 `/actuator/conditions` 或 `--debug`，确认是缺类、属性不匹配、已有 Bean、应用类型不匹配，还是配置类被排除。

> **重点：** 自动配置不是运行时随机猜测，而是启动期基于条件的确定性决策。
>
> **难点：** 自动配置类顺序影响 BeanDefinition 注册时机，但不等于 Bean 实例化顺序。

### 2. starter 引入了，但 Bean 没有创建，你怎么排查？

**参考答案：**

我会按六步排查：第一看依赖树确认 starter 和 auto-configuration 是否真的进入类路径；第二看 conditions 报告中的 negative match；第三查配置属性和 Profile 是否满足；第四查是否已有同类型 Bean 触发 `@ConditionalOnMissingBean`；第五查自动配置是否被 exclude；第六看 Bean 创建阶段是否因为连接、权限、版本冲突失败。

> **易错：** 直接复制网上配置手动创建 Bean，可能掩盖真正的依赖或配置问题。

## 二、配置体系深度题

### 3. 生产配置和 YAML 文件不一致，你怎么定位最终值来自哪里？

**参考答案：**

不能只看仓库里的 `application-prod.yml`。Spring Boot 会把配置文件、Profile 文件、环境变量、Java 系统属性、命令行参数、配置导入和测试覆盖属性合并到 `Environment`。我会通过 `/actuator/env` 查目标 key 的最终值和属性源，再检查 `SPRING_PROFILES_ACTIVE`、环境变量映射、启动参数和配置中心。

例如 `APP_CACHE_MAX_SIZE` 会通过松散绑定映射到 `app.cache.max-size`。容器环境里经常是环境变量覆盖了文件值。

> **重点：** 配置问题的核心问题是“最终值来自哪个 PropertySource”。

### 4. 为什么复杂配置推荐 `@ConfigurationProperties` 而不是到处写 `@Value`？

**参考答案：**

`@ConfigurationProperties` 可以把一组相关配置建模成类型安全对象，支持嵌套结构、类型转换、Bean Validation 校验、IDE 元数据提示和集中测试。`@Value` 适合少量简单值，但大量使用会导致配置分散、校验困难、重构成本高。

项目里像线程池、第三方服务、JWT、存储服务这类配置，我会建单独 properties 类，并让关键字段缺失时启动失败。

> **重点：** 配置错误应尽早在启动期失败，而不是运行到业务路径才失败。

## 三、Web 链路深度题

### 5. 一次请求进入 Spring MVC 后发生了什么？

**参考答案：**

请求先进入 Servlet 容器，再经过 Filter chain，包括 Spring Security 过滤器。之后进入 `DispatcherServlet`，由 `HandlerMapping` 根据路径、HTTP 方法、Header、参数、consumes、produces 等条件选择 Controller 方法。`HandlerAdapter` 调用方法，期间 `ArgumentResolver` 解析参数，类型转换和 Bean Validation 校验入参。方法返回后由 `ReturnValueHandler` 和 `HttpMessageConverter` 把对象序列化成 HTTP 响应。如果过程中抛异常，会进入 `@RestControllerAdvice` 或默认错误处理。

> **难点：** 路由匹配不只看 URL，还看方法、媒体类型、Header 和参数条件。

### 6. 为什么不建议 Controller 直接返回 JPA Entity？

**参考答案：**

Entity 是持久化模型，不是 API 契约。直接返回会把数据库字段暴露给客户端，字段变更会破坏兼容性；懒加载字段可能在 JSON 序列化时触发 N+1 或 LazyInitializationException；双向关联还可能循环序列化；敏感字段也容易泄漏。更稳妥的做法是 Request DTO、Response DTO、领域对象/Entity 分离。

> **重点：** DTO 是接口边界，Entity 是持久化边界，两者职责不同。

## 四、事务深度题

### 7. `@Transactional` 为什么会失效？

**参考答案：**

常见原因包括：同类内部方法调用没有经过代理；方法不是 public 或事务注解放在不合适的位置；异常被捕获后方法正常返回；默认回滚规则不覆盖受检异常；异步线程不继承原线程事务；数据库引擎或操作本身不支持事务；对象不是 Spring 容器管理的 Bean。

我会先确认调用是否经过代理，再看异常是否抛出到事务拦截器，最后看数据库和事务管理器能力。

> **重点：** `@Transactional` 的核心是代理拦截方法调用，不是给任意代码块自动加事务。

### 8. `REQUIRES_NEW` 有什么风险？

**参考答案：**

`REQUIRES_NEW` 会挂起外层事务并开启新事务，通常会额外获取数据库连接。高并发下，如果外层事务已经持有连接，内层新事务再取连接，连接池可能更快耗尽。它适合审计、outbox、独立流水等确实需要独立提交的场景，但不能随意使用。

> **难点：** 事务传播不只是语义问题，也会影响连接池资源。

### 9. 订单创建成功后要发 Kafka 消息，如何保证一致性？

**参考答案：**

不能用本地数据库事务直接覆盖 Kafka。常见方案是 outbox：订单和待发送消息在同一个本地事务中落库，事务提交后由后台任务或发布器把 outbox 消息发送到 Kafka，成功后标记已发送。消费者按业务键做幂等。这样即使发送失败，也能通过 outbox 重试，不会丢消息。

如果使用事务消息，也仍要设计重复消费、补偿和监控。

> **重点：** 分布式一致性要设计失败恢复路径，不能只说“加事务”。

## 五、安全深度题

### 10. 401、403、CORS、CSRF 问题如何区分？

**参考答案：**

401 是未认证，通常是没带凭据、Token 过期或签名无效。403 是已认证但权限不足，也可能是 CSRF 校验失败。CORS 是浏览器跨域策略，常表现为浏览器拦截响应，后端可能实际返回了 401 或 403。CSRF 主要防基于 Cookie 的跨站请求伪造；无状态 Bearer Token API 常见禁用 CSRF，但 Cookie Session 应保留。

排查时先看后端状态码和安全日志，再看浏览器预检 OPTIONS 和响应头。

> **易错：** 看到浏览器 CORS 报错就只改 CORS，忽略真实后端认证失败。

### 11. 多个 `SecurityFilterChain` 如何匹配？

**参考答案：**

每条 `SecurityFilterChain` 先用 `securityMatcher` 判断是否处理当前请求，匹配到对应链后，再用链内部的 `requestMatchers` 做授权规则判断。多条链需要用 `@Order` 控制顺序。常见做法是 Actuator 一条链、API 一条链，管理端点和业务接口分开授权。

> **重点：** `securityMatcher` 是选链，`requestMatchers` 是链内授权。

## 六、测试和生产排障深度题

### 12. 为什么不应该所有测试都用 `@SpringBootTest`？

**参考答案：**

`@SpringBootTest` 会加载完整上下文，能发现装配问题，但慢且容易受配置影响。Controller 的参数绑定、校验、异常处理适合 `@WebMvcTest`；Repository 和 SQL 适合 `@DataJpaTest` 加真实数据库容器；跨层流程再用 `@SpringBootTest`。这样反馈更快，也能更准确定位失败原因。

> **重点：** 测试范围应匹配验证目标，范围越大不是越好。

### 13. 线上连接池耗尽，你怎么处理？

**参考答案：**

先止血，必要时限流、降级、回滚新版本或关闭入口。证据上看 Hikari active、idle、pending、timeout，结合数据库慢查询、锁等待、线程 dump 和最近变更。根因可能是慢 SQL、事务范围过大、连接泄漏、实例总连接数超过数据库上限，或 `REQUIRES_NEW` 嵌套事务额外消耗连接。不能只调大连接池，因为可能把数据库压垮。

> **重点：** 连接池耗尽通常是症状，不是根因。

### 14. 接口 P99 突然升高，你的完整排查路径是什么？

**参考答案：**

先确认影响面：单接口、全局、单实例还是全实例。再查最近发布、配置和流量变化。指标上看 HTTP 延迟、错误率、CPU、内存、GC、线程池、连接池。Trace 找慢 span，SQL/APM 查慢查询，下游指标看外部 API，线程 dump 看阻塞点。恢复动作优先回滚、限流、降级、熔断、扩容，根因修复放在服务恢复后做。

> **难点：** P99 升高可能来自数据库、GC、线程阻塞、下游慢、锁竞争、日志 IO 或容器资源限制。
