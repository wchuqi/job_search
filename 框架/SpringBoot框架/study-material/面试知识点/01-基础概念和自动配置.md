## 一、基础概念

### 1. Spring Boot 和 Spring Framework 有什么区别？

**参考答案：**

Spring Framework 提供 IoC、AOP、事务、Web、资源抽象等底层能力；Spring Boot 基于 Spring Framework，提供 starter、依赖版本管理、自动配置、外部化配置、Actuator、构建插件和可执行 Jar，让 Spring 应用更容易创建、配置、部署和运维。

项目中，Spring MVC、Spring Security、Spring Data 仍然负责具体能力，Boot 负责把这些能力按约定装配起来。

> **重点：** Boot 不是 Spring 的替代品，而是 Spring 应用的工程化装配体系。

### 2. `@SpringBootApplication` 做了什么？

**参考答案：**

它通常组合了 `@SpringBootConfiguration`、`@EnableAutoConfiguration` 和 `@ComponentScan`。前者标记配置类，中间启用自动配置，后者从主类所在包向下扫描组件。

主启动类位置很重要。如果放在很深的子包，其他包的 Controller、Service、Repository 可能扫描不到。

> **易错：** 只回答“三个注解组合”不够，应补充组件扫描边界和自动配置作用。

### 3. Starter 和自动配置有什么区别？

**参考答案：**

Starter 是依赖聚合，解决“引入哪些 Jar”的问题；自动配置是运行时配置类，解决“容器里创建哪些 Bean”的问题。Starter 提供类路径条件，自动配置根据这些条件创建默认 Bean。

例如引入 Web MVC starter 后，类路径上有 MVC 和内嵌服务器相关类，Boot 才能自动配置 DispatcherServlet、消息转换器和 Web 服务器。

> **重点：** 依赖在构建期，自动配置在应用启动装配期。

## 二、自动配置

### 4. Spring Boot 自动配置的生效条件有哪些？

**参考答案：**

常见条件包括类路径是否存在某类、配置属性是否满足、容器中是否已有某 Bean、当前是否 Web 应用、当前 Profile、资源是否存在等。对应注解包括 `@ConditionalOnClass`、`@ConditionalOnProperty`、`@ConditionalOnMissingBean`、`@ConditionalOnWebApplication`、`@Profile`。

排查时看 conditions 报告，而不是猜。

> **重点：** 类路径、属性、已有 Bean、应用类型、Profile 是高频判断维度。

### 5. 如何覆盖 Spring Boot 默认配置？

**参考答案：**

优先级通常是：

1. 通过官方配置属性调整。
2. 声明自己的 Bean，让 `@ConditionalOnMissingBean` 的默认 Bean 退让。
3. 使用定制器，例如 Jackson、WebMvc、Security 的 customizer。
4. 必要时排除自动配置类。

不建议一开始就排除自动配置，因为容易丢掉相关默认基础设施。

> **易错：** 为了改一个细节直接 `exclude` 整个自动配置类。

### 6. 自动配置为什么没有生效，怎么排查？

**参考答案：**

先确认 starter 是否引入，再看配置属性和 Profile，然后检查是否已有用户 Bean 覆盖，最后查看 conditions 报告。可以启动加 `--debug`，也可以使用 Actuator `conditions`、`beans`、`configprops`、`env`。

如果是运行时类找不到或方法不存在，要检查依赖版本冲突和第三方 starter 兼容性。

> **难点：** 很多问题不是某一个配置错，而是依赖、属性、Bean 和 Profile 共同导致。

### 7. 如何设计一个自定义 starter？

**参考答案：**

通常拆成两部分：业务库提供核心能力，starter 提供自动配置。自动配置要使用 `@ConfigurationProperties` 暴露配置，使用条件注解控制生效，使用 `@ConditionalOnMissingBean` 允许用户覆盖，并提供配置元数据。

设计时要避免无条件创建重量级连接、线程池或外部客户端，避免污染使用方应用上下文。

> **重点：** 自定义 starter 的目标是提供合理默认值和清晰覆盖点。
