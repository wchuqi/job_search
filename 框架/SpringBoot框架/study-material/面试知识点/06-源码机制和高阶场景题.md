## 一、源码机制和高阶场景题

### 1. 请按源码主线讲一下自动配置如何导入。

**参考答案：**

`@SpringBootApplication` 启用 `@EnableAutoConfiguration`，核心由 `AutoConfigurationImportSelector` 读取 `AutoConfiguration.imports` 中的候选配置类，去重、处理 exclude、应用过滤器、排序后返回给 Spring 配置类解析流程。之后类级和方法级条件再决定具体 Bean 是否注册。

> **重点：** 导入候选配置类和创建 Bean 是两个阶段。

### 2. 为什么 `@ConditionalOnMissingBean` 可能受顺序影响？

**参考答案：**

它判断当前 BeanFactory 中是否已经存在匹配 Bean。如果某个默认 Bean 先注册，另一个条件判断结果就会变化。因此自动配置之间需要排序，并且用户配置通常应优先于自动配置。

> **难点：** 条件判断不是脱离上下文的静态判断，它依赖当时容器状态。

### 3. ConfigData 为什么发生在 refresh 之前？

**参考答案：**

配置会影响自动配置条件、BeanDefinition 注册、Profile、日志、Web 类型、端口等启动决策，所以必须在 ApplicationContext refresh 前准备好 Environment。

> **重点：** 配置加载不是普通 Bean 初始化阶段的事情。

### 4. 一个 Starter 引入后导致线上 Bean 被替换，你怎么排查？

**参考答案：**

先看 dependency tree 确认新增 Starter 带来的依赖和自动配置，再看 conditions 报告确认哪些自动配置新生效，检查是否注册了同类型 Bean、`@Primary`、`@ConditionalOnMissingBean`、自动配置顺序，最后写回归测试固定期望 Bean。

> **易错：** 只删依赖不分析根因，后续升级还会复发。

### 5. Actuator 指标为什么不能把 orderId 放进 tag？

**参考答案：**

orderId 是高基数标签，会让指标时序数量快速膨胀，导致 Prometheus 或指标后端内存和查询压力暴涨。指标 tag 应使用有限枚举维度，例如 result、method、status。

> **重点：** 指标系统适合聚合趋势，不适合记录每个业务实体。

### 6. 如何设计一个企业内部 Starter？

**参考答案：**

Starter 只聚合依赖和自动配置，不直接绑定业务应用。自动配置要有条件开关、属性类、默认 Bean、`@ConditionalOnMissingBean`、配置元数据、测试用例和文档。敏感默认值不能硬编码。

> **难点：** 企业 Starter 的质量取决于是否可覆盖、可诊断、可升级。

### 7. Spring Boot 应用启动很慢，你会怎么定位？

**参考答案：**

先看启动日志和 ApplicationStartup 指标，检查组件扫描范围、Bean 数量、初始化阶段外部调用、JPA 和数据库迁移、自动配置数量、日志输出、类加载、Testcontainers 或远程依赖。必要时拆分初始化逻辑或使用 AOT。

> **重点：** 启动慢要分阶段定位，不要只盲目加 lazy initialization。

### 8. 从 Boot 3 升到 Boot 4，你会如何制定计划？

**参考答案：**

先升级到 Boot 3 最新补丁并清理 warning，补齐测试，再升级 Boot 4 BOM，处理 JDK、Spring Framework、Jakarta、第三方 Starter、Spring Cloud、Actuator、测试和部署兼容性。灰度发布并保留回滚镜像和数据库兼容方案。

> **易错：** 直接改版本上线，不检查第三方 Starter 和 Spring Cloud 版本矩阵。
