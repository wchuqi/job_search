## 一、迁移升级

### 1. 从 Spring Boot 3.x 升级到 4.x，你会怎么做？

**参考答案：**

先确认 JDK、Spring Framework、Spring Cloud、第三方 starter、数据库驱动、构建插件是否支持 Boot 4。然后把项目升级到当前 3.x 最新维护版，修复弃用警告和测试问题，再切换到 4.x。升级后运行完整测试、查看自动配置报告、验证 Actuator、压测关键接口，并准备回滚版本。

> **重点：** 主版本升级是依赖生态和运行行为验证，不是只改版本号。

### 2. 升级后出现 `NoSuchMethodError` 怎么排查？

**参考答案：**

这通常是依赖版本冲突或二进制不兼容。先用 Maven/Gradle 查看依赖树，确认报错类来自哪个 Jar，再检查是否手工覆盖了 Boot BOM 管理版本，或第三方 starter 引入了旧版本依赖。解决时优先遵循 Boot BOM 和官方兼容矩阵。

> **易错：** 只改一个库版本压住错误，可能引入更深的兼容问题。

## 二、排障

### 3. 应用启动失败，你会按什么顺序排查？

**参考答案：**

先看错误栈最底层根因，再判断是配置绑定、Bean 创建、端口占用、数据库连接、依赖冲突还是自动配置条件不满足。必要时加 `--debug` 查看 conditions 报告，使用 `mvn dependency:tree` 或 Gradle dependencies 分析依赖。

> **重点：** 启动失败先读根因，不要只看最外层 `ApplicationContext` 失败。

### 4. 线上接口突然变慢，如何定位？

**参考答案：**

先确认影响范围和最近变更，再看 HTTP 请求耗时指标、错误率、实例资源、GC、线程、连接池。然后用 trace 定位慢 span，用 SQL 日志或 APM 查慢查询，用线程 dump 查阻塞。恢复优先可采用回滚、扩容、限流、降级，再做根因修复。

> **难点：** 慢接口可能是数据库、连接池、线程池、下游 API、GC 或网络任一层导致。

### 5. 自动配置报告里某配置 negative match，怎么处理？

**参考答案：**

先看 negative match 的具体条件：缺类就检查 starter，属性不匹配就检查配置来源和 Profile，已有 Bean 导致不创建就检查自定义 Bean，非 Web 应用条件不满足就检查应用类型。不要直接复制网上配置覆盖。

> **重点：** conditions 报告已经给了判断依据，按条件逐项排查。

## 三、场景题

### 6. 订单服务创建订单后要发消息，如何保证一致性？

**参考答案：**

常见方案是 outbox 模式：订单和待发送消息记录在同一个本地事务中提交，后台任务或消息转发器可靠发送消息，发送成功后标记状态。消费端还要按业务键做幂等。也可以根据消息中间件能力使用事务消息，但仍要设计补偿和重复消费处理。

> **重点：** 不要只说“加事务”，数据库事务不能天然覆盖外部消息系统。

### 7. 多实例部署下定时任务重复执行怎么办？

**参考答案：**

可以使用分布式锁、任务调度平台、数据库任务表抢占、Kubernetes CronJob，或让任务逻辑幂等。选择方案取决于任务重要性、频率、失败重试和运维体系。

> **易错：** 以为 `@Scheduled` 在集群里天然只执行一次。

### 8. 如何保护 Actuator 端点？

**参考答案：**

只暴露必要端点，如 `health`、`info`、`metrics`、`prometheus`。敏感端点不暴露或仅内网加认证访问。可将管理端口和业务端口隔离，配置 Spring Security 对 `/actuator/**` 单独授权，并确保敏感值脱敏。

> **重点：** Actuator 端点一旦泄漏，可能暴露配置、内存、线程和运行状态。

### 9. 如何解释 Spring Boot 的“约定优于配置”？

**参考答案：**

Boot 为常见场景提供默认约定，例如目录结构、端口、日志、JSON、错误处理、内嵌服务器和 starter 依赖组合。开发者遵循约定时无需大量配置；需要变化时可以用属性、Bean、定制器或排除自动配置覆盖默认行为。

> **重点：** 约定不是不可改，而是提供默认起点和标准覆盖方式。
