# Spring Boot 学习资料：集成能力：缓存、消息、定时任务

[返回索引](../SpringBoot学习资料.md)

## 学习目标

- 掌握 Boot 对缓存、消息、任务调度和异步执行的常见自动配置。
- 理解集成外部系统时的幂等、重试、超时和降级。
- 能设计稳定的后台任务和消息消费逻辑。

## 理论导读

真实业务服务通常不只是 HTTP 加数据库，还会接入 Redis、Kafka、RabbitMQ、邮件、对象存储、第三方 API、定时任务和异步线程池。Boot 能简化接入，但系统稳定性取决于你是否处理超时、重试、幂等、重复消费、顺序和故障恢复。

## 核心心智模型

集成外部系统时，默认假设外部系统会慢、会失败、会重复、会乱序。你的代码要能承受这些现实。

## 知识点详解

### 缓存

Spring Cache 抽象通过 `@Cacheable`、`@CacheEvict`、`@CachePut` 隔离缓存实现。底层可以是 Caffeine、Redis 等。

```java
@Cacheable(cacheNames = "products", key = "#id")
public Product findProduct(String id) {
    return productClient.findById(id);
}
```

缓存适合读多写少、允许短暂不一致的数据。强一致业务不要盲目加缓存。

### 消息

消息系统解决异步解耦、削峰和事件驱动。消费端必须考虑：

- 重复消费：使用业务幂等键。
- 消费失败：重试和死信队列。
- 顺序性：按业务 key 分区或设计补偿。
- 事务一致性：本地事务和消息发送之间可能不一致。

### 定时任务

`@Scheduled` 适合单实例简单任务。多实例部署时，可能每个实例都执行一次，需要分布式锁、任务平台或只在一个实例运行。

### 异步执行

`@Async` 需要显式线程池。默认线程池配置不适合生产高负载。

## 例子

```java
@Configuration
@EnableScheduling
class SchedulingConfig {
}

@Component
class OrderCleanupJob {
    @Scheduled(cron = "0 */10 * * * *")
    void closeExpiredOrders() {
        // close expired orders idempotently
    }
}
```

## 练习

1. 给商品查询增加 Caffeine 本地缓存。
2. 订单创建后发送领域事件。
3. 消费事件时实现幂等表。
4. 写一个关闭超时订单的定时任务，并说明多实例部署方案。

## 验收

- 缓存有 TTL 和容量上限。
- 消息消费可重复执行且结果一致。
- 定时任务不会在多实例下重复造成业务错误。
- 外部调用配置超时。

## 重点

- 缓存提升读性能，但会引入一致性问题。
- 消息消费一定要设计幂等。
- 定时任务在集群部署中不是天然单例。

## 难点

- 本地事务提交和消息发送之间的一致性，需要 outbox、事务消息或补偿设计。
- 重试会放大下游压力，必须配合退避、限流和死信。

## 易错

> **易错：** 消费消息时只按消息 ID 去重。
>
> 正确做法：以业务幂等键去重，例如订单号加事件类型，因为同一业务事件可能被重新投递或重新生成消息。
