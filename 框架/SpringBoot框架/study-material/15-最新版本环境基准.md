# Spring Boot 学习资料：最新版本环境基准

[返回索引](../SpringBoot学习资料.md)

## 学习目标

- 明确 2026-06-06 核对到的 Spring Boot 最新稳定版本。
- 能为新项目选择 JDK、构建工具、Servlet 容器和 Native Image 工具链。
- 能解释为什么不能只看 Spring Boot 版本号，还要同时检查 Spring Framework、第三方 starter、Spring Cloud 和运行环境。

## 版本基准

截至 2026-06-06，Spring Boot 官方文档页面标记的最新稳定版本是 `4.0.6`。同一版本选择器中仍列出 `3.5.14`、`3.4.13`、`3.3.13` 稳定维护线，`4.1.0-RC1` 是预览版本，`4.1.0-SNAPSHOT`、`4.0.7-SNAPSHOT`、`3.5.15-SNAPSHOT` 是快照版本。

> **重点：** 新项目优先评估 `4.0.6`；生产项目不要使用 RC 或 SNAPSHOT，除非明确做兼容性验证或框架适配测试。

## 官方系统要求速查

| 类别 | Spring Boot 4.0.6 要求 |
| --- | --- |
| Java | 至少 Java 17，兼容到 Java 26 |
| Spring Framework | Spring Framework 7.0.7 或更高 |
| Maven | 3.6.3 或更高 |
| Gradle | Gradle 8.x，其中 8.14 或更高；以及 Gradle 9.x |
| 内嵌 Tomcat | Tomcat 11.0.x，Servlet 6.1 |
| 内嵌 Jetty | Jetty 12.1.x，Servlet 6.1 |
| 外部 Servlet 容器 | Servlet 6.1+ 兼容容器 |
| GraalVM Native Image | GraalVM 25 或更高 |
| Native Build Tools | 0.11.5 |

## 理论导读

Spring Boot 是一个集成型框架：它本身的版本只是入口，真正运行时还包含 Spring Framework、Spring Security、Spring Data、日志、JSON、数据库驱动、Servlet 容器、构建插件和测试工具。Boot 的 BOM 会把这些依赖组合成一套经过验证的版本矩阵，所以选型时要看“整套生态是否支持”，不能只把 `spring-boot-starter-parent` 改成最新版本。

## 核心心智模型

把 Spring Boot 版本看成一张“兼容性菜单”。菜单里已经搭配好了主菜和配菜：Spring Framework、Tomcat、Jackson、Hibernate、Micrometer、测试库等。如果你私自替换其中一个关键版本，就像把某个配件换成不同规格，编译可能通过，但运行时会出现方法找不到、类找不到、自动配置条件不匹配或行为变化。

## 新项目选型建议

1. 普通后端 REST 服务：选择 Spring Boot `4.0.6`、Java 21 或 Java 25/26 中团队已标准化的 LTS/运行版本、Maven 3.9+ 或 Gradle 8.14+。
2. 企业存量系统：如果依赖 Spring Cloud、商业中间件、公司内部 starter，先确认它们是否支持 Boot 4；不支持时可停留在官方仍列出的 3.5.x 稳定维护线。
3. 容器部署：优先使用 Boot 插件、buildpacks 或明确的 Dockerfile；运行镜像的 JDK 版本必须和编译、测试环境一致。
4. Native Image：只在启动时间、内存占用、弹性伸缩有明确收益时采用；所有反射、动态代理、资源加载和第三方库都要验证。

## 例子

Maven 父版本：

```xml
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>4.0.6</version>
    <relativePath/>
</parent>
```

Java 版本属性：

```xml
<properties>
    <java.version>21</java.version>
</properties>
```

Gradle 插件版本：

```kotlin
plugins {
    id("org.springframework.boot") version "4.0.6"
    id("io.spring.dependency-management") version "1.1.7"
    java
}
```

## 练习

1. 用 Spring Initializr 创建一个 Boot `4.0.6` 项目，选择 Web、Validation、Actuator、Data JDBC 或 JPA。
2. 分别用 Maven 和 Gradle 查看依赖树，找到 Spring Framework、Tomcat、Jackson、Micrometer 的实际版本。
3. 把项目切换到 Java 17、Java 21 两个运行环境，分别执行测试和启动。
4. 增加一个第三方 starter，检查它是否声明支持 Boot 4。

## 验收

- 能说出 Spring Boot `4.0.6` 的 Java、Spring Framework、Maven、Gradle、Servlet 容器、GraalVM 要求。
- 能判断新项目是否适合直接采用 Boot 4。
- 能为存量项目列出升级前必须检查的依赖兼容项。
- 能解释为什么生产环境不应使用 RC 或 SNAPSHOT。

## 重点

- 最新稳定版不是最新预览版；稳定线、RC、SNAPSHOT 用途不同。
- Boot 4.0.6 要求 Java 17+ 和 Spring Framework 7.0.7+。
- Servlet 栈进入 Servlet 6.1 生态，Tomcat 11 和 Jetty 12.1 是重点容器版本。

## 难点

- 编译通过不代表运行兼容，很多问题会在自动配置、条件匹配、运行 Profile、容器镜像或集成测试阶段暴露。
- Spring Cloud、公司内部 starter、老版本 ORM 插件通常是升级 Boot 主版本的主要阻力。

## 易错

> **易错：** 看到 `4.1.0-RC1` 比 `4.0.6` 新，就直接用于生产。
>
> 正确做法：生产默认选择最新稳定版；RC 用于提前适配和验证，SNAPSHOT 用于框架开发或问题复现。

> **易错：** 只升级 Spring Boot，不升级构建工具、JDK 镜像和 CI 环境。
>
> 正确做法：把本地、CI、测试、生产镜像里的 JDK、Maven/Gradle、插件和运行参数一起纳入版本基准。
