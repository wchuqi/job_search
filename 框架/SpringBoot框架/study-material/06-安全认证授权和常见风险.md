# Spring Boot 学习资料：安全认证、授权和常见风险

[返回索引](../SpringBoot学习资料.md)

## 学习目标

- 理解 Spring Security 在 Boot 应用中的默认行为和自动配置边界。
- 掌握认证、授权、`SecurityFilterChain`、过滤器顺序、CSRF、CORS、Session、JWT、OAuth2 Resource Server 和方法级权限。
- 能设计默认拒绝、最小授权、端点隔离、密码安全和 Actuator 保护策略。
- 能排查 401、403、CORS 失败、CSRF 失败、Token 校验失败和多安全链匹配错误。

## 理论导读

引入 `spring-boot-starter-security` 后，Boot 会启用默认安全配置：大多数请求需要认证，开发时可能生成默认用户密码。真实项目不能依赖默认配置，而要显式定义自己的 `SecurityFilterChain`，明确哪些路径公开、哪些路径需要认证、哪些权限可访问、认证失败和权限不足如何返回、Actuator 如何隔离。

Spring Security 的本质是一条过滤器链。请求还没到 Controller，就会经过多个安全过滤器：提取凭据、认证身份、保存安全上下文、检查 CSRF、处理异常、做授权判断。很多“接口没进 Controller”的问题，本质是被安全链拦截了。

## 核心心智模型

```text
HTTP Request
  -> SecurityFilterChain 匹配
  -> 认证过滤器提取凭据
  -> AuthenticationManager 验证
  -> SecurityContext 保存身份
  -> AuthorizationManager 判断权限
  -> Controller
```

认证回答“你是谁”，授权回答“你能做什么”。认证失败是 401，认证成功但权限不足是 403。

## 知识点详解

### 1. 默认安全行为

引入 Security starter 后，如果你不声明自己的配置，Boot 会提供默认安全行为。默认行为适合提醒你“应用需要安全”，不适合作为生产策略。

生产必须明确：

- 哪些接口匿名访问。
- 哪些接口需要认证。
- 权限模型是角色、scope、authority 还是 ABAC。
- Token、Session 或 mTLS 如何认证。
- Actuator 是否与业务端点分离。
- 认证失败和授权失败返回格式。

### 2. `SecurityFilterChain` 和匹配顺序

可以有多个 `SecurityFilterChain`。每条链先按 security matcher 判断是否适用，匹配到后使用该链的规则。顺序非常关键。

```java
@Configuration
class SecurityConfig {

    @Bean
    @Order(1)
    SecurityFilterChain actuatorSecurity(HttpSecurity http) throws Exception {
        return http
                .securityMatcher(EndpointRequest.toAnyEndpoint())
                .authorizeHttpRequests(auth -> auth
                        .requestMatchers(EndpointRequest.to("health", "info")).permitAll()
                        .anyRequest().hasRole("OPS"))
                .httpBasic(Customizer.withDefaults())
                .build();
    }

    @Bean
    @Order(2)
    SecurityFilterChain apiSecurity(HttpSecurity http) throws Exception {
        return http
                .securityMatcher("/api/**")
                .csrf(csrf -> csrf.disable())
                .authorizeHttpRequests(auth -> auth
                        .requestMatchers(HttpMethod.GET, "/api/orders/**").hasAuthority("SCOPE_order:read")
                        .requestMatchers(HttpMethod.POST, "/api/orders/**").hasAuthority("SCOPE_order:write")
                        .anyRequest().authenticated())
                .oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults()))
                .build();
    }
}
```

> **难点：** `securityMatcher` 决定某条安全链是否处理请求；`requestMatchers` 决定在这条链内部如何授权。两者不是一回事。

### 3. Session、JWT 和 OAuth2 Resource Server

| 模式 | 适合 | 风险 |
| --- | --- | --- |
| Session Cookie | 服务端 Web、传统后台 | CSRF、防固定会话、集群 Session |
| JWT Bearer Token | 前后端分离、资源服务器 | Token 泄漏、过期、撤销困难 |
| OAuth2 Resource Server | 接入统一身份提供方 | issuer、audience、scope 映射要准确 |

JWT 资源服务器配置：

```yaml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://issuer.example.com
```

权限映射要和身份提供方约定一致。很多 403 问题不是 Token 无效，而是 scope/authority 前缀或 claim 映射不一致。

### 4. CSRF 和 CORS 的边界

CORS 是浏览器跨域访问控制，防的是浏览器随意读取跨域响应。CSRF 是跨站请求伪造防护，防的是浏览器自动携带 Cookie 发起恶意请求。

| 机制 | 关注点 | 常见误解 |
| --- | --- | --- |
| CORS | 浏览器是否允许跨域请求/读取响应 | CORS 不是认证 |
| CSRF | 攻击者借用户 Cookie 发起请求 | 不是所有 API 都能无脑关闭 |

无状态 Bearer Token API 常见禁用 CSRF，因为 Token 不应由浏览器自动附带；基于 Cookie Session 的表单应用通常应开启 CSRF。

### 5. 密码存储

密码不能明文、不能可逆加密、不能自己发明算法。使用 `PasswordEncoder`，例如 bcrypt、argon2、pbkdf2。

```java
@Bean
PasswordEncoder passwordEncoder() {
    return PasswordEncoderFactories.createDelegatingPasswordEncoder();
}
```

密码校验是“比较哈希”，不是解密。密码重置应走一次性 token 和过期时间，不应把原密码发回用户。

### 6. 方法级权限

URL 权限适合粗粒度路径控制，方法级权限适合业务对象级控制。

```java
@PreAuthorize("hasAuthority('SCOPE_order:read')")
public OrderResponse findById(Long id) {
    return orderRepository.findById(id)
            .map(OrderResponse::from)
            .orElseThrow(OrderNotFoundException::new);
}
```

对象级权限例如“只能查看自己的订单”，通常需要业务字段参与判断，不能只靠 URL。

### 7. Actuator 安全

可公开的通常只有健康探针的有限信息。以下端点必须严格限制：

- `/actuator/env`
- `/actuator/configprops`
- `/actuator/heapdump`
- `/actuator/threaddump`
- `/actuator/loggers`
- `/actuator/conditions`

管理端点最好使用独立端口、独立网络策略和独立权限。

### 8. 安全排查路径

401：

1. 请求是否带凭据。
2. Token 是否过期、签名是否正确。
3. issuer、audience 是否匹配。
4. 认证过滤器是否处理了该请求。

403：

1. 是否已经认证成功。
2. authority/scope 是否符合规则。
3. 方法级权限是否额外拒绝。
4. CSRF 是否拒绝了非 GET 请求。

CORS 失败：

1. 浏览器控制台看预检请求。
2. 后端是否允许 Origin、Method、Header。
3. 是否允许 credentials。
4. Security 是否放行 OPTIONS。

## 例子：统一安全错误响应

```java
@Bean
AuthenticationEntryPoint authenticationEntryPoint(ObjectMapper objectMapper) {
    return (request, response, authException) -> {
        response.setStatus(HttpStatus.UNAUTHORIZED.value());
        response.setContentType(MediaType.APPLICATION_JSON_VALUE);
        objectMapper.writeValue(response.getOutputStream(),
                new ApiError("UNAUTHORIZED", "未认证", request.getHeader("X-Trace-Id"), List.of()));
    };
}
```

## 练习

1. 配置两条安全链：Actuator 链和 API 链。
2. 健康检查匿名可访问，其他 Actuator 端点需要 `OPS`。
3. API 使用 JWT，GET 需要 `order:read`，POST 需要 `order:write`。
4. 为 Cookie Session 示例开启 CSRF，为 Bearer Token API 禁用 CSRF。
5. 写安全测试覆盖 401、403、CORS 预检和健康检查。

## 验收

- 能解释 401 和 403 的区别。
- 能说明 `securityMatcher` 与 `requestMatchers` 的区别。
- 能判断一个 CSRF 配置是否适合当前认证模式。
- Actuator 敏感端点不匿名暴露。
- 密码使用强哈希存储。

## 重点

- 默认拒绝，明确放行。
- 认证、授权、CORS、CSRF 是不同层次的问题。
- 多安全链必须明确匹配范围和顺序。

## 难点

- JWT 的 claim 到 authority 映射容易和身份提供方不一致。
- 浏览器 CORS 报错可能掩盖真实后端 401/403。
- URL 权限无法覆盖所有对象级授权需求。

## 易错

> **易错：** `permitAll` 写得过宽，例如 `/**` 或 `/api/**`。
>
> 正确做法：默认认证或拒绝，只对白名单路径明确放行。

> **易错：** 把 CORS 当成安全认证。
>
> 正确做法：CORS 只控制浏览器跨域访问，接口仍需要认证授权。
