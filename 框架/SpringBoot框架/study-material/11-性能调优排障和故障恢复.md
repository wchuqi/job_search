# Spring Boot 学习资料：性能调优、排障和故障恢复

[返回索引](../SpringBoot学习资料.md)

## 学习目标

- 建立从现象到证据再到根因的排障路径，而不是凭经验乱改配置。
- 能排查启动失败、自动配置不生效、Bean 冲突、配置覆盖、请求慢、连接池耗尽、线程阻塞、内存异常和依赖故障。
- 能设计生产故障恢复动作：回滚、扩容、降级、限流、熔断、关闭开关和数据修复。
- 能把日志、指标、链路追踪、线程 dump、堆 dump、数据库慢查询和自动配置报告串起来使用。

## 理论导读

性能调优和故障排查不是“参数大全”。真正有效的方法是先判断影响面，再收集证据，然后缩小范围。Spring Boot 提供了 Actuator、自动配置报告、健康检查、指标、日志和可观测性集成，但这些工具只回答特定问题：指标告诉你哪里异常，trace 告诉你慢在哪一段，日志告诉你发生了什么，线程栈告诉你线程卡在哪里，堆 dump 告诉你对象占用，conditions 告诉你自动配置为什么生效或不生效。

生产故障处理要先恢复服务，再追根因。恢复动作可以很务实：回滚版本、回滚配置、临时扩容、关闭功能开关、限制流量、熔断下游、切只读、清理异常数据。根因分析则要在服务恢复后完成，避免在高压线上环境里做无边界实验。

## 核心心智模型

排障像二分搜索：

```text
确认现象
  -> 判断影响面
  -> 找最近变更
  -> 看黄金指标
  -> 分层定位：应用 / JVM / 数据库 / 缓存 / 网络 / 下游
  -> 验证假设
  -> 执行恢复动作
  -> 固化根因和防复发措施
```

黄金指标：

- 流量：QPS、并发、队列长度。
- 延迟：平均、P95、P99。
- 错误：5xx、4xx、业务错误码。
- 饱和度：CPU、内存、GC、线程、连接池、磁盘、数据库连接。

## 知识点详解

### 1. 启动失败排查

启动失败优先读完整异常栈的最底层 cause，再结合 Boot 的 FailureAnalyzer 输出。

常见类型：

| 现象 | 可能原因 | 证据 |
| --- | --- | --- |
| 端口占用 | 端口已有进程 | `PortInUseException` |
| 配置绑定失败 | 类型不匹配、缺必填、校验失败 | `ConfigurationPropertiesBindException` |
| Bean 冲突 | 同名 Bean 或多候选 Bean | `NoUniqueBeanDefinitionException` |
| 循环依赖 | A 依赖 B，B 又依赖 A | dependency cycle |
| 数据库不可达 | URL、密码、网络、驱动错误 | datasource 初始化失败 |
| 依赖冲突 | 方法或类找不到 | `NoSuchMethodError`、`ClassNotFoundException` |
| 自动配置未匹配 | 类路径、属性、Bean 条件不满足 | conditions 报告 |

排查命令：

```powershell
java -jar app.jar --debug
```

`--debug` 会输出自动配置条件报告。不要把它长期用于生产高流量环境日志，但在复现和诊断时很有价值。

### 2. 自动配置不生效

诊断顺序：

1. starter 是否引入。
2. 类路径中条件类是否存在。
3. 当前应用类型是否正确。
4. 配置属性是否被正确加载。
5. 是否已有用户 Bean 导致默认 Bean 退让。
6. 自动配置是否被排除。
7. Bean 创建阶段是否失败。

Actuator 入口：

- `/actuator/conditions`
- `/actuator/beans`
- `/actuator/env`
- `/actuator/configprops`

### 3. 请求慢排查

请求慢要区分是“所有接口慢”还是“单接口慢”。

全局慢常见原因：

- CPU 打满。
- Full GC 或频繁 GC。
- 线程池耗尽。
- 数据库连接池耗尽。
- 下游依赖大面积慢。
- 日志、磁盘或网络异常。

单接口慢常见原因：

- 慢 SQL。
- N+1 查询。
- 锁等待。
- 第三方 API 慢。
- 请求体过大。
- 序列化大对象。
- 业务循环或算法低效。

排查流程：

```text
metrics 看 P95/P99
  -> trace 找最慢 span
  -> SQL/下游指标定位耗时段
  -> 线程 dump 看阻塞
  -> 连接池指标看等待
  -> 对照最近发布和配置变更
```

### 4. 连接池耗尽

连接池耗尽的表象通常是请求排队、超时、数据库连接获取失败。不要第一反应就把 `maximum-pool-size` 调大。

可能原因：

- SQL 执行慢，占用连接时间长。
- 事务范围过大，把远程调用放在事务内。
- 连接泄漏，没有正确关闭底层资源。
- 并发超过数据库承载能力。
- 应用实例数增加后，总连接数超过数据库上限。
- `REQUIRES_NEW` 嵌套事务额外消耗连接。

估算思路：

```text
数据库最大连接数 >= 应用实例数 * 每实例最大连接池 + 运维和其他服务连接
```

如果数据库最多 200 连接，10 个应用实例每个配 50，理论上就已经超了。

正确处理：

1. 看 Hikari 指标：active、idle、pending、timeout。
2. 看数据库慢查询和锁等待。
3. 缩小事务范围。
4. 给下游调用设置超时，避免事务内等待。
5. 必要时扩数据库能力，而不是只扩应用连接池。

### 5. 线程阻塞和线程池问题

线程 dump 可以回答“线程现在卡在哪里”。

```powershell
jcmd <pid> Thread.print
```

关注：

- 大量线程 `BLOCKED` 在同一锁。
- 大量线程 `WAITING` 等连接池。
- Web 线程等待下游 HTTP 调用。
- 定时任务线程长时间执行。
- ForkJoinPool 被阻塞任务占满。

常见修复：

- 给远程调用设置连接超时和读取超时。
- 给业务线程池设置队列、拒绝策略和隔离。
- 避免在公共线程池运行阻塞任务。
- 把慢任务异步化，但要配套限流和监控。

### 6. JVM 内存和 GC

内存问题不等于只调大堆。先看现象：

- OOM：堆、元空间、直接内存、线程数都有可能。
- GC 频繁：对象创建过多、堆太小、缓存无上限。
- 响应变慢：Stop-The-World、内存抖动。
- 容器被杀：超过 cgroup 内存限制。

命令：

```powershell
jcmd <pid> GC.heap_info
jcmd <pid> VM.native_memory summary
jcmd <pid> GC.class_histogram
```

堆 dump 用于离线分析，生产上抓 dump 要评估磁盘空间和暂停影响。

常见根因：

- 无上限本地缓存。
- 一次性读取大文件或大响应。
- 日志聚合对象过大。
- 线程池创建过多线程。
- 大量临时对象导致 GC 压力。

### 7. 日志、指标、Trace 如何配合

日志适合回答“发生了什么”，指标适合回答“是否异常和异常程度”，Trace 适合回答“一次请求慢在哪里”。

实践建议：

- 所有请求带 traceId/requestId。
- 错误日志包含业务主键，但不要包含密码、Token、身份证等敏感信息。
- 指标按接口、状态码、下游、数据库连接池维度拆分。
- 慢查询日志和应用 trace 能关联。
- 告警基于趋势和 SLO，不只看单点阈值。

### 8. 故障恢复优先级

线上故障处理顺序：

1. 止血：回滚、降级、限流、熔断、扩容、切流量。
2. 稳定：确认错误率、延迟、饱和度恢复。
3. 保全证据：日志、trace、dump、指标截图、变更记录。
4. 根因分析：复盘触发条件和缺失防线。
5. 防复发：测试、监控、限流、配置校验、发布策略。

恢复动作选择：

| 情况 | 优先动作 |
| --- | --- |
| 新版本引入错误 | 回滚版本 |
| 新配置引入错误 | 回滚配置 |
| 下游依赖慢 | 熔断、降级、缩短超时 |
| 流量突增 | 限流、扩容、排队 |
| 数据异常 | 暂停写入、修复数据、补偿 |
| 数据库慢 | 限制高成本接口、修 SQL、加索引需评估 |

### 9. Spring Boot 生产就绪检查

上线前至少确认：

- Actuator health/readiness/liveness 配置正确。
- 管理端点访问受控。
- 关键指标有 dashboard 和告警。
- 日志包含 traceId，敏感信息脱敏。
- 数据库连接池、HTTP 客户端、线程池都有超时和上限。
- 关键配置有校验，缺失时启动失败。
- 有回滚版本、回滚配置和数据库迁移策略。

## 例子：连接池耗尽故障时间线

现象：10:03 开始订单创建接口 P99 从 200ms 升到 8s，错误率升高。

证据：

- Hikari pending connections 增加。
- 数据库慢查询日志显示订单查询缺少索引。
- 最近发布新增按 `status + created_at` 查询。
- 线程 dump 显示大量 Web 线程等待数据库连接。

止血：

1. 关闭新查询入口或回滚版本。
2. 降低超时时间，避免请求堆积。
3. 临时限制该接口流量。

根因：

- 新查询未建联合索引。
- 发布前集成测试没有覆盖百万级数据量。
- 告警只监控错误率，没有监控连接池 pending。

防复发：

- 增加索引评审。
- 增加连接池指标告警。
- 增加压测数据集。

## 练习

1. 制造配置绑定失败，定位具体字段。
2. 引入两个同类型 Bean，观察注入失败并修复。
3. 模拟慢 SQL，观察接口 P99、Hikari pending、线程 dump。
4. 写一个没有超时的 HTTP 调用，再加上连接超时和读取超时。
5. 用 Actuator 查看 `metrics`、`threaddump`、`conditions`。

## 验收

- 能按证据链定位启动失败。
- 能解释连接池耗尽为什么不能只调大池子。
- 能从线程 dump 判断等待数据库、等待锁或等待下游。
- 能写出线上故障止血动作和复盘项。
- 能列出 Spring Boot 应用上线前生产就绪检查项。

## 重点

- 先恢复服务，再深入根因。
- 指标、日志、Trace、dump、conditions 各有边界，必须组合使用。
- 超时、限流、熔断、降级、回滚是生产系统基本能力。

## 难点

- 同一个现象可能来自应用、JVM、数据库、容器、网络或下游依赖。
- 临时扩容可能掩盖根因，也可能把压力转移给数据库。
- 抓 dump、开 debug、改日志级别都可能影响生产环境，需要评估范围。

## 易错

> **易错：** 看到连接池耗尽就盲目把最大连接数调大。
>
> 正确做法：先确认慢 SQL、事务范围、数据库容量、连接泄漏和实例总连接数。

> **易错：** 线上故障时一边改代码一边重启，缺少证据保全。
>
> 正确做法：先止血，保留日志、指标、dump、变更记录，再做根因分析。
