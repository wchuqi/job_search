# Vue3学习资料：组件渲染、Patch 算法、Key 和更新队列深度解析

[返回索引](../Vue3学习资料.md)

## 学习目标

- 理解组件挂载、更新、卸载的运行链路。
- 掌握 patch、key、最长递增子序列在列表 diff 中的作用。
- 能排查列表状态串位和渲染性能问题。

## 理论导读

Vue 渲染器的核心工作是把 vnode 同步到宿主环境。浏览器环境中宿主就是 DOM。首次渲染是 mount，后续渲染是 patch。patch 会根据 vnode 类型、key、props、children 判断复用、更新、移动或销毁。

## 核心心智模型

`key` 是组件实例和 DOM 节点的身份证。没有稳定身份证时，渲染器只能根据位置猜测身份，列表变化后就容易复用错对象。

## 列表 diff 简化过程

Vue3 对 keyed children 的处理可以粗略理解为：

1. 从头部同步相同节点。
2. 从尾部同步相同节点。
3. 处理新增或删除的简单区间。
4. 对中间乱序部分建立 key 到新索引的映射。
5. 判断旧节点是否可复用，不可复用则卸载。
6. 对需要移动的节点计算最长递增子序列，尽量少移动 DOM。

## 例子

```vue
<UserRow
  v-for="user in users"
  :key="user.id"
  :user="user"
/>
```

如果改成 `:key="index"`，排序后原本第一个位置的组件实例可能被复用到新的第一个用户上，导致输入框内部状态错配。

## 更新队列

同一个组件多次状态变更会进入同一个刷新队列。父子组件更新也会按一定顺序调度，避免重复无意义更新。`nextTick` 等待的是 Vue 当前刷新队列完成。

## 练习

构造列表 `[A, B, C, D]` 调整为 `[B, A, D, C]`，手动画出哪些节点可复用、哪些需要移动，并解释稳定 key 的作用。

## 验收

- 能解释 patch 不是简单全量重绘。
- 能说明最长递增子序列用于减少 DOM 移动。
- 能解释 `nextTick` 和更新队列的关系。

## 重点

- key 影响身份识别和状态保留。
- Vue3 编译优化能减少动态比较范围，但不能弥补错误状态设计。

## 难点

- patch flag、block tree、动态子节点收集需要结合编译产物理解。

## 易错

> **易错：** 看到页面显示没问题，就忽略 index key。
>
> 正确做法：只要列表会增删排序，并且子项有状态，就必须使用稳定业务 key。
