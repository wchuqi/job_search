## 使用方式

这份文件不是题库数量堆叠，而是训练深度回答。每题都按“直接答案、机制展开、工程边界、追问方向”准备。

## 1. Vue3 响应式为什么用 Proxy？和 Vue2 有什么差异？

**直接答案：**

Vue3 使用 Proxy 代理对象读写，可以拦截属性读取、设置、删除、`in`、枚举、数组和集合类型操作。Vue2 主要基于 `Object.defineProperty` 劫持已有属性，新增属性、删除属性、数组索引等场景需要额外处理。

**机制展开：**

Vue3 在 getter 中 track，在 setter/delete/collection mutation 中 trigger。依赖关系按 target、key、effect 存储。组件 render 是 effect，因此响应式状态变化会触发组件更新。

**工程边界：**

Proxy 不能代理已经解构出来的普通值。对大型外部对象深度代理可能带来性能和行为问题，需要 `markRaw`、`shallowRef`。

**追问：**

- 为什么解构 `reactive` 会丢响应式？
- `ref` 为什么脚本中需要 `.value`？
- `Map`、`Set` 响应式怎么触发？

## 2. `computed` 为什么有缓存？什么时候失效？

**直接答案：**

`computed` 内部有 lazy effect 和 dirty 标记。依赖未变化时直接返回缓存值；依赖变化时 scheduler 把 dirty 置为 true，下次读取才重新计算。

**机制展开：**

`computed` 读取时收集外层依赖，执行 getter 时收集自身依赖。依赖 trigger 后不会立刻重复计算，而是标记脏值并通知使用 computed 的 effect。

**工程边界：**

`computed` 应保持无副作用。异步请求、写 localStorage、修改其他状态不应放 computed。

**追问：**

- `computed` 和 `watch` 如何选择？
- computed 返回对象时会有什么问题？

## 3. Vue3 更新 DOM 的完整链路是什么？

**直接答案：**

状态写入触发 trigger，组件 render effect 进入 scheduler 队列；队列批量刷新时重新执行 render 生成新 vnode；patch 比较新旧 vnode；最后更新真实 DOM。`nextTick` 等待本轮刷新完成。

**机制展开：**

Vue3 会利用编译器生成的 patch flag 和 block tree 跳过静态节点，只处理动态部分。列表中 key 决定 vnode 身份，错误 key 会导致组件实例复用错位。

**工程边界：**

状态更新不是同步 DOM 更新。读取 DOM 尺寸、focus 新出现元素、滚动到新节点，都需要等 `nextTick`。

**追问：**

- patch flag 是什么？
- block tree 解决什么问题？
- index key 为什么危险？

## 4. 组合式 API 的优势和风险是什么？

**直接答案：**

优势是按逻辑能力组织代码，利于复用、测试和 TypeScript 推导。风险是 composable 滥用后形成隐式依赖、共享状态污染和难以追踪的副作用。

**机制展开：**

setup 执行时创建组件实例相关响应式状态，组件卸载时清理作用域内 effect。composable 如果在函数外创建状态，就可能变成跨组件共享。

**工程边界：**

composable 应表达稳定能力，例如请求、分页、权限、表单 dirty。不要把一个页面所有逻辑粗暴塞进 `usePage`。

**追问：**

- composable 如何清理副作用？
- 什么时候使用 `effectScope`？

## 5. Router 权限系统怎么设计？

**直接答案：**

登录态判断用户是否认证；权限判断用户可访问哪些页面；菜单是权限结果的 UI 投影；接口鉴权由后端保证。前端可用路由 meta、动态路由和全局守卫控制体验。

**机制展开：**

刷新后 Pinia 内存状态会丢，需要根据 token 重新拉用户和权限，再 addRoute。添加动态路由后要重新进入目标路由，否则 matcher 还没命中。

**工程边界：**

退出登录要清 token、用户信息、权限 store、动态路由和缓存页面。403、404、登录页要在白名单和匹配顺序上设计清楚。

**追问：**

- 动态路由刷新丢失怎么办？
- 如何避免无限重定向？
- 前端按钮权限安全吗？

## 6. Pinia 和组件本地状态怎么划分？

**直接答案：**

跨页面共享、有登录会话生命周期的状态放 Pinia；组件私有 UI 状态放本地；列表筛选适合 URL query；服务端列表和详情数据有缓存和过期语义，不应无脑放全局 store。

**机制展开：**

Pinia store 在同一 Pinia 实例下按 id 单例，订阅、插件、DevTools 都基于这个实例。SSR 中每个请求必须创建新的 Pinia，避免跨用户污染。

**追问：**

- `storeToRefs` 解决什么问题？
- Pinia 持久化有哪些安全风险？

## 7. Vue 项目卡顿怎么排查？

**直接答案：**

先复现并录 Performance，判断瓶颈是 JS、渲染、绘制、网络还是内存。再用 Vue DevTools 看组件更新时间，用构建分析看依赖体积，用 Memory 看泄漏。

**机制展开：**

如果是组件更新问题，用 `onRenderTriggered` 找触发字段；如果是大列表，减少 DOM、组件实例和深度响应式；如果是首屏，拆包和减少关键路径资源。

**追问：**

- 怎么定位内存泄漏？
- 虚拟列表有什么副作用？
- `v-memo` 适合什么场景？

## 8. SSR 水合不一致怎么处理？

**直接答案：**

找出服务端 HTML 和客户端首轮 render 不一致的组件，检查时间、随机数、localStorage、window、用户状态、异步数据是否一致。客户端专属逻辑放 mounted 后，或使用 client-only。

**机制展开：**

hydration 是在已有 DOM 上绑定事件和恢复组件树，不是重新创建一遍 DOM。结构不一致会导致 hydration mismatch，严重时会丢弃服务端 DOM 重建。

**追问：**

- SSR 为什么可能跨请求污染？
- Nuxt 适合什么项目？
