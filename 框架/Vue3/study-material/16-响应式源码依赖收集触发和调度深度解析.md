# Vue3学习资料：响应式源码依赖收集、触发和调度深度解析

[返回索引](../Vue3学习资料.md)

## 学习目标

- 从机制层理解 Proxy、effect、track、trigger。
- 能解释响应式失效和重复触发。
- 能把源码模型用于排障。

## 理论导读

Vue3 响应式并不是“数据绑定魔法”。它是运行时依赖图：当 effect 执行时，如果读取了响应式属性，这个 effect 就登记到该属性的依赖集合；当属性写入时，依赖集合里的 effect 被触发。

## 核心心智模型

可以把依赖关系想象成通信录：

```text
WeakMap(target)
  -> Map(key)
    -> Set(effect)
```

某个对象的某个字段变化，只通知订阅这个字段的人。

## 简化伪代码

```ts
const bucket = new WeakMap<object, Map<PropertyKey, Set<Function>>>()
let activeEffect: Function | null = null

function effect(fn: Function) {
  activeEffect = fn
  fn()
  activeEffect = null
}

function track(target: object, key: PropertyKey) {
  if (!activeEffect) return
  let depsMap = bucket.get(target)
  if (!depsMap) bucket.set(target, depsMap = new Map())
  let deps = depsMap.get(key)
  if (!deps) depsMap.set(key, deps = new Set())
  deps.add(activeEffect)
}

function trigger(target: object, key: PropertyKey) {
  const deps = bucket.get(target)?.get(key)
  deps?.forEach(fn => fn())
}
```

真实 Vue 会处理嵌套 effect、清理依赖、调度队列、数组、集合类型、只读、浅响应式等复杂情况。

## 调度机制

组件 effect 不会在每次状态写入时立即同步执行 DOM 更新，而是进入 job queue。同一轮同步代码中的多次状态修改会被合并刷新。

```ts
count.value++
count.value++
count.value++
// 通常只触发一轮组件更新
```

## 排障场景

| 现象 | 原因 | 修复 |
| --- | --- | --- |
| 解构后不更新 | 丢失 getter 代理 | 使用 `toRefs` |
| Map/Set 更新异常 | 没有通过响应式代理操作 | 保持使用 reactive 包装后的引用 |
| 第三方实例变慢 | 深度代理了复杂对象 | 使用 `markRaw` 或 `shallowRef` |
| watch 触发太多 | deep 监听大对象 | 监听明确字段 |

## 练习

手写一个最小响应式系统，支持 `reactive`、`effect`、`computed`，再加入微任务队列批处理。

## 验收

- 能解释为什么读取才会收集依赖。
- 能解释为什么 `computed` 有缓存。
- 能解释为什么多次赋值通常只导致一次 DOM 更新。

## 重点

- 响应式依赖图是理解 Vue3 的底层钥匙。

## 难点

- 嵌套 effect、依赖清理和调度优先级是源码阅读重点。

## 易错

> **易错：** 以为 Vue 会追踪普通局部变量。
>
> 正确做法：只有通过响应式 API 创建或包装的状态才参与依赖追踪。
