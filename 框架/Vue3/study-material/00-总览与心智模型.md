# Vue3学习资料：总览与心智模型

[返回索引](../Vue3学习资料.md)

## 学习目标

- 理解 Vue3 的定位：用于构建用户界面的渐进式框架。
- 建立“状态、模板、组件、运行时”的整体模型。
- 知道 Vue3 与 React、传统 DOM 操作、服务端模板的差异。

## 理论导读

Vue3 的核心是把页面看成状态的函数：状态改变，框架负责计算视图差异并更新 DOM。开发者不再把精力放在“找到 DOM，再改 DOM”，而是放在“状态如何建模，组件如何拆分，用户事件如何改变状态”。

Vue3 是渐进式的：可以只在一个页面局部增强，也可以配合 Vite、Vue Router、Pinia、测试工具、SSR 框架构建完整应用。组合式 API、响应式系统和编译优化是 Vue3 相比 Vue2 最重要的变化。

## 核心心智模型

把 Vue 应用想成一个工作台：

- 状态是原材料，例如用户列表、当前页码、加载状态。
- 模板是装配图，描述状态如何变成界面。
- 组件是工位，负责一块明确的 UI 和交互。
- 响应式系统是传送带，记录谁使用了状态，状态变化后通知对应组件。
- 渲染器是施工队，把新的虚拟节点结果同步到真实 DOM。

## 知识点详解

| 概念 | 解决的问题 | 常见边界 |
| --- | --- | --- |
| 声明式渲染 | 用数据描述界面，减少手动 DOM 操作 | 复杂第三方 DOM 库仍可能需要封装指令或组件 |
| 响应式系统 | 状态变更后自动触发依赖更新 | 解构、浅层响应式、非响应式外部对象容易造成失效 |
| 组件化 | 拆分 UI 和业务责任 | 组件过大难维护，组件过碎会增加通信成本 |
| 组合式 API | 按逻辑能力组织代码 | composable 过度隐式依赖会降低可测性 |
| 编译优化 | 模板编译时标记动态节点，减少运行时成本 | 手写 render 或过度动态模板会降低优化效果 |

## 例子

```vue
<script setup>
import { ref, computed } from 'vue'

const count = ref(0)
const doubleCount = computed(() => count.value * 2)
</script>

<template>
  <button @click="count++">count: {{ count }}</button>
  <p>double: {{ doubleCount }}</p>
</template>
```

这个例子里，`count` 是状态，`doubleCount` 是派生状态，模板声明了状态如何展示，点击事件只改变状态。

## 练习

实现一个计数器组件，包含加、减、重置、奇偶提示、最大值限制。

## 验收

- 页面不直接使用 `document.querySelector` 修改显示文本。
- 派生值使用 `computed`，不是每次事件里手动同步。
- 能说清楚 `ref` 为什么在脚本中需要 `.value`。

## 重点

- Vue3 默认推荐组合式 API 和 SFC。
- 状态驱动视图是理解所有高级能力的基础。

## 难点

- 模板更新不是立即同步 DOM，涉及调度和批处理。
- 响应式依赖是运行时收集的，不是简单字符串替换。

## 易错

> **易错：** 把所有页面数据放到全局 store。
>
> 正确做法：组件私有状态留在组件，跨页面共享状态才放 Pinia。
