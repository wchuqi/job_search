# Vue3学习资料：性能剖析、构建拆包、内存泄漏和大列表深度解析

[返回索引](../Vue3学习资料.md)

## 学习目标

- 建立 Vue 应用性能问题的分类和测量方法。
- 掌握首屏、交互、组件更新、大列表、内存泄漏、构建体积的定位路径。
- 能输出可复核的性能分析报告，而不是凭经验“优化一下”。

## 理论导读

性能问题不是一个问题，而是一组问题：资源下载慢、JS 执行慢、组件更新多、布局频繁抖动、图片过大、接口慢、内存泄漏、长列表 DOM 过多、第三方依赖体积过大。Vue 只是其中一层。有效优化的顺序是：测量、归类、定位、修复、对比。

## 性能分类

| 类型 | 指标 | 常见工具 |
| --- | --- | --- |
| 首屏性能 | FCP、LCP、TTFB、资源瀑布 | Lighthouse、WebPageTest、Network |
| 交互性能 | INP、长任务、输入延迟 | Performance |
| 渲染性能 | 组件更新时间、layout、paint | Vue DevTools、Performance |
| 构建体积 | chunk size、依赖占比 | rollup visualizer |
| 内存 | heap 增长、detached DOM | Memory |
| 接口 | latency、重复请求、失败率 | Network、APM |

## Performance 火焰图阅读方法

一次卡顿录制后按顺序看：

1. 有没有 Long Task。
2. 长任务里是 scripting、rendering 还是 painting。
3. scripting 中是否出现大量组件更新、JSON 处理、第三方库计算。
4. rendering 中是否有强制同步布局。
5. paint 中是否是大面积重绘或复杂样式。
6. 对照用户操作时间点，确认因果关系。

> **重点：** 火焰图要和用户动作对应，不要只看最大函数名。

## 大列表优化

大列表性能瓶颈来自 DOM 节点数量、响应式追踪数量、组件实例数量和布局绘制成本。优化顺序：

- 后端分页或前端分页。
- 虚拟列表，只渲染视口附近节点。
- 降低每行组件复杂度。
- 使用稳定 key。
- 对大型原始数据使用 `shallowRef`，只让可见切片进入深度响应式。
- 避免每行创建复杂 watch。

```ts
const rawRows = shallowRef<User[]>([])
const visibleRows = computed(() => {
  return rawRows.value.slice(start.value, end.value)
})
```

## 构建拆包

常见策略：

- 路由级懒加载。
- 管理后台按模块拆分。
- 大依赖按需引入。
- 对编辑器、图表、地图等低频模块异步加载。
- 使用构建分析工具查看真实体积。
- 避免重复引入不同版本依赖。

```ts
const RichTextEditor = defineAsyncComponent(() => import('@/components/RichTextEditor.vue'))
```

> **易错：** 为了拆包拆出大量小 chunk，反而增加请求和调度成本。拆包要结合缓存、HTTP/2、首屏路径和用户访问频率。

## 内存泄漏定位

Vue 项目常见泄漏源：

- 全局事件监听未移除。
- 定时器未清理。
- WebSocket 未关闭。
- 图表、地图、编辑器实例未 dispose。
- KeepAlive 缓存无上限。
- 组件外 watch 未 stop。
- 闭包保存大对象引用。

排查步骤：

1. 打开页面，拍 heap snapshot。
2. 进入/离开页面多次。
3. 手动 GC 后再拍 snapshot。
4. 对比对象数量是否持续增长。
5. 查 detached DOM、组件实例、第三方实例引用链。

## Vue 更新性能定位

如果怀疑 Vue 组件重复更新：

- Vue DevTools 查看组件更新时间。
- 使用 `onRenderTriggered` 找触发字段。
- 检查是否把大对象作为 props 传给很多子组件。
- 检查 Pinia 状态是否过大，任一字段变化导致大范围依赖更新。
- 检查模板中复杂表达式是否每次 render 重算。

## 练习

构建一个包含 10000 行表格、图表、搜索和编辑弹窗的页面，分别完成：

- 首屏拆包。
- 虚拟列表。
- 搜索防抖和请求竞态保护。
- 图表实例卸载清理。
- 输出性能对比报告。

## 验收

- 优化前后有可复核指标。
- 大列表滚动不卡顿。
- 多次进入离开页面，内存不持续增长。
- 首屏 chunk 体积下降。

## 易错

> **易错：** 把“感觉快了”当作性能优化完成。
>
> 正确做法：每次优化都要有指标、证据、风险和回滚方案。
