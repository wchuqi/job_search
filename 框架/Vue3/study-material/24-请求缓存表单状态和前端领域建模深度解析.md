# Vue3学习资料：请求缓存、表单状态和前端领域建模深度解析

[返回索引](../Vue3学习资料.md)

## 学习目标

- 理解请求不是简单 `axios.get`，而是状态机和缓存一致性问题。
- 掌握列表、详情、表单草稿、乐观更新、错误恢复的建模方式。
- 能区分 UI 状态、领域状态、服务端缓存和持久化状态。

## 理论导读

很多 Vue 项目后期混乱，不是组件写法问题，而是状态没有分类。接口返回的数据、用户正在编辑的草稿、页面筛选条件、全局用户信息、按钮 loading、缓存快照，都被混成一个对象。结果就是刷新丢状态、编辑污染列表、重复请求、旧请求覆盖新请求、保存失败无法回滚。

## 状态分类

| 类型 | 例子 | 生命周期 | 推荐位置 |
| --- | --- | --- | --- |
| UI 状态 | 弹窗开关、tab、展开行 | 组件内短期 | 组件本地 |
| URL 状态 | page、keyword、sort | 可分享可恢复 | route query |
| 领域草稿 | 用户编辑表单 | 编辑会话期间 | 组件/composable |
| 应用状态 | 当前用户、权限 | 登录会话期间 | Pinia |
| 服务端缓存 | 用户列表、详情 | 有过期语义 | 请求缓存层 |
| 持久化状态 | 主题、remember me | 跨刷新 | Pinia 插件或存储 |

## 请求状态机

一个健壮请求至少有这些状态：

```text
idle -> loading -> success
              -> error
success -> refreshing -> success/error
```

区别：

- `loading`：首次没有数据。
- `refreshing`：已有旧数据，正在刷新。
- `error`：首次失败和刷新失败的 UI 应不同。
- `stale`：数据可展示但可能过期。

## 请求竞态

搜索输入是典型竞态场景：用户先搜索 `a`，再搜索 `alice`，如果 `a` 的请求后返回，可能覆盖 `alice` 结果。

```ts
let requestId = 0

async function loadUsers(query: Query) {
  const current = ++requestId
  loading.value = true
  try {
    const result = await fetchUsers(query)
    if (current !== requestId) return
    users.value = result.items
  } finally {
    if (current === requestId) loading.value = false
  }
}
```

也可以用 `AbortController` 取消旧请求。

## 缓存键设计

缓存键要包含决定结果的所有输入：

```ts
const key = computed(() => [
  'users',
  route.query.page ?? '1',
  route.query.keyword ?? '',
  route.query.role ?? 'all'
].join(':'))
```

缺少任何输入都会造成缓存串数据。包含不稳定输入又会造成缓存命中率低。

## 表单草稿和原始数据

错误示例：

```ts
const user = userStore.currentUser
// 表单直接修改 user.name，列表、详情、缓存都被污染
```

推荐：

```ts
const source = ref<User | null>(null)
const draft = reactive<UserForm>({ name: '', email: '', roleIds: [] })

function resetDraft(user: User) {
  source.value = user
  draft.name = user.name
  draft.email = user.email
  draft.roleIds = [...user.roleIds]
}
```

保存成功后再更新原始数据或使缓存失效。

## 乐观更新和回滚

适合点赞、收藏、开关状态等低风险操作：

```ts
async function toggleEnabled(user: User) {
  const previous = user.enabled
  user.enabled = !previous
  try {
    await updateEnabled(user.id, user.enabled)
  } catch (error) {
    user.enabled = previous
    throw error
  }
}
```

不适合金额、库存、权限等高风险数据，除非后端有幂等和冲突处理。

## 前端领域建模

领域模型不是后端实体原样搬到前端。前端要围绕交互建模：

- 列表项可能只需要摘要字段。
- 详情页需要完整字段。
- 表单需要草稿和校验错误。
- 权限需要可见、可编辑、可提交三种状态。
- 批量操作需要 selected、pending、failed 映射。

## 练习

实现一个用户管理模块：

- 列表 query 同步到 URL。
- 用户详情有缓存和刷新。
- 编辑表单使用草稿。
- 保存成功使列表和详情缓存失效。
- 搜索请求有竞态保护。

## 验收

- 快速输入搜索不会被旧请求覆盖。
- 编辑未保存不会污染列表。
- 刷新 URL 能恢复列表条件。
- 保存失败能保留草稿并显示字段错误。

## 重点

- 先分类状态，再选择 Vue、Router、Pinia 或请求缓存工具。

## 易错

> **易错：** 认为接口返回什么，前端就应该用什么结构直接渲染和编辑。
>
> 正确做法：根据交互场景建立视图模型、表单模型和缓存模型。
