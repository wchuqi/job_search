# MyBatis 学习资料：插件、拦截器和扩展点

[返回索引](../Mybatis学习资料.md)

## 学习目标

- 理解 MyBatis 插件能拦截哪些对象和方法。
- 掌握 Interceptor、Invocation、Plugin.wrap 的基本机制。
- 能解释分页插件、SQL 日志插件、审计插件的原理。
- 能避免插件顺序、SQL 改写和参数映射不一致导致的问题。

## 理论导读

MyBatis 插件是基于动态代理的拦截机制。它不是任意位置都能插入，而是只能拦截 MyBatis 规定的四类核心对象：Executor、StatementHandler、ParameterHandler、ResultSetHandler。分页插件通常拦截 StatementHandler 或 Executor，在 SQL 执行前改写 SQL 或执行 count。

插件能力强，也很危险。错误 SQL 改写、插件顺序冲突、参数映射不一致，都会造成生产事故。

## 可拦截对象

| 对象 | 常见用途 |
| --- | --- |
| Executor | 查询/更新入口、缓存、分页 count |
| StatementHandler | SQL 改写、分页、SQL 日志 |
| ParameterHandler | 参数处理、加密脱敏 |
| ResultSetHandler | 结果处理、字段脱敏 |

## 插件示例：慢 SQL 日志

```java
@Intercepts({
    @Signature(type = StatementHandler.class, method = "query", args = {Statement.class, ResultHandler.class})
})
public class SlowSqlInterceptor implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            return invocation.proceed();
        } finally {
            long cost = System.currentTimeMillis() - start;
            if (cost > 500) {
                System.out.println("Slow SQL cost=" + cost + "ms");
            }
        }
    }
}
```

真实项目还需要输出 statementId、BoundSql、参数、traceId，并注意不要泄漏敏感信息。

## 插件代理链机制

MyBatis 插件会对目标对象层层包装：

```text
Executor 原始对象
  -> 插件 A 代理
  -> 插件 B 代理
  -> 插件 C 代理
```

调用时顺序与包装顺序相关。多个插件都改写 SQL 时，最终 SQL 取决于插件链顺序。常见冲突：

- 分页插件先改 SQL，数据权限插件后加条件，count SQL 不准确。
- 多租户插件和逻辑删除插件都改 where，导致条件顺序或括号错误。
- SQL 日志插件记录的是改写前 SQL，不是最终执行 SQL。

> **难点：** 插件不是并列执行，而是代理链。排查插件问题要确认每个插件看到的是哪一版 SQL。

## 分页插件原理

典型分页插件做两件事：

1. 根据原 SQL 生成 count SQL。
2. 给原 SQL 添加 limit/offset 或数据库方言分页语法。

风险：

- 复杂 SQL 自动生成 count 不准确。
- order by、group by、distinct 会影响 count。
- SQL 改写后参数映射不一致。
- 多个插件顺序冲突。

## 分页插件的 count 陷阱

自动 count 对简单 SQL 有效：

```sql
select * from orders where status = ?
```

但复杂 SQL 风险很高：

```sql
select distinct u.id
from users u
left join orders o on u.id = o.user_id
group by u.id
having count(o.id) > 3
```

简单包一层 count 可能正确但性能差；移除 order by 可能正确；错误改写 group by/distinct 可能直接算错。生产中对复杂分页查询应手写 count SQL 或单独优化。

## 数据权限插件风险

数据权限插件常在 SQL 后追加：

```sql
and tenant_id = ?
```

风险：

- 原 SQL 没有 where。
- 原 SQL 有 union。
- 子查询中也需要权限条件。
- 参数映射要新增 tenantId。
- 管理员绕过逻辑要明确。

复杂权限 SQL 不建议完全靠字符串拼接，应有清晰的 SQL 解析或业务层约束。

## TypeHandler

TypeHandler 负责类型转换。适合：

- 枚举 code 映射。
- JSON 字段映射。
- 加密字段。
- 数据库特殊类型。

示例：

```java
public class StatusTypeHandler extends BaseTypeHandler<OrderStatus> {
    @Override
    public void setNonNullParameter(PreparedStatement ps, int i, OrderStatus parameter, JdbcType jdbcType)
            throws SQLException {
        ps.setString(i, parameter.getCode());
    }

    @Override
    public OrderStatus getNullableResult(ResultSet rs, String columnName) throws SQLException {
        return OrderStatus.fromCode(rs.getString(columnName));
    }
}
```

## 验收

- 能列出四类可拦截对象。
- 能解释分页插件基本原理。
- 能写简单 Interceptor。
- 能说明插件 SQL 改写的风险。

## 重点

- 插件是 MyBatis 核心对象的动态代理。
- 分页插件不是魔法，本质是 SQL 改写和 count。
- TypeHandler 是类型转换关键扩展点。

## 难点

- 插件顺序和 SQL 改写可能互相影响。
- BoundSql 和 ParameterMapping 不一致会导致参数绑定错误。

## 易错

> **易错：** 在插件中简单字符串拼接 SQL，不考虑参数映射和数据库方言。
>
> 正确做法：理解 BoundSql、ParameterMapping 和方言差异，复杂 SQL 谨慎改写。
