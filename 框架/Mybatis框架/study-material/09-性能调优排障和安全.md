# MyBatis 学习资料：性能调优、排障和安全

[返回索引](../Mybatis学习资料.md)

## 学习目标

- 能排查慢 SQL、N+1、分页慢、批量写慢、映射错误和参数绑定错误。
- 掌握 SQL 日志、BoundSql、执行计划、索引、批处理和流式查询。
- 理解 MyBatis 中 SQL 注入、`${}`、动态排序、模糊查询和日志泄密风险。
- 能制定 MyBatis 生产规范。

## 排障心智模型

```text
接口慢
  -> 看 SQL 次数
  -> 看单条 SQL 耗时
  -> 看执行计划和索引
  -> 看 ResultMap 是否 N+1
  -> 看分页和排序
  -> 看数据库锁和连接池
```

## 慢 SQL

处理步骤：

1. 打开 SQL 日志或慢 SQL 插件。
2. 记录 statementId、SQL、参数、耗时、traceId。
3. 使用数据库 `EXPLAIN`。
4. 检查索引、扫描行数、排序、临时表。
5. 改写 SQL 或增加索引。
6. 加压测验证。

## 慢 SQL 证据链

生产排障不要只记录“SQL 慢”，要记录足够定位的信息：

```text
traceId
statementId
BoundSql
参数摘要
耗时
返回行数
数据库实例
连接池等待时间
执行计划 hash 或 explain 结果
```

statementId 很关键。只看到 SQL 文本时，很难定位是哪一个 Mapper 方法触发；同一 SQL 片段可能来自多个接口。statementId 能直接定位到 XML 和方法。

## N+1

典型 N+1：

```text
select orders limit 100
for each order:
  select items where order_id = ?
```

解决：

- join + collection resultMap。
- 两次批量查询后 Java 组装。
- 限制分页大小。
- 禁止列表接口懒加载明细。

## N+1 证据链

N+1 不一定表现为单条慢 SQL，而是 SQL 次数异常多。判断方式：

```text
一次 HTTP 请求
  -> orderMapper.selectPage 执行 1 次
  -> orderItemMapper.selectByOrderId 执行 100 次
  -> 每条 5ms，总计 500ms
```

排查时按 traceId 聚合 statementId 次数。如果同一个 statementId 在一次请求内出现几十次，就要检查嵌套查询、循环调用 Mapper 或延迟加载序列化。

## 分页优化

大 offset 分页会越来越慢：

```sql
select * from orders order by id limit 20 offset 1000000
```

优化：

- 使用基于游标/seek 的分页。
- 用覆盖索引。
- 先查 id 再回表。
- 限制最大页数。

## 索引和动态 SQL 的关系

动态 SQL 会改变 where 条件组合。索引设计不能只看某一个固定 SQL，要看高频条件组合。

例如订单查询：

```sql
where user_id = ?
  and status = ?
  and created_at >= ?
order by created_at desc
```

可能需要 `(user_id, status, created_at)` 这样的联合索引。但如果动态条件允许只按 `status` 查全表，索引策略就不同。生产规范应限制查询条件，避免“万能搜索接口”拖垮数据库。

## 批量写

批量写方式：

- 单条循环 insert：简单但慢。
- foreach 拼多 values：SQL 过长风险。
- BatchExecutor：性能好但错误定位复杂。
- 数据库导入工具：大批量离线导入。

## 批量和事务的性能边界

批量写太小没有收益，太大又可能造成：

- SQL 过长。
- 单事务锁持有时间过长。
- binlog/redo 压力过大。
- 出错后回滚成本高。
- 应用内存占用高。

推荐按批次提交或 flush，并记录失败批次。具体批大小要通过压测确定。

## 安全风险

### SQL 注入

危险：

```xml
where name like '%${keyword}%'
```

安全：

```xml
where name like concat('%', #{keyword}, '%')
```

### 动态排序

排序列必须白名单：

```java
String column = allowedColumns.getOrDefault(request.sort(), "id");
```

### 日志泄密

SQL 日志可能包含手机号、身份证、Token、密码。生产日志应脱敏，并控制 SQL 参数输出级别。

## SQL 注入审计清单

必须审计：

- 所有 `${}`。
- `order by ${}`。
- `${tableName}`。
- `${column}`。
- like 拼接。
- Provider 拼接字符串。
- Wrapper 或工具类中的 raw SQL。

每一处 `${}` 都要回答：来源是否可信，是否白名单，是否有测试覆盖。

## 常见错误

| 错误 | 排查方向 |
| --- | --- |
| `Invalid bound statement` | namespace/id/XML 加载 |
| `Parameter not found` | `@Param`、参数名、OGNL |
| 结果字段 null | 列名、别名、驼峰映射、resultMap |
| TooManyResults | 方法返回单对象但 SQL 多行 |
| SQL 语法错误 | 动态 SQL 空 where、foreach 空集合 |

## 练习

1. 制造 N+1，统计 SQL 次数。
2. 用 explain 优化订单分页查询。
3. 写安全排序白名单。
4. 写慢 SQL 拦截器。
5. 检查日志是否输出敏感参数。

## 验收

- 能从 statementId 定位慢 SQL。
- 能解释 N+1 并给两种修复方式。
- 能处理大 offset 分页。
- 能识别 `${}` 注入风险。
- 能写 MyBatis 生产 SQL 规范。
- 能建立 statementId 级别的慢 SQL 证据链。
- 能按一次请求聚合 SQL 次数发现 N+1。

## 重点

- MyBatis 性能核心仍是 SQL 和索引。
- 动态 SQL 必须安全拼接。
- 排障要看最终 BoundSql 和执行计划。

## 难点

- 复杂 ResultMap 可能让 SQL 次数和对象数量不直观。
- 分页插件不能替代数据库分页优化。

## 易错

> **易错：** 只看 Java 代码，不看实际 SQL 和执行计划。
>
> 正确做法：以 BoundSql、参数和数据库执行计划为准。
