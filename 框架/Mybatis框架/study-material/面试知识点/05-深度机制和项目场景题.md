## 一、执行链路深度题

### 1. MyBatis 查询时 CacheKey 由什么决定？

**参考答案：**

CacheKey 通常由 statementId、RowBounds、最终 SQL、参数值和 Environment 等因素组成。它不是只看 SQL 字符串，也不是只看 Mapper 方法名。动态 SQL 生成的最终 SQL 不同，或者参数不同，CacheKey 都不同。

> **重点：** 一级缓存命中依赖 CacheKey，排查缓存问题要看最终 BoundSql 和参数。

### 2. MyBatis 更新后为什么要清一级缓存？

**参考答案：**

如果同一个 SqlSession 内先查询一条数据，再执行 update，然后再次查询，如果不清缓存就会读到旧对象。insert/update/delete 会清理当前 SqlSession 的本地缓存，以保证后续查询更接近数据库当前状态。

### 3. 插件改写 SQL 为什么容易出问题？

**参考答案：**

因为 BoundSql 不只有 SQL 文本，还有 ParameterMapping。插件如果只改 SQL，不同步处理参数映射，就可能导致 `?` 数量和参数数量不一致。分页、数据权限、多租户插件都可能踩这个坑。

## 二、参数绑定深度题

### 4. 多参数 Mapper 方法为什么必须重视 `@Param`？

**参考答案：**

MyBatis 会用 ParamNameResolver 解析方法参数。多个参数如果没有 `@Param`，XML 中稳定可用的往往是 `param1`、`param2` 这类名字，依赖编译参数名不可靠。`@Param` 能让 XML 中的 OGNL 表达式明确引用参数。

### 5. `#{}` 的参数最终是怎么绑定到 SQL 的？

**参考答案：**

动态 SQL 解析后生成 BoundSql，其中 SQL 文本里是 `?`，ParameterMapping 记录每个 `?` 对应的属性名和类型。ParameterHandler 按顺序取出参数值，再通过 TypeHandler 调用 PreparedStatement 的 set 方法完成绑定。

> **重点：** `#{}` 最终不是字符串拼接，而是 PreparedStatement 参数绑定。

## 三、ResultMap 深度题

### 6. 一对多 join 结果如何变成 Java 对象集合？

**参考答案：**

join 结果是多行扁平数据。ResultSetHandler 根据 ResultMap 中主对象 `<id>` 判断哪些行属于同一个父对象，根据 collection 中子对象 `<id>` 判断哪些子对象需要追加。如果 `<id>` 配错或列别名冲突，就会出现重复对象、集合重复或数据合并错误。

### 7. 为什么一对多 join 不能直接随便分页？

**参考答案：**

因为 limit 限制的是 join 后的行数，不是父对象数量。一个订单有 10 个明细时，`limit 20` 可能只返回两个订单。正确做法是先分页查主表 id，再按 id 查明细，或用子查询/窗口函数保证先分页主表。

## 四、缓存和 Spring 集成深度题

### 8. Spring 中一级缓存什么时候明显生效？

**参考答案：**

Spring 集成下，SqlSession 通常与事务绑定。同一个 `@Transactional` 方法内多次调用 Mapper，可能复用同一个 SqlSession，因此一级缓存可能生效。没有事务时，每次 Mapper 调用可能使用不同 SqlSession，一级缓存效果不明显。

### 9. SqlSessionTemplate 为什么线程安全？

**参考答案：**

SqlSessionTemplate 本身是线程安全代理，它不会把同一个原始 SqlSession 暴露给多个线程共享，而是根据当前 Spring 事务上下文获取或创建合适的 SqlSession，并在事务同步中提交、回滚和释放资源。

### 10. 二级缓存为什么不适合多表 join 查询？

**参考答案：**

二级缓存按 namespace 管理。OrderMapper 的 join 查询可能缓存了 users 表字段，但 UserMapper 更新用户后不会自动清理 OrderMapper 的缓存。这样会产生跨 namespace 脏数据。除非能明确失效策略，否则不建议缓存多表 join。

## 五、插件和分页深度题

### 11. MyBatis 插件是如何形成代理链的？

**参考答案：**

MyBatis 会对 Executor、StatementHandler、ParameterHandler、ResultSetHandler 这些目标对象按插件顺序层层包装。调用时经过代理链。多个插件都改 SQL 时，最终 SQL 取决于代理链顺序。排查时要确认每个插件看到的是改写前还是改写后的 SQL。

### 12. 分页插件自动 count 有什么坑？

**参考答案：**

简单 SQL 可以自动生成 count，但复杂 SQL 如 distinct、group by、having、union、多表 join，自动 count 可能性能差或结果不准。生产复杂分页应考虑手写 count SQL 或专门优化。

## 六、生产排障场景题

### 13. 一个接口 P99 升高，你如何判断是不是 MyBatis 导致？

**参考答案：**

先按 traceId 聚合 SQL 次数和耗时，记录 statementId、BoundSql、参数摘要、返回行数和连接池等待时间。如果单条 SQL 慢，看 EXPLAIN、索引、锁等待；如果 SQL 次数多，看 N+1 或循环 Mapper；如果数据库等待高，看连接池和事务范围；如果 ResultMap 映射耗时高，看一对多 join 行膨胀。

> **重点：** MyBatis 排障要定位到 statementId，而不是只说数据库慢。

### 14. 如何审计 MyBatis SQL 注入风险？

**参考答案：**

扫描所有 `${}`、Provider 字符串拼接、order by 动态字段、表名列名拼接、like 拼接和工具类 raw SQL。普通值必须用 `#{}`。必须拼接标识符时，用服务端白名单映射，并写测试覆盖非法输入。

### 15. 批量插入怎么选方案？

**参考答案：**

小批量可以 foreach 多 values；中等批量可以 BatchExecutor 分批 flush；超大批量考虑数据库导入工具。要结合事务大小、SQL 长度、内存、binlog/redo 压力和错误定位。批量越大不一定越好，需要压测确定批大小。
