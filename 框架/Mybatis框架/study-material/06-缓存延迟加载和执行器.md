# MyBatis 学习资料：缓存、延迟加载和执行器

[返回索引](../Mybatis学习资料.md)

## 学习目标

- 理解一级缓存、二级缓存、localCacheScope 和缓存失效规则。
- 掌握 SimpleExecutor、ReuseExecutor、BatchExecutor 的差异。
- 理解延迟加载的适用场景和风险。
- 能排查缓存脏读、批处理未生效、查询重复和延迟加载触发时机。

## 理论导读

MyBatis 缓存不是通用缓存系统。一级缓存绑定 SqlSession，默认存在；二级缓存绑定 namespace，需要显式配置且风险较高。缓存能减少重复查询，但也可能带来脏数据、事务边界混乱和难以排查的问题。

执行器决定 SQL 执行策略。SimpleExecutor 简单可靠；ReuseExecutor 复用 Statement；BatchExecutor 适合批量写，但需要理解 flush 时机和返回结果。

## 一级缓存

一级缓存是 SqlSession 级别缓存。同一个 SqlSession 内相同 statement 和参数，可能命中缓存。

```text
SqlSession A
  select user id=1 -> 查数据库
  select user id=1 -> 命中一级缓存
```

更新、提交、回滚、关闭会清理缓存。Spring 集成下，SqlSession 生命周期通常和事务绑定，不能按裸 MyBatis 的长会话理解。

`localCacheScope=STATEMENT` 可缩小一级缓存范围，避免复杂查询中的缓存干扰。

## 一级缓存深入：为什么会影响嵌套查询

一级缓存不仅用于普通查询，也会参与嵌套查询和循环引用处理。MyBatis 在处理复杂 ResultMap 时，需要避免同一对象重复创建，也要防止嵌套查询互相引用造成无限递归。

这也是 `localCacheScope` 存在的原因：

| 值 | 含义 |
| --- | --- |
| SESSION | 同一 SqlSession 内缓存查询结果 |
| STATEMENT | 每条语句执行后清理本地缓存 |

如果你遇到同一事务内重复查询返回旧对象，或复杂嵌套查询缓存干扰，可以评估 `STATEMENT`，但不要把它当成性能优化默认项。

## 二级缓存

二级缓存是 namespace 级别，需要配置：

```xml
<cache/>
```

风险：

- 多表 join 查询很难判断哪个 namespace 更新后应清理缓存。
- 分布式多实例下本地二级缓存不一致。
- 缓存对象序列化和可变对象风险。

生产中如果需要缓存，通常更推荐业务层使用 Redis/Caffeine，并明确 key、TTL、失效策略。

## 二级缓存深入：namespace 边界

二级缓存按 namespace 管理。假设：

```text
OrderMapper.selectOrderWithUser -> join orders + users
UserMapper.updateUserName       -> 更新 users
```

如果 `OrderMapper` 开了二级缓存，`UserMapper.updateUserName` 默认不会清理 `OrderMapper` 的缓存。这样订单详情中的用户名可能是旧的。

解决方式：

- 不对多表 join 开二级缓存。
- 使用 `cache-ref` 共享缓存，但会增加耦合。
- 在业务缓存层统一管理失效。
- 对高一致性数据不要使用 MyBatis 二级缓存。

## 延迟加载

延迟加载让关联对象在访问时再查询。它可以减少不必要查询，但风险是：

- 访问属性时突然发 SQL。
- 序列化对象时触发大量查询。
- Session 已关闭时访问失败。
- 难以控制查询次数。

MyBatis 的延迟加载不应作为默认优化手段。复杂接口更推荐明确 SQL 和查询边界。

## 执行器

| Executor | 适合 | 注意 |
| --- | --- | --- |
| SimpleExecutor | 默认场景 | 每次创建 Statement |
| ReuseExecutor | 重复 SQL | Statement 复用收益有限 |
| BatchExecutor | 批量 insert/update | flush 时机、事务和错误定位复杂 |

批处理示意：

```java
try (SqlSession session = sqlSessionFactory.openSession(ExecutorType.BATCH)) {
    UserMapper mapper = session.getMapper(UserMapper.class);
    for (User user : users) {
        mapper.insert(user);
    }
    session.flushStatements();
    session.commit();
}
```

Spring 中更常通过专门配置的 `SqlSessionTemplate` 使用 batch。

## BatchExecutor 深入

BatchExecutor 并不是“把所有 SQL 合成一条”。它会把相同或相近的 PreparedStatement 加入批处理，等 flush 时统一发送。

注意点：

- insert 后立刻依赖自增主键，可能需要 flush。
- 批处理中的某条失败时，错误定位比单条执行难。
- 大批量要分批 flush，避免内存和数据库压力。
- 与 Spring 事务结合时，flush 不等于 commit。

推荐批量写策略：

```text
每 500 或 1000 条 flush 一次
  -> 捕获批次错误
  -> 事务边界按业务决定
  -> 记录失败数据
```

## 练习

1. 同一事务内查询两次相同 SQL，观察一级缓存。
2. 更新后再查，观察缓存清理。
3. 开启二级缓存，模拟多表 join 缓存脏数据。
4. 用 BatchExecutor 批量插入 1000 条数据。
5. 开启延迟加载并观察触发 SQL 的时机。

## 验收

- 能解释一级缓存作用域。
- 能说明二级缓存生产风险。
- 能区分三种 Executor。
- 能判断延迟加载是否适合某接口。

## 重点

- 一级缓存默认存在，作用域是 SqlSession。
- 二级缓存不是分布式缓存，生产使用要谨慎。
- BatchExecutor 要理解 flush 和事务边界。

## 难点

- Spring 事务管理会改变你对 SqlSession 生命周期的直觉。
- 缓存命中和清理依赖 statement、参数、事务和更新操作。

## 易错

> **易错：** 开启二级缓存解决所有性能问题。
>
> 正确做法：先优化 SQL、索引和查询次数；缓存要有明确一致性策略。
