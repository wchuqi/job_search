# MyBatis 学习资料：总览与心智模型

[返回索引](../Mybatis学习资料.md)

## 学习目标

- 理解 MyBatis 的定位：SQL Mapper，而不是完整 ORM。
- 建立从 Mapper 方法到 JDBC 执行的运行时心智模型。
- 区分 MyBatis、JDBC、JPA/Hibernate、MyBatis-Plus、MyBatis Dynamic SQL 的边界。
- 明确学习重点：SQL、参数绑定、结果映射、缓存、插件、事务和性能。

## 理论导读

MyBatis 解决的是 Java 对象和 SQL 之间的映射问题。JDBC 要手写连接、PreparedStatement、参数设置、ResultSet 遍历和对象封装；MyBatis 把这些重复工作标准化，同时保留 SQL 的可控性。你写 SQL 和映射规则，MyBatis 负责执行、绑定参数、映射结果、管理缓存和提供扩展点。

MyBatis 不是 Hibernate 那样的完整 ORM。它不会自动追踪对象状态，不会自动脏检查，不会自动根据对象关系生成复杂 SQL。它适合团队重视 SQL 可控、数据库调优和明确查询行为的场景。

## 核心心智模型

```text
Mapper 接口方法
  -> MapperProxy 动态代理
  -> 找到 MappedStatement
  -> 解析动态 SQL 得到 BoundSql
  -> Executor 执行
  -> StatementHandler 创建 PreparedStatement
  -> ParameterHandler 绑定参数
  -> JDBC 查询数据库
  -> ResultSetHandler 映射结果
  -> 返回 Java 对象
```

把 MyBatis 想成“SQL 执行和映射引擎”：Mapper 方法只是入口，真正的核心是 `MappedStatement + BoundSql + Executor + ResultSetHandler`。

## 知识点详解

### MyBatis 和 JDBC

| 维度 | JDBC | MyBatis |
| --- | --- | --- |
| SQL | 手写 | 手写 |
| 参数绑定 | 手写 setXxx | 自动绑定 |
| 结果映射 | 手写 ResultSet | ResultMap/自动映射 |
| 事务 | 手动或 Spring | 通常交给 Spring |
| 扩展 | 自己封装 | 插件、TypeHandler、ObjectFactory |

MyBatis 没有消灭 SQL，它让 SQL 的执行和映射更工程化。

### MyBatis 和 JPA/Hibernate

| 维度 | MyBatis | JPA/Hibernate |
| --- | --- | --- |
| 核心模型 | SQL Mapper | ORM |
| SQL 控制 | 强 | 可控但常由框架生成 |
| 对象状态管理 | 不做脏检查 | 持久化上下文和脏检查 |
| 复杂查询 | SQL 直接表达 | JPQL/Criteria/原生 SQL |
| 学习难点 | SQL、映射、动态 SQL | 对象状态、懒加载、缓存 |

如果系统查询复杂、报表多、SQL 调优频繁，MyBatis 更直观。如果领域对象关系复杂，希望用对象模型驱动，JPA 可能更合适。

### MyBatis 和 MyBatis-Plus

MyBatis-Plus 是 MyBatis 的增强工具，提供通用 CRUD、Wrapper 条件构造、分页插件等能力。它不是 MyBatis 官方核心的一部分。学习顺序应先掌握 MyBatis，再使用 MyBatis-Plus，否则容易只会写 Wrapper，不懂执行链路和 SQL 风险。

## 例子

Mapper 接口：

```java
public interface UserMapper {
    User selectById(Long id);
}
```

XML：

```xml
<mapper namespace="com.example.UserMapper">
    <select id="selectById" parameterType="long" resultType="com.example.User">
        select id, username, email
        from users
        where id = #{id}
    </select>
</mapper>
```

调用：

```java
User user = userMapper.selectById(1L);
```

这一次调用不是直接执行接口方法，而是由动态代理找到 XML 中 namespace 和 id 对应的 SQL。

## 练习

1. 写一个 `UserMapper.selectById`。
2. 打开 SQL 日志，观察最终执行 SQL。
3. 把 `#{id}` 改成 `${id}`，理解 SQL 注入风险。
4. 把数据库列名改成 `user_name`，学习字段映射。

## 验收

- 能解释 MyBatis 不是完整 ORM。
- 能画出 Mapper 方法到 JDBC 的执行链路。
- 能说出 `MappedStatement` 和 `BoundSql` 的作用。
- 能说明为什么 SQL 可控既是优势也是责任。

## 重点

- MyBatis 的核心是 SQL 映射和执行。
- Mapper 接口通过动态代理绑定 XML 或注解 SQL。
- 深入 MyBatis 必须理解执行链路和映射规则。

## 难点

- 参数绑定、动态 SQL、结果映射、缓存和插件都发生在执行链路不同阶段。
- MyBatis 很灵活，也意味着规范不足时容易写出难维护 SQL。

## 易错

> **易错：** 以为 MyBatis 会像 Hibernate 一样自动维护对象关系和更新。
>
> 正确做法：把 MyBatis 当 SQL Mapper 使用，明确写 SQL、明确映射、明确事务边界。
