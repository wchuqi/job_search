# MyBatis 学习资料：核心架构和执行流程

[返回索引](../Mybatis学习资料.md)

## 学习目标

- 深入理解 Mapper 方法如何转换为 JDBC 执行。
- 掌握 Configuration、MappedStatement、SqlSource、BoundSql、Executor、StatementHandler、ParameterHandler、ResultSetHandler、TypeHandler 的职责。
- 能解释 MyBatis 插件为什么能拦截 Executor、StatementHandler 等对象。
- 能根据执行链路定位参数、SQL、结果映射和性能问题。

## 理论导读

MyBatis 表面上是调用 Mapper 接口，内部其实是一条精细的执行流水线。启动时，MyBatis 解析 XML/注解，把每个 SQL 语句注册成 `MappedStatement`。运行时，Mapper 接口的动态代理根据方法找到对应 `MappedStatement`，动态 SQL 生成 `BoundSql`，Executor 负责查询或更新，StatementHandler 创建 JDBC Statement，ParameterHandler 绑定参数，ResultSetHandler 把 ResultSet 变成 Java 对象。

理解这条链路后，很多问题不再神秘：为什么 namespace/id 错会找不到 statement，为什么 `#{}` 能防注入，为什么插件能做分页，为什么 resultMap 配错会映射为空。

## 核心心智模型

```text
MapperProxy
  -> MapperMethod
  -> SqlSession
  -> MappedStatement
  -> SqlSource 生成 BoundSql
  -> Executor
  -> StatementHandler
  -> ParameterHandler
  -> JDBC PreparedStatement
  -> ResultSetHandler
  -> TypeHandler
```

## 知识点详解

### 1. Configuration

`Configuration` 是 MyBatis 运行时总配置，里面保存：

- MappedStatement 集合。
- ResultMap 集合。
- TypeHandlerRegistry。
- MapperRegistry。
- 插件 Interceptor。
- 缓存配置。
- 全局 settings。

可以把它理解为 MyBatis 的“运行时注册表”。

### 2. MappedStatement

每个 `<select>`、`<insert>`、`<update>`、`<delete>` 都会变成一个 `MappedStatement`。它包含：

- statement id。
- SQL 类型。
- SqlSource。
- 参数类型。
- 结果映射。
- 缓存策略。
- 超时、fetchSize 等。

statement id 通常是：

```text
namespace + "." + sqlId
```

例如：

```text
com.example.UserMapper.selectById
```

这就是 `Invalid bound statement` 常见原因：接口方法全限定名和 XML namespace/id 对不上。

### 3. SqlSource 和 BoundSql

`SqlSource` 代表 SQL 来源。静态 SQL 直接生成 BoundSql；动态 SQL 会先根据参数计算 `<if>`、`<foreach>`、`<choose>` 等节点，再生成最终 SQL。

`BoundSql` 包含：

- 最终 SQL。
- 参数映射列表。
- 参数对象。
- 动态 SQL 产生的附加参数。

例子：

```xml
<select id="selectByStatus" resultType="Order">
    select id, status
    from orders
    where 1 = 1
    <if test="status != null">
        and status = #{status}
    </if>
</select>
```

当 `status = "PAID"` 时，BoundSql 才会包含 `and status = ?`。

### 4. Executor

Executor 负责真正执行查询/更新和缓存协调。常见类型：

| Executor | 特点 |
| --- | --- |
| SimpleExecutor | 每次执行创建 Statement |
| ReuseExecutor | 复用 Statement |
| BatchExecutor | 批处理更新 |

分页插件通常拦截 Executor 或 StatementHandler，在 SQL 执行前改写 SQL 或统计总数。

### 5. Executor 查询的内部步骤

一次查询不是直接执行 JDBC，它会先经过缓存、延迟加载和查询栈控制。

典型查询路径：

```text
Executor.query
  -> 根据 MappedStatement、SQL、参数、RowBounds 创建 CacheKey
  -> 查一级缓存 localCache
  -> 命中则返回
  -> 未命中则 queryFromDatabase
  -> StatementHandler.prepare 创建 Statement
  -> ParameterHandler.setParameters 绑定参数
  -> StatementHandler.query 执行 JDBC
  -> ResultSetHandler.handleResultSets 映射结果
  -> 写入一级缓存
```

CacheKey 通常由这些因素组成：

- statementId。
- RowBounds offset/limit。
- 最终 SQL。
- 参数值。
- Environment id。

这解释了为什么“同一 SQL 不同参数”不会命中同一缓存，也解释了为什么动态 SQL 最终文本不同会影响缓存 key。

> **难点：** 一级缓存不是只按 SQL 字符串缓存，而是按 statement、分页、SQL、参数等组合生成 CacheKey。

### 6. 更新为什么会清缓存

insert/update/delete 会影响数据一致性，所以 MyBatis 通常会清理当前 SqlSession 的一级缓存。否则同一个事务中先查一次用户，再更新用户，再查用户，如果缓存不清理，就可能读到旧值。

```text
selectById(1) -> 缓存 User(id=1, name=A)
updateName(1, B) -> 清理 localCache
selectById(1) -> 重新查数据库
```

二级缓存的清理按 namespace 进行，这也是它在多表查询中风险较高的原因：一个 namespace 的更新不一定能清理另一个 namespace 的缓存。

### 7. StatementHandler

StatementHandler 负责创建 JDBC Statement、设置 SQL、执行查询。它下面还会使用 ParameterHandler 和 ResultSetHandler。

插件拦截 StatementHandler 常用于：

- SQL 改写。
- 分页。
- SQL 日志。
- 性能统计。

### 8. StatementHandler 的四个关键动作

StatementHandler 内部通常围绕四个动作：

```text
prepare       -> 创建 Statement/PreparedStatement
parameterize  -> 绑定参数
query/update  -> 执行 SQL
batch         -> 加入批处理
```

分页插件、SQL 审计插件、慢 SQL 插件常拦截 `prepare` 或 `query`。如果插件在 prepare 前改写 SQL，就必须保证 BoundSql 中的 ParameterMapping 仍然和 SQL 中的 `?` 数量一致。

> **易错：** 插件只改 SQL 文本，不同步处理参数映射，导致参数数量和占位符不一致。

### 9. ParameterHandler 和 TypeHandler

ParameterHandler 根据 BoundSql 的参数映射，把 Java 参数绑定到 PreparedStatement。

TypeHandler 负责 Java 类型和 JDBC 类型转换：

- String -> VARCHAR。
- Long -> BIGINT。
- LocalDateTime -> TIMESTAMP。
- Enum -> VARCHAR/INTEGER。

自定义枚举、JSON 字段、加密字段常需要自定义 TypeHandler。

### 10. TypeHandler 选择规则

TypeHandler 选择通常依赖 Java 类型和 JDBC 类型：

```text
Java 参数类型 + JdbcType
  -> TypeHandlerRegistry 查找处理器
  -> setParameter / getResult
```

如果同一个 Java 类型对应多种数据库表达，例如枚举可以存 name、ordinal、code，就必须显式指定或注册清晰的 TypeHandler。否则不同 Mapper 使用不同存储方式，会造成结果不可预测。

### 11. ResultSetHandler

ResultSetHandler 把 ResultSet 映射为 Java 对象。它会结合 resultType/resultMap、构造器、属性名、列名、TypeHandler、嵌套结果或嵌套查询。

复杂关联映射、字段为 null、集合重复、N+1 查询，大多和 ResultSetHandler 阶段有关。

## 例子：执行链路定位问题

现象：调用 `userMapper.selectById(1L)` 报 `Invalid bound statement`。

排查：

1. Mapper 接口全限定名：`com.example.UserMapper`。
2. XML namespace 是否等于该全限定名。
3. XML 中是否有 `<select id="selectById">`。
4. XML 是否被 `mapper-locations` 加载。
5. 多模块项目中 XML 是否被打包到 classpath。

## 练习

1. 打开 MyBatis 日志，观察 BoundSql。
2. 故意写错 XML id，触发 `Invalid bound statement`。
3. 写一个自定义 TypeHandler 映射枚举。
4. 使用 BatchExecutor 做批量插入，并观察 SQL 执行。

## 重点

- MyBatis 启动期注册 MappedStatement，运行期执行 BoundSql。
- `#{}` 最终变成 PreparedStatement 参数。
- 结果映射是 ResultSetHandler 的核心职责。

## 难点

- 动态 SQL 会改变最终 BoundSql，排查必须看运行时 SQL。
- TypeHandler 既影响参数绑定，也影响结果读取。

## 易错

> **易错：** 只看 XML 原始 SQL，不看动态 SQL 计算后的 BoundSql。
>
> 正确做法：打开日志或插件，观察最终 SQL 和参数。
