# 分布式系统学习资料：复制、分片和数据迁移深度推演

[返回索引](../分布式系统学习资料.md)

## 学习目标

- 能从读写路径、故障切换、迁移和校验角度设计数据分布。
- 理解副本滞后、热点、跨分片查询和再均衡的真实成本。
- 掌握不停机迁移的常见步骤和风险点。

## 理论导读

复制和分片是扩展系统的核心手段，但它们不是“加机器”这么简单。复制会引入副本一致性和故障切换问题；分片会引入路由、热点、跨分片事务、跨分片查询和数据迁移问题。

## 复制深度推演

### 主从复制的数据丢失窗口

主库写入成功后，如果异步复制尚未到从库，主库宕机并切换到从库，则这部分数据可能丢失。

```text
Client -> Primary: write x=1 ok
Primary -- async not yet --> Replica
Primary crash
Replica promoted, x still 0
```

解决方向：

- 半同步复制，至少一个从库确认。
- 关键写入等待更多副本。
- 故障切换前检查复制位点。
- 业务允许时用对账补偿。

### 副本滞后下的读异常

常见异常：

- 写后读旧值：用户改资料后读从库。
- 单调读破坏：第一次读到新值，第二次路由到更落后的副本读到旧值。
- 前缀读破坏：读到后续事件但没读到前置事件。

处理方式：

- 写后短时间读主。
- 按用户会话固定副本。
- 读请求携带最小版本。
- 副本落后超过阈值就不参与读流量。

## 分片键选择

分片键决定系统未来很多年能否扩展。

| 分片键 | 优点 | 风险 |
| --- | --- | --- |
| userId | 用户维度查询快 | 大客户或超级用户热点 |
| orderId 哈希 | 写入均匀 | 用户订单列表跨分片 |
| tenantId | 租户隔离好 | 大租户压垮单分片 |
| 时间 | 范围归档方便 | 最新时间段写热点 |
| 组合键 | 可兼顾维度 | 路由和查询复杂 |

## 热点治理

热点可能来自：

- 单个 key 被高频访问。
- 单个租户数据量远超平均。
- 时间范围分片导致最新分片写入集中。
- 营销活动导致某商品库存成为热点。

治理方案：

- 读热点：本地缓存、多级缓存、热点副本、请求合并。
- 写热点：预分片、令牌桶、队列串行化、库存分桶。
- 大租户：独立分片或独立集群。
- 时间热点：时间桶加哈希后缀。

## 不停机迁移流程

从单库迁移到分库分表，典型流程：

1. 建新表和新路由规则。
2. 历史数据批量迁移。
3. 开启增量同步，例如 binlog 或双写。
4. 双读校验，比较新旧结果。
5. 小流量读新库。
6. 全量读新库。
7. 写入切到新库。
8. 保留旧库回滚窗口。
9. 完成一致性校验后下线旧路径。

## 双写风险

双写最常见的问题是两边成功状态不一致：

```text
写旧库成功 -> 写新库失败
写旧库失败 -> 写新库成功
写旧库成功 -> 写新库成功 -> 返回前超时
```

安全策略：

- 明确一个主写源，另一个通过日志同步。
- 双写失败进入补偿队列。
- 所有写入有幂等键。
- 校验工具能按范围和版本比较。
- 切流有开关和回滚路径。

## 跨分片查询

跨分片查询会带来扇出成本：

```text
查询用户最近订单，如果按 orderId 分片：
请求 -> 所有分片查询 -> 合并排序 -> 返回
```

优化方式：

- 按查询主维度分片。
- 建读模型或搜索索引。
- 保存冗余索引表。
- 限制查询范围和分页深度。
- 异步构建报表。

## 练习

为“在线招聘平台投递记录”设计分片：

1. 候选人查自己的投递列表。
2. 招聘方查某职位收到的投递。
3. 运营按时间统计投递量。
4. 大客户职位突然收到海量投递。

说明你选择的主存储分片键、读模型、索引和热点治理方案。

## 验收

- 能解释异步复制的数据丢失窗口。
- 能设计读己之写和单调读策略。
- 能给出不停机迁移步骤和回滚方案。
- 能识别热点并选择治理手段。

## 重点

- 复制提高可用性，但引入滞后和切换复杂度。
- 分片提高容量，但牺牲跨分片查询和事务便利性。
- 迁移方案必须包含校验和回滚。

## 难点

- 分片键不是技术选择，而是业务访问模式选择。
- 数据迁移期间新旧系统同时存在，最容易出现隐性不一致。

## 易错

> **易错：** “先双写，后面慢慢校验”。
>
> 正确做法：双写前就定义主写源、失败补偿、校验指标、切流比例和回滚窗口。
