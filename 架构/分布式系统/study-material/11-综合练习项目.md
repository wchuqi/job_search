# 分布式系统学习资料：综合练习项目

[返回索引](../分布式系统学习资料.md)

## 项目：分布式订单事件系统

## 学习目标

- 综合练习服务拆分、消息、幂等、最终一致、弹性和可观测性。
- 输出一份能用于面试讲解的架构设计文档。
- 通过故障注入验证系统不是只在正常路径可用。

## 背景

设计一个简化电商订单系统，包含订单、库存、支付、通知四个能力。要求创建订单后扣减库存，支付成功后通知用户。库存、支付、通知可以异步处理，但订单状态必须可查询、可恢复、可对账。

## 功能范围

- 创建订单。
- 扣减或冻结库存。
- 支付成功或失败回调。
- 发送通知。
- 查询订单状态。
- 失败重试、补偿和对账。

## 推荐架构

```text
Client
  |
API Gateway
  |
Order Service -- DB(order, outbox)
  |
Message Broker
  |-- Inventory Service -- DB(inventory, consume_log)
  |-- Payment Service   -- DB(payment, consume_log)
  |-- Notify Service    -- DB(notification, consume_log)
```

## 必做任务

### 任务 1：定义状态机

订单状态至少包含：

- CREATED
- INVENTORY_RESERVED
- WAITING_PAYMENT
- PAID
- NOTIFIED
- CANCELLED
- FAILED_REQUIRES_MANUAL_CHECK

验收：

- 每个状态有允许的前置状态。
- 每次状态变更都有事件记录。
- 重复事件不会让状态倒退。

### 任务 2：设计事件契约

至少定义：

- OrderCreated
- InventoryReserved
- InventoryFailed
- PaymentSucceeded
- PaymentFailed
- NotificationSent

验收：

- 每个事件有 eventId、eventType、eventVersion、aggregateId、occurredAt。
- 消费者用 eventId 去重。
- 同一订单事件按 aggregateId 分区。

### 任务 3：实现 Outbox 思路

不要求真实接入 Kafka，也可以用数据库表模拟消息发布。

验收：

- 创建订单和写 outbox 在同一个本地事务中。
- 发布任务可以重复扫描未发布事件。
- 发布成功后更新 outbox 状态。

### 任务 4：故障注入

模拟以下故障：

- 库存服务处理超时。
- 支付回调重复到达。
- 通知服务失败 3 次后进入死信。
- 消息消费者处理成功但确认失败。

验收：

- 业务状态最终可恢复。
- 重复消息不会重复扣库存或重复通知。
- dashboard 或日志能解释故障过程。

## 扩展任务

- 增加租户限流。
- 增加读模型，提供订单列表查询。
- 增加对账任务，修复长时间停留在中间状态的订单。
- 使用 Docker Compose 启动数据库和消息队列。

## 重点

- 项目目标不是堆组件，而是验证分布式语义。
- 所有失败都要能被观测、重试、补偿或人工介入。

## 难点

- 消息处理成功但确认失败会导致重复消费，必须依赖幂等。
- 补偿流程要遵守业务规则，不能简单删除数据。

## 易错

> **易错：** 只实现正常链路，没有故障注入。
>
> 正确做法：把超时、重复、乱序、失败和恢复作为项目验收条件。
