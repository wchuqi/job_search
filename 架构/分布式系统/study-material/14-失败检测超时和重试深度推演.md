# 分布式系统学习资料：失败检测、超时和重试深度推演

[返回索引](../分布式系统学习资料.md)

## 学习目标

- 理解失败检测为什么无法给出绝对结论，只能给出基于超时的怀疑。
- 能把一次远程调用拆成排队、连接、传输、执行、响应和确认阶段。
- 能设计端到端超时预算、重试策略、幂等策略和故障证据链。

## 理论导读

分布式系统中，“节点是否失败”不是一个可以瞬间确认的事实。调用方无法区分远端进程崩溃、网络丢包、远端正在执行但很慢、响应已经发出但在网络中丢失、调用方线程被调度延迟这些情况。失败检测器本质上是“超时后怀疑对方失败”的机制。

这意味着工程设计不能依赖“准确知道失败”。正确设计要承认不确定性：超时后要么查询状态，要么幂等重试，要么进入待确认状态，要么交给对账和人工处理。

## 核心心智模型

一次 RPC 不是一条直线，而是一段有多个等待点的流水线：

```text
上游排队 -> 获取连接 -> 网络发送 -> 下游队列 -> 下游执行
        -> 下游提交状态 -> 响应发送 -> 上游读取 -> 上游确认结果
```

任意阶段变慢，上游都可能只看到一个笼统的 timeout。你看到 timeout，不代表业务没有发生。

## 失败检测的关键结论

### 1. 超时是本地决策，不是全局事实

超时由调用方本地时钟决定。调用方说“我等不下去了”，不是说“对方一定失败了”。这也是为什么支付、下单、转账、库存扣减这类操作不能在超时后直接发起无保护重试。

### 2. 心跳失败只能证明“当前路径不可用”

心跳经过同样的网络和调度路径。心跳没回来，可能是：

- 对方进程死了。
- 网络分区。
- 对方 Stop-The-World GC。
- 对方 CPU 或 IO 饱和。
- 心跳线程池和业务线程池互相抢资源。
- 本机接收线程被阻塞。

因此心跳只适合做怀疑信号，不能单独作为高风险切主依据。

### 3. 慢比死更危险

节点崩溃通常容易摘除；慢节点会持续占用连接、线程和队列，诱发级联故障。很多生产事故不是“服务挂了”，而是“服务还活着但响应越来越慢”。

## 超时预算设计

假设入口接口 SLO 是 300ms 内返回，链路如下：

```text
Gateway -> Order -> Inventory -> Redis
                 -> Payment
                 -> Outbox DB
```

不能让每个依赖都独立设置 300ms 超时，否则最坏情况会远超用户预算。应该做端到端预算：

| 阶段 | 预算示例 | 原因 |
| --- | --- | --- |
| Gateway 到 Order | 280ms | 给网关留返回和日志时间 |
| Order 内部总预算 | 240ms | 业务编排需要整体截止时间 |
| Inventory | 80ms | 库存是核心依赖，但不能无限等待 |
| Redis | 20ms | 缓存慢时应快速失败或降级 |
| Payment 创建单 | 100ms | 可进入待支付状态异步确认 |
| Outbox DB | 40ms | 写本地事务必须可控 |

> **重点：** 超时应该逐层递减，并传递 deadline，而不是每层重新开始计时。

## 重试策略推演

### 能重试的条件

一个操作适合自动重试，至少满足：

- 错误大概率是临时错误。
- 操作是只读、天然幂等，或带业务幂等键。
- 重试不会突破全链路 deadline。
- 下游没有明确提示“不要重试”。
- 系统有退避、抖动、限流和熔断保护。

### 不适合自动重试的情况

- 支付扣款、发放权益、发送短信等有外部副作用且无幂等键。
- 下游返回业务校验失败，例如余额不足、库存不足。
- 当前请求已经超过用户等待预算。
- 依赖整体处于熔断状态。

## 重试风暴的形成

假设 A 调 B，B 调 C。A、B、网关都设置 3 次重试：

```text
Gateway: 3
  A: 每次 3
    B: 每次 3
      C: 最多被打 27 次
```

如果 C 已经因为慢查询接近饱和，这种重试会把 C 的队列塞满，同时拖垮 B 和 A。

正确做法：

- 只在最了解语义的一层重试。
- 设置全链路 deadline。
- 使用指数退避和随机抖动。
- 对依赖错误率升高启用熔断。
- 重试前检查请求是否仍有意义。

## 深度场景：订单创建超时

### 场景

客户端调用创建订单接口，5 秒后超时。服务端可能处于以下状态：

| 服务端真实状态 | 客户端看到 | 风险 | 正确处理 |
| --- | --- | --- | --- |
| 请求未到达 | timeout | 订单未创建 | 可用幂等键重试 |
| 订单写入成功，响应丢失 | timeout | 重试可能重复创建 | 用 requestId 查询或幂等重试 |
| 订单写入成功，库存处理中 | timeout | 用户不知是否成功 | 返回“处理中”并异步确认 |
| 订单写入失败但 outbox 写入成功 | timeout | 数据不一致 | 本地事务必须避免这种状态 |
| 服务写入后进程崩溃 | timeout | 后续事件可能未发布 | Outbox 扫描补发 |

### 设计方案

1. 客户端生成 `requestId`。
2. 订单表对 `userId + requestId` 建唯一索引。
3. 创建订单和写 Outbox 在同一事务。
4. 超时后客户端使用 `requestId` 查询结果。
5. 重试时如果唯一索引冲突，返回已有订单。
6. 后台对长时间中间态订单做对账推进。

## 诊断证据链

排查超时不能只看一条日志，应串起：

- 入口请求量、错误率、P95/P99。
- trace 中慢 span 的位置。
- 上游连接池等待时间。
- 下游线程池队列长度。
- 数据库慢查询、锁等待、连接数。
- 消息队列堆积和消费延迟。
- 最近发布、配置和流量变化。

## 练习

为“支付回调处理”设计失败处理：

1. 第三方回调超时后会重试，系统如何幂等？
2. 回调已写库但订单状态更新失败怎么办？
3. 回调重复 10 次是否会重复发权益？
4. 你会设置哪些日志字段和告警？

## 验收

- 能说明失败检测只能产生怀疑，不能证明远端真实状态。
- 能给出端到端超时预算，而不是随意写 3 秒、5 秒。
- 能解释某个操作为什么能重试或不能重试。
- 能设计 requestId、唯一索引、状态查询和对账机制。

## 重点

- 超时是观察结果，不是业务结果。
- 重试是放大器，既能提高成功率，也能放大故障。
- 幂等和状态查询是处理超时歧义的核心手段。

## 难点

- 多层框架默认重试可能叠加，必须统一治理。
- 慢节点造成的资源耗尽常常比直接失败更隐蔽。

## 易错

> **易错：** “超时了就重试一次”。
>
> 正确做法：先判断操作是否幂等、是否有剩余 deadline、下游是否熔断、重试是否会造成外部副作用。
