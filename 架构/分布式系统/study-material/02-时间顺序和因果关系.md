# 分布式系统学习资料：时间、顺序和因果关系

[返回索引](../分布式系统学习资料.md)

## 学习目标

- 理解分布式系统中不存在可靠的全局物理时钟。
- 掌握逻辑时钟、因果关系、版本向量和冲突检测的用途。
- 能判断业务是否需要全局顺序、分区内顺序或因果顺序。

## 理论导读

单机程序可以用执行顺序理解状态变化。分布式系统中，不同节点各自执行，消息在网络中延迟和乱序，物理时钟还会漂移。因此“哪个事件先发生”并不总是有唯一答案。系统只能根据消息传递关系建立因果顺序，或者通过共识、单主写入、全局序列号等机制构造一个人为顺序。

## 核心心智模型

如果两个事件之间没有信息传递，就像两个城市同时发生的新闻，单靠本地钟表很难证明谁先谁后。只有当 A 的结果被 B 看到并用于下一步时，才能说 A 因果上先于 B。

## 知识点详解

### 物理时钟

物理时钟来自机器时间，受 NTP 同步、时钟漂移、闰秒、虚拟化暂停影响。它适合记录日志时间、设置过期时间、做粗略排序，但不适合单独判断并发写冲突的真实顺序。

### Lamport Clock

Lamport Clock 用递增计数器表达“如果 A 因果先于 B，则 A 的时间戳小于 B”。它能给事件排序，但不能判断两个事件是否并发。

基本规则：

```text
本地事件：counter = counter + 1
发送消息：携带 counter
接收消息：counter = max(localCounter, messageCounter) + 1
```

### Vector Clock

Vector Clock 为每个节点维护一个计数，可以判断两个版本是否存在因果关系。

| 关系 | 含义 |
| --- | --- |
| A 全部计数 <= B 且至少一个 < | A 发生在 B 之前 |
| B 全部计数 <= A 且至少一个 < | B 发生在 A 之前 |
| 两者互有大于对方的项 | A 和 B 并发，需要冲突解决 |

### 顺序的常见层次

- 全局顺序：所有操作都有同一个顺序，通常需要单主、共识或中心化序列。
- 分区内顺序：同一 key 或同一分区内有序，例如 Kafka 单分区消息顺序。
- 因果顺序：先看到 A 再产生 B，则所有观察者不能先看到 B 后看到 A。
- 无序：系统只承诺最终处理，业务自己做幂等和去重。

## 例子

协作文档中，两个人离线编辑同一段文字。版本 A 和版本 B 互相没有看到对方，属于并发更新。系统不能简单用“最后写入时间”覆盖，否则会丢失其中一方内容。更合理的做法是保存两个版本，提示用户合并，或使用 CRDT/OT 等冲突合并算法。

## 练习

判断以下场景需要什么顺序：

1. 同一用户的账户流水。
2. 不同用户的浏览日志。
3. 同一订单的状态流转。
4. 全站热搜计数。

## 验收

- 能解释物理时钟不可靠的原因。
- 能说明 Lamport Clock 和 Vector Clock 的区别。
- 能为业务选择全局顺序、分区顺序或因果顺序。

## 重点

- 时间戳不是因果关系。
- 并发写不一定能自动合并，必须定义冲突策略。
- 全局顺序昂贵，优先判断是否只需要局部顺序。

## 难点

- 很多系统实际只需要同一实体内有序，而不是全局有序。
- “最后写入获胜”简单但可能丢数据。

## 易错

> **易错：** 用数据库更新时间字段解决所有冲突。
>
> 正确做法：先判断是否允许覆盖；不允许覆盖时使用版本号、条件更新、合并策略或人工冲突处理。
