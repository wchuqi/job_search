# 分布式系统学习资料：系统设计面试表达框架

[返回索引](../分布式系统学习资料.md)

## 学习目标

- 能把分布式系统知识转化为系统设计面试中的清晰表达。
- 能围绕需求、容量、数据、一致性、故障、观测和演进展开设计。
- 避免只堆 Redis、Kafka、微服务、分库分表等组件名。

## 总体表达顺序

系统设计建议按以下顺序讲：

1. 澄清需求和非功能目标。
2. 做容量估算。
3. 定义核心数据模型和读写路径。
4. 选择数据分布、复制和分片策略。
5. 明确一致性和事务边界。
6. 设计异步消息、幂等和补偿。
7. 设计弹性、限流、降级和故障恢复。
8. 设计可观测性、安全和发布回滚。
9. 说明取舍、风险和演进路径。

## 一、需求澄清

不要直接画架构。先问：

- 核心功能是什么？
- 读多还是写多？
- 数据规模和增长速度？
- 延迟目标是多少？
- 哪些数据不能错？
- 是否跨地域？
- 是否需要实时性？
- 是否允许最终一致？
- 峰值流量和热点场景是什么？

## 二、容量估算

示例：设计短链接系统。

```text
日新增短链：1000 万
读写比：100:1
读 QPS 平均：10000 万 / 86400 ≈ 1157
峰值按 10 倍：约 12000 QPS
每条记录 200B，日增约 2GB，年增约 730GB，不含索引和副本
```

容量估算的作用不是精确，而是决定：

- 单库是否够。
- 是否需要缓存。
- 是否要分片。
- 是否需要异步化。
- 备份和成本规模。

## 三、读写路径表达

用路径讲系统，比堆组件更清楚。

```text
写路径：
Client -> API -> Service -> DB
                    -> Outbox -> MQ -> Search/Cache/Notify

读路径：
Client -> API -> Cache -> DB/ReadModel
```

每条路径都要说明：

- 成功条件。
- 失败处理。
- 是否可重试。
- 数据一致性。
- 观测指标。

## 四、一致性表达模板

不要只说“最终一致”。要这样讲：

```text
订单主状态需要强约束，使用订单服务单主写入和状态机条件更新。
库存扣减允许短暂最终一致，但必须防超卖，所以对同一 sku 使用库存分桶或预扣令牌。
通知是非核心副作用，使用消息异步处理，至少一次投递，消费端用 eventId 幂等。
搜索索引允许分钟级延迟，通过 Outbox 事件异步更新，并有补偿扫描。
```

## 五、故障模型表达模板

系统设计里主动说故障，会显得更成熟：

```text
我假设网络调用可能超时，消息可能重复，消费者可能处理成功但 ack 失败，数据库主从可能有复制延迟。
因此创建订单用 requestId 做幂等；订单和 outbox 在同一事务提交；
消费端有 inbox 去重；超时订单由对账任务扫描；核心接口有限流和降级。
```

## 六、常见题型切入点

### 设计秒杀系统

重点：

- 热点库存。
- 限流和排队。
- 库存预扣或令牌。
- 防重复下单。
- 异步创建订单。
- 用户查询最终结果。

容易错：

- 所有请求直接打数据库扣库存。
- 只用缓存扣减但没有对账。
- 没有处理支付超时和订单关闭。

### 设计消息通知系统

重点：

- 模板、渠道、用户偏好。
- 异步队列和重试。
- 供应商限流。
- 幂等和去重。
- 死信和人工处理。
- 送达率、失败率和延迟指标。

容易错：

- 同步发送通知拖垮主链路。
- 失败无限重试。
- 不记录外部供应商流水。

### 设计分布式任务调度

重点：

- 任务分片。
- 租约和抢占。
- 失败重试。
- 幂等执行。
- 进度 checkpoint。
- 防止同一分片被旧 worker 写入，必要时用 fencing token。

容易错：

- 只用心跳判断 worker 死亡。
- 锁过期后旧 worker 继续写。
- 任务执行无幂等。

## 七、面试高分表达

### 从约束推导组件

普通表达：

```text
我会用 Redis、Kafka、MySQL 分库分表。
```

更好表达：

```text
因为读 QPS 峰值约 2 万，且允许 1 秒内的短暂不一致，所以详情读路径加缓存。
因为创建后要通知、建索引、更新统计，这些不是主链路强一致动作，所以用 Outbox 加 MQ 异步发布。
因为订单量按用户查询和按商家查询都有需求，主表按 orderId 分片，再为用户和商家建立读模型。
```

### 主动说明取舍

```text
这里我不选择跨服务 2PC，因为流程包含支付和通知等外部副作用，锁资源时间长且可用性差。
我会用 Saga 状态机，每一步本地事务提交，失败后补偿，并通过对账任务处理长期中间态。
```

## 练习

任选一个题目写设计：

1. 分布式订单系统。
2. 短链接系统。
3. 秒杀系统。
4. 分布式任务调度系统。
5. 聊天消息系统。

要求输出：

- 需求澄清。
- 容量估算。
- 核心模型。
- 读写路径。
- 一致性选择。
- 故障处理。
- 可观测指标。
- 取舍和演进。

## 验收

- 能用业务约束解释组件选择。
- 能明确哪些路径强一致，哪些最终一致。
- 能主动覆盖超时、重复、乱序、堆积和故障恢复。
- 能给出可观测指标和回滚策略。

## 重点

- 系统设计不是背标准答案，是在约束下做取舍。
- 分布式系统面试最看重异常路径和边界条件。
- 好答案要能落到状态机、幂等键、读写路径和告警指标。

## 难点

- 面试中要控制层次：先讲主路径，再讲高风险异常，再讲扩展。
- 不要过早陷入某个中间件细节，除非面试官追问。

## 易错

> **易错：** 架构图很复杂，但没有说数据一致性和失败处理。
>
> 正确做法：每个组件都要回答它解决什么问题、引入什么风险、失败后如何恢复。
