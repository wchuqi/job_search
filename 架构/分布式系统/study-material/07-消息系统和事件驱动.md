# 分布式系统学习资料：消息系统和事件驱动

[返回索引](../分布式系统学习资料.md)

## 学习目标

- 掌握消息队列的削峰、解耦、异步、顺序和重放能力。
- 理解至少一次、至多一次、精确一次效果、幂等消费和死信队列。
- 能设计事件驱动系统的事件契约和消费语义。

## 理论导读

消息系统把同步调用改造成异步协作。生产者只需要发布事实或命令，消费者按自己的节奏处理。它能削峰填谷、隔离故障、支持重放和多消费者扩展，但也引入延迟、乱序、重复消费、消息堆积和事件版本治理问题。

## 核心心智模型

消息队列像传送带。上游把包裹放上去，下游按速度取走。传送带可以缓冲高峰，但如果下游长期处理不过来，包裹会堆积；如果包裹标签不规范，下游会误处理；如果上游重复放包裹，下游必须识别重复。

## 知识点详解

### 投递语义

| 语义 | 含义 | 风险 |
| --- | --- | --- |
| 至多一次 | 消息可能丢，但不会重复 | 适合可丢指标 |
| 至少一次 | 消息不轻易丢，但可能重复 | 消费端必须幂等 |
| 精确一次效果 | 端到端结果看起来只执行一次 | 依赖事务、幂等和去重，不只是 MQ 开关 |

### 顺序

消息队列通常只能保证同一分区内有序。要保证同一订单事件顺序，应以 orderId 作为分区键。不同订单之间通常不需要全局顺序。

### 消费者处理流程

1. 拉取或接收消息。
2. 校验事件版本和字段。
3. 根据幂等键判断是否处理过。
4. 执行业务本地事务。
5. 记录消费结果。
6. 提交 offset 或确认消息。

### 死信和重试

重试适合临时故障，例如网络抖动。死信适合长期无法处理的消息，例如字段格式错误、业务状态不允许、下游数据缺失。死信队列必须有告警、排查和重放工具。

## 例子

订单创建事件：

```json
{
  "eventId": "evt-001",
  "eventType": "OrderCreated",
  "eventVersion": 1,
  "occurredAt": "2026-06-06T10:00:00Z",
  "aggregateId": "order-1001",
  "payload": {
    "userId": "u1",
    "amount": 9900
  }
}
```

消费者用 `eventId` 做去重，用 `aggregateId` 做顺序分区，用 `eventVersion` 做兼容处理。

## 练习

为“投递职位后通知招聘方”设计事件：

1. 事件名和字段。
2. 幂等键。
3. 重试策略。
4. 死信处理。
5. 事件版本升级方式。

## 验收

- 能说明消息队列解决的问题和引入的问题。
- 能解释至少一次投递下为什么必须幂等。
- 能设计分区键、事件契约、重试和死信策略。

## 重点

- 事件表达已经发生的事实，命令表达希望对方执行的动作。
- 消费者必须能处理重复、乱序和延迟。
- offset 提交位置决定失败后的重复或丢失风险。

## 难点

- 精确一次效果是端到端设计，不是只配置一次。
- 事件版本治理需要兼容旧消费者。

## 易错

> **易错：** 消费业务逻辑成功前就提交 offset。
>
> 正确做法：业务本地事务成功后再确认消息；重复消费通过幂等表或状态条件更新处理。
