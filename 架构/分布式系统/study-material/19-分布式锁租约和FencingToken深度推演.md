# 分布式系统学习资料：分布式锁、租约和 fencing token 深度推演

[返回索引](../分布式系统学习资料.md)

## 学习目标

- 理解分布式锁的风险来自租约过期、暂停、网络分区和外部资源不受锁服务控制。
- 掌握 fencing token 为什么能阻止过期持有者写入。
- 能判断什么时候不该使用分布式锁。

## 理论导读

分布式锁常被用来保护互斥操作，但“拿到锁”不等于“之后所有操作都安全”。锁服务只能控制锁的发放，无法控制客户端在锁过期后是否还继续执行，也无法让外部数据库、文件系统或第三方接口自动知道谁的锁更新。

## 租约问题

常见 Redis 锁形态：

```text
SET lock_key random_value NX PX 30000
```

它表达的是：“如果锁不存在，则设置一个 30 秒过期的锁”。问题在于客户端可能在拿锁后暂停超过 30 秒。

```text
T0  Client A 获取锁，租约 30s
T1  Client A GC 暂停 40s
T30 锁过期
T31 Client B 获取锁并执行业务
T41 Client A 恢复，继续以为自己持有锁，写入旧结果
```

此时锁本身没有阻止 A 的过期写入。

## fencing token

fencing token 是锁服务每次发锁时返回的单调递增令牌。受保护资源只接受 token 更大的写入。

```text
Client A 获取锁 token=10
Client A 暂停
Client B 获取锁 token=11
Client B 写资源，资源记录 last_token=11
Client A 恢复，用 token=10 写资源
资源拒绝，因为 10 < 11
```

关键是：受保护资源必须参与校验 token。只有锁服务返回 token 不够，数据库、存储或业务服务也要拒绝旧 token。

## 分布式锁适用场景

适合：

- 低频管理任务，例如定时任务主实例。
- 可重试、可补偿、可校验的互斥操作。
- 锁失效后不会造成不可逆损失的流程。

谨慎：

- 扣款、发券、转账等高风险外部副作用。
- 长时间任务。
- 依赖本地进程续约的任务。
- 没有 fencing token 的共享资源写入。

## 替代方案

很多场景不需要锁：

- 用数据库唯一约束防重复创建。
- 用状态机条件更新防重复推进。
- 用队列按 key 串行化处理。
- 用乐观锁版本号处理并发更新。
- 用幂等键处理重复请求。
- 用分片单线程模型处理同一实体命令。

## 深度场景：定时任务重复执行

多个实例都可能执行“结算昨日账单”。使用锁时：

1. 任务开始前获取锁和 token。
2. 结算批次表记录 batchId 和 token。
3. 写入结算结果时带 token 条件。
4. 任务可分片，每个分片有独立状态。
5. 锁过期后旧任务继续写入会被 token 拒绝。

更好的方案可能是任务表抢占：

```sql
UPDATE job_shards
SET owner = 'worker-1', version = version + 1
WHERE shard_id = 1
  AND status IN ('READY', 'TIMEOUT')
  AND version = 7;
```

数据库条件更新本身就是互斥控制。

## Redlock 争议的学习重点

不必把面试答成论文争论，但要说清楚：

- 多 Redis 节点加锁试图降低单点风险。
- 仍然依赖时钟、超时和租约假设。
- 对高风险一致性场景，应该优先使用共识系统或数据库约束。
- 锁之外仍需要 fencing token 或下游条件校验。

判断以下场景是否适合分布式锁：

1. 防止重复创建订单。
2. 每天凌晨只有一个实例生成报表。
3. 多实例同时扣减同一商品库存。

## 验收

- 能解释租约过期后旧客户端继续执行的风险。
- 能说明 fencing token 的工作方式和前提。
- 能为高风险业务选择唯一约束、条件更新或队列串行化替代锁。
- 能指出分布式锁不适合解决所有并发问题。

## 重点

- 锁服务只能发锁，不能自动约束外部资源。
- 租约模型必须考虑客户端暂停。
- fencing token 要在资源写入点校验。

## 难点

- 看似互斥的锁，在网络分区和暂停下可能只提供有限保护。
- 续约线程成功不代表业务线程没有卡住。

## 易错

> **易错：** “Redis 锁加过期时间就安全了”。
>
> 正确做法：过期时间只防止死锁，不防止过期持有者继续写入；高风险写入需要 token 或条件更新。
