# 分布式锁与事务学习资料：生产选型、性能、安全和排障

[返回索引](../分布式锁与事务学习资料.md)

## 学习目标

- 能根据业务风险选择锁和事务方案。
- 掌握生产环境中的指标、日志、告警和排障路径。
- 理解性能、安全和运维风险。

## 选型原则

| 问题 | 优先方案 |
| --- | --- |
| 单表防重复 | 唯一索引 |
| 单资源余额或库存不为负 | 数据库原子更新或行锁 |
| 多实例定时任务防重复 | Redis 锁或调度平台锁，任务幂等 |
| 选主和元数据协调 | ZooKeeper/etcd |
| 短事务跨少量数据库强一致 | XA/2PC，谨慎评估 |
| 长业务流程最终一致 | Saga、TCC、Outbox、对账 |
| 外部支付和资金 | 幂等流水、回调去重、对账、人工兜底 |

## 性能关注点

- 锁粒度：全局锁吞吐最低，资源级锁更合理。
- 临界区长度：不要把慢 RPC 放进锁内。
- 热点 key：高并发抢同一个 Redis 锁会造成热点。
- 事务时长：长事务会持有数据库锁，阻塞其他请求。
- 消息积压：Outbox 发布器和消费者要有批量、限流和重试退避。
- 对账窗口：窗口太短容易误判，太长恢复慢。

## 安全和风险点

- 锁 key 不能直接拼接未校验用户输入，避免 key 污染和越权。
- 幂等键应绑定用户、业务类型和请求摘要，不能让攻击者复用他人幂等键。
- 管理后台补偿操作必须有权限、审批和审计。
- 死信消息可能包含敏感数据，日志和告警要脱敏。

## 排障剧本

### 锁竞争严重

检查：

- 锁 key 是否过粗。
- 临界区是否包含慢调用。
- TTL 是否频繁过期。
- 是否有任务重试风暴。

处理：

- 缩小锁粒度。
- 将远程调用移出临界区。
- 加入退避和限流。
- 用数据库状态机替代高频锁。

### Saga 卡住

检查：

- 当前 Saga 状态。
- 最后一次成功步骤。
- 失败原因和重试次数。
- 下游服务是否幂等。
- 是否可自动补偿。

处理：

- 重试可重入步骤。
- 对不可逆动作创建反向补偿。
- 进入人工处理队列。

### 消息重复或乱序

检查：

- 消息 ID 是否唯一。
- 消费者是否有 Inbox。
- 状态机是否允许旧事件覆盖新状态。
- 是否存在并发消费者处理同一聚合根。

处理：

- 增加去重表。
- 按聚合根路由消息。
- 使用版本号或事件序号。

## 练习

写一份“支付成功但订单未支付”的排障手册：

- 需要查哪些表。
- 需要查哪些日志和 trace。
- 自动修复和人工修复分别如何做。
- 如何避免同类问题再次发生。

## 验收

- 能给出方案选型理由和风险边界。
- 能列出锁和事务关键指标。
- 能设计告警、死信、补偿和人工处理流程。

## 重点

- 生产系统要关注可观测性和恢复闭环。
- 方案选型必须结合业务风险、吞吐、延迟和运维能力。

## 易错

> **易错：** 只设计成功路径，没有失败后的可观测和补偿入口。
>
> 正确做法：每个跨服务步骤都要有状态、日志、trace、重试、告警和人工处理策略。
