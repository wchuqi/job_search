# 分布式锁与事务学习资料：幂等、去重、状态机和对账

[返回索引](../分布式锁与事务学习资料.md)

## 学习目标

- 掌握幂等设计的常见手段。
- 理解状态机如何防止重复、乱序和迟到请求。
- 能设计对账任务发现并修复最终一致流程中的遗漏。

## 理论导读

分布式系统中，重复不是异常，而是常态。HTTP 重试、消息重投、任务重跑、用户重复点击、支付回调多次到达都会造成重复请求。幂等的目标是让同一个业务意图执行一次和执行多次的最终结果一致。

状态机用于限制状态只能按合法路径推进。对账用于发现自动流程遗漏、卡住和跨系统差异。

## 幂等手段

| 手段 | 适用场景 | 示例 |
| --- | --- | --- |
| 请求幂等键 | API 防重复提交 | `Idempotency-Key` |
| 唯一索引 | 防重复创建 | `payment_no` 唯一 |
| 去重表 | 消息消费 | `message_id + consumer` |
| 状态机条件更新 | 防重复流转 | `WHERE status='CREATED'` |
| 业务流水 | 外部支付和权益 | 一笔业务一条流水 |
| fencing token | 拒绝旧持锁者 | `last_token < token` |

## 状态机设计原则

- 所有状态要可枚举。
- 每个状态只允许有限动作。
- 每次状态变化写审计日志。
- 重复动作返回当前最终状态。
- 迟到动作不能覆盖新状态。

## 对账任务

对账不是失败后的补丁，而是最终一致架构的组成部分。对账任务应回答：

- 哪些流程超过 SLA 仍未完成。
- 哪些系统状态不一致。
- 哪些补偿重试多次失败。
- 哪些外部系统成功但本地未感知。
- 哪些记录需要人工介入。

## 例子

支付对账 SQL 思路：

```sql
SELECT p.payment_no, p.order_id, p.status AS payment_status, o.status AS order_status
FROM payment_order p
JOIN orders o ON p.order_id = o.id
WHERE p.status = 'SUCCESS'
  AND o.status <> 'PAID'
  AND p.updated_at < CURRENT_TIMESTAMP - INTERVAL '5 minutes';
```

发现后不要直接无脑更新，应进入修复流程：

1. 查询支付通道确认真实支付状态。
2. 检查订单是否已取消、退款或关闭。
3. 按状态机执行补偿动作。
4. 记录修复流水和操作审计。

## 练习

为“订单支付成功但权益未开通”设计对账：

- 对账周期。
- 查询条件。
- 自动修复动作。
- 失败重试上限。
- 人工处理字段和操作日志。

## 验收

- 能为 API、消息、任务、回调分别设计幂等键。
- 能用状态机条件更新防止乱序覆盖。
- 能写出对账任务的输入、输出和处理闭环。

## 重点

- 幂等要落到数据库或可靠存储，不能只靠内存缓存。
- 对账要有 SLA、告警、审计和人工兜底。

## 易错

> **易错：** 用 Redis 记录去重，过期后认为重复不会再来。
>
> 正确做法：关键业务去重应持久化，并根据业务追溯周期设置保留时间。
