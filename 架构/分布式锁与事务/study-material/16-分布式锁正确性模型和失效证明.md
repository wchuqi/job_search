# 分布式锁与事务学习资料：分布式锁正确性模型和失效证明

[返回索引](../分布式锁与事务学习资料.md)

## 学习目标

- 不只会说“加分布式锁”，而是能说明锁在什么故障模型下成立。
- 能用时间线证明 Redis、ZK、etcd 锁可能失效的边界。
- 能把锁的正确性转移到下游资源的版本校验和状态机上。

## 理论导读

分布式锁要讨论三个层次：

1. 锁服务是否只把锁授予一个客户端。
2. 客户端是否仍然有资格执行临界区。
3. 下游资源是否只接受最新、合法、幂等的写入。

很多方案只证明了第一层，忽略第二层和第三层。现实中即使锁服务没有明显 bug，客户端暂停、网络延迟、租约过期和下游乱序写入也会让业务出错。

## 正确性目标

| 目标 | 严格含义 | 工程实现难点 |
| --- | --- | --- |
| Safety 安全性 | 不会有两个有效持有者同时对资源产生冲突写入 | 租约过期后旧客户端仍可能写 |
| Liveness 活性 | 持锁者宕机后其他客户端最终能继续 | TTL 太长恢复慢，太短安全风险高 |
| Fencing 可隔离旧持有者 | 旧持有者恢复后不能覆盖新结果 | 下游必须校验单调 token |
| Observability 可观测 | 能知道谁持锁、持多久、为何失败 | 需要日志、指标、trace 和审计 |

> **重点：** 分布式锁的安全目标不是“代码块一定只有一个线程执行”，而是“资源最终不会被旧请求或并发请求破坏”。

## 基础故障模型

### 1. 进程暂停

```text
t0  A 获取锁，TTL=30s
t1  A 进入临界区
t2  A 被 STW、宿主机冻结或 CPU 饥饿，暂停 60s
t30 锁过期
t31 B 获取锁并完成写入
t62 A 恢复，继续执行旧写入
```

如果下游没有 token 或版本校验，A 的旧写入可能覆盖 B。此时“锁服务是否正确删除过期 key”已经不是关键，关键是客户端 A 在恢复后无法可靠知道自己是否仍有执行权。

### 2. 网络延迟

```text
A 获取锁成功响应在网络中延迟
A 业务侧超时重试，B 获取锁
旧响应到达 A，A 误以为自己刚拿到锁
```

客户端必须把请求生命周期、锁生命周期和业务执行生命周期分开，不能只凭“收到成功响应”就无限期执行。

### 3. 主从切换

Redis 主从复制如果是异步的，可能出现：

```text
A 在 master 写入 lock key 成功
master 尚未复制给 replica 就宕机
replica 提升为新 master
B 在新 master 获取同一把锁成功
A 和 B 都认为自己拿到锁
```

这不是 Lua 或释放校验能解决的问题。释放校验解决的是误删别人锁，不解决 failover 造成的双持有者。

## 证明一个锁方案是否足够

可以按这组问题审查：

1. 锁服务是否线性一致？
2. 锁授予是否有租约？租约由谁的时间判断？
3. 客户端暂停超过租约后会发生什么？
4. 续期失败时客户端是否立即停止业务？
5. 下游资源是否能拒绝旧持有者？
6. 重复执行是否幂等？
7. 锁服务 failover 后是否可能出现双持有？
8. 业务失败后是否有补偿和对账？

## fencing token 的正确性作用

锁服务发放的 token 必须单调递增，下游资源记录最大已接受 token：

```sql
UPDATE resource
SET owner = :owner,
    payload = :payload,
    last_token = :token
WHERE id = :resource_id
  AND last_token < :token;
```

如果影响行数为 0，说明当前请求是旧 token 或状态不允许，业务必须停止并记录异常。

> **难点：** token 不是为了证明“只有一个客户端执行”，而是为了证明“旧客户端即使执行也写不进去”。

## 锁粒度和临界区

锁粒度越粗，冲突越多；临界区越长，租约失效概率越大。

| 错误设计 | 风险 | 改进 |
| --- | --- | --- |
| 全局锁处理所有订单 | 吞吐低，热点严重 | 按订单 ID 或用户 ID 加锁 |
| 锁内调用支付、库存、通知多个 RPC | TTL 难设置，故障放大 | 锁内只做短本地状态推进 |
| 锁保护最终正确性 | 锁失效后数据错 | 数据库状态机和 token 兜底 |

## 练习

给出一个“锁服务正确但业务仍出错”的时间线，要求包含：

- 客户端 A 获取锁。
- A 暂停或网络延迟。
- B 获取锁并完成写入。
- A 恢复后旧写入。
- 用 SQL 条件说明如何拒绝 A。

## 验收

- 能区分锁服务互斥和业务资源互斥。
- 能用时间线解释进程暂停造成的旧持有者问题。
- 能写出下游 fencing token 校验条件。

## 易错

> **易错：** 把锁成功响应当作长期执行许可证。
>
> 正确做法：锁是有租约的短期许可，关键写入必须携带 token 或版本，并由下游资源判断是否接受。
