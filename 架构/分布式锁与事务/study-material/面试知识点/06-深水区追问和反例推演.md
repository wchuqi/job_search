## 一、深水区追问

### 1. 你说 Redis 锁能防并发，如果持锁线程 STW 超过 TTL 怎么办？

**参考答案：**

这时锁已经过期，其他客户端可能已经获得锁并完成写入。旧线程恢复后不能继续被信任。解决方式是缩短临界区、任务幂等，并让写下游资源时携带 fencing token 或版本号。下游只接受 token 更大的请求。

```sql
UPDATE resource
SET last_token = :token,
    value = :value
WHERE id = :id
  AND last_token < :token;
```

> **重点：** 不能只说“看门狗续期”，因为 STW 或网络异常时续期也可能停止。

### 2. Redlock 到底能不能用？

**参考答案：**

要按业务风险回答。Redlock 可以降低单 Redis 节点故障造成的锁丢失概率，适合重复执行代价可接受的任务。但它依赖时间窗口，不等同于共识协议。对资金、库存、权益等关键写入，必须加 fencing token、数据库约束、状态机和对账。

> **易错：** 绝对化回答“能”或“不能”。成熟回答要说明故障模型和业务风险。

### 3. 2PC 中协调器宕机后，参与者为什么不能直接回滚？

**参考答案：**

参与者 prepare 成功后，可能已有其他参与者收到 commit 并提交。如果某个参与者自行回滚，就破坏全局原子性。参与者处于 in-doubt 状态时应等待协调器恢复或通过恢复协议确认全局决议。长期无法恢复时才可能走启发式决策，并需要人工修复和审计。

> **难点：** 2PC 阻塞的核心就是 prepare 后参与者已承诺可提交但还不知道最终决议。

### 4. TCC Cancel 先到，Try 后到，你怎么防悬挂？

**参考答案：**

Cancel 找不到 Try 记录时不能直接返回成功，而要写一条 `CANCEL_BEFORE_TRY` 分支事务记录。迟到的 Try 执行前必须查询分支状态，发现已取消就拒绝预留资源。Confirm 和 Cancel 也都要按状态机保证幂等。

### 5. Outbox 发送成功但标记 PUBLISHED 前宕机，会不会重复？

**参考答案：**

会重复。Outbox 只能保证业务数据和事件记录同事务提交，不能保证只发布一次。发布器恢复后可能再次发送同一事件，所以消费者必须使用 Inbox 去重表、业务唯一键和状态机。

### 6. 支付成功但订单取消了，怎么处理？

**参考答案：**

不能简单把订单改成已支付。要先确认业务规则：如果订单取消发生在支付成功之前但支付回调迟到，通常需要发起退款或进入人工处理。处理时查询支付通道真实状态、本地支付流水和订单状态，按状态机执行补偿，并记录审计。

### 7. 如何证明你的最终一致方案能收敛？

**参考答案：**

要说明四点：

1. 每个本地状态变化由事务和约束保护。
2. 每个跨服务事件通过 Outbox 或事务消息可靠产生。
3. 每个消费者幂等，重复和乱序不会破坏状态。
4. 对账任务能发现长时间未完成或状态不一致的记录，并自动修复或人工兜底。

> **重点：** “能重试”不等于“能收敛”，还要有幂等、状态机和停止条件。
