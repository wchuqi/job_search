## 一、分布式锁

### 1. Redis 分布式锁如何正确实现？

**参考答案：**

获取锁使用 `SET key value NX PX ttl`，value 是唯一 requestId。释放锁时不能直接 `DEL`，必须用 Lua 校验 value 后删除，避免删除其他客户端新获得的锁。业务执行要有超时控制，临界区要短，关键写入还要用幂等、状态机或 fencing token 兜底。

```bash
SET lock:order:o1001 request-id NX PX 30000
```

> **重点：** 获取锁和设置过期时间必须是原子操作，释放锁必须校验持有者。

### 2. Redis 锁有哪些失效场景？

**参考答案：**

常见失效场景包括客户端执行超过 TTL、GC 或容器暂停、网络分区、Redis 主从切换导致锁丢失、续期失败、释放锁误删他人锁。最危险的是旧客户端以为自己还持有锁，恢复后继续写下游资源。

解决思路不是只把 TTL 调大，而是缩短临界区、业务幂等、状态机保护，并在高风险场景加入 fencing token。

> **易错：** 认为续期线程可以彻底解决锁过期。
>
> 正确做法：续期只能降低概率，不能替代下游防旧请求。

### 3. 什么是 fencing token？

**参考答案：**

fencing token 是每次获得执行权时生成的单调递增令牌。客户端写下游资源时携带 token，下游只接受比已记录 token 更大的请求。这样即使旧持锁者恢复后继续执行，也会因为 token 更小被拒绝。

```sql
UPDATE resource
SET value = 'new',
    last_token = 11
WHERE id = 'r1'
  AND last_token < 11;
```

> **重点：** fencing token 必须由下游资源校验，只在锁服务里生成 token 没有保护效果。

### 4. Redlock 能不能保证强一致？

**参考答案：**

Redlock 通过多个独立 Redis 节点和多数派获取锁，降低单节点故障风险。但它依赖时间窗口，不等同于共识协议。面对进程暂停、网络分区、时钟漂移和下游无 token 校验时，仍可能出现旧请求写入问题。

普通防重复任务可以考虑 Redis 锁或 Redlock；资金、库存、权益等关键场景不能只靠 Redlock，要结合数据库约束、fencing token、状态机和对账。
