## 学习目标

- 把幂等、状态机、对账从口号落到字段、SQL 和运行机制。
- 能处理重复、乱序、迟到、重放和人工修复。
- 能设计生产中可审计、可恢复、可解释的最终一致闭环。

## 幂等记录表

```sql
CREATE TABLE idempotency_record (
  idempotency_key VARCHAR(128) PRIMARY KEY,
  tenant_id VARCHAR(64) NOT NULL,
  user_id VARCHAR(64) NOT NULL,
  business_type VARCHAR(64) NOT NULL,
  request_hash VARCHAR(128) NOT NULL,
  status VARCHAR(32) NOT NULL,
  response_snapshot TEXT,
  created_at TIMESTAMP NOT NULL,
  updated_at TIMESTAMP NOT NULL
);
```

幂等键不能只由客户端随便传。服务端要绑定租户、用户、业务类型和请求摘要。相同 key 但请求内容不同，应返回冲突，防止错误复用或恶意重放。

## API 幂等流程

```text
1. 客户端传 Idempotency-Key
2. 服务端计算 request_hash
3. 尝试插入 idempotency_record
4. 插入成功，执行业务
5. 业务完成，保存响应快照
6. 重复请求到达，校验 request_hash
7. hash 相同返回已有结果，hash 不同返回冲突
```

## 状态机条件更新

状态机要防止迟到事件覆盖新状态：

```sql
UPDATE orders
SET status = 'PAID',
    version = version + 1
WHERE id = :order_id
  AND status = 'CREATED'
  AND version = :expected_version;
```

如果订单已经 `CANCELED` 或 `REFUNDED`，支付成功事件不能直接把它改成 `PAID`，必须进入异常处理或退款流程。

## 对账任务设计

对账任务不是简单 SQL 扫描。完整设计包括：

| 维度 | 内容 |
| --- | --- |
| 对账对象 | 支付单、订单、权益、库存、外部通道 |
| 对账窗口 | 排除刚发生且仍在传播中的记录 |
| 差异类型 | 漏处理、状态冲突、金额不一致、重复记录 |
| 修复动作 | 自动重试、补偿、退款、人工处理 |
| 审计 | 修复前状态、修复后状态、原因、操作人 |
| SLA | 多久必须收敛，超过多久告警 |

## 人工兜底表

```sql
CREATE TABLE manual_case (
  case_id VARCHAR(64) PRIMARY KEY,
  business_key VARCHAR(128) NOT NULL,
  case_type VARCHAR(64) NOT NULL,
  status VARCHAR(32) NOT NULL,
  severity VARCHAR(32) NOT NULL,
  snapshot TEXT NOT NULL,
  reason TEXT,
  assignee VARCHAR(64),
  created_at TIMESTAMP NOT NULL,
  updated_at TIMESTAMP NOT NULL
);
```

人工处理不是“直接改库”。后台操作应调用受控补偿接口，记录审批和审计。

## 支付对账推演

场景：支付通道显示成功，本地订单仍是 `CREATED`。

处理：

1. 查询本地支付流水是否 `SUCCESS`。
2. 如果支付流水不是成功，先根据通道结果修复支付流水。
3. 查询订单状态。
4. 如果订单仍 `CREATED`，按状态机推进到 `PAID` 并发权益事件。
5. 如果订单已 `CANCELED`，判断是否需要退款。
6. 如果权益未开通，重新发布或补偿开通事件。
7. 全部操作写审计。

## 练习

为“支付成功但权益未开通”写完整闭环：

- 差异查询 SQL。
- 自动修复动作。
- 什么情况不能自动修复。
- 人工 case 字段。
- 修复后如何验证。

## 验收

- 能设计幂等记录表，并解释 request_hash 的作用。
- 能用状态机条件更新处理乱序事件。
- 能设计对账和人工处理闭环。

## 易错

> **易错：** 对账任务发现不一致就直接 update 状态。
>
> 正确做法：先确认真实业务事实，再按状态机和补偿接口修复，并记录审计。
