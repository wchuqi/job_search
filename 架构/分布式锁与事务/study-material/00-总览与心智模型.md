# 分布式锁与事务学习资料：总览与心智模型

[返回索引](../分布式锁与事务学习资料.md)

## 学习目标

- 建立分布式锁与分布式事务的整体地图。
- 理解锁、事务、幂等、补偿、对账分别解决的问题。
- 能从业务不变量出发选择方案，而不是从中间件出发套方案。

## 理论导读

分布式系统里的正确性问题通常来自四类事实：网络会超时，进程会暂停或宕机，多个副本状态可能短暂不一致，外部系统不会跟你的数据库一起回滚。分布式锁试图让多个节点在某个时间窗口内只让一个执行者进入临界区；分布式事务试图让多个资源的状态变化在业务上保持一致。

两者都不是银弹。锁偏向“谁能做”，事务偏向“做成后状态是否一致”。真实业务还需要幂等防重复、状态机防乱序、补偿处理失败、对账发现遗漏、人工兜底处理不可自动恢复的异常。

## 核心心智模型

把分布式业务流程想成多个柜台办理一笔业务：

- 锁：发号器，尽量避免多个人同时办理同一个资源。
- 本地事务：单个柜台内部记账要么全写成，要么全不写。
- 分布式事务：多个柜台尝试按同一张流程单协作完成。
- 幂等：同一张凭证重复提交，只处理一次。
- 补偿：已经发生的外部动作不能撤销时，做反向业务动作修正。
- 对账：定期核对多个柜台的账本，发现漏单、错单和卡住的流程。

## 知识覆盖图

| 领域 | 必学内容 | 关键问题 |
| --- | --- | --- |
| 基础一致性 | ACID、隔离级别、CAP、BASE、最终一致 | 什么必须立刻一致，什么允许延迟一致 |
| 本地并发控制 | 行锁、唯一索引、乐观锁、版本号、状态机 | 单库内如何避免重复和并发覆盖 |
| 分布式锁 | 租约、TTL、续期、释放校验、锁服务高可用 | 锁失效时是否仍能保护正确性 |
| Redis 锁 | `SET NX PX`、Lua、Redlock、主从切换 | Redis 锁适合哪些风险等级 |
| ZK/etcd 锁 | 会话、临时节点、lease、revision、watch | 如何获得更强的会话和顺序语义 |
| 分布式事务 | 2PC、3PC、XA、事务协调器 | 强一致代价是什么 |
| 最终一致方案 | Saga、TCC、Outbox、Inbox、事务消息 | 如何让流程失败后可恢复 |
| 业务兜底 | 幂等、去重、补偿、对账、人工处理 | 生产异常如何闭环 |
| 运维排障 | 指标、日志、trace、死信、告警、审计 | 如何定位卡住和错乱的流程 |

## 例子

订单支付流程的正确性不是一个锁能解决的：

1. 订单只能从 `CREATED` 流转到 `PAID` 一次，需要数据库状态机和幂等键。
2. 库存不能超卖，需要库存服务本地事务、预留记录或乐观扣减。
3. 支付回调可能重复，需要支付流水唯一约束和回调去重。
4. 支付成功但订单更新失败，需要补偿任务或对账任务修复。
5. 多个任务同时处理同一订单，可以用分布式锁降低冲突，但正确性仍要靠状态机。

## 练习

列出“用户购买会员”的业务不变量：

- 同一支付单不能重复扣款。
- 同一订单不能重复开通权益。
- 支付成功后最终必须有权益或退款。
- 权益开通失败必须可重试或可人工处理。

为每条不变量写出保护方式：本地事务、唯一约束、锁、消息、补偿或对账。

## 验收

- 能说清楚锁和事务的边界。
- 能把一个跨服务流程拆成局部强一致、流程最终一致和异常恢复。
- 能为每个关键状态设计幂等键和状态机。

## 重点

- 先识别业务不变量，再选技术方案。
- 分布式锁减少并发冲突，但不能替代数据库约束和幂等。
- 分布式事务不是唯一选择，很多业务更适合 Saga、TCC、Outbox 和对账。

## 难点

- 分布式系统没有可靠的“我一定还持有锁”的直觉。TTL 过期、进程暂停、网络分区都会破坏这种直觉。
- 最终一致不是放弃正确性，而是把正确性分散到状态机、补偿、重试和对账。

## 易错

> **易错：** 认为“加了分布式锁就不会并发错乱”。
>
> 正确做法：锁只作为协调层，数据库约束、幂等、状态机和 fencing token 才是关键保护层。
