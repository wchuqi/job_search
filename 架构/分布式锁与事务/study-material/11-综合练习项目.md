# 分布式锁与事务学习资料：综合练习项目

## 项目：课程购买与会员开通一致性设计

## 目标

设计一个包含订单、库存、支付、会员权益、通知的后端流程，要求支持重复请求、支付回调重复、消息重复、库存预留超时、权益开通失败和对账修复。

## 业务流程

1. 用户创建订单。
2. 库存服务预留课程名额或活动名额。
3. 支付服务创建支付单。
4. 支付通道回调支付结果。
5. 订单服务标记已支付。
6. 权益服务开通会员或课程。
7. 通知服务发送站内信或短信。

## 必做设计

- 业务不变量清单。
- 数据表草图。
- 状态机图。
- 幂等键设计。
- Saga 或 TCC 流程。
- Outbox/Inbox 可靠消息。
- 对账任务。
- 死信和人工处理后台。
- 关键指标和告警。

## 推荐表结构

```sql
CREATE TABLE orders (
  id VARCHAR(64) PRIMARY KEY,
  user_id VARCHAR(64) NOT NULL,
  status VARCHAR(32) NOT NULL,
  amount NUMERIC(18,2) NOT NULL,
  version INT NOT NULL DEFAULT 0,
  created_at TIMESTAMP NOT NULL,
  updated_at TIMESTAMP NOT NULL
);

CREATE TABLE payment_order (
  payment_no VARCHAR(64) PRIMARY KEY,
  order_id VARCHAR(64) NOT NULL,
  status VARCHAR(32) NOT NULL,
  channel_trade_no VARCHAR(128),
  amount NUMERIC(18,2) NOT NULL,
  created_at TIMESTAMP NOT NULL,
  updated_at TIMESTAMP NOT NULL
);

CREATE TABLE idempotency_record (
  idempotency_key VARCHAR(128) PRIMARY KEY,
  user_id VARCHAR(64) NOT NULL,
  request_hash VARCHAR(128) NOT NULL,
  response_snapshot TEXT,
  status VARCHAR(32) NOT NULL,
  created_at TIMESTAMP NOT NULL
);
```

## 故障注入任务

- 支付回调重复 10 次。
- 权益服务开通成功后响应超时。
- Outbox 消息发送成功但发布器在标记成功前宕机。
- 库存 Try 成功后 Confirm 长时间未到。
- Redis 锁过期后旧任务继续执行。

## 验收标准

- 重复请求不会重复创建订单或重复扣款。
- 支付成功后最终订单和权益一致，或进入可解释的人工处理状态。
- 消息重复不会重复开通权益。
- 补偿失败会进入告警和人工处理队列。
- 所有关键状态变化有审计日志。

## 重点

- 练习时必须写失败路径，不能只画成功流程。
- 每个外部动作都要有业务流水和幂等键。

## 易错

> **易错：** 把通知失败当成订单失败。
>
> 正确做法：通知通常是非核心副作用，应异步重试，不应回滚支付和权益主流程。
