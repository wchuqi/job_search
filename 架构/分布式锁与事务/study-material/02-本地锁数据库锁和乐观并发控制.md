# 分布式锁与事务学习资料：本地锁、数据库锁和乐观并发控制

[返回索引](../分布式锁与事务学习资料.md)

## 学习目标

- 区分 JVM 本地锁、数据库锁、乐观锁和分布式锁。
- 掌握悲观锁、乐观锁、唯一约束的适用场景。
- 能避免把本地并发问题误判为分布式问题。

## 理论导读

锁的本质是控制临界区。临界区可以在一个进程内、一个数据库内，也可以跨多个进程。作用范围越大，成本和失败场景越复杂。因此选型原则是：能用数据库约束解决，就不要升级成分布式锁；能用单服务本地事务解决，就不要引入跨服务事务。

## 锁和并发控制对比

| 工具 | 作用范围 | 适合场景 | 主要风险 |
| --- | --- | --- | --- |
| JVM 锁 | 单进程 | 内存对象、单实例任务 | 多实例无效 |
| 数据库行锁 | 单数据库 | 更新同一行资源 | 长事务阻塞 |
| 乐观锁 | 单数据库或单资源 | 冲突少、读多写少 | 冲突重试复杂 |
| 唯一约束 | 单数据库 | 防重复创建 | 需要处理冲突异常 |
| 分布式锁 | 多进程/多节点 | 跨实例协调临界区 | TTL、网络、锁服务故障 |

## 数据库悲观锁

```sql
BEGIN;

SELECT *
FROM coupon
WHERE id = 'c1001'
FOR UPDATE;

UPDATE coupon
SET status = 'USED'
WHERE id = 'c1001'
  AND status = 'UNUSED';

COMMIT;
```

悲观锁适合冲突概率高、临界区短、资源明确的场景。它不适合包住慢接口、远程调用或长时间业务流程。

## 乐观锁和版本号

乐观锁假设冲突不常发生，用版本号或更新时间检测并发修改：

```sql
UPDATE orders
SET status = 'PAID',
    version = version + 1
WHERE id = 'o1001'
  AND status = 'CREATED'
  AND version = 3;
```

影响行数为 1 才表示成功。影响行数为 0 时，应用要重新读取状态并决定重试、忽略或报错。

## 状态机是并发控制的一部分

订单状态不能随意更新，而应定义允许的转移：

| 当前状态 | 允许动作 | 下一状态 |
| --- | --- | --- |
| CREATED | 支付成功 | PAID |
| CREATED | 用户取消 | CANCELED |
| PAID | 发货 | SHIPPED |
| PAID | 退款 | REFUNDING |

状态机能防止迟到消息把状态改回旧值。

## 例子

支付回调重复到达时，不需要先抢分布式锁：

```sql
UPDATE payment_order
SET status = 'SUCCESS',
    paid_at = CURRENT_TIMESTAMP
WHERE payment_no = 'p1001'
  AND status = 'PROCESSING';
```

如果第二次回调影响行数为 0，说明已经处理过，返回成功即可。

## 练习

为“优惠券核销”设计并发控制：

- 同一张券只能核销一次。
- 核销成功后不能再次核销。
- 支付失败时是否允许恢复未使用状态。
- 回调重复时如何幂等。

## 验收

- 能解释本地锁为什么在多实例部署下无效。
- 能用 `UPDATE ... WHERE status = ?` 表达状态机保护。
- 能判断悲观锁和乐观锁的取舍。

## 重点

- 数据库状态机和唯一约束通常比分布式锁更可靠。
- 锁不要包住远程调用，否则阻塞、死锁和超时都会放大。

## 易错

> **易错：** 在应用层 `synchronized` 包住下单逻辑，以为部署多个实例后仍有效。
>
> 正确做法：本地锁只保护单进程；多实例必须依靠数据库约束、分布式锁或服务端原子操作。
