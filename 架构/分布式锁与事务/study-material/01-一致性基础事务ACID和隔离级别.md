# 分布式锁与事务学习资料：一致性基础、事务 ACID 和隔离级别

[返回索引](../分布式锁与事务学习资料.md)

## 学习目标

- 理解事务 ACID 的真实含义。
- 掌握常见隔离级别和并发异常。
- 能判断哪些一致性需求必须在单库内强约束。

## 理论导读

事务是数据库提供的局部正确性工具。它保证一组操作在一个资源管理器内满足原子性、一致性、隔离性和持久性。分布式事务之所以复杂，是因为多个资源管理器无法天然共享同一个提交点。

ACID 中的一致性不是“数据库自动理解业务正确”，而是数据库在约束、触发器、外键、唯一索引和事务逻辑配合下，从一个合法状态进入另一个合法状态。业务约束设计不完整时，事务也会提交错误数据。

## 核心概念

| 概念 | 含义 | 常见误解 |
| --- | --- | --- |
| Atomicity 原子性 | 事务内操作整体成功或失败 | 误以为能自动覆盖外部 API |
| Consistency 一致性 | 约束不被破坏 | 误以为数据库知道业务规则 |
| Isolation 隔离性 | 并发事务之间的可见和干扰规则 | 误以为默认隔离足够所有场景 |
| Durability 持久性 | 提交后即使崩溃也能恢复 | 误以为消息和缓存也一起持久 |

## 隔离级别和并发异常

| 隔离级别 | 可能问题 | 适合场景 |
| --- | --- | --- |
| Read Uncommitted | 脏读、不可重复读、幻读 | 很少用于核心业务 |
| Read Committed | 不可重复读、幻读、写偏斜风险 | 常见默认级别 |
| Repeatable Read | 幻读或写偏斜取决于数据库实现 | 多数 OLTP 足够 |
| Serializable | 并发效果等价串行 | 正确性强，冲突和性能代价高 |

### 典型异常

- 脏读：读到未提交数据。
- 不可重复读：同一事务内两次读同一行结果不同。
- 幻读：同一范围查询前后出现新行。
- 丢失更新：两个事务读同一值后分别写回，后者覆盖前者。
- 写偏斜：两个事务各自看到条件满足，分别修改不同记录，最终整体约束被破坏。

## 例子

用乐观锁避免丢失更新：

```sql
UPDATE account
SET balance = balance - 100,
    version = version + 1
WHERE id = 'A'
  AND version = 7
  AND balance >= 100;
```

如果影响行数为 0，说明余额不足或版本已变化，应用必须重试或返回失败。

## 实际场景

库存扣减不一定需要分布式锁。单个库存服务拥有库存表时，可以用数据库原子更新保护库存不为负：

```sql
UPDATE sku_stock
SET available = available - 1
WHERE sku_id = 'sku-1'
  AND available >= 1;
```

这比“所有请求先抢 Redis 锁再扣库存”更直接，因为不变量在数据所有者内部被保护。

## 练习

设计“一个用户对一个职位只能投递一次”的表结构：

- 使用唯一索引保护 `user_id + job_id`。
- 使用状态字段表示 `SUBMITTED`、`WITHDRAWN`、`REJECTED`。
- 思考撤回后是否允许再次投递，如果允许，唯一约束应该如何调整。

## 验收

- 能解释 ACID 每个字母的边界。
- 能举例说明丢失更新和写偏斜。
- 能使用唯一约束、状态机和原子更新保护单库业务不变量。

## 重点

- 单资源不变量优先放在数据所有者内部保护。
- 数据库约束是最后防线，应用层检查只是体验优化。

## 难点

- 写偏斜常发生在“读一组记录判断条件，再写另一组记录”的场景，普通行锁不一定保护整体约束。

## 易错

> **易错：** 先查询“有没有记录”，再插入，认为这样能防重复。
>
> 正确做法：必须使用唯一索引，并正确处理唯一冲突。
