# 分布式锁与事务学习资料：Saga、TCC 和补偿事务

[返回索引](../分布式锁与事务学习资料.md)

## 学习目标

- 掌握 Saga 和 TCC 的核心模型。
- 理解补偿、空回滚、悬挂、幂等的工程要求。
- 能为跨服务业务流程设计状态机和恢复路径。

## 理论导读

Saga 把一个长事务拆成多个本地事务，每一步成功后推进流程，失败时执行补偿动作。TCC 把一个动作拆成 Try、Confirm、Cancel 三个阶段，先预留资源，再确认或取消。它们都不依赖所有参与者长时间持有数据库锁，更适合业务流程长、参与服务多、允许最终一致的场景。

补偿不是数据库回滚。很多动作已经对外发生，例如发券、扣款、发货、开通权益，只能通过反向业务动作修正。

## Saga

| 类型 | 工作方式 | 优点 | 风险 |
| --- | --- | --- | --- |
| 编排式 Saga | 中央协调器调用各服务并记录状态 | 流程清晰，便于排障 | 协调器可能承载过多业务 |
| 协同式 Saga | 服务通过事件相互触发 | 松耦合，扩展灵活 | 全局流程分散，排障困难 |

Saga 每一步都需要：

- 本地事务。
- 幂等键。
- 明确状态。
- 可重试策略。
- 失败补偿。
- 超时和对账。

## TCC

| 阶段 | 作用 | 示例 |
| --- | --- | --- |
| Try | 检查并预留资源 | 冻结库存、冻结余额 |
| Confirm | 确认使用资源 | 扣减冻结库存、完成扣款 |
| Cancel | 取消预留资源 | 解冻库存、解冻余额 |

TCC 的三大高频问题：

- 空回滚：Try 未成功，Cancel 先到。Cancel 必须能识别并安全返回。
- 悬挂：Cancel 已执行，迟到的 Try 又成功。Try 必须检查全局事务状态。
- 幂等：Try、Confirm、Cancel 都可能重复调用。

## 例子

库存 TCC 表可以包含：

```sql
CREATE TABLE stock_reservation (
  xid VARCHAR(64) PRIMARY KEY,
  sku_id VARCHAR(64) NOT NULL,
  quantity INT NOT NULL,
  status VARCHAR(32) NOT NULL,
  created_at TIMESTAMP NOT NULL,
  updated_at TIMESTAMP NOT NULL
);
```

Try 阶段插入 `TRYING` 记录并扣减可用库存、增加冻结库存；Confirm 把冻结库存转为已售；Cancel 释放冻结库存。

## Saga 订单例子

1. 订单服务创建订单 `PENDING`。
2. 库存服务预留库存。
3. 支付服务创建支付单并等待支付成功。
4. 权益服务开通权益。
5. 订单服务标记完成。
6. 任一步失败，按已完成步骤逆序补偿。

## 练习

分别用 Saga 和 TCC 设计“酒店预订 + 优惠券 + 支付”：

- 哪些资源需要 Try 预留。
- 哪些动作只能补偿。
- 超时未支付如何释放资源。
- 支付成功但订单状态未更新如何对账。

## 验收

- 能区分 Saga 和 TCC。
- 能解释空回滚、悬挂和幂等。
- 能画出正常路径、失败路径、补偿路径和超时路径。

## 重点

- TCC 要求业务资源支持预留模型。
- Saga 的核心是状态机和补偿，不是简单异步调用。

## 易错

> **易错：** 认为补偿就是把数据库状态改回去。
>
> 正确做法：补偿是业务动作，必须考虑外部影响、审计记录、用户可见状态和不可逆动作。
