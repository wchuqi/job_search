# 分布式锁与事务学习资料：架构选型决策树和事故复盘深度训练

[返回索引](../分布式锁与事务学习资料.md)

## 学习目标

- 能从业务不变量推导技术方案，而不是背方案。
- 能用决策树快速判断锁、事务、消息和补偿的组合。
- 能写出有工程价值的事故复盘。

## 决策树

### 1. 不变量是否只涉及一个数据库资源？

是：优先使用本地事务、唯一索引、行锁、乐观锁、状态机。

否：继续判断。

### 2. 是否必须同步强一致？

是：判断参与方是否都支持短事务和 XA。如果参与方可控、事务短、吞吐可接受，可考虑 XA/2PC；否则需要重新设计业务边界。

否：继续判断。

### 3. 是否存在可预留资源？

是：考虑 TCC，例如库存冻结、余额冻结、名额预留。

否：考虑 Saga、Outbox、补偿和对账。

### 4. 是否只是多实例防重复执行？

是：Redis 锁、ZK/etcd 选主或调度平台锁都可以，但任务必须幂等。

否：不要把锁当一致性方案，回到不变量分析。

### 5. 是否涉及外部系统？

是：必须有业务流水、幂等回调、对账、补偿和人工兜底。外部系统通常不能纳入本地事务或 XA。

## 方案组合示例

| 业务 | 推荐组合 |
| --- | --- |
| 防重复投递简历 | 唯一索引 + 状态机 |
| 秒杀扣库存 | Redis 预热/限流 + 库存服务原子扣减 + 订单幂等 |
| 订单支付开通权益 | 支付流水 + Outbox + Inbox + 状态机 + 对账 |
| 酒店预订 | TCC 预留房源 + Saga 支付 + 超时取消 |
| 主调度器执行任务 | etcd 选主 + revision token + 任务幂等 |
| 跨库短事务 | XA/2PC + 事务日志监控 + 超时处理 |

## 深度训练题

### 题 1：支付重复回调 100 次

要求证明：

- 不会重复扣款。
- 不会重复开通权益。
- 重复消息会被 Inbox 拦截。
- 状态机不会倒退。

### 题 2：锁过期后旧任务继续执行

要求证明：

- 旧任务写入会被 token 拒绝。
- 即使 token 没拒绝，状态机也不会重复推进。
- 对账能发现异常状态。

### 题 3：Outbox 发布器宕机

要求证明：

- 业务状态和 outbox 事件已经同事务提交。
- 发布器恢复后能继续发布。
- 重复发布不会导致重复消费。

## 验收

- 能根据决策树解释为什么不用某个热门方案。
- 能在事故复盘中指出“现有保护为什么没挡住”。
- 能为每个方案写出失败证明和恢复路径。

## 易错

> **易错：** 面试回答只给方案名，不给失败路径。
>
> 正确做法：每个方案至少说明一个失败场景、一个兜底机制和一个监控指标。
