# 分布式锁与事务学习路线图

## 阶段 1：基础认知

- 目标：理解为什么单机锁、本地事务和数据库约束进入分布式场景后会失效或边界变窄。
- 需要掌握：ACID、隔离级别、并发异常、悲观锁、乐观锁、唯一约束、业务不变量。
- 例子：同一用户重复下单、库存超卖、重复支付回调。
- 练习：用一张表设计“订单只能支付一次”的约束和状态流转。
- 验收：能说明“加锁”和“事务”分别保护什么，不能保护什么。
- 重点：先用本地事务和数据库约束保护单资源不变量。
- 易错：一上来就用分布式锁解决所有并发问题。

## 阶段 2：分布式锁核心能力

- 目标：掌握租约锁、TTL、续期、释放校验、fencing token 和锁服务选择。
- 需要掌握：Redis `SET NX PX`、Lua 释放、Redlock 边界、ZooKeeper 临时顺序节点、etcd lease 和 revision。
- 例子：定时任务只允许一个实例执行，订单关闭任务防重复执行。
- 练习：设计一个带 requestId、TTL、续期、超时和幂等保护的锁执行模板。
- 验收：能解释锁过期后旧持有者继续写入为什么危险，以及 fencing token 如何兜底。
- 重点：分布式锁通常是协调手段，不应是最后一道正确性防线。
- 易错：只校验拿锁，不校验释放锁，也不保护下游资源。

## 阶段 3：分布式事务核心能力

- 目标：掌握 XA/2PC、Saga、TCC、可靠消息、Outbox/Inbox 的适用边界。
- 需要掌握：全局事务协调器、prepare/commit、阻塞问题、补偿、空回滚、悬挂、幂等、事务消息。
- 例子：订单创建、库存预留、支付扣款、权益开通、通知发送。
- 练习：分别用 Saga、TCC、Outbox 设计“购买课程”流程。
- 验收：能给出每种方案的失败恢复路径，而不是只画正常流程。
- 重点：跨服务一致性是业务流程设计问题，不只是中间件问题。
- 易错：把补偿理解为数据库 rollback。

## 阶段 4：生产应用和排障

- 目标：能为真实业务选择方案，并对重复、丢失、乱序、超时、脑裂、补偿失败做预案。
- 需要掌握：幂等键、去重表、状态机、对账任务、死信队列、告警、审计、人工处理后台。
- 例子：支付成功但订单未更新、库存预留成功但订单取消、消息发出但本地事务回滚。
- 练习：写一份事故推演：如果消息重复投递 100 次，系统是否仍正确？
- 验收：能产出方案选型表、状态机图、补偿表、对账 SQL 和故障处理手册。
- 重点：生产系统靠“可恢复”获得最终正确。
- 易错：只考虑正常路径，不设计观测、补偿和人工兜底。

## 推荐学习节奏

| 周期 | 学习重点 | 产出 |
| --- | --- | --- |
| 第 1 周 | 事务、锁、隔离级别、数据库约束 | 并发异常案例和 SQL 约束设计 |
| 第 2 周 | Redis/ZooKeeper/etcd 分布式锁 | 锁模板、失效场景清单、fencing token 方案 |
| 第 3 周 | 2PC、XA、Saga、TCC | 订单流程事务方案对比 |
| 第 4 周 | Outbox、幂等、对账、排障 | 可恢复业务流程设计文档 |

## 最终能力清单

- 能判断一个问题应该用唯一约束、本地事务、乐观锁、分布式锁、消息最终一致还是 TCC。
- 能设计锁获取、释放、续期、超时、fencing token 和下游拒绝旧请求。
- 能画出 Saga/TCC 正常路径、失败路径、补偿路径和对账路径。
- 能解释 2PC 的阻塞、单点、资源占用和启发式提交风险。
- 能设计幂等消费、去重表、状态机流转、死信处理和人工补偿入口。
