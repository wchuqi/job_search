# RESTful API学习路线图

## 阶段 1：HTTP和REST基础认知

- 目标：理解 REST 不是 URL 命名风格，而是一组围绕资源、表示、统一接口和无状态交互的架构约束。
- 需要掌握：客户端/服务端、资源、表示、URI、HTTP 方法、状态码、请求头、响应头、JSON、幂等、安全方法。
- 例子：把订单系统抽象为 `/orders`、`/orders/{id}`、`/orders/{id}/items` 等资源，而不是只设计 `/createOrder`、`/queryOrder`。
- 练习：为用户、订单、商品三个领域对象画出资源关系图。
- 验收：能解释 `GET`、`POST`、`PUT`、`PATCH`、`DELETE` 的语义差异，并能判断一个接口是否符合 HTTP 语义。
- 重点：资源建模、HTTP 方法语义、状态码语义。
- 易错：把 RESTful API 等同于“路径不用动词”；路径只是表象，核心是统一接口和协议语义。

## 阶段 2：资源建模和接口设计

- 目标：能够设计清晰、稳定、可演进的 API。
- 需要掌握：集合资源、单个资源、子资源、查询参数、分页、过滤、排序、批量操作、异步任务资源。
- 例子：`GET /orders?status=PAID&page=1&pageSize=20&sort=-createdAt` 表示查询订单集合，不应该设计成 `POST /orders/searchPaidOrders`。
- 练习：设计一套博客系统 API，包括文章、评论、标签、草稿、发布流程。
- 验收：接口命名统一，能解释每个路径为什么是资源，能说明查询参数和路径参数的边界。
- 重点：URI 表达资源层级，查询参数表达资源集合的筛选视图。
- 易错：为了“纯 REST”过度嵌套路径，导致 `/users/{uid}/orders/{oid}/items/{iid}/...` 难以维护。

## 阶段 3：请求响应、错误和安全

- 目标：让 API 对调用方稳定、可诊断、可保护。
- 需要掌握：请求体设计、响应体设计、错误模型、Problem Details、认证、授权、CORS、输入校验、敏感信息保护。
- 例子：认证失败返回 `401`，权限不足返回 `403`，资源不存在返回 `404`，业务冲突返回 `409`。
- 练习：为订单创建接口设计成功响应、参数错误响应、库存不足响应、重复提交响应。
- 验收：错误响应包含错误码、可读信息、字段错误和追踪 ID，且不会泄露堆栈或密钥。
- 重点：状态码表达通用结果，业务错误码表达业务细节。
- 难点：既让前端易处理，又不把内部异常模型暴露给外部。

## 阶段 4：缓存、并发、版本和可观测

- 目标：让 API 在生产环境中稳定演进。
- 需要掌握：HTTP 缓存、ETag、条件请求、乐观锁、API 版本、兼容性、限流、日志、指标、链路追踪、契约测试。
- 例子：更新资料时使用 `If-Match: "<etag>"`，避免两个客户端互相覆盖修改。
- 练习：为商品详情接口增加缓存头，为用户资料修改接口增加 ETag 并发控制。
- 验收：能解释缓存失效、并发冲突、版本兼容、灰度发布和问题定位方案。
- 重点：向后兼容优先；只有破坏性变更才需要新版本。
- 易错：随意修改字段含义、删除字段或改变状态码，导致调用方线上故障。

## 推荐学习节奏

| 周期 | 学习重点 | 产出 |
| --- | --- | --- |
| 第 1-2 天 | HTTP、REST、资源建模 | 资源关系图和基础 API 清单 |
| 第 3-4 天 | 方法语义、状态码、请求响应 | 一套 CRUD API 草案 |
| 第 5-6 天 | 安全、错误、分页、过滤 | 可被前端调用的接口契约 |
| 第 7-8 天 | 缓存、并发、版本、文档 | OpenAPI 文档和测试用例 |
| 第 9-10 天 | 网关、限流、观测、排障 | 生产化检查清单 |

## 最终能力清单

- 能从业务模型抽象资源模型。
- 能选择正确的 HTTP 方法和状态码。
- 能设计一致的 URI、请求体、响应体和错误结构。
- 能处理分页、过滤、排序、批量、异步任务等常见场景。
- 能说明幂等、安全、缓存、条件请求和并发控制。
- 能设计认证授权、CORS、防重放、限流和审计方案。
- 能用 OpenAPI 描述契约，并通过契约测试降低联调风险。
- 能在面试中解释 REST 与 RPC、GraphQL、WebSocket 的边界。
