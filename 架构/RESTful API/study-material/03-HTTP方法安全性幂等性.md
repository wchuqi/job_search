# RESTful API学习资料：HTTP方法安全性幂等性

[返回索引](../RESTful API学习资料.md)

## 学习目标

- 掌握常用 HTTP 方法的语义。
- 理解安全性和幂等性的区别。
- 能设计可重试、防重复提交的 API。

## 理论导读

HTTP 方法不是随便选的动作标签，它们携带协议层语义。客户端、浏览器、网关、缓存、监控和重试组件都可能根据方法做决策。如果所有接口都用 `POST`，系统会丢失一大部分协议能力。

安全方法指不会改变服务端资源状态的方法。幂等方法指执行一次和执行多次的最终效果相同。安全关注“是否修改”，幂等关注“重复执行是否造成额外影响”。

## 方法语义表

| 方法 | 常见用途 | 安全 | 幂等 | 请求体 | 常见成功状态码 |
| --- | --- | --- | --- | --- | --- |
| `GET` | 查询资源 | 是 | 是 | 通常不用 | `200`、`304` |
| `HEAD` | 只查询响应头 | 是 | 是 | 不用 | `200`、`304` |
| `POST` | 创建、提交、触发非幂等处理 | 否 | 通常否 | 常用 | `201`、`202`、`200` |
| `PUT` | 整体替换或指定 ID 创建 | 否 | 是 | 常用 | `200`、`201`、`204` |
| `PATCH` | 局部修改 | 否 | 取决于补丁语义 | 常用 | `200`、`204` |
| `DELETE` | 删除资源 | 否 | 是 | 少用 | `204`、`202` |

## 核心心智模型

幂等像“设置空调到 26 度”：执行一次和执行多次结果一样。非幂等像“转账 100 元”：重复执行会重复扣款。

## 知识点详解

### `POST` 与 `PUT`

`POST /orders` 表示向订单集合提交创建请求，服务端通常生成 ID。`PUT /orders/1001` 表示把 `1001` 这个资源替换成请求体中的完整表示，客户端知道资源地址。

### `PATCH` 的风险

`PATCH` 是局部修改，但是否幂等取决于补丁格式。设置字段通常幂等，增量操作通常不幂等。

```http
PATCH /products/p1
Content-Type: application/json

{
  "price": 19900
}
```

这个补丁趋向幂等。

```json
{
  "stockIncrement": -1
}
```

这个补丁重复执行会多扣库存，不幂等。

### 幂等键

对支付、下单、发券等高风险 `POST` 操作，可以使用幂等键：

```http
POST /payments
Idempotency-Key: 8fd777a2-4d79-4e8b-8c5e-1a5b7d8b8f01
Content-Type: application/json

{
  "orderId": "1001",
  "amount": 29900
}
```

服务端应记录幂等键、请求摘要和处理结果。相同幂等键和相同请求返回同一结果；相同键但请求内容不同应拒绝。

## 练习

为“创建支付单、取消支付单、查询支付单、退款、查询退款结果”选择 HTTP 方法，并说明哪些操作必须支持幂等。

## 验收

- 能准确区分安全和幂等。
- 能解释为什么 `GET` 不应该修改状态。
- 能设计幂等键存储和冲突处理逻辑。

> **重点：** 幂等性是 API 可重试的基础，重试是分布式系统中绕不开的现实。
>
> **难点：** `PATCH` 和 `POST` 是否幂等取决于业务和实现，不要只背方法表。
>
> **易错：** 用 `GET /sendEmail` 触发发邮件，可能被浏览器预取、爬虫、缓存或监控误触发。
