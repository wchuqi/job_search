# RESTful API学习资料：资源建模决策树和反模式深度解析

[返回索引](../RESTful API学习资料.md)

## 学习目标

- 掌握从业务流程抽象资源的决策树。
- 能处理动作、关系、状态、搜索、批量、异步任务等复杂场景。
- 能识别常见资源建模反模式并给出修正方案。

## 理论导读

资源建模不是把数据库表搬到 URL 上，也不是把方法名改成名词。资源是调用方可理解、可定位、可操作的业务对象或业务结果。一个好的资源模型要满足三个条件：语义稳定、权限边界清楚、生命周期可解释。

## 资源建模决策树

### 第一步：这个概念是否可命名

如果业务中有稳定名词，例如订单、支付单、退款申请、审批单、导入任务，通常可以成为资源。

如果只是临时动作，例如“重新发送一次通知”，先判断它是否需要记录结果。如果需要记录，就可以建模为通知发送任务；如果不需要记录，命令式接口可能更合适。

### 第二步：是否有独立生命周期

有独立生命周期的对象适合成为顶级资源：

- 支付单可以独立查询、失败、重试、关闭。
- 退款单可以独立审核、打款、失败。
- 导入任务可以排队、运行、成功、失败。

没有独立生命周期的对象更适合作为子资源或字段：

- 订单收货地址快照通常附属于订单。
- 订单项通常依赖订单存在。

### 第三步：权限是否独立

如果某对象的访问权限和父对象不同，要谨慎深嵌套。比如客服可以看售后单，但不一定能看用户全部资料；售后单适合作为独立资源。

### 第四步：查询入口是否独立

如果需要跨父资源查询，应该提供集合入口：

```http
GET /order-items?productId=p1
```

而不是只提供：

```http
GET /orders/{orderId}/items
```

### 第五步：动作是否可转化为状态转移或动作资源

取消订单可以是：

- 修改订单状态：`PATCH /orders/{id}`
- 创建取消申请：`POST /orders/{id}/cancellation-requests`
- 创建取消任务：`POST /order-cancellation-jobs`

选择依据是业务复杂度、是否异步、是否需要审核、是否需要记录失败原因。

## 常见复杂资源模式

### 关系资源

用户加入团队不是简单把 userId 塞到团队里。如果成员关系有角色、加入时间、邀请人、状态，就应建模为资源。

```http
POST /team-memberships
Content-Type: application/json

{
  "teamId": "t1",
  "userId": "u1",
  "role": "ADMIN"
}
```

### 状态资源

当状态变化有复杂规则，可以暴露状态转移入口：

```http
POST /orders/1001/status-transitions
Content-Type: application/json

{
  "targetStatus": "CANCELLED",
  "reason": "USER_REQUEST"
}
```

这不是最纯粹的 REST，但比 `POST /cancelOrder` 更容易记录、审计和查询。

### 异步任务资源

耗时任务不应让 HTTP 请求一直阻塞。

```http
POST /export-jobs
```

```http
GET /export-jobs/job-1
```

任务状态至少包括 `PENDING`、`RUNNING`、`SUCCEEDED`、`FAILED`、`CANCELLED`。

### 搜索资源

简单筛选用 `GET /products?...`。复杂搜索条件过长或包含复杂 JSON 时，可以创建搜索请求：

```http
POST /product-searches
Content-Type: application/json

{
  "filters": {
    "brandIds": ["b1", "b2"],
    "price": {
      "gte": 10000,
      "lte": 50000
    }
  },
  "sort": ["-sales", "id"]
}
```

如果搜索请求本身不需要保存，也可以使用 `POST /products/search`，但要明确这是搜索动作，不要误用缓存语义。

## 反模式和修正

| 反模式 | 问题 | 修正 |
| --- | --- | --- |
| `GET /createOrder` | 安全方法产生副作用，可能被预取误触发 | `POST /orders` |
| `POST /getUser` | 查询无法被缓存和标准监控识别 | `GET /users/{id}` |
| `POST /api` + `action` | 退化成私有 RPC 隧道 | 按资源拆分路径和方法 |
| 深度嵌套 5 层以上 | 权限和查询入口复杂，路径脆弱 | 提供顶级资源入口 |
| 数据库表直接暴露 | 内部结构泄露，演进困难 | 设计面向调用方的资源表示 |
| 用状态字段代替流程资源 | 审核、原因、失败记录丢失 | 建模为申请、任务或转移记录 |

## 例子：审批系统资源建模

业务对象：

- 审批模板：`/approval-templates`
- 审批实例：`/approval-instances`
- 审批任务：`/approval-tasks`
- 审批动作记录：`/approval-actions`

审批通过不是简单 `PATCH status=APPROVED`，因为它涉及操作者、意见、节点、时间和下一步流转：

```http
POST /approval-actions
Content-Type: application/json

{
  "taskId": "task-1",
  "action": "APPROVE",
  "comment": "Looks good"
}
```

## 练习

把“用户申请退款，客服审核，财务打款，第三方支付回调，用户查询进度”建模为资源。要求列出：

- 资源名和 URI。
- 每个资源的生命周期。
- 哪些资源需要顶级集合入口。
- 哪些操作需要幂等键。
- 哪些状态转移要记录审计日志。

## 验收

- 能用决策树判断资源、子资源、关系资源、任务资源。
- 能解释为什么某动作应该建模为资源或命令。
- 能识别路径动词、深嵌套、表结构泄露等反模式。

> **重点：** 资源建模的关键不是“名词化”，而是找出稳定业务对象、生命周期、权限和查询入口。
>
> **难点：** 复杂业务流程通常需要动作资源或任务资源，简单 CRUD 思维不够。
>
> **易错：** 把所有子对象都嵌套在父对象下面，会导致跨父查询、权限控制和缓存策略变复杂。
