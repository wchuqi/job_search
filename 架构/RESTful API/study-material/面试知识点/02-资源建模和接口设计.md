## 一、资源建模和接口设计

### 1. 如何把业务需求设计成 RESTful API？

**参考答案：**

先识别业务名词和生命周期，抽象集合资源、单个资源、子资源和关系资源；再确定每个资源支持的 HTTP 方法；然后设计请求响应、状态码、错误模型、安全和版本策略。不要从 Controller 方法名开始设计。

> **重点：** 资源建模先于方法选择。

### 2. 路径参数和查询参数怎么区分？

**参考答案：**

路径参数用于定位具体资源或资源层级，例如 `/orders/1001`。查询参数用于描述集合资源的筛选、排序、分页、搜索和字段投影，例如 `/orders?status=PAID&page=1`。

> **易错：** 把筛选条件全部塞进路径，会导致路径数量膨胀且难以组合。

### 3. 分页接口怎么设计？

**参考答案：**

后台管理类列表可以用页码分页：`page`、`pageSize`。大数据量、时间线和信息流更适合游标分页：`cursor`、`limit`。需要限制最大页大小，排序字段要有白名单，深分页要评估性能。

```http
GET /orders?status=PAID&sort=-createdAt&page=1&pageSize=20
```

> **难点：** 页码分页在数据实时变化时可能重复或漏数据，游标分页更稳定但实现和调试更复杂。

### 4. 批量操作如何设计？

**参考答案：**

简单批量查询可用查询参数或请求体；高风险批量修改建议建模为批量任务或请求资源，例如 `POST /orders/batch-cancellation-requests`。响应要说明成功、失败和失败原因，必要时使用异步任务。

> **重点：** 批量操作要明确是否事务性、是否允许部分成功、如何重试。
