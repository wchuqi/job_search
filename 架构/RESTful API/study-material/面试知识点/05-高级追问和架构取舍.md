## 一、高级追问和架构取舍

### 1. REST 的核心约束有哪些？实际项目为什么很少完整实现 HATEOAS？

**参考答案：**

REST 核心约束包括 Client-Server、Stateless、Cacheable、Uniform Interface、Layered System，以及可选的 Code on Demand。HATEOAS 属于统一接口约束中的高级形式，要求客户端通过响应中的链接发现下一步操作。

实际项目很少完整实现 HATEOAS，因为前端页面流程、OpenAPI、SDK、网关路由和权限配置通常依赖静态契约。团队需要额外设计链接关系、超媒体格式和客户端解释器，成本较高。多数业务 API 达到 Richardson Level 2，即资源、HTTP 方法和状态码使用正确，已经能获得主要工程收益。

> **重点：** 不要说 HATEOAS 没用，而要说它理论上降低耦合，但工程生态和成本让多数项目停留在 Level 2。

### 2. RESTful API 和 RPC 该怎么取舍？

**参考答案：**

面向稳定资源、开放平台、多端复用、需要 HTTP 缓存和通用工具时，RESTful API 更合适。内部服务调用、复杂命令式动作、强类型低延迟调用、大量过程接口时，RPC 更直接。两者不是优劣关系，而是建模方式不同：REST 以资源为中心，RPC 以动作或方法为中心。

> **易错：** 为了追求 REST，把所有复杂命令伪装成资源，反而让 API 语义含糊。

### 3. “取消订单”应该用 `PATCH /orders/{id}` 还是 `POST /orders/{id}/cancellation-requests`？

**参考答案：**

如果取消只是简单状态修改，没有审核、原因、失败记录、补偿流程，可以用 `PATCH /orders/{id}` 修改状态。如果取消本身是一个业务流程，有原因、操作者、审核、失败原因、幂等、审计和可查询进度，更适合建模为取消申请资源：`POST /orders/{id}/cancellation-requests`。

> **重点：** 判断依据不是“哪个更 REST”，而是动作是否有独立生命周期和审计价值。

### 4. 复杂搜索条件很长，GET 放不下怎么办？

**参考答案：**

简单过滤仍然优先 `GET /products?...`，因为查询语义、安全性和缓存更明确。如果搜索条件复杂到 URL 过长，或者包含嵌套条件，可以使用 `POST /product-searches` 创建搜索资源，或 `POST /products/search` 做搜索动作。若搜索结果需要复用、分享、审计，建模为搜索资源更合理；若只是一次性查询，搜索动作也可以接受，但要明确缓存和幂等边界。

> **难点：** `POST` 搜索不是绝对错误，错误的是不解释语义和后续治理。

### 5. 为什么不能直接把数据库表暴露成 REST 资源？

**参考答案：**

数据库表是内部存储模型，API 资源是外部业务契约。直接暴露表会泄露内部字段、权限边界和实现细节，也会让数据库重构变成 API 破坏性变更。正确做法是面向调用方设计资源表示，隐藏内部表结构。

> **易错：** 把 ORM Entity 直接返回给前端，容易泄露密码哈希、删除标记、内部状态和审计字段。

### 6. RESTful API 如何表达长时间运行的任务？

**参考答案：**

提交请求后返回 `202 Accepted`，并通过 `Location` 返回任务资源地址。客户端查询任务资源获取状态、进度、结果和失败原因。

```http
POST /export-jobs
```

```http
HTTP/1.1 202 Accepted
Location: /export-jobs/job-1
```

> **重点：** 长任务要资源化，避免 HTTP 请求一直阻塞。
