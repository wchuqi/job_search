## 一、生产级可靠性和治理

### 1. 支付接口如何设计幂等？

**参考答案：**

支付接口通常使用 `POST /payments`，客户端传 `Idempotency-Key`。服务端保存幂等键、调用方、接口、请求摘要、处理状态、业务支付单 ID、响应结果和过期时间。相同键相同请求返回第一次结果；相同键不同请求返回 `409`。同时还要用数据库唯一约束限制同一订单同一支付阶段只能有一个有效支付单，并用支付单状态机处理重复回调和乱序事件。

> **重点：** 幂等键、唯一约束、状态机三者要配合，单靠 Redis 键不够。

### 2. 客户端调用创建订单超时了，怎么办？

**参考答案：**

超时不代表失败，可能服务端已经创建成功但响应丢失。客户端应使用幂等键重试，服务端返回第一次处理结果；或者客户端通过查询接口按业务请求号查询订单创建结果。服务端不能因为重试就重复创建订单。

> **难点：** 处理的是“结果未知”，不是简单“失败重试”。

### 3. API 版本变更怎么判断是否破坏兼容？

**参考答案：**

删除字段、字段改名、类型变化、格式变化、新增必填请求字段、状态码变化、错误码变化、权限语义变化、默认排序变化通常都是破坏性或高风险变更。新增可选字段通常兼容，但枚举新增值也可能破坏客户端 switch 逻辑。要通过 OpenAPI diff、契约测试和调用方统计评估。

> **易错：** 认为只要路径没变就兼容，实际上字段语义变化可能更危险。

### 4. 如何做 API 契约测试？

**参考答案：**

提供方可以根据 OpenAPI 在 CI 中校验响应是否符合 Schema，这叫 Provider Contract Test。多团队协作时，消费者可以提交自己依赖的请求和响应期望，提供方发布前验证不破坏消费者，这叫 Consumer-Driven Contract。还可以做 OpenAPI diff 检测删除字段、新增必填字段、类型变化和状态码变化。

> **重点：** 契约测试是防止“文档写着能用，实现已经变了”。

### 5. JWT 的优缺点是什么？如何处理注销和权限变更？

**参考答案：**

JWT 自包含、便于跨服务验证，减少每次查会话的成本。但未过期前难以主动撤销，权限变化无法天然实时生效，payload 不能放敏感信息。常见方案是 access token 短有效期、refresh token 可撤销、关键权限变化维护用户权限版本或会话版本，服务端必要时做黑名单或 introspection。

> **难点：** JWT 的无状态和实时撤销是天然冲突，不要说“用 JWT 就不用存任何状态”。

### 6. 大列表接口为什么不建议深分页？怎么优化？

**参考答案：**

深分页如 `offset 100000 limit 20` 通常需要扫描和跳过大量数据，性能差；实时数据变化时还可能重复或漏数据。优化方式包括游标分页、稳定排序键、合理索引、限制最大页码或 pageSize、异步导出、缓存热点查询。

> **重点：** 游标分页通常基于稳定排序字段，例如 `(createdAt, id)`。

### 7. 线上 API 延迟突然升高，你怎么排查？

**参考答案：**

先看 QPS、错误率、P95/P99 延迟和状态码分布，再用 traceId 分解网关、应用、数据库、缓存和外部依赖耗时。检查最近发布、配置变更、流量变化、慢查询、连接池耗尽、线程池队列、GC、依赖超时、缓存击穿和客户端重试风暴。

> **易错：** 只看平均耗时会掩盖长尾延迟。

### 8. 多租户 API 如何防止越权？

**参考答案：**

每次访问资源都要校验身份中的租户、角色、数据范围和资源本身的租户归属。不能信任客户端传入的 `tenantId` 或 `userId`。平台客服等跨租户角色需要临时授权、原因记录和审计日志。

> **重点：** 多租户安全核心是资源级授权和数据隔离，不是路径里带租户 ID。
