## 一、场景题和系统设计

### 1. 设计一个订单创建 API，你会考虑什么？

**参考答案：**

接口可以设计为 `POST /orders`。需要考虑请求体字段、商品和库存校验、价格快照、收货地址、认证授权、幂等键、事务边界、状态码、错误响应、审计日志和 traceId。成功可返回 `201 Created` 和 `Location`。库存不足可返回 `409` 或 `422`，取决于团队错误模型。

> **重点：** 面试官通常不只考路径，更考状态机、幂等、安全和异常处理。

### 2. 支付回调 API 如何设计？

**参考答案：**

支付回调通常是第三方调用服务端的 Webhook。需要验证签名、时间戳和 nonce，防止伪造和重放；回调处理必须幂等；根据支付单状态机处理乱序和重复事件；响应应简单明确，失败时让对方可重试。

> **难点：** 支付回调不是浏览器 CORS 问题，而是服务端到服务端的签名、幂等和状态机问题。

### 3. 如何设计异步导入接口？

**参考答案：**

提交导入请求返回 `202 Accepted`，并返回任务资源地址。客户端查询任务状态。

```http
POST /import-jobs
Content-Type: application/json

{
  "fileId": "file-1",
  "type": "ORDER_IMPORT"
}
```

```http
HTTP/1.1 202 Accepted
Location: /import-jobs/job-1
```

任务资源包含状态、进度、成功数、失败数和错误文件地址。

> **重点：** 长任务不要让 HTTP 请求一直阻塞到完成。

### 4. RESTful API 如何和微服务网关配合？

**参考答案：**

网关负责统一入口、认证、路由、限流、灰度、跨域、日志和审计。服务内部仍要做资源级授权和输入校验，不能完全信任网关。网关日志和应用日志需要共享 traceId，方便跨服务排障。

> **易错：** 只在网关做鉴权，内部服务完全裸奔，一旦绕过网关或内部调用失控就会越权。
