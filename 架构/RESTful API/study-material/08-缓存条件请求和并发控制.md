# RESTful API学习资料：缓存条件请求和并发控制

[返回索引](../RESTful API学习资料.md)

## 学习目标

- 理解 HTTP 缓存基础。
- 掌握 ETag、Last-Modified 和条件请求。
- 能用条件请求实现乐观并发控制。

## 理论导读

RESTful API 可以利用 HTTP 缓存减少重复请求，也可以利用条件请求避免并发覆盖。缓存关注“能不能少传”，并发控制关注“能不能防止旧版本覆盖新版本”。两者都依赖资源表示的版本信息。

## 核心心智模型

ETag 像文件指纹。客户端说“如果文件还是这个指纹，就不用再发给我”或“只有文件还是这个指纹，才允许我覆盖”。

## 知识点详解

### 缓存头

```http
Cache-Control: public, max-age=60
ETag: "order-1001-v3"
Last-Modified: Sun, 07 Jun 2026 10:00:00 GMT
```

### 条件 GET

```http
GET /products/p1
If-None-Match: "product-p1-v8"
```

如果资源未变化，服务端返回：

```http
HTTP/1.1 304 Not Modified
```

### 条件更新

```http
PATCH /users/u1
If-Match: "user-u1-v3"
Content-Type: application/json

{
  "nickname": "new-name"
}
```

如果 ETag 不匹配，返回：

```http
HTTP/1.1 412 Precondition Failed
```

### 缓存策略

| 场景 | 推荐策略 |
| --- | --- |
| 公共静态数据 | `public, max-age=...` |
| 用户私有数据 | `private, max-age=...` 或谨慎禁用 |
| 强一致敏感数据 | `no-store` |
| 可验证缓存 | `ETag` + `If-None-Match` |

## 练习

为商品详情、用户资料、订单详情、首页配置四类 API 设计缓存策略，并说明原因。

## 验收

- 能解释 `no-store`、`no-cache`、`max-age` 的区别。
- 能用 `ETag` 处理缓存和乐观锁。
- 能解释 `304` 和 `412` 的触发条件。

> **重点：** 条件请求既能节省带宽，也能防止并发覆盖。
>
> **难点：** 需要区分资源版本、表示版本和业务版本；不同表示格式可能有不同 ETag。
>
> **易错：** 给用户隐私数据设置公共缓存，可能造成数据泄露。
