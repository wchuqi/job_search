# RESTful API学习资料：REST约束成熟度和架构取舍深度解析

[返回索引](../RESTful API学习资料.md)

## 学习目标

- 从架构约束层面理解 REST，而不是停留在 URL 命名规范。
- 能解释 REST、HTTP API、RPC、GraphQL、事件驱动接口各自的边界。
- 能用成熟度模型判断一个 API 的 RESTful 程度，并说明工程上为什么不一定追求“满分 REST”。

## 理论导读

REST 是一种架构风格，不是某个框架或协议。它强调通过统一接口操作资源，使客户端、服务端、缓存、代理和网关可以在较少耦合下协作。REST 的价值来自约束：约束越清晰，系统越容易被通用组件理解；约束越弱，短期开发可能更快，但长期治理、缓存、重试、监控、权限和兼容性会变难。

工程里的“RESTful API”通常是“基于 HTTP 的资源风格 API”。它未必完整实现 HATEOAS，但仍应认真使用资源建模、HTTP 方法、状态码、缓存、内容协商和无状态交互。

## REST核心约束

| 约束 | 含义 | 工程收益 | 违反后的问题 |
| --- | --- | --- | --- |
| Client-Server | 客户端和服务端职责分离 | 前后端、移动端、第三方可独立演进 | 客户端依赖内部实现，重构困难 |
| Stateless | 每个请求自包含处理所需上下文 | 易扩展、易负载均衡、易重试 | 请求依赖隐式会话步骤，排障困难 |
| Cacheable | 响应声明是否可缓存 | 降低延迟和服务压力 | 所有请求打到源站，性能差 |
| Uniform Interface | 统一接口操作资源 | 工具、网关、监控能理解 | 所有行为变成私有协议 |
| Layered System | 支持分层代理和网关 | 网关、CDN、安全层可插入 | 客户端必须知道服务端拓扑 |
| Code on Demand | 可选，服务端下发可执行代码 | Web 场景扩展 | 普通 API 很少使用 |

## Richardson成熟度模型

| 等级 | 特征 | 示例 | 评价 |
| --- | --- | --- | --- |
| Level 0 | 一个端点承载所有操作 | `POST /api` + action 字段 | 本质是隧道式 RPC |
| Level 1 | 有资源，但方法语义弱 | `/orders/1001` 全部用 `POST` | 资源边界出现，但协议价值没用起来 |
| Level 2 | 资源 + HTTP 方法 + 状态码 | `GET /orders/1001`、`201`、`404` | 多数工程目标应至少达到这里 |
| Level 3 | 超媒体驱动状态转移 | 响应里包含可用操作链接 | 更松耦合，但实现和客户端习惯成本高 |

## HATEOAS为什么少见

HATEOAS 要求客户端通过响应里的链接发现下一步可执行操作，而不是写死所有 URI。它理论上降低客户端和服务端路径耦合，但工程落地难点明显：

- 前端和移动端通常需要明确页面流程和埋点，动态发现链接并不自然。
- OpenAPI、SDK、网关路由、权限配置更习惯静态契约。
- 团队很少愿意为超媒体格式、链接关系、客户端解释器投入成本。

所以面试中更稳妥的回答是：完整 REST 包含 HATEOAS，但实际业务 API 多数落在 Richardson Level 2。我们不必机械追求 Level 3，但要避免退化成 Level 0。

## REST、RPC、GraphQL、事件接口对比

| 类型 | 适合场景 | 优势 | 风险 |
| --- | --- | --- | --- |
| RESTful API | 面向资源的 CRUD、开放平台、前后端通用接口 | 协议语义清晰，工具生态强，缓存和状态码自然 | 复杂动作和多资源聚合可能笨重 |
| RPC | 内部服务调用、命令式业务、低延迟强类型接口 | 方法直接，代码生成方便，调用像本地方法 | 容易过度耦合内部实现 |
| GraphQL | 前端多端差异大、字段组合灵活 | 客户端按需取字段，减少接口数量 | 鉴权、缓存、复杂查询成本高 |
| 事件/Webhook | 异步通知、状态变更广播 | 解耦生产者和消费者，适合最终一致 | 幂等、顺序、重放、追踪复杂 |

## 架构取舍决策

### 什么时候优先 RESTful API

- 资源边界清楚，例如用户、订单、商品、工单、文章。
- 需要对外开放或多端复用。
- 需要 HTTP 缓存、状态码、网关治理和标准文档。
- 调用方希望通过 OpenAPI、Mock、SDK、Postman 等工具协作。

### 什么时候不要强行 REST

- 操作明显是复杂命令，例如“重新计算全量账单并生成补偿记录”。
- 一次请求需要高度定制多个资源字段，且前端组合变化频繁。
- 内部服务间需要强类型、低延迟、大量方法调用。
- 长时间状态流、实时双向通信，例如聊天、行情推送。

## 例子：取消订单到底怎么设计

三种常见设计：

```http
PATCH /orders/1001
Content-Type: application/json

{
  "status": "CANCELLED"
}
```

适合状态修改非常简单，且取消不需要独立审核、原因、操作者、补偿任务。

```http
POST /orders/1001/cancellation-requests
Content-Type: application/json

{
  "reason": "USER_CHANGED_MIND"
}
```

适合取消是一个业务流程，有原因、审核、状态机、失败原因、幂等和审计。

```http
POST /order-cancellation
Content-Type: application/json

{
  "orderId": "1001"
}
```

命令味道更重，资源边界较弱，但如果团队明确把它当命令 API，也可以接受。关键是不要伪装成 REST 后又不遵守资源语义。

## 深度练习

选择“退款、审批、导入、导出、发送验证码、重算报表”六个场景，分别判断：

- 能否抽象成资源。
- 是否应使用异步任务资源。
- 是否需要幂等键。
- 是否适合 REST、RPC 或事件。

## 验收

- 能解释 REST 的六个约束以及常见工程落地程度。
- 能说清 Richardson Level 2 和 Level 3 的区别。
- 能在复杂动作场景下做架构取舍，而不是机械套模板。

> **重点：** REST 的工程价值来自资源边界和协议语义，不是来自路径格式本身。
>
> **难点：** 很多业务动作既可以设计成资源，也可以设计成命令，关键看它是否有生命周期、状态、审计和可查询结果。
>
> **易错：** 为了“看起来 RESTful”把所有命令强行改名成名词，但底层仍然忽略方法语义、状态码、幂等和资源边界。
