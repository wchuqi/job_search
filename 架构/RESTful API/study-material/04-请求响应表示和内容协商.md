# RESTful API学习资料：请求响应表示和内容协商

[返回索引](../RESTful API学习资料.md)

## 学习目标

- 掌握 JSON 请求响应设计。
- 理解表示、媒体类型和内容协商。
- 能设计稳定、易扩展的字段结构。

## 理论导读

资源和表示不是一回事。服务端内部可能是多张表、缓存对象或聚合模型，API 返回的是面向调用方的表示。表示设计一旦发布，就成为外部契约，不能像内部 DTO 一样随意调整。

内容协商让客户端通过 `Accept` 表示想要的格式，服务端通过 `Content-Type` 声明实际返回格式。常见 RESTful API 使用 JSON，但也可能返回 CSV、PDF、图片或二进制流。

## 核心心智模型

API 响应像合同附件，不是数据库截图。合同附件必须面向读者稳定表达信息，而不是暴露内部字段怎么存。

## 知识点详解

### 字段设计

- 字段名保持一致，常见选择是 `camelCase`。
- 金额避免浮点数，使用最小货币单位或字符串十进制。
- 时间使用 ISO 8601 并包含时区。
- 枚举值使用稳定字符串，如 `PAID`、`CANCELLED`。
- 不返回调用方不需要的敏感字段。

### 响应包装

两种常见风格：

```json
{
  "id": "1001",
  "status": "PAID"
}
```

```json
{
  "data": {
    "id": "1001",
    "status": "PAID"
  },
  "traceId": "abc"
}
```

裸资源风格更贴近 HTTP，统一包装风格方便某些前端框架和网关处理。团队要统一，不要同一系统混用。

### 创建响应

```http
HTTP/1.1 201 Created
Location: /orders/1001
Content-Type: application/json

{
  "id": "1001",
  "status": "CREATED"
}
```

### 空响应

删除成功或更新成功且无需返回资源时，使用 `204 No Content`，响应体应为空。

## 练习

为用户资料接口设计请求和响应，包含昵称、头像、手机号脱敏、邮箱验证状态、创建时间、更新时间。

## 验收

- 能解释资源和表示的区别。
- 能设计金额、时间、枚举、可空字段。
- 能说明 `Content-Type` 和 `Accept` 的作用。

> **重点：** API 表示是外部契约，稳定性优先于内部实现方便。
>
> **难点：** 字段新增通常兼容，字段删除、改名、类型变化、语义变化通常不兼容。
>
> **易错：** 直接返回数据库实体，容易泄露密码哈希、内部状态、删除标记和审计字段。
