# RESTful API学习资料：缓存分页性能网关和排障深度解析

[返回索引](../RESTful API学习资料.md)

## 学习目标

- 深入理解缓存、分页、查询性能和网关治理。
- 能针对高流量 API 设计缓存和限流策略。
- 能建立线上排障路径，而不是只看业务代码。

## 理论导读

API 性能问题常常不是单个接口“写得慢”，而是缓存策略、查询模型、分页方式、序列化大小、依赖调用、网关限流和客户端重试共同造成。RESTful API 的生产设计要从“请求进入网关”一直看到“响应返回客户端”。

## 缓存层次

| 层次 | 示例 | 适合内容 | 风险 |
| --- | --- | --- | --- |
| 浏览器缓存 | `Cache-Control` | 静态或低敏数据 | 私密数据泄露 |
| CDN | 商品详情、图片 | 公共热点内容 | 失效传播延迟 |
| 网关缓存 | 公共查询接口 | 多服务共用缓存 | 权限差异难处理 |
| 应用缓存 | Redis、本地缓存 | 热点业务数据 | 一致性和击穿 |
| 数据库缓存 | Buffer Pool | 数据页 | 慢查询仍会拖垮 |

## Cache-Control深度

| 指令 | 含义 | 注意 |
| --- | --- | --- |
| `public` | 共享缓存可缓存 | 不用于用户私密数据 |
| `private` | 仅用户代理可缓存 | 用户数据可谨慎使用 |
| `max-age` | 新鲜时间 | 过期前可能不访问源站 |
| `no-cache` | 可缓存但使用前必须重新验证 | 名字容易误解 |
| `no-store` | 不应存储 | 登录、支付、隐私数据 |
| `stale-while-revalidate` | 可返回旧数据并后台刷新 | 适合容忍短暂旧数据 |

## 分页性能

### 页码分页问题

```sql
select * from orders order by created_at desc limit 20 offset 100000;
```

深分页需要扫描和跳过大量记录，性能差。实时插入数据时，用户翻页还可能重复或漏数据。

### 游标分页设计

```http
GET /orders?limit=20&cursor=eyJjcmVhdGVkQXQiOiIyMDI2LTA2LTA3VDEwOjAwOjAwKzA4OjAwIiwiaWQiOiIxMDAxIn0
```

游标应基于稳定排序字段，例如 `(createdAt, id)`。查询条件类似：

```sql
where (created_at, id) < (:lastCreatedAt, :lastId)
order by created_at desc, id desc
limit 20
```

### 排序和过滤白名单

不要允许客户端传任意字段排序。每个可排序字段都应评估：

- 是否有索引。
- 是否能和过滤条件组合。
- 是否会造成不稳定排序。
- 是否涉及敏感或内部字段。

## 字段投影和N+1问题

`include=items,payment` 可以减少接口数量，但可能引发 N+1 查询和响应过大。应设定：

- 可展开关系白名单。
- 最大展开深度。
- 响应大小限制。
- 慢查询监控。

## 网关治理

网关常见职责：

- TLS 终止。
- 路由和负载均衡。
- 认证前置校验。
- 限流和配额。
- CORS。
- 请求体大小限制。
- 灰度发布。
- 访问日志和审计。

网关不应替代业务服务的资源级授权。服务端仍要做最终权限判断。

## 限流策略

| 策略 | 特点 | 场景 |
| --- | --- | --- |
| 固定窗口 | 简单，但边界瞬时流量高 | 低风险接口 |
| 滑动窗口 | 更平滑 | 登录、搜索 |
| 令牌桶 | 允许突发，控制平均速率 | API 网关常用 |
| 漏桶 | 稳定输出 | 下游保护 |

限流返回：

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 30
```

## 排障剧本

### 4xx突然升高

- `401`：token 过期、认证服务异常、时钟偏移。
- `403`：权限配置变更、租户关系异常。
- `404`：路由变更、资源 ID 错误、灰度版本不一致。
- `409/412`：并发冲突或重复提交增多。
- `429`：流量异常、客户端重试风暴。

### 5xx突然升高

- 最近发布引入异常。
- 数据库连接池耗尽。
- 外部依赖超时。
- 缓存服务不可用。
- 序列化或反序列化异常。
- 网关路由到错误版本。

### 延迟升高

- 看 P95/P99，不只看平均值。
- 拆分网关、应用、数据库、缓存、外部调用耗时。
- 检查慢查询、锁等待、连接池、GC、线程池队列。
- 检查客户端是否无退避重试。

为“商品搜索 API”设计生产方案：

## 验收

- 能解释页码分页和游标分页的性能差异。
- 能设计不同数据敏感级别的缓存策略。
- 能按状态码、延迟、依赖、发布和流量变化进行排障。

> **重点：** 性能不是只靠代码优化，API 查询模型、缓存、网关、限流和客户端重试都会影响系统稳定性。
>
> **难点：** 缓存提高性能但引入一致性和隐私风险，分页提高可用性但影响排序稳定性和查询成本。
>
> **易错：** 给用户私密数据设置共享缓存，或开放无限 pageSize 和任意排序字段。
