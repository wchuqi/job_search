# RESTful API学习资料：HTTP语义状态码和错误边界深度解析

[返回索引](../RESTful API学习资料.md)

## 学习目标

- 深入掌握 HTTP 方法、状态码、Header 和错误模型的边界。
- 能解释容易混淆的状态码选择。
- 能设计既符合协议语义又便于业务处理的错误响应。

## 理论导读

HTTP 语义不是装饰，它会影响缓存、重试、代理、浏览器行为、网关策略、监控统计和客户端错误处理。状态码过粗会丢失诊断能力，状态码过细又可能让团队难以统一。合理做法是使用一组稳定状态码表达通用语义，再用业务错误码表达业务细节。

## 状态码决策框架

### 第一步：请求是否到达正确资源

- 路径不存在：`404`
- 方法不支持：`405`
- 媒体类型不支持：`415`
- 响应格式无法满足：`406`

### 第二步：调用方身份是否有效

- 未登录、token 缺失、token 过期：`401`
- 已登录但无权限：`403`

### 第三步：请求内容是否可理解

- JSON 语法错误、参数类型错误、缺少必要参数：`400`
- 语法正确但业务规则不满足：`422`

### 第四步：是否与资源当前状态冲突

- 资源版本冲突、重复创建、状态不允许转移：`409`
- 条件请求前提不满足：`412`

### 第五步：服务端是否可用

- 未预期程序错误：`500`
- 依赖不可用、服务维护、过载：`503`
- 网关到上游超时：`504`

## 易混状态码边界

### `400` vs `422`

`400` 更偏请求格式、协议层参数或解析失败。`422` 更偏请求语法正确，但业务无法处理。

```json
{
  "quantity": "abc"
}
```

数量类型错误，倾向 `400`。

```json
{
  "quantity": 999999
}
```

类型正确，但超过库存或业务上限，倾向 `422` 或 `409`，取决于是否强调资源状态冲突。

### `409` vs `412`

`409` 是一般资源冲突。`412` 专门用于条件请求失败，例如 `If-Match` 不匹配。

```http
PATCH /documents/d1
If-Match: "v3"
```

如果当前版本是 `v4`，返回 `412` 比 `409` 更精确。

### `404` vs `403`

如果暴露资源存在性会造成安全风险，可以对无权访问的资源返回 `404`。但内部系统或管理后台通常返回 `403` 更利于排障。关键是团队安全策略一致。

### `201` vs `202`

`201` 表示资源已经创建。`202` 表示请求已接受，但处理尚未完成。导入、导出、批处理、异步审核更适合 `202` + 任务资源。

## Header的深度作用

| Header | 作用 | 常见风险 |
| --- | --- | --- |
| `Location` | 指向新建资源或异步任务 | 创建后不返回位置，客户端只能猜 |
| `Content-Type` | 声明请求或响应格式 | 格式和实际内容不一致 |
| `Accept` | 客户端声明期望格式 | 服务端忽略导致内容协商失效 |
| `ETag` | 表示版本指纹 | 生成规则不稳定 |
| `If-Match` | 条件更新 | 不校验导致并发覆盖 |
| `Retry-After` | 告知重试时间 | 限流或维护时不给提示 |
| `X-Request-Id`/`traceId` | 请求追踪 | 错误响应和日志对不上 |

## Problem Details落地模型

建议把错误分为四层：

1. HTTP 状态码：通用协议语义。
2. `type`：错误类别文档地址或稳定类型。
3. `code`：业务错误码，用于程序分支。
4. `traceId` 和 `errors`：排障和字段级错误。

```json
{
  "type": "https://api.example.com/problems/order-state-conflict",
  "title": "Order state conflict",
  "status": 409,
  "code": "ORDER_ALREADY_SHIPPED",
  "detail": "The order has been shipped and cannot be cancelled.",
  "traceId": "req-20260607-001",
  "errors": []
}
```

## 错误码设计原则

- 稳定，不随文案变化。
- 可分类，例如 `ORDER_`、`PAYMENT_`、`AUTH_`。
- 不包含动态信息。
- 不暴露内部异常类名。
- 文案可本地化，错误码不能本地化。

## 练习

为以下场景选择状态码、错误码和响应结构：

- 用户取消已发货订单。
- 客户端提交了非法 JSON。
- token 有效但访问他人订单。
- 更新资料时 ETag 已过期。
- 支付服务临时不可用。
- 请求被限流。

## 验收

- 能用决策框架选择状态码。
- 能解释至少 6 组易混状态码边界。
- 能设计 Problem Details 风格错误响应。

> **重点：** 状态码表达通用协议结果，业务错误码表达业务规则，不要互相替代。
>
> **难点：** 同一场景可能有多个合理状态码，关键是团队模型一致并能解释取舍。
>
> **易错：** 所有业务失败都返回 `200`，会污染监控、网关策略和客户端错误处理。
