# RESTful API学习资料：契约版本治理和评审清单深度解析

[返回索引](../RESTful API学习资料.md)

## 学习目标

- 把 API 设计从“写接口”提升到“治理契约”。
- 能判断兼容变更、破坏性变更和灰色风险变更。
- 能建立接口评审、OpenAPI、契约测试、废弃下线的流程。

## 理论导读

API 契约包括路径、方法、请求参数、请求体、响应体、状态码、错误码、Header、认证方式、权限语义、分页规则、排序规则、速率限制和兼容承诺。只要调用方依赖了它，它就是契约。契约治理的核心目标是减少“服务端改得合理，但调用方线上坏了”的事故。

## 兼容性深度分类

| 变更 | 风险 | 原因 |
| --- | --- | --- |
| 新增响应字段 | 通常低 | 客户端应忽略未知字段 |
| 新增请求可选字段 | 通常低 | 不传时保持旧行为 |
| 新增必填请求字段 | 高 | 旧客户端无法调用 |
| 删除响应字段 | 高 | 旧客户端可能读取 |
| 字段改名 | 高 | 等同删除旧字段加新增字段 |
| 字段类型变化 | 高 | 解析失败 |
| 字段格式变化 | 高 | 日期、金额、ID 格式可能被依赖 |
| 枚举新增值 | 中高 | 客户端 switch 可能没有 default |
| 默认排序变化 | 中高 | 用户看到的数据顺序改变 |
| 分页语义变化 | 高 | 可能重复或漏数据 |
| 状态码变化 | 高 | 客户端错误分支改变 |
| 错误码变化 | 高 | 前端提示和业务处理失败 |
| 权限语义变化 | 高 | 安全和业务流程都可能受影响 |

## API评审清单

### 资源和URI

- 资源名是否稳定。
- 是否存在动词路径。
- 是否有过深嵌套。
- 是否暴露数据库表或内部枚举。
- 是否需要顶级查询入口。

### 方法和状态码

- 方法是否符合安全性和幂等性。
- 创建是否返回 `201` 和 `Location`。
- 异步处理是否返回 `202` 和任务资源。
- 错误状态码是否能被网关和监控识别。

### 请求响应

- 字段是否有类型、含义、必填、可空说明。
- 金额、时间、枚举、ID 格式是否稳定。
- 是否返回敏感字段。
- 是否有成功和失败示例。

### 安全

- 是否认证。
- 是否资源级授权。
- 是否多租户隔离。
- 是否记录审计。
- 是否限制请求体大小。
- 是否有输入校验和输出脱敏。

### 可靠性

- 高风险写操作是否幂等。
- 是否定义超时和重试边界。
- 是否处理并发冲突。
- 是否有状态机校验。

### 性能

- 是否有分页上限。
- 过滤和排序是否有白名单。
- 是否评估索引。
- 是否有缓存策略。
- 是否限制 include 展开深度。

### 治理

- 是否写入 OpenAPI。
- 是否有契约测试。
- 是否有监控指标。
- 是否有 traceId。
- 是否有版本和废弃策略。

## OpenAPI治理

OpenAPI 不应只是生成 Swagger 页面。更完整的用法：

- 设计阶段评审契约。
- 联调前生成 Mock。
- CI 中校验实现响应是否符合 Schema。
- 发布前检测破坏性变更。
- SDK 生成和版本管理。
- 文档示例和错误码同步。

## 契约测试策略

### Provider Contract Test

服务提供方根据 OpenAPI 校验自己的响应，确保实现没有偏离文档。

### Consumer-Driven Contract

消费者提交自己依赖的请求和响应期望，提供方发布前验证这些期望不被破坏。适合多团队、多服务协作。

### 兼容性扫描

对比新旧 OpenAPI：

- 删除路径。
- 删除方法。
- 删除字段。
- 新增必填字段。
- 类型变化。
- 枚举变化。
- 状态码变化。

## 废弃和下线流程

1. 标记 deprecated。
2. 文档提供替代接口。
3. 响应头提示废弃和下线日期。
4. 统计调用方和调用频率。
5. 通知调用方并提供迁移支持。
6. 灰度拒绝或降低配额验证影响。
7. 到期下线并保留故障回滚预案。

示例响应头：

```http
Deprecation: true
Sunset: 2026-12-31T23:59:59+08:00
Link: <https://api.example.com/docs/migration/orders-v2>; rel="deprecation"
```

## 练习

评审下面变更是否安全：

- 删除 `order.totalAmount` 字段。
- 把 `order.status` 从 `PAID` 改成 `PAYMENT_SUCCESS`。
- 给订单列表新增默认过滤 `status != DELETED`。
- 把 `pageSize` 默认值从 20 改成 100。
- 新增枚举 `PARTIALLY_REFUNDED`。
- `404` 改成 `403`。

对每项写出风险、兼容方案和迁移方式。

## 验收

- 能列出一份接口评审清单。
- 能判断 API 变更的兼容性风险。
- 能设计契约测试和废弃下线流程。

> **重点：** API 是多方协作契约，破坏性变更必须被识别、测试、通知和迁移。
>
> **难点：** 最危险的变更往往不是删接口，而是字段含义、默认排序、枚举值、权限语义和错误码变化。
>
> **易错：** 认为“后端代码能跑”就代表接口没问题，忽略调用方解析逻辑、缓存逻辑和错误分支。
