# RESTful API学习资料：总览与心智模型

[返回索引](../RESTful API学习资料.md)

## 学习目标

- 建立 RESTful API 的整体知识地图。
- 理解 REST 的核心不是“好看的 URL”，而是围绕资源进行统一交互。
- 能判断一个接口设计问题属于资源建模、HTTP 语义、错误模型、安全、兼容性还是生产治理。

## 理论导读

REST 是 Representational State Transfer 的缩写，可以理解为“通过资源的表示来转移应用状态”。客户端不直接操作服务端内部对象，而是通过 URI 定位资源，通过 HTTP 方法表达动作，通过表示格式传递数据，通过状态码和响应头理解结果。

一个好的 RESTful API 像一套稳定的公共交通规则：路线代表资源，站点代表资源位置，车票和信号灯代表认证、状态码和缓存规则。调用方不需要知道司机怎么换挡，也就是不需要知道服务端内部代码结构。

## 核心心智模型

### 资源

资源是业务中可以被命名、定位、读取或操作的对象或集合，例如用户、订单、商品、评论、库存记录、支付单。

### 表示

表示是资源在网络上传输的样子，常见是 JSON。资源本身在服务端，表示是某个时刻的快照。

### 统一接口

统一接口指使用有限且语义明确的方法操作资源，例如 `GET` 读取、`POST` 创建或提交、`PUT` 整体替换、`PATCH` 局部修改、`DELETE` 删除。

### 无状态

每个请求必须携带服务端处理它所需的信息。服务端可以保存业务状态，但不应该依赖“上一次 HTTP 请求临时留下的会话步骤”才能理解当前请求。

## 知识覆盖地图

| 领域 | 必学内容 | 容易遗漏 |
| --- | --- | --- |
| HTTP 基础 | 请求行、方法、状态码、头、请求体、响应体 | 安全方法、幂等、缓存语义 |
| 资源建模 | 集合、单资源、子资源、关系资源 | 动作资源、异步任务资源 |
| URI 设计 | 路径、查询参数、命名规范 | 嵌套深度、稳定标识符 |
| 请求响应 | JSON、字段命名、内容协商、响应包装 | 空响应、部分字段、字段兼容 |
| 错误模型 | 状态码、业务错误码、字段错误、追踪 ID | 409/422/429/503 的边界 |
| 安全 | TLS、认证、授权、CORS、输入校验 | IDOR、敏感字段泄露、重放攻击 |
| 数据查询 | 分页、过滤、排序、搜索、批量 | 游标分页、总数成本、批量局部失败 |
| 缓存并发 | Cache-Control、ETag、If-Match | 条件请求和乐观锁 |
| 演进 | 版本、兼容变更、废弃策略 | 字段含义变化比新增字段更危险 |
| 工程化 | OpenAPI、契约测试、Mock、SDK | 文档和实现漂移 |
| 生产治理 | 网关、限流、日志、指标、追踪 | 幂等键、超时重试、排障剧本 |

## 例子

```http
GET /orders/1001 HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer <token>
```

```json
{
  "id": "1001",
  "status": "PAID",
  "totalAmount": 29900,
  "currency": "CNY",
  "createdAt": "2026-06-07T10:00:00+08:00"
}
```

这个例子里 `/orders/1001` 是资源标识，`GET` 是读取语义，JSON 是表示，`Authorization` 提供认证信息。

## 练习

为“图书借阅系统”列出至少 10 个资源，包括读者、图书、借阅记录、预约、罚款、馆藏副本。

## 验收

- 能区分资源、表示、动作和业务流程。
- 能把一个 RPC 风格接口改写为资源风格接口。
- 能说清 RESTful API 在安全、缓存、版本和生产治理中的关注点。

> **重点：** RESTful API 的核心是资源、表示、统一接口、无状态和超媒体约束，其中实际工程最常落地的是前四项。
>
> **难点：** 业务流程经常是动作，但 API 设计要先判断动作背后是否存在可命名的资源或状态转移。
>
> **易错：** 只把动词从路径里删掉，却用 `POST` 承载所有读写操作，会失去 HTTP 方法、缓存、幂等和状态码的价值。
