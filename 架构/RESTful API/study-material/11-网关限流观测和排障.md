# RESTful API学习资料：网关限流观测和排障

[返回索引](../RESTful API学习资料.md)

## 学习目标

- 理解 API 在生产环境中的治理能力。
- 掌握网关、限流、超时、重试、日志、指标和链路追踪。
- 能定位常见 API 故障。

## 理论导读

设计良好的 RESTful API 不只看接口路径，还要看线上运行质量。网关承担统一入口、认证、路由、限流、审计、灰度和协议转换等职责。服务端需要输出结构化日志、指标和 trace，方便定位问题。

## 核心心智模型

API 生产治理像机场安检和调度：入口统一、身份检查、流量控制、异常隔离、全程留痕。

## 知识点详解

### 限流

常见维度：

- 按用户、租户、应用、IP、接口限流。
- 固定窗口、滑动窗口、令牌桶、漏桶。
- 超限返回 `429 Too Many Requests`，可带 `Retry-After`。

### 超时和重试

客户端应设置超时，服务端应给慢依赖设置超时。只对幂等或有幂等键保护的请求自动重试。

### 观测字段

每次请求至少记录：

- `traceId`
- HTTP 方法和路径模板
- 状态码
- 延迟
- 调用方身份
- 错误码
- 上游依赖耗时

不要记录明文密码、Token、身份证号、完整银行卡号。

### 排障路径

1. 看状态码分布：4xx 是调用或权限问题居多，5xx 是服务端或依赖问题居多。
2. 看延迟分位数：平均值不够，要看 P95/P99。
3. 用 traceId 串起网关、应用、数据库、缓存、外部服务。
4. 检查最近发布、配置变更、流量变化和依赖状态。
5. 对重试风暴、缓存击穿、慢查询和连接池耗尽做专项排查。

## 练习

设计一个订单查询接口的监控面板，包含 QPS、成功率、P95 延迟、错误码 TopN、下游依赖耗时和限流次数。

## 验收

- 能解释 `429`、`503`、超时和重试的关系。
- 能说明哪些请求可以自动重试。
- 能设计 traceId 贯穿日志和错误响应。

> **重点：** 生产 API 必须可观测，否则出问题只能猜。
>
> **难点：** 重试能提高短暂故障下的成功率，也能在依赖故障时放大流量。
>
> **易错：** 自动重试非幂等创建接口，可能造成重复下单、重复扣款或重复发券。
