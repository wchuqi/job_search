# RESTful API学习资料：幂等重试并发和一致性深度解析

[返回索引](../RESTful API学习资料.md)

## 学习目标

- 理解幂等、重试、超时、并发和一致性之间的关系。
- 能为支付、下单、退款、发券等高风险接口设计幂等方案。
- 能区分 HTTP 方法幂等、业务幂等和分布式最终一致。

## 理论导读

线上系统中“请求发出后不知道结果”非常常见：客户端超时、网关断开、服务处理成功但响应丢失、依赖半成功、消息重复投递。RESTful API 如果不设计幂等和并发控制，重试就可能把短暂故障变成重复扣款、重复下单或重复发券。

## 三层幂等

| 层次 | 含义 | 示例 | 风险 |
| --- | --- | --- | --- |
| HTTP 方法幂等 | 协议语义上重复执行效果相同 | `PUT /users/u1` | 实现不当仍可能破坏 |
| 业务幂等 | 同一业务请求重复提交只生效一次 | 同一支付请求只生成一笔支付单 | 需要唯一约束和幂等记录 |
| 消息幂等 | 重复事件或回调只处理一次 | 支付回调重复通知 | 需要事件 ID、状态机和去重 |

## 幂等键设计

### 请求格式

```http
POST /payments
Idempotency-Key: pay-req-20260607-0001
Content-Type: application/json

{
  "orderId": "1001",
  "amount": 29900,
  "currency": "CNY"
}
```

### 服务端存储

幂等记录至少包含：

- 幂等键。
- 调用方身份或租户 ID。
- 接口名称。
- 请求摘要 hash。
- 处理状态：`PROCESSING`、`SUCCEEDED`、`FAILED`。
- 业务资源 ID。
- 响应摘要或完整响应。
- 过期时间。

### 决策规则

| 情况 | 处理 |
| --- | --- |
| 幂等键不存在 | 创建记录并处理请求 |
| 键存在，请求摘要相同，已成功 | 返回第一次成功结果 |
| 键存在，请求摘要相同，处理中 | 返回 `409`、`202` 或等待，取决于接口约定 |
| 键存在，请求摘要不同 | 返回 `409 Conflict` |
| 键已过期 | 可按新请求处理，但要评估业务唯一约束 |

## 幂等键和唯一约束

只靠幂等键不够。支付接口还应有业务唯一约束，例如同一订单同一支付阶段只能有一个有效支付单。原因是客户端可能每次重试都生成新幂等键。

```text
unique(order_id, payment_purpose, active_flag)
```

幂等键解决“同一请求重复”，唯一约束解决“不同请求重复做同一业务”。

## 并发控制

### 乐观锁

资源带版本号或 ETag，更新时校验版本。

```http
PATCH /orders/1001
If-Match: "order-1001-v5"
```

不匹配返回 `412`。适合用户资料、文档编辑、库存配置等并发编辑。

### 状态机保护

订单状态转移不能只靠前端判断，服务端必须校验：

```text
CREATED -> PAID -> SHIPPED -> COMPLETED
CREATED -> CANCELLED
PAID -> REFUNDING -> REFUNDED
```

重复回调或乱序事件必须按当前状态决定是否忽略、拒绝或补偿。

### 分布式锁不是首选答案

面试中常有人直接说“加分布式锁”。更稳妥的顺序是：

1. 数据库唯一约束保证不重复。
2. 状态机保证不乱转。
3. 幂等表保证重复请求返回相同结果。
4. 乐观锁防止覆盖。
5. 分布式锁只用于确实需要串行化且无法靠约束表达的场景。

## 超时和重试策略

| 请求类型 | 是否自动重试 | 条件 |
| --- | --- | --- |
| `GET` 查询 | 可以 | 设置超时和最大次数 |
| `PUT` 整体替换 | 可以 | 实现保持幂等 |
| `DELETE` 删除 | 可以 | 删除语义幂等 |
| `POST` 创建支付 | 谨慎 | 必须有幂等键 |
| `POST` 发送短信 | 谨慎 | 需要业务去重或频控 |
| Webhook 回调处理 | 必须幂等 | 事件 ID 和状态机 |

## 例子：支付回调乱序

可能事件：

- `PAYMENT_SUCCEEDED`
- `PAYMENT_FAILED`
- `REFUND_SUCCEEDED`

处理规则：

- 已成功支付后，迟到的失败事件不能把订单改回失败。
- 重复成功事件只更新回调日志，不重复发货。
- 退款成功只能作用于已支付或退款中订单。

## 练习

设计“发优惠券 API”的幂等方案。要求说明：

- 幂等键粒度。
- 唯一约束。
- 重复请求返回什么。
- 处理中请求返回什么。
- 发券成功但响应超时时客户端如何查询。

## 验收

- 能区分 HTTP 幂等、业务幂等和消息幂等。
- 能设计幂等表字段和处理状态。
- 能解释为什么唯一约束、状态机和幂等键要配合使用。

> **重点：** 幂等不是一个注解或一个缓存键，而是一套包含请求识别、业务约束、状态机和结果复用的设计。
>
> **难点：** 超时后“客户端不知道服务端是否成功”是最核心场景，必须提供查询结果或复用结果的能力。
>
> **易错：** 自动重试没有幂等保护的 `POST`，线上可能造成重复扣款、重复发货、重复发券。
