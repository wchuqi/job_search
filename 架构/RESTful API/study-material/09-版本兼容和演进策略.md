# RESTful API学习资料：版本兼容和演进策略

[返回索引](../RESTful API学习资料.md)

## 学习目标

- 理解 API 契约的兼容性。
- 能判断哪些变更是破坏性变更。
- 能设计版本、废弃和迁移策略。

## 理论导读

API 一旦被外部调用，就不再只是服务端代码的一部分，而是多方协作的契约。演进策略的核心不是“怎么快速改”，而是“怎么让已有调用方不坏”。新增能力要尽量兼容，破坏性变更要有版本和迁移窗口。

## 核心心智模型

API 像公共道路。你可以加车道和标识，但不能突然把左转车道改成掉头专用而不通知司机。

## 兼容性分类

| 变更 | 通常是否兼容 | 说明 |
| --- | --- | --- |
| 新增可选响应字段 | 兼容 | 客户端应忽略未知字段 |
| 新增可选请求字段 | 兼容 | 服务端应提供默认行为 |
| 删除字段 | 不兼容 | 旧客户端可能依赖 |
| 字段改名 | 不兼容 | 等同删除旧字段 |
| 字段类型变化 | 不兼容 | 解析失败 |
| 枚举新增值 | 可能不兼容 | 客户端如果写死枚举会失败 |
| 状态码语义变化 | 不兼容 | 错误处理逻辑改变 |
| 字段含义变化 | 高风险不兼容 | 最隐蔽，最容易线上出事 |

## 版本策略

### URI 版本

```http
GET /v1/orders/1001
```

直观、易路由，缺点是容易让版本污染资源路径。

### Header 版本

```http
Accept: application/vnd.example.order.v2+json
```

更贴近内容协商，但使用门槛较高。

### 无版本优先

内部系统可以优先通过兼容演进减少大版本。只有破坏性变更无法兼容时才引入新版本。

## 废弃策略

- 标记 deprecated，并在文档中写明替代接口。
- 通过响应头提示废弃和下线时间。
- 统计调用方和调用量。
- 提供迁移窗口和兼容层。
- 下线前进行告警和灰度。

## 练习

订单状态原来是 `PAID`，现在要拆成 `PAID_PENDING_SETTLEMENT` 和 `PAID_SETTLED`。设计兼容方案和最终迁移方案。

## 验收

- 能判断常见变更是否兼容。
- 能设计双写、双读、兼容字段和废弃流程。
- 能解释 URI 版本与 Header 版本的取舍。

> **重点：** API 版本不是每次发布都递增，而是处理破坏性契约变更的手段。
>
> **难点：** 枚举新增和字段含义变化常被误判为安全变更。
>
> **易错：** 只升级服务端，不统计客户端使用情况，会导致隐蔽调用方故障。
