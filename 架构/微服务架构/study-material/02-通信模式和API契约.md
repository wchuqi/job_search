# 微服务架构学习资料：通信模式和 API 契约

[返回索引](../微服务架构学习资料.md)

## 学习目标

- 区分同步调用、异步消息和事件驱动。
- 掌握 API 契约、版本管理、幂等和超时设计。
- 能设计调用链的选择、路由、匹配和错误处理规则。

## 理论导读

微服务之间不能靠内部方法调用协作，而要通过网络协议、消息或事件表达意图。通信设计决定系统耦合程度、故障传播方式和一致性模型。同步调用简单直观，但会把调用方和被调用方的可用性绑定在一起；异步消息能削峰和解耦，但会引入重复消费、顺序、延迟和最终一致性问题。

API 契约是服务之间的边界文档。契约不只是 URL 和字段，还包括状态码、错误码、幂等规则、超时、重试、鉴权、版本兼容和废弃策略。

## 核心心智模型

同步调用像打电话：需要对方当场接听并回答。异步消息像投递工单：发送方提交后可以继续做事，接收方稍后处理。事件像广播事实：发布方说明“发生了什么”，订阅方自行决定怎么响应。

## 知识点详解

### 通信模式对比

| 模式 | 适用场景 | 优点 | 风险 |
| --- | --- | --- | --- |
| REST/HTTP | 外部 API、资源操作、通用集成 | 易理解、生态成熟 | 过多同步链路会放大延迟 |
| RPC/gRPC | 内部高性能调用、强类型契约 | 性能好、IDL 明确 | 跨语言和版本治理要求高 |
| 消息队列 | 削峰、异步任务、可靠投递 | 解耦、缓冲 | 重复消费、堆积、顺序问题 |
| 领域事件 | 状态变化通知、最终一致性 | 低耦合、易扩展 | 事件语义和版本需要治理 |
| 批处理 | 大批量同步、离线补偿 | 成本可控 | 实时性差 |

### API 路由和匹配规则

常见网关路由选择顺序：

1. 根据 Host、Path、Method、Header、Query 等谓词匹配路由。
2. 多条路由匹配时按优先级或定义顺序选择。
3. 路由命中后执行全局过滤器和路由过滤器。
4. 如果目标是服务名，负载均衡器从可用实例列表中选择实例。
5. 选择实例后转发请求，并记录 routeId、serviceId、instanceId、traceId。

> **重点：** 路由规则必须避免模糊匹配。例如 `/api/order/**` 和 `/api/**` 同时存在时，要明确优先级，否则会出现请求落到错误服务的风险。

### 深度机制：一次服务调用的完整决策链

一次 `POST /api/applications` 请求从客户端到目标实例，通常经历这些决策：

```text
客户端
  -> DNS 解析入口域名
  -> LB 或 Ingress 选择网关实例
  -> Gateway 根据 Host/Path/Method/Header 匹配 Route
  -> Gateway 执行认证、限流、灰度、路径改写
  -> 根据 serviceId 查询可用实例列表
  -> 过滤不可用实例、非目标版本实例、非同区域实例
  -> 负载均衡算法选择一个实例
  -> 建立或复用连接
  -> 下游服务处理并返回
  -> Gateway 记录指标、日志、trace
```

每一步都有失败模式：

- DNS 缓存未刷新，流量仍打到旧入口。
- 路由规则过宽，请求命中错误服务。
- 灰度 header 丢失，用户在新旧版本之间跳变。
- 注册中心实例缓存滞后，请求发到已下线实例。
- 连接池耗尽，请求在客户端排队。
- 下游返回业务错误，但调用方当成系统错误重试。

> **难点：** “服务不可用”不是一个原因，而是路由、发现、负载均衡、连接、协议、超时、业务错误中的任意一环失败。

### 契约版本

推荐原则：

- 新增字段通常向后兼容。
- 删除字段、改字段含义、改错误码通常不兼容。
- 请求字段应区分必填、选填和默认值。
- 响应字段不能随意改变语义。
- 事件版本要包含 schemaVersion 或 eventVersion。
- 废弃旧版本时要有统计、通知和迁移窗口。

### 兼容性矩阵

| 变更 | 是否兼容 | 风险 | 推荐做法 |
| --- | --- | --- | --- |
| 响应新增可选字段 | 通常兼容 | 老消费者忽略即可 | 保持字段语义稳定 |
| 请求新增必填字段 | 不兼容 | 老消费者无法调用 | 先新增可选字段，再逐步强制 |
| 删除响应字段 | 不兼容 | 老消费者解析失败或逻辑错误 | 标记 deprecated，统计使用方后删除 |
| 字段类型变化 | 不兼容 | 反序列化失败 | 新增字段替代旧字段 |
| 字段含义变化 | 高风险不兼容 | 表面不报错但业务错 | 新增版本或新字段 |
| 错误码变化 | 可能不兼容 | 消费者重试和提示逻辑错误 | 错误码作为契约治理 |
| 事件新增字段 | 通常兼容 | 订阅方忽略 | 保持 eventVersion |
| 事件改名 | 不兼容 | 订阅方收不到 | 新旧事件并行一段时间 |

### 契约测试

契约测试用于验证提供者和消费者对 API 的理解是否一致。它不替代单元测试和集成测试，而是专门防止“服务 A 改了接口，服务 B 到线上才发现坏了”。

契约测试关注：

- 请求字段是否仍被接受。
- 响应字段和类型是否符合消费者期望。
- 错误码是否稳定。
- 认证和 header 是否符合约定。
- 事件 schema 是否向后兼容。

> **重点：** 契约测试应该进入 CI，而不是发布后人工联调。

### 幂等设计

幂等是指同一个请求重复执行多次，结果与执行一次一致。微服务中重试、网络抖动、消息重复投递都会导致重复请求。

常见做法：

- 使用业务唯一键，例如订单号、支付流水号。
- 使用幂等键 `Idempotency-Key`。
- 数据库唯一索引兜底。
- 消息消费记录表去重。
- 状态机只允许合法状态迁移。

### 错误分类和重试规则

| 错误类型 | 示例 | 是否重试 | 说明 |
| --- | --- | --- | --- |
| 参数错误 | 缺少必填字段 | 不重试 | 调用方修正请求 |
| 权限错误 | token 无效 | 不重试 | 重新认证或拒绝 |
| 业务冲突 | 重复投递 | 不重试或返回已有结果 | 依赖业务幂等语义 |
| 临时网络错误 | 连接重置 | 可重试 | 必须有幂等和退避 |
| 下游限流 | 429 | 谨慎重试 | 遵守 Retry-After |
| 下游过载 | 503 | 谨慎重试 | 过度重试会放大故障 |
| 未知 500 | 服务异常 | 少量重试 | 结合熔断和告警 |

调用方不能只按 HTTP 状态码粗暴处理。例如 409 可能表示业务冲突，也可能表示幂等请求已经处理过；429 不是让所有客户端立即重试，而是要求退避。

## 例子

HTTP 幂等请求示例：

```http
POST /api/orders
Idempotency-Key: create-order-20260606-0001
Content-Type: application/json

{
  "candidateId": "c1001",
  "positionId": "p9001"
}
```

服务端处理伪代码：

```java
Order createOrder(CreateOrderRequest request, String idempotencyKey) {
    IdempotencyRecord record = idempotencyRepo.find(idempotencyKey);
    if (record != null) {
        return orderRepo.find(record.getResourceId());
    }

    Order order = orderRepo.insertIfAbsent(request.toOrder());
    idempotencyRepo.save(idempotencyKey, order.getId());
    return order;
}
```

## 练习

设计一个“创建投递申请”的 API 契约，写出：

- URL、Method、请求字段、响应字段。
- 成功和失败状态码。
- 幂等键规则。
- 超时和重试建议。
- 版本升级策略。

## 验收

- 能说明何时使用同步调用，何时使用异步事件。
- 能识别接口变更是否破坏兼容性。
- 能为关键写操作设计幂等机制。
- 能解释路由匹配和服务实例选择过程。

## 重点

- 契约是服务之间的边界，不是文档装饰。
- 所有跨服务写操作都要考虑重复请求。
- 路由、负载均衡、超时和重试是同一条调用链上的联动策略。

## 难点

- 异步事件不是“无依赖”，只是把时间维度和一致性模型改变了。
- API 兼容性需要从消费者角度判断，不是只看服务端能不能运行。

## 易错

> **易错：** 重试所有失败请求。
>
> 正确做法：只重试可安全重试的请求，并配合幂等键、退避策略和最大次数。

> **易错：** 事件只写 `OrderUpdated`，不说明变化原因和关键字段。
>
> 正确做法：事件名表达业务事实，例如 `OrderPaid`、`OrderCancelled`，并提供稳定 schema。

> **易错：** 把内部实体对象直接作为 API 响应。
>
> 正确做法：API DTO 是对外契约，应和内部领域模型解耦，避免内部字段变更破坏消费者。
