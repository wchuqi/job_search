# 微服务架构学习资料：生产故障推演和排障手册

[返回索引](../微服务架构学习资料.md)

## 学习目标

- 掌握微服务生产故障的推演和定位方法。
- 能建立从症状到根因的证据链。
- 能区分止血、恢复、根因修复和复盘。

## 排障总原则

1. 先确认用户影响，再定位技术细节。
2. 先止血，再深挖根因。
3. 用证据链定位，不靠猜测。
4. 每个动作都要可回滚。
5. 复盘要转化为监控、测试、流程或架构改进。

## 故障 1：下单接口错误率升高

### 现象

- 网关 `/api/orders` 5xx 从 0.1% 升到 8%。
- 只影响 v2 灰度用户。
- P99 从 500ms 升到 2.5s。

### 排查路径

1. 按 version 维度查看错误率，确认集中在 order-service v2。
2. 查最近发布，确认 v2 增加了优惠券校验调用。
3. 用 trace 查看慢 span，发现 coupon-service 平均耗时 1.8s。
4. 看 order-service 线程池，发现等待队列升高。
5. 看 coupon-service，发现数据库慢查询。

### 止血

- 暂停 v2 灰度。
- 对优惠券校验降级或短超时。
- 必要时临时关闭优惠券入口。

### 根因修复

- 优惠券查询加索引或缓存。
- order-service 对 coupon-service 设置 300ms 超时。
- 优惠券从 P0 主链路降为 P1 可降级依赖。
- 增加发布前压测和灰度 SLO 自动回滚。

## 故障 2：消息堆积导致通知延迟

### 现象

- notification topic 堆积 200 万条。
- 投递成功但用户长时间收不到邮件。
- 下单主流程成功率正常。

### 排查路径

1. 看消费速率和生产速率，确认消费能力不足。
2. 查看消费者错误日志，发现第三方邮件 API 限流。
3. 查看 DLQ，确认大量 `RATE_LIMITED`。
4. 查看重试策略，发现立即重试造成放大。

### 止血

- 暂停低优先级通知。
- 对第三方 API 增加退避和速率限制。
- 临时扩容消费者，但不能超过供应商限额。

### 根因修复

- 通知按优先级分 topic。
- 按供应商限额做令牌桶。
- 失败重试使用指数退避。
- 增加通知延迟 SLO 和堆积告警。

## 故障 3：服务下线导致请求中断

### 现象

- 滚动发布期间出现大量连接重置。
- 错误集中在被替换实例。

### 排查路径

1. 查看实例生命周期事件。
2. 查看 readiness 变化和流量摘除时间。
3. 查看客户端负载均衡缓存刷新间隔。
4. 查看应用关闭钩子，发现进程收到终止信号后立即退出。

### 止血

- 暂停发布。
- 延长 termination grace period。
- 手动摘除异常实例。

### 根因修复

- 下线前 readiness 置为 false。
- 等待负载均衡和注册发现缓存刷新。
- 停止接收新请求，等待存量请求完成。
- 消息消费者先停止拉取新消息，再完成当前处理。

## 故障 4：缓存雪崩导致数据库打满

### 现象

- 数据库 CPU 100%。
- 多个服务读延迟升高。
- 缓存命中率突然下降。

### 排查路径

1. 查看缓存命中率和 key 过期分布。
2. 查看数据库慢查询和连接池等待。
3. 查看是否有批量缓存失效或发布清缓存动作。
4. 查看热点 key 是否集中。

### 止血

- 限流非核心读请求。
- 临时延长热点缓存。
- 对不存在数据加入空值缓存。
- 启用降级读模型。

### 根因修复

- 缓存 TTL 加随机抖动。
- 热点 key 提前刷新。
- 缓存重建加互斥锁。
- 数据库连接池和查询索引优化。

## 故障 5：跨服务数据不一致

### 现象

- 支付成功，但课程权益未开通。
- 用户投诉集中在某一时间段。

### 排查路径

1. 查 payment-service 支付流水。
2. 查 outbox 事件是否生成。
3. 查消息是否发布。
4. 查 entitlement-service 消费记录。
5. 查是否进入 DLQ。
6. 查对账任务是否发现异常。

### 止血

- 对受影响订单人工或批量补开权益。
- 暂停相关发布。
- 保留证据，避免重复开通。

### 根因修复

- 修复 Outbox 发布器。
- 消费者加入幂等和重试。
- 对账任务缩短周期。
- 用户可见状态增加“开通中”和“异常处理中”。

## 通用排障清单

| 检查项 | 具体问题 |
| --- | --- |
| 影响范围 | 哪些接口、租户、用户、地区、版本受影响？ |
| 时间线 | 从什么时候开始？是否对应发布或配置变更？ |
| 入口指标 | QPS、错误率、P95/P99 是否异常？ |
| 调用链 | 慢 span 或失败 span 在哪里？ |
| 实例维度 | 是否集中在某个实例、版本、可用区？ |
| 依赖维度 | 数据库、缓存、队列、第三方是否异常？ |
| 资源饱和 | CPU、内存、线程池、连接池、队列是否满？ |
| 数据一致性 | 是否有未发布事件、未消费消息、补偿失败？ |
| 止血动作 | 回滚、限流、降级、扩容、摘实例是否可行？ |
| 复盘动作 | 监控、测试、流程、架构如何改？ |

## 练习

选择一个故障场景，写出：

- 现象。
- 初始假设。
- 需要收集的证据。
- 止血方案。
- 根因修复。
- 复盘改进。

## 验收

- 能按影响范围、时间线、证据链排障。
- 能把止血和根因修复分开。
- 能从一次故障反推架构改进。
- 能识别监控缺口和流程缺口。

## 重点

- 生产排障先保用户影响，再做根因分析。
- trace、日志、指标、发布记录要能串起来。
- 微服务故障经常是多个小问题叠加。

## 难点

- 局部故障可能只影响某个版本、租户或可用区，聚合指标会掩盖问题。
- 止血动作本身也可能扩大影响，所以必须提前演练和验证。

## 易错

> **易错：** 复盘只写“加强测试”“加强监控”。
>
> 正确做法：明确补哪类测试、补哪个指标、谁负责、什么时候完成、如何验收。
