## 一、架构设计场景

### 1. 让你设计一个电商下单微服务链路，你会怎么拆？

**参考答案：**

可以拆成商品、库存、订单、支付、优惠、通知等服务。订单服务负责订单生命周期，库存服务负责库存预留和释放，支付服务负责支付流水和回调，通知服务异步发送消息。下单主链路应控制同步依赖数量，非核心动作例如通知、积分可以异步处理。

关键设计包括：写接口幂等、库存预留状态、支付回调去重、订单 Saga、Outbox 事件、失败补偿和对账。

> **重点：** 不要只列服务名，要说明每个服务的数据所有权和失败处理。

### 2. 微服务之间应该用同步调用还是异步消息？

**参考答案：**

取决于业务对实时性、一致性和失败隔离的要求。需要立即返回结果、强依赖下游判断的场景适合同步调用；需要削峰、解耦、允许延迟处理的场景适合异步消息。核心链路要减少同步调用层级，异步链路要处理重复消费、顺序、堆积和最终一致性。

> **易错：** 异步不是没有问题，它把问题从等待变成延迟、一致性和补偿。

### 3. 跨服务事务怎么处理？

**参考答案：**

首先判断业务是否真的需要强一致。多数微服务场景采用最终一致，通过 Saga、Outbox、消息重试、消费幂等、补偿和对账实现。Saga 负责多步骤流程和补偿，Outbox 保证本地业务数据和待发布事件原子写入，消费者通过事件 ID 或业务唯一键去重。

> **重点：** 面试中要说出“状态记录、幂等、补偿、对账”，否则答案容易停留在概念层。

### 4. 如何设计 API 契约版本？

**参考答案：**

新增响应字段通常兼容，删除字段、改变字段含义、改变错误码通常不兼容。应明确请求字段必填规则、默认值、错误码、版本号和废弃窗口。多服务协作时要通过契约测试验证消费者和提供者是否兼容。事件也要有 schemaVersion，避免订阅方被破坏。

> **难点：** API 是否兼容要从消费者视角判断。

### 5. 如何从单体逐步迁移到微服务？

**参考答案：**

先梳理模块边界和依赖，优先拆变化频繁、边界清晰、风险可控的能力。使用 Strangler Fig 思路，新功能走新服务，旧功能逐步迁移。新旧系统之间加反腐层，避免旧模型污染新服务。数据迁移要分阶段，保持接口和数据库变更可回滚。

> **重点：** 不建议一次性重写。渐进式迁移更符合真实工程风险控制。
