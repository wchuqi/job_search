# 微服务架构学习资料：服务拆分和边界设计

[返回索引](../微服务架构学习资料.md)

## 学习目标

- 掌握按业务能力拆分服务的方法。
- 理解限界上下文、聚合、上下文映射和数据所有权。
- 能识别错误拆分和过度拆分。

## 理论导读

服务边界是微服务架构中最关键、也最难返工的决策。拆得太粗，仍然会被大服务拖慢；拆得太细，调用链、事务、团队协作和排障成本会急剧上升。好的边界应该让服务内部高内聚、服务之间低耦合，并让一个团队可以对该服务的业务结果负责。

领域驱动设计中的限界上下文可以帮助拆分。限界上下文不是简单的模块名，而是一组业务概念在特定语境下的含义边界。例如“候选人”在招聘投递上下文中代表求职者资料，在合同结算上下文中可能代表可计费对象，两者字段、规则和生命周期不一定相同。

## 核心心智模型

把服务想成一个有明确职责的业务柜台。每个柜台拥有自己的账本和规则，其他柜台不能直接修改它的账本，只能通过公开窗口提交请求或订阅它发出的事件。

## 知识点详解

### 拆分依据

| 拆分依据 | 是否推荐 | 原因 |
| --- | --- | --- |
| 业务能力 | 推荐 | 与团队职责、数据所有权和变更频率一致 |
| 限界上下文 | 推荐 | 能处理同一概念在不同业务语境下的差异 |
| 数据表 | 不推荐作为唯一依据 | 表结构通常反映实现细节，容易造成贫血服务 |
| 技术层 | 不推荐 | 会形成跨服务分层调用，增加网络成本 |
| 团队结构 | 可参考 | 团队边界会影响服务边界，但不能替代业务建模 |

### 深度判断：服务边界不是“名词分类”，而是“变化边界”

很多初学者会把“订单、商品、用户”这些名词直接映射成服务，但真实系统里更可靠的拆分依据是变化原因。一个服务内部的代码、数据和规则应该因为同一类业务原因一起变化；如果两个模块总是在不同原因下变化，它们适合分开；如果两个模块每次都同时变化，强行拆开会制造分布式单体。

判断一个候选边界时，可以用下面的权重打分：

| 维度 | 问题 | 偏向拆分的信号 | 偏向合并的信号 |
| --- | --- | --- | --- |
| 业务规则 | 两块能力的规则是否不同源？ | 由不同业务团队定义 | 同一个业务流程一步都不能少 |
| 变更频率 | 是否经常独立变化？ | 一边频繁改，一边稳定 | 每次需求都一起改 |
| 数据所有权 | 是否能明确谁拥有数据？ | 数据可由单一服务权威维护 | 多方都要强一致写同一组数据 |
| 性能特征 | 负载是否差异明显？ | 一个高并发，一个低频 | 负载模式接近且互相强依赖 |
| 故障影响 | 是否需要隔离故障？ | 非核心能力失败不应影响核心链路 | 失败必须作为整体失败处理 |
| 团队边界 | 是否有独立负责人？ | 不同团队可独立负责结果 | 无人能承担独立服务运维 |

> **重点：** 这个表不是数学公式，而是架构评审中的证据清单。分数高不代表立刻拆，分数低也不代表永远不拆。

### 服务边界判断问题

- 这个服务对外提供什么业务能力？
- 它拥有哪些数据？其他服务是否能直接写这些数据？
- 它的核心业务状态如何变化？
- 它依赖哪些上游和下游？
- 它失败时影响哪些用户旅程？
- 它是否可以独立部署而不要求其他服务同步发布？
- 它的变化频率是否与其他能力明显不同？

### 深度机制：限界上下文和聚合的关系

限界上下文定义语言和模型的边界，聚合定义一致性边界。一个限界上下文里可以有多个聚合，但一个聚合内部通常要求本地事务保护。微服务拆分时经常把“限界上下文”作为服务候选边界，把“聚合”作为服务内部事务边界。

例子：

- 投递上下文里有“投递申请”聚合，维护投递状态从 `SUBMITTED` 到 `SCREENING`、`INTERVIEWING`、`OFFERED`、`REJECTED` 的合法流转。
- 职位上下文里有“职位”聚合，维护职位是否开放、岗位人数、招聘负责人等规则。
- 投递服务可以读取职位状态，但不能直接修改职位聚合。

如果“投递申请”和“职位”必须在一个数据库事务里同时更新，说明要么服务拆早了，要么业务规则还没有被重新建模。例如可以把“职位开放状态校验”变成调用或快照，把“投递人数统计”变成异步读模型，而不是跨服务强事务。

### 上下文映射

常见上下文关系：

- 客户方/供应方：下游依赖上游接口，上游需要稳定契约。
- 反腐层：在新系统和旧系统之间转换模型，避免旧模型污染新边界。
- 共享内核：多个上下文共享少量模型，需要严格治理。
- 开放主机服务：某上下文提供标准 API 给多个下游。
- 发布语言：团队约定统一术语，减少同名异义。

## 例子

在线招聘系统可以初步拆成：

- 账号服务：登录、认证、用户基础身份。
- 候选人服务：简历、求职意向、候选人资料。
- 职位服务：职位发布、职位状态、招聘需求。
- 投递服务：投递记录、投递状态流转。
- 面试服务：面试安排、评价、结果。
- 通知服务：短信、邮件、站内信。

其中投递服务不应该直接修改职位服务的职位表，只能调用职位服务检查职位状态，或订阅职位关闭事件后拒绝新投递。

### 深度例子：错误拆分如何演变成分布式单体

假设把招聘系统拆成 `resume-service`、`application-service`、`interview-service`，但三者共用 `candidate` 表，并且 `application-service` 创建投递时必须直接更新简历完整度字段，`interview-service` 安排面试时必须直接改投递状态。

表面上有三个服务，实际问题是：

- 数据所有权不清：任何服务都能改候选人相关数据。
- 发布耦合：字段变更会同时影响三个服务。
- 排障困难：候选人状态异常时无法判断谁写坏了数据。
- 事务幻觉：开发者以为共享表简单，实际在并发写入、锁竞争和回滚时更复杂。

修正方式：

- candidate-service 拥有简历和候选人资料。
- application-service 拥有投递状态。
- interview-service 拥有面试安排。
- 跨服务只通过 API 或事件更新事实，例如 `ApplicationSubmitted`、`InterviewScheduled`。
- 如果需要候选人完整度，application-service 可调用 candidate-service 查询，或维护只读快照。

## 练习

为“在线教育平台”拆分服务，至少包含课程、订单、支付、学习进度、作业、通知。为每个服务写出：

- 业务职责。
- 数据所有权。
- 对外 API 或事件。
- 不能负责的事项。
- 失败影响范围。

## 验收

- 能为每个服务写出一句清晰职责。
- 能说明为什么某两个能力应该合并或拆开。
- 能识别跨服务共享数据库带来的耦合。
- 能用上下文映射描述新旧系统迁移关系。

## 重点

- 服务拥有自己的业务规则和数据。
- 服务之间通过契约协作，不通过共享表协作。
- 边界要围绕变更原因，而不是围绕代码目录。

## 难点

- 同一实体在不同上下文含义不同，不能强行抽成一个“通用对象”。
- 边界不是一次确定的，需要通过业务变化、故障和发布经验修正。

## 易错

> **易错：** 所有服务共用一套 `common-domain`，里面放满核心实体。
>
> 正确做法：共享库只放稳定、无业务状态的基础能力；业务模型应属于具体服务。

> **易错：** 一个服务拆出来后仍然必须和其他 5 个服务同时发布。
>
> 正确做法：通过契约版本、向后兼容、事件解耦和灰度发布降低同步发布需求。

> **易错：** 只画服务框图，不画数据所有权和状态流转。
>
> 正确做法：服务边界评审至少要同时看服务职责、数据所有权、核心状态机、接口契约和依赖方向。
