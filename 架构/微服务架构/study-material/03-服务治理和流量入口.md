# 微服务架构学习资料：服务治理和流量入口

[返回索引](../微服务架构学习资料.md)

## 学习目标

- 理解 API Gateway、注册发现、负载均衡和服务网格的责任边界。
- 掌握流量入口、认证、路由、限流和灰度的基本治理策略。
- 能解释服务实例选择、下线和灰度发布中的边缘情况。

## 理论导读

服务拆多以后，调用方需要知道服务在哪里、哪些实例健康、请求应该走哪条路由、如何鉴权、如何限流以及如何灰度。服务治理就是对这些跨服务协作规则进行统一管理。

API Gateway 通常负责外部流量入口，包括认证、路由、协议转换、限流、聚合和审计。服务注册发现解决“服务名到实例列表”的问题。负载均衡解决“从可用实例中选择哪个”的问题。服务网格则把一部分治理能力下沉到 sidecar 或数据平面中，适合更大规模和多语言场景。

## 核心心智模型

网关像园区大门，负责外来访客的身份检查和路线指引。注册中心像楼宇目录，记录每栋楼的地址和健康状态。负载均衡像调度员，在多个可用窗口之间分配请求。

## 知识点详解

### API Gateway 职责

- 外部统一入口。
- 认证和基础授权。
- 路由匹配和路径改写。
- 限流和黑白名单。
- CORS 和协议适配。
- 请求日志、审计和 traceId 注入。
- 灰度路由，例如按用户、Header、租户或比例分流。

### 注册发现生命周期

1. 服务实例启动后向注册中心注册，提交 serviceId、host、port、metadata、health 信息。
2. 注册中心通过心跳或健康检查维护实例状态。
3. 调用方或网关按 serviceId 拉取实例列表，并缓存一段时间。
4. 负载均衡器根据算法和过滤条件选择实例。
5. 实例下线时先停止接收新流量，再等待存量请求完成，最后注销。

> **难点：** 注册中心显示实例已下线，不代表所有客户端缓存立即更新。优雅下线要考虑缓存、连接池和负载均衡刷新间隔。

### 深度机制：健康检查不是一个布尔值

健康状态至少要区分：

- liveness：进程是否还活着，失败后可重启。
- readiness：是否可以接收流量，失败后应从流量池摘除。
- dependency health：依赖是否健康，用于诊断，不一定用于重启。
- startup：启动是否完成，避免慢启动时被误杀。

错误做法是把所有依赖都放进 liveness。例如数据库短暂异常时，如果 liveness 失败导致服务重启，会让连接、缓存和实例稳定性更差。更合理的方式是：liveness 只检查进程关键状态，readiness 检查服务是否具备处理请求的能力。

### 负载均衡选择规则

常见选择因素：

- 健康状态：只选择健康实例。
- 权重：新版本或高配置实例可设置不同权重。
- 区域：优先选择同机房、同可用区实例。
- 负载：避开连接数、延迟或错误率过高的实例。
- 灰度标签：根据版本、租户、用户或请求 Header 选择实例。

### 负载均衡算法和适用场景

| 算法 | 机制 | 适用场景 | 风险 |
| --- | --- | --- | --- |
| Round Robin | 轮询实例 | 实例能力接近 | 不考虑实时负载 |
| Weighted | 按权重分配 | 实例规格不同或灰度 | 权重配置错误会打爆实例 |
| Least Connections | 选连接少的实例 | 长连接或请求耗时差异大 | 需要实时连接信息 |
| Random | 随机选择 | 简单高效 | 小流量下分布不稳定 |
| Consistent Hash | 按 key 固定路由 | 会话、缓存亲和 | 扩缩容会迁移部分 key |
| Locality Aware | 优先同区域 | 多机房低延迟 | 区域故障时要能跨区 |

灰度发布时，负载均衡不是单独工作，而是先过滤出符合灰度条件的实例，再在候选实例中选择。过滤顺序通常是健康状态、区域、版本/标签、权重、算法选择。

### Gateway、Ingress、Service Mesh 对比

| 能力 | API Gateway | Kubernetes Ingress | Service Mesh |
| --- | --- | --- | --- |
| 外部入口 | 强 | 中 | 通常不是主职责 |
| 业务路由 | 强 | 中 | 中 |
| 服务间治理 | 中 | 弱 | 强 |
| 多语言透明治理 | 中 | 弱 | 强 |
| 运维复杂度 | 中 | 中 | 高 |

### 限流维度

限流不是只设置一个全局 QPS。常见维度包括：

- 按接口：保护高成本接口。
- 按用户：防止单用户刷接口。
- 按租户：保障多租户公平。
- 按 IP：防止入口攻击。
- 按服务：保护下游服务。
- 按资源：保护数据库连接、队列和第三方额度。

限流后的返回也要设计：

- 对外 API 返回明确错误码和重试建议。
- 内部调用可触发降级。
- 高优先级请求应保留配额。
- 低优先级任务可以排队或丢弃。

> **重点：** 限流保护的是瓶颈资源和核心业务，不是为了让指标看起来整齐。

## 例子

灰度路由规则：

```yaml
routes:
  - id: order-v2-canary
    match:
      path: /api/orders/**
      header:
        name: X-Canary
        value: "true"
    target: order-service-v2
  - id: order-v1-default
    match:
      path: /api/orders/**
    target: order-service-v1
```

这类规则必须保证更具体的灰度路由优先于默认路由。

## 练习

为一个有订单、支付、库存服务的系统设计流量入口：

- 哪些接口暴露给外部？
- 哪些服务只允许内部访问？
- 哪些接口要限流？
- 如何按租户或用户灰度？
- 服务下线前如何排空流量？

## 验收

- 能画出从客户端到网关再到服务实例的调用路径。
- 能解释服务实例选择的过滤和排序规则。
- 能说明 Gateway、Ingress、Service Mesh 的区别。
- 能设计优雅下线和灰度路由。

## 重点

- 网关是外部入口治理，不应承载大量业务逻辑。
- 注册发现和负载均衡要配合健康检查、缓存和下线策略。
- 灰度规则要可观测、可回滚、可验证。

## 易错

> **易错：** 把所有业务判断放在网关里。
>
> 正确做法：网关处理横切能力，核心业务规则仍应留在业务服务中。

> **易错：** 服务直接杀进程下线。
>
> 正确做法：先从流量池摘除，再等待请求完成，最后停止进程。

> **易错：** 灰度只按流量比例做，不绑定用户。
>
> 正确做法：用户体验相关功能要尽量按用户、租户或会话稳定分流，否则同一用户可能一会儿看到 v1，一会儿看到 v2。
