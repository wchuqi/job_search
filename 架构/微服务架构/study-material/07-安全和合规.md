# 微服务架构学习资料：安全和合规

[返回索引](../微服务架构学习资料.md)

## 学习目标

- 理解微服务中的身份认证、授权、服务间信任和密钥管理。
- 掌握最小权限、零信任、审计和敏感数据保护。
- 能识别网关认证和内部授权的边界。

## 理论导读

服务拆分后，安全边界从“一个应用内部”变成“多个服务之间的网络协作”。外部用户要认证，服务之间也要证明身份；网关能做入口认证，但不能替代业务服务内部的细粒度授权。安全设计应默认网络不可信、服务不天然可信、令牌可能泄漏、权限需要最小化。

## 核心心智模型

微服务安全像办公楼门禁加内部权限。大门验证来访者身份，但不能因为进了大门就允许进入财务室、机房和档案室。每个敏感区域仍要检查权限，并留下访问记录。

## 知识点详解

### 认证和授权

- 认证：确认“你是谁”，例如用户名密码、OAuth2、OIDC、JWT。
- 授权：确认“你能做什么”，例如角色、权限、资源归属、租户边界。
- 网关认证：适合统一验证 token、基础拦截和用户上下文注入。
- 服务内授权：适合业务规则，例如只能查看自己的订单、HR 只能看本公司候选人。

### 服务间安全

常见方案：

- 内部网络隔离和安全组。
- mTLS 验证服务身份。
- 服务账号和短期凭证。
- Token Relay 或 Token Exchange。
- Secret 管理系统，避免密钥写入代码库。

### 敏感数据

- 传输加密：TLS。
- 存储加密：数据库、对象存储、备份。
- 脱敏展示：手机号、身份证、邮箱。
- 日志脱敏：避免 token、密码、证件号进入日志。
- 审计：记录谁在什么时候访问了哪些敏感资源。

## 例子

内部接口授权伪代码：

```java
boolean canViewCandidate(UserContext user, Candidate candidate) {
    if (!user.hasPermission("candidate:read")) {
        return false;
    }
    return user.getTenantId().equals(candidate.getTenantId());
}
```

## 练习

为“招聘系统候选人简历查看”设计权限：

- 外部用户如何认证？
- HR、面试官、管理员权限如何区分？
- 服务之间如何传递用户和租户上下文？
- 日志中哪些字段必须脱敏？
- 哪些操作需要审计？

## 验收

- 能区分认证和授权。
- 能说明网关认证不能替代服务内授权。
- 能设计服务间身份验证。
- 能列出敏感数据保护和审计要求。

## 重点

- 微服务内部网络不应被视为完全可信。
- 权限校验必须靠近业务资源。
- Secret 不应进入代码、镜像和普通配置文件。

## 难点

- Token 传播要避免权限扩大。服务间调用时要明确代表用户、代表服务，还是二者结合。
- 多租户场景下，租户边界常比角色本身更关键。

## 易错

> **易错：** 网关校验登录后，内部服务不再校验资源权限。
>
> 正确做法：网关做入口认证，业务服务做资源级授权。

> **易错：** 把 JWT、数据库密码和第三方密钥打印到日志。
>
> 正确做法：日志脱敏，密钥由 Secret 管理系统注入，并定期轮换。
