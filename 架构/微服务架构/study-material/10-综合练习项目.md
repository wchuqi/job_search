# 微服务架构学习资料：综合练习项目

[返回索引](../微服务架构学习资料.md)

## 项目目标

设计并实现一个“招聘投递平台”的微服务架构练习，覆盖服务边界、API 契约、事件驱动、数据一致性、服务治理、可观测性、安全和发布。

## 推荐服务

- account-service：账号、登录、用户身份。
- candidate-service：候选人资料、简历。
- job-service：职位发布、职位状态。
- application-service：投递记录、投递状态流转。
- interview-service：面试安排和评价。
- notification-service：短信、邮件、站内信。
- gateway：统一入口、认证、路由、限流。

## 核心流程

1. HR 发布职位。
2. 候选人完善简历。
3. 候选人投递职位。
4. 投递服务校验职位状态。
5. 投递创建成功后发布 `ApplicationSubmitted` 事件。
6. 通知服务消费事件，发送通知。
7. 面试服务创建面试安排。

## 必做任务

### 任务 1：服务边界

- 输出服务职责表。
- 输出每个服务拥有的数据。
- 输出上下文关系图。

验收：

- 没有服务直接写其他服务的数据表。
- 每个服务有清晰的业务能力。

### 任务 2：API 契约

- 设计投递创建 API。
- 设计职位查询 API。
- 设计投递状态变更 API。
- 为写接口加入幂等键。

验收：

- 契约包含状态码、错误码、版本和兼容策略。

### 任务 3：事件和一致性

- 设计 `ApplicationSubmitted`、`InterviewScheduled`、`NotificationSent` 事件。
- 使用 Outbox 保存待发送事件。
- 消费端使用事件 ID 去重。

验收：

- 能模拟消息重复投递，不产生重复通知。

### 任务 4：弹性策略

- 为 gateway 到 application-service 设置限流。
- 为 application-service 到 job-service 设置超时和熔断。
- notification-service 异步处理，不阻塞投递主链路。

验收：

- job-service 不可用时，投递服务能按业务策略返回明确错误或降级结果。

### 任务 5：可观测性

- 所有请求带 traceId。
- 日志记录 service、version、operation、status、durationMs。
- 建立核心 dashboard。

验收：

- 能通过 traceId 查到一次投递请求经过的所有服务。

### 任务 6：发布和回滚

- 设计 application-service v2 灰度。
- 新增字段时保持向后兼容。
- 设定错误率和 P99 回滚阈值。

验收：

- v1 和 v2 能短期共存。
- 回滚后旧版本仍能处理新版本写入的数据。

## 扩展任务

- 引入 CQRS，为 HR 提供跨职位的投递统计读模型。
- 引入服务目录，记录服务负责人、SLO 和依赖。
- 做一次故障演练：通知服务堆积、职位服务超时、网关限流误配置。

## 最终验收

- 能画出整体架构图。
- 能说明每个服务的边界和数据所有权。
- 能演示一次投递链路。
- 能演示消息重复消费的幂等处理。
- 能通过日志、指标、trace 定位故障。
- 能给出灰度和回滚方案。
