# 微服务架构学习资料：服务边界深度推演

[返回索引](../微服务架构学习资料.md)

## 学习目标

- 从业务流程推导服务边界，而不是从名词列表拆服务。
- 掌握事件风暴、状态机、依赖方向和数据所有权分析。
- 能识别边界设计中的深层耦合。

## 案例：招聘投递平台

核心流程：

1. HR 发布职位。
2. 候选人创建简历。
3. 候选人投递职位。
4. 系统校验职位开放、候选人资格和重复投递。
5. HR 筛选投递。
6. 安排面试。
7. 面试官提交评价。
8. HR 发 offer 或拒绝。
9. 系统发送通知。

## 第一步：找领域事件

领域事件表达已经发生的业务事实：

- `JobPublished`
- `JobClosed`
- `ResumeCompleted`
- `ApplicationSubmitted`
- `ApplicationScreened`
- `InterviewScheduled`
- `InterviewFeedbackSubmitted`
- `OfferIssued`
- `ApplicationRejected`
- `NotificationRequested`

事件风暴时不要先画服务，先贴出事件、命令、参与者、规则和外部系统。

## 第二步：识别命令和规则

| 命令 | 触发者 | 关键规则 |
| --- | --- | --- |
| 发布职位 | HR | HR 必须属于该公司，职位字段完整 |
| 提交投递 | 候选人 | 职位开放、不能重复投递、简历满足最低完整度 |
| 安排面试 | HR | 投递状态必须进入面试阶段，面试官可用 |
| 提交评价 | 面试官 | 面试官必须被分配到该面试 |
| 发放 Offer | HR | 投递状态必须允许发 offer |

规则来源不同，是拆分线索。职位规则、投递规则、面试规则来自不同业务负责人时，更适合作为不同上下文。

## 第三步：画状态机

投递状态机：

```text
SUBMITTED
  -> SCREENING
  -> INTERVIEWING
  -> OFFERED
  -> HIRED

SUBMITTED / SCREENING / INTERVIEWING
  -> REJECTED

任意终态不能回到处理中状态，除非有明确 reopen 流程。
```

状态机通常属于一个服务。不要让多个服务分别修改同一个状态字段，否则状态合法性会分散。

## 第四步：确定数据所有权

| 数据 | 所有者 | 其他服务如何使用 |
| --- | --- | --- |
| 职位基本信息和状态 | job-service | API 查询或职位事件同步快照 |
| 简历和候选人资料 | candidate-service | API 查询或只读快照 |
| 投递记录和投递状态 | application-service | 事件通知其他服务 |
| 面试安排和评价 | interview-service | 订阅投递状态，发布面试事件 |
| 通知发送记录 | notification-service | 消费通知请求事件 |

## 第五步：识别耦合

### 强耦合信号

- A 服务需要直接写 B 服务的表。
- A 服务改字段，B 服务必须同日发布。
- A 服务的一个请求同步调用 5 个服务才能完成。
- 多个服务共享一个核心状态枚举。
- 业务错误码没有所有者，多个服务随意定义。

### 合理依赖方向

- 投递服务可以依赖职位服务的“职位是否开放”能力。
- 职位服务不应该依赖投递服务来决定自己是否能发布。
- 通知服务应该依赖事件，不应该被投递主流程同步等待。

## 第六步：处理跨服务查询

HR 查看候选人投递列表，需要职位、候选人、投递和面试信息。不要让前端直接调用所有服务拼装复杂页面，也不要让 application-service 直接读所有表。

可选方案：

1. API 聚合：由 BFF 或查询服务调用多个服务聚合，适合低频实时查询。
2. 读模型：订阅事件维护宽表，适合高频列表和报表。
3. 搜索索引：维护 Elasticsearch 等索引，适合筛选和搜索。

权衡：

- API 聚合实时性好，但延迟和依赖多。
- 读模型性能好，但有同步延迟。
- 搜索索引查询强，但一致性和运维成本更高。

## 练习

为“在线教育平台”做同样推演：

- 列领域事件。
- 列命令和规则。
- 画课程购买状态机。
- 确定课程、订单、支付、权益、学习进度的数据所有权。
- 设计跨服务查询方案。

## 验收

- 能从事件和规则推导服务，而不是先拍服务名。
- 能说明状态机归属哪个服务。
- 能为跨服务查询选择 API 聚合、读模型或搜索索引。
- 能指出共享表、共享枚举和同步链路过长的问题。

## 重点

- 服务边界来自业务行为、规则和状态，而不是实体名。
- 数据所有权必须和状态机放在一起看。
- 查询模型不一定等于写模型。

## 难点

- 一个业务名词可能跨多个上下文，但含义不同。
- 读模型会引入延迟，需要业务接受并设计提示或刷新机制。

## 易错

> **易错：** 为了避免重复数据，强行所有服务都实时查同一个权威服务。
>
> 正确做法：重复只读快照可以接受，关键是明确来源、版本、同步延迟和修复机制。
