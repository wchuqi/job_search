# 微服务架构学习资料：部署发布和平台工程

[返回索引](../微服务架构学习资料.md)

## 学习目标

- 掌握微服务交付、发布、回滚和环境治理。
- 理解容器、Kubernetes、CI/CD、配置和服务目录的作用。
- 能设计灰度、蓝绿、金丝雀和回滚方案。

## 理论导读

微服务让服务可以独立部署，但也要求团队具备成熟的交付能力。几十个服务如果没有统一构建、测试、发布、配置、监控和回滚流程，独立部署会变成混乱部署。平台工程的目标是把通用能力产品化，让业务团队以低认知成本完成发布和治理。

## 核心心智模型

平台像机场运行系统。每架飞机可以独立起降，但航线、塔台、安检、跑道、维护和应急流程必须标准化，否则独立性会带来事故风险。

## 知识点详解

### 交付流水线

典型阶段：

1. 代码提交。
2. 静态检查和单元测试。
3. 契约测试和集成测试。
4. 构建镜像。
5. 扫描漏洞和依赖风险。
6. 部署到测试环境。
7. 灰度发布。
8. 指标验证。
9. 全量发布或回滚。

### 发布策略

| 策略 | 说明 | 适用场景 | 风险 |
| --- | --- | --- | --- |
| 滚动发布 | 逐步替换实例 | 常规发布 | 新旧版本短暂共存 |
| 蓝绿发布 | 两套环境切换 | 需要快速回滚 | 成本较高 |
| 金丝雀发布 | 小比例流量验证 | 高风险变更 | 需要精细流量控制 |
| 灰度发布 | 按用户、租户、地区放量 | 业务验证 | 规则复杂 |

### 深度机制：数据库变更的 expand-contract 策略

数据库变更最容易破坏回滚。推荐使用 expand-contract：

1. Expand：新增兼容结构，例如新增列、新增表、新增索引，不删除旧字段。
2. Dual write 或 backfill：新旧结构短期并存，回填历史数据。
3. Read switch：读取逻辑切到新结构。
4. Verify：对账确认新旧数据一致。
5. Contract：确认没有旧版本依赖后删除旧字段或旧表。

例子：把 `candidate.full_name` 拆成 `first_name` 和 `last_name`。

- 第一次发布：新增 `first_name`、`last_name`，写入时同时写新旧字段。
- 后台回填历史数据。
- 第二次发布：读取优先使用新字段，旧字段作为兜底。
- 确认所有服务升级后，停止写旧字段。
- 最后删除旧字段。

> **重点：** 应用回滚比数据库回滚容易得多，所以数据库变更要向前兼容和向后兼容。

### 配置管理

- 配置分层：默认配置、环境配置、服务配置、实例配置、Secret。
- 配置变更要有审计和回滚。
- 动态配置只适合部分参数，不适合所有启动期参数。
- 配置和代码版本要能关联，便于排查。

### 配置变更风险

配置不是低风险变更。限流阈值、路由规则、熔断参数、数据库连接池、消息消费并发都可能直接造成故障。

配置治理要求：

- 配置有 owner。
- 配置变更有审计。
- 高风险配置支持灰度。
- 配置生效范围明确。
- 配置能快速回滚。
- 关键配置变更触发事件，进入排障时间线。

### 平台能力成熟度

| 成熟度 | 表现 |
| --- | --- |
| L1 手工部署 | 发布靠人工命令，回滚靠经验 |
| L2 自动流水线 | 构建、测试、部署标准化 |
| L3 可观测发布 | 发布自动关联指标和 trace |
| L4 自动防护 | 灰度异常自动暂停或回滚 |
| L5 自服务平台 | 团队通过平台自助创建服务、配置、发布、监控 |

### 服务目录

服务目录应记录：

- 服务负责人。
- 服务职责。
- API 和事件契约。
- 依赖关系。
- SLO。
- 运行环境。
- dashboard 和告警。
- 最近发布版本。

## 例子

Kubernetes 部署关注点：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
  template:
    spec:
      containers:
        - name: order-service
          image: registry.example.com/order-service:1.8.0
          readinessProbe:
            httpGet:
              path: /actuator/health/readiness
              port: 8080
          livenessProbe:
            httpGet:
              path: /actuator/health/liveness
              port: 8080
```

## 练习

为订单服务设计一次高风险发布：

- 发布前检查清单。
- 灰度规则。
- 验证指标。
- 自动回滚条件。
- 数据库变更兼容方案。

## 验收

- 能说明不同发布策略的取舍。
- 能设计可回滚的数据库和 API 变更。
- 能说明 readiness 和 liveness 的区别。
- 能列出服务目录需要维护的信息。

## 重点

- 独立部署能力依赖自动化测试、契约治理和可观测性。
- 发布前必须知道如何回滚。
- 新旧版本共存是微服务发布中的常态。

## 难点

- 数据库变更经常比代码回滚更难，需要 expand-contract 策略。
- 平台标准化不能压死业务灵活性，需要提供默认路径和扩展点。

## 易错

> **易错：** 发布代码后再考虑回滚。
>
> 正确做法：发布方案必须提前定义回滚条件、回滚步骤和数据兼容策略。

> **易错：** liveness 检查依赖下游服务，导致下游异常时本服务被反复重启。
>
> 正确做法：liveness 只判断进程是否存活，readiness 判断是否可以接流量。

> **易错：** 把配置中心当成万能动态开关。
>
> 正确做法：区分启动期配置和运行期配置。线程池大小、连接池、路由、限流等动态变更要经过验证和审计。
