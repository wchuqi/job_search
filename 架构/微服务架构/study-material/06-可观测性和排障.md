# 微服务架构学习资料：可观测性和排障

[返回索引](../微服务架构学习资料.md)

## 学习目标

- 掌握日志、指标和链路追踪的分工。
- 能建立 SLI、SLO 和告警。
- 能从用户问题追踪到具体服务、实例、依赖或发布变更。

## 理论导读

服务数量增加后，单机日志已经不足以解释系统行为。一个用户请求可能经过网关、认证、订单、库存、支付、消息队列和数据库。可观测性就是让系统在运行中留下足够证据，使团队能回答：发生了什么、影响多大、从什么时候开始、在哪个环节、是否和发布或流量有关。

三大基础信号：

- 日志：离散事件和上下文，适合查具体请求。
- 指标：聚合趋势，适合看健康状态和告警。
- 链路追踪：跨服务调用路径，适合定位慢点和断点。

## 核心心智模型

排障像还原一条快递路线。日志是每个站点的扫描记录，指标是每个站点的吞吐和异常率，链路追踪是一张把所有站点串起来的路线图。

## 知识点详解

### 必备上下文

每条请求应尽量带上：

- traceId：跨服务请求链路。
- spanId：链路中的单个步骤。
- userId 或 tenantId：注意脱敏。
- routeId：网关路由。
- serviceId：服务名。
- instanceId：实例。
- version：服务版本。
- requestId 或 idempotencyKey：业务请求标识。

### 深度机制：上下文传播不能只覆盖 HTTP

微服务链路通常跨越 HTTP、RPC、消息队列、异步线程和定时任务。traceId 如果只在 HTTP 请求里传播，一旦进入消息队列或线程池，链路就会断。

传播规则：

- HTTP/RPC：通过标准 trace header 或自定义 header 传播。
- 消息队列：把 traceId、spanId、业务 key 放入 message header。
- 异步线程：提交任务时复制上下文，执行结束后清理，避免上下文串线。
- 定时任务：生成新的 traceId，并记录 jobName、scheduleId、batchId。
- 第三方回调：记录外部 requestId，并和内部 traceId 建立映射。

> **易错：** 使用线程本地变量存 traceId，但在线程池复用时不清理，导致 A 用户日志串到 B 用户链路。

### 核心指标

| 类别 | 指标 | 说明 |
| --- | --- | --- |
| 流量 | QPS、请求数 | 判断负载和突增 |
| 错误 | 4xx、5xx、业务失败率 | 区分用户错误和系统错误 |
| 延迟 | P50、P95、P99 | 长尾延迟比平均值更有价值 |
| 饱和度 | CPU、内存、线程池、连接池 | 判断资源瓶颈 |
| 依赖 | 下游延迟、错误率 | 定位故障传播 |
| 队列 | 堆积、消费延迟、DLQ | 判断异步链路健康 |

### SLI、SLO 和告警

- SLI：可量化的服务水平指标，例如可用性、P99 延迟、成功率。
- SLO：团队承诺的目标，例如“投递创建接口 99.9% 请求在 800ms 内成功”。
- SLA：对外承诺，通常带有合同或赔偿约束。

告警设计应优先从用户影响出发：

| 告警类型 | 示例 | 处理优先级 |
| --- | --- | --- |
| 症状告警 | 下单成功率低于 99% | 高 |
| 依赖告警 | 支付服务 P99 超过 2s | 中高 |
| 资源告警 | CPU 超过 85% | 中 |
| 信息告警 | 某服务发生滚动发布 | 低 |

> **重点：** 症状告警告诉你用户正在受影响，资源告警只能帮助解释原因。

### 排障流程

1. 明确用户影响：哪些接口、哪些用户、从何时开始。
2. 看入口指标：网关 QPS、错误率、延迟是否异常。
3. 查变更：最近发布、配置、限流规则、依赖变更。
4. 用 traceId 找调用链：定位慢 span 或失败 span。
5. 看服务实例：是否集中在某个版本、实例或可用区。
6. 看依赖：数据库、缓存、消息队列、第三方服务是否异常。
7. 先止血：回滚、降级、扩容、限流、摘除异常实例。
8. 后复盘：补监控、补测试、补演练。

### 深度排障：把“现象”拆成四类

| 现象 | 常见根因 | 优先证据 |
| --- | --- | --- |
| 错误率升高 | 发布缺陷、依赖失败、权限变更、数据异常 | 错误码分布、版本维度、trace |
| 延迟升高 | 下游慢、连接池等待、GC、锁竞争、队列堆积 | P99、span duration、资源饱和度 |
| 吞吐下降 | 限流、实例减少、线程池耗尽、入口异常 | QPS、限流计数、实例状态 |
| 数据不一致 | 消息堆积、重复消费、补偿失败、缓存脏读 | 事件状态、消费位点、对账结果 |

排障时要避免只找一个证据。更可靠的方式是构建证据链：发布时间点、受影响版本、错误码、慢 span、依赖指标和业务结果相互印证。

## 例子

结构化日志示例：

```json
{
  "time": "2026-06-06T14:00:00Z",
  "traceId": "4f9c1a",
  "service": "order-service",
  "version": "v2.3.1",
  "operation": "createOrder",
  "orderId": "o1001",
  "status": "FAILED",
  "errorCode": "INVENTORY_TIMEOUT",
  "durationMs": 812
}
```

## 练习

模拟“下单接口 P99 从 300ms 升到 3s”：

- 写出你会看的 10 个指标。
- 写出你需要的 5 类日志字段。
- 写出可能的 5 个根因。
- 写出止血动作和长期修复动作。

## 验收

- 能解释日志、指标、链路追踪分别解决什么问题。
- 能为核心接口设计 SLO。
- 能基于 traceId 还原一次跨服务请求。
- 能区分止血动作和根因修复。

## 重点

- 没有可观测性，微服务故障会变成猜谜。
- 指标用于发现异常，追踪用于定位路径，日志用于还原细节。
- 告警应围绕用户影响和 SLO，而不是只围绕机器资源。

## 难点

- 高基数标签会让指标系统成本暴涨，例如把 userId 直接作为指标标签。
- 日志过多会增加成本和噪音，关键是结构化、可关联和可采样。

## 易错

> **易错：** 只在出错服务打印日志，不传递 traceId。
>
> 正确做法：入口生成或接收 traceId，并在所有下游调用、消息和日志中传播。

> **易错：** 告警条件只写 CPU 超过 80%。
>
> 正确做法：优先关注错误率、延迟、成功率和核心业务指标，再结合资源指标判断原因。

> **易错：** dashboard 只按服务看，不按版本、路由、租户或可用区切分。
>
> 正确做法：核心指标至少要能按 service、version、route、region、instance 维度定位，否则灰度问题和局部故障很难发现。
