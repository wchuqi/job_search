# 微服务架构学习资料：一致性和事件驱动深度推演

[返回索引](../微服务架构学习资料.md)

## 学习目标

- 从业务不变量推导一致性模型。
- 掌握事件设计、Outbox、消费幂等、补偿和对账。
- 能解释消息乱序、重复、丢失和延迟的处理方式。

## 一致性推导方法

### 1. 列出业务不变量

以“购买课程”为例：

- 同一支付流水只能开通一次权益。
- 支付成功后最终必须开通课程权益或进入人工处理。
- 已退款订单不能继续访问付费课程。
- 通知可以延迟，但不能影响权益开通。

### 2. 标注一致性等级

| 规则 | 一致性要求 | 方案 |
| --- | --- | --- |
| 支付流水不重复 | 强一致 | payment-service 本地唯一索引 |
| 权益最终开通 | 最终一致 | Saga + Outbox + 重试 |
| 退款后不能访问 | 较强一致 | entitlement-service 状态机 + 缓存短 TTL |
| 通知送达 | 最终一致 | 异步消息 + 重试 + DLQ |

### 3. 设计失败恢复

每个步骤都要回答：

- 失败是否可重试？
- 重试是否幂等？
- 是否需要补偿？
- 用户看到什么状态？
- 多久未完成要告警？
- 如何人工修复？

## 事件设计

### 事件命名

推荐使用业务事实：

- `PaymentSucceeded`
- `EntitlementGranted`
- `CourseAccessRevoked`

避免模糊事件：

- `PaymentUpdated`
- `OrderChanged`
- `DataSynced`

### 事件内容

事件应包含：

- eventId。
- eventType。
- eventVersion。
- occurredAt。
- aggregateId。
- producer。
- traceId。
- 幂等相关业务 key。
- 必要业务快照。

示例：

```json
{
  "eventId": "evt-1001",
  "eventType": "PaymentSucceeded",
  "eventVersion": 1,
  "occurredAt": "2026-06-06T14:00:00Z",
  "aggregateId": "pay-9001",
  "orderId": "order-1001",
  "userId": "user-1001",
  "amount": 19900,
  "currency": "CNY",
  "traceId": "trace-abc"
}
```

## 消息异常处理

### 重复

原因：

- 生产者重发。
- 消费者处理成功但 ack 前宕机。
- Broker 重新投递。

处理：

- 消费者保存 eventId。
- 使用业务唯一键。
- 状态机只允许合法迁移。

### 乱序

原因：

- 多分区并行消费。
- 不同事件发布路径不同。
- 重试导致旧事件迟到。

处理：

- 按 aggregateId 分区。
- 事件带版本号或序列号。
- 消费端检查当前状态，拒绝非法倒退。
- 对乱序事件延迟重试或进入待处理表。

### 丢失

原因：

- 写库成功但发消息失败。
- 发布器宕机。
- 配置错误。

处理：

- Outbox 或 CDC。
- 发布状态表。
- 定期扫描未发布事件。
- 对账任务比对业务表和事件处理结果。

### 堆积

原因：

- 消费者处理变慢。
- 下游依赖异常。
- 消息突增。

处理：

- 扩容消费者。
- 暂停非核心消费。
- 降低单条处理耗时。
- 拆分 topic 或分区。
- 设置 DLQ 和重放策略。

## Saga 推演：购买课程

```text
OrderCreated
  -> PaymentRequested
  -> PaymentSucceeded
  -> EntitlementGrantRequested
  -> EntitlementGranted
  -> NotificationRequested
```

失败路径：

- 支付失败：订单关闭。
- 权益开通失败：重试；超过阈值进入人工处理，订单保持 `PAID_PENDING_ENTITLEMENT`。
- 通知失败：不影响订单和权益，进入通知重试队列。

用户可见状态：

- `待支付`
- `支付处理中`
- `开通中`
- `已开通`
- `处理异常，请联系客服`

> **重点：** 用户可见状态必须能表达中间态。否则最终一致会被误解为系统错误。

## 对账设计

对账任务检查：

- payment-service 中支付成功但 order-service 未标记已支付。
- order-service 已支付但 entitlement-service 未开通权益。
- outbox 中长期未发布事件。
- 消费记录中长期失败事件。

对账处理：

- 自动补发事件。
- 自动调用补偿接口。
- 进入人工处理队列。
- 输出审计记录。

## 练习

设计“退款后撤销课程权益”：

- 业务不变量。
- 事件列表。
- Saga 流程。
- 幂等键。
- 乱序处理。
- 对账逻辑。

## 验收

- 能根据业务不变量选择一致性方案。
- 能设计事件 schema 和版本。
- 能处理重复、乱序、丢失和堆积。
- 能给出用户可见中间状态。
- 能设计对账任务和人工兜底。

## 重点

- 事件是业务事实，不是数据库变更通知。
- 消费者必须按重复、乱序和延迟设计。
- 对账是最终一致的最后防线。

## 难点

- 事件字段太少会导致消费者必须回查，字段太多又会造成兼容压力。
- 编排式和协同式 Saga 的选择取决于流程可见性和耦合度。

## 易错

> **易错：** 消费失败后无限重试，把队列堵死。
>
> 正确做法：区分临时失败和永久失败，设置重试次数、退避、DLQ 和人工处理。
