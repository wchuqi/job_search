# 微服务架构学习资料：数据管理和分布式事务

[返回索引](../微服务架构学习资料.md)

## 学习目标

- 理解服务数据所有权和跨服务一致性。
- 掌握 Saga、Outbox、TCC、事件驱动和 CQRS 的适用场景。
- 能设计幂等、补偿、去重和数据对账方案。

## 理论导读

微服务最大的变化之一是数据不再天然处于同一个本地事务中。每个服务应拥有自己的数据，其他服务通过 API 或事件访问。如果多个服务共享一个数据库并随意读写彼此的表，系统表面上是微服务，实际上仍是强耦合单体。

跨服务一致性通常不追求所有步骤瞬间一致，而是通过业务流程、事件、补偿和对账实现最终一致。关键问题不是“如何让分布式事务像本地事务一样简单”，而是“哪些业务必须强一致，哪些可以接受短暂不一致，以及不一致时如何恢复”。

## 核心心智模型

本地事务像一个柜台一次性完成收款和出票。跨服务事务像多个柜台协作，每个柜台只能保证自己的账本正确，整条流程需要凭证、状态流转、补偿和对账来保证最终正确。

## 知识点详解

### 数据所有权

- 一个服务拥有自己的核心表和业务规则。
- 其他服务不能直接写该服务的数据。
- 跨服务查询可以通过 API、事件同步读模型或搜索索引实现。
- 共享数据库只适合迁移过渡期，并应有明确退出计划。

### 深度原则：先识别业务不变量

一致性设计的第一步不是选择 Saga、TCC 或消息队列，而是识别业务不变量。业务不变量是系统必须始终满足的规则。

例子：

- 一个支付流水不能被重复扣款。
- 同一个职位对同一个候选人不能重复投递。
- 库存不能被卖成负数。
- 已取消订单不能再支付成功。

不同不变量需要不同保护方式：

| 不变量类型 | 推荐保护方式 | 说明 |
| --- | --- | --- |
| 单服务内部唯一性 | 本地事务、唯一索引、状态机 | 必须在数据所有者内部保证 |
| 跨服务流程最终完成 | Saga、Outbox、补偿、对账 | 允许中间状态，但必须可恢复 |
| 外部资金或权益 | 幂等键、业务流水、对账、人工兜底 | 不能只靠内存状态或消息投递 |
| 查询展示一致性 | 读模型、缓存失效、版本号 | 允许短暂延迟，要标明来源和刷新策略 |

> **重点：** 微服务一致性不是“牺牲正确性”，而是把正确性拆成局部强一致、流程最终一致和异常可恢复。

### 一致性方案对比

| 方案 | 适用场景 | 优点 | 风险 |
| --- | --- | --- | --- |
| 本地事务 | 单服务内部 | 简单可靠 | 不能跨服务解决问题 |
| Saga | 多步骤业务流程 | 适合长事务和补偿 | 补偿逻辑复杂 |
| Outbox | 本地写库同时可靠发事件 | 避免写库成功但消息丢失 | 需要轮询或 CDC |
| TCC | 资源预留、确认、取消 | 一致性强于普通 Saga | 接口设计复杂，资源悬挂风险 |
| CQRS | 读写模型差异大 | 查询灵活，扩展性好 | 数据延迟和模型同步成本 |

### Outbox 机制

Outbox 的核心是把业务数据和待发布事件写入同一个本地事务。之后由后台任务或 CDC 把 outbox 表中的事件投递到消息队列。

```sql
BEGIN;

INSERT INTO orders(id, user_id, status)
VALUES ('o1001', 'u1001', 'CREATED');

INSERT INTO outbox_events(id, aggregate_id, event_type, payload, status)
VALUES ('e1001', 'o1001', 'OrderCreated', '{"orderId":"o1001"}', 'NEW');

COMMIT;
```

这样可以避免“订单创建成功，但事件没有发出去”的不一致。

### Saga 状态机

订单 Saga 示例：

1. 创建订单，本地状态为 `PENDING_PAYMENT`。
2. 请求库存服务预留库存。
3. 请求支付服务扣款。
4. 支付成功后确认订单。
5. 任一步失败，触发补偿，例如释放库存、关闭订单。

> **重点：** Saga 的每一步都要有状态记录、幂等键、重试策略和补偿动作。

### 编排式 Saga 和协同式 Saga

| 类型 | 工作方式 | 优点 | 风险 |
| --- | --- | --- | --- |
| 编排式 Saga | 一个协调器明确调用每一步，并记录流程状态 | 流程清晰、便于排障 | 协调器容易承载过多业务 |
| 协同式 Saga | 服务发布事件，其他服务订阅后推进流程 | 松耦合、扩展方便 | 流程分散，排障和全局状态更难 |

招聘投递场景中，“投递成功后发通知”适合协同式，因为通知失败不应影响投递主流程。“购买课程后开通权益”更适合编排式或至少有明确流程状态，因为支付、权益和订单状态需要更强的业务可追踪性。

### TCC 的悬挂、空回滚和幂等问题

TCC 包含 Try、Confirm、Cancel 三个阶段：

- Try：预留资源。
- Confirm：确认提交。
- Cancel：取消预留。

高风险边界：

- 空回滚：Try 还没执行成功，Cancel 先到达。Cancel 必须能识别并安全返回。
- 悬挂：Cancel 已执行后，迟到的 Try 又成功预留资源。Try 必须检查事务状态，不能在已取消后继续成功。
- 幂等：Confirm 或 Cancel 可能重复到达，必须重复执行安全。

> **难点：** TCC 不是简单加三个接口，它要求资源模型支持预留、确认和取消。很多业务动作没有天然 Try 阶段，强行 TCC 会变复杂。

## 例子

消息消费幂等伪代码：

```java
void handle(OrderPaidEvent event) {
    if (processedMessageRepo.exists(event.getEventId())) {
        return;
    }

    transactionTemplate.execute(status -> {
        orderRepo.markPaid(event.getOrderId());
        processedMessageRepo.save(event.getEventId());
        return null;
    });
}
```

### 深度例子：Outbox 投递失败如何恢复

Outbox 表建议至少包含：

```sql
CREATE TABLE outbox_events (
  id              VARCHAR(64) PRIMARY KEY,
  aggregate_type  VARCHAR(64) NOT NULL,
  aggregate_id    VARCHAR(64) NOT NULL,
  event_type      VARCHAR(128) NOT NULL,
  event_version   INT NOT NULL,
  payload         TEXT NOT NULL,
  status          VARCHAR(32) NOT NULL,
  retry_count     INT NOT NULL DEFAULT 0,
  next_retry_at   TIMESTAMP NULL,
  created_at      TIMESTAMP NOT NULL,
  published_at    TIMESTAMP NULL
);
```

后台发布器的关键规则：

1. 只扫描 `status = NEW` 或到达 `next_retry_at` 的失败事件。
2. 发送消息成功后标记 `PUBLISHED`。
3. 发送失败增加 `retry_count`，按退避时间设置下一次投递。
4. 超过最大次数进入 `FAILED`，触发告警和人工处理。
5. 消费者仍必须幂等，因为发布器可能在“消息已发出但标记 PUBLISHED 前”宕机。

这说明 Outbox 解决的是“业务数据和事件记录原子提交”，不保证消费者只收到一次。

## 练习

设计“用户购买课程”的一致性方案：

- 订单创建。
- 支付扣款。
- 开通课程权益。
- 发送通知。
- 支付失败和权益开通失败的补偿。
- 对账任务如何发现漏处理。

## 验收

- 能说明为什么跨服务不能依赖单库事务。
- 能为关键事件设计 Outbox。
- 能为消息消费者设计去重。
- 能画出 Saga 状态机和补偿路径。
- 能区分强一致、最终一致和读模型延迟。

## 重点

- 数据所有权是服务自治的基础。
- 最终一致不是不一致，而是有恢复机制的一致性。
- 幂等、补偿、对账是分布式业务流程的三件套。

## 难点

- 补偿不是简单回滚。真实业务中扣款、发货、发票等动作可能无法完全撤销，只能做反向业务动作。
- 查询需求经常跨服务，需要在实时 API、读模型、搜索索引和报表系统之间取舍。

## 易错

> **易错：** 服务拆分后继续共用数据库表。
>
> 正确做法：明确数据所有者，通过 API、事件或读模型共享数据。

> **易错：** 以为消息队列保证 exactly once 后就不需要幂等。
>
> 正确做法：业务侧必须按“可能重复”设计，使用事件 ID、业务唯一键和状态机兜底。

> **易错：** 把最终一致理解为“不用管什么时候一致”。
>
> 正确做法：必须定义最大可接受延迟、失败重试策略、对账频率、人工处理入口和用户可见状态。
