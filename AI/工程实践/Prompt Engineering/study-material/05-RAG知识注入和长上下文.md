# Prompt Engineering 学习资料：RAG、知识注入和长上下文

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 理解何时需要 RAG，而不是只改 prompt。
- 能设计带引用、可拒答、可追溯的知识问答 prompt。
- 掌握长上下文材料的组织方式。

## 理论导读

RAG 是 Retrieval-Augmented Generation，即先检索相关资料，再让模型基于资料生成答案。它适合私有知识、时效信息、长文档问答和需要引用依据的任务。

Prompt 在 RAG 中的作用是规定资料优先级、引用规则、拒答条件、冲突处理和输出结构。

## RAG Prompt 关键规则

```text
请只依据 <context> 中的资料回答。
如果资料不足，输出“资料中没有足够信息”。
每个关键结论必须引用资料编号。
如果资料互相冲突，请指出冲突，不要自行调和。
```

## 长上下文组织

| 问题 | 解决方式 |
| --- | --- |
| 材料太多 | 先检索，再摘要，再生成 |
| 来源混乱 | 每段资料加 id、source、date |
| 资料冲突 | 要求列出冲突来源 |
| 模型忽略中间内容 | 分块编号，先抽取相关片段 |
| 用户问题模糊 | 先澄清或给出可回答范围 |

## 例子

```text
你是公司制度问答助手。

规则：
- 只依据 <docs> 回答。
- 必须引用 doc_id。
- 不允许使用常识补充公司政策。
- 资料不足时输出固定句：“资料中没有足够信息。”

<docs>
<doc id="hr-001" source="员工手册" date="2026-01-01">
...
</doc>
</docs>

用户问题：
试用期员工可以休年假吗？
```

## RAG 与微调的取舍

| 需求 | 更适合 |
| --- | --- |
| 需要最新知识 | RAG |
| 需要引用来源 | RAG |
| 固定输出风格 | Prompt 或微调 |
| 大量稳定标签任务 | 微调 |
| 知识经常变化 | RAG |

## 练习

准备 5 段短文档，设计一个 RAG prompt，要求：

- 每条回答带引用。
- 资料不足时拒答。
- 资料冲突时列出冲突。
- 输出包含 `answer`、`citations`、`confidence`。

## 验收

- 能说明 RAG 解决的核心问题。
- 能写出引用约束和拒答规则。
- 能解释长上下文不等于可靠引用。

## 易错

> **易错：** 检索到了资料就认为答案一定正确。
>
> 正确做法：同时评估检索质量、上下文组织、生成答案和引用准确性。
