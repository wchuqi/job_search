# Prompt Engineering 学习资料：解码、采样、约束生成和结构化输出机制深度解析

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 理解 temperature、top-p、max tokens、stop sequence 对输出的影响。
- 掌握约束生成和结构化输出的工程机制。
- 能解释为什么 prompt 约束、API schema 和程序校验要同时存在。

## 从 logits 到输出

模型每一步会给词表中每个 token 一个分数，通常称为 logits。解码过程把这些分数变成下一个 token。

简化流程：

```text
上下文 -> 模型前向计算 -> token logits -> 采样/选择 -> 追加 token -> 重复
```

Prompt 影响的是 logits 分布，解码参数影响的是从分布中如何选择 token。

## 常见解码参数

### Temperature

temperature 控制分布平滑程度。

- 低温：更确定、更稳定，适合分类、抽取、JSON。
- 高温：更发散、更有创意，适合头脑风暴、文案。

### Top-p

top-p 只从累计概率达到 p 的候选 token 集合中采样。它控制候选范围。

### Max tokens

限制输出长度。太短会截断 JSON、代码或推理结果；太长会增加成本并可能输出多余内容。

### Stop sequence

遇到特定序列停止。适合对话分隔、批量生成和固定格式，但误设会截断合法输出。

## 结构化输出的三层约束

### 1. Prompt 语义约束

```text
只输出 JSON，不要解释。
```

作用：告诉模型任务目标。

风险：模型可能仍然漂移。

### 2. API/schema 约束

如果平台支持结构化输出或函数调用，可以用 schema 限制字段和类型。

作用：减少非法格式。

风险：schema 只能约束形状，不保证事实正确。

### 3. 程序校验

```text
JSON parse -> schema validate -> business validate
```

作用：最终兜底。

风险：需要设计重试和降级。

## 约束生成和自由生成的差异

| 方式 | 优点 | 缺点 |
| --- | --- | --- |
| 自由文本 | 表达丰富 | 难解析 |
| prompt 要求 JSON | 简单 | 仍可能漂移 |
| schema 约束 | 格式稳定 | 灵活性下降 |
| 函数调用 | 适合工具参数 | 需要后端执行 |
| 程序后处理 | 可控 | 可能丢语义 |

## 解析失败根因

| 根因 | 表现 | 处理 |
| --- | --- | --- |
| max tokens 太小 | JSON 截断 | 增大上限或缩短字段 |
| prompt 允许解释 | JSON 外有文本 | 禁止额外文本 |
| schema 太复杂 | 字段缺失 | 拆任务或简化 schema |
| 输入资料不足 | `null` 未处理 | 规定缺失值 |
| 任务冲突 | 字段含义混乱 | 拆分 prompt |

## 重试策略

推荐不是简单无限重试，而是分级：

1. 第一次正常生成。
2. 格式失败：只修复格式。
3. 业务校验失败：带错误原因重答。
4. 多次失败：降级到人工或宽松格式。

```text
你上一次输出未通过校验：
- `score` 必须是 0-100 整数。
- `decision` 必须是 recommend/hold/reject。

请只重新输出合法 JSON。
```

## 练习

设计一个“候选人风险评估 JSON 输出”的 schema，并写出：

- 解码参数建议。
- max tokens 风险。
- 解析失败重试 prompt。
- 业务校验规则。

## 验收

- 能解释 temperature 和 top-p 的作用。
- 能区分 prompt 约束、schema 约束和程序校验。
- 能设计解析失败恢复链路。

## 难点

结构化输出的正确性分两层：格式正确和语义正确。schema 主要解决格式，语义仍要靠资料约束、引用、业务规则和评估。
