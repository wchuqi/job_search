# Prompt Engineering 学习资料：Prompt 评估、实验设计、统计和线上监控深度解析

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 掌握 prompt 评估的实验设计。
- 能建立离线评测、人工评分、LLM-as-judge 和线上监控。
- 能避免“看起来变好”的评估陷阱。

## 理论导读

Prompt 的每次修改都可能同时改善部分样例、破坏另一部分样例。评估要回答的问题不是“这次输出好不好”，而是“在目标输入分布上，整体指标是否显著改善，且没有引入不可接受风险”。

## 评估集分层

| 层 | 内容 | 用途 |
| --- | --- | --- |
| smoke set | 5-10 条核心样例 | 快速检查格式和基本能力 |
| regression set | 历史失败案例 | 防止旧 bug 回来 |
| benchmark set | 代表性真实分布 | 比较版本 |
| adversarial set | 恶意和边界输入 | 安全测试 |
| golden set | 人工标注高质量答案 | 精准评估 |

## 指标体系

不能只看一个总分。建议同时看：

- 任务正确率。
- 格式合规率。
- 证据引用准确率。
- 拒答正确率。
- 幻觉率。
- 安全违规率。
- 平均成本。
- P95 延迟。
- 人工复核率。

## LLM-as-judge 使用边界

适合：

- 主观质量初筛。
- 大规模样例排序。
- 摘要可读性评分。
- 方案完整性评分。

需要谨慎：

- 法律、医疗、金融等高风险结论。
- 需要精确事实判断。
- 评分规则模糊。
- judge 模型与被评模型同源导致偏差。

## 实验设计

### 单变量原则

一次只改一个主要因素：

- prompt 文案。
- 示例。
- 模型版本。
- 温度。
- RAG top-k。
- 工具描述。

否则无法知道效果来自哪里。

### A/B 测试

线上 A/B 要记录：

- 用户输入类型。
- prompt 版本。
- 模型版本。
- 成本和延迟。
- 用户反馈。
- 人工复核结果。

```text
根因假设：
修复方案：
是否加入回归集：
```

## 练习

为“知识库问答 prompt”设计评估方案：

- 50 条 benchmark。
- 20 条 adversarial。
- 10 条历史失败回归。
- 评分表和上线阈值。

## 验收

- 能区分 smoke、regression、benchmark、adversarial。
- 能设计多指标评估。
- 能说明为什么单次示例不能证明 prompt 变好。

## 易错

> **易错：** 只优化平均分，忽略安全违规和格式崩溃。
>
> 正确做法：关键风险指标设置硬门槛，例如安全违规率必须为 0。
