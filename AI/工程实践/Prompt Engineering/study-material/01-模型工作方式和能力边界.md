# Prompt Engineering 学习资料：模型工作方式和能力边界

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 理解 token、上下文窗口、非确定性和幻觉。
- 能解释模型为什么会“看起来懂了但答错”。
- 能根据模型边界设计提示词和评估方法。

## 理论导读

大语言模型根据输入上下文生成下一个 token 的概率分布。它不是数据库，也不是规则引擎。它擅长语言理解、模式归纳、摘要、分类、改写、推理辅助和代码生成，但对事实时效性、隐藏业务规则、精确计算、权限判断和未提供资料存在天然限制。

## 核心概念

### token 和上下文窗口

token 是模型处理文本的基本单位。上下文窗口是模型一次请求能看到的最大信息量。窗口越大，不代表所有信息都会被同等重视；长上下文里仍需要清晰结构、引用编号和任务聚焦。

> **易错：** 把大量资料无结构地塞给模型。正确做法：分块、编号、标注来源，告诉模型如何使用资料。

### 非确定性

同一个 prompt 可能生成不同答案。温度、采样策略、模型版本、上下文顺序和系统指令都会影响输出。

> **重点：** 生产环境要降低随机性、固定模型版本、建立评估集，而不是依赖单次结果。

### 幻觉

幻觉是模型生成了看似合理但缺乏依据的内容。幻觉常见于缺少上下文、要求模型回答超出资料的问题、提示词奖励“完整答案”而不允许“不知道”的场景。

## 能力边界

| 场景 | 模型适合吗 | 处理方式 |
| --- | --- | --- |
| 总结一段会议记录 | 适合 | 给出摘要维度和输出格式 |
| 查询公司最新制度 | 只靠模型不适合 | 使用 RAG 或工具查询 |
| 精确财务计算 | 只靠模型风险高 | 调用计算工具或程序 |
| 法律/医疗结论 | 高风险 | 限定资料、引用来源、人工审核 |
| 多步骤业务操作 | 可以辅助 | 使用工具调用、权限和确认机制 |

## 例子

风险 prompt：

```text
我们公司的年假政策是什么？
```

更稳妥：

```text
请只依据 <policy_docs> 中的制度片段回答年假政策。
如果材料没有提到，请输出：“资料中没有足够信息”。
回答必须包含引用片段编号。
```

## 练习

列出 5 个你希望 AI 完成的任务，并判断每个任务需要：

- 只用 prompt。
- prompt + 外部资料。
- prompt + 工具调用。
- prompt + 人工审核。

## 验收

- 能解释上下文窗口和 token 的影响。
- 能识别模型可能幻觉的场景。
- 能设计“不知道时如何回答”的规则。

## 难点

模型输出流畅会掩盖不确定性。工程上要把“看起来合理”转成“有依据、可检查、可回归”。
