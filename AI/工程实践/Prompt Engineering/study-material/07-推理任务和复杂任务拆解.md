# Prompt Engineering 学习资料：推理任务和复杂任务拆解

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 掌握复杂推理任务的提示策略。
- 能用计划、证据、检查和修正降低错误。
- 理解“要求推理”与“暴露完整思维过程”的边界。

## 理论导读

复杂任务包括多约束规划、代码调试、数学分析、架构评审、合同条款比对等。它们容易失败，因为模型需要同时记住约束、选择步骤、执行推理并生成答案。

更稳妥的方式是让模型输出可检查的中间产物，例如计划、依据、候选方案、校验结果，而不是要求它展示完整隐藏推理。

## 常用策略

| 策略 | 作用 |
| --- | --- |
| 先计划后执行 | 降低遗漏步骤 |
| 先提取事实再判断 | 防止直接下结论 |
| 列约束清单 | 防止忽略限制 |
| 多方案比较 | 防止单一路径错误 |
| 结果自检 | 检查格式、约束、证据 |
| 分步调用 | 每一步单独评估 |

## 示例

```text
请完成以下任务，但不要输出冗长推理过程。

输出结构：
1. 关键信息提取
2. 约束清单
3. 推荐方案
4. 风险和不确定项
5. 自检结果
```

## 代码任务示例

```text
你是代码审查助手。

流程：
1. 先列出代码意图。
2. 再找可能的 bug、边界条件和测试缺口。
3. 最后给出最小修改建议。

要求：
- 只引用给定代码。
- 每个问题要说明触发条件。
- 没有问题时明确说“未发现明确问题”。
```

## 练习

设计一个“系统设计方案评审”prompt，要求输出：

- 需求理解。
- 关键约束。
- 架构方案。
- 风险。
- 替代方案。
- 自检。

## 验收

- 能把复杂任务拆成可检查步骤。
- 能避免让模型只给最终结论。
- 能让模型输出证据和不确定项。

## 易错

> **易错：** 以为“请一步步思考”能自动解决复杂任务。
>
> 正确做法：规定可观察的中间产物和最终验收标准。
