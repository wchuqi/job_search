# Prompt Engineering 学习资料：工具调用决策、路由、参数校验和权限深度解析

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 掌握工具调用中的决策规则和权限边界。
- 能设计工具路由、参数校验、重试和审计。
- 能识别工具调用中的高风险失败模式。

## 理论导读

工具调用把模型从“生成答案”扩展到“选择动作”。风险也随之扩大：模型可能选错工具、填错参数、重复调用、越权调用或把工具结果理解错。

## 工具调用决策树

```text
是否需要外部事实/计算/操作？
  否 -> 直接回答
  是 -> 信息是否足够？
    否 -> 追问
    是 -> 是否有权限？
      否 -> 拒绝或请求授权
      是 -> 是否为写操作/高风险？
        是 -> 先生成计划并请求确认
        否 -> 调用工具
```

## 工具描述要可路由

坏描述：

```text
search: 搜索东西。
```

好描述：

```text
search_jobs: 根据岗位、城市、年限查询公开职位。不能查询候选人隐私信息。
```

工具描述要包含：

- 能做什么。
- 不能做什么。
- 必填参数。
- 返回结果含义。
- 失败状态。

## 参数校验

模型填参前：

- 缺必填字段就追问。
- 枚举值必须映射到合法值。
- 日期、金额、城市等实体要规范化。
- 用户自然语言不要直接拼进 SQL 或 shell。

工具执行前：

- 后端重新校验权限。
- 后端验证参数类型。
- 高风险操作二次确认。
- 记录审计日志。

## 工具结果解释

工具返回应结构化：

```json
{
  "status": "ok",
  "items": [],
  "source": "jobs_db",
  "retrieved_at": "2026-06-07T10:00:00Z"
}
```

Prompt 要规定：

```text
只根据工具返回的 items 输出职位。
如果 items 为空，不要编造职位。
```

## 高频失败模式

| 失败 | 例子 | 防护 |
| --- | --- | --- |
| 错选工具 | 用邮件工具查数据库 | 工具描述更清晰，路由评估 |
| 参数幻觉 | 自造用户没提供的城市 | 缺参追问 |
| 重复调用 | 循环搜索 | 调用次数限制 |
| 越权调用 | 删除他人数据 | 后端权限校验 |
| 结果误读 | 把空结果当成功 | 明确 status 和空结果语义 |

## 练习

设计一个“投递管理 Agent”的工具层：

- `search_jobs`
- `match_resume`
- `draft_email`
- `submit_application`

要求写出每个工具的权限、参数、调用条件、失败处理和审计字段。

## 验收

- 能写出工具调用决策树。
- 能区分 prompt 约束和后端权限。
- 能说明参数校验和审计为什么不能省。

## 易错

> **易错：** 认为模型决定调用工具后就可以直接执行。
>
> 正确做法：模型只提出动作，系统负责授权、校验、执行和记录。
