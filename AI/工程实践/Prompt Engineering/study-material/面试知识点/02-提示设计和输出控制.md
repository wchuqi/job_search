## 一、提示设计

### 1. 如何设计一个稳定的 prompt？

**参考答案：**

先定义成功标准，再组织 prompt。一个稳定 prompt 通常包含身份、任务、上下文、规则、输出格式、示例、反例和失败处理。对于生产任务，还要配套评估集、版本管理和安全规则。

可以按以下顺序设计：

1. 明确任务目标和用户场景。
2. 确定输入资料和边界。
3. 写清楚必须做和不能做的规则。
4. 规定输出 schema。
5. 添加代表性示例和反例。
6. 用评估集测试并迭代。

### 2. 为什么要使用分隔符或 XML 标签？

**参考答案：**

分隔符可以帮助模型区分指令、上下文、示例和用户输入，减少把资料内容误当成指令的概率。XML 标签还可以携带 id、source、date 等元数据，方便引用和冲突处理。

### 3. 如何控制模型只输出 JSON？

**参考答案：**

Prompt 中要明确“只输出合法 JSON，不要 Markdown 和解释”，并定义字段、类型、枚举和缺失值规则。工程上还要用 JSON parser 校验，解析失败时重试或进入修复流程。

> **重点：** 只靠一句“输出 JSON”不够，程序校验是必要的。

### 4. 示例应该怎么选？

**参考答案：**

示例要覆盖典型输入、边界输入和容易错的输入。示例必须和规则一致，输出格式要完全符合要求。数量不一定多，但要有代表性。

### 5. prompt 越长越好吗？

**参考答案：**

不是。prompt 越长，成本越高，模型也可能忽略关键约束。好的 prompt 应该信息充分但结构清晰。长资料要分块、摘要、检索或编号，而不是无差别堆叠。
