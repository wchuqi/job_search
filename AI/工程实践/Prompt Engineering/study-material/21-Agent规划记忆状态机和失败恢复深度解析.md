# Prompt Engineering 学习资料：Agent 规划、记忆、状态机和失败恢复深度解析

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 深入理解 Agent 的工程结构。
- 掌握计划、状态、记忆、工具、检查点和失败恢复。
- 能设计不会无限循环、不会越权、可观测的 Agent。

## 理论导读

Agent 的核心不是“让模型自己想办法”，而是把模型放进一个受控工作流中。可靠 Agent 通常像状态机，而不是自由聊天。

## Agent 状态机

```text
INIT -> CLARIFY -> PLAN -> ACT -> OBSERVE -> VERIFY -> DONE
                         \-> RECOVER -> CLARIFY/PLAN/DONE
```

每个状态要定义：

- 输入。
- 允许动作。
- 输出。
- 转移条件。
- 失败处理。

## 计划设计

计划要短、可执行、可检查：

```json
{
  "goal": "筛选 5 个匹配职位",
  "steps": [
    {"id": 1, "action": "extract_preferences", "status": "pending"},
    {"id": 2, "action": "search_jobs", "status": "pending"},
    {"id": 3, "action": "rank_jobs", "status": "pending"}
  ]
}
```

> **重点：** 计划不是越详细越好。过细会增加循环和脆弱性。

## 记忆分类

| 记忆 | 内容 | 风险 |
| --- | --- | --- |
| 会话短期记忆 | 当前任务上下文 | 被历史信息污染 |
| 用户偏好记忆 | 城市、风格、禁忌 | 隐私和过期 |
| 任务状态 | 已完成步骤、待确认项 | 状态不一致 |
| 工具结果 | 查询返回、执行状态 | 过期或误读 |

记忆必须有生命周期和来源。不要把全部对话历史直接当长期记忆。

## 失败恢复

| 失败 | 恢复策略 |
| --- | --- |
| 信息不足 | 追问最少必要问题 |
| 工具失败 | 重试、换工具、降级 |
| 结果冲突 | 列冲突，要求用户确认 |
| 循环调用 | 设置最大步数和停止条件 |
| 低置信度 | 交给人工或输出不确定项 |
| 高风险操作 | 停止并请求确认 |

## 可观测性

Agent 日志要记录：

- 用户目标。
- 状态转移。
- 工具调用和参数。
- 工具返回摘要。
- 决策原因摘要。
- 失败和重试。
- 用户确认。

## 练习

把“求职助手 Agent”设计成状态机，列出每个状态的输入、输出、允许工具和转移条件。

## 验收

- 能用状态机描述 Agent。
- 能定义记忆类型和生命周期。
- 能设计失败恢复和最大步数。

## 难点

Agent 最大风险不是回答错，而是“带着错误状态继续行动”。因此要把关键状态结构化，并在关键步骤做验证和确认。
