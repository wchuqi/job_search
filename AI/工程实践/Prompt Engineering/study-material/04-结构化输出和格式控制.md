# Prompt Engineering 学习资料：结构化输出和格式控制

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 掌握 JSON、Markdown 表格、枚举值和 schema 约束。
- 能减少格式漂移和多余解释。
- 能为下游程序设计可解析输出。

## 理论导读

结构化输出是 Prompt Engineering 进入工程系统的关键。人读的答案可以有弹性，程序读的答案必须稳定。只写“请输出 JSON”通常不够，还要定义字段、类型、枚举、缺失值处理和禁止额外文本。

## 输出格式设计

| 控制项 | 说明 | 示例 |
| --- | --- | --- |
| 字段 | 固定字段名 | `decision`, `reasons`, `next_steps` |
| 类型 | 字符串、数组、数字、布尔 | `score`: 0-100 |
| 枚举 | 限定取值 | `recommend` / `hold` / `reject` |
| 缺失值 | 资料不足时怎么填 | `null` 或 `"unknown"` |
| 长度 | 避免冗长 | 每条建议不超过 40 字 |
| 额外文本 | 禁止解释包裹 | 只输出 JSON |

## JSON 示例

```text
只输出合法 JSON，不要 Markdown，不要解释。

Schema:
{
  "decision": "recommend | hold | reject",
  "score": "integer, 0-100",
  "reasons": ["string"],
  "missing_evidence": ["string"],
  "interview_questions": ["string"]
}
```

## Markdown 表格示例

适合人工阅读，但不适合严肃程序解析：

```text
输出 Markdown 表格：
| 维度 | 结论 | 依据 | 风险 |
| --- | --- | --- | --- |
```

## 格式校验

生产系统中应尽量在模型外部做校验：

- JSON 解析失败时重试或进入修复流程。
- 枚举值不合法时拒收。
- 缺失字段时补充默认值或让模型重答。
- 高风险字段由人工复核。

## 练习

为“客服工单分类”设计一个 JSON 输出格式，包含：

- `category`
- `priority`
- `sentiment`
- `summary`
- `needs_human_review`
- `evidence`

## 验收

- 能写出固定字段、类型和枚举。
- 能说明 JSON 和 Markdown 表格的适用差异。
- 能设计格式校验和失败重试策略。

## 易错

> **易错：** 只靠 prompt 保证格式。
>
> 正确做法：prompt 规定格式，程序负责解析、校验和兜底。
