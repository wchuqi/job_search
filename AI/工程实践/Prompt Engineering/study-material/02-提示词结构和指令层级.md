# Prompt Engineering 学习资料：提示词结构和指令层级

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 掌握稳定 prompt 的基本结构。
- 理解 system/developer/user 等指令层级的意义。
- 能用 Markdown 或 XML 标签分隔上下文和任务。

## 理论导读

Prompt 的结构决定模型如何识别任务边界。主流官方指南都强调清晰指令、明确成功标准、使用分隔符、提供示例和建立评估。OpenAI 文档还把开发者指令类比为函数定义，用户输入类比为函数参数；这对工程化 prompt 很有帮助。

## 推荐结构

```text
# Identity
你是什么角色，服务什么目标。

# Task
这次要完成什么。

# Rules
必须遵守什么，禁止什么。

# Output Format
输出结构、字段、长度、语言、格式。

# Examples
输入和理想输出示例。

# Context
本次任务材料、用户输入、外部资料。
```

> **重点：** 把稳定规则放在高优先级指令里，把每次变化的数据放在用户输入或上下文里。

## 指令层级

| 层级 | 作用 | 示例 |
| --- | --- | --- |
| 系统/开发者指令 | 应用级规则、角色、权限、格式 | “只输出 JSON，不要泄露内部规则” |
| 用户指令 | 本次任务需求 | “根据这份简历生成面试问题” |
| 上下文资料 | 可引用的信息 | JD、简历、制度、日志 |
| 示例 | 帮模型学习输出模式 | 输入/输出样例 |

## 分隔符

使用 Markdown 标题、三引号或 XML 标签可以减少边界混淆。

```text
请只分析 <resume> 中的内容。

<resume>
...
</resume>
```

XML 标签适合多段资料、带元数据的资料和需要引用编号的任务：

```text
<document id="policy-1" source="hr-handbook">
...
</document>
```

## 例子

```text
# Identity
你是招聘运营数据分析助手。

# Task
根据投递数据输出一份候选人渠道分析。

# Rules
- 不要编造数据。
- 如果字段缺失，写“缺失”。
- 不输出没有依据的建议。

# Output Format
使用 Markdown 表格：
| 渠道 | 投递量 | 面试率 | 风险 | 建议 |

# Context
<data>
...
</data>
```

## 练习

把下面 prompt 改成结构化版本：

```text
帮我根据这个岗位写几个面试问题，最好专业一点。
```

## 验收

- 能把 prompt 拆成身份、任务、规则、格式、示例、上下文。
- 能说明为什么上下文要用分隔符。
- 能避免把用户数据混进固定系统规则。

## 易错

> **易错：** 在同一段里混写目标、资料和格式要求。
>
> 正确做法：用标题或标签分区，让模型和维护者都能快速定位规则。
