# Prompt Engineering 学习资料：总览与心智模型

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 理解 Prompt Engineering 的定位：它是人和模型之间的任务规格设计。
- 建立“输入规格 -> 模型生成 -> 校验反馈 -> 迭代”的闭环心智模型。
- 明确 prompt、RAG、工具调用、微调和评估之间的边界。

## 理论导读

Prompt Engineering 是设计输入，使模型更稳定地产生符合目标的输出。它不是让模型突然拥有不存在的知识，也不能替代业务规则、权限控制、数据库查询和测试流程。

一个好 prompt 通常包含：任务目标、受众、上下文、约束、输出格式、示例、失败处理和验收标准。对于生产系统，还要包含评估集、版本记录、安全防护和回滚策略。

## 核心心智模型

把模型想成一个“强大的概率型协作者”。它可以综合上下文、归纳模式、生成文本、调用工具，但它不会天然知道你的业务边界，也不会自动理解哪个目标最重要。

Prompt 的职责不是“求模型聪明”，而是减少歧义：

| 问题 | Prompt 要补上的信息 |
| --- | --- |
| 模型不知道任务目标 | 明确角色、任务和成功标准 |
| 模型不知道该依赖什么资料 | 标注上下文、来源和优先级 |
| 模型输出太散 | 规定结构、字段、长度和语气 |
| 模型容易编造 | 要求引用依据、允许回答不知道 |
| 模型多步任务失败 | 拆成计划、执行、检查、修正 |
| 模型上线后不稳定 | 建立测试集、评分规则和版本管理 |

## 知识点详解

### 1. Prompt 是任务规格

写 prompt 类似写接口契约。你要告诉模型输入是什么、应该输出什么、不能输出什么、遇到不确定时怎么办。

> **重点：** 好 prompt 的核心不是“优美”，而是可执行、可验证、可复用。

### 2. Prompt 优化必须围绕成功标准

没有成功标准时，任何修改都只是感觉。成功标准可以是准确率、召回率、格式合规率、人工评分、引用命中率、误拒率、成本、延迟或用户满意度。

> **易错：** 只看一次聊天效果。正确做法：准备代表性样例，记录每次改动对指标的影响。

### 3. Prompt 和其他技术的边界

| 技术 | 适合解决 | 不适合解决 |
| --- | --- | --- |
| Prompt | 指令清晰度、格式控制、任务拆解 | 缺失知识、权限控制、长期稳定学习 |
| RAG | 依赖外部知识、私有文档、时效信息 | 复杂业务操作、模糊评分标准 |
| 工具调用 | 查数据库、执行计算、调用 API | 无权限边界的开放式行动 |
| 微调 | 固定风格、特定标签、领域模式 | 临时知识更新、一次性任务 |
| 评估 | 判断改动是否有效 | 代替需求澄清 |

## 例子

含糊 prompt：

```text
帮我分析这个候选人怎么样。
```

改写后：

```text
你是技术招聘面试官。请根据下面的 JD 和候选人经历，输出一份面试筛选建议。

目标：
- 判断是否进入下一轮技术面。
- 只依据给定材料，不要补充未提供经历。

输出格式：
1. 结论：推荐 / 待定 / 不推荐
2. 匹配点：最多 3 条
3. 风险点：最多 3 条
4. 下一轮追问：5 个问题

JD：
<job_description>
...
</job_description>

候选人经历：
<resume>
...
</resume>
```

## 验收

- 能说明 Prompt Engineering 和普通聊天的区别。
- 能指出一个 prompt 的目标、上下文、约束和输出格式。
- 能判断某个问题应该靠 prompt、RAG、工具调用、微调还是评估解决。

## 难点

真正难的不是写一句“请一步步思考”，而是把业务目标拆成模型能执行、系统能校验、团队能维护的规格。
