# Prompt Engineering 学习资料：Few-shot 示例选择和 Prompt 模式库深度解析

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 深入理解示例如何影响模型行为。
- 掌握示例选择、排序、覆盖和反例设计。
- 能为分类、抽取、评审、生成任务建立 prompt 模式库。

## 理论导读

few-shot 的作用不是“给模型补知识”，而是给模型当前任务的局部模式：输入长什么样、输出长什么样、边界怎么判、异常怎么处理。示例是隐式规则，因此示例和明文规则冲突时，模型可能优先模仿示例。

## 示例选择原则

| 原则 | 说明 |
| --- | --- |
| 代表性 | 覆盖真实输入分布 |
| 边界性 | 包含最容易误判的样例 |
| 一致性 | 示例必须严格遵守 schema |
| 精简性 | 每个示例承担一个教学目标 |
| 可维护 | 示例变更要和评估集同步 |

## 示例排序

常见策略：

- 从简单到复杂：适合教学型任务。
- 从最相似到最一般：适合动态检索示例。
- 正例后接反例：适合防止固定错误。
- 最关键格式示例靠近输出要求：适合 JSON 输出。

## 反例设计

反例要说明“错误在哪里”和“正确处理是什么”：

```text
反例：资料没有项目规模时，不要推测。
错误输出：候选人有大型项目经验。
正确输出：资料未提供项目规模，加入 missing_evidence。
```

## Prompt 模式库

### 1. Extract-Then-Reason

先抽取事实，再做判断。适合招聘、审计、合规、法律条款比对。

```text
第一步：只提取事实，不评价。
第二步：只基于事实给结论。
```

### 2. Critique-Then-Revise

先评审，再修改。适合文案、代码、方案优化。

```text
先列出问题，再输出修改版。修改版必须解决已列问题。
```

### 3. Rubric Scoring

按评分标准打分。适合面试、作文、客服质量评估。

```text
按 rubrics 中每一项独立评分，不能因为总体印象替代单项评分。
```

### 4. Plan-Execute-Verify

计划、执行、校验。适合复杂任务、Agent 和代码生成。

```text
先给计划，再执行，最后检查是否满足约束。
```

### 5. Grounded Answer

基于资料回答并引用。适合 RAG。

```text
每个关键结论必须引用 doc_id。
资料不足时拒答。
```

## 动态示例检索

当任务种类多时，可以从示例库检索最相似样例放入 prompt。注意：

- 示例不能泄露隐私。
- 示例必须和当前 schema 一致。
- 动态示例也要参与评估。
- 不要放入会诱导偏见的样例。

## 练习

为“客服投诉分类”建立 8 条 few-shot 示例：

- 4 条常规。
- 2 条边界。
- 1 条资料不足。
- 1 条恶意输入。

## 验收

- 能说明示例为什么有效。
- 能设计正例、边界例和反例。
- 能识别示例和规则冲突。

## 易错

> **易错：** 只放完美样例，不放边界样例。
>
> 正确做法：示例要覆盖模型最容易错的输入。
