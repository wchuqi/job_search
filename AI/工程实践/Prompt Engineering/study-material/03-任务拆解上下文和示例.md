# Prompt Engineering 学习资料：任务拆解、上下文和示例

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 能把复杂任务拆成模型更容易执行的小任务。
- 掌握 zero-shot、few-shot、反例和上下文排序。
- 能选择何时给示例、给多少示例、示例怎么写。

## 理论导读

复杂任务失败时，原因经常不是模型“不会”，而是任务中同时包含理解、分类、推理、生成、格式化和校验。把任务拆开后，每一步都更容易评估。

## 任务拆解

常见拆解方式：

| 拆解方式 | 适用场景 | 示例 |
| --- | --- | --- |
| 按阶段拆 | 多步骤流程 | 先提取事实，再判断风险，再生成建议 |
| 按字段拆 | 结构化输出 | 每个字段都有定义和取值范围 |
| 按证据拆 | 高准确要求 | 先列证据，再给结论 |
| 按角色拆 | 多视角评审 | 产品、技术、安全分别审查 |
| 按失败模式拆 | 排障优化 | 找格式错误、事实错误、遗漏项 |

## 示例类型

### zero-shot

不给示例，只给规则。适合简单、通用、格式要求不严格的任务。

### few-shot

给少量输入输出示例。适合分类、风格迁移、复杂格式、行业术语和容易歧义的任务。

### 反例

告诉模型什么是不合格输出。适合减少常见错误。

```text
反例：
输入没有薪资信息时，不要猜测薪资范围。
错误输出：“预计薪资 20k-30k。”
正确输出：“资料中未提供薪资信息。”
```

## 上下文排序

推荐顺序：稳定规则在前，任务输入在后；长资料要编号；输出要求要靠近任务说明；关键约束可以在格式前再强调一次。

> **难点：** 长上下文任务中，模型可能忽略中间材料。需要分块摘要、编号引用和阶段性处理。

## 例子

```text
任务分两步：
1. 从候选人经历中提取事实，不做评价。
2. 根据事实和 JD 输出匹配度评分。

输出：
{
  "facts": [],
  "score": 0-100,
  "evidence": [],
  "uncertainties": []
}
```

## 练习

设计一个“简历与岗位匹配评分”prompt，要求：

- 至少 3 个 few-shot 示例。
- 一个反例。
- 输出包含评分、证据、不确定项和下一轮追问。

## 验收

- 能判断任务是否需要拆解。
- 能写出能影响模型行为的示例，而不是随便举例。
- 能用反例约束高频错误。

## 易错

> **易错：** 示例和规则互相冲突。
>
> 正确做法：示例必须遵守规则；如果规则变了，示例也要同步更新。
