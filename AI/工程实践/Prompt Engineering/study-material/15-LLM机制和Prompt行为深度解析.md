# Prompt Engineering 学习资料：LLM 机制和 Prompt 行为深度解析

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 理解 prompt 为什么会影响模型输出，而不是只会背模板。
- 掌握 token、注意力、位置、采样、指令微调和上下文竞争对 prompt 的影响。
- 能从机制角度解释“同一个 prompt 为什么换模型、换顺序、换参数就变了”。

## 理论导读

大语言模型不是按规则执行 prompt，而是在当前上下文条件下生成下一个 token。Prompt Engineering 的本质，是通过上下文设计影响模型对任务、约束、格式、证据和语气的概率分布。

工程上最重要的结论是：prompt 不是命令行参数，模型也不是解释器。prompt 更像“把任务、资料、约束和示例写进模型当前工作记忆”，让模型在生成时更可能沿着正确轨道继续。

## 核心机制

### 1. token 化会影响边界识别

模型看到的是 token 序列，不是你脑中的“词”。同一个中文词、代码标识符、JSON 字段、特殊符号，可能被切成不同 token。长字段名、罕见符号和混合语言会增加不稳定性。

> **重点：** 结构化输出字段名要稳定、短、无歧义。不要在同一 prompt 中混用 `candidateScore`、`score`、`评分` 三种字段。

### 2. 注意力不是无限均匀阅读

上下文窗口允许模型“看到”很多内容，但不同位置、不同格式、不同重复程度的信息权重不同。长上下文里，中间信息、重复模板、低信号文本和格式混乱内容更容易被忽略。

实践规则：

- 核心任务和输出格式靠前且清晰。
- 长资料分块编号。
- 关键限制在任务前后都可轻量重复。
- 对长资料先做检索、抽取或摘要。

### 3. 指令微调影响“听话方式”

现代对话模型经过指令微调和偏好训练，会倾向于遵循高层指令、保持有帮助、避免危险内容。但不同模型对“角色扮演、格式约束、拒答、安全边界、工具调用”的偏好不同。

> **易错：** 以为一个 prompt 在 A 模型好用，换 B 模型也必然好用。正确做法：把模型版本纳入评估维度。

### 4. 采样参数影响稳定性

温度越高，输出越发散；温度越低，越稳定但可能更保守。对于分类、抽取、JSON 输出、合规审查，通常要低温；对于创意写作、头脑风暴，可适当升高。

| 任务 | 推荐倾向 |
| --- | --- |
| JSON 抽取 | 低温，强 schema |
| 分类打标 | 低温，枚举约束 |
| 摘要 | 低到中 |
| 方案头脑风暴 | 中到高 |
| 文案创意 | 中到高 |

### 5. 上下文竞争导致指令冲突

模型同时看到系统规则、开发者规则、用户输入、外部文档、示例和历史对话。若这些内容互相冲突，模型会尝试在上下文中寻找最合理延续，但结果可能不符合你的系统期望。

常见冲突：

- 示例输出和 schema 不一致。
- 用户要求和安全规则冲突。
- 外部文档包含“忽略之前指令”。
- 历史对话的旧要求和当前任务冲突。

## 机制到工程策略

| 机制问题 | 工程策略 |
| --- | --- |
| token 和字段不稳定 | 固定 schema、短字段、parser 校验 |
| 长上下文稀释 | 检索、编号、摘要、任务聚焦 |
| 模型版本差异 | 固定版本、回归测试 |
| 采样随机 | 低温、评估多次运行 |
| 指令冲突 | 分层指令、上下文隔离、安全规则 |

## 例子：为什么“资料不足就说不知道”要写清楚

如果只写：

```text
回答用户问题。
```

模型会倾向于生成一个完整答案，因为很多训练样本奖励“有帮助的完整回答”。如果写成：

```text
只依据资料回答。资料中没有答案时，必须输出：“资料中没有足够信息”。
```

你改变了任务目标：从“尽量回答”变成“基于证据回答”。这会显著降低无依据补全。

## 练习

选择一个 prompt，做 4 组实验：

- 改变上下文顺序。
- 去掉输出格式。
- 增加反例。
- 调低温度。

记录每组输出的格式合规率、事实错误和稳定性。

## 验收

- 能从机制解释 prompt 为什么有效。
- 能解释长上下文失败的原因。
- 能说明模型版本、温度和上下文顺序为什么要纳入评估。

## 难点

Prompt 行为不是单因果。一个输出变差，可能同时由模型版本、输入分布、上下文位置、示例冲突、温度和下游 parser 共同造成。排障时要做单变量实验。
