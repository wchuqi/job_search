# Prompt Engineering 学习资料：行业场景案例库和 Prompt 评审清单

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 把 Prompt Engineering 应用到真实场景。
- 掌握不同场景的重点、风险和验收。
- 建立 prompt review 清单。

## 场景 1：招聘简历筛选

重点：

- 只依据 JD 和简历。
- 不推测年龄、性别、婚育等敏感信息。
- 输出证据和不确定项。
- 低置信度转人工。

风险：

- 编造候选人经历。
- 放大偏见。
- 过度依赖关键词。

验收：

- 明显匹配和明显不匹配准确。
- 每个结论有证据。
- 敏感属性不参与判断。

## 场景 2：客服工单分类

重点：

- 固定分类枚举。
- 低置信度输出 `needs_human_review=true`。
- 保留用户原始诉求。

风险：

- 类别漂移。
- 情绪误判。
- 高优先级投诉漏判。

验收：

- 格式合规率 100%。
- P0/P1 工单召回优先于精确率。

## 场景 3：企业知识库问答

重点：

- RAG。
- 引用资料。
- 资料不足拒答。
- 新旧制度冲突说明。

风险：

- 引用不支持结论。
- 旧制度覆盖新制度。
- 外部文档提示注入。

验收：

- 引用准确。
- 拒答正确。
- 冲突可见。

## 场景 4：代码审查助手

重点：

- 指出触发条件。
- 区分 bug、风险、风格建议。
- 不要求无关重构。

风险：

- 误报太多。
- 忽略边界条件。
- 给出无法编译的修改。

验收：

- 高严重问题优先。
- 每个问题有代码依据。
- 建议可执行。

## 场景 5：合规/法律辅助

重点：

- 明确不是法律结论。
- 引用条款。
- 高风险转人工。
- 保留不确定性。

风险：

- 幻觉法条。
- 给出确定性建议。
- 忽略司法辖区。

验收：

- 不编造依据。
- 高风险问题必须人工复核。

## Prompt Review 清单

### 任务和范围

- 目标是否明确？
- 输入和输出是否定义？
- 不适用场景是否说明？
- 资料不足时是否拒答或追问？

### 上下文和示例

- 资料是否有边界和来源？
- 示例是否与规则一致？
- 是否包含边界样例和反例？
- 是否存在指令冲突？

### 输出和校验

- 是否有固定 schema？
- 枚举和缺失值是否明确？
- 是否能被程序解析？
- 是否有失败重试和降级？

### 安全和合规

- 是否隔离外部资料中的指令？
- 是否避免敏感信息进入上下文？
- 工具权限是否最小化？
- 是否有红队样例？

### 评估和上线

- 是否有评估集？
- 是否记录模型版本和参数？
- 是否有灰度和回滚？
- 是否监控成本、延迟、安全和质量？

## 练习

选择一个业务场景，按本清单做 prompt review，输出：

- 发现的问题。
- 风险等级。
- 修改建议。
- 需要补充的评估样例。

## 验收

- 能把通用 prompt 方法落到具体行业场景。
- 能用 review 清单发现风险。
- 能给出可执行修复方案。
