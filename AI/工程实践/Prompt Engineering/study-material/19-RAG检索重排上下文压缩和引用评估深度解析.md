# Prompt Engineering 学习资料：RAG 检索、重排、上下文压缩和引用评估深度解析

[返回索引](../Prompt Engineering学习资料.md)

## 学习目标

- 深入理解 RAG 不只是“把文档塞给模型”。
- 掌握检索、重排、上下文拼装、引用和评估的关键环节。
- 能定位 RAG 失败是检索问题、上下文问题还是生成问题。

## RAG 流程

```text
用户问题 -> 查询改写 -> 检索 -> 重排 -> 上下文压缩 -> Prompt 拼装 -> 生成 -> 引用校验 -> 评估
```

每一步都可能失败。Prompt 只覆盖后半段，不能弥补完全错误的检索结果。

## 查询改写

用户问题可能太短或含糊。查询改写可以补全同义词、实体、时间范围和领域术语。

例子：

```text
用户问：试用期能休假吗？
改写为：试用期 员工 年假 休假 请假 政策
```

> **易错：** 改写时加入用户没问的限定条件。正确做法：扩展检索词，不改变问题意图。

## Chunk 设计

| 维度 | 取舍 |
| --- | --- |
| chunk 太小 | 容易丢上下文 |
| chunk 太大 | 检索噪声高，token 成本高 |
| 无标题 | 模型难判断来源 |
| 无编号 | 引用不可追踪 |

推荐每个 chunk 带：

- `doc_id`
- `title`
- `source`
- `date`
- `section`
- `text`

## 重排和上下文压缩

检索返回的 top-k 不一定都适合放进 prompt。重排负责按问题相关性重新排序；上下文压缩负责去掉无关段落，只保留能回答问题的证据。

Prompt 中可要求：

```text
先从 <context> 中列出与问题直接相关的证据片段，再回答。
不要引用无法支持结论的资料。
```

## 引用评估

引用不是装饰。要评估：

- 答案是否有引用。
- 引用是否真实存在。
- 引用是否支持该结论。
- 是否遗漏关键冲突资料。
- 是否把多个来源混成一个结论。

## RAG 失败归因

| 现象 | 可能原因 | 修复 |
| --- | --- | --- |
| 答案完全错 | 检索没召回正确文档 | 优化索引、查询改写 |
| 答案有资料但乱用 | 上下文太噪 | 重排、压缩、分块 |
| 答案编造 | prompt 未要求拒答 | 加引用和拒答规则 |
| 引用不支持结论 | 生成阶段问题 | 引用校验、二次审查 |
| 遗漏冲突资料 | top-k 或重排问题 | 增加冲突检测 |

## 练习

设计一个公司知识库 RAG 评估集，包含：

- 有明确答案。
- 资料不足。
- 多文档冲突。
- 旧制度和新制度冲突。
- 用户问题含糊。

## 验收

- 能画出 RAG 全链路。
- 能区分检索失败和生成失败。
- 能设计引用准确性评估。

## 难点

很多团队只优化生成 prompt，却不评估检索召回和引用正确性。RAG 的质量上限经常由检索和文档治理决定。
