# Prompt Engineering 学习路线图

## 阶段 1：基础认知

- 目标：理解 Prompt Engineering 解决什么问题，以及它不能解决什么问题。
- 需要掌握：大语言模型、上下文窗口、token、非确定性、幻觉、指令遵循、任务目标、成功标准。
- 例子：让模型“写一份简历建议”和让模型“按给定 JD、候选人经历、评分标准输出可执行修改建议”是两个完全不同的任务。
- 练习：把一个含糊请求改写为包含目标、受众、上下文、限制和输出格式的请求。
- 验收：能解释为什么“提示词更长”不等于“效果更好”。
- 重点：Prompt 的本质是任务规格说明，不是魔法咒语。
- 易错：只追求炫技模板，不定义可验证的成功标准。

## 阶段 2：提示结构和输出控制

- 目标：能写出稳定、可读、可调试的提示词。
- 需要掌握：角色、任务、上下文、约束、步骤、示例、反例、输出格式、Markdown/XML 分隔、JSON Schema。
- 例子：客服总结任务中，把“对话原文”“分类规则”“输出字段”分开，减少模型混淆。
- 练习：为“面试反馈生成器”写一个提示词，要求输出 `strengths`、`risks`、`next_steps` 三个字段。
- 验收：同一输入多次运行时，输出字段稳定且没有多余说明。
- 重点：清晰边界、明确格式、少量高质量示例比堆砌形容词更有效。
- 难点：既要限制输出，又不能把模型限制到无法完成任务。

## 阶段 3：上下文、示例和知识注入

- 目标：能处理模型不知道、记不准或需要依赖私有资料的任务。
- 需要掌握：few-shot、negative examples、检索增强生成、引用约束、长上下文规划、上下文压缩。
- 例子：生成公司制度问答时，不让模型凭常识回答，而是要求只依据检索到的制度片段回答。
- 练习：给模型 3 段政策文本，设计提示词让它回答并标注依据。
- 验收：模型能区分“资料中有答案”和“资料中没有答案”。
- 易错：把资料塞进上下文但不说明优先级和引用规则，导致模型混合常识和资料。

## 阶段 4：工具调用、Agent 和工作流

- 目标：能让模型在受控边界内调用工具、执行多步骤任务并维护状态。
- 需要掌握：函数调用、工具 schema、参数校验、工具选择、计划-执行-检查、记忆、多轮状态、失败重试。
- 例子：旅行规划 Agent 不能只输出建议，还要查航班、查酒店、校验预算，并在用户确认后再预订。
- 练习：设计一个“简历匹配职位”的工具调用流程，包括解析简历、检索职位、评分、输出建议。
- 验收：每个工具都有清晰输入输出、权限边界和失败处理。
- 难点：Agent 的问题通常不在“模型会不会想”，而在工具权限、状态污染和错误恢复。

## 阶段 5：评估、安全和生产化

- 目标：把提示词从一次性聊天推进到可上线、可回归、可治理的工程资产。
- 需要掌握：离线评测集、人工评分、LLM-as-judge、A/B 测试、版本管理、提示注入、防泄露、成本和延迟。
- 例子：修改提示词前先跑 100 条代表性样例，比较准确率、格式合规率、幻觉率和人工评分。
- 练习：为一个分类任务建立 20 条测试样例和评分规则。
- 验收：能说明一次 prompt 改动为什么变好，能回滚，能定位失败模式。
- 重点：没有评估的 prompt 优化只是主观试错。
- 易错：把安全要求只写在 prompt 中，不做输入过滤、权限隔离和输出审计。

## 推荐学习节奏

| 周期 | 学习重点 | 产出 |
| --- | --- | --- |
| 第 1 周 | 基础概念、提示结构、输出格式 | 5 个可复用任务 prompt |
| 第 2 周 | 示例、上下文、RAG 和长上下文 | 带引用的知识问答 prompt |
| 第 3 周 | 工具调用、Agent 和多轮状态 | 一个工具调用流程设计 |
| 第 4 周 | 评估、安全和生产化 | 测试集、评分表、版本记录 |
| 第 5 周 | 综合项目和面试复盘 | 可演示 Demo、设计说明、面试答案 |

## 最终能力清单

- 能把模糊需求转成可执行 prompt 规格。
- 能控制输出结构，减少格式漂移。
- 能用示例、反例和上下文提升稳定性。
- 能设计 RAG 和长上下文提示策略。
- 能设计工具调用、Agent 和失败恢复边界。
- 能建立评估集并迭代 prompt。
- 能识别提示注入、数据泄露和越权风险。
- 能在面试中讲清楚 prompt、RAG、tool use、fine-tuning、evaluation 的取舍。
