# SDD与Harness Engineering学习资料：规格解析冲突消解和版本治理

[返回索引](../SDD与Harness Engineering学习资料.md)

## 学习目标

- 掌握多份规格同时存在时的解析、优先级和冲突消解。
- 理解规格版本、兼容性和变更影响分析。
- 能为 Agent 定义明确的规格选择规则。

## 理论导读

真实项目中不会只有一份规格。Agent 可能同时看到产品需求、接口契约、架构规则、历史 ADR、测试说明和团队规范。如果没有优先级，Agent 会在冲突信息中随机选择一条看起来合理的规则。

规格治理的核心是让 Agent 和人都知道：冲突出现时听谁的，旧规格和新规格怎么区分，变更会影响哪些地方。

## 核心心智模型

```text
规格库不是文档仓库，而是规则系统。
规则系统必须有来源、优先级、版本、作用域和冲突处理。
```

## 知识点详解

### 1. 规格元数据

每份规格都应包含元数据：

```yaml
spec_id: order-cancel-v2
domain: order
version: 2.1.0
status: active
owner: order-team
updated_at: 2026-06-07
scope:
  services:
    - order-service
  apis:
    - POST /orders/{id}/cancel
depends_on:
  - payment-refund-v1
  - inventory-release-v1
supersedes:
  - order-cancel-v1
```

这些字段不是装饰，它们给 Agent 提供解析依据。

### 2. 规格优先级

默认优先级可以这样设计：

```text
法律合规和安全策略
> 当前任务 feature spec
> API/事件契约
> 领域状态机和不变量
> 架构边界和 ADR
> 团队代码规范
> 历史 PR 说明
> 非权威聊天记录
```

注意：这只是默认顺序。特定团队可以改，但必须显式化。

### 3. 冲突类型

| 冲突类型 | 例子 | 处理方式 |
| --- | --- | --- |
| 时间冲突 | 新规格允许取消，旧规格禁止取消 | 根据版本和状态选择 active 规格 |
| 作用域冲突 | 移动端允许字段为空，后端契约不允许 | 检查作用域和接口边界 |
| 目标冲突 | 性能要求低延迟，合规要求强审计 | 高优先级规则优先 |
| 实现冲突 | 任务要求改 payment，但权限禁止 | 转人工审批 |
| 术语冲突 | refund 在不同团队含义不同 | 建立术语表和领域边界 |

### 4. 冲突消解决策树

```text
发现冲突
  -> 是否有明确版本关系？
     -> 有：选择 active 且最新的适用版本
     -> 没有：继续
  -> 是否作用域不同？
     -> 是：按服务、API、用户群体拆开
     -> 否：继续
  -> 是否涉及安全、合规、数据正确性？
     -> 是：高优先级规则胜出并转人工确认
     -> 否：继续
  -> 是否能通过兼容策略同时满足？
     -> 是：制定过渡方案
     -> 否：记录冲突并阻止 Agent 自动执行
```

### 5. 版本治理

规格版本可以借鉴语义化版本：

- 主版本：破坏兼容的业务或接口变更。
- 次版本：新增功能或规则，不破坏旧行为。
- 补丁版本：澄清文字、修复歧义、补测试说明。

示例：

```text
order-cancel 1.2.0 -> 2.0.0
原因：已发货订单从“允许客服强制取消”改为“禁止取消，只能售后”。
影响：客服后台、订单 API、售后流程、退款策略、测试用例。
```

### 6. 变更影响分析

Agent 在修改规格前后应执行影响分析：

```text
规格变更 -> 影响领域模型 -> 影响 API -> 影响测试 -> 影响数据迁移 -> 影响监控和告警
```

影响矩阵：

| 变更 | 影响模块 | 必须检查 |
| --- | --- | --- |
| 新增订单取消原因 | order-service | 状态机、数据库字段、API响应 |
| 修改退款失败状态 | payment-service | 退款状态枚举、消息消费者 |
| 增加审计日志 | audit-service | 日志格式、隐私脱敏 |

## 练习

设计一套规格优先级和冲突消解规则，用于“订单、支付、库存”三个团队共同维护的电商系统。

## 验收

- 能定义规格元数据。
- 能解释冲突优先级。
- 能阻止 Agent 在规格冲突时盲目执行。

## 重点

Agent 不应该在高风险规格冲突中自由发挥。Harness 必须把冲突识别为阻塞或审批点。

## 难点

难点是版本和作用域。很多冲突并不是真冲突，而是不同版本、不同用户群或不同系统边界下的规则。

## 易错

> **易错：** 让 Agent 自动“综合”两条冲突规格。
>
> 正确做法：低风险冲突可以建议方案，高风险冲突必须转人工决策。
