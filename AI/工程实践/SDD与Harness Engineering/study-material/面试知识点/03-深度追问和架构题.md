## 一、深度追问

### 1. 如何判断一份 SDD 规格是“正确”的？

**参考答案：**

我会从五个维度判断：完备性、一致性、可验证性、可追踪性和可演进性。完备性看是否覆盖主流程、异常、权限和副作用；一致性看是否与 API、状态机和架构规则冲突；可验证性看是否能转成测试或检查器；可追踪性看需求、实现和测试能否对应；可演进性看变更后是否能做影响分析。

> **重点：** 规格正确性不是文档格式，而是能不能驱动实现、验证和维护。

### 2. 多份规格冲突时，Agent 应该怎么处理？

**参考答案：**

Agent 不应该自由综合高风险冲突。Harness 应先检查版本、状态和作用域，再按优先级处理。一般安全合规高于当前任务，当前 feature spec 高于旧文档，API 契约和领域不变量高于历史 PR 评论。如果冲突涉及安全、数据正确性或兼容性，应进入阻塞状态并转人工决策。

> **易错：** 让模型自己猜哪个规格更合理。正确做法是用元数据、优先级和审批机制消解。

### 3. 你会如何设计 Agent Harness 的运行时？

**参考答案：**

我会把运行时设计成状态机，包括 CREATED、CONTEXT_LOADING、PLANNING、WAITING_APPROVAL、EXECUTING、VERIFYING、REPAIRING、SUMMARIZING、COMPLETED，以及 BLOCKED_BY_SPEC_CONFLICT、BLOCKED_BY_PERMISSION、FAILED_BY_VALIDATION 等异常状态。每次工具调用都经过 Tool Gateway 做权限、参数、审计和脱敏。验证失败进入有限重试，重复失败转人工。

> **重点：** 深度回答要讲状态、控制循环、停止条件和重试预算。

### 4. 上下文选择如何做，不能只说“给更多上下文”？

**参考答案：**

我会把上下文分为必选和打分选择。必选包括当前规格、领域状态机、API 契约、相关测试和权限策略。其他候选按相关性、权威性、新鲜度、具体性和历史使用信号打分。长日志和大文件要压缩成错误摘要、相关函数和决策结论，而不是简单塞进上下文窗口。

> **难点：** 上下文选择是排序和压缩问题，不是堆材料。

### 5. 如何评估 Harness 是否真的有效？

**参考答案：**

要建立 Eval 任务集和指标。任务集覆盖功能新增、bug 修复、边界补全、规格冲突、权限拒绝、高风险任务和测试失败修复。指标包括任务成功率、首次通过率、回归率、越界修改率、审批违规率、测试选择准确率、上下文 precision/recall、平均修复次数和转人工质量。

> **重点：** 只看成功率不够，还要看风险拦截和质量成本。

### 6. 如果 Agent 修改代码后测试通过但线上出问题，你怎么复盘？

**参考答案：**

我会先对照规格和 trace，检查是否是规格缺失、上下文缺失、工具缺失、权限过宽、验证缺失、Runtime 问题或模型误判。然后补对应的 Harness 能力：例如补规格边界、补上下文检索规则、补测试或架构检查、收紧权限、增加 Eval，并把事故纳入回归任务集。

> **重点：** 复盘目标是让同类问题更难再次发生。

## 二、架构设计题

### 7. 设计一个组织级 SDD + Harness 平台，你会有哪些核心模块？

**参考答案：**

核心模块包括 Spec Store、Policy Store、Knowledge Index、Agent Gateway、Harness Runtime、Tool Gateway、Verifier Runner、Trace Store、Metrics/Eval Store 和 Review/Approval。Spec Store 管版本和作用域，Policy Store 管权限和审批，Knowledge Index 做上下文检索，Runtime 管状态和控制循环，Tool Gateway 管工具安全，Verifier Runner 管测试和检查，Trace/Metrics/Eval 支持复盘和持续改进。

> **重点：** 架构题要讲模块职责和数据流，不只列工具名。

### 8. 哪些任务适合 Agent 自动做，哪些必须人工审批？

**参考答案：**

低风险、规格清晰、验证充分的任务适合自动化，例如小范围 bug 修复、测试补全、文档同步、局部重构。涉及支付、认证授权、数据库迁移、生产配置、隐私数据、安全策略、依赖新增和跨系统架构变更的任务必须审批。高风险任务可以让 Agent 辅助分析，但不能无审批执行。

> **易错：** 不是按“Agent 能不能做”判断，而是按风险、规格清晰度和验证能力判断。
