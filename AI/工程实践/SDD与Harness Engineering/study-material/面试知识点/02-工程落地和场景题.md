## 一、工程落地

### 1. 如果让你在团队中落地 SDD + Harness，你会怎么做？

**参考答案：**

我会从低风险功能开始，不直接平台化。第一步建立 feature spec 模板，要求写用户场景、约束、验收标准和不做范围。第二步定义 Agent 规则，包括可读目录、可改目录、禁止目录、测试命令和审批操作。第三步把关键验证接入 CI。第四步建立失败复盘模板，把失败归类为规格、上下文、工具、权限、验证或模型问题。跑通闭环后再逐步沉淀成平台。

> **重点：** 先跑通闭环，再平台化。

### 2. Agent 经常修改不该修改的模块，你如何处理？

**参考答案：**

先判断是规格边界没写，还是权限控制缺失。如果规格没有说明禁止修改范围，就补充模块边界。如果 Agent 已经知道边界仍然修改，就把规则从 prompt 升级为 Harness 权限：限制可编辑路径，增加架构依赖检查，要求高风险模块修改审批。同时把这次失败加入复盘和回归检查。

> **重点：** 不只修这次输出，要修导致错误再次发生的系统缺口。

### 3. 如何判断 Agent 输出已经完成？

**参考答案：**

不能只看 Agent 自己说完成。应该对照规格检查验收标准，运行测试、lint、类型检查和必要的架构检查；对于业务高风险功能，还要检查异常路径、幂等、兼容性、安全和回滚。最后由人评审变更摘要和未解决风险。

> **易错：** 测试通过不等于完全正确，测试只是完成证据之一。

### 4. Agent 失败后，你如何做复盘？

**参考答案：**

我会记录任务、规格版本、读取的上下文、工具调用、修改文件、测试结果和失败现象。然后把失败分类为规格缺失、上下文缺失、工具缺失、权限问题、验证缺失或模型误判。最后必须产出改进行动，例如补规格、补测试、限制权限、增加上下文检索规则或设置转人工阈值。

> **重点：** 复盘的产物应该是 Harness 改进，而不只是问题记录。

### 5. 哪些任务不适合直接交给 Agent？

**参考答案：**

高风险、规格不清、验证不完善或涉及敏感权限的任务不适合直接交给 Agent。例如支付、认证授权、数据库迁移、生产配置、安全策略、隐私数据处理、大规模删除和跨系统架构调整。可以让 Agent 辅助分析和生成草案，但最终执行需要更强审批和人工控制。

> **难点：** 不是所有任务都不能用 Agent，而是要匹配风险级别和 Harness 成熟度。

### 6. 如何避免 SDD 变成文档负担？

**参考答案：**

规格要围绕当前变更，不追求一次写完整个系统。模板要短而关键，重点写场景、约束、验收和风险。规格要直接连接任务、测试和评审，否则就会变成额外负担。团队还应定期删除或更新过期规格。

> **易错：** 文档越多越好。正确做法是规格越能驱动实现和验证越好。
