# SDD与Harness Engineering学习资料：总览与心智模型

[返回索引](../SDD与Harness Engineering学习资料.md)

## 学习目标

- 建立 SDD 与 Harness Engineering 的整体认知。
- 理解为什么 AI 编码时代需要“规格 + Harness”。
- 能用一句话解释二者关系。

## 理论导读

在传统软件开发中，需求、设计、代码、测试和发布通常由人串联。AI 编码 Agent 出现后，代码生成速度提升了，但一个更严重的问题暴露出来：Agent 不一定理解业务边界、架构约束、历史上下文、团队规范和完成标准。

SDD 的思路是把“想做什么”前置为规格，让规格成为开发的中心工件。Harness Engineering 的思路是把 Agent 放进一个工程化外壳里，让它只能在受控上下文、受控工具、受控权限和受控验证下工作。

## 核心心智模型

可以把 AI Agent 想象成一个很强但不了解公司规章的新工程师：

- SDD 是任务说明、验收标准、设计约束和变更记录。
- Harness 是工位、权限、代码库入口、测试环境、代码评审规则和事故复盘流程。
- 模型能力是个人能力，Harness 能力是组织能力。

单独有 SDD，Agent 可能知道目标但执行失控。单独有 Harness，Agent 被管住了但可能不知道真正意图。二者组合后，才能形成“意图清晰、执行受控、结果可验”的工作方式。

## 知识覆盖图

| 领域 | 需要掌握 |
| --- | --- |
| 基础术语 | SDD、spec、task、contract、agent、harness、feedback loop |
| 规格机制 | 需求、设计、任务、验收、约束、变更、版本 |
| Agent机制 | 上下文窗口、工具调用、权限、状态、轨迹、失败重试 |
| Harness机制 | 沙箱、脚手架、检查器、测试、观测、记忆、策略 |
| 验证机制 | TDD、属性测试、静态分析、架构规则、回归样例 |
| 团队治理 | 审批、权限、审计、合规、成本、维护、知识沉淀 |
| 面试评估 | 概念区分、落地方案、风险识别、故障复盘 |

## 例子

需求：“做一个订单取消功能。”

弱 prompt：

```text
帮我加一个订单取消功能。
```

SDD 规格片段：

```text
用户场景：
- 已支付未发货订单允许取消。
- 已发货订单不允许取消，只能申请售后。

业务约束：
- 取消成功后必须释放库存。
- 取消成功后必须发起退款流程。
- 重复取消请求必须幂等。

验收标准：
- 未发货订单取消后状态变为 CANCELLED。
- 重复调用取消接口不产生重复退款。
- 发货订单返回明确错误码 ORDER_ALREADY_SHIPPED。
```

Harness 规则片段：

```text
Agent执行规则：
- 只能修改 order 模块和相关测试。
- 必须先读取 OrderStatus、PaymentService、InventoryService。
- 修改后必须运行订单模块单元测试。
- 如果涉及事件发布，必须补充幂等性测试。
```

## 练习

选择一个你熟悉的功能，例如登录、搜索、文件上传、优惠券使用，分别写：

- 3 条用户场景。
- 5 条业务约束。
- 5 条验收标准。
- 5 条 Agent 执行规则。

## 验收

- 能说清楚 SDD 和 Harness 分别解决的问题。
- 能把一个模糊需求拆成规格和执行约束。
- 能指出 Agent 输出失败时是规格问题、上下文问题、工具问题还是验证问题。

## 重点

SDD 的核心不是写文档，而是把需求变成可执行、可审查、可验证的工程契约。

Harness Engineering 的核心不是写更长提示词，而是设计 Agent 的运行环境和反馈系统。

## 难点

真正困难的是把隐性团队知识显性化：哪些目录不能改、哪些 API 不能破坏、哪些测试必须跑、哪些业务规则不能违反。这些知识过去存在于资深工程师脑中，现在要进入规格和 Harness。

## 易错

> **易错：** 以为模型越强就越不需要规格和 Harness。
>
> 正确做法：模型越强，越需要明确边界。能力越强的 Agent 能更快完成任务，也能更快造成大范围错误。
