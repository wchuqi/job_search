# SDD与Harness Engineering学习资料：综合练习项目

[返回索引](../SDD与Harness Engineering学习资料.md)

## 项目目标

设计一个“AI Agent 修改订单取消功能”的最小 SDD + Harness 方案。目标不是真的接入某个模型，而是产出可复用的工程材料。

## 项目背景

电商系统已有订单模块。现在要新增订单取消功能，要求：

- 已支付未发货订单可取消。
- 已发货订单不可取消。
- 取消成功要释放库存。
- 取消成功要发起退款。
- 重复取消必须幂等。
- 操作必须记录审计日志。

## 任务 1：编写 Feature Spec

产出文件：`specs/order-cancel.md`

必须包含：

- 背景。
- 用户场景。
- 功能需求。
- 非功能需求。
- 边界条件。
- 验收标准。
- 不做范围。

## 任务 2：编写 Harness 规则

产出文件：`agent-rules/order-cancel-harness.yml`

必须包含：

- 允许读取目录。
- 允许修改目录。
- 禁止修改目录。
- 必须先读的文件。
- 必须运行的测试。
- 需要审批的操作。
- 失败重试策略。

示例骨架：

```yaml
task: order-cancel
context:
  must_read:
    - specs/order-cancel.md
    - docs/order-state-machine.md
    - src/order/OrderStatus.java
permissions:
  allow_edit:
    - src/order/**
    - test/order/**
  deny_edit:
    - infra/**
    - src/payment/**
approval:
  required_for:
    - database_migration
    - dependency_change
validation:
  commands:
    - mvn test -Dtest=OrderCancel*
retry:
  max_attempts: 2
  escalate_on:
    - same_test_failed_twice
    - permission_boundary_needed
```

## 任务 3：设计验证矩阵

产出文件：`verification/order-cancel-matrix.md`

| 场景 | 输入 | 预期 | 验证方式 |
| --- | --- | --- | --- |
| 未发货取消 | PAID + NOT_SHIPPED | CANCELLED | 单元测试 |
| 已发货取消 | SHIPPED | 409错误 | 单元测试 |
| 重复取消 | 两次请求 | 只退款一次 | 幂等测试 |
| 库存释放失败 | 库存服务异常 | 状态不乱改 | 集成测试 |

## 任务 4：设计失败复盘模板

产出文件：`retrospective/agent-failure-template.md`

必须能区分：

- 规格缺失。
- 上下文缺失。
- 工具缺失。
- 权限问题。
- 验证缺失。
- 模型误判。

## 任务 5：答辩说明

用 5 分钟讲清楚：

- 为什么这个功能适合 SDD。
- 哪些规则属于 Harness。
- 哪些风险不能交给 Agent 自行判断。
- 如果 Agent 失败，你如何改进 Harness。

## 验收标准

- Feature Spec 可直接指导实现。
- Harness 规则有明确允许、禁止和审批边界。
- 验证矩阵覆盖 happy path、异常、幂等和失败恢复。
- 复盘模板能推动后续改进。

## 重点

这个项目考察的是工程控制能力，不是模型使用技巧。

## 难点

难点是把“业务正确性”变成测试和规则，而不是只写自然语言说明。

## 易错

> **易错：** 只写 Agent 提示词，不写验证矩阵。
>
> 正确做法：没有验证矩阵，Agent 输出无法稳定评估。
