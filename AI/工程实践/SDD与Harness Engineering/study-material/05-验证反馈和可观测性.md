# SDD与Harness Engineering学习资料：验证反馈和可观测性

[返回索引](../SDD与Harness Engineering学习资料.md)

## 学习目标

- 理解验证闭环在 Harness 中的核心地位。
- 掌握常见验证方式和失败分类。
- 能设计 Agent 轨迹和复盘机制。

## 理论导读

没有验证的 Agent 输出只是候选答案。Harness Engineering 要求把“是否完成”从主观判断转成可执行检查。测试、静态分析、属性测试、架构规则和人工评审都可以成为验证器。

可观测性则负责记录 Agent 如何到达结果。没有轨迹，就无法复盘失败；无法复盘，就无法改进 Harness。

## 核心心智模型

```text
规格给出预期
Agent 产生变更
验证器检查结果
观测系统记录过程
复盘把失败转成新规则
```

## 知识点详解

### 1. 验证方式

| 验证方式 | 能发现的问题 | 局限 |
| --- | --- | --- |
| 单元测试 | 局部逻辑错误 | 覆盖不到集成行为 |
| 集成测试 | 模块协作问题 | 成本较高 |
| 端到端测试 | 用户路径问题 | 运行慢，定位难 |
| 属性测试 | 边界和不变量问题 | 需要抽象能力 |
| 静态分析 | 类型、风格、安全问题 | 不理解全部业务语义 |
| 架构检查 | 依赖方向、模块边界 | 需要规则维护 |
| LLM评审 | 语义和一致性问题 | 不能作为唯一依据 |
| 人工评审 | 业务判断和风险兜底 | 成本高，不可无限扩展 |

### 2. 反馈循环

失败不是终点，而是 Harness 的训练材料。

失败分类：

- 规格缺失：需求或边界没写。
- 上下文缺失：Agent 没看到关键文件。
- 工具缺失：Agent 无法运行必要检查。
- 权限过宽：Agent 改了不该改的东西。
- 权限过窄：Agent 无法完成必要操作。
- 验证缺失：错误没有被测试或检查器捕获。
- 模型误判：已有信息足够，但 Agent 推理错误。

### 3. 可观测字段

建议记录：

- 任务 ID。
- 使用的规格版本。
- 读取的上下文文件。
- 调用的工具和命令。
- 修改的文件。
- 测试结果。
- 失败原因。
- 重试次数。
- 人工介入点。
- 最终结论。

## 例子

失败复盘模板：

```markdown
# Agent失败复盘

## 任务
实现订单取消。

## 失败现象
重复取消产生两条退款记录。

## 分类
验证缺失 + 规格缺失。

## 根因
规格只写了取消成功，没有写幂等要求；测试没有覆盖重复请求。

## Harness改进
- 在订单状态规格中加入幂等约束。
- 新增重复取消测试。
- Agent 修改订单状态机时必须运行幂等测试。
```

## 练习

找一个你过去遇到的 bug，用上述模板复盘，并写出至少 3 条 Harness 改进。

## 验收

- 能把失败归类到正确原因。
- 能把失败转化为规格、上下文、工具、权限或验证改进。
- 能设计一套最小观测字段。

## 重点

Harness Engineering 的复利来自“错误不白犯”。每个失败都应该留下新的规则、测试或上下文策略。

## 难点

难点是区分模型错误和系统错误。很多看似模型犯傻的问题，本质是规格、上下文或验证缺失。

## 易错

> **易错：** 测试通过就认为 Agent 输出一定正确。
>
> 正确做法：测试只是证据之一。还要检查规格覆盖、架构边界、兼容性、安全和未测试风险。
