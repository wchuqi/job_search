# SDD与Harness Engineering学习资料：组织落地成熟度和平台化架构

[返回索引](../SDD与Harness Engineering学习资料.md)

## 学习目标

- 理解 SDD + Harness 从个人实践到组织平台的演进。
- 能设计组织级平台架构。
- 能识别平台化的成本和边界。

## 理论导读

个人使用 Agent 可以靠经验和手工控制，团队级使用必须平台化。平台化不是一开始造大系统，而是在多次任务中把重复出现的规格、上下文、工具、验证、审批和复盘流程沉淀成可复用能力。

## 成熟度模型

| 级别 | 名称 | 特征 | 主要风险 |
| --- | --- | --- | --- |
| L0 | 随机使用 | 个人随手 prompt | 不可复盘 |
| L1 | 规格先行 | 有 feature spec | 质量依赖个人 |
| L2 | 规则文件 | 有路径权限和测试命令 | 缺少统一执行 |
| L3 | Harness运行时 | 有状态机、工具网关、验证器 | 维护成本上升 |
| L4 | Eval驱动 | 有任务集和质量指标 | 指标设计偏差 |
| L5 | 组织平台 | 规格库、权限、审计、审批、知识库集成 | 平台复杂度和治理负担 |

## 平台化架构

```text
                  +----------------------+
                  | Developer / Reviewer |
                  +----------+-----------+
                             |
                             v
 +-------------+     +---------------+     +--------------+
 | Spec Store  | --> | Agent Gateway | <-- | Policy Store |
 +-------------+     +-------+-------+     +--------------+
                             |
                             v
                     +---------------+
                     | Harness Runtime|
                     +---+-------+---+
                         |       |
                         v       v
                  Tool Gateway  Context Builder
                         |       |
                         v       v
                    Workspace  Knowledge Index
                         |
                         v
                  Verifier Runner
                         |
                         v
              Trace / Metrics / Eval Store
                         |
                         v
                 Review and Governance
```

## 核心平台能力

### 1. Spec Store

职责：

- 存储规格。
- 管理版本。
- 标记状态。
- 支持作用域。
- 提供搜索和冲突检测。

### 2. Policy Store

职责：

- 路径权限。
- 工具权限。
- 审批规则。
- 风险分级。
- 组织安全策略。

### 3. Knowledge Index

职责：

- 索引代码、文档、接口契约、测试、历史 PR。
- 支持上下文检索。
- 标记权威性和新鲜度。

### 4. Tool Gateway

职责：

- 统一工具入口。
- 权限校验。
- 命令审计。
- 输出摘要。
- 敏感信息脱敏。

### 5. Verifier Runner

职责：

- 根据规格和修改范围选择测试。
- 运行 lint、类型检查、安全扫描。
- 汇总失败。
- 将失败反馈给 Runtime。

### 6. Trace / Metrics / Eval Store

职责：

- 保存执行轨迹。
- 统计质量指标。
- 维护 Eval 任务集。
- 支持事故复盘。

## 组织流程

```text
需求进入 -> 生成/更新规格 -> 规格评审 -> Agent执行 -> Harness验证 -> 人工评审 -> 合并 -> 指标统计 -> 复盘改进
```

## 平台化边界

不要平台化的内容：

- 未稳定的个人实验。
- 偶发任务的复杂规则。
- 无法复用的特殊审批。
- 没有质量指标支撑的自动化。

应该平台化的内容：

- 重复出现的任务类型。
- 高风险权限控制。
- 常用上下文检索。
- 强制验证。
- 失败复盘。
- Eval 评估。

## 例子：小团队落地路线

```text
第1周：统一 feature spec 模板和 Agent 规则文件
第2周：为核心仓库接入必跑测试和路径权限
第3周：记录 Agent 轨迹和失败原因
第4周：建立 10 个 Eval 任务
第2个月：接入审批、指标和知识索引
第3个月：沉淀组织级 Harness 平台
```

## 练习

为一个 20 人后端团队设计 Harness 平台 MVP。必须说明：

- 先支持哪些任务。
- 哪些模块禁止 Agent 自动改。
- 如何存储规格。
- 如何选择上下文。
- 如何运行验证。
- 如何统计指标。
- 如何接入人工审批。

## 验收

- 能画出组织级架构。
- 能区分个人工作流和团队平台。
- 能提出渐进落地方案，而不是一开始搭大而空的平台。

## 重点

平台化的目标是降低重复风险和提高组织学习速度，不是把所有开发动作都自动化。

## 难点

难点是成本控制。Harness 平台本身也会成为系统，需要维护、观测和治理。

## 易错

> **易错：** 把工具采购当成 Harness Engineering。
>
> 正确做法：工具只是载体。真正的 Harness 是围绕规格、上下文、权限、验证和反馈建立的组织能力。
