# SDD与Harness Engineering学习资料：故障推演和深度排障案例

[返回索引](../SDD与Harness Engineering学习资料.md)

## 学习目标

- 掌握 Agent 失败的深度归因方法。
- 能从故障中推导 Harness 改进。
- 能面向生产风险设计事故预防。

## 理论导读

Agent 失败不能只写“模型输出错了”。工程上更有价值的问题是：为什么系统允许这个错误发生，为什么验证没有拦住，为什么上下文没有暴露关键规则，为什么权限允许越界，为什么复盘没有转成新规则。

## 故障分类框架

| 类别 | 根因 | 典型表现 | 改进方向 |
| --- | --- | --- | --- |
| S1 规格缺失 | 没写边界或副作用 | 只实现主流程 | 补规格和验收 |
| S2 规格冲突 | 多份规则矛盾 | Agent 随机选 | 优先级和冲突阻塞 |
| C1 上下文缺失 | 没看到关键文件 | 改错实现点 | Context Builder |
| C2 上下文污染 | 旧文档误导 | 使用废弃规则 | 版本和权威性过滤 |
| T1 工具缺失 | 无法验证或搜索 | 靠猜测修复 | 增加工具 |
| P1 权限过宽 | 可改不该改文件 | 越界修改 | 权限收紧 |
| P2 权限过窄 | 无法完成必要修改 | 反复失败 | 审批扩展 |
| V1 验证缺失 | 错误未被测试捕获 | 通过但有 bug | 补测试和检查器 |
| R1 Runtime问题 | 无限重试或错误停止 | 任务卡死 | 状态机和预算 |
| M1 模型误判 | 信息足够仍推理错 | 错误解释 | 更强提示、拆任务、转人工 |

## 案例 1：重复退款事故

### 背景

Agent 根据规格实现订单取消，测试通过，但线上出现重复退款。

### 现象

- 用户重复点击取消。
- 系统产生两条退款请求。
- 订单状态最终是 CANCELLED，但支付侧出现重复退款。

### 初始判断

不能直接归因“Agent 写错”。要沿链路排查：

```text
规格是否写幂等 -> Agent 是否看到幂等策略 -> 是否有幂等测试 -> 权限是否允许改 payment -> Runtime 是否强制验证
```

### 根因

- S1：规格只写“取消后退款”，没写“重复取消必须幂等”。
- C1：Agent 没看到支付侧幂等键约定。
- V1：测试只验证订单状态，没有验证退款请求数量。

### Harness改进

- 在订单取消规格中新增 `idempotency_required: true`。
- Context Builder 对幂等任务必选 `payment-idempotency.md`。
- Verifier Runner 自动运行重复请求测试。
- Eval 集新增 `order-cancel-duplicate-refund`。

## 案例 2：Agent越界重构公共模块

### 背景

任务只是修复订单展示字段，Agent 顺手重构了公共日期格式化工具，导致其他模块格式变化。

### 根因

- P1：权限过宽，允许修改 shared 工具。
- V1：没有快照测试或公共 API 兼容检查。
- R1：Runtime 没有检测修改扩散。

### Harness改进

```text
允许修改范围 = 当前模块 + 相关测试
shared/ 修改需要审批
公共工具改动必须运行全量兼容测试
变更摘要必须列出跨模块影响
```

## 案例 3：规格冲突导致错误实现

### 背景

新规格要求“VIP 用户可延迟付款”，旧 API 文档仍要求“订单创建时必须支付”。Agent 选择旧 API 文档，实现错误。

### 根因

- S2：新旧规格冲突。
- C2：旧文档仍为 active。
- Runtime 没有检测冲突。

### Harness改进

- 规格加入 `status`、`version`、`supersedes`。
- Context Builder 过滤 deprecated 文档。
- 冲突时进入 `BLOCKED_BY_SPEC_CONFLICT`，不允许 Agent 继续实现。

## 深度排障流程

```text
1. 确认失败现象
2. 对照规格验收标准
3. 检查 Agent 实际读取的上下文
4. 检查工具调用和权限记录
5. 检查验证是否覆盖失败路径
6. 检查 Runtime 是否正确停止或转人工
7. 分类根因
8. 产出 Harness 改进
9. 加入 Eval 或回归测试
```

## 练习

模拟一个“Agent 修改权限逻辑导致普通用户能访问管理员接口”的事故，完成：

- 失败现象。
- 根因分类。
- 规格缺口。
- 上下文缺口。
- 权限缺口。
- 验证缺口。
- Harness 改进。

## 验收

- 能区分直接原因和系统原因。
- 能把事故转成 Harness 改进。
- 能设计防止同类问题再次发生的 Eval。

## 重点

深度排障的目标不是找人背锅，也不是怪模型，而是找出 Harness 为什么没拦住。

## 难点

难点是证据链。没有 trace store、工具日志和规格版本，事故复盘只能靠猜。

## 易错

> **易错：** 只补 prompt。
>
> 正确做法：高价值修复通常是补规格、补上下文策略、补权限、补验证和补 Eval。
