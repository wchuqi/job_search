# SDD与Harness Engineering学习资料：SDD开发模式

[返回索引](../SDD与Harness Engineering学习资料.md)

## 学习目标

- 理解 SDD 的定义、价值和适用边界。
- 能区分 SDD、TDD、BDD、传统需求文档和系统设计文档。
- 能写出一份最小可用规格。

## 理论导读

SDD 通常指 `Specification-Driven Development` 或 `Spec-Driven Development`。在 AI 辅助开发语境中，它强调先把功能意图、约束、验收标准和任务拆解写成规格，再让人或 Agent 基于规格实现。

SDD 不是新瓶装旧文档。传统文档经常在实现前写一次，之后逐渐过期。SDD 要求规格持续参与开发过程：实现时引用规格，测试时验证规格，评审时对照规格，需求变化时更新规格。

## 核心心智模型

SDD 把“需求”升级成“契约”：

- 需求像一句愿望：“用户要能取消订单。”
- 规格像一份契约：“哪些订单可取消，取消后哪些状态变化必须发生，什么情况下拒绝，如何验证。”
- 任务像施工清单：“改状态机、补库存释放、接退款服务、写幂等测试。”

## 知识点详解

### 1. 规格的组成

一份实用规格通常包含：

- 背景：为什么要做。
- 用户场景：谁在什么情况下做什么。
- 功能需求：系统必须提供什么行为。
- 非功能需求：性能、安全、可靠性、兼容性、可观测性。
- 边界条件：异常、权限、并发、幂等、失败恢复。
- 验收标准：怎么判断完成。
- 任务拆解：实现步骤和依赖顺序。
- 变更记录：规格如何演进。

### 2. SDD 的常见流程

```text
idea -> clarify -> spec -> plan -> tasks -> implement -> verify -> review -> update spec
```

在 AI Agent 参与时，流程会变成：

```text
人定义意图 -> Agent辅助澄清 -> 生成规格 -> 人审查规格 -> Agent拆任务 -> Agent实现 -> Harness验证 -> 人评审结果
```

### 3. SDD 与相关方法对比

| 方法 | 中心工件 | 主要问题 | 与 SDD 的关系 |
| --- | --- | --- | --- |
| TDD | 测试 | 如何先定义可执行验证 | SDD 可把验收标准转成测试 |
| BDD | 行为场景 | 业务行为如何表达 | SDD 可吸收 Given/When/Then |
| DDD | 领域模型 | 业务概念如何建模 | SDD 可描述领域规则和边界 |
| 传统需求文档 | 文档 | 做什么 | SDD 要求文档可驱动实现和验证 |
| 系统设计文档 | 架构方案 | 如何设计系统 | SDD 中的设计规格可引用它 |

## 例子

最小 SDD 模板：

```markdown
# 功能规格：订单取消

## 背景
用户希望在未发货前取消订单，减少客服介入。

## 用户场景
- 用户取消已支付未发货订单。
- 用户重复点击取消按钮。
- 用户尝试取消已发货订单。

## 功能需求
- 系统必须校验订单状态。
- 系统必须释放库存。
- 系统必须发起退款。
- 系统必须保证重复请求幂等。

## 验收标准
- 未发货订单取消成功。
- 已发货订单取消失败并返回明确错误。
- 重复取消不会重复退款。
- 取消失败时订单状态不被错误修改。
```

## 练习

把“用户可以上传头像”写成 SDD 规格。必须覆盖：

- 文件大小限制。
- 文件格式限制。
- 鉴权。
- 覆盖旧头像。
- 上传失败回滚。
- CDN 或对象存储异常。

## 验收

- 规格能指导开发者写实现。
- 规格能指导测试人员写测试。
- 规格能让代码评审者判断改动是否完整。

## 重点

规格必须可验证。不可验证的规格只能表达愿望，不能驱动工程。

## 难点

SDD 的难点不是模板，而是把隐性需求挖出来。真正的业务 bug 往往来自“大家以为很明显，所以没人写”的边界条件。

## 易错

> **易错：** 把 SDD 写成大而全的需求说明，导致无人维护。
>
> 正确做法：规格要足够完整，但要围绕当前变更。每次变更都更新对应规格，不追求一次写完整个系统百科。
