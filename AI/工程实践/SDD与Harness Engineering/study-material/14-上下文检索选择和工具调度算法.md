# SDD与Harness Engineering学习资料：上下文检索选择和工具调度算法

[返回索引](../SDD与Harness Engineering学习资料.md)

## 学习目标

- 掌握上下文选择的排序规则。
- 理解工具调度不是随意调用，而是有前置条件和风险控制。
- 能设计一个简单的 Context Builder 和 Tool Scheduler。

## 理论导读

Agent 失败经常不是因为模型不会，而是因为上下文错了：读到了旧文档，漏了关键状态机，看了无关代码，或者把历史讨论当成当前规则。上下文工程的深度在于选择算法：在有限上下文窗口中选择最相关、最新、最权威、最可执行的材料。

工具调度也类似。Agent 不是想调用什么就调用什么，而应该按任务状态、权限、风险和验证需求选择工具。

## 核心心智模型

```text
Context Builder 决定 Agent 看什么。
Tool Scheduler 决定 Agent 能做什么、何时做、做完怎么验。
```

## 上下文选择算法

### 1. 候选来源

- 当前任务规格。
- 相关领域规格。
- API 契约。
- 状态机和不变量。
- 代码文件。
- 测试文件。
- 架构规则。
- 最近变更。
- 失败日志。
- 团队规范。

### 2. 打分模型

可以用简单加权分：

```text
score = relevance * 0.40
      + authority * 0.25
      + freshness * 0.15
      + specificity * 0.15
      + usage_signal * 0.05
```

字段含义：

- relevance：与任务关键词、模块、API、错误栈的相关度。
- authority：规格、契约、ADR 高于聊天记录。
- freshness：active 新版本高于旧版本。
- specificity：具体到当前模块的规则高于通用规范。
- usage_signal：历史上解决类似任务时常用的上下文。

### 3. 必选和可选

不是所有上下文都靠打分。高风险任务需要必选上下文：

```yaml
must_include:
  - current_feature_spec
  - domain_state_machine
  - api_contract
  - related_tests
  - permission_policy
optional_by_score:
  - recent_prs
  - old_issues
  - style_guides
```

### 4. 上下文压缩

上下文过大时不能简单截断。优先压缩：

- 长日志压缩成错误类型、堆栈顶部、失败断言。
- 大文件压缩成相关函数和调用关系。
- 多份文档压缩成冲突点和决策结论。
- 历史 PR 压缩成变更原因和风险。

## 工具调度算法

### 1. 工具生命周期

```text
discover -> validate arguments -> permission check -> execute -> summarize -> record -> trigger verification
```

### 2. 前置条件

| 工具 | 前置条件 |
| --- | --- |
| edit_file | 已读取规格和目标文件 |
| run_tests | 工作区有可验证变更 |
| install_dependency | 有审批和理由 |
| modify_migration | 有数据风险说明 |
| browser_test | 前端服务已启动 |
| external_fetch | 网络访问已允许 |

### 3. 调度策略

```pseudo
if task.state == PLANNING:
    allow(search, read)
    deny(edit)

if task.state == EXECUTING:
    allow(edit, search, read)
    require_approval(high_risk_tools)

if modified_files.changed:
    schedule(required_verifiers)

if verifier.failed:
    allow(read_failure, edit_related_files)
    deny(unrelated_expansion)
```

### 4. 修改范围控制

Agent 容易“顺手重构”。Harness 应限制修改扩散：

```text
允许修改范围 = 规格影响范围 + 测试覆盖范围 + 审批扩展范围
```

如果 Agent 需要改额外模块，必须说明：

- 为什么当前任务需要。
- 哪条规格支持。
- 会影响哪些测试。
- 是否需要人工审批。

## 深度例子

任务：“修复订单取消幂等性 bug。”

上下文必选：

- 订单取消规格。
- 订单状态机。
- 退款请求模型。
- 幂等键策略。
- 失败测试日志。
- 相关测试。

不应优先包含：

- 全站 UI 风格指南。
- 三年前订单重构讨论。
- 无关促销模块代码。

工具调度：

```text
先 search/read -> 再 edit order/refund idempotency相关文件 -> 运行幂等测试 -> 运行订单模块测试 -> 输出 traceability 摘要
```

## 练习

为“修复用户登录偶发超时”设计上下文选择算法。至少包含：

- 必选上下文。
- 候选上下文打分。
- 日志压缩策略。
- 工具调度顺序。
- 禁止扩散范围。

## 验收

- 能定义上下文打分字段。
- 能区分必选上下文和可选上下文。
- 能给每个工具定义前置条件和审批策略。

## 重点

上下文质量决定 Agent 判断质量；工具调度决定 Agent 行动风险。

## 难点

难点是选择，而不是堆叠。给 Agent 全部材料通常不等于更好，反而可能增加误判。

## 易错

> **易错：** 把最近文件当作最相关文件。
>
> 正确做法：最近只是一个信号，权威性、作用域和规格关联更重要。
