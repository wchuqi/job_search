# SDD与Harness Engineering学习资料：规格文档层次和工作流

[返回索引](../SDD与Harness Engineering学习资料.md)

## 学习目标

- 掌握规格文档的分层方式。
- 能设计从规格到任务再到测试的工作流。
- 理解规格冲突、版本和变更治理。

## 理论导读

规格不是单一文档。一个项目中的规格可以分成产品规格、领域规格、接口规格、架构规格、测试规格和运行规格。层次清楚，Agent 才能获取正确上下文；层次混乱，Agent 会把产品目标、代码约束、测试要求混成一团。

## 核心心智模型

把规格看作“工程地图”：

- 产品规格说明目的地。
- 领域规格说明地形。
- 架构规格说明道路。
- 接口规格说明路口规则。
- 测试规格说明检查站。
- Harness 规则说明施工队能用什么工具、走哪条路、哪里不能碰。

## 知识点详解

### 1. 规格分层

| 层次 | 内容 | 典型产物 |
| --- | --- | --- |
| 产品规格 | 用户目标、业务价值、范围 | feature spec |
| 领域规格 | 领域实体、状态机、规则 | domain model spec |
| 接口规格 | API、事件、消息、错误码 | OpenAPI、AsyncAPI |
| 架构规格 | 模块边界、依赖方向、技术约束 | architecture decision record |
| 测试规格 | 验收标准、测试数据、属性约束 | test plan、property tests |
| Harness规格 | Agent 规则、工具、权限、验证命令 | agent rules、CI checks |

### 2. 从规格到任务

好的任务拆解应满足：

- 每个任务都有输入规格。
- 每个任务有明确输出。
- 每个任务能独立验证。
- 任务之间有依赖顺序。
- 高风险任务先加测试或保护网。

任务示例：

```text
任务1：补充订单取消状态机测试
依赖：订单状态规格
输出：OrderCancelStateTest
验证：订单模块测试通过

任务2：实现取消服务
依赖：任务1
输出：OrderCancelService.cancel()
验证：状态机测试通过
```

### 3. 冲突处理

规格冲突常见于：

- 产品希望简单，安全要求更严格。
- 接口兼容性要求与新功能冲突。
- 性能目标与一致性目标冲突。
- Agent 生成的任务与架构边界冲突。

处理顺序：

```text
法规和安全 > 数据正确性 > 兼容性 > 业务目标 > 性能优化 > 开发便利
```

这不是绝对规则，但适合作为默认判断框架。

## 例子

API 规格片段：

```yaml
POST /orders/{orderId}/cancel:
  auth: required
  idempotency: required
  success:
    status: 200
    body:
      orderStatus: CANCELLED
  errors:
    - code: ORDER_ALREADY_SHIPPED
      status: 409
    - code: ORDER_NOT_FOUND
      status: 404
```

Harness 读取规则：

```text
上下文优先级：
1. 当前 feature spec
2. 相关 domain spec
3. 相关 API spec
4. 项目代码风格
5. 历史 issue 和 PR 摘要
```

## 练习

为“优惠券核销”设计 6 层规格，每层至少写 3 条内容。

## 验收

- 能说明每层规格的责任边界。
- 能把规格转成任务。
- 能发现并处理规格冲突。

## 重点

规格的分层是为了减少上下文噪音。Agent 不需要每次读完整个项目百科，但必须读到当前任务的关键约束。

## 难点

难点在于上下文选择：给少了，Agent 会误判；给多了，Agent 会淹没在无关信息里。

## 易错

> **易错：** 把所有规则堆到一个大文档。
>
> 正确做法：按责任分层，并为 Agent 定义检索顺序和引用方式。
