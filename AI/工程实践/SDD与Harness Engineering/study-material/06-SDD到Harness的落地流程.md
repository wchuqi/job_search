# SDD与Harness Engineering学习资料：SDD到Harness的落地流程

[返回索引](../SDD与Harness Engineering学习资料.md)

## 学习目标

- 能把 SDD 规格转成 Agent Harness 规则。
- 掌握小团队落地步骤。
- 能识别从试验到生产的成熟度阶段。

## 理论导读

SDD 和 Harness Engineering 的关系可以理解为：SDD 生产“可执行意图”，Harness 消费这些意图并驱动 Agent 完成任务。落地时不要一开始就追求复杂平台，先从一个小功能、一个规则文件、一组测试命令和一次复盘开始。

## 核心心智模型

```text
规格 -> 计划 -> 任务 -> 执行约束 -> 验证 -> 复盘 -> 规格更新
```

## 知识点详解

### 1. 最小落地步骤

1. 选择低风险功能。
2. 写一份 feature spec。
3. 标注相关目录和禁止修改目录。
4. 定义测试命令。
5. 让 Agent 先生成计划，不直接改代码。
6. 人审查计划。
7. Agent 小步实现。
8. Harness 强制验证。
9. 人做代码评审。
10. 复盘失败并更新规则。

### 2. 成熟度模型

| 阶段 | 特征 | 风险 |
| --- | --- | --- |
| L0 随手提示 | 直接让 Agent 改代码 | 漂移、误改、不可复盘 |
| L1 规格先行 | 先写 feature spec | 规格质量不稳定 |
| L2 固定规则 | 有 Agent 规则和测试入口 | 上下文仍靠人工挑选 |
| L3 自动验证 | CI、lint、测试、架构检查自动化 | 检查器维护成本上升 |
| L4 可观测复盘 | 有轨迹和失败分类 | 需要团队纪律 |
| L5 组织级 Harness | 权限、审计、知识库、评估体系完整 | 平台复杂度高 |

### 3. 团队角色变化

- 产品人员：更明确地定义场景和验收。
- 开发人员：更多设计规格、边界、验证和 Harness。
- 测试人员：从手工验收转向规格覆盖和自动化验证。
- 架构师：维护模块边界、依赖规则和技术决策。
- 安全人员：把安全要求前置到规格和权限系统。

## 例子

落地清单：

```text
Feature: 用户头像上传

规格：
- 文件大小 <= 2MB
- 仅允许 jpg/png/webp
- 上传成功后旧头像异步删除
- 上传失败不更新用户资料

Harness：
- 允许修改 user-profile 模块
- 禁止修改 auth 模块
- 必须运行 UserAvatarServiceTest
- 必须运行文件类型校验测试
- 涉及对象存储配置时转人工审批
```

## 练习

把你当前项目中的一个小功能，按成熟度 L1 到 L3 落地：

- 写规格。
- 写 Agent 规则。
- 写验证命令。
- 写复盘模板。

## 验收

- Agent 的任务不是一句 prompt，而是一组规格和规则。
- 测试不是 Agent 自愿运行，而是流程强制要求。
- 失败能产生可维护改进。

## 重点

从 SDD 到 Harness 的关键，是把规格中的“必须”变成 Harness 中的“强制检查”。

## 难点

难点是组织习惯。团队必须愿意把口头规则写下来，并在失败后更新规则。

## 易错

> **易错：** 一开始就搭大平台。
>
> 正确做法：先用 Markdown 规格、脚本、CI 和代码评审跑通闭环，再逐步平台化。
