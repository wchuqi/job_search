# SDD与Harness Engineering学习资料：团队治理安全和风险

[返回索引](../SDD与Harness Engineering学习资料.md)

## 学习目标

- 理解 Agent 参与开发后的治理和安全问题。
- 能设计权限、审计、审批和风险控制。
- 能识别常见反模式。

## 理论导读

AI Agent 不只是新的 IDE 插件，而是能读取上下文、调用工具、修改代码、运行命令甚至影响交付链路的执行主体。因此，团队必须把它纳入工程治理：谁发起任务，Agent 能看到什么，能改什么，谁审批，如何审计，如何回滚。

## 核心心智模型

把 Agent 当作“权限受控的工程参与者”，而不是聊天窗口。

## 知识点详解

### 1. 安全风险

| 风险 | 例子 | 控制方式 |
| --- | --- | --- |
| 密钥泄露 | Agent 读取 `.env` 并写入日志 | secrets denylist、脱敏 |
| 供应链风险 | 随意安装依赖 | 依赖审批、锁文件审查 |
| 数据破坏 | 执行危险删除或迁移 | 沙箱、审批、只读数据库 |
| 权限扩大 | 修改认证授权逻辑 | 高风险模块转人工 |
| 合规风险 | 把用户数据放入外部模型上下文 | 数据分类、脱敏、本地化策略 |
| 架构漂移 | 为完成任务绕过模块边界 | 架构规则和依赖检查 |

### 2. 审批策略

需要审批的情况：

- 修改认证、授权、支付、风控、数据迁移。
- 安装或升级依赖。
- 访问外部网络。
- 读取敏感配置。
- 修改 CI/CD、部署脚本、基础设施配置。
- 删除大量文件。

### 3. 成本和效率治理

Agent 使用也有成本：

- token 成本。
- 运行测试成本。
- 失败重试成本。
- 人工评审成本。
- 上下文维护成本。

治理方法：

- 限制任务粒度。
- 建立高价值上下文库。
- 用脚本摘要日志。
- 对重复失败设转人工阈值。
- 用评估集衡量 Harness 改进。

## 例子

风险规则片段：

```text
高风险目录：
- infra/
- migrations/
- auth/
- payment/
- security/

策略：
- Agent 可以读取，但修改必须审批。
- 涉及依赖新增必须说明原因、许可证和替代方案。
- 所有高风险修改必须生成回滚说明。
```

## 练习

为一个电商系统写 Agent 风险控制表。至少覆盖：

- 支付。
- 库存。
- 用户隐私。
- 优惠券。
- 订单状态。
- 数据库迁移。

## 验收

- 能列出高风险操作。
- 能为不同风险设置不同审批策略。
- 能解释安全要求如何进入 SDD 和 Harness。

## 重点

Agent 的权限边界必须工程化，而不是靠口头提醒。

## 难点

难点在于不过度限制。安全规则过宽会出事故，过窄会让 Agent 没有实际产能。

## 易错

> **易错：** 认为“只在本地运行 Agent”就没有安全风险。
>
> 正确做法：本地仍可能泄露密钥、误删文件、污染依赖、提交敏感数据或引入供应链问题。
