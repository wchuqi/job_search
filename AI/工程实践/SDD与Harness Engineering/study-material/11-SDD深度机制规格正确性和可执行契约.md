# SDD与Harness Engineering学习资料：SDD深度机制规格正确性和可执行契约

[返回索引](../SDD与Harness Engineering学习资料.md)

## 学习目标

- 理解“规格正确性”不是文档好看，而是规格能否被实现、验证和维护。
- 掌握可执行契约、traceability、验收标准形式化、状态机和不变量。
- 能判断一份规格是否足够让 Agent 执行。

## 理论导读

SDD 的深度在于把自然语言需求逐步收敛成工程契约。自然语言适合表达意图，但它模糊、可歧义、难验证。可执行契约则要求每条关键规则都能映射到代码、测试、检查器或人工审批。

一份成熟规格至少要回答：

- 这个功能属于哪个业务目标。
- 哪些状态、输入、输出和副作用被允许。
- 哪些边界和异常必须处理。
- 哪些行为必须保持不变。
- 哪些验收标准能被自动检查。
- 哪些风险需要人工评审。

## 核心心智模型

```text
自然语言需求 -> 结构化规格 -> 可测试验收 -> 可执行检查 -> 可追踪变更
```

规格越接近右侧，越适合 Agent 执行；越停留在左侧，越依赖人工解释。

## 知识点详解

### 1. 规格正确性的五个维度

| 维度 | 判断问题 | 失败表现 |
| --- | --- | --- |
| 完备性 | 是否覆盖主要路径、异常、权限和副作用 | Agent 只实现 happy path |
| 一致性 | 规格之间是否互相矛盾 | API 允许取消，状态机禁止取消 |
| 可验证性 | 是否能通过测试或检查证明 | “体验更好”无法判断 |
| 可追踪性 | 需求、设计、任务、测试是否能对应 | 改了代码但不知道满足哪条规格 |
| 可演进性 | 变更后是否能定位影响范围 | 新规则破坏旧场景 |

### 2. 可执行契约

可执行契约不是要求所有规格都写成代码，而是要求每条关键规则都有执行载体。

| 规格类型 | 执行载体 |
| --- | --- |
| API 输入输出 | OpenAPI、契约测试 |
| 状态迁移 | 状态机测试、迁移表 |
| 幂等要求 | 重复请求测试、幂等键检查 |
| 安全权限 | 权限测试、策略检查 |
| 架构边界 | 依赖扫描、ArchUnit、模块规则 |
| 性能目标 | benchmark、压测阈值 |
| 合规要求 | 审批、审计、数据脱敏检查 |

### 3. Traceability 矩阵

Traceability 是从需求到实现和验证的可追踪关系。它解决的问题是：某段代码为什么存在，某个测试覆盖了哪条需求，某条规格变更会影响哪些实现。

示例：

| 规格ID | 规则 | 实现位置 | 测试 | 风险 |
| --- | --- | --- | --- | --- |
| OC-001 | 未发货订单可取消 | OrderCancelService | OrderCancelServiceTest.canCancelPaidOrder | 库存和退款副作用 |
| OC-002 | 已发货订单不可取消 | OrderStateMachine | OrderCancelServiceTest.rejectShippedOrder | 状态码兼容 |
| OC-003 | 重复取消必须幂等 | RefundRequestRepository | OrderCancelIdempotencyTest | 重复退款 |

Agent 执行任务时，应尽量维护这张矩阵：新增规则要新增实现和测试，删除规则要检查相关测试是否仍有意义。

### 4. 状态机和不变量

很多业务规格本质是状态机。只用文字描述时，Agent 容易漏掉非法迁移。

订单状态示例：

```text
CREATED -> PAID -> SHIPPED -> COMPLETED
PAID -> CANCELLED
CREATED -> CANCELLED
```

不变量示例：

```text
如果 order.status == CANCELLED:
- shippedAt 必须为空
- cancelReason 必须存在
- refundRequestId 必须存在，除非订单未支付
- inventoryReleaseEvent 必须至少发布一次
```

> **重点：** 不变量比单个场景更深，因为它描述系统必须始终满足的性质。

### 5. 规格中的歧义检测

常见歧义词：

- 尽快。
- 更好。
- 合理。
- 默认。
- 必要时。
- 保持兼容。
- 正常情况。

修正方式：

| 模糊表达 | 可执行表达 |
| --- | --- |
| 查询要更快 | P95 响应时间小于 200ms |
| 保持兼容 | v1 API 字段不删除，不改变类型，不改变错误码含义 |
| 必要时重试 | 对 5xx 最多重试 2 次，间隔 100ms、300ms |
| 默认排序 | 按 created_at 降序，id 降序作为 tie-breaker |

## 深度例子

需求：“退款要支持失败重试。”

浅规格：

```text
退款失败时重试。
```

深规格：

```text
规格ID：RF-003
场景：退款请求调用第三方支付网关失败。
触发条件：支付网关返回 5xx 或网络超时。
重试策略：最多 3 次，指数退避 1s、3s、9s。
幂等要求：每次重试必须使用同一个 refundRequestId。
禁止行为：不得重复创建退款单。
最终状态：
- 成功则 RefundRequest.status = SUCCESS。
- 3 次失败后 RefundRequest.status = FAILED_RETRYABLE。
验收：
- 模拟两次 5xx 后一次成功，只产生一条退款单。
- 模拟三次 5xx 后进入 FAILED_RETRYABLE。
- 重放同一个 refundRequestId 不创建重复退款。
```

## 练习

把“文件上传失败后要回滚”改写成深规格。至少包含：

- 失败类型。
- 状态机。
- 不变量。
- 幂等规则。
- Traceability 矩阵。
- 验收测试。

## 验收

- 你能把模糊需求改写成可执行契约。
- 你能为规格建立需求、实现、测试之间的追踪关系。
- 你能识别规格中的歧义和不可验证表达。

## 重点

规格正确性必须能落到执行载体。只有文字没有检查器，Agent 仍然可能把它当作建议。

## 难点

难点是把业务语言转成状态、不变量、约束和验收标准，同时不让规格变成过重的官僚流程。

## 易错

> **易错：** 认为规格写得越详细越正确。
>
> 正确做法：深规格不是无限细节，而是关键规则可执行、可验证、可追踪。
