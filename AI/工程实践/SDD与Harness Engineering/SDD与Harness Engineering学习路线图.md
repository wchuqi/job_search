# SDD与Harness Engineering学习路线图

## 阶段 1：基础认知

- 目标：区分 SDD、TDD、BDD、传统需求文档、提示词工程和 Harness Engineering。
- 需要掌握：规格、契约、上下文、Agent、工具调用、验证闭环、权限边界。
- 例子：把“做一个登录功能”改写成用户场景、约束、验收标准、测试任务。
- 练习：选择一个已有小功能，写出 `需求 -> 设计 -> 任务 -> 验证` 四层规格。
- 验收：能说明“规格不是给人看的说明书，而是给人和 Agent 共同执行的工程契约”。
- 重点：SDD 解决“做什么、为什么、怎么验收”；Harness 解决“让 Agent 在什么环境中做、怎么防偏、怎么确认完成”。
- 易错：把 SDD 当作更长的 prompt。正确理解是：SDD 是规格治理流程，prompt 只是执行入口之一。

## 阶段 2：规格驱动开发能力

- 目标：能把模糊业务需求转成可执行规格。
- 需要掌握：用户场景、功能需求、非功能需求、边界条件、验收标准、任务拆解、变更记录。
- 例子：把“提高搜索体验”拆成查询解析、排序规则、性能目标、空结果策略和埋点指标。
- 练习：为一个 REST API 写规格，并根据规格生成任务列表。
- 验收：规格中没有“应该差不多”“体验更好”这类不可验收表达。
- 重点：规格要能驱动实现、测试、评审和回滚。
- 易错：只写 happy path。正确做法是同时写异常路径、权限边界、性能约束和兼容性约束。

## 阶段 3：Harness Engineering能力

- 目标：能为 AI 编码 Agent 设计一个可控运行环境。
- 需要掌握：上下文装配、工具白名单、权限模型、沙箱、测试命令、代码风格约束、状态记录、任务恢复。
- 例子：让 Agent 修改 Java 项目时，只允许读取仓库、编辑指定目录、运行单元测试和生成 PR 摘要。
- 练习：为一个仓库写 Agent 运行规约，包括允许命令、禁止命令、测试入口、失败重试策略。
- 验收：Agent 失败时能留下可复盘轨迹，而不是只留下一堆不可解释的修改。
- 重点：Harness 的价值是把不可预测的模型行为压进可审计、可重复、可恢复的工程过程。
- 易错：只靠模型“自觉遵守”。正确做法是用脚本、测试、权限和检查器把关键规则外部化。

## 阶段 4：验证、观测和生产治理

- 目标：能评估 Agent 输出是否可靠，并持续改进 Harness。
- 需要掌握：单元测试、集成测试、属性测试、静态检查、架构规则、日志、轨迹、失败分类、回归样例库。
- 例子：Agent 经常误改公共 API，就增加 API 兼容性测试和架构边界检查。
- 练习：设计一个“Agent 失败复盘模板”，把失败原因归类为规格缺失、上下文缺失、工具缺失、验证缺失或权限缺失。
- 验收：每次失败都能转化为规格、工具、检查器或流程的改进。
- 重点：Harness Engineering 的长期收益来自反馈闭环，不来自单次 prompt 技巧。
- 易错：把人工修 bug 当作终点。正确做法是把 bug 变成 Harness 的新能力。

## 阶段 5：深度机制和架构设计

- 目标：能设计 SDD + Harness 的组织级方案，而不是只会使用工具。
- 需要掌握：规格正确性、规格冲突消解、traceability、Agent 控制循环、上下文选择算法、工具调度、评估集、质量指标、平台化架构。
- 例子：为一个多服务电商系统设计 Agent Harness，支持订单、库存、支付三个领域的规格检索、权限控制和验证矩阵。
- 练习：写一份“Agent 修改支付退款逻辑”的深度设计，必须包含风险分级、规格版本、工具权限、验证矩阵、失败复盘和上线门禁。
- 验收：能回答“为什么这套 Harness 能降低错误率、如何度量、失败后如何定位、如何持续改进”。
- 重点：深度不在术语，而在可执行决策规则。
- 易错：只谈流程，不谈状态、算法、指标和故障模式。

## 推荐学习节奏

| 周期 | 学习重点 | 产出 |
| --- | --- | --- |
| 第 1 天 | SDD 和 Harness 基础概念 | 一页概念对照表 |
| 第 2-3 天 | 规格拆解和验收标准 | 一个功能规格文档 |
| 第 4-5 天 | Agent 运行环境和权限 | 一个 Agent 工作规约 |
| 第 6-7 天 | 测试、观测和复盘 | 一个最小 Harness 方案 |
| 第 2 周 | 综合项目 | 可展示的 SDD + Harness 案例 |
| 第 3 周 | 深度机制 | 规格冲突规则、Agent 控制循环和验证指标 |
| 第 4 周 | 平台化设计 | 组织级 Harness 架构方案 |

## 最终能力清单

- 能把模糊需求转成可执行规格。
- 能判断规格是否足以驱动实现和测试。
- 能设计 Agent 的上下文、工具、权限和验证边界。
- 能解释 Harness Engineering 与 prompt engineering、context engineering、DevOps、平台工程的关系。
- 能用失败复盘持续改进 Agent 工作流。
- 能设计规格正确性检查、上下文选择、工具权限、质量评估和平台化治理方案。
