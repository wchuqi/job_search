# AI Context Engineering 学习资料：总览和心智模型

[返回索引](../AI Context Engineering学习资料.md)

## 学习目标

- 理解 Context Engineering 与 Prompt Engineering、RAG、Agent Engineering 的关系。
- 建立“上下文是运行时信息架构”的心智模型。
- 能识别一次模型调用中各类上下文的职责、风险和生命周期。

## 理论导读

大模型不会直接访问你的数据库、文件系统、业务状态或历史记忆。它只能基于本次调用里出现的上下文进行预测和生成。因此，一个 AI 应用的稳定性很大程度上取决于：你是否把正确的信息放到了正确的位置，是否清除了过期和危险信息，是否让模型看到了足够的证据，是否能在失败时解释“模型到底看见了什么”。

Prompt Engineering 主要解决“如何表达任务”。Context Engineering 进一步解决“任务执行时，模型应获得哪些信息”。当 AI 应用从单轮问答进入 RAG、工具调用、长期记忆和 Agent 工作流后，失败往往不再是 prompt 文案问题，而是上下文系统问题。

## 核心心智模型

把一次模型调用想象成一次临时会议：

- 系统规则是会议章程。
- 用户当前请求是本次议题。
- 历史摘要是会议纪要。
- 检索资料是证据材料。
- 工具结果是外部系统回执。
- 长期记忆是用户档案。
- 输出 schema 是会议纪要格式。
- 安全策略是合规红线。

Context Engineering 的工作不是把所有材料堆到桌上，而是整理会议包：哪些材料必须带，哪些材料只能作为参考，哪些材料过期，哪些材料有安全风险，哪些材料需要附出处，哪些材料应该压缩成结构化状态。

## 知识点详解

### 1. 上下文不是“更多文本”

上下文越多，成本越高，延迟越高，噪声越多，冲突概率越高。长上下文窗口可以缓解容量问题，但不能自动解决注意力分散、证据冲突、旧指令残留和不可信内容注入。

> **重点：** 上下文工程首先是选择工程，其次才是拼接工程。

### 2. 上下文有权限等级

系统规则、开发者规则、用户输入、外部资料、工具结果不应拥有同等权限。外部网页中的“忽略之前指令”只是网页内容，不是系统指令。用户上传文档里的操作要求也不能越过系统和权限网关。

> **易错：** 只用自然语言提醒模型“不许被注入攻击”，但没有在上下文格式、工具权限和调用网关上隔离不可信内容。

### 3. 上下文有生命周期

有些信息只在当前调用有效，例如当前用户问题。有些信息在当前任务有效，例如已确认的需求。有些信息可以长期保留，例如用户偏好。还有些信息必须过期，例如一次性验证码、临时搜索结果、旧版本 API 文档。

### 4. 上下文需要可观测

生产系统必须能记录：本次调用使用了哪些上下文槽位、每个槽位来源是什么、占用多少 token、是否被压缩、是否触发安全过滤、最终传给模型的版本是什么。否则失败后只能猜。

## 例子

一个简化的上下文包：

```json
{
  "instruction": {
    "role": "career_assistant",
    "policy": "do not fabricate company facts"
  },
  "task": {
    "user_request": "帮我改这份 Java 后端简历，突出分布式系统经验"
  },
  "state": {
    "target_role": "Java Backend Engineer",
    "confirmed_constraints": ["中文简历", "一页内"]
  },
  "evidence": [
    {
      "source": "uploaded_resume",
      "trust": "user_provided",
      "content": "..."
    }
  ],
  "output_contract": {
    "format": "markdown",
    "sections": ["修改建议", "改写版本", "风险提示"]
  }
}
```

## 练习

选一个你常用的 AI 场景，列出：

- 当前请求。
- 必须保留的事实。
- 可摘要的历史。
- 需要检索的外部资料。
- 不可信但需要引用的资料。
- 不允许进入上下文的敏感信息。

## 重点

- Context Engineering 是 AI 应用的运行时信息架构。
- 上下文包括指令、历史、状态、证据、工具结果、记忆、权限和输出契约。
- 上下文质量直接影响准确性、稳定性、安全性、成本和可解释性。

## 难点

难点在于冲突和取舍：两个来源说法不一致时，系统必须有权威度、时效性和适用范围的决策规则，而不是把冲突文本都塞给模型。

## 易错

> **易错：** 认为使用长上下文模型就不需要 RAG、压缩和记忆建模。
>
> 正确做法：长上下文解决容量，Context Engineering 解决选择、可信度、排序、冲突和治理。
