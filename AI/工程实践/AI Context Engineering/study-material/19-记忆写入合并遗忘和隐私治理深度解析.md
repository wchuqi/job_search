# AI Context Engineering 学习资料：记忆写入、合并、遗忘和隐私治理深度解析

[返回索引](../AI Context Engineering学习资料.md)

## 学习目标

- 深入理解长期记忆不是聊天历史。
- 掌握记忆写入、合并、冲突、遗忘和隐私治理。
- 能设计可控的用户记忆系统。

## 理论导读

记忆系统的核心风险是“错误长期化”。一次误解、一次恶意输入、一次过期偏好，如果被写入长期记忆，就会持续污染后续上下文。成熟记忆系统必须控制写入条件、更新规则、删除机制、隐私边界和注入时机。

## 记忆数据模型

```json
{
  "memory_id": "m_001",
  "type": "preference",
  "content": "用户偏好中文技术资料，结构要细。",
  "source": "explicit_user_statement",
  "confidence": 0.95,
  "scope": "learning_material",
  "sensitivity": "low",
  "created_at": "2026-06-07",
  "updated_at": "2026-06-07",
  "expires_at": null,
  "confirmed": true
}
```

## 写入策略

| 信号 | 是否写入 | 说明 |
| --- | --- | --- |
| 用户明确说“记住” | 可以 | 仍需隐私判断 |
| 用户一次性任务约束 | 通常不写 | 保留在任务状态 |
| 模型推断出的性格 | 不建议 | 容易偏见和误解 |
| 多次稳定偏好 | 可候选 | 最好让用户确认 |
| 敏感身份和隐私 | 默认不写 | 除非必要且授权 |

## 合并策略

当新记忆和旧记忆相似：

```text
同义补充 -> 合并
更具体 -> 更新旧记忆
明显冲突 -> 标记冲突并请求确认
不同范围 -> 并存但加 scope
过期信息 -> 归档或删除
```

## 检索和注入

长期记忆不应每次全部进入上下文。注入前要判断：

- 是否与当前任务相关。
- 是否在适用范围内。
- 是否未过期。
- 是否没有被用户撤销。
- 是否敏感且当前任务确实需要。

## 隐私治理

记忆系统应支持：

- 用户查看。
- 用户修改。
- 用户删除。
- 敏感信息分类。
- 记忆导出。
- 审计写入来源。
- 默认最小化保存。

## 例子：冲突处理

旧记忆：

```text
用户偏好 Java 后端岗位。
```

新请求：

```text
我现在转向 AI 工程岗位，后续资料围绕 AI Agent。
```

处理：当前任务使用 AI 工程目标；长期记忆更新为“当前目标从 Java 后端转向 AI 工程”，旧偏好归档或标注历史。

## 练习

设计 8 条求职助手记忆，标注 type、scope、sensitivity、confidence、confirmed、expires_at。再给 3 条新输入，说明是新增、合并、更新还是拒绝写入。

## 验收

- 能区分任务状态、历史日志和长期记忆。
- 能设计记忆冲突处理规则。
- 能说明隐私和用户控制机制。

## 重点

记忆系统不是为了“记得多”，而是为了在未来任务中稳定、可控、合规地注入有用信息。

## 难点

难点是写入判断。模型觉得“有用”的信息未必应该长期保存，尤其涉及隐私、偏见和短期上下文时。

## 易错

> **易错：** 把每轮对话摘要都写入长期记忆。
>
> 正确做法：长期记忆只保存未来稳定有用且经过来源、范围、隐私判断的信息。
