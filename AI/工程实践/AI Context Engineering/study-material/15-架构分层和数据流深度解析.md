# AI Context Engineering 学习资料：架构分层和数据流深度解析

[返回索引](../AI Context Engineering学习资料.md)

## 学习目标

- 能把 Context Engineering 拆成可实现的系统层次。
- 理解上下文从业务状态到模型输入的完整数据流。
- 能区分上下文设计中的控制平面和数据平面。

## 理论导读

Context Engineering 不是一个 prompt 模板文件，而是一套运行时系统。它连接业务数据、会话状态、检索系统、工具协议、安全策略、模型调用和评估日志。深度理解它，需要把“最终 prompt”往前拆：每一段文本为什么出现、谁决定它出现、是否经过授权、是否经过压缩、失败后如何复现。

一个成熟系统至少有三层：

- 数据平面：真实进入模型的内容，如用户请求、证据、工具结果、状态摘要。
- 控制平面：决定哪些内容进入模型的规则，如预算、排序、权限、安全策略。
- 观测平面：记录和评估上下文构造过程，如 trace、token、命中证据、过滤原因。

## 分层架构

```text
业务入口层
  -> 任务理解层
  -> 状态和记忆层
  -> 检索和工具层
  -> 上下文选择层
  -> 安全和权限层
  -> 模板渲染层
  -> 模型调用层
  -> 评估和观测层
```

### 1. 业务入口层

负责接收用户请求、文件、会话 ID、权限身份和业务场景。这里不能急着拼 prompt，而要先把请求转成结构化任务。

```json
{
  "intent": "resume_revision",
  "user_goal": "突出分布式系统经验",
  "required_inputs": ["resume", "target_jd"],
  "risk_level": "medium"
}
```

### 2. 任务理解层

判断任务类型、缺失信息、是否需要检索、是否需要工具、是否需要人工确认。它决定后续启用哪些上下文槽位。

### 3. 状态和记忆层

读取短期任务状态和长期记忆，但不会全部注入。它会先做相关性过滤、权限过滤和时效过滤。

### 4. 检索和工具层

检索系统返回候选证据，工具系统返回外部观察。两者都不应直接成为最终上下文，必须进入清洗、压缩和安全包装环节。

### 5. 上下文选择层

这是 Context Engineering 的核心：基于相关性、权威度、时效、风险和 token 成本选择内容。

### 6. 安全和权限层

负责敏感信息过滤、不可信资料包装、工具权限检查、数据越权检查和高风险动作审批。

### 7. 模板渲染层

把已经选好的结构化槽位渲染成模型可读的输入。模板层不应承担复杂决策。

### 8. 评估和观测层

记录本次上下文的每个槽位来源、长度、版本、过滤原因和最终模型输出，用于回放和归因。

## 控制平面和数据平面

| 类型 | 内容 | 典型问题 |
| --- | --- | --- |
| 数据平面 | 证据片段、历史摘要、工具结果 | 内容是否正确、完整、可信 |
| 控制平面 | 预算规则、权限规则、排序规则 | 是否选择了正确内容 |
| 观测平面 | trace、指标、评测样本 | 失败后能否复现 |

## 例子：一次 RAG Agent 调用的数据流

```text
用户问题
-> 分类为“文档问答”
-> 读取用户权限
-> query rewrite
-> 检索候选文档
-> 权限过滤和版本过滤
-> rerank
-> 去重和上下文压缩
-> 不可信资料包装
-> 加入输出引用规则
-> 调用模型
-> 记录证据命中和引用准确率
```

## 深入机制：为什么要分层

如果检索、权限、压缩、模板渲染都写在一个 prompt 构造函数里，系统很难回答这些问题：

- 为什么这段资料进入了上下文？
- 为什么另一段资料被丢弃？
- 资料是否越权？
- 压缩前后丢了什么？
- 本次失败能否回放？
- 新版本策略是否比旧版本好？

分层的意义是让每个决策点可测试。Context Engineering 的深度不在“模板多复杂”，而在“信息进入模型前经过了哪些可解释决策”。

## 练习

为一个“企业知识库问答 + 工单操作 Agent”画出架构图，至少包含：

- 用户身份和权限。
- 文档检索。
- 工具调用。
- 上下文选择。
- 安全隔离。
- 模型调用。
- 观测日志。

## 验收

- 能区分数据平面、控制平面和观测平面。
- 能解释上下文进入模型前的每个处理环节。
- 能为每层写出输入、输出和失败模式。

## 重点

Context Engineering 是系统工程。最终 prompt 只是结果，不是设计本身。

## 难点

难点是责任边界：哪些由模型判断，哪些由程序规则判断，哪些必须由人工确认。越高风险的决策，越不能只交给模型。

## 易错

> **易错：** 把所有上下文策略都写成 prompt 里的自然语言要求。
>
> 正确做法：选择、权限、过滤、审计和高风险动作审批应由外部系统实现，prompt 只表达模型需要遵守的任务契约。
