# AI Context Engineering 学习资料：RAG 证据工程、查询规划和上下文打包深度解析

[返回索引](../AI Context Engineering学习资料.md)

## 学习目标

- 深入理解 RAG 不只是检索，而是证据工程。
- 掌握查询规划、多跳检索、证据覆盖和上下文打包。
- 能设计可引用、可回放、可评估的 RAG 上下文。

## 理论导读

RAG 的失败经常不是模型不会回答，而是模型没有看到正确证据，或看到的证据不完整、不可信、互相冲突。RAG 深度工程要回答四个问题：

- 应该查什么？
- 从哪里查？
- 哪些证据足以回答？
- 如何把证据打包给模型？

## 查询规划

简单问题可以直接检索，复杂问题需要拆分：

```text
用户问题：对比 A 版本和 B 版本 API 的认证方式变化。

子查询：
1. A 版本认证方式
2. B 版本认证方式
3. 认证参数变化
4. 兼容性和迁移说明
```

查询规划适合：

- 对比问题。
- 时间变化问题。
- 多条件筛选。
- 需要多个文档合成的答案。

## 检索策略

| 策略 | 适用 |
| --- | --- |
| Dense retrieval | 语义相似问题 |
| Keyword search | 专有名词、错误码、函数名 |
| Hybrid search | 技术文档和企业知识库 |
| Metadata filter | 权限、版本、时间、产品线 |
| Multi-hop retrieval | 需要多个证据链 |
| Parent-child retrieval | 小片段召回，大段落注入 |

## 证据覆盖

证据不只要相关，还要覆盖答案所需维度。

```json
{
  "answer_requirements": ["定义", "适用场景", "限制", "示例"],
  "evidence_coverage": {
    "定义": ["doc_1#p3"],
    "适用场景": ["doc_2#p1"],
    "限制": [],
    "示例": ["doc_3#code"]
  }
}
```

如果关键维度缺证据，系统应拒答、追问或继续检索。

## 上下文打包

推荐证据格式：

```text
<evidence id="E1" source="docs/api-v2.md" version="2.1" trust="official">
认证请求必须包含 Authorization header。
</evidence>

<evidence id="E2" source="docs/migration-v2.md" version="2.1" trust="official">
v2 移除了 query token 认证方式。
</evidence>
```

回答时要求引用 `E1`、`E2`，而不是生成模糊来源。

## RAG 失败深度归因

| 层级 | 失败原因 | 修复 |
| --- | --- | --- |
| 数据层 | 文档缺失、旧版本、解析错误 | 数据治理 |
| 切分层 | chunk 破坏语义 | 结构化切分 |
| 检索层 | query 不匹配 | query rewrite、hybrid |
| 排序层 | 相关证据不靠前 | rerank |
| 打包层 | 证据太多或缺来源 | 压缩和引用 ID |
| 生成层 | 未按证据回答 | 忠实性约束 |
| 安全层 | 恶意资料进入 | 准入和隔离 |

## 练习

选一个技术文档问答场景，设计 10 条需要多证据合成的问题。为每题标注必须命中的证据 ID，并评估 RAG 是否覆盖全部证据。

## 验收

- 能拆复杂问题为子查询。
- 能设计 evidence coverage 表。
- 能区分证据相关和证据充分。
- 能让引用与证据 ID 绑定。

## 重点

RAG 的答案质量上限由证据质量决定。证据不足时，模型再强也容易补全和编造。

## 难点

难点是多跳和冲突：复杂问题经常需要多个来源拼接，同时还要判断哪个来源更权威。

## 易错

> **易错：** 检索 top-k 后直接拼给模型。
>
> 正确做法：做查询规划、权限过滤、重排、证据覆盖检查、压缩和引用绑定。
