# AI Context Engineering 学习资料：上下文建模、槽位和拼装流水线

[返回索引](../AI Context Engineering学习资料.md)

## 学习目标

- 能把 AI 调用拆成上下文槽位。
- 掌握上下文拼装流水线的基本步骤。
- 能用结构化方式处理排序、裁剪和冲突。

## 理论导读

上下文拼装不应该散落在业务代码里。生产系统通常需要一条明确流水线：读取任务状态，选择相关记忆，检索证据，压缩历史，格式化工具结果，检查安全风险，最后按固定契约生成模型输入。

## 上下文槽位设计

```yaml
slot:
  name: retrieved_evidence
  source: vector_store
  trust_level: external_verified
  max_tokens: 6000
  lifecycle: current_turn
  conflict_policy: prefer_newer_version
  security_policy: untrusted_instruction
```

## 拼装流水线

```text
1. 解析当前任务
2. 读取会话状态和长期记忆
3. 判断需要哪些上下文槽位
4. 检索或调用工具获取资料
5. 对资料重排、去重、压缩
6. 检查敏感信息和注入风险
7. 按指令层级和输出契约拼装
8. 记录上下文版本和 token 分布
```

## 简化代码

```python
def build_context(request, state, stores, tools):
    slots = []
    slots.append(system_rules())
    slots.append(task_request(request))
    slots.append(compact_state(state))

    if needs_evidence(request):
        docs = stores.docs.search(request.query)
        docs = rerank(request.query, docs)
        slots.append(wrap_untrusted(compress_docs(docs)))

    if needs_tools(request):
        result = tools.run_safe(request)
        slots.append(summarize_tool_result(result))

    return assemble_with_budget(slots)
```

## 冲突处理

| 冲突类型 | 处理规则 |
| --- | --- |
| 当前请求 vs 历史偏好 | 当前请求优先，必要时提示偏好冲突 |
| 新文档 vs 旧文档 | 版本号和更新时间优先 |
| 用户资料 vs 官方资料 | 取决于任务；事实判断通常官方优先 |
| 工具结果 vs 模型推测 | 工具结果优先，但要校验工具是否成功 |
| 资料内容 vs 系统规则 | 系统规则优先 |

## 练习

为“根据公司 JD 修改简历”的任务设计上下文流水线，包括用户简历、目标岗位、历史偏好、行业术语库和输出格式。

## 验收

- 能画出拼装流程。
- 能为每个槽位写出来源、预算、权限和生命周期。
- 能处理至少三类上下文冲突。

## 重点

槽位化的价值在于可替换、可评估、可监控。不要把上下文拼接写成一段不可解释的字符串拼接代码。

## 难点

难点是让上下文流水线既稳定又动态：固定的是槽位契约，动态的是本次任务实际启用哪些槽位以及每个槽位的预算。

## 易错

> **易错：** 把上下文构造逻辑藏在 prompt 模板字符串里。
>
> 正确做法：把选择、过滤、压缩、排序、安全检查和模板渲染分成可测试步骤。
