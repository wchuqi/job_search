# AI Context Engineering 学习资料：Agent 工作流和上下文生命周期

[返回索引](../AI Context Engineering学习资料.md)

## 学习目标

- 理解 Agent 中上下文随计划、行动、观察和反思循环变化。
- 掌握任务状态、checkpoint 和恢复机制。
- 能设计多步骤任务的上下文生命周期。

## 理论导读

Agent 不只是多次调用 LLM。每一步都会产生新状态：计划、工具调用、观察结果、错误、用户确认、文件变更、下一步动作。Context Engineering 要决定哪些状态进入下一步、哪些写入日志、哪些压缩、哪些丢弃。

## Agent 循环

```text
goal -> plan -> act -> observe -> update_state -> decide_next
```

每一轮都要更新上下文：

- 目标是否变化。
- 计划是否仍有效。
- 工具结果是否成功。
- 是否出现阻塞。
- 是否需要用户确认。
- 是否需要压缩历史。

## 生命周期设计

| 内容 | 生命周期 | 进入上下文方式 |
| --- | --- | --- |
| 用户最终目标 | 整个任务 | 原文或稳定摘要 |
| 当前步骤 | 当前循环 | 结构化状态 |
| 工具原始输出 | 短期 | 摘要后进入 |
| 操作日志 | 长期审计 | 不一定注入 |
| 错误信息 | 修复期间 | 精简保留关键错误 |
| 已完成动作 | 整个任务 | 列表化状态 |

## checkpoint 示例

```json
{
  "goal": "修复测试失败",
  "plan": ["运行测试", "定位失败", "修改代码", "复测"],
  "completed": ["运行测试", "定位失败"],
  "current_blocker": null,
  "last_observation": "UserServiceTest 在空邮箱场景失败",
  "next_action": "检查 UserService.validateEmail"
}
```

## 练习

设计一个代码修复 Agent 的上下文生命周期，说明：测试输出、文件 diff、用户需求、错误堆栈、计划和待办分别保留多久。

## 验收

- 能画出 Agent 每轮上下文变化。
- 能从中断点恢复任务。
- 能避免旧计划覆盖新目标。

## 重点

Agent 的可靠性取决于状态更新质量。没有状态机的 Agent 很容易在长任务中重复、跑偏或忘记约束。

## 难点

难点是任务切换：用户中途改变目标时，系统必须区分“继承旧状态”和“清空旧状态”。

## 易错

> **易错：** 把 Agent 所有中间思考和工具输出都作为下一轮上下文。
>
> 正确做法：保留可验证状态、关键观察和下一步决策依据，压缩或归档冗余过程。
