# AI Context Engineering 学习资料：长上下文、压缩和摘要

[返回索引](../AI Context Engineering学习资料.md)

## 学习目标

- 理解长上下文能力的边界。
- 掌握摘要、抽取、状态化和查询聚焦压缩。
- 能评估压缩是否丢失关键事实。

## 理论导读

长上下文模型让我们能放入更多资料，但资料越多越需要结构。压缩不是简单“变短”，而是在保留任务所需信息的前提下降低噪声和成本。压缩的最大风险是丢失约束、来源、否定条件和例外情况。

## 压缩方式

| 方式 | 适用场景 | 风险 |
| --- | --- | --- |
| 滚动摘要 | 多轮对话 | 早期细节被改写 |
| 抽取式压缩 | 需要保留原文证据 | 片段不连贯 |
| 生成式摘要 | 长文档快速概览 | 幻觉和遗漏 |
| 查询聚焦压缩 | 问答和 RAG | 对未来问题不通用 |
| 状态化压缩 | Agent 和工作流 | schema 设计不当 |

## 状态化摘要示例

```json
{
  "goal": "准备 Java 后端面试",
  "confirmed_facts": [
    "用户目标岗位是中高级 Java 后端",
    "重点准备分布式系统和数据库"
  ],
  "decisions": [
    "资料按主题拆分到 Markdown 文件"
  ],
  "open_questions": [],
  "risks": [
    "部分 AI 工程术语变化快，需要注明日期"
  ]
}
```

## 压缩评估

压缩后要检查：

- 目标是否保留。
- 硬约束是否保留。
- 数字、日期、版本、路径是否保留。
- 否定条件是否保留。
- 证据来源是否保留。
- 未解决问题是否保留。
- 安全风险是否保留。

## 练习

把一段包含需求、限制、用户偏好、旧决定和工具结果的长对话压缩成结构化状态。然后用压缩状态回答 5 个问题，检查是否能恢复关键事实。

## 验收

- 能选择合适压缩方式。
- 能说明压缩损失。
- 能保留来源、时间和适用范围。

## 重点

压缩应该服务于下一步任务，而不是追求文本短。对 Agent 来说，结构化状态通常比自然语言摘要更可靠。

## 难点

难点是压缩的不可逆性。被摘要掉的细节如果没有原文回溯机制，后续模型可能永远不知道它存在。

## 易错

> **易错：** 把所有历史压缩成一段流畅总结。
>
> 正确做法：把事实、偏好、决定、待办、风险、证据分字段保存。
