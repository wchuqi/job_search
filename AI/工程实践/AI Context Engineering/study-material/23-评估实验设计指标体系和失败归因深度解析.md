# AI Context Engineering 学习资料：评估实验设计、指标体系和失败归因深度解析

[返回索引](../AI Context Engineering学习资料.md)

## 学习目标

- 掌握 Context Engineering 的分层评估方法。
- 能设计离线评测集、线上指标和失败归因流程。
- 能用实验比较不同上下文策略。

## 理论导读

如果只评估最终答案，就无法知道失败来自哪里。Context Engineering 的评估必须分层：数据是否正确、检索是否命中、排序是否合理、压缩是否保真、上下文是否安全、模型是否忠实、工具是否正确调用。

## 分层评估框架

```text
数据层 -> 检索层 -> 选择层 -> 压缩层 -> 打包层
-> 生成层 -> 工具层 -> 安全层 -> 用户效果层
```

## 指标设计

| 层级 | 指标 | 判断 |
| --- | --- | --- |
| 数据 | 文档覆盖率、版本正确率 | 知识库是否可用 |
| 检索 | recall@k、MRR、nDCG | 是否找得到 |
| 选择 | 上下文相关率、冲突率 | 是否选得对 |
| 压缩 | 关键事实保留率 | 是否丢信息 |
| 打包 | 引用 ID 完整率、token 利用率 | 是否组织得好 |
| 生成 | faithfulness、answer relevance | 是否基于证据 |
| 工具 | tool precision、参数正确率 | 是否会行动 |
| 安全 | 注入成功率、泄露率 | 是否可靠 |

## 评测集设计

一套好的评测集应覆盖：

- 正常简单问题。
- 多证据合成问题。
- 资料缺失问题。
- 版本冲突问题。
- 权限边界问题。
- 恶意注入问题。
- 工具失败问题。
- 长对话恢复问题。

每条样本至少包含：

```json
{
  "input": "...",
  "expected_evidence": ["E1", "E3"],
  "expected_behavior": "answer_with_citations",
  "must_not": ["fabricate", "leak_sensitive_data"],
  "scoring_rubric": {}
}
```

## 失败归因流程

```text
1. 查看最终答案
2. 查看最终上下文包
3. 检查 expected evidence 是否进入上下文
4. 检查上下文是否被压缩丢失
5. 检查指令和 schema 是否冲突
6. 检查工具调用和错误恢复
7. 检查安全过滤是否触发
8. 标记失败层级
```

## A/B 实验

比较上下文策略时，一次只改一个核心变量：

- chunk size。
- rerank 模型。
- evidence top-k。
- 压缩方式。
- 记忆注入规则。
- 工具描述。
- prompt 模板。

否则无法知道改进来自哪里。

## 练习

设计一个 50 条样本的评测集，分为 RAG、Agent、记忆、安全、工具五类。每类至少定义两个指标，并说明通过标准。

## 验收

- 能设计分层评估。
- 能从失败样本定位责任层级。
- 能设计可比较的实验。

## 重点

评估的对象不仅是模型输出，还包括模型看到的上下文。

## 难点

难点是自动评分不完全可靠。高价值样本需要人工标注 expected evidence 和行为标准，再结合模型评审或规则评审。

## 易错

> **易错：** 更换模型后发现效果变好，就认为上下文策略正确。
>
> 正确做法：模型能力和上下文质量要分别评估；强模型可能掩盖检索和上下文问题。
