## 一、Agent、记忆和工具

### 1. Agent 的上下文和普通聊天有什么不同？

**参考答案：**

Agent 的上下文会随计划、行动、观察、状态更新和恢复不断变化。普通聊天主要保留对话历史，Agent 还需要保留目标、计划、已完成动作、工具结果、错误、待办、权限和 checkpoint。

> **重点：** Agent 的上下文工程本质上是状态工程。

### 2. 如何设计 Agent 的状态？

**参考答案：**

可以设计结构化状态：

```json
{
  "goal": "完成任务目标",
  "constraints": [],
  "plan": [],
  "completed": [],
  "observations": [],
  "open_questions": [],
  "risks": [],
  "next_action": ""
}
```

状态应支持更新、压缩、恢复和冲突处理。

> **难点：** 状态不是日志，日志记录一切，状态只保存下一步决策需要的信息。

### 3. 长期记忆应该如何写入？

**参考答案：**

长期记忆应来自用户确认、稳定偏好或对未来任务明显有用的信息。写入时记录来源、时间、确认状态、作用范围和隐私等级。敏感信息默认不写入，用户应能查看、修改和删除。

> **易错：** 把模型推断出的用户画像直接写入长期记忆。

### 4. 工具调用为什么是上下文工程问题？

**参考答案：**

模型选择工具依赖工具名称、描述、参数 schema、历史工具结果和当前任务上下文。工具结果又会成为后续模型输入。如果工具描述含糊，模型会误选；如果工具结果未经清洗，可能导致注入；如果权限不受控，可能执行高风险动作。

> **重点：** 工具描述、参数、结果和权限都是上下文。

### 5. 如何处理工具调用失败？

**参考答案：**

应把失败状态、错误类型、可重试条件、关键错误信息和已执行副作用写入状态。不要把完整敏感堆栈直接给模型。对于有副作用工具，重试前必须检查幂等性和是否需要人工确认。

> **易错：** 工具失败后让模型盲目重试，导致重复发送、重复扣费或重复写入。
