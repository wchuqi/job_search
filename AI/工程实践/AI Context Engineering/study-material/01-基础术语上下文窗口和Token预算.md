# AI Context Engineering 学习资料：基础术语、上下文窗口和 Token 预算

[返回索引](../AI Context Engineering学习资料.md)

## 学习目标

- 掌握 Context Engineering 的基础术语。
- 理解上下文窗口、token、预算和信息密度之间的关系。
- 能为一次模型调用设计基础预算表。

## 理论导读

模型每次调用都有一个上下文窗口。窗口越大，能放入的信息越多，但并不等于模型会更准确。长上下文会带来成本、延迟、噪声、冲突和注意力分散问题。工程上必须用预算约束每类信息，而不是等超限后再粗暴截断。

## 核心概念

| 概念 | 含义 | 工程关注点 |
| --- | --- | --- |
| Context window | 本次调用能容纳的输入和输出总量 | 容量、成本、截断 |
| Token budget | 给每类上下文分配 token 上限 | 稳定性、延迟、质量 |
| Context slot | 上下文中有明确职责的一块内容 | 来源、权限、生命周期 |
| Salience | 信息对当前任务的重要性 | 排序、压缩、保留 |
| Recency | 信息的新旧程度 | 过期、冲突 |
| Authority | 信息来源的可信等级 | 冲突解决 |
| Grounding | 让答案基于可验证证据 | RAG、引用 |

## 预算设计

一个通用预算表：

| 槽位 | 建议预算 | 原则 |
| --- | --- | --- |
| 系统规则 | 小而稳定 | 保留不可违反的规则 |
| 当前用户请求 | 原文优先 | 不要摘要到丢失意图 |
| 任务状态 | 中等 | 用结构化状态替代全量历史 |
| 检索证据 | 弹性 | 按问题需要动态分配 |
| 工具结果 | 中等 | 保留结论、错误和关键字段 |
| 示例 | 小 | 只保留高价值样例 |
| 输出契约 | 小而明确 | schema、格式、评分标准 |

## 例子

```python
context_budget = {
    "system_rules": 800,
    "current_request": 1200,
    "task_state": 1500,
    "retrieved_evidence": 6000,
    "tool_results": 2000,
    "output_schema": 800,
    "reserved_for_answer": 3000,
}
```

## 练习

为“AI 面试辅导助手”设计预算表，至少包含：当前问题、候选人背景、目标岗位、历史弱点、题库证据、评分标准、输出格式。

## 验收

- 能说明每类上下文的预算理由。
- 能解释哪些信息应该原文保留，哪些应该压缩。
- 能在预算不足时按优先级裁剪。

## 重点

Token 预算不是纯成本优化，它是质量控制手段。没有预算的上下文系统会在长对话、RAG 和 Agent 中逐渐退化。

## 难点

难点是动态预算：同一个系统在简单闲聊、文档问答、代码修复、工具调用时，预算分配应完全不同。

## 易错

> **易错：** 先拼接所有内容，再按长度从尾部截断。
>
> 正确做法：先按槽位分预算，再对低优先级内容做摘要、抽取、重排或丢弃。
