# AI Context Engineering 学习资料：上下文窗口、注意力限制和信息位置效应深度解析

[返回索引](../AI Context Engineering学习资料.md)

## 学习目标

- 理解上下文窗口不是无限可靠的记忆。
- 掌握长上下文中的噪声、位置、冲突和召回问题。
- 能根据任务设计信息位置和重复策略。

## 理论导读

上下文窗口表示模型本次调用最多能接收多少 token，但它不承诺每个 token 都被同等利用。长输入会带来多个问题：关键证据被噪声稀释，远距离依赖变弱，多个相似片段造成冲突，输出预算挤占输入预算，模型可能对开头和结尾更敏感。

因此，Context Engineering 不能只关心“放得下”，还要关心“模型能不能稳定使用”。

## 长上下文的四类限制

### 1. 容量限制

输入和输出共享上下文窗口。预留输出 token 不足时，模型可能截断答案或无法完成格式。

### 2. 注意力限制

模型处理长文本时，关键事实可能被大量弱相关内容淹没。尤其是多个片段语义相近但细节不同，会增加错误组合概率。

### 3. 位置效应

关键信息放在上下文中间可能更容易被忽略。工程上常把任务目标、硬约束、输出格式放在靠前或靠近生成位置的位置；把证据按相关性排序，并在回答前重新强调引用要求。

### 4. 冲突效应

当上下文里存在旧版本 API、重复文档、过期用户偏好和当前请求冲突时，模型可能混用信息。

## 信息位置策略

| 信息 | 推荐位置 | 原因 |
| --- | --- | --- |
| 不可违反规则 | 开头稳定位置 | 建立行为边界 |
| 当前任务目标 | 靠前且简洁 | 减少跑题 |
| 输出 schema | 靠近生成要求 | 提高格式遵循 |
| 高相关证据 | 靠近问题或答案指令 | 便于引用 |
| 低可信资料 | 单独包裹 | 防止指令混淆 |
| 历史摘要 | 任务状态之后 | 防止旧历史抢占当前目标 |

## 上下文窗口预算公式

```text
total_window >= system + task + state + evidence + tool_results + schema + answer_budget
```

如果预算不足，优先裁剪：

```text
低相关证据 -> 重复证据 -> 旧历史 -> 示例 -> 冗余工具日志
```

不要裁剪：

```text
当前请求、硬约束、权限边界、关键证据来源、输出 schema
```

## 例子：错误的位置设计

```text
开头：大量历史聊天
中间：当前用户真正问题
后面：20 个检索片段
最后：请回答
```

风险：当前问题被历史和证据淹没，输出格式不稳定，引用规则弱。

更稳的设计：

```text
稳定规则
当前任务和硬约束
结构化任务状态
证据片段，按相关性排序并带来源
输出格式和引用规则
```

## 练习

给一个 50 页 API 文档问答任务设计上下文打包策略，说明：

- 文档是全量放入还是检索后放入。
- 输出预算预留多少。
- schema 和引用规则放在哪里。
- 如何处理多个版本 API 文档。

## 验收

- 能解释长上下文的容量、注意力、位置和冲突问题。
- 能设计输出预算。
- 能说明为什么关键信息不能藏在长上下文中间。

## 重点

上下文窗口是容量，不是理解质量。信息位置、结构和噪声控制同样重要。

## 难点

难点是任务差异：总结任务可能适合放大量全文，精确问答任务更适合检索和证据打包，Agent 任务更适合状态化摘要。

## 易错

> **易错：** 因为模型支持大窗口，就把完整历史和完整文档全部塞进去。
>
> 正确做法：根据任务目标选择信息，保留输出预算，并用结构、排序和引用规则提高可用性。
