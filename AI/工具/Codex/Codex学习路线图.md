# Codex学习路线图

> 资料更新时间：2026-06-08。Codex 是持续演进的 OpenAI 编程代理，本路线图以 OpenAI 官方 Codex 文档和 Codex CLI 文档为主要依据。

## 阶段 1：基础认知

- 目标：理解 Codex 的定位、适合任务、交互入口和协作边界。
- 需要掌握：Codex App、IDE 扩展、CLI、Cloud tasks、ChatGPT 中的 Codex 入口、模型能力、任务上下文。
- 例子：让 Codex 阅读一个小项目，解释模块关系，提出一个小修复，并生成测试建议。
- 练习：选一个已有仓库，让 Codex 执行“只读代码审查”，要求它给出文件路径、风险和建议。
- 验收：能判断一个任务适合交给 Codex、应给多少上下文、哪些操作必须人工确认。
- 重点：Codex 是能操作代码和工具的代理，不只是补全器或问答机器人。
- 易错：把 Codex 当成“自动写完整项目”的黑盒，缺少边界、验收和版本控制保护。

## 阶段 2：仓库上下文和提示词

- 目标：学会给 Codex 足够但不过量的上下文，让它按仓库现有风格工作。
- 需要掌握：任务说明、验收标准、限制条件、文件范围、AGENTS.md、引用文件、复现步骤、错误日志。
- 例子：把“修 bug”改写为“在 `src/auth` 内修复登录失败时状态未重置的问题，保留现有 API，补充单元测试”。
- 练习：为一个仓库写一份 `AGENTS.md`，包含构建、测试、代码风格、安全注意事项。
- 验收：Codex 能按你的说明定位相关文件，少走弯路，并在最终答复中报告验证结果。
- 重点：好提示词不是长，而是边界清楚、目标可验收、证据可追踪。
- 难点：在开放任务中逐步收敛范围，而不是一次性要求 Codex 猜完整需求。

## 阶段 3：CLI、IDE 和 Cloud 工作流

- 目标：掌握 Codex 在本地和远程环境中的常见使用方式。
- 需要掌握：CLI 交互模式、非交互运行、IDE 中的代码导航与编辑、Cloud tasks、PR/分支工作流。
- 例子：本地用 CLI 修复测试失败；IDE 中审查变更；Cloud 中并行探索两个实现方案。
- 练习：把同一任务分别用 IDE 和 CLI 完成一次，比较上下文、审批、验证和输出差异。
- 验收：能根据任务类型选择入口：快速问答用 IDE，仓库级修改用 CLI，大型异步任务用 Cloud。
- 重点：入口不同，但核心能力都依赖明确任务、可访问上下文和可执行验证。
- 易错：远程任务没有同步本地最新分支，导致 Codex 基于旧代码工作。

## 阶段 4：权限、沙箱和安全

- 目标：理解 Codex 执行命令、修改文件、访问网络时的风险控制。
- 需要掌握：approval mode、sandbox mode、workspace-write、read-only、danger-full-access、网络限制、秘密信息处理。
- 例子：允许 Codex 运行测试，但要求安装依赖、删除文件、访问外网前请求审批。
- 练习：为一个仓库配置保守权限，并模拟三类操作：只读搜索、写入代码、需要网络的依赖安装。
- 验收：能解释“哪些命令可以自动执行，哪些必须审批，为什么”。
- 重点：权限策略是工程安全边界，不是使用障碍。
- 易错：为了省事长期使用最高权限，导致误删文件、泄露密钥或执行不可信脚本的风险上升。

## 阶段 5：工程化协作和质量控制

- 目标：把 Codex 纳入日常开发流程，而不是只做一次性生成。
- 需要掌握：小步任务拆分、测试驱动、代码审查、回滚、提交说明、CI、文档同步、团队规范。
- 例子：让 Codex 先写失败测试，再实现修复，最后根据 diff 做自审。
- 练习：完成一个“修复 bug + 添加测试 + 更新文档 + 生成 PR 描述”的完整流程。
- 验收：最终变更可读、可测、可回滚，且 Codex 的推理和验证结果能被人复核。
- 重点：Codex 可以加速工程活动，但质量责任仍在开发者。
- 难点：不要让 Codex 同时处理过多目标；大型任务要拆成可验证的小批次。

## 阶段 6：进阶与排障

- 目标：处理 Codex 上下文不足、命令失败、误改、测试不稳定、权限阻塞等问题。
- 需要掌握：日志与错误复现、最小复现、上下文压缩、冲突处理、恢复策略、网络和依赖问题、执行生命周期、权限决策和 MCP 工具风险。
- 例子：Codex 反复改错文件时，要求它先列出文件证据和调用链，再做最小修改。
- 练习：准备一个故意失败的项目，让 Codex 定位失败原因，并要求它写出排障记录。
- 验收：遇到失败时能收敛问题，而不是不断追加模糊指令。
- 重点：排障的核心是把“代理猜测”转化为“可验证证据”。
- 易错：在 Codex 已经偏离目标时继续让它扩大修改范围。

## 阶段 7：机制级掌握

- 目标：能解释 Codex 为什么这样行动，并能设计团队级安全和治理机制。
- 需要掌握：任务生命周期、上下文优先级、sandbox/approval/network/auto review 关系、MCP 工具设计、prompt injection、模型选择、长上下文和成本治理、OTel 审计。
- 例子：为一个支付服务设计 Codex 使用策略，区分只读审计、修复单测、安全修复和生产事故分析。
- 练习：写一份企业级 Codex 治理方案，包含风险分级、权限策略、PR 标注、审计日志和事故复盘模板。
- 验收：能回答“为什么这个命令需要审批”“为什么这个 MCP 工具不能暴露任意 SQL”“为什么 Cloud 测试通过本地还会失败”。
- 重点：机制级掌握要求能设计边界，不只是会用命令。
- 难点：效率、安全、成本和开发体验之间需要权衡。

## 推荐学习节奏

| 周期 | 学习重点 | 产出 |
| --- | --- | --- |
| 第 1-2 天 | 基本定位、入口、上下文写法 | 1 份任务提示词模板 |
| 第 3-4 天 | CLI/IDE 基础工作流 | 完成 1 个小 bug 修复 |
| 第 5-6 天 | AGENTS.md、测试和验证 | 仓库级 Codex 使用规范 |
| 第 7-8 天 | 权限、沙箱、安全 | 1 份权限策略清单 |
| 第 9-10 天 | Cloud tasks、PR、团队协作 | 1 个可审查的 Codex 任务流程 |
| 第 11-14 天 | 排障、代码审查、自动化 | 1 套面试/复盘问答 |
| 第 15-18 天 | 生命周期、权限决策、MCP 安全 | 1 份机制级总结 |
| 第 19-21 天 | 团队治理、审计、成本治理 | 1 份企业落地规范 |

## 最终能力清单

- 能说明 Codex 与普通代码补全、聊天问答、CI 自动化的区别。
- 能为任务编写目标、上下文、限制和验收标准。
- 能配置或选择合适的权限和沙箱策略。
- 能让 Codex 做代码阅读、实现、测试、审查和调试。
- 能识别 Codex 输出中的幻觉、过度修改、测试不足和安全风险。
- 能把 Codex 接入团队工作流，并制定复核规则。
- 能回答 Codex 使用、工程化、安全和面试场景问题。
- 能解释 Codex 执行生命周期、上下文决策和权限审批机制。
- 能设计 MCP 工具安全边界和团队审计方案。
