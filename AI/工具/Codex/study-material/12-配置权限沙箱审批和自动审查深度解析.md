# Codex学习资料：配置权限沙箱审批和自动审查深度解析

[返回索引](../Codex学习资料.md)

## 学习目标

- 深入理解 sandbox mode、approval policy、网络访问、自动审查和配置文件之间的关系。
- 能为不同项目设计权限策略。
- 能解释命令为什么被允许、被阻止或要求审批。

## 理论导读

Codex 的安全控制不是一个开关，而是多层组合：沙箱决定“技术上能做什么”，审批策略决定“什么时候必须问人”，自动审查决定“某些审批请求是否先交给审查代理判断”。这些层叠加后，才形成实际运行边界。

OpenAI 官方文档明确把 Codex 安全控制分为两层：sandbox mode 控制技术能力，approval policy 控制何时请求批准。CLI/IDE 默认常见策略是工作区可写、网络默认关闭、越界写入或网络访问需要审批。

## 核心层次

```text
用户任务
  -> 配置解析
  -> 沙箱边界
  -> 命令/工具风险分类
  -> 审批策略
  -> 可选自动审查
  -> 执行或拒绝
```

## 一、沙箱模式

沙箱回答的是“Codex 技术上能不能做”。

| 模式 | 能力 | 适合场景 | 风险 |
| --- | --- | --- | --- |
| read-only | 读取文件和回答问题 | 代码审查、学习、方案分析 | 不能自动修复 |
| workspace-write | 可写当前工作区 | 日常开发、测试修复 | 可能误改项目文件 |
| danger-full-access | 基本绕过沙箱限制 | 受控容器、临时高权限任务 | 高风险，不建议默认 |

### 工作区边界

workspace-write 通常意味着当前项目目录可写，但不代表所有路径都可写。官方文档中还提到一些受保护路径，例如 `.git`、`.codex`、`.agents` 在默认工作区写入策略下会被保护为只读。

这背后的设计逻辑是：允许 Codex 完成代码修改，但降低破坏版本控制元数据和代理配置的风险。

> **重点：** workspace-write 不是无限写入，它是“给项目变更留通道，同时保护关键控制面”。

## 二、审批策略

审批策略回答的是“Codex 什么时候必须问人”。

| 策略 | 含义 | 适用 |
| --- | --- | --- |
| on-request | 遇到越界或高风险请求时询问 | 日常推荐 |
| untrusted | 只自动运行已知安全读操作，其他更谨慎 | 陌生仓库、安全敏感仓库 |
| never | 不弹审批，能做就做，不能做就失败 | CI 只读检查或严格自动化 |
| granular | 对不同审批类别分别控制 | 企业或复杂团队 |

### 决策示例

```text
命令：rg "password" src
沙箱：workspace-write
审批：on-request
结果：通常可自动运行，因为是只读搜索。

命令：npm install
沙箱：workspace-write，网络关闭
审批：on-request
结果：需要审批或失败，因为涉及网络和依赖脚本。

命令：Remove-Item -Recurse .
沙箱：workspace-write
审批：untrusted
结果：应请求审批或被拒绝，因为 destructive。
```

## 三、网络访问

网络访问是单独风险维度。即使允许写工作区，也不应默认允许网络。网络打开后风险包括：

- 下载未审查依赖。
- 执行安装脚本。
- 外传源码、日志或密钥。
- 访问内部服务。
- 调用生产 API。

### 安全策略

- 默认关闭网络。
- 需要下载依赖时临时审批。
- 企业环境使用 allowlist。
- 日志和遥测只发往受控 collector。
- 不在 Codex 输出中暴露 token。

## 四、自动审查

自动审查是对“已经需要审批的请求”增加一层审查代理。它不是替代沙箱，也不是让所有操作自动通过。根据官方说明，自动审查关注数据外传、凭据探测、持久性安全削弱、破坏性操作等风险。

### 心智模型

```text
普通安全边界：
请求 -> 人工审批 -> 执行

自动审查边界：
请求 -> 审查代理判断风险 -> 批准/拒绝/仍需用户授权 -> 执行或停止
```

### 适合场景

- 团队希望减少低风险审批噪音。
- 企业希望对审批请求做统一策略判断。
- 安全敏感团队需要记录风险级别。

### 不适合场景

- 生产破坏性操作。
- 需要业务负责人判断的变更。
- 密钥和权限变更。

> **难点：** 自动审查降低审批负担，但不能替代业务授权和最终责任。

## 五、配置文件和策略落地

Codex 支持通过配置文件保存默认策略。一个保守配置示例：

```toml
approval_policy = "untrusted"
sandbox_mode = "read-only"
allow_login_shell = false
```

一个日常开发配置示例：

```toml
approval_policy = "on-request"
sandbox_mode = "workspace-write"

[sandbox_workspace_write]
network_access = false
```

一个团队可审计配置示例：

```toml
approval_policy = "on-request"
sandbox_mode = "workspace-write"
approvals_reviewer = "auto_review"

[otel]
environment = "staging"
exporter = "none"
log_user_prompt = false
```

## 六、权限策略设计模式

### 模式 1：陌生仓库评估

- 沙箱：read-only。
- 审批：never 或 on-request。
- 网络：关闭。
- 允许：读文件、搜索、总结。
- 禁止：安装依赖、运行未知脚本、写文件。

### 模式 2：日常开发

- 沙箱：workspace-write。
- 审批：on-request。
- 网络：默认关闭。
- 允许：读写项目文件、运行已知测试。
- 审批：依赖安装、外网访问、大规模删除。

### 模式 3：受控容器高权限

- 外层：devcontainer/Docker 提供隔离。
- 内层：可按需放宽 Codex 沙箱。
- 网络：容器防火墙或域名 allowlist。
- 审计：记录命令、工具、审批。

> **易错：** 在宿主机直接使用高权限，和在受控容器内使用高权限，风险完全不同。

## 练习

为以下项目写权限策略：

1. 开源前端组件库。
2. 公司内部支付服务。
3. 只读代码审计任务。
4. 需要下载依赖的临时迁移任务。

每个策略说明沙箱、审批、网络、密钥、日志和回滚。

## 验收

- 能解释 sandbox 和 approval 的区别。
- 能预测一个命令会自动执行、请求审批还是失败。
- 能写出适合团队的 `config.toml` 策略。
- 能说明自动审查的边界。

## 重点

- 沙箱控制能力，审批控制决策，自动审查控制审批请求的风险判断。
- 权限策略必须与任务风险和仓库可信度匹配。

## 难点

- 真实项目中脚本副作用常隐藏在 `package.json`、构建插件、Git hooks、依赖安装脚本和测试初始化里。

## 易错

> **易错：** 认为 `--ask-for-approval never` 等于“更自动、更高效”。
>
> 正确做法：它只是关闭交互审批；如果沙箱很窄，任务会失败；如果沙箱很宽，风险会急剧上升。
