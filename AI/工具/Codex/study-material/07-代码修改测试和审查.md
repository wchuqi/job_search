# Codex学习资料：代码修改测试和审查

[返回索引](../Codex学习资料.md)

## 学习目标

- 掌握 Codex 修改代码的可靠流程。
- 学会让 Codex 生成和运行测试。
- 能审查 Codex 的 diff、解释和残余风险。

## 理论导读

Codex 生成代码的速度很快，但工程质量来自验证。一次可靠修改应包括：定位根因、最小变更、测试覆盖、回归检查、变更解释和风险说明。

## 推荐流程

1. 让 Codex 先复现或解释问题。
2. 要求它列出相关文件和调用链。
3. 要求最小修改，不做无关重构。
4. 补充或更新测试。
5. 运行最小相关测试。
6. 必要时运行更大范围测试。
7. 审查 diff。
8. 要求 Codex 总结残余风险。

## 测试策略

| 场景 | 优先测试 | 补充验证 |
| --- | --- | --- |
| bug 修复 | 失败用例或回归测试 | 相关模块单测 |
| 重构 | 原有测试全通过 | 行为快照或契约测试 |
| 性能优化 | 基准测试 | 正确性测试 |
| 依赖升级 | 单元测试和集成测试 | 兼容性检查 |
| 安全修复 | 攻击用例 | 正常路径回归 |

## 代码审查清单

- 修改是否集中在任务范围内。
- 是否改变公共 API 或外部行为。
- 是否引入新依赖。
- 是否删除了重要错误处理。
- 是否只让测试“看起来通过”，而不是修复根因。
- 是否补了失败用例。
- 是否有残余风险说明。

## 例子

```text
请修复这个测试失败。
要求：
1. 先说明失败断言和实际行为。
2. 找到导致行为的代码路径。
3. 写一个能在修复前失败、修复后通过的测试。
4. 修改范围越小越好。
5. 最后给出 diff 摘要和测试命令。
```

## 练习

找一个历史 bug，删除它的回归测试，然后让 Codex 重新补测试和修复。比较它是否能覆盖原始问题。

## 验收

- Codex 的修改有对应测试。
- 你能解释每个文件为什么被改。
- 你能判断验证范围是否足够。

## 重点

- Codex 的最终答复必须包含验证命令和结果。
- 没有测试的“已修复”只能算候选结果。

## 难点

- 有些问题无法用单元测试完全覆盖，需要结合集成测试、手工复现或日志验证。

## 易错

> **易错：** 让 Codex 为了通过测试而修改测试期望。
>
> 正确做法：先确认业务期望，再决定改生产代码还是测试。
