# Codex学习资料：MCP工具和自动化集成

[返回索引](../Codex学习资料.md)

## 学习目标

- 理解 MCP 在 Codex 工具扩展中的作用。
- 知道何时接入外部工具、文档、数据库或内部系统。
- 掌握自动化集成的安全边界。

## 理论导读

MCP 可以让 Codex 通过标准方式连接外部工具和上下文，例如文档检索、Issue 系统、数据库查询、浏览器自动化、内部服务等。它的价值是把“人工复制粘贴上下文”变成“受控工具调用”。

## 使用场景

- 查询官方文档或内部知识库。
- 读取 Issue、PR、CI 结果。
- 运行浏览器测试或截图。
- 查询只读数据库。
- 调用内部诊断工具。

## 工具设计原则

| 原则 | 说明 |
| --- | --- |
| 最小能力 | 工具只暴露任务需要的操作 |
| 明确参数 | 不让 Codex 拼接任意危险命令 |
| 可审计 | 记录调用、参数和结果 |
| 分级权限 | 只读、写入、生产操作分开 |
| 脱敏输出 | 不返回密钥和隐私数据 |

## 自动化边界

适合自动化：

- 查询文档。
- 拉取测试报告。
- 创建草稿 PR。
- 生成变更摘要。
- 对非生产环境执行测试。

不宜完全自动化：

- 生产发布。
- 生产数据修复。
- 权限变更。
- 财务或合规相关操作。
- 删除不可恢复数据。

## 例子

MCP 工具说明应像 API 合同，而不是泛化 shell：

```text
工具：get_ci_status
输入：repository、commit_sha
输出：测试套件、状态、失败日志摘要
限制：只读，不触发重新运行
```

## 练习

为团队设计 3 个 Codex 可用工具：

1. 查询内部文档。
2. 查询 CI 失败日志。
3. 创建草稿 PR 描述。

为每个工具写出输入、输出、权限和风险控制。

## 验收

- 能解释 MCP 解决的问题。
- 能判断一个工具是否过度授权。
- 能设计只读优先、可审计的工具接口。

## 重点

- MCP 扩展的是 Codex 的行动能力，因此安全设计比接入速度更重要。

## 难点

- 内部工具往往带有隐式权限。接入 Codex 前要把权限、数据范围和审计要求显式化。

## 易错

> **易错：** 给 Codex 暴露一个“执行任意 SQL”或“执行任意 shell”的工具。
>
> 正确做法：暴露任务级工具，例如“查询某 commit 的 CI 失败摘要”，而不是暴露无限制执行能力。
