# Codex学习资料：Codex完整知识点清单

[返回索引](../Codex学习资料.md)

## 一、基础和术语

- Codex 的定位：软件开发编码代理。
- 与代码补全、聊天问答、脚本自动化、CI 的区别。
- 主要入口：CLI、IDE、Cloud、ChatGPT、code review。
- 任务、上下文、工具、权限、验证、最终汇报。
- 模型能力和产品能力的区别。

## 二、产品形态和工作流

- CLI 本地交互。
- CLI 非交互和自动化运行。
- IDE 代码阅读和局部编辑。
- Cloud tasks 异步执行。
- PR/分支工作流。
- ChatGPT 中讨论方案和解释代码。
- 代码审查场景。
- 多任务并行探索。

## 九、排障和恢复

- 环境不一致。

- 测试不稳定。

- diff 过大。

- 分阶段重试。
- 生命周期定位：任务解析、上下文获取、假设计划、工具选择、状态修改、观察迭代、最终汇报。

## 十三、容易遗漏的高级点

- 代理不是确定性编译器，输出需要验证。
- 工具权限会改变风险模型。
- Cloud 和本地环境可能不一致。
- `AGENTS.md` 适合稳定规则，不适合临时需求。
- 测试通过不等于业务正确。
- 自动化工具接口不应暴露无限制 shell/SQL。
- 高风险任务中 Codex 适合分析，不适合独立执行。
- 多个 Codex 任务并行可能产生冲突实现。
- 依赖安装脚本可能执行任意代码。
- 最终责任仍属于开发团队。
- 外部文档、Issue、日志都可能包含 prompt injection。
- 高权限在宿主机和受控容器中的风险不同。
- 自动审查不是业务审批。
- 配置文件、`AGENTS.md` 和单次提示词分别承担不同职责。
- 遥测本身也可能泄露敏感信息。

## 快速自查

- 我是否明确了目标和验收？
- 我是否限制了修改范围？
- 我是否保护了密钥和生产数据？
- 我是否看过 Codex 的 diff？
- 我是否运行了足够测试？
- 我是否知道如何回滚？
- 我是否能解释每处修改的原因？
