# Codex学习资料：排障恢复和反模式

[返回索引](../Codex学习资料.md)

## 学习目标

- 掌握 Codex 任务失败时的排障方法。
- 能识别常见反模式。
- 知道如何恢复、收敛和重新定义任务。

## 理论导读

Codex 失败通常不是单点问题，而是上下文、权限、环境、任务边界或验证标准出了偏差。排障目标不是让 Codex“继续试”，而是找出失败链路并缩小搜索空间。

## 常见失败类型

| 失败类型 | 表现 | 处理方式 |
| --- | --- | --- |
| 上下文不足 | 改错文件、猜接口 | 补充日志、路径、调用链 |
| 任务过大 | diff 发散、无关重构 | 拆成阶段任务 |
| 权限阻塞 | 命令无法执行 | 明确允许范围或手动执行 |
| 环境不一致 | 本地过、CI 不过 | 对齐依赖、系统版本、环境变量 |
| 测试不足 | 看似修复但回归 | 增加回归测试 |
| 错误假设 | 解释和事实不符 | 要求 Codex 引用文件证据 |

## 恢复流程

1. 停止继续扩大修改。
2. 查看当前 diff。
3. 要求 Codex 总结它已做的假设。
4. 找出第一个无证据假设。
5. 回到只读分析模式。
6. 明确新的最小目标。
7. 重新执行并验证。

## 反模式

### 反模式 1：无限追加指令

表现：

```text
不对，再改一下。
还是不对，继续。
```

问题：Codex 不知道“不对”具体指什么，会不断扩大搜索和修改。

正确做法：

```text
当前问题是测试 A 仍失败，失败断言是 X。
请不要继续修改，先解释为什么你的上一版没有改变这个断言。
```

### 反模式 2：一次交付大型重构

问题：修改范围大，难以审查，失败后难以定位。

正确做法：先让 Codex 输出调用图和风险，再选择一个低风险切片。

### 反模式 3：忽略工作区状态

问题：Codex 可能在有未提交改动的工作区继续编辑，难以区分人和代理的修改。

正确做法：任务前查看 git status，任务后查看 diff。

## 练习

构造一个失败场景：

- 给 Codex 不完整错误日志。
- 观察它是否猜测。
- 再补充完整堆栈和复现步骤。
- 比较两次输出质量。

## 验收

- 能从失败中定位是上下文问题、环境问题还是实现问题。
- 能要求 Codex 回到证据驱动模式。
- 能防止 diff 继续发散。

## 重点

- 失败时先收敛，不要加速。
- 要求 Codex 引用文件、命令输出和测试结果。

## 难点

- 不稳定测试、外部服务依赖和并发问题会让 Codex 难以单次定位，需要人为提供复现策略。

## 易错

> **易错：** 在 Codex 已经偏离目标时，继续给它更多权限。
>
> 正确做法：先降低权限和范围，重新做只读分析。
