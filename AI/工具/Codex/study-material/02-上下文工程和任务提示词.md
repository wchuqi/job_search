# Codex学习资料：上下文工程和任务提示词

[返回索引](../Codex学习资料.md)

## 学习目标

- 学会为 Codex 编写高质量任务说明。
- 掌握上下文选择、范围控制和验收标准。
- 避免提示词过短、过宽、过度暗示或缺少证据。

## 理论导读

上下文工程不是把所有资料塞给 Codex，而是把“完成任务所需的关键证据和约束”组织好。Codex 的工作方式通常是：理解目标、探索代码、形成假设、编辑文件、运行验证、汇报结果。提示词要服务于这条链路。

## 任务提示词结构

```text
目标：
  说明要解决什么问题，以及完成后的行为。

背景：
  给出错误日志、复现步骤、用户场景、相关提交或设计约束。

范围：
  指定优先查看或允许修改的目录/文件。

限制：
  说明不能改的 API、兼容性、性能、安全、依赖和代码风格要求。

验证：
  指定要运行的测试、构建、lint 或手工检查。

输出：
  要求 Codex 报告修改文件、验证结果、未解决风险。
```

## 心智模型

把提示词看成“发给工程师的工单”。工单不需要教对方每一行怎么写，但必须说清楚成功标准。Codex 会探索细节，但不能替你猜业务边界。

## 好提示词示例

```text
请修复用户退出登录后刷新页面仍然显示旧用户名的问题。

背景：
- 复现步骤：登录 -> 点击退出 -> 刷新页面。
- 期望：刷新后显示未登录状态。
- 当前现象：页面仍从本地缓存读到了旧用户名。

范围：
- 优先查看 src/auth、src/store、src/api/session。
- 不要改后端接口。

限制：
- 保持现有 public API。
- 不引入新依赖。

验证：
- 运行 auth/session 相关测试。
- 如果没有覆盖，请补一个最小单元测试。

输出：
- 先说明根因和调用链。
- 修改后列出文件、测试命令和残余风险。
```

## 常见上下文材料

| 材料 | 作用 | 注意事项 |
| --- | --- | --- |
| 错误日志 | 定位失败点 | 保留完整堆栈，不只截最后一行 |
| 复现步骤 | 明确行为差异 | 写出输入、动作、期望、实际 |
| 相关文件 | 减少搜索成本 | 不要过度限制到错误文件 |
| 设计约束 | 防止错误修复 | 包括兼容性、安全、性能 |
| 测试命令 | 让结果可验证 | 指定最小测试和必要的全量测试 |
| 历史背景 | 避免回退旧问题 | 标注哪些是事实，哪些是猜测 |

## 练习

把下面模糊任务改写成高质量任务：

```text
这个接口有问题，帮我看看。
```

验收答案应包含：

- 接口名称或路径。
- 复现请求和响应。
- 期望行为。
- 相关日志。
- 修改范围。
- 验证方式。

## 验收

- 能写出结构化 Codex 任务。
- 能区分背景、目标、限制和验收。
- 能要求 Codex 先读代码再修改，而不是直接生成猜测代码。

## 重点

- 明确验收标准比“详细描述怎么实现”更重要。
- 任务越大，越要先要求 Codex 做计划、证据和风险分析。

## 难点

- 给足上下文但不替 Codex 过早下结论。错误的根因暗示会把它引向错误路径。

## 易错

> **易错：** 只要求“优化”“重构”“完善”，没有说明成功标准。
>
> 正确做法：把抽象目标转换成可观察行为、性能指标、测试结果或代码结构约束。
