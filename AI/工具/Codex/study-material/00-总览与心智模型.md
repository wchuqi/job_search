# Codex学习资料：总览与心智模型

[返回索引](../Codex学习资料.md)

## 学习目标

- 理解 Codex 是什么，不是什么。
- 建立“代理协作”的心智模型。
- 知道 Codex 工作质量由任务、上下文、权限、验证共同决定。

## 理论导读

Codex 是面向软件开发的编码代理。它可以阅读代码、理解仓库结构、提出修改方案、编辑文件、运行命令、分析测试结果、做代码审查和调试。与普通代码补全不同，Codex 的核心不是“在光标位置补几行代码”，而是围绕一个工程目标执行多步任务。

可以把 Codex 想成一个临时加入项目的工程师：它需要知道需求、项目结构、编码规范、可执行命令、风险边界和验收方式。你给出的信息越像一个清晰的任务单，它越容易产出可审查的结果。

## 核心心智模型

### 1. 任务单模型

一次高质量 Codex 任务通常包含：

- 背景：当前问题是什么。
- 目标：最终要达到什么状态。
- 范围：允许修改哪些文件或模块。
- 约束：不能改变哪些 API、性能、安全或兼容行为。
- 验收：要跑哪些测试，输出什么证据。

> **重点：** Codex 不只是“回答问题”，它会根据你的授权探索仓库和执行操作。任务描述必须可执行、可验证。

### 2. 上下文窗口模型

Codex 不能天然知道整个项目的所有细节。它会通过读取文件、搜索符号、运行命令、查看日志等方式构建上下文。上下文不足时，它容易猜错；上下文过载时，它可能抓不住重点。

正确做法是先给关键线索：相关文件、错误日志、复现步骤、近期改动、约束条件。让 Codex 自己补充探索，但要求它基于证据行动。

### 3. 人机分工模型

Codex 适合承担重复、探索、草拟、验证和局部实现工作；人负责目标判断、架构取舍、安全审批、最终审查和上线责任。

| 工作 | Codex适合度 | 人的责任 |
| --- | --- | --- |
| 阅读陌生代码并总结 | 高 | 判断总结是否覆盖关键路径 |
| 修复局部 bug | 高 | 确认修复符合业务规则 |
| 执行危险命令 | 低 | 审批、备份、回滚 |
| 跨团队架构决策 | 中 | 明确约束和最终拍板 |
| 生成测试用例 | 高 | 检查测试是否真的覆盖风险 |

## 例子

不推荐：

```text
帮我优化这个项目。
```

推荐：

```text
请检查登录流程里 token 刷新失败后页面没有退出登录的问题。
范围限制在 src/auth、src/api 和相关测试。
不要改公共 API。
请先说明你找到的调用链，再做最小修改，最后运行 auth 相关测试。
```

## 练习

1. 选一个你熟悉的仓库。
2. 写一个只读任务，让 Codex 总结模块边界。
3. 写一个修改任务，让 Codex 修复一个小问题。
4. 要求 Codex 在最终答复中列出修改文件、验证命令和残余风险。

## 验收

- 能把模糊需求改写为可执行任务单。
- 能解释 Codex 为什么需要仓库上下文。
- 能区分 Codex 输出中的事实、推断和建议。
- 能决定哪些步骤可以自动执行，哪些必须人工审批。

## 重点

- Codex 的产出质量等于任务质量、上下文质量、工具权限和验证强度的综合结果。
- 最可靠的用法是“小任务、强约束、可验证、可回滚”。

## 难点

- 难点不是写提示词本身，而是把真实工程任务拆成清楚的目标、边界和证据链。

## 易错

> **易错：** 只看 Codex 的自然语言解释，不看实际 diff 和测试结果。
>
> 正确做法：把 Codex 输出当成候选方案，必须通过代码审查、测试和业务规则复核。
