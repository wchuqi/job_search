# Codex学习资料：执行生命周期和上下文决策深度解析

[返回索引](../Codex学习资料.md)

## 学习目标

- 不是只会“让 Codex 改代码”，而是能解释一次 Codex 任务从输入到输出的生命周期。
- 能识别上下文不足、错误假设、验证不足和任务发散发生在哪个环节。
- 能把开放任务拆成可控制、可审查、可回滚的执行链。

## 理论导读

Codex 的工作不是一次性生成答案，而是一个闭环：接收目标、构建上下文、形成计划、调用工具、修改状态、观察结果、修正假设、输出总结。这个闭环和传统编译器不同：编译器按确定规则处理输入，Codex 是基于上下文和模型推理选择下一步动作。

理解生命周期的价值在于：当结果不好时，你能定位问题发生在“目标不清”“上下文缺失”“工具受限”“验证错误”还是“风险边界缺失”，而不是简单归因于“模型不行”。

## 生命周期总览

```text
用户任务
  -> 任务解析
  -> 上下文获取
  -> 假设和计划
  -> 工具选择
  -> 文件修改或命令执行
  -> 观察结果
  -> 迭代修正
  -> 最终汇报
  -> 人工审查和合并
```

## 阶段一：任务解析

Codex 首先从任务中提取目标、约束、范围和验收标准。如果任务只写“帮我优化”，它无法知道优化目标是性能、可读性、内存、接口稳定性还是用户体验。

### 机制重点

- 目标越抽象，Codex 越需要先进入探索模式。
- 范围越不明确，修改越容易扩散。
- 验收越模糊，最终答复越容易停留在“我已完成”。

### 例子

低质量任务：

```text
帮我重构用户模块。
```

机制级改写：

```text
请先只读分析用户模块的 controller/service/repository 边界。
不要修改文件。
输出：
1. 当前职责耦合点。
2. 可独立重构的最小切片。
3. 每个切片的风险和测试要求。
```

> **重点：** 对大型任务，第一步应是“形成可验证计划”，不是直接修改代码。

## 阶段二：上下文获取

Codex 会通过读取文件、搜索符号、查看配置、运行命令等方式建立任务上下文。上下文获取有一个隐含决策：先看哪里、读取多少、是否运行命令、是否相信已有线索。

### 上下文优先级

| 优先级 | 上下文 | 为什么重要 |
| --- | --- | --- |
| 最高 | 当前任务明确给出的文件、日志、复现步骤 | 直接限定问题边界 |
| 高 | `AGENTS.md`、构建脚本、测试配置 | 决定如何安全工作 |
| 高 | 调用链和接口边界 | 防止只修表象 |
| 中 | 相邻模块和历史测试 | 判断兼容性 |
| 低 | 大范围全仓扫描 | 只在范围不明时需要 |

### 错误上下文的后果

- 只看失败测试，不看生产代码：可能改测试期望。
- 只看当前文件，不看调用方：可能破坏 API。
- 只看错误日志最后一行：可能漏掉根因堆栈。
- 只看文档，不跑测试：可能忽略真实行为。

> **难点：** 上下文不是越多越好，关键是让 Codex 拿到“能排除错误假设”的证据。

## 阶段三：假设和计划

Codex 会基于上下文形成若干假设，例如“失败原因是缓存未清理”“类型错误来自空值”“依赖版本冲突”。好的任务设计应要求 Codex 把假设显式化。

### 推荐输出格式

```text
观察：
- 文件 A 中退出登录只清理 token。
- 文件 B 中用户名来自 localStorage。

假设：
- 刷新后显示旧用户名，是因为 logout 没清理 localStorage.user。

验证：
- 搜索所有 localStorage.user 写入点。
- 添加退出登录后刷新状态的回归测试。
```

这样做的好处是：如果假设错了，审查者可以在修改前发现，而不是等 diff 变大后才发现。

## 阶段四：工具选择

Codex 可以选择不同工具：搜索、读取文件、应用补丁、运行测试、启动服务、访问 MCP、请求浏览器截图等。工具选择决定了任务的风险和成本。

### 工具选择规则

- 能用只读工具确认的，不先写文件。
- 能用最小测试验证的，不先跑全量耗时测试。
- 需要网络、安装依赖、删除文件、读敏感配置时先审批。
- 工具失败后先解释失败原因，再选择替代路径。

### 例子

如果 `npm test` 很慢，合理步骤是：

1. 先运行目标测试文件。
2. 修复后运行目标测试。
3. 再运行相关 test suite。
4. 最后在 PR/CI 中跑全量测试。

> **易错：** 把“跑了某个命令”当作充分验证，而不检查这个命令覆盖了哪些路径。

## 阶段五：状态修改

状态修改包括改文件、生成文件、安装依赖、改变配置、创建分支等。对 Codex 来说，状态修改越多，回滚和审查成本越高。

### 最小变更原则

- 修 bug 时优先改根因所在代码，不顺手重构。
- 补测试时覆盖失败路径，不随意重写测试框架。
- 配置修改要说明影响范围。
- 删除文件要说明替代路径和回滚方式。

### 状态边界

| 状态 | 风险 |
| --- | --- |
| 工作区文件 | 可通过 diff 审查和回滚 |
| 依赖锁文件 | 可能改变大量间接依赖 |
| 数据库状态 | 可能不可逆 |
| git 历史 | 可能影响协作 |
| 生产配置 | 可能造成线上事故 |

## 阶段六：观察和迭代

工具执行后，Codex 会读取输出并决定下一步。这里最容易发生“选择性读取”：只关注通过信息，忽略 warning、跳过的测试或部分失败。

### 好的观察应包含

- 命令是否真的运行成功。
- 是否有跳过测试。
- 是否有警告。
- 失败是否和任务相关。
- 修复是否改变了原始断言。

## 阶段七：最终汇报

最终汇报不是礼貌性总结，而是审查入口。高质量汇报应包含：

- 修改文件。
- 变更意图。
- 验证命令和结果。
- 未运行的验证及原因。
- 残余风险。
- 后续建议。

## 练习

拿一个真实 bug，要求 Codex 按下面模板输出，不允许跳步：

```text
1. 任务理解
2. 已读取上下文
3. 根因假设
4. 修改计划
5. 实际修改
6. 验证结果
7. 残余风险
```

## 验收

- 能把 Codex 失败定位到生命周期中的具体阶段。
- 能要求 Codex 输出证据链。
- 能把大型任务拆成只读分析、最小修改、验证、审查四段。

## 重点

- Codex 的可靠性来自闭环，不来自单次回答。
- 每个阶段都要有可复核证据。

## 难点

- 模型可能输出看似合理的解释，但没有代码证据。深度使用 Codex 的关键是让解释和文件、命令、测试结果绑定。

## 易错

> **易错：** 只在最后问“你改了什么”，而不是在计划阶段要求证据。
>
> 正确做法：对非平凡任务，在修改前就要求 Codex 显式列出假设和验证方法。
