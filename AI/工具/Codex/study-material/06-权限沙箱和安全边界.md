# Codex学习资料：权限沙箱和安全边界

[返回索引](../Codex学习资料.md)

## 学习目标

- 理解 Codex 权限和沙箱的基本概念。
- 能判断哪些操作需要审批。
- 掌握密钥、网络、文件写入和危险命令的防护原则。

## 理论导读

Codex 能运行命令和修改文件，因此安全边界非常重要。权限不是“信不信任 Codex”的问题，而是“把自动化能力限制在任务需要的范围内”。工程上应默认最小权限，按需提升。

## 核心概念

| 概念 | 含义 | 风险 |
| --- | --- | --- |
| approval mode | 决定哪些命令需要人工确认 | 太宽会自动执行高风险操作 |
| sandbox mode | 限制文件系统、网络或命令能力 | 太窄会阻塞必要验证 |
| read-only | 只读探索 | 无法修改和运行部分验证 |
| workspace-write | 可写当前工作区 | 可能误改文件 |
| danger-full-access | 高权限运行 | 误删、泄密、执行不可信脚本风险高 |
| network access | 是否可访问外部网络 | 依赖下载、数据外传、供应链风险 |

## 审批原则

应该请求人工确认的操作：

- 安装或升级依赖。
- 删除、移动大量文件。
- 修改权限、密钥、部署配置。
- 访问外部网络。
- 执行未知脚本。
- 连接生产数据库或生产服务。
- 改变 git 历史，例如 reset、rebase、force push。

通常可自动执行的低风险操作：

- 搜索文本。
- 读取文件。
- 查看 git diff。
- 运行本地、已知、无副作用的单元测试。

## 密钥和敏感信息

Codex 任务中不要直接粘贴长期密钥、个人隐私数据、生产凭据或客户数据。需要调试认证问题时，优先使用：

- 本地 mock。
- 临时 token。
- 最小权限凭据。
- 脱敏日志。
- 明确过期时间和撤销机制。

## 例子

安全任务说明：

```text
请运行本地单元测试定位失败原因。
如果需要安装依赖、访问网络、删除文件或读取 .env，请先说明原因并等待确认。
不要输出任何密钥或 token。
```

## 练习

设计三种权限策略：

1. 只读代码审查。
2. 本地修复测试失败。
3. 云端生成 PR。

分别写出允许操作和必须审批的操作。

## 验收

- 能解释权限和沙箱的区别。
- 能为不同任务选择合理权限。
- 能识别高风险命令和敏感信息风险。

## 重点

- 最小权限原则是 Codex 使用的默认安全策略。
- 权限放宽必须和具体任务绑定，不能成为长期默认。

## 难点

- 有些测试脚本看似安全，但内部可能访问网络、删除文件或改数据库。运行前要了解脚本副作用。

## 易错

> **易错：** 把“本地仓库”误认为一定安全。
>
> 正确做法：第三方项目、未知脚本、安装钩子和依赖构建步骤都可能执行任意代码，必须谨慎授权。
