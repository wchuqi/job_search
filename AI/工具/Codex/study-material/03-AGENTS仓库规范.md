# Codex学习资料：AGENTS.md仓库规范

[返回索引](../Codex学习资料.md)

## 学习目标

- 理解 `AGENTS.md` 的作用。
- 能为仓库编写清晰的 Codex 工作规范。
- 知道规范应该写什么、不应该写什么。

## 理论导读

`AGENTS.md` 是给编码代理看的仓库说明文件。它相当于“项目入职手册”：告诉 Codex 如何构建、测试、格式化、命名、提交、处理风险，以及哪些目录有特殊规则。

如果没有这类文件，Codex 只能从现有文件和命令中推断规范。推断并非总是可靠，尤其是多语言仓库、历史包袱项目、私有构建系统和团队约定较多的项目。

## 推荐内容

```markdown
# AGENTS.md

## 项目概览
- 项目用途：
- 主要语言和框架：
- 关键目录：

## 构建和测试
- 安装依赖：
- 单元测试：
- 集成测试：
- lint/format：

## 代码规范
- 命名规则：
- 错误处理：
- 日志规范：
- API 兼容要求：

## 安全和风险
- 不要读取或输出的文件：
- 不要执行的命令：
- 涉及密钥、生产数据、删除操作时必须请求审批。

## 工作方式
- 修改前先搜索相关调用。
- 大改前先给计划。
- 最终答复必须包含修改文件和验证命令。
```

## 作用机制

Codex 在仓库中工作时，会把这类仓库说明作为长期上下文的一部分。好的 `AGENTS.md` 能减少重复说明，让不同任务保持一致风格。

可以按目录分层设置规范。例如根目录写通用规则，前端子目录写前端构建和样式规范，后端子目录写 API、数据库和测试规范。

## 例子

适合写进 `AGENTS.md`：

```markdown
## Java 后端
- 使用 JDK 21。
- 运行单测：mvn test。
- 禁止修改 public REST API，除非任务明确要求。
- 数据库迁移必须包含回滚说明。
```

不适合写进 `AGENTS.md`：

```markdown
请这次修复登录 bug。
```

单次需求应写在当前任务提示词里，不应污染长期仓库规范。

## 练习

为你当前最常用的仓库写一份 `AGENTS.md`，至少包含：

- 项目概览。
- 构建命令。
- 测试命令。
- 风格要求。
- 安全限制。
- 最终输出要求。

## 验收

- Codex 能根据 `AGENTS.md` 找到正确测试命令。
- Codex 不会随意引入新依赖或改变公共 API。
- 团队成员能读懂并维护这份规范。

## 重点

- `AGENTS.md` 应写稳定规则，不写一次性任务。
- 规则要具体到命令、路径和边界。

## 难点

- 多模块仓库需要分层规范，避免根目录规则过长或互相冲突。

## 易错

> **易错：** 在 `AGENTS.md` 中写大量抽象口号，例如“代码要优雅”“注意性能”。
>
> 正确做法：写可执行规则，例如“新增接口必须补充 controller/service 测试，不允许吞异常”。
