# Codex学习资料：CLI工作流和配置

[返回索引](../Codex学习资料.md)

## 学习目标

- 掌握 Codex CLI 的基本工作流。
- 理解配置、模型、权限、沙箱、命令执行和最终汇报。
- 能在本地仓库中安全使用 CLI 完成任务。

## 理论导读

Codex CLI 是在终端中使用 Codex 的方式。它适合处理本地仓库任务：搜索文件、读取代码、修改文件、运行测试、查看 git diff、排查失败。CLI 的优势是贴近真实开发环境，缺点是权限风险也更直接。

## 基本工作流

1. 进入目标仓库。
2. 检查当前 git 状态，确认不要覆盖他人改动。
3. 启动 Codex，并给出任务说明。
4. Codex 探索文件和命令。
5. 对需要审批的命令进行确认。
6. Codex 修改文件并运行验证。
7. 人工审查 diff 和测试结果。

## 配置维度

常见配置包括：

- 模型选择：影响推理能力、速度和成本。
- approval mode：决定哪些命令需要人工批准。
- sandbox mode：限制文件写入和网络访问。
- MCP servers：接入外部工具或文档源。
- 项目说明：通过 `AGENTS.md` 或配置提供长期上下文。

## 命令使用心智模型

Codex CLI 的命令执行可以看成三层：

| 层次 | 作用 | 示例 |
| --- | --- | --- |
| 读取层 | 收集上下文 | 搜索文件、读取源码、查看日志 |
| 修改层 | 改变工作区 | 编辑代码、更新测试、改配置 |
| 执行层 | 验证或产生副作用 | 运行测试、安装依赖、启动服务 |

越往下风险越高，越需要权限控制和人工判断。

## 例子

```text
请在当前仓库修复 npm test 中 auth.spec.ts 的失败。
限制：
- 只修改 auth 相关代码和测试。
- 不引入新依赖。
- 修改前先说明失败断言和相关调用链。
- 修改后运行最小相关测试。
```

## 练习

1. 在一个测试项目中用 CLI 让 Codex 做只读代码总结。
2. 再让它修复一个简单测试失败。
3. 审查它的 diff，要求它解释每个修改的必要性。
4. 如果测试失败，让它基于失败日志继续排查。

## 验收

- 能安全启动 Codex CLI 并限制任务范围。
- 能看懂 Codex 请求执行命令的原因。
- 能根据测试输出判断任务是否完成。

## 重点

- CLI 的价值在于能把“读代码、改代码、跑验证”连起来。
- 使用 CLI 前要确认当前工作区状态，避免覆盖未提交改动。

## 难点

- 长任务中 Codex 可能积累错误假设。遇到偏离时要要求它重新列证据，而不是继续扩大修改。

## 易错

> **易错：** 在没有看 diff 的情况下直接接受 Codex 结果。
>
> 正确做法：每次修改后查看变更范围、测试结果和残余风险。
