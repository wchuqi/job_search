## 一、工程质量和团队实践

### 1. 如何审查 Codex 生成的代码？

**参考答案：**

审查应看五类内容：变更范围是否符合任务，是否改变公共 API 或业务行为，是否有测试覆盖，是否引入安全或性能风险，是否有无关重构。还要检查 Codex 的最终汇报是否包含修改文件、验证命令、测试结果和残余风险。

> **重点：** Codex 输出是候选变更，不是自动合格代码。

### 2. Codex 生成测试通过后是否就可以合并？

**参考答案：**

不能只凭测试通过合并。测试可能覆盖不足，或 Codex 修改了测试期望而非修复根因。还需要人工审查业务语义、兼容性、安全风险、代码风格和回滚影响。

> **易错：** “测试通过”只能证明已运行测试覆盖的路径没有失败。

### 3. 团队如何规范 Codex 使用？

**参考答案：**

团队应建立 `AGENTS.md`、权限策略、PR 标注规则、测试要求、安全强审场景和失败处理流程。Codex 参与的变更应进入正常 code review 和 CI 流程，高风险任务必须人工审批。

> **重点：** 团队落地 Codex 的核心是流程化，而不是个人随意使用。

### 4. 如何衡量 Codex 落地效果？

**参考答案：**

可以看任务完成率、人工返工率、测试补充率、PR 审查发现数、CI 通过率、高风险审批次数和线上缺陷变化。指标应同时关注效率和质量，不能只看生成代码数量。

> **难点：** 速度指标容易掩盖质量问题，必须配合回归率和缺陷率。
