## 一、深度场景题

### 1. Codex 修复 bug 时产生了大量无关修改，你怎么处理？

**参考答案：**

先停止继续修改，查看当前 diff，要求 Codex 总结它的假设和每处修改的原因。然后回到只读分析模式，让它重新列出失败断言、调用链和最小修复点。无关修改应丢弃或拆成单独任务，不能混入 bug 修复 PR。

> **重点：** 任务发散时要收敛范围，而不是继续追加模糊指令。

### 2. Cloud 中 Codex 说测试通过，但本地失败，如何排查？

**参考答案：**

先比较分支、commit、依赖版本、系统环境、环境变量、网络访问和测试命令。要求 Codex 或开发者提供云端命令和日志。本地复现时应使用相同命令和依赖锁文件。如果差异来自环境，要更新文档或 CI 配置，避免结果不可复现。

> **难点：** 云端成功不等于本地成功，环境一致性是关键。

### 3. 你会让 Codex 自动升级依赖吗？

**参考答案：**

可以让 Codex 辅助升级，但不应无约束自动升级。应限定依赖范围，查看 changelog，运行测试，检查破坏性变更和安全影响。安装依赖和修改锁文件需要审批，关键依赖升级要有回滚方案。

> **易错：** 只看版本号升级成功，不看行为兼容和供应链风险。

### 4. 如何用 Codex 做安全修复？

**参考答案：**

先让 Codex 只读分析漏洞路径、输入来源、权限边界和已有测试。然后要求它写攻击用例或回归测试，再做最小修复。修复后运行安全用例和正常路径回归。涉及密钥、权限、认证、授权的修改必须人工强审。

> **重点：** 安全修复必须先证明漏洞路径，再证明修复有效。

### 5. 如何把 Codex 接入公司内部工具？

**参考答案：**

应优先通过受控工具接口，例如 MCP。工具设计要最小权限、参数明确、只读优先、输出脱敏、调用可审计。不要直接暴露任意 shell、任意 SQL 或生产写操作。生产操作应保留人工审批。

> **易错：** 工具接入越强，越要重视权限和审计，而不是只追求方便。
