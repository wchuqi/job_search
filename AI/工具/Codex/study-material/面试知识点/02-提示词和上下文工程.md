## 一、提示词和上下文工程

### 1. 如何给 Codex 写一个高质量任务？

**参考答案：**

高质量任务应包含目标、背景、范围、限制、验证和输出要求。目标说明完成后的行为，背景提供错误日志和复现步骤，范围限制文件或模块，限制说明不能破坏的兼容性和安全要求，验证指定测试命令，输出要求 Codex 汇报修改文件和残余风险。

示例：

```text
请修复 src/auth 中 token 过期后未退出登录的问题。
不要改公共 API，不引入新依赖。
请先说明调用链，再做最小修改。
修改后运行 auth 相关测试，并报告命令结果。
```

> **重点：** 验收标准是提示词的核心。

### 2. 上下文给得越多越好吗？

**参考答案：**

不是。上下文应足够完成任务，但要优先提供关键证据。过少会导致猜测，过多会稀释重点。最有价值的上下文通常是复现步骤、错误日志、相关路径、业务约束、测试命令和近期相关改动。

> **难点：** 上下文工程的目标是提高证据密度，而不是堆材料。

### 3. 为什么要让 Codex 先分析再修改？

**参考答案：**

先分析可以让 Codex 建立代码证据链，降低猜测修改的概率。特别是 bug 修复、重构和安全问题，应先要求它说明相关文件、调用链、失败断言和根因，再做最小修改。

> **易错：** 直接要求“修复”可能让 Codex 根据表面现象改测试或绕过问题。

### 4. `AGENTS.md` 和单次提示词有什么区别？

**参考答案：**

`AGENTS.md` 存放稳定的仓库级规则，例如构建命令、测试命令、代码风格、安全限制和输出要求。单次提示词描述当前任务，例如要修哪个 bug、限制哪个模块、运行哪些测试。长期规则不应混入一次性需求。

> **重点：** `AGENTS.md` 是项目手册，单次提示词是任务单。
