# Codex学习资料：团队治理可观测性和审计深度解析

[返回索引](../Codex学习资料.md)

## 学习目标

- 从团队和企业角度理解 Codex 管理问题。
- 能设计审计、遥测、权限、PR 规范和风险分级。
- 能把 Codex 使用纳入工程质量体系。

## 理论导读

个人使用 Codex 关注效率，团队使用 Codex 必须关注一致性、审计性和责任边界。否则很容易出现：每个人权限不同、提示词不同、测试要求不同、PR 中看不出哪些由 Codex 生成、出现事故后无法追踪。

治理的目标不是压制使用，而是让 Codex 的速度进入可控流程。

## 治理模型

```text
使用前：规范、权限、模板、培训
使用中：审批、日志、工具限制、上下文边界
使用后：PR 审查、CI、审计、指标、复盘
```

## 风险分级

| 等级 | 任务 | 要求 |
| --- | --- | --- |
| L1 | 只读解释、文档摘要 | 可自动，记录即可 |
| L2 | 小范围测试和代码修改 | 正常 review 和 CI |
| L3 | API、权限、安全、数据迁移 | 强制人工 owner 审查 |
| L4 | 生产发布、生产数据、密钥、账单 | Codex 只做分析，不独立执行 |

## PR 标注规范

建议 PR 模板增加：

```markdown
## Codex 使用情况
- 是否使用 Codex：
- 使用入口：CLI / IDE / Cloud / ChatGPT
- Codex 参与范围：
- 人工修改范围：
- 验证命令：
- 未验证内容：
- 风险等级：
```

这样可以让审查者知道哪些变更需要重点复核。

## 可观测性

官方文档提到 Codex 支持可选 OpenTelemetry，用于审计使用、调查问题和满足合规需求；提示词内容默认应保持脱敏或不记录。团队落地时可以记录：

- 会话开始。
- 模型和权限策略。
- 工具调用。
- 审批决策。
- 命令结果。
- 错误和重试。
- 任务耗时。

不应默认记录：

- 完整源码。
- 密钥。
- 用户隐私数据。
- 未脱敏 prompt。

## 指标体系

| 指标 | 解释 | 风险信号 |
| --- | --- | --- |
| 任务一次通过率 | Codex 首轮完成并通过验证比例 | 低说明任务拆分或上下文差 |
| 平均重试次数 | 每个任务被纠正多少次 | 高说明提示词或工具有问题 |
| 无测试变更比例 | 代码变更但无测试 | 高说明质量风险 |
| 高风险审批次数 | 安全敏感操作次数 | 异常升高需排查 |
| 回滚率 | Codex 相关变更被回滚 | 高说明审查不足 |
| 线上缺陷关联率 | 线上问题是否来自 Codex 变更 | 用于风险评估 |

## 团队模板

### 任务模板

```text
目标：
背景：
范围：
限制：
验证：
风险等级：
输出要求：
```

### 审查模板

```text
请审查这个 diff：
1. 是否有无关修改。
2. 是否改变公共 API。
3. 是否缺测试。
4. 是否有安全、性能、并发、兼容风险。
5. 给出必须修改和可选建议。
```

## 事故复盘

Codex 相关事故复盘不应停留在“模型改错了”，而应问：

- 任务是否过大？
- `AGENTS.md` 是否缺规则？
- 权限是否过宽？
- 是否跳过测试？
- 审查者是否知道 Codex 参与范围？
- PR 模板是否暴露风险？
- 是否需要工具层或 CI 层防护？

## 练习

为一个 10 人团队设计 Codex 治理方案：

- L1-L4 风险分级。
- 默认权限。
- PR 模板。
- 审批规则。
- OTel 或日志策略。
- 月度复盘指标。

## 验收

- 能区分个人效率问题和团队治理问题。
- 能制定可执行的 PR 和权限规范。
- 能设计不泄露敏感信息的可观测性方案。

## 重点

- 团队使用 Codex 必须可追踪、可审查、可回滚。
- 安全和质量规则要写进流程，而不是依赖个人习惯。

## 难点

- 遥测和审计本身也可能泄露源码和 prompt。治理方案必须同时考虑可观测性和隐私。

## 易错

> **易错：** 只统计 Codex 节省了多少时间。
>
> 正确做法：同时统计返工率、缺陷率、测试覆盖、审批风险和回滚情况。
