# Codex学习资料：Codex产品形态和适用场景

[返回索引](../Codex学习资料.md)

## 学习目标

- 了解 Codex 的主要入口。
- 能根据任务选择 CLI、IDE、Cloud 或 ChatGPT 中的 Codex。
- 知道哪些任务适合 Codex，哪些任务需要谨慎。

## 理论导读

Codex 不是单一界面，而是一组围绕软件开发的代理能力。常见形态包括本地 CLI、IDE 集成、云端任务、代码审查和 ChatGPT 中的 Codex 入口。不同入口面对的上下文、权限、运行环境和协作方式不同。

## 产品形态

| 形态 | 适合任务 | 关键特点 | 风险点 |
| --- | --- | --- | --- |
| CLI | 本地仓库修改、测试、排障、自动化 | 能直接读写当前工作区并运行命令 | 权限配置不当会误执行命令 |
| IDE | 代码阅读、局部编辑、交互式解释 | 接近开发者当前上下文 | 可能只关注打开文件，遗漏仓库级影响 |
| Cloud | 异步任务、并行探索、PR 工作流 | 可在云环境中执行较长任务 | 需要确认分支、环境变量和依赖状态 |
| ChatGPT 中的 Codex | 需求讨论、方案评审、代码解释 | 适合轻量对话和设计推演 | 不一定拥有完整仓库和执行环境 |
| Code review | PR 风险识别、测试建议 | 适合第二视角审查 | 不能替代人工业务审查 |

## 适用场景

### 高适配

- 修复局部 bug。
- 为已有代码补测试。
- 迁移小范围 API 使用。
- 解释陌生模块。
- 根据错误日志定位失败原因。
- 生成脚本、配置、文档初稿。
- 做 PR 自审和风险清单。

### 中等适配

- 跨模块重构。
- 性能优化。
- 依赖升级。
- CI/CD 流程调整。
- 复杂数据迁移脚本。

这些任务可以交给 Codex 做探索和草案，但必须拆分阶段，并设置严格验收。

### 谨慎使用

- 删除大量文件。
- 执行生产数据库操作。
- 处理密钥、凭据、个人信息。
- 自动部署到生产环境。
- 修改权限、账单、安全策略。

> **重点：** Codex 可参与高风险任务的分析和方案设计，但执行环节需要人工审批、备份和回滚计划。

## 例子

任务：修复 CI 中某个测试偶发失败。

推荐入口：

- CLI：本地复现、搜索相关测试、运行单测。
- Cloud：并行尝试两个修复方向。
- IDE：审查最终修改是否符合当前模块风格。

不推荐做法：直接让 Codex “修复所有 flaky tests”，范围过大且验收不清。

## 练习

给下面任务选择入口，并说明理由：

1. 解释一个新接手的支付模块。
2. 修复本地 1 个单元测试失败。
3. 给一个大型重构生成 3 个方案。
4. 审查一个 PR 是否遗漏权限校验。

## 验收

- 能按任务类型选择 Codex 入口。
- 能说明本地和云端任务的环境差异。
- 能识别高风险操作并要求审批。

## 重点

- Codex 的入口选择影响上下文、权限、速度和风险。
- 越接近真实代码和测试环境，越适合做实现和验证。

## 难点

- Cloud 和本地环境可能不一致，尤其是依赖、密钥、网络、系统工具和分支状态。

## 易错

> **易错：** 以为 Codex 在任何入口都能看到同样的文件和运行同样的命令。
>
> 正确做法：每次任务开始都明确工作区、分支、环境和验证命令。
