# Codex学习资料：模型选择长上下文和成本治理深度解析

[返回索引](../Codex学习资料.md)

## 学习目标

- 理解 Codex 模型选择和任务复杂度之间的关系。
- 学会管理长上下文、长任务和成本。
- 能设计“便宜任务用低成本路径，关键任务用高质量路径”的策略。

## 理论导读

Codex 的模型能力会持续更新。官方模型页显示 Codex 系列模型面向 agentic coding，支持大上下文和推理能力。学习时不要死记某个模型名，而要理解选择维度：任务复杂度、代码规模、推理深度、工具调用、上下文长度、稳定性和成本。

## 模型选择维度

| 维度 | 简单任务 | 复杂任务 |
| --- | --- | --- |
| 代码范围 | 单文件、小函数 | 多模块、跨服务 |
| 推理要求 | 明确 bug、明确测试 | 架构权衡、长链路排障 |
| 上下文 | 少量文件 | 大仓库、日志、历史背景 |
| 风险 | 低风险 | API、安全、数据、性能 |
| 验证 | 单测 | 多层测试和人工评审 |

## 任务分层策略

### 低成本路径

适合：

- 解释代码。
- 写小脚本。
- 生成测试草案。
- 格式化文档。
- 局部重命名。

策略：

- 范围小。
- 验证简单。
- 使用较低 reasoning effort。

### 高质量路径

适合：

- 跨模块 bug。
- 安全修复。
- 复杂重构。
- 性能问题。
- 生产事故复盘。

策略：

- 先只读分析。
- 要求证据链。
- 使用更强模型或更高推理设置。
- 分阶段验证。

## 长上下文管理

长上下文不是把整个仓库一次性喂给模型。更可靠的方式是分层摘要：

```text
仓库结构摘要
  -> 相关模块摘要
  -> 关键文件原文
  -> 错误日志
  -> 测试结果
  -> 约束和验收
```

### 上下文压缩原则

- 保留接口签名、调用关系、错误路径。
- 删除重复日志和无关文件。
- 区分事实与推断。
- 对长日志提取首个错误、堆栈、环境信息。
- 对大型 diff 提取文件列表和关键修改块。

## 成本治理

成本来自：

- 输入上下文 token。
- 输出 token。
- 推理 token。
- 工具调用。
- 重试和长任务。
- 自动审查或额外模型调用。

### 控制方法

- 任务前先拆分。
- 先只读定位，再写代码。
- 使用最小相关测试。
- 避免反复让 Codex 猜测。
- 对重复任务沉淀模板和 `AGENTS.md`。
- 对团队启用指标：任务时长、重试次数、审批次数、通过率。

## 例子

低成本任务：

```text
请解释 src/utils/date.ts 中 parseDate 的边界条件，不修改文件。
```

高质量任务：

```text
请只读分析订单支付成功但库存未扣减的问题。
先输出调用链、事务边界、可能竞态点和需要验证的假设。
不要修改文件。
```

## 练习

把下面任务拆成低成本探索和高质量修复两段：

```text
支付偶尔重复扣款，帮我修。
```

验收答案应包含：

- 只读证据收集。
- 幂等性分析。
- 日志和事务边界。
- 回归测试。
- 安全审查。

## 验收

- 能按任务风险选择模型和推理强度。
- 能减少不必要上下文。
- 能用任务拆分降低成本和失败率。

## 重点

- 模型越强，不代表任务可以越模糊。
- 长上下文要结构化管理，不能无差别堆材料。

## 难点

- 成本治理不是单纯省 token，而是减少无效循环、错误假设和不可审查的大 diff。

## 易错

> **易错：** 一开始就把整仓和所有日志交给 Codex。
>
> 正确做法：先让 Codex 找到关键路径，再逐步提供关键原文和验证结果。
