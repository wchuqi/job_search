# Claude Code 学习路线图

> 资料日期：2026-06-08。Claude Code 是快速演进的产品，具体命令参数、权限模式、IDE 支持和订阅能力以官方文档与本机 `claude --help`、`/help`、`claude doctor` 为准。

## 阶段 1：基础认知

- 目标：理解 Claude Code 是什么、适合解决什么开发任务，以及它和普通聊天机器人的区别。
- 需要掌握：终端式 Agent、代码库上下文、工具调用、文件编辑、命令执行、权限确认、会话状态、项目记忆。
- 例子：让 Claude Code “修复这个测试失败”时，它会先读代码、定位失败、编辑文件、运行验证；普通聊天更多停留在建议层面。
- 练习：在一个小项目里让 Claude Code 解释目录结构、找入口文件、总结测试命令。
- 验收：能说清楚 Claude Code 的核心边界：它可以操作工作区，但每一步都受权限、上下文和工具可用性限制。
- 重点：Claude Code 的本质是“在开发环境里工作的 AI 工程协作者”，不是脱离仓库的问答机器人。
- 易错：把它当成万能自动化脚本，忽略权限、成本、上下文污染和误改风险。

## 阶段 2：安装、登录和基本使用

- 目标：能完成安装、登录、项目启动、交互式会话和一次性命令。
- 需要掌握：安装渠道、账户认证、`claude` 交互模式、`claude -p` 非交互模式、`--help`、`doctor`、输出格式、恢复会话。
- 例子：在项目根目录运行 `claude` 后，让它读取 README、识别技术栈、建议第一个可验证任务。
- 练习：分别用交互模式和 `-p` 模式完成“总结最近一次提交影响”。
- 验收：能独立启动会话，知道如何退出、清理上下文、恢复历史会话和查看帮助。
- 重点：先让 Claude Code 读取项目事实，再让它改代码。
- 易错：在错误目录启动，导致上下文、权限和文件路径全部偏离。

## 阶段 3：上下文和项目记忆

- 目标：让 Claude Code 持续理解项目约定，而不是每次重复解释。
- 需要掌握：`CLAUDE.md`、用户级/项目级/子目录级记忆、导入规则、上下文窗口、压缩、清理、会话边界。
- 例子：在 `CLAUDE.md` 写明测试命令、代码风格、禁止改动目录、提交规范。
- 练习：为一个 Java 或前端项目写一份 `CLAUDE.md`，让 Claude Code 按它修复一个小 bug。
- 验收：新会话中 Claude Code 能自动遵守项目约定，并在不确定时先检查本地命令。
- 难点：把规则写得可执行，而不是写成愿望清单。
- 易错：把密钥、内部账号或临时调试信息写进长期记忆。

## 阶段 4：权限、设置和安全边界

- 目标：理解 Claude Code 何时能读写文件、执行命令、访问网络、调用 MCP 工具，以及如何收紧权限。
- 需要掌握：权限提示、允许/拒绝、配置层级、团队策略、危险模式、敏感目录、密钥保护、审计。
- 例子：允许运行 `npm test`，但不允许直接执行删除数据、上传文件或修改生产配置的命令。
- 练习：为一个项目定义允许命令清单、禁止目录清单和高风险操作确认规则。
- 验收：能解释为什么 `--dangerously-skip-permissions` 只能用于隔离容器或一次性实验环境。
- 重点：权限设计要服务于“可恢复、可审计、最小授权”。
- 易错：为了省确认，把所有工具和命令永久放行。

## 阶段 5：日常开发工作流

- 目标：把 Claude Code 用在真实开发流程中，而不是只让它回答概念问题。
- 需要掌握：代码阅读、缺陷定位、测试驱动修复、重构、文档更新、PR 摘要、代码审查、分支协作。
- 例子：给出失败测试日志，让 Claude Code 先复现、再定位、再最小改动、最后运行验证。
- 练习：完成一个“失败测试 -> 修复 -> 更新说明 -> 生成 PR 描述”的闭环。
- 验收：每次代码变更都有验证命令或明确说明未验证原因。
- 重点：要求 Claude Code 按“读事实、做计划、改小步、跑验证、汇报差异”的节奏工作。
- 难点：复杂任务要拆分，不要一次给出过宽需求。

## 阶段 6：MCP、Hooks、Skills、Subagents 和插件

- 目标：会扩展 Claude Code 的工具边界和复用能力。
- 需要掌握：MCP 服务器、外部工具/数据源、Hooks 事件、Skills 技能包、Slash Commands、Subagents、插件结构、权限隔离。
- 例子：用 MCP 接入公司文档检索，用 Hooks 在提交前自动检查格式，用 Subagent 并行做测试排查和文档整理。
- 练习：为一个项目设计“数据库只读查询 MCP + 测试前 Hook + 代码审查 Subagent”的组合方案。
- 验收：能解释每种扩展机制的职责边界：MCP 提供工具，Hooks 管事件，Skills 封装流程，Subagents 分工执行。
- 易错：所有能力都塞进一个大 prompt，导致不可测试、不可复用、不可审计。

## 阶段 7：团队化、生产治理和面试表达

- 目标：能把 Claude Code 放进团队开发规范，而不是个人玩具。
- 需要掌握：标准 `CLAUDE.md` 模板、权限策略、成本监控、日志审计、CI 集成、模型选择、版本升级、故障恢复、面试表达。
- 例子：团队约定 Claude Code 不能自动修改数据库迁移文件，必须先生成风险说明和回滚方案。
- 练习：写一份“团队 Claude Code 使用规范”，覆盖权限、密钥、代码评审、验证和事故处理。
- 验收：能在面试中讲清楚 Claude Code 如何提升效率，以及如何防止越权、误改和不可追溯变更。
- 重点：团队采用的关键不是“会不会用”，而是“能否稳定、可控、可回滚地用”。

## 阶段 8：机制深水区

- 目标：能从机制层解释 Claude Code 的执行循环、上下文、权限、工具、扩展和平台化。
- 需要掌握：Agent 状态机、上下文装载和压缩、配置层级、权限决策算法、工具路由、验证闭环、MCP schema、Hooks 生命周期、Subagents 编排、SDK 评估和可观测性。
- 例子：面试官问“Claude Code 修 bug 修错了怎么办”，不能只说“重新提示”，而要能定位是 ContextBuild 不足、PermissionCheck 缺失、Observe 误读还是 Reflect 没有停止。
- 练习：把一次失败任务复盘成状态机，并写出对应治理措施：补上下文、改权限、加 Hook、加测试或重开会话。
- 验收：能设计一套团队级 Claude Code 架构方案，并说明每个控制点的风险和指标。
- 难点：从“工具使用者”升级为“Agent 工作流设计者”。

## 推荐学习节奏

| 周期 | 学习重点 | 产出 |
| --- | --- | --- |
| 第 1 周 | 安装、CLI、基本会话、帮助命令 | 完成 5 个小任务记录 |
| 第 2 周 | `CLAUDE.md`、上下文、权限和设置 | 项目级 Claude Code 规范 |
| 第 3 周 | 修 bug、跑测试、重构、写文档 | 一个完整开发闭环 |
| 第 4 周 | MCP、Hooks、Skills、Subagents | 一个扩展能力设计方案 |
| 第 5 周 | 团队治理、成本、安全、面试题 | 使用规范、排障清单、面试答案 |
| 第 6 周 | 深度机制、SDK 平台化和团队治理 | 状态机图、权限矩阵、MCP 工具设计、评估方案 |

## 最终能力清单

- 能独立安装、登录、启动和恢复 Claude Code 会话。
- 能用 `CLAUDE.md` 管理项目上下文和长期约定。
- 能配置权限，识别高风险命令和敏感数据。
- 能让 Claude Code 完成阅读、修改、测试、解释和文档闭环。
- 能用 MCP、Hooks、Skills、Slash Commands、Subagents 扩展工作流。
- 能设计团队级使用规范、审计策略和成本控制方式。
- 能在面试中清楚说明 Agentic Coding 的价值、边界和风险。
- 能用状态机分析 Claude Code 任务失败原因。
- 能设计 MCP、Hooks、Subagents 和 SDK 的生产级边界。
