# Claude Code 学习资料：上下文装载、记忆检索和压缩深度解析

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 理解 Claude Code 的上下文不是“所有文件自动进入模型”，而是经过选择、读取、摘要和压缩。
- 能设计高质量 `CLAUDE.md` 和任务上下文。
- 能排查上下文污染、记忆冲突和压缩失真。

## 理论导读

上下文是 Claude Code 的工作内存。它包含用户消息、系统规则、项目记忆、文件片段、工具输出、MCP 返回数据、会话摘要和当前计划。上下文质量决定 Agent 的判断质量。

错误心智模型：“Claude Code 已经知道整个仓库。”

正确心智模型：“Claude Code 会根据任务和工具主动抽取仓库事实，并把有限事实放进上下文。”

## 上下文来源分层

```text
全局规则：产品/系统/安全边界
用户偏好：个人习惯、输出风格
项目记忆：CLAUDE.md、团队规则、命令
会话目标：本轮用户任务和约束
工作证据：文件内容、搜索结果、命令输出、MCP 数据
中间状态：计划、已修改文件、失败日志、验证结果
最终摘要：压缩后的任务状态
```

这些来源可能冲突。越具体、越接近当前任务、越高优先级的约束越重要。

## 装载机制：从“可能相关”到“足够相关”

Claude Code 构建上下文时常见路径：

1. 读取入口文件：README、配置、`CLAUDE.md`、包管理配置。
2. 搜索关键词：错误类名、函数名、路由、测试名。
3. 沿调用链扩展：controller -> service -> repository -> schema。
4. 读取测试和失败日志。
5. 根据新证据继续扩展或停止。

关键不是一次读完，而是逐步降低不确定性。

## `CLAUDE.md` 深度设计

### 1. 好的 `CLAUDE.md` 是约束系统

它应提供“行动规则”，例如：

```markdown
- 修改后端 Java 代码后运行 `mvn -q -pl service test`。
- 不要修改 `generated/` 和 `target/`。
- 修改数据库迁移前先生成风险和回滚方案，等待确认。
- 代码审查时优先报告 bug、安全、回归和缺失测试，不做风格建议。
```

这些规则有明确触发条件和可检查行为。

### 2. 差的 `CLAUDE.md` 是愿望清单

```markdown
- 请写高质量代码。
- 请尽量全面考虑。
- 请保持专业。
```

这些规则无法验证，也不能帮助 Agent 做选择。

### 3. 子目录记忆的作用

在 monorepo 中，不同子项目可能有不同构建命令、语言和发布规则。子目录级 `CLAUDE.md` 应写局部事实：

- 当前包的测试命令。
- 当前模块的边界。
- 特殊生成文件。
- 特殊安全规则。

> **重点：** 根目录写全局规则，子目录写局部差异；不要两边重复维护同一规则。

## 压缩机制和风险

长会话会触发摘要或压缩。压缩的目标是减少 token，但会带来信息损失。

### 压缩应保留

- 当前目标。
- 用户明确禁止事项。
- 已修改文件。
- 已运行验证和结果。
- 失败假设。
- 剩余 TODO。
- 高风险人工确认点。

### 压缩容易丢失

- 细微业务边界。
- 用户临时限定。
- 失败日志中的关键一行。
- “不要做”的约束。
- 修改前的原因链。

> **易错：** 压缩后继续让 Agent 大范围改代码。
>
> 正确做法：压缩或重开后，先让它复述目标、约束、已知事实和下一步，确认无误再继续。

## 上下文污染

上下文污染指过期、错误或无关信息影响当前判断。

常见来源：

- 前一个任务的目标残留。
- 失败修复中的错误假设。
- 旧测试输出。
- 用户改变需求但没有清理上下文。
- `CLAUDE.md` 中过期命令。

排查方式：

```text
请停止修改，只读复述：
1. 当前目标
2. 当前禁止事项
3. 你认为已验证的事实
4. 你还不确定的事实
5. 下一步需要读取哪些文件
```

## 上下文预算策略

| 任务类型 | 上下文策略 |
| --- | --- |
| 小 bug | 错误日志 + 相关测试 + 相关实现 |
| 代码审查 | 当前 diff + 涉及接口契约 + 测试结果 |
| 重构 | 当前行为测试 + 调用方 + 类型/接口定义 |
| 架构理解 | README + 目录树 + 入口 + 配置 + 少量核心文件 |
| 迁移升级 | 版本说明 + 依赖配置 + 编译错误 + 迁移指南 |

## 例子：上下文最小化提示

```text
请用最小上下文定位这个测试失败：
1. 只读取失败测试、错误栈直接指向的文件、相关接口定义
2. 不要全仓库搜索，除非你说明为什么需要
3. 每读取一个文件后说明它回答了哪个问题
4. 修改前列出仍不确定的事实
```

## 练习

1. 把一个长任务压缩成 10 行状态摘要，检查是否保留禁止事项。
2. 为 monorepo 设计根目录和子目录两级 `CLAUDE.md`。
3. 故意让 Claude Code 带着旧上下文做新任务，观察污染表现。

## 验收

- 能列出 Claude Code 的上下文来源。
- 能设计可执行的 `CLAUDE.md`。
- 能识别压缩失真和上下文污染。
- 能用最小上下文完成任务。

## 重点

- 上下文不是越多越好，而是越准越好。
- 长期规则放记忆，临时约束放当前任务。
- 压缩后必须校验目标和禁止事项。

## 难点

- 真正困难的是判断“还缺哪条事实”，而不是把所有文件塞进去。

## 易错

- **易错：** 让 Agent 在没读测试的情况下修 bug。
  正确做法：失败测试是最重要的上下文之一。
- **易错：** 把安全密钥写进长期记忆。
  正确做法：长期记忆只写变量名、边界和获取方式，不写值。
