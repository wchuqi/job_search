# Claude Code 学习资料：Skills、Subagents、插件和并行编排深度解析

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 理解复用能力和并行能力的底层边界。
- 能设计 Skill、Slash Command、Subagent 和插件的职责分工。
- 能避免“多个 Agent 同时忙但结果不可整合”的伪并行。

## 理论导读

Claude Code 的扩展机制可以分成两类：

```text
复用机制：Slash Commands、Skills、插件
并行机制：Subagents、后台会话
```

复用解决“同类任务重复做”的问题；并行解决“复杂任务可拆分调查”的问题。它们经常组合使用，例如一个 `/review` 命令启动安全、测试、架构三个子 Agent，再由主 Agent 汇总。

## 复用机制分层

| 层级 | 粒度 | 典型内容 |
| --- | --- | --- |
| Slash Command | 单个入口 | `/review`、`/pr-summary`、`/test-plan` |
| Skill | 一类任务方法 | 学习资料生成、事故复盘、安全审计 |
| Plugin | 一组能力分发 | 团队开发插件、安全插件、后端插件 |

## Skill 深度设计

一个高质量 Skill 应包含：

- 触发条件：什么时候必须使用。
- 输入要求：需要哪些上下文。
- 工作流：分几步做。
- 输出标准：格式、深度、验收。
- 工具建议：优先读什么、跑什么、查什么。
- 禁止事项：不能做什么。
- 失败处理：信息不足、权限不足、结果冲突怎么办。

### Skill 反例

```text
当用户让你写文档时，写得详细一点。
```

问题：没有输入、步骤、质量标准和边界。

### Skill 正例

```text
当用户请求代码审查时：
1. 只读检查当前 diff
2. 优先 bug、安全、回归、缺失测试
3. 每条发现必须有文件引用和影响说明
4. 不做纯风格建议
5. 输出按严重程度排序
```

## Subagent 编排模型

```text
主 Agent
  -> 定义任务边界
  -> 拆分子任务
  -> 分配上下文
  -> 收集结构化结果
  -> 冲突消解
  -> 决定最终行动
```

子 Agent 的价值在于“专项视角”，不是“更多 token”。每个子 Agent 都应有明确角色：

- Security Reviewer：认证、授权、输入校验、密钥。
- Test Analyst：覆盖率、边界测试、回归风险。
- Migration Analyst：版本兼容、配置变化、破坏性变更。
- Docs Verifier：文档是否和代码一致。

## 并行适用性判断

适合并行：

- 多个独立模块调查。
- 多个审查维度。
- 读取大量资料后汇总。
- 互不修改文件的只读分析。

不适合并行：

- 同一文件修改。
- 强顺序依赖任务。
- 高风险命令执行。
- 目标尚未明确。

> **重点：** 并行读，串行改。并行审查，集中决策。

## 输出契约

子 Agent 输出必须结构化，否则主 Agent 难以整合。

```markdown
## 结论

## 证据

## 文件/来源

## 风险等级

## 建议动作

## 不确定事项
```

## 冲突消解

常见冲突：

- 安全 Agent 要求收紧权限，产品 Agent 要求兼容旧行为。
- 测试 Agent 认为缺测试，架构 Agent 认为不应扩大范围。
- 两个 Agent 对根因判断不同。

消解规则：

1. 证据优先于猜测。
2. 高风险安全问题优先于便利性。
3. 可验证方案优先于口头判断。
4. 修改范围小的方案优先。
5. 冲突无法消解时请求用户确认。

## 插件治理

插件是团队级资产，必须治理：

- manifest/版本。
- 包含哪些 commands、skills、hooks、MCP 配置。
- 权限声明。
- 变更日志。
- 负责人。
- 兼容 Claude Code 版本范围。
- 安全审查。

> **易错：** 插件一安装就放开高权限工具。
>
> 正确做法：插件默认低权限，高权限功能显式启用。

## 例子：多 Agent 代码审查

```text
主任务：审查支付模块 PR。

子 Agent A 安全审查：
  只检查认证、授权、签名、密钥、输入校验。

子 Agent B 测试审查：
  检查新增行为是否有单测、集成测试和回归测试。

子 Agent C 兼容性审查：
  检查接口、数据库 schema、消息格式是否破坏旧版本。

主 Agent：
  合并 findings，按严重程度输出，不自动修改。
```

## 练习

1. 为 `/review` 设计一个使用 3 个 Subagents 的流程。
2. 给每个 Subagent 写输入、禁止事项和输出契约。
3. 设计一个插件发布 checklist。

## 验收

- 能区分复用机制和并行机制。
- 能设计子 Agent 的职责边界。
- 能处理多 Agent 结论冲突。
- 能说明插件治理要求。

## 重点

- 复用机制解决一致性，并行机制解决复杂度。
- 主 Agent 必须负责整合和最终决策。
- 插件必须版本化和权限声明。

## 易错

- **易错：** 子 Agent 输出自由散文。
  正确做法：输出结构化证据和建议。
- **易错：** 为了并行而并行。
  正确做法：只有任务独立且结果可合并时才并行。
