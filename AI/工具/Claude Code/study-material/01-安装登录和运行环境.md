# Claude Code 学习资料：安装、登录和运行环境

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 能按官方文档完成安装、登录、升级和健康检查。
- 能确认 Claude Code 当前版本、平台支持、认证状态和项目根目录。
- 能区分个人使用、团队使用、CI/自动化使用的环境要求。

## 理论导读

Claude Code 的运行环境由本机 CLI、账号认证、项目目录、可用工具链和权限设置组成。安装成功只代表 CLI 可启动，不代表它能正确完成项目任务。真正可用还需要：项目依赖能安装、测试命令能跑、Git 状态可读、编辑器或 IDE 集成正常、外部工具或 MCP 服务可访问。

## 核心心智模型

把安装理解为三次检查：

1. CLI 检查：`claude --version`、`claude --help` 能正常输出。
2. 账号检查：登录方式、订阅或 API 权限可用。
3. 项目检查：在正确目录启动，依赖、测试、构建、Git 状态可用。

## 知识点详解

### 1. 安装和升级

官方安装方式可能随版本调整，优先使用官方文档给出的命令。常见方式包括包管理器安装、官方安装脚本或平台特定安装方式。安装后立即执行：

```bash
claude --version
claude --help
claude doctor
```

`doctor` 类命令用于检查环境问题，例如认证、版本、路径、平台支持、IDE 连接或配置问题。具体检查项以当前版本输出为准。

> **重点：** 不要把网上旧教程中的安装命令永久写入团队文档，团队文档应链接官方安装页并记录本团队验证过的版本。

### 2. 登录和认证

Claude Code 通常需要与 Claude 账号、控制台、订阅或 API 访问权限关联。个人开发机可以使用交互式登录；自动化环境更适合使用受控凭据和环境变量。

认证设计要考虑：

- 凭据不能写入仓库。
- 团队共享机器不能混用个人账号。
- CI 中的凭据必须有最小权限和过期机制。
- 日志中不能打印 token、cookie、内部 URL 参数。

### 3. 项目根目录

Claude Code 会基于当前工作目录理解项目。错误目录会带来两个问题：看不到正确文件，或者权限范围大于预期。

推荐启动前检查：

```bash
pwd
git status --short
ls
```

Windows PowerShell 对应：

```powershell
Get-Location
git status --short
Get-ChildItem
```

### 4. 本地工具链

Claude Code 能运行测试，但前提是本机已有对应工具链。例如 Node、Python、Java、Go、Docker、数据库客户端、包管理器、浏览器驱动。它不能凭空绕过环境缺失。

> **易错：** 让 Claude Code 调试测试失败，但项目依赖根本没有安装。
>
> 正确做法：先让它识别项目技术栈和安装命令，再确认是否允许安装依赖。

## 例子

### 新项目启动检查

```text
请只读检查当前项目：
1. 识别技术栈和包管理器
2. 找到启动、测试、构建命令
3. 检查是否有 CLAUDE.md
4. 不要修改文件
```

### 环境排障请求

```text
我运行测试失败，先不要改代码。
请根据错误日志判断是代码问题、依赖问题、环境变量问题还是平台路径问题。
```

## 练习

1. 安装 Claude Code 后记录版本、平台、认证方式、项目检查命令。
2. 在一个项目根目录启动 Claude Code，让它生成本项目的“开发环境检查清单”。
3. 故意在上级目录启动，再让它解释可能的风险。

## 验收

- 能在新机器上完成安装和健康检查。
- 能确认当前账号和项目目录是否正确。
- 能判断失败来自 Claude Code、项目依赖、权限还是平台环境。

## 重点

- 安装成功不等于项目可用。
- 在正确目录启动比“提示词写得漂亮”更重要。
- 团队环境必须记录验证过的版本和最小权限认证方式。

## 难点

- 区分 CLI 自身问题、模型权限问题、本地项目问题和外部服务问题。

## 易错

- **易错：** 把密钥写进 `CLAUDE.md` 方便模型使用。
  正确做法：用环境变量、密钥管理器或 MCP 服务隔离，文档只写变量名和使用边界。
- **易错：** 在 monorepo 根目录直接让 Claude Code 全局搜索和修改。
  正确做法：先限定包、目录、模块和验收命令。
