# Claude Code 学习资料：权限模式、设置层级和安全边界

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 理解 Claude Code 为什么需要权限系统。
- 能配置文件读写、命令执行、网络访问、MCP 工具等边界。
- 能设计个人、项目和团队级的安全策略。

## 理论导读

Claude Code 运行在你的开发环境里，能接触真实文件、真实命令和真实凭据。因此权限不是形式问题，而是工程安全的核心。权限系统要解决三个问题：

1. 哪些操作允许自动执行？
2. 哪些操作必须人工确认？
3. 哪些操作永远禁止？

## 核心心智模型

把权限分成四圈：

```text
只读分析 -> 可恢复修改 -> 受控命令 -> 高风险操作
```

- 只读分析：读文件、搜索代码、查看 Git diff。
- 可恢复修改：编辑代码、文档、测试，可通过 Git 回滚。
- 受控命令：测试、构建、lint、格式化。
- 高风险操作：删除、发布、上传、数据库写入、密钥操作、修改权限、绕过确认。

## 知识点详解

### 1. 设置层级

Claude Code 通常有用户级、项目级、工作区级或企业策略等配置层。具体文件名和优先级以官方 settings 文档和当前版本为准。设计时应遵循：

- 用户级：个人偏好，例如默认模型、编辑器、界面习惯。
- 项目级：项目命令、权限建议、工具启用范围。
- 团队级：安全策略、禁用项、审计要求。
- 临时会话：本轮任务的额外限制。

> **重点：** 团队安全策略不要只写在 prompt 中，应该进入可审计的配置和流程。

### 2. 允许清单和拒绝清单

权限设计优先使用允许清单。例如允许：

```text
npm test
npm run lint
mvn test
pytest
git status
git diff
```

高风险命令应默认拒绝或需要确认：

```text
rm -rf
git reset --hard
git push --force
kubectl apply
terraform apply
psql -c "DELETE ..."
```

Windows 环境还要注意 PowerShell 的递归删除、移动、脚本执行策略和路径通配符。

### 3. 危险模式

Claude Code 可能提供跳过权限确认的模式。这类模式只能用于隔离、可丢弃、无密钥、无生产连接的环境，例如临时容器或本地实验仓库。

> **易错：** 为了提高效率，在日常开发机上长期启用跳过权限确认。
>
> 正确做法：只在隔离环境中短时使用，并确保没有生产凭据、SSH key、云账号、内部网络访问。

### 4. 密钥和敏感数据

敏感数据保护规则：

- 不把密钥写入 prompt、`CLAUDE.md`、日志和提交。
- 不让 Claude Code 打印完整 token。
- 只给只读或最小权限凭据。
- 外部 MCP 服务要隔离权限。
- 输出报告前检查是否包含隐私数据。

## 例子

### 项目权限说明

```markdown
## Claude Code 权限建议

- 允许自动运行：`npm test`、`npm run lint`、`git diff --stat`
- 修改文件前必须说明计划。
- 禁止自动运行：部署命令、数据库写入、删除目录、修改 `.env`
- 如需安装依赖，先说明包名、版本、理由和影响。
```

### 高风险请求改写

不建议：

```text
直接帮我清理所有没用文件。
```

建议：

```text
只读列出疑似无用文件，按证据分类。
不要删除。给出每个文件为什么可能可删、删除风险和验证方法。
```

## 练习

1. 为一个项目列出 10 条允许命令和 10 条禁止命令。
2. 设计一个“数据库只读查询”的 MCP 权限边界。
3. 让 Claude Code 审查当前仓库是否存在密钥泄漏风险。

## 验收

- 能区分只读、可恢复修改、受控命令和高风险操作。
- 能解释设置层级和团队策略的关系。
- 能写出项目级权限说明。

## 重点

- 最小权限、可恢复、可审计。
- 高风险操作必须人工确认。
- 跳过权限确认不是效率开关，而是隔离实验选项。

## 难点

- 自动化越强，权限边界越重要；否则效率提升会转换成事故风险。

## 易错

- **易错：** 只在自然语言里要求“不要做危险操作”。
  正确做法：结合配置、允许清单、拒绝清单、代码审查和 CI 约束。
- **易错：** 把测试命令和部署命令放在同一自动允许级别。
  正确做法：测试可自动，部署必须进入发布流程。
