# Claude Code 学习资料：配置层级、权限决策和工具路由深度解析

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 理解 Claude Code 配置不是单一文件，而是多层规则合成。
- 能设计权限决策流程，避免危险命令和越权工具。
- 能解释工具路由：为什么某个任务该读文件、运行命令、调 MCP 或请求用户。

## 理论导读

Claude Code 的每个动作都可以看成一次决策：

```text
任务意图 -> 候选工具 -> 配置约束 -> 权限判断 -> 执行或拒绝 -> 记录结果
```

配置层级决定“规则从哪里来”，权限决策决定“能不能做”，工具路由决定“用什么做”。

## 配置层级模型

常见配置来源可抽象为：

```text
产品默认规则
  < 用户级设置
  < 项目级设置
  < 团队/企业策略
  < 当前会话显式指令
```

具体优先级以当前 Claude Code 官方 settings 文档为准，但工程原则稳定：越高风险的限制越不应被低层随意覆盖；越具体的任务约束越应被显式遵守。

## 配置内容分类

| 配置类别 | 例子 | 风险 |
| --- | --- | --- |
| 模型和输出 | 默认模型、输出格式 | 成本、稳定性 |
| 文件范围 | 允许目录、忽略目录 | 误读、误改 |
| 命令权限 | 允许测试命令、拒绝删除命令 | 本机破坏、生产事故 |
| MCP | server、工具、环境变量 | 外部系统越权 |
| Hooks | 事件、匹配、脚本 | 误阻断、隐藏副作用 |
| 记忆 | `CLAUDE.md`、规则导入 | 上下文污染 |

## 权限决策算法

一个实用权限决策流程：

```text
1. 识别动作类型
   - 读文件、写文件、执行命令、网络访问、MCP 调用

2. 识别目标对象
   - 文件路径、命令字符串、MCP 工具名、外部系统

3. 计算风险等级
   - 只读 / 可恢复修改 / 本机副作用 / 外部副作用 / 生产高风险

4. 匹配配置
   - 是否命中禁止清单？
   - 是否命中允许清单？
   - 是否需要人工确认？

5. 检查当前任务约束
   - 用户是否说“只读”？
   - 是否限制目录？
   - 是否禁止安装依赖？

6. 执行决策
   - 允许、拒绝、询问、降级为只读、提出替代方案
```

## 风险等级表

| 等级 | 动作 | 默认策略 |
| --- | --- | --- |
| L0 | 读取仓库文件、`git diff` | 可允许 |
| L1 | 编辑代码和测试 | 需任务相关，可恢复 |
| L2 | 运行测试、lint、构建 | 可允许但要限制命令 |
| L3 | 安装依赖、生成文件、修改配置 | 需说明影响 |
| L4 | 删除、重命名、强制 Git 操作 | 需人工确认 |
| L5 | 部署、数据库写入、密钥操作 | 默认禁止或受控流水线 |

> **重点：** 风险来自“动作 + 对象 + 环境”，不是只看命令名。

## 工具路由策略

Claude Code 面对任务时有多个工具选择：

| 任务 | 首选工具 | 不推荐 |
| --- | --- | --- |
| 找函数定义 | 文件搜索/读取 | 让模型凭记忆猜 |
| 修测试失败 | 读日志、读测试、运行单测 | 直接全量重构 |
| 查业务规则 | 文档 MCP、代码引用 | 凭常识补业务 |
| 查询生产状态 | 只读 MCP/监控 | 直接运行生产 CLI |
| 格式化代码 | 项目格式化命令 | 手工大面积改格式 |
| 不确定权限 | 请求用户确认 | 尝试绕过 |

工具路由的关键问题是：“哪个工具能以最小风险获得最多证据？”

## 允许清单设计

### 好的允许清单

```text
git status
git diff
git log --oneline -n 20
npm test
npm run lint
pytest
mvn test
```

### 脆弱允许清单

```text
npm *
git *
powershell *
bash *
```

脆弱清单看似方便，但实际放开了发布、删除、重写历史等危险动作。

## 命令风险识别

高风险特征：

- 递归删除：`rm -rf`、PowerShell `Remove-Item -Recurse -Force`。
- Git 破坏：`reset --hard`、`clean -fd`、`push --force`。
- 部署：`deploy`、`release`、`kubectl apply`、`terraform apply`。
- 数据写入：`DELETE`、`UPDATE`、`DROP`、迁移执行。
- 网络上传：`curl -X POST`、上传脚本。
- 权限改变：`chmod 777`、IAM、云权限。

## 例子：请求降级

用户请求：

```text
帮我清理所有无用文件。
```

安全改写：

```text
我先只读列出疑似无用文件，并按证据分类：
1. 构建产物
2. 缓存
3. 未引用源码
4. 风险不明
不会删除。每项给出验证方法。
```

## 练习

1. 为你的项目设计 L0-L5 权限分级。
2. 把 20 条常用命令分到允许、确认、禁止三类。
3. 设计一个 Hook，在命令执行前按上述算法输出风险等级。

## 验收

- 能解释配置层级、权限决策和工具路由的关系。
- 能设计允许清单和禁止清单。
- 能把高风险请求降级为只读分析。

## 重点

- 工具越强，默认权限越要窄。
- 允许清单应精确到命令意图，而不是放开整个命令家族。
- 用户本轮“只读”约束应覆盖项目默认行为。

## 难点

- 权限判断需要理解上下文意图。例如 `npm run build` 通常低风险，`npm run deploy` 可能是生产发布。

## 易错

- **易错：** 只用自然语言说“谨慎操作”。
  正确做法：用配置、Hook、审计和人工确认共同约束。
- **易错：** 团队成员各自维护权限。
  正确做法：把项目级权限策略版本化，进入代码审查。
