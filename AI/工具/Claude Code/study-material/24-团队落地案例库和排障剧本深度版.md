# Claude Code 学习资料：团队落地案例库和排障剧本深度版

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 掌握团队引入 Claude Code 的分阶段方法。
- 能把安全、成本、质量和协作纳入治理。
- 能用排障剧本处理常见失败和事故。

## 落地阶段

### 阶段 1：个人试点

目标：验证适合本团队的任务类型。

允许：

- 只读代码解释。
- PR 摘要。
- 测试补全。
- 小 bug 修复。

禁止：

- 自动部署。
- 生产数据库操作。
- 批量改全仓。

产出：

- 任务记录。
- 成功/失败样例。
- 初版 `CLAUDE.md`。

### 阶段 2：小组规范

目标：把个人经验变成小组规则。

动作：

- 统一 `CLAUDE.md`。
- 建立允许命令和禁止命令。
- PR 模板增加 AI 使用记录。
- 定义 review 要求。
- 收集成本和返工数据。

### 阶段 3：团队治理

目标：让 Claude Code 进入稳定研发流程。

动作：

- 标准化 Skills/Commands。
- 接入只读 MCP。
- 添加 Hooks 审计和风险阻断。
- 建立评估集。
- 规定高风险人工确认。

### 阶段 4：平台化

目标：把稳定流程产品化。

动作：

- SDK 应用。
- 统一策略引擎。
- 统一审计日志。
- 在线指标。
- 人工反馈闭环。

## 案例 1：CI 失败排查

### 场景

PR 中 CI 单测失败，失败日志较长。

### 推荐流程

```text
1. 只读读取失败日志摘要。
2. 定位失败测试和最近相关 diff。
3. 判断是代码、测试、环境还是依赖问题。
4. 修改前给出根因假设。
5. 运行最小相关测试。
6. 输出修复说明和验证结果。
```

### 验收

- 引用失败测试。
- 修改范围最小。
- 没有无关格式化。
- 不声称未运行测试通过。

## 案例 2：依赖升级

### 风险

- API 破坏。
- lockfile 大面积变化。
- 构建脚本变化。
- 隐式安全策略变化。

### 推荐流程

```text
1. 读取升级前后版本说明。
2. 生成破坏性变更清单。
3. 先升级单个包或模块。
4. 运行编译和测试。
5. 记录迁移步骤。
```

### 易错

直接让 Claude Code “升级到最新版”，导致多个依赖同时变化，回归范围不可控。

## 案例 3：数据库迁移

### 默认策略

Claude Code 可以生成迁移计划和草稿，但不应自动执行生产迁移。

### 必要内容

- schema 变化。
- 数据回填。
- 兼容旧代码。
- 回滚方案。
- 锁表和耗时风险。
- 灰度和监控。

## 案例 4：安全修复

### 推荐流程

- 只读确认漏洞路径。
- 查认证、授权、输入校验。
- 补回归测试。
- 检查类似模式。
- 生成安全说明。

### 禁止

- 打印真实 token。
- 把漏洞利用细节写进公开 PR。
- 跳过人工安全 review。

## 排障剧本

### 剧本 1：Claude Code 找错文件

处理：

```text
停止修改 -> 要求列出证据 -> 重新读取调用链 -> 限定文件范围 -> 再计划
```

根因：

- 上下文不足。
- 关键词搜索误命中。
- monorepo 目录错误。

### 剧本 2：连续修复失败

处理：

```text
停止修改 -> 保存 diff -> 复盘假设 -> 回滚本次改动 -> 缩小复现 -> 重新开始
```

重点：不要继续叠加补丁。

### 剧本 3：误改用户文件

处理：

```text
检查 git diff -> 区分用户原有改动和 AI 改动 -> 局部恢复 -> 更新权限规则
```

禁止：盲目 `git reset --hard`。

### 剧本 4：MCP 返回错误数据

处理：

```text
检查查询参数 -> 检查环境和租户 -> 查看 source_url -> 缩小范围 -> 修复 MCP 工具
```

### 剧本 5：成本异常

处理：

```text
查看任务类型 -> 检查上下文大小 -> 检查并行 Agent 数量 -> 设置预算 -> 拆小任务
```

## 团队评审清单

| 项目 | 检查问题 |
| --- | --- |
| 上下文 | 是否有 `CLAUDE.md`？是否过期？ |
| 权限 | 是否有允许/禁止命令？ |
| 安全 | 是否禁止密钥和生产写操作？ |
| 验证 | 是否有最小和全量测试命令？ |
| 审计 | 是否记录 AI 使用范围和结果？ |
| 成本 | 是否有预算和指标？ |
| 反馈 | 是否记录采纳、误报和返工？ |

## 练习

1. 选一个真实 PR，按案例 1 让 Claude Code 排查 CI 失败。
2. 设计团队 Claude Code 试点计划，包含 4 周目标。
3. 写一份事故复盘：AI 误改、发现、恢复、预防。

## 验收

- 能分阶段推进 Claude Code 团队落地。
- 能处理常见失败剧本。
- 能建立团队级评审清单。

## 重点

- 团队落地是流程变革，不是工具安装。
- 先只读和低风险任务，再扩展到受控修改。
- 每次失败都要沉淀为规则、Hook、评估样例或文档。

## 易错

- **易错：** 一开始就平台化。
  正确做法：先从真实任务试点，稳定后再 SDK 化。
- **易错：** 只追求节省时间。
  正确做法：同时统计质量、安全、返工和事故。
