# Claude Code 学习资料：文件编辑、命令执行和验证闭环深度解析

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 理解 Claude Code 修改代码时的风险点不在“能否生成代码”，而在编辑范围、状态一致性和验证。
- 能设计小步可恢复的文件编辑流程。
- 能把测试、构建、lint、人工检查组合成可靠验收链。

## 理论导读

代码修改是外部世界状态变化。Claude Code 每次写文件都可能引入以下风险：

- 修改了不相关文件。
- 覆盖了用户未提交改动。
- 引入格式化噪声。
- 修复表象但破坏业务语义。
- 测试命令选错，导致假通过。
- 命令执行依赖错误环境。

因此，文件编辑和命令执行必须绑定验证闭环。

## 编辑前状态检查

在动手前应检查：

```text
1. 当前 Git 状态是否干净？
2. 是否存在用户未提交改动？
3. 任务涉及哪些文件？
4. 哪些文件禁止修改？
5. 修改失败后如何恢复？
```

> **重点：** Claude Code 不应默认回滚或覆盖用户已有改动。任何回滚都必须只针对本次任务的改动，并经过确认。

## 文件编辑策略

### 1. 最小修改

最小修改不是代码行数最少，而是行为影响范围最小。比如：

- 只改失败路径，不顺手重构成功路径。
- 只补对应测试，不改整个测试框架。
- 不把格式化和逻辑修改混在一起。

### 2. 结构化 diff

修改后应检查：

```bash
git diff --stat
git diff
```

关注：

- 是否有无关文件。
- 是否有格式化大面积变化。
- 是否改了生成文件。
- 是否改了锁文件、配置、迁移文件。

### 3. 并发修改处理

如果用户在同一文件里有未提交改动，Claude Code 应：

- 先读取当前内容。
- 只在必要位置编辑。
- 不使用覆盖式写入。
- 修改后检查 diff，确认没有移除用户代码。

## 命令执行机制

命令执行分为四类：

| 类型 | 例子 | 使用策略 |
| --- | --- | --- |
| 只读诊断 | `git diff`、`ls`、`rg` | 可频繁使用 |
| 验证命令 | `npm test`、`pytest` | 修改后必须优先 |
| 环境变更 | `npm install`、数据库迁移 | 需说明影响 |
| 高风险命令 | 删除、部署、强推 | 默认禁止 |

## 验证闭环

可靠验证通常分三层：

```text
最小验证 -> 邻近验证 -> 全量验证
```

- 最小验证：失败测试、单文件测试、局部编译。
- 邻近验证：模块测试、相关集成测试。
- 全量验证：全项目测试、构建、lint、E2E。

并不是每次都必须全量验证，但必须说明为什么当前验证足够，或说明未验证风险。

## 假通过和假失败

### 假通过

测试通过但问题未解决，常见原因：

- 跑错测试。
- 测试没有覆盖业务边界。
- mock 掩盖真实依赖。
- 环境和生产差异大。
- 只验证编译，没有验证行为。

### 假失败

代码正确但测试失败，常见原因：

- 环境变量缺失。
- 依赖未安装。
- 缓存或端口占用。
- 平台路径差异。
- 测试顺序依赖。

> **难点：** 面试中不能简单说“跑测试通过了”。要能说明测试覆盖了哪个风险，没有覆盖哪个风险。

## 例子：修复流程模板

```text
请按以下流程修复：
1. 只读检查 git status，确认是否有用户改动。
2. 读取失败测试和相关实现。
3. 说明根因假设和计划。
4. 做最小修改。
5. 运行最小相关测试。
6. 检查 git diff，确认无关文件。
7. 汇报：改动、验证、未覆盖风险。
```

## 回滚策略

回滚不是简单 `git reset --hard`。安全回滚流程：

1. 确认哪些改动是 Claude Code 本次产生的。
2. 确认是否混有用户改动。
3. 优先使用手工反向补丁或局部 checkout。
4. 回滚后运行最小验证。
5. 记录失败原因，避免重复。

## 练习

1. 让 Claude Code 修复一个测试，要求每次编辑后说明 diff 范围。
2. 设计“最小验证、邻近验证、全量验证”三层命令表。
3. 故意制造一个假通过场景，让 Claude Code 发现测试覆盖不足。

## 验收

- 能解释文件编辑前为什么要检查 Git 状态。
- 能区分最小验证、邻近验证和全量验证。
- 能识别假通过和假失败。
- 能安全回滚 AI 改动而不丢失用户改动。

## 重点

- 文件编辑必须和验证绑定。
- 不要混合格式化、重构和行为修改。
- 回滚必须保护用户已有改动。

## 易错

- **易错：** 看到测试通过就认为任务完成。
  正确做法：检查测试是否覆盖目标行为，再检查 diff 是否合理。
- **易错：** 用全量格式化掩盖小逻辑修改。
  正确做法：格式化和逻辑改动分开提交或至少分开说明。
