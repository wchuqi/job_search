# Claude Code 学习资料：MCP 外部工具和数据源集成

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 理解 MCP 在 Claude Code 中的作用。
- 能设计只读、安全、可审计的外部工具接入方式。
- 能区分 MCP 与普通命令、Hooks、Skills、Subagents 的边界。

## 理论导读

MCP 即 Model Context Protocol，用来把外部工具和数据源以统一协议暴露给 AI 应用。对 Claude Code 来说，MCP 可以接入文档检索、数据库查询、Issue 系统、设计稿、内部 API、浏览器自动化、日志平台等能力。

MCP 的价值是把“模型不知道、终端不方便查、需要统一权限控制”的信息变成可调用工具。

## 核心心智模型

```text
Claude Code -> MCP 客户端 -> MCP 服务器 -> 外部系统
```

Claude Code 不应该直接拿到所有外部系统权限。MCP 服务器应该提供窄接口，例如“按 issue id 读取标题和评论”“只读查询最近 100 条日志”“检索文档片段”，而不是暴露完整管理员 API。

## 知识点详解

### 1. MCP 适用场景

- 检索公司内部文档。
- 查询 Issue、PR、CI 状态。
- 读取只读数据库。
- 调用测试环境 API。
- 获取日志、指标、告警。
- 访问设计系统或接口契约。

### 2. 工具设计原则

好的 MCP 工具应该：

- 参数清晰，有 schema。
- 默认只读。
- 返回结构化数据。
- 限制数据量和查询范围。
- 不返回密钥、隐私和大段无关日志。
- 有审计日志。
- 错误信息可诊断但不泄漏敏感细节。

> **重点：** MCP 服务器是权限边界，不是让 Claude Code 绕过权限边界。

### 3. 与命令行工具的区别

命令行适合本地项目操作，例如运行测试、搜索文件。MCP 适合受控访问外部系统。不要为了省事让 Claude Code 直接执行带生产凭据的 CLI 命令。

### 4. MCP 配置和排障

排障步骤：

1. 确认 MCP server 能独立启动。
2. 确认 Claude Code 配置能找到 server。
3. 确认工具列表可见。
4. 用最小参数调用一次只读工具。
5. 检查权限、路径、环境变量、网络和日志。

具体配置文件、传输方式和命令以官方 MCP 文档及当前版本为准。

## 例子

### 只读数据库 MCP 工具设计

```json
{
  "name": "query_orders_readonly",
  "description": "只读查询测试环境订单，最多返回 50 行",
  "input": {
    "order_id": "可选订单 ID",
    "created_after": "ISO 日期，最多查询最近 30 天"
  },
  "guards": [
    "禁止 INSERT/UPDATE/DELETE/DDL",
    "限制返回字段，不返回手机号和地址",
    "记录调用人、参数和行数"
  ]
}
```

### 文档检索 MCP

```text
请用公司文档 MCP 检索“退款状态机”，只根据检索到的文档回答。
如果资料不足，请列出缺口，不要凭经验补全。
```

## 练习

1. 为项目设计 3 个只读 MCP 工具：文档检索、Issue 查询、CI 状态查询。
2. 给每个工具写参数、返回结构、权限限制和失败处理。
3. 让 Claude Code 使用 MCP 信息修复一个需求理解问题，但禁止访问生产写接口。

## 验收

- 能解释 MCP 的价值和边界。
- 能设计只读、结构化、有审计的 MCP 工具。
- 能排查 MCP server 不可见、调用失败、返回过大等问题。

## 重点

- MCP 是外部能力接入层。
- 工具越强，权限越要窄。
- 返回数据要少、准、可追溯。

## 易错

- **易错：** MCP 工具返回整张表、完整日志或原始密钥。
  正确做法：限制字段、时间范围、行数和脱敏策略。
- **易错：** 把 MCP 当成脚本执行器。
  正确做法：MCP 暴露业务安全接口，脚本执行仍受本地权限控制。
