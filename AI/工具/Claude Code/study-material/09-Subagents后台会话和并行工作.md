# Claude Code 学习资料：Subagents、后台会话和并行工作

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 理解 Subagents 如何分解复杂任务。
- 能设计不同 Agent 的职责、上下文和输出契约。
- 能避免并行工作带来的冲突、重复和上下文分裂。

## 理论导读

Subagent 是把复杂任务拆给专门角色处理的机制。主会话负责目标、协调和最终决策；子 Agent 负责某个明确子任务，例如测试排查、安全审查、文档整理、性能分析、迁移影响评估。

并行不是越多越好。只有当任务可以拆成相互独立的调查、审查或生成工作时，并行才有价值。

## 核心心智模型

```text
主 Agent：目标、边界、整合、决策
子 Agent：局部调查、专项审查、候选方案
```

子 Agent 的输出应该是结构化结论，而不是继续制造开放问题。

## 知识点详解

### 1. 适合 Subagents 的任务

- 一个 Agent 查测试失败，一个 Agent 查最近改动。
- 一个 Agent 做安全审查，一个 Agent 做性能审查。
- 一个 Agent 阅读后端，一个 Agent 阅读前端。
- 一个 Agent 生成文档草稿，一个 Agent 校验代码事实。

### 2. 不适合 Subagents 的任务

- 多个 Agent 同时改同一文件。
- 目标不清楚的探索。
- 强顺序依赖的迁移。
- 高风险命令执行。

### 3. 子 Agent 输出契约

建议格式：

```markdown
## 结论

## 证据

## 涉及文件

## 风险

## 建议下一步
```

> **难点：** 并行调查的难点不是启动多个 Agent，而是把它们的结论合并成一致、可执行的计划。

## 例子

### 并行排障设计

```text
主任务：定位登录接口 500。

子 Agent A：只读检查最近 3 次提交中登录相关改动。
子 Agent B：只读检查测试、日志和错误栈。
子 Agent C：检查配置和环境变量变化。

主 Agent 汇总后再决定是否修改代码。
```

### 代码审查分工

```text
安全审查 Agent：只看认证、授权、输入校验、密钥泄漏。
测试审查 Agent：只看新增行为是否有测试覆盖。
架构审查 Agent：只看模块边界和兼容性。
```

## 练习

1. 设计 3 个子 Agent：测试排查、安全审查、文档事实校验。
2. 给每个子 Agent 写职责、输入、禁止事项和输出格式。
3. 模拟一次多 Agent 结论冲突，写主 Agent 的合并规则。

## 验收

- 能判断任务是否适合并行。
- 能防止多个 Agent 修改同一范围。
- 能用结构化输出整合结论。

## 重点

- Subagents 适合并行调查和专项审查。
- 主 Agent 必须保留最终决策权。
- 子 Agent 应有窄职责和清晰输出契约。

## 易错

- **易错：** 为了显得高级，任何任务都开多个 Agent。
  正确做法：只有独立、可并行、结果可合并的任务才拆分。
- **易错：** 让多个 Agent 同时改同一文件。
  正确做法：并行读，串行改。
