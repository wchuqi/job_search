# Claude Code 学习资料：总览与心智模型

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 理解 Claude Code 与普通聊天、Copilot 式补全、脚本自动化的差异。
- 建立“Agent + 工具 + 权限 + 上下文 + 验证”的工作模型。
- 知道哪些任务适合 Claude Code，哪些任务应该谨慎或不用。

## 理论导读

Claude Code 是运行在开发环境中的编码 Agent。它不是只生成代码片段，而是可以读取项目文件、理解目录结构、调用工具、编辑文件、运行命令、根据测试反馈继续修复。它的能力来自三层组合：

1. 模型推理：理解需求、拆解任务、解释代码、判断下一步。
2. 工具执行：读写文件、搜索代码、运行命令、调用 MCP 工具。
3. 运行边界：权限确认、设置文件、工作目录、上下文窗口、会话状态。

心智模型可以理解为“一个有工具箱的远程同事坐在你的终端里”。这个同事很快、很勤奋，但它只能看到你给它的上下文和它主动读取到的文件；它会犯错；它可能误解项目规则；它执行高风险操作前必须受约束。

## 核心心智模型

| 维度 | Claude Code 中的含义 | 需要你控制的点 |
| --- | --- | --- |
| 目标 | 用户给出的开发任务 | 任务边界、验收标准、优先级 |
| 上下文 | 对话、文件、`CLAUDE.md`、工具输出 | 是否准确、是否过期、是否包含敏感信息 |
| 工具 | 文件、命令、MCP、浏览器或内部系统 | 权限、参数、失败处理 |
| 状态 | 当前会话、压缩摘要、后台任务 | 何时清理、何时新开会话 |
| 验证 | 测试、构建、lint、人工检查 | 让结果可证明，不靠主观感觉 |

## 知识点详解

### 1. Claude Code 解决的问题

Claude Code 适合处理需要在代码库中来回穿梭的任务：理解陌生项目、修 bug、补测试、重构、升级依赖、写文档、生成 PR 摘要、排查日志、迁移配置。它的优势不是“生成第一版代码”，而是能在现有工程约束下推进。

> **重点：** Claude Code 的核心价值是缩短“读代码 -> 找位置 -> 改动 -> 验证 -> 解释”的循环。

### 2. Claude Code 不适合的问题

不适合直接处理不可恢复、高权限、缺少验收标准的任务。例如生产数据库变更、密钥轮换、批量删除、自动发布、涉及隐私数据的导出。如果一定要用，必须用只读工具、隔离环境、审批流程和人工确认。

> **易错：** 看到 Claude Code 能执行命令，就把它当成自动运维工具。
>
> 正确做法：高风险环境中只允许只读命令和模拟输出，真正执行放到人工审批或 CI/CD 控制面。

### 3. Agentic Coding 的工作闭环

可靠的 Claude Code 工作流通常是：

```text
明确任务 -> 读取上下文 -> 制定小计划 -> 编辑少量文件 -> 运行验证 -> 根据反馈迭代 -> 汇报改动和风险
```

如果跳过“读取上下文”，模型容易用通用经验覆盖项目事实。如果跳过“验证”，看似合理的代码可能无法编译、破坏测试或违背业务规则。

## 例子

### 简单任务

```text
请阅读这个项目的 README 和 package.json，总结启动、测试、构建命令。
先不要修改文件。
```

### 修复任务

```text
这个测试失败了：<粘贴日志>
请先定位相关代码和测试，说明原因，再做最小修改。
修改后运行对应测试；如果无法运行，说明原因。
```

### 代码审查任务

```text
请以代码审查视角检查当前 git diff。
优先找 bug、回归风险、安全问题和缺失测试。
不要做风格建议，除非会造成真实维护风险。
```

## 练习

1. 找一个已有项目，让 Claude Code 只读地总结项目结构、入口文件、测试命令和风险区域。
2. 让 Claude Code 修复一个小 bug，但要求它先列出会修改哪些文件。
3. 对同一个任务分别用“模糊请求”和“有验收标准的请求”运行，比较结果差异。

## 验收

- 能解释 Claude Code 与聊天式代码生成的差异。
- 能把开发任务拆成可验证步骤。
- 能识别高风险操作并设计权限边界。
- 能要求 Claude Code 汇报“改了什么、为什么、如何验证、剩余风险”。

## 重点

- Claude Code 是工具执行型 Agent，不只是文本生成器。
- 好结果来自准确上下文、清晰边界、小步修改和验证闭环。
- 团队使用必须有规则、权限、审计和回滚策略。

## 难点

- 判断什么时候该让 Claude Code 自主推进，什么时候要收紧到只读分析。
- 把模糊开发意图转成可执行、可验证、可回滚的任务。

## 易错

- **易错：** 一开始就让 Claude Code 大范围重构。
  正确做法：先让它画出依赖和风险，再分模块实施。
- **易错：** 把失败验证当作模型失败。
  正确做法：失败验证是反馈信号，让 Agent 根据日志继续定位。
- **易错：** 把一次会话里的上下文长期复用。
  正确做法：复杂任务结束后新开会话或压缩上下文，避免旧假设污染。
