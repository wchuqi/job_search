# Claude Code 学习资料：Hooks 自动化和事件治理

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 理解 Hooks 如何把 Claude Code 的关键事件接入自动化。
- 能区分 Hooks、MCP、Skills、Slash Commands 的职责。
- 能设计安全、可观测、可失败回退的 Hook。

## 理论导读

Hooks 是围绕 Claude Code 生命周期事件触发的自动化机制。它适合做轻量、确定性的工程动作，例如格式化、审计、通知、阻断高风险命令、记录工具调用、在任务结束后生成摘要。

Hooks 不应该承载复杂推理。复杂判断交给 Claude Code 或专门工具，Hook 负责“事件发生时执行确定动作”。

## 核心心智模型

```text
事件 -> 条件匹配 -> 执行动作 -> 记录结果 -> 允许继续或阻断
```

一个 Hook 的质量取决于三点：触发准确、动作幂等、失败可解释。

## 知识点详解

### 1. 适合 Hooks 的场景

- 命令执行前检查是否包含危险模式。
- 文件编辑后运行格式化或 lint。
- 会话结束后保存摘要。
- 调用 MCP 工具前记录审计信息。
- 任务完成后通知团队渠道。

### 2. 不适合 Hooks 的场景

- 大规模重构。
- 复杂业务判断。
- 长时间运行的端到端测试。
- 需要大量上下文推理的代码审查。
- 生产部署或数据库写入。

### 3. Hook 设计原则

- 快：不要阻塞主工作流太久。
- 窄：只处理一个明确事件。
- 幂等：重复执行不会破坏状态。
- 可观测：输出清楚、日志可查。
- 可禁用：出现误阻断时有恢复路径。

> **重点：** Hook 是自动化护栏，不是隐藏的第二个 Agent。

## 例子

### 高风险命令检查思路

```text
事件：执行 Bash/PowerShell 命令前
规则：
- 包含递归删除、强制 push、生产部署命令时阻断或要求人工确认
- 测试、lint、git diff 允许继续
输出：
- 命令
- 风险类型
- 建议替代动作
```

### 任务结束摘要

```text
事件：会话或任务结束
动作：
- 收集修改文件
- 收集验证命令
- 生成 summary.md 或发送通知
```

## 练习

1. 设计一个 Hook：当 Claude Code 尝试修改 `.env`、部署配置或数据库迁移时提醒人工确认。
2. 设计一个 Hook：每次任务结束自动输出改动文件、测试结果、未验证原因。
3. 写出 Hook 误阻断时的恢复流程。

## 验收

- 能解释 Hook 的触发事件、输入、输出和失败行为。
- 能避免 Hook 执行不可恢复操作。
- 能把 Hook 纳入团队审计和版本管理。

## 重点

- Hooks 适合确定性、轻量级自动化。
- 高风险 Hook 必须可解释、可禁用、可审计。
- 不要在 Hook 中藏复杂业务逻辑。

## 易错

- **易错：** Hook 自动修改大量文件，Claude Code 不知道上下文。
  正确做法：Hook 只做格式化、检查、记录等可预测动作。
- **易错：** Hook 失败时沉默。
  正确做法：失败要返回清晰原因和处理建议。
