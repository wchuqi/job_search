# Claude Code 学习资料：CLI 命令、交互模式和批处理

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 掌握 Claude Code CLI 的常见使用形态：交互式会话、一次性命令、恢复会话、输出格式化。
- 能用 `/help` 和 `claude --help` 发现当前版本支持的命令。
- 能设计适合脚本和 CI 的非交互任务边界。

## 理论导读

Claude Code 的 CLI 有两类入口：终端命令和会话内 Slash Commands。终端命令负责启动、配置、非交互执行、健康检查、会话恢复等；Slash Commands 更像会话控制面，用于清理上下文、查看权限、管理记忆、选择模型、触发内置或自定义命令。

由于命令随版本变化，本资料只固定讲稳定使用模式。具体参数以当前版本的帮助输出为准：

```bash
claude --help
claude -p --help
```

会话内：

```text
/help
```

## 核心心智模型

| 模式 | 适合场景 | 风险 |
| --- | --- | --- |
| 交互式 `claude` | 复杂开发、逐步确认、排障 | 上下文越来越长，可能积累旧假设 |
| 非交互 `claude -p` | 脚本、摘要、只读检查、批处理 | 如果权限过宽，可能难以及时干预 |
| 恢复会话 | 继续未完成任务 | 旧上下文可能过期 |
| Slash Commands | 会话控制、记忆、权限、扩展 | 不同版本命令名称可能变化 |

## 知识点详解

### 1. 交互式会话

最常见方式是在项目根目录运行：

```bash
claude
```

适合需求不完全明确、需要模型读代码、提出计划、逐步修改、验证反馈的任务。交互式会话中要经常要求 Claude Code 先说明计划和文件范围，再修改。

### 2. 非交互模式

一次性任务可使用 print 模式，常见写法类似：

```bash
claude -p "总结当前 git diff 的风险，按严重程度列出"
```

非交互模式适合只读分析、生成摘要、格式化输出、CI 中的检查报告。它不适合默认执行高风险写操作。

### 3. 输出格式

Claude Code 通常支持选择输出格式，常用于脚本处理。例如文本、JSON、流式 JSON 等。具体参数名称以 CLI 帮助为准。

输出格式化适合：

- 让 CI 读取检查结果。
- 生成 PR 评论。
- 把代码审查发现转换为结构化报告。
- 对批量仓库执行一致的只读检查。

### 4. 会话控制

会话中常见控制动作包括清理上下文、压缩上下文、查看权限、管理记忆、恢复或继续会话。遇到以下情况应考虑清理或重开会话：

- 任务目标改变。
- 前面做过错误假设。
- 上下文太长，模型开始忽略关键约束。
- 已完成一个阶段，需要进入新的独立任务。

> **重点：** 长会话不是越长越好。上下文越多，越需要显式整理事实和当前目标。

## 例子

### 只读摘要

```bash
claude -p "只读分析当前仓库：识别技术栈、测试命令和主要模块，不要修改文件。"
```

### 结构化代码审查

```bash
claude -p "审查当前 git diff，只输出 JSON：findings、risk、tests_needed。不要修改文件。"
```

### 交互式修复

```text
请修复这个失败测试。先复现或解释如何复现，然后定位原因。
修改前列出计划，修改后运行最小相关测试。
```

## 练习

1. 用交互式模式完成一次 bug 修复。
2. 用 `-p` 模式生成当前 diff 的 PR 摘要。
3. 用 `/help` 找到当前版本支持的权限、记忆、会话相关命令，并记录到本地笔记。

## 验收

- 能解释 CLI 命令和 Slash Commands 的边界。
- 能选择交互式或非交互式模式。
- 能让 Claude Code 输出适合人读或机器处理的结果。

## 重点

- 命令细节以当前 CLI 帮助为准。
- 非交互模式要默认只读，写操作必须有清晰权限和验证。
- 会话太长时要清理、压缩或重开。

## 难点

- 在自动化脚本中既获得 AI 的灵活性，又不让它越权执行不可控操作。

## 易错

- **易错：** 在 CI 中让 Claude Code 直接修代码并提交。
  正确做法：先从只读审查、摘要、建议开始；自动改动必须进入独立分支和人工审查。
- **易错：** 把 `/help` 中某个命令名写死进长期教程。
  正确做法：教程写能力分类，命令名保留“以当前版本帮助为准”的校验步骤。
