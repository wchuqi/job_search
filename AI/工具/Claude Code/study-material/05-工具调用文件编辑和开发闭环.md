# Claude Code 学习资料：工具调用、文件编辑和开发闭环

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 掌握 Claude Code 在真实开发中的可靠任务流程。
- 能要求 Claude Code 小步读写、运行验证、解释失败和汇报风险。
- 能识别 AI 代码变更中的常见质量问题。

## 理论导读

Claude Code 的优势不是“一次生成很多代码”，而是能在工具反馈中迭代。可靠流程应强调事实读取、最小修改和验证闭环。你要像给同事分配任务一样给它明确边界：目标、文件范围、验收命令、禁止事项、汇报格式。

## 核心心智模型

```text
读 -> 计划 -> 改 -> 验 -> 复盘
```

- 读：读取相关代码、测试、配置、错误日志。
- 计划：说明原因、方案、影响范围。
- 改：小步修改，避免无关重构。
- 验：运行最小相关测试，再必要时运行全量验证。
- 复盘：说明改动、验证、风险、后续建议。

## 知识点详解

### 1. 代码阅读任务

让 Claude Code 读代码时，不要只问“解释这个项目”。更好的请求是指定输出结构：

```text
请只读分析当前项目：
1. 技术栈
2. 入口文件
3. 主要模块职责
4. 测试和构建命令
5. 最容易出错的边界
```

### 2. 缺陷修复任务

修 bug 要提供错误日志、复现步骤、期望行为和禁止事项。让 Claude Code 先定位再修改，能减少“凭猜测改代码”。

```text
请根据下面失败日志定位原因。
先不要修改文件，先说明相关代码路径和根因。
确认方案后再做最小修改。
```

### 3. 测试策略

Claude Code 生成测试时应覆盖：

- 成功路径。
- 边界输入。
- 错误处理。
- 回归用例。
- 与历史 bug 对应的最小复现。

### 4. 重构策略

重构任务要先锁定行为不变的验证方式。没有测试时，应先补 characterization tests，记录当前行为，再重构。

> **难点：** AI 很容易把“重构”做成“顺手改行为”。重构必须先定义行为边界。

## 例子

### Bug 修复提示

```text
目标：修复用户搜索为空时后端返回 500 的问题。
约束：
- 先读 controller、service 和相关测试。
- 最小修改，不重构无关模块。
- 修改后运行对应测试。
- 汇报：根因、改动文件、验证命令、剩余风险。
```

### PR 摘要提示

```text
请基于当前 git diff 生成 PR 描述：
- 背景
- 主要改动
- 测试结果
- 风险和回滚
不要编造未运行的测试。
```

## 练习

1. 找一个失败测试，让 Claude Code 完成定位、修复、验证。
2. 让 Claude Code 给已有函数补 3 个边界测试。
3. 让 Claude Code 审查自己刚才的改动，找潜在回归。

## 验收

- 每次修改都有明确目标和最小范围。
- 每次修改后有验证命令或未验证说明。
- 汇报能说明根因、影响范围和剩余风险。

## 重点

- 小步修改比一次大改更可靠。
- 不运行验证的代码修改不能视为完成。
- Claude Code 的输出要和 Git diff、测试结果互相校验。

## 易错

- **易错：** 让 Claude Code 同时修 bug、重构、优化性能、补文档。
  正确做法：拆成独立任务，每个任务有验收标准。
- **易错：** 只看最终回答，不看实际 diff。
  正确做法：最终一定检查 git diff 和测试输出。
