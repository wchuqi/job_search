# Claude Code 学习资料：IDE、桌面、Web 和团队协作

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 理解 Claude Code 在终端、IDE、桌面和 Web 中的不同使用方式。
- 能把个人工作流迁移到团队协作流程。
- 能设计代码评审、PR、Issue、CI 中的 Claude Code 使用规范。

## 理论导读

Claude Code 的入口可能包括终端、IDE 集成、桌面应用和 Web 体验。不同入口的核心差异不是模型能力，而是上下文来源、工具权限、协作方式和审计能力。

终端适合深度开发，IDE 适合贴近编辑器上下文，Web/桌面适合跨设备、计划和协作。团队落地时要统一规则，否则每个人的使用方式会产生不可控差异。

## 核心心智模型

| 入口 | 优势 | 注意事项 |
| --- | --- | --- |
| 终端 | 最接近项目工具链 | 权限和命令风险最高 |
| IDE | 文件上下文清晰，改动可见 | 注意自动编辑范围 |
| 桌面/Web | 适合计划、总结、远程任务 | 本地工具和仓库状态可能不同 |
| CI/自动化 | 可重复、可审计 | 默认只读，避免自动写生产 |

## 知识点详解

### 1. IDE 协作

IDE 集成适合：

- 当前文件解释。
- 局部重构。
- 测试补全。
- 错误定位。
- 代码审查建议。

但 IDE 当前文件上下文可能过窄，复杂任务仍需让 Claude Code 搜索项目事实。

### 2. 团队规范

团队使用建议：

- 仓库根目录维护 `CLAUDE.md`。
- 统一允许命令和禁止命令。
- 约定 AI 生成代码必须经过人类 review。
- PR 描述标明使用了 Claude Code 的范围和验证结果。
- 高风险文件变更要求人工确认。

### 3. PR 和 Issue 工作流

Claude Code 可以帮助：

- 从 Issue 生成实现计划。
- 从 diff 生成 PR 摘要。
- 从 CI 失败日志定位问题。
- 从 review 评论生成修复计划。

它不应该绕过代码评审、测试和发布流程。

## 例子

### 团队使用规范片段

```markdown
## Claude Code 使用规范

- AI 修改代码必须进入普通 PR 流程。
- 不允许自动 push 到 main。
- 不允许自动执行部署、数据库写入和密钥操作。
- PR 中必须写明验证命令；未运行测试必须说明原因。
- 对安全、权限、账单、支付、数据迁移相关变更必须人工复核。
```

## 练习

1. 为团队写一份 1 页 Claude Code 使用规范。
2. 设计 PR 模板，要求填写 AI 使用范围和验证结果。
3. 设计 CI 只读检查：让 Claude Code 审查 diff 并输出风险报告。

## 验收

- 能解释不同入口的上下文和权限差异。
- 能把 Claude Code 放入现有 PR/CI/Issue 流程。
- 能防止 AI 自动绕过团队质量门禁。

## 重点

- 团队采用要标准化，不靠个人习惯。
- AI 辅助不替代代码评审和发布流程。
- 上下文入口不同，风险也不同。

## 易错

- **易错：** 在 Web 中规划的任务，直接假设本地仓库状态一致。
  正确做法：执行前让 Claude Code 在本地重新读取事实。
- **易错：** AI 生成 PR 后跳过 review。
  正确做法：AI 改动更需要看 diff、测试和风险说明。
