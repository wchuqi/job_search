# Claude Code 学习资料：Agent 执行循环和状态机深度解析

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 理解 Claude Code 不是“一问一答”，而是围绕目标、工具、反馈循环运行的 Agent。
- 能画出一次任务从输入到完成的状态流转。
- 能判断任务卡住时卡在感知、规划、执行、验证还是恢复阶段。

## 理论导读

Claude Code 的一次有效工作不是模型直接输出代码，而是一个循环：

```text
接收目标 -> 建立上下文 -> 形成计划 -> 选择工具 -> 执行动作 -> 观察结果 -> 更新状态 -> 决定继续/停止/求助
```

这套循环的核心是“模型推理”和“外部反馈”交替出现。模型不能凭空知道仓库状态，它要通过读文件、搜索、运行命令、调用 MCP 等动作把外部状态拉回上下文。每个动作又会改变环境或产生新证据，下一步推理必须基于这些证据更新。

## 核心状态机

```text
Idle
  -> Intake          接收任务和约束
  -> ContextBuild    读取文件、规则、历史、工具可用性
  -> Plan            拆解步骤和风险
  -> PermissionCheck 检查工具、命令、文件权限
  -> Act             编辑文件、运行命令、调用 MCP
  -> Observe         读取结果、错误、diff、测试输出
  -> Reflect         判断是否满足目标
  -> Finalize        汇报改动、验证和风险

异常分支：
  PermissionDenied -> AskUser / ReduceScope
  ToolFailure      -> Diagnose / AlternativeTool
  VerificationFail -> DebugLoop / RollbackPlan
  ContextDrift     -> RebuildContext / NewSession
```

## 阶段详解

### 1. Intake：任务接收

这一阶段决定 Claude Code 会不会从一开始就跑偏。好的 Intake 包含：

- 目标：要达成什么结果。
- 范围：哪些文件、模块、分支、环境。
- 禁止事项：不能修改什么、不能执行什么。
- 验收：用什么测试、构建、人工检查证明完成。
- 输出：最终要给什么格式的报告。

> **重点：** 如果 Intake 模糊，后面的计划再漂亮也可能解决错问题。

### 2. ContextBuild：上下文构建

Claude Code 需要主动收集事实，包括：

- 项目结构：入口、模块、配置、测试。
- 任务相关代码：调用链、数据结构、错误路径。
- 项目规则：`CLAUDE.md`、README、贡献指南。
- 当前状态：Git diff、失败日志、依赖版本。
- 工具可用性：测试命令、包管理器、MCP server、权限。

上下文构建不是读取越多越好，而是读取“能减少不确定性”的证据。过宽读取会增加噪声和成本。

### 3. Plan：计划

计划的作用不是形式化汇报，而是暴露模型的假设。一个可审查的计划应包含：

- 根因假设。
- 证据来源。
- 修改文件范围。
- 验证命令。
- 回滚或替代方案。

计划的粒度要适配风险。低风险文档修改可以短计划；数据库迁移、认证逻辑、安全策略必须详细计划。

### 4. PermissionCheck：权限决策

任何工具调用前都应经过权限边界。权限决策至少考虑：

- 工具是否启用。
- 文件是否在允许范围。
- 命令是否属于允许清单。
- 是否涉及密钥、生产、删除、网络写入。
- 是否需要用户确认。

> **难点：** 权限检查不是只看命令字符串，还要看上下文。例如 `npm run deploy` 和 `npm test` 都是 npm 命令，但风险完全不同。

### 5. Act：执行

执行阶段包括文件编辑、命令运行和 MCP 调用。可靠执行有三个原则：

- 小步：一次修改一组相关文件。
- 可观察：每步执行后能看到结果。
- 可恢复：失败后能定位和回滚。

### 6. Observe：观察

观察不是把命令输出贴进上下文，而是抽取关键信号：

- 测试是否通过。
- 失败堆栈指向哪里。
- diff 是否包含无关改动。
- 生成文件是否被错误修改。
- 工具是否因为权限、环境或参数失败。

### 7. Reflect：反思和停止条件

Claude Code 需要判断是否继续。停止条件包括：

- 验收命令通过。
- 目标已达成且无未处理风险。
- 权限或信息不足，需要用户输入。
- 继续尝试会扩大风险。
- 任务目标改变，应重开会话。

## 失败模式分析

| 失败表现 | 可能卡住阶段 | 典型修复 |
| --- | --- | --- |
| 改错文件 | ContextBuild 不充分 | 重新读取调用链和测试 |
| 反复修不好 | Observe/Reflect 弱 | 停止修改，复盘证据和假设 |
| 运行危险命令 | PermissionCheck 弱 | 增加允许清单、Hook、人工确认 |
| 代码能编译但业务错 | Intake 验收不清 | 补业务场景和回归测试 |
| 输出与事实不符 | ContextDrift | 重开会话，重新读取文件 |

## 例子：失败测试修复的状态流

```text
Intake:
  修复 OrderServiceTest 的空订单失败。

ContextBuild:
  读失败日志 -> 读测试 -> 读 OrderService -> 读 DTO。

Plan:
  假设空列表没有被处理，修改 service 边界分支，补测试。

PermissionCheck:
  允许编辑 service/test，允许运行单测。

Act:
  编辑代码和测试。

Observe:
  运行单测失败，发现断言期望不是空列表而是业务错误码。

Reflect:
  更新根因，回退错误分支，按业务规则返回错误码。

Finalize:
  汇报根因、文件、测试命令、剩余风险。
```

## 练习

1. 找一个 Claude Code 任务记录，把每一步标注到状态机。
2. 故意给一个模糊需求，观察它在哪个阶段开始跑偏。
3. 让 Claude Code 在每次工具调用前说明“为什么需要这个工具、预期看到什么、失败后怎么办”。

## 验收

- 能画出 Claude Code 的任务状态机。
- 能把失败定位到具体阶段。
- 能用状态机改写更可靠的任务提示。

## 重点

- Agent 的本质是循环，不是单轮生成。
- 状态机能帮助你诊断失败，而不是只评价“模型聪不聪明”。
- 最关键的控制点是 Intake、PermissionCheck 和 Reflect。

## 难点

- 高级使用不是让 Agent 完全自主，而是知道在哪些状态必须收紧权限、要求证据或停止。

## 易错

- **易错：** 看到测试失败就让 Claude Code 连续改。
  正确做法：失败后先进入 Observe/Reflect，提取新证据再决定是否修改。
- **易错：** 只要求最终结果，不约束中间状态。
  正确做法：对高风险任务明确计划、权限、验证和停止条件。
