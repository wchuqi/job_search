# Claude Code 学习资料：Skills、Slash Commands 和插件复用

[返回索引](../Claude Code学习资料.md)

## 学习目标

- 理解如何把常用工作流封装成可复用能力。
- 能区分 Skills、Slash Commands、插件和 `CLAUDE.md`。
- 能设计团队可维护的 Claude Code 扩展包。

## 理论导读

当同一类任务反复出现，例如代码审查、故障复盘、接口文档生成、测试补全、PR 描述，就不应该每次手写长 prompt。Claude Code 的复用机制可以把流程、模板、工具说明和约束打包，让团队共享一致做法。

## 核心心智模型

| 机制 | 适合内容 | 不适合内容 |
| --- | --- | --- |
| `CLAUDE.md` | 项目长期规则 | 一次性任务模板 |
| Slash Command | 可触发的短流程 | 大量背景知识 |
| Skill | 专门领域流程、参考资料、脚本 | 项目私密凭据 |
| Plugin | 分发一组命令、技能、配置 | 未审计的高权限自动化 |

## 知识点详解

### 1. Slash Commands

Slash Commands 适合把常用请求封装成命令，例如：

- `/review`：按代码审查格式检查当前 diff。
- `/pr-summary`：生成 PR 描述。
- `/test-plan`：根据改动生成测试计划。
- `/incident`：生成故障复盘草稿。

好命令要明确输入、输出、约束和不做什么。

### 2. Skills

Skills 更适合包含较多说明、参考资料和脚本的专门能力。一个好 Skill 通常包括：

- 触发场景。
- 工作流程。
- 输出格式。
- 可复用脚本或模板。
- 注意事项和边界。

### 3. 插件化分发

插件适合团队统一分发一组命令、技能和配置。例如“后端服务开发插件”“前端代码审查插件”“安全审计插件”。插件必须有版本、负责人、审计和更新说明。

> **重点：** 复用能力的目标是降低认知负担，而不是隐藏风险。

## 例子

### `/review` 命令规格

```markdown
目标：审查当前 git diff。
输出顺序：
1. 严重 bug
2. 回归风险
3. 安全问题
4. 缺失测试

约束：
- 不做纯风格建议。
- 每条发现必须引用文件路径和原因。
- 不修改文件。
```

### Skill 适用场景

```text
当任务涉及“生成面试学习资料”时，使用学习资料 Skill：
- 先做知识覆盖图
- 再拆分阶段
- 每章包含目标、理论、例子、练习、验收、重点、难点、易错
```

## 练习

1. 为团队写 3 个 Slash Command：代码审查、PR 摘要、测试计划。
2. 把一个重复任务抽象成 Skill，写明触发条件和输出标准。
3. 设计插件版本更新流程，包含破坏性变更说明。

## 验收

- 能说明复用机制之间的职责边界。
- 能写出可执行、可测试的命令规格。
- 能把团队流程从个人 prompt 中迁移到可维护资产。

## 重点

- Slash Commands 封装触发入口。
- Skills 封装领域工作法。
- 插件封装分发和治理。

## 易错

- **易错：** 把团队所有流程都塞到 `CLAUDE.md`。
  正确做法：长期项目规则放 `CLAUDE.md`，重复任务流程放命令或 Skill。
- **易错：** 插件启用高权限工具但没有说明。
  正确做法：插件必须声明工具、权限、风险和审计方式。
