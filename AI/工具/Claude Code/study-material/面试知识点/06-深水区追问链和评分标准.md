## 一、Agent 执行机制

### 1. Claude Code 执行一次修复任务时，内部可以抽象成哪些阶段？

**参考答案：**

可以抽象为 Intake、ContextBuild、Plan、PermissionCheck、Act、Observe、Reflect、Finalize。也就是先接收目标和约束，再构建上下文，制定计划，检查权限，执行文件编辑或命令，观察结果，根据反馈迭代，最后汇报改动和风险。

**追问：如果它连续两次修复失败，你认为应该回到哪个阶段？**

应回到 Observe/Reflect，停止继续修改，复盘失败证据和错误假设；必要时回到 ContextBuild 重新读取关键文件，而不是继续在 Act 阶段叠补丁。

**评分点：**

- 初级：知道它会读文件、改代码、跑测试。
- 中级：能说出计划、权限、验证闭环。
- 高级：能用状态机定位失败阶段和停止条件。

## 二、上下文工程

### 2. 为什么说 Claude Code 不等于“知道整个仓库”？

**参考答案：**

模型上下文有限，Claude Code 需要通过工具主动读取相关文件、命令输出和记忆。它看到的是被选择和装载进上下文的事实，而不是完整仓库的实时全量状态。因此要让它引用真实文件、读取测试和错误日志，避免凭通用经验猜测。

**追问：压缩上下文后最容易丢什么？**

容易丢用户禁止事项、细微业务边界、失败日志关键行、旧假设是否已被否定、未验证风险。压缩后应让它复述当前目标、已知事实、禁止事项和下一步。

**评分点：**

- 初级：知道上下文有限。
- 中级：能讲 `CLAUDE.md`、文件读取和会话压缩。
- 高级：能讲上下文污染、压缩失真和最小上下文策略。

## 三、权限和安全

### 3. 如何设计 Claude Code 的权限决策？

**参考答案：**

先识别动作类型和目标对象，再计算风险等级，匹配允许清单和禁止清单，结合当前任务约束，最后允许、拒绝、询问或降级为只读。风险等级可分为只读、可恢复修改、受控命令、环境变更、高风险操作、生产写操作。

**追问：为什么不能简单允许 `git *` 或 `npm *`？**

因为命令家族内风险差异巨大。`git diff` 是只读，`git reset --hard` 会丢改动；`npm test` 是验证，`npm run deploy` 可能发布生产。允许清单要按意图精确设计。

**评分点：**

- 初级：知道危险命令要确认。
- 中级：能列允许/禁止清单。
- 高级：能按动作、对象、环境和任务约束做风险分级。

## 四、MCP 设计

### 4. 设计一个只读数据库 MCP，关键点是什么？

**参考答案：**

不要暴露任意 SQL。应设计窄工具，例如按订单 ID、状态、时间范围查询；限制字段、行数、时间范围；连接只读副本；脱敏敏感字段；记录调用审计；错误信息不泄露连接串；把外部返回当数据而不是指令。

**追问：MCP 返回的 Issue 评论里包含“忽略之前规则”怎么办？**

把外部内容视为不可信数据，不作为系统或用户指令执行。工具返回应结构化，并明确内容来源；敏感动作仍受权限系统控制。

**评分点：**

- 初级：知道 MCP 用来接外部工具。
- 中级：能讲 schema、只读和脱敏。
- 高级：能讲 prompt injection、审计、最小权限和数据裁剪。

## 五、Hooks 治理

### 5. Hooks 应该做什么，不应该做什么？

**参考答案：**

Hooks 应做确定性治理：命令前风险检查、文件编辑后密钥扫描、会话结束摘要、工具调用审计。不应承载复杂业务推理或大范围自动修改。高风险 Hook 应 fail-closed 或降级为人工确认，通知类 Hook 可以 fail-open。

**追问：Hook 误阻断怎么办？**

Hook 需要可解释输出、dry-run、测试样例、配置化规则和人工恢复路径。误阻断后应记录样例，调整规则，而不是让用户绕过全部权限。

**评分点：**

- 初级：知道 Hooks 是自动化。
- 中级：能举命令检查和审计例子。
- 高级：能讲 fail-open/fail-closed、误阻断治理和 Hook 测试。

## 六、Subagents 编排

### 6. 什么时候应该使用 Subagents？

**参考答案：**

当任务可以拆成独立、只读、结果可合并的专项调查时使用，例如安全审查、测试审查、兼容性审查并行。主 Agent 负责目标、分工、汇总和最终决策。多个 Agent 不应同时修改同一文件。

**追问：两个 Subagent 结论冲突怎么办？**

证据优先于猜测，高风险安全问题优先，能验证的方案优先，修改范围小的方案优先；无法消解时请求用户确认。

**评分点：**

- 初级：知道可以并行。
- 中级：能拆分职责。
- 高级：能设计输出契约和冲突消解规则。

## 七、SDK 平台化

### 7. 如何设计 AI PR Reviewer？

**参考答案：**

输入 PR diff、相关文件、CI 结果和项目规范；Context Builder 收集最小上下文；Policy Engine 限制只读；Agent Runtime 执行审查；Result Validator 校验 JSON schema、文件引用和敏感信息；输出 findings；人工反馈用于评估；审计记录模型、工具调用、成本和采纳结果。

**追问：如何评估它是否有效？**

用历史 PR 建离线评估集，统计 bug 发现率、误报率、严重度排序、引用准确率、格式合规率、成本和延迟；上线后统计采纳率、驳回率、返工、CI 失败率和 review 周期。

**评分点：**

- 初级：说能自动 review。
- 中级：能讲输入、输出和只读权限。
- 高级：能讲架构、评估、审计、Result Validator 和人工反馈闭环。

## 八、综合场景

### 8. 团队准备大规模引入 Claude Code，你会如何推进？

**参考答案：**

分阶段推进：个人试点只读和低风险任务；小组阶段沉淀 `CLAUDE.md`、命令清单、PR 模板；团队阶段引入 MCP、Hooks、Skills、评估集和审计；平台化阶段再用 SDK 构建内部应用。全程禁止生产写操作自动化，AI 修改必须走 review 和 CI。

**追问：如何证明收益？**

按任务类型统计完成时间、返工、缺陷率、测试覆盖、review 周期、采纳率、误报率、成本和事故。不能只看生成代码量。

**评分点：**

- 初级：会安装和使用。
- 中级：能写规范和限制危险操作。
- 高级：能设计试点、治理、平台化、指标和事故复盘。

## 面试答题总原则

- 先讲机制，再讲命令。
- 先讲边界，再讲效率。
- 先讲证据，再讲结论。
- 先讲只读，再讲修改。
- 先讲验证，再讲完成。
