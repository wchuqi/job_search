## 一、配置和记忆

### 1. `CLAUDE.md` 应该写什么？

**参考答案：**

`CLAUDE.md` 应写给 Claude Code 的项目操作指南，包括技术栈、常用命令、测试方式、禁止修改目录、代码风格、验证要求、高风险操作确认规则。它不应包含密钥、账号、临时日志或过期背景。

> **重点：** `CLAUDE.md` 是 Agent 操作手册，不是 README 的复制品。

### 2. 如何处理用户级、项目级和子目录级规则？

**参考答案：**

用户级适合个人偏好，项目级适合仓库通用规则，子目录级适合局部模块规则。冲突时通常遵守更具体、更贴近当前任务的规则，同时用户本轮明确指令应优先。例如本轮说“只读分析”，即使项目规则写了“修改后跑测试”，也不能修改文件。

### 3. 长会话有什么问题？如何治理？

**参考答案：**

长会话会积累旧假设、无关日志和过期上下文，导致模型忽略当前目标或混淆文件状态。治理方式包括阶段性总结、压缩上下文、清理会话、重开会话，并在新阶段重新读取关键文件。

## 二、权限和安全

### 4. Claude Code 为什么需要权限控制？

**参考答案：**

因为它能读写真实文件、执行本机命令、调用外部工具，可能造成误删、泄密、错误部署或生产数据破坏。权限控制用于区分可自动执行、需确认和禁止执行的操作，保障最小权限、可恢复和可审计。

> **重点：** 权限是工程安全边界，不是交互体验细节。

### 5. 如何设计 Claude Code 的允许命令和禁止命令？

**参考答案：**

允许命令一般是测试、lint、构建、只读 Git 命令，例如 `npm test`、`pytest`、`mvn test`、`git diff`。禁止或需确认命令包括递归删除、强制 push、生产部署、数据库写入、修改密钥文件、权限变更。团队应把这些规则写进项目文档和配置。

### 6. `--dangerously-skip-permissions` 这类跳过权限模式能不能用？

**参考答案：**

只能在隔离、可丢弃、无生产凭据、无敏感数据的环境中短时使用，例如临时容器或实验仓库。不应该在日常开发机、生产网络或有 SSH key、云凭据的机器上长期使用。

> **易错：** 把危险模式当成效率优化。

### 7. 如何防止 Claude Code 泄露密钥？

**参考答案：**

不把密钥写进 prompt、`CLAUDE.md` 或仓库；用环境变量和密钥管理器；限制 MCP 返回字段；日志脱敏；禁止打印完整 token；在 CI 中使用最小权限短期凭据；增加 secret scanning。

### 8. 如果 Claude Code 准备执行高风险命令，你怎么处理？

**参考答案：**

先阻断或要求它只读解释命令目的、影响范围、回滚方式和替代方案。对删除、部署、数据库写入这类操作，要求人工确认，并尽量改成 dry-run、只读检查或在隔离环境执行。
