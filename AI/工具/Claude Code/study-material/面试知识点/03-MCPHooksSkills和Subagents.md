## 一、MCP

### 1. MCP 在 Claude Code 中解决什么问题？

**参考答案：**

MCP 用统一协议把外部工具和数据源接入 Claude Code，例如文档检索、Issue 查询、CI 状态、日志平台、只读数据库。它解决的是“模型需要外部事实或工具，但不能直接无边界访问系统”的问题。

> **重点：** MCP 是受控工具接入层。

### 2. 如何设计一个安全的只读数据库 MCP？

**参考答案：**

只暴露白名单查询，不允许任意 SQL；限制时间范围、返回行数和字段；脱敏手机号、地址、身份证等敏感信息；记录调用人和参数；只连接测试或只读副本；错误信息不泄露连接串。

### 3. MCP 和直接让 Claude Code 执行数据库 CLI 有什么区别？

**参考答案：**

MCP 可以提供结构化参数、权限校验、脱敏、审计和限流；直接 CLI 容易暴露完整权限和凭据，也难以限制危险 SQL。生产或准生产环境应优先用受控 MCP，而不是裸命令。

## 二、Hooks

### 4. Hooks 适合做什么？

**参考答案：**

Hooks 适合在 Claude Code 生命周期事件上执行确定性动作，例如命令前风险检查、文件修改后格式化、任务结束摘要、工具调用审计、通知。它不适合复杂推理或长时间业务流程。

### 5. Hooks 和 MCP 的边界是什么？

**参考答案：**

MCP 提供外部工具和数据源；Hooks 响应 Claude Code 内部事件并执行自动化动作。一个是工具能力，一个是事件治理。比如“查询 Issue”是 MCP，“每次查询 Issue 后记录审计日志”可以是 Hook。

## 三、Skills、Slash Commands 和插件

### 6. 什么时候应该把 prompt 封装成 Slash Command 或 Skill？

**参考答案：**

当任务重复出现、输出格式固定、团队希望统一流程时，应封装。例如代码审查、PR 摘要、测试计划适合 Slash Command；复杂领域流程、带参考资料和脚本的任务适合 Skill。

### 7. 插件化的价值和风险是什么？

**参考答案：**

价值是团队可以统一分发命令、技能和配置，降低重复维护成本。风险是插件可能引入高权限工具或过期规则，所以必须有版本、权限声明、负责人、变更记录和审计。

## 四、Subagents

### 8. Subagents 适合什么场景？

**参考答案：**

适合可并行、职责清晰、结果可合并的任务，例如一个子 Agent 查测试失败，一个查最近改动，一个做安全审查。主 Agent 负责整合和最终决策。

### 9. 如何避免多个 Subagents 互相冲突？

**参考答案：**

优先让子 Agent 只读调查；如果要修改，必须串行执行并明确文件范围。给每个子 Agent 规定职责、禁止事项和输出格式，主 Agent 汇总结论后再决定改动。

> **易错：** 多个 Agent 同时改同一个文件，造成冲突和难以审计。
