## 一、工程实践

### 1. 你如何用 Claude Code 修复一个失败测试？

**参考答案：**

先提供失败日志和复现命令，让 Claude Code 只读定位相关测试和代码；修改前要求它说明根因、计划和影响范围；实施最小改动；修改后运行相关测试；最后汇报改动文件、验证命令和剩余风险。

> **重点：** 先复现和定位，再修改，最后验证。

### 2. 如何让 Claude Code 做代码审查？

**参考答案：**

让它以 review 视角检查当前 diff，优先找 bug、回归风险、安全问题和缺失测试。要求每条发现引用文件和原因，不做纯风格建议，不修改文件。最后输出按严重程度排序的 findings。

### 3. Claude Code 生成的代码是否需要人工 review？

**参考答案：**

需要。AI 生成代码可能看似合理但破坏边界条件、安全规则或业务语义。人工 review、测试和 CI 是必要质量门禁。团队中 AI 代码应进入普通 PR 流程。

### 4. 如何处理 Claude Code 无法运行测试？

**参考答案：**

让它说明无法运行的具体原因，例如依赖缺失、平台问题、权限阻断、环境变量缺失；给出替代验证方式，例如静态检查、局部编译、人工检查清单。最终汇报不能写“测试通过”。

## 二、生产排障

### 5. Claude Code 连续两次修复失败，你怎么办？

**参考答案：**

停止继续修改，进入只读复盘：当前目标是什么、做过哪些改动、哪些假设失败、测试日志说明什么、最小复现是什么。必要时回滚 AI 改动，重新读取关键文件，再制定更小步骤。

> **易错：** 失败后让它无限“再试一次”。

### 6. 如何排查 Claude Code 输出和项目事实不符？

**参考答案：**

要求它引用具体文件和代码位置，重新读取相关文件，检查是否启动目录错误、上下文过期、长会话污染或未读取实际代码。必要时重开会话并提供最小上下文。

### 7. 如何控制 Claude Code 成本？

**参考答案：**

拆小任务、限制文件范围、先只读分析、避免长会话堆积、减少并行 Agent、设置预算和超时、使用结构化输出、把常见流程封装成命令或 Skill。成本评估还要包括人工返工和误报成本。

### 8. 如何把 Claude Code 接入团队 PR 流程？

**参考答案：**

可以让它生成 PR 摘要、风险清单、测试建议和 review 初稿，但不能绕过人工 review 和 CI。PR 模板应记录 AI 使用范围、验证命令、未验证原因和高风险变更。

### 9. 如果 Claude Code 修改了不该改的文件，如何恢复？

**参考答案：**

先查看 `git diff` 确认范围；如果改动未提交，可只回滚对应文件或手工撤销；分析为什么权限或任务边界没阻止它；更新 `CLAUDE.md`、权限策略或 Hook，防止重复发生。不要盲目 `git reset --hard`，避免丢失用户其他改动。
