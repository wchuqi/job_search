# OpenCode 学习路线图

## 阶段 1：基础认知

- 目标：理解 OpenCode 是什么、适合什么、不适合什么。
- 需要掌握：AI coding agent、TUI、CLI、Provider、模型、工具、权限、Agent、AGENTS.md。
- 例子：在一个已有项目中运行 `opencode`，用 `@` 引用文件询问认证逻辑。
- 练习：让 OpenCode 解释一个模块，但不允许它改文件。
- 验收：能说清 OpenCode 与普通聊天、IDE 补全、脚本自动化的区别。
- 重点：OpenCode 的核心价值是把 LLM 接入代码库、终端、文件编辑和团队规则。
- 易错：把 OpenCode 当作“自动程序员”。正确做法：把它当成带工具权限的开发协作者。

## 阶段 2：安装和日常使用

- 目标：完成安装、Provider 配置、项目初始化和基础交互。
- 需要掌握：安装方式、API key、`/connect`、`/init`、`AGENTS.md`、Plan/Build 切换、`/undo` 和 `/redo`。
- 例子：先要求 OpenCode 生成实现计划，再切换 Build 执行。
- 练习：修复一个小 bug，并在执行前要求它列出要改的文件。
- 验收：能完成一次“阅读代码 -> 制定计划 -> 修改 -> 运行测试 -> 回滚”的闭环。
- 重点：复杂改动先 Plan，低风险小改动才直接 Build。
- 易错：不给上下文直接说“修一下”。正确做法：给目标、相关文件、约束和验收命令。

## 阶段 3：配置和权限

- 目标：能为个人、项目和团队设计安全的 OpenCode 配置。
- 需要掌握：`opencode.json`、`opencode.jsonc`、配置优先级、`permission`、模型配置、shell、server、MCP、插件、TUI 配置。
- 例子：项目配置默认 `bash` 为 `ask`，允许 `git status`，拒绝 `rm *`。
- 练习：为一个前端项目写一个只允许读文件、运行测试、禁止写入的 Plan 配置。
- 验收：能解释“最后匹配规则生效”和“配置合并而不是整体替换”的影响。
- 重点：权限配置是 OpenCode 落地的安全阀。
- 难点：权限既要够用，又要防止误删、越权访问和秘密泄露。
- 易错：全局配置 `permission: "allow"` 后在生产仓库直接使用。

## 阶段 4：工作流工程化

- 目标：把重复任务固化为命令、规则、Agent 和工具。
- 需要掌握：`.opencode/commands/`、自定义 Agent、Rules、AGENTS.md、MCP server、自定义工具、插件 hooks。
- 例子：创建 `/review-changes` 命令，自动读取最近提交并按团队规范做 review。
- 练习：为项目写一个“生成测试 + 运行测试 + 输出风险”的自定义命令。
- 验收：能把一个团队重复流程封装成可审查、可复用、权限可控的 OpenCode 入口。
- 重点：好的 OpenCode 工程化不是放权，而是把流程和边界写清楚。
- 难点：MCP、插件、自定义工具会增加上下文和攻击面，需要按需启用。

## 阶段 5：团队治理和高级排障

- 目标：能在团队中推广 OpenCode，并处理质量、安全、成本和稳定性问题。
- 需要掌握：配置分层、共享会话、变更审计、模型选择、上下文管理、排障命令、token 成本、失败恢复。
- 例子：为团队仓库制定 `AGENTS.md`、`.opencode/commands/test.md`、`permission` 策略和 PR 审查规则。
- 练习：设计一套“新人用 OpenCode 修改业务代码”的安全流程。
- 验收：能让 OpenCode 的输出进入常规代码审查、测试、CI 和发布流程，而不是绕过流程。
- 重点：OpenCode 只能加速研发流程，不能替代需求确认、代码审查、测试和安全审批。
- 易错：把 AI 生成代码直接合入主干。

## 阶段 6：机制深挖和平台化

- 目标：从“会用 OpenCode”升级到“能设计和治理 OpenCode 工作体系”。
- 需要掌握：执行生命周期、会话状态、配置合并、权限匹配算法、Agent 调度、上下文压缩、MCP 安全、插件供应链、事故复盘、度量体系。
- 例子：为一个团队设计“只读探索 Agent + 安全审查 Agent + PR review 命令 + 托管权限配置 + 成本指标”的完整方案。
- 练习：复盘一次 AI 误改代码事故，补齐权限、命令、`AGENTS.md` 和 PR 模板。
- 验收：能解释一次 OpenCode 行为从用户输入到工具执行的完整链路，并能指出每个阶段的风险控制点。
- 重点：深度能力不是记字段，而是理解配置、权限、上下文和工具如何共同决定 Agent 行为。
- 难点：多层配置、多 Agent、多工具、多轮上下文共同作用时，故障定位需要分层排查。
- 易错：只优化 prompt，不治理工具权限、审计和团队流程。

## 推荐学习节奏

| 周期 | 学习重点 | 产出 |
| --- | --- | --- |
| 第 1 天 | 安装、初始化、TUI 基础 | 在一个项目中完成 `/init` 和一次只读问答 |
| 第 2-3 天 | Plan/Build、文件引用、回滚 | 完成一个小改动并能 `/undo` |
| 第 4-5 天 | 配置、权限、模型 | 写出个人和项目级 `opencode.jsonc` |
| 第 6-7 天 | 自定义命令和 Agent | 固化 2 个团队常用命令 |
| 第 2 周 | MCP、插件、安全治理 | 设计一套团队落地规范和排障手册 |
| 第 3 周 | 深度机制和平台化治理 | 输出执行链路图、权限模板、扩展评审表和度量指标 |

## 最终能力清单

- 能独立安装和配置 OpenCode。
- 能在真实仓库中用 OpenCode 阅读、计划、修改、测试和回滚。
- 能解释配置优先级、权限匹配、Agent 模式和工具边界。
- 能编写项目级命令、Agent、规则和 AGENTS.md。
- 能判断哪些场景需要 MCP、自定义工具或插件。
- 能设计团队安全策略、审计流程和成本控制方法。
- 能从执行生命周期、配置解析和权限匹配角度定位复杂问题。
- 能评审 MCP、自定义工具和插件的供应链风险。
- 能用度量指标判断 OpenCode 是否真正提升团队效率和质量。
