# OpenCode 学习资料：调试、回滚、排障和成本控制

[返回索引](../OpenCode学习资料.md)

## 学习目标

- 掌握 OpenCode 常见失败类型。
- 会使用回滚、debug、配置检查和会话总结。
- 能控制模型成本、上下文膨胀和无效工具调用。

## 理论导读

OpenCode 的失败通常不是单点原因，而是上下文、权限、配置、模型、工具、项目状态共同作用。排障时要先分类：是模型理解错、上下文缺失、权限被拒、命令失败、配置未生效，还是工具输出误导。

## 常见问题分类

| 问题 | 现象 | 排查方向 |
| --- | --- | --- |
| Provider 未配置 | 无法调用模型 | `/connect`、API key、模型名 |
| 配置未生效 | 权限或模型不符合预期 | `opencode debug config` |
| 权限被拒 | 工具无法执行 | `permission`、Agent 权限、外部目录 |
| 命令失败 | 测试或构建失败 | shell、依赖、工作目录、环境变量 |
| 误改文件 | diff 超出范围 | `/undo`、Git diff、权限收紧 |
| 上下文膨胀 | 响应慢、成本高、跑偏 | 减少 MCP、拆任务、总结上下文 |
| 模型跑偏 | 答案不按要求 | 补充约束、例子、验收标准 |

## 回滚和恢复

OpenCode 提供 `/undo` 和 `/redo`，适合撤销会话内修改。但它不是 Git 的替代品。

推荐恢复顺序：

1. 查看当前 diff。
2. 如果是最近一次 OpenCode 修改，优先 `/undo`。
3. 如果涉及多次修改，用 Git 分块检查。
4. 对用户已有改动，不要直接覆盖。
5. 回滚后让 OpenCode总结错误原因和新的约束。

## 配置排障

```bash
opencode debug config
```

检查重点：

- 最终 `model` 是否正确。
- `permission` 是否被项目或托管配置覆盖。
- `shell` 是否符合当前系统。
- MCP、插件、Agent 是否从预期目录加载。
- `share`、server、CORS 等是否符合安全要求。

## 成本控制

- 使用 `small_model` 处理标题、总结等轻量任务。
- 大任务分阶段，避免一次性塞入太多文件。
- 用 `@file` 精确引用，不要让 Agent 大范围搜索。
- 限制 MCP server 数量。
- 对子 Agent 输出要求简洁结构化。
- 让 Agent 先列计划，不要反复无效执行。

## 质量控制

每次实现后要求输出：

```text
请给出变更总结：
1. 修改文件
2. 行为变化
3. 已运行验证
4. 未验证内容
5. 剩余风险
```

## 练习

模拟以下故障并写排障记录：

- 权限拒绝 `bash`。
- 模型读错文件导致实现错误。
- MCP 工具太多导致响应变慢。
- 测试命令在你的 shell 中不可用。
- OpenCode 修改了不该修改的文件。

## 验收

- 能用 `/undo` 恢复一次错误修改。
- 能用 `debug config` 判断配置来源。
- 能给出三种降低 token 成本的方法。
- 能写出一次 AI 修改的风险总结。

## 重点

- 排障先分类，不要直接怪模型。
- 回滚要尊重用户已有改动和 Git 状态。
- 成本控制本质是上下文控制和工具控制。

## 难点

- 多层配置、插件和 MCP 同时存在时，问题可能来自隐式加载。需要检查最终配置和启动日志。

## 易错

> **易错：** 测试失败后让 OpenCode 盲目“继续修”。
>
> 正确做法：要求它先解释失败原因、关联改动、最小修复方案，再执行。
