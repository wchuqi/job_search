# OpenCode 学习资料：配置解析、覆盖顺序和版本治理深度解析

[返回索引](../OpenCode学习资料.md)

## 学习目标

- 深入理解配置加载、合并、覆盖和排障。
- 能为个人、项目、组织设计配置分层。
- 能处理 OpenCode 版本升级带来的字段变更和行为变化。

## 理论导读

OpenCode 配置不是“读取一个 JSON 文件”这么简单。它会从多个来源加载配置，并把它们合并成最终运行配置。后加载配置覆盖同名字段，但不删除不冲突字段。

这带来两个结果：

- 优点：组织、项目、个人可以分层管理。
- 风险：实际生效配置可能不是你眼前看到的那一个文件。

## 配置解析流程

可以把配置解析理解为流水线：

```text
默认内置配置
  -> 组织远程配置
  -> 用户全局配置
  -> 自定义配置路径
  -> 项目根配置
  -> .opencode/ 目录内容
  -> 环境变量内联配置
  -> 管理员托管配置
  -> 最终运行配置
```

具体加载源以官方文档和当前版本为准，但排障思路稳定：永远检查最终解析结果。

## 合并规则的影响

假设全局配置：

```jsonc
{
  "model": "provider-a/model-x",
  "permission": {
    "*": "ask",
    "read": "allow"
  }
}
```

项目配置：

```jsonc
{
  "permission": {
    "bash": {
      "*": "ask",
      "pnpm test*": "allow"
    }
  }
}
```

最终效果通常不是“项目配置完全替换全局配置”，而是合并后同时存在 `model`、`read`、`bash` 等字段。

> **重点：** 配置合并让默认策略可复用，但也可能让旧配置残留生效。

## 配置分层设计

| 层级 | 应放内容 | 不应放内容 |
| --- | --- | --- |
| 托管配置 | 强制安全、禁用分享、限制 Provider | 个人主题 |
| 组织远程配置 | 默认权限、通用 Agent、推荐命令 | 项目私有路径 |
| 项目配置 | 测试命令、目录边界、项目 MCP | 个人 API key |
| 个人配置 | 默认模型、TUI 偏好、个人快捷命令 | 团队强制规则 |
| 临时内联配置 | CI 或一次性实验 | 长期规则 |

## Schema 和 JSONC

建议所有配置加 `$schema`：

```jsonc
{
  "$schema": "https://opencode.ai/config.json"
}
```

价值：

- 编辑器能提示字段。
- 版本变化时更容易发现废弃字段。
- 团队 review 时能降低拼写错误。

JSONC 支持注释，适合解释权限意图：

```jsonc
{
  "$schema": "https://opencode.ai/config.json",
  // 默认询问，项目只放行只读和测试命令
  "permission": {
    "*": "ask"
  }
}
```

## 版本治理

OpenCode 仍在快速演进，团队使用时要考虑：

- 固定 CLI 安装方式。
- 记录推荐版本或升级窗口。
- 升级前用样例仓库回归权限、命令、MCP 和插件。
- 关注官方文档中废弃字段，例如旧 `tools` 配置与新 `permission` 模型的迁移。
- 对项目配置做定期清理，避免旧字段误导新人。

## 配置排障剧本

### 现象：模型不是预期模型

检查：

1. `opencode debug config`。
2. 全局配置是否设置了 `model`。
3. 项目配置是否覆盖。
4. CLI 启动参数是否覆盖。
5. 托管配置是否强制限制。

### 现象：权限规则没生效

检查：

1. 工具名是否写对。
2. 规则是否被更晚配置覆盖。
3. 对象语法是否存在后续通配规则。
4. Agent 级权限是否覆盖或收紧。
5. 当前命令是否由 MCP 或自定义工具触发。

### 现象：MCP 或插件未加载

检查：

1. 配置文件路径。
2. `.opencode/` 是否在项目根。
3. `enabled` 是否为 true。
4. 本地命令是否能单独运行。
5. 依赖是否已安装。

## 练习

设计一个团队配置方案：

- 托管配置禁止分享和部署命令。
- 组织配置允许只读工具和测试命令。
- 项目配置添加项目专属命令和 Agent。
- 个人配置只保留 TUI 和默认模型。
- 写一份升级检查清单。

## 验收

- 能解释配置为什么会“看着没写但实际生效”。
- 能用最终配置定位覆盖来源。
- 能设计安全的配置分层。
- 能处理字段迁移和版本升级。

## 重点

- 配置排障只看单个文件是不够的。
- 团队配置要区分推荐值和强制值。
- 版本升级必须验证权限和插件行为。

## 难点

- 合并规则会让旧配置长期残留。需要定期用最终配置反推哪些规则仍有必要。

## 易错

> **易错：** 把项目 `opencode.jsonc` 当作唯一真相。
>
> 正确做法：以 `debug config` 的最终结果为真相，再回溯来源。
