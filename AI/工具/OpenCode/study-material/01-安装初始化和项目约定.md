# OpenCode 学习资料：安装、初始化和项目约定

[返回索引](../OpenCode学习资料.md)

## 学习目标

- 掌握 OpenCode 的安装入口和 Windows 使用注意事项。
- 理解 Provider/API key、模型和项目初始化的关系。
- 会维护项目级 `AGENTS.md` 和基础配置。

## 理论导读

OpenCode 的基本落地顺序是：安装 CLI -> 配置模型 Provider -> 进入项目 -> 启动 TUI -> `/init` 生成项目说明 -> 用 Plan/Build 处理任务。

官方文档建议 Windows 用户优先使用 WSL，因为终端体验、shell 行为和工具兼容性更接近 Linux/macOS。也可以通过 Chocolatey、Scoop、npm 或 Docker 使用，但复杂项目开发时 WSL 更稳。

## 安装方式

常见方式：

```bash
curl -fsSL https://opencode.ai/install | bash
npm install -g opencode-ai
brew install anomalyco/tap/opencode
```

Windows 常见方式：

```powershell
choco install opencode
scoop install opencode
npm install -g opencode-ai
```

> **重点：** 实际安装命令以官方文档为准；团队环境应固定安装方式和版本升级策略。

## 初始化流程

```bash
cd /path/to/project
opencode
```

在 TUI 中连接 Provider：

```text
/connect
```

初始化项目：

```text
/init
```

初始化会让 OpenCode 分析项目，并在根目录生成 `AGENTS.md`。这个文件应提交到 Git，因为它是团队共享的项目上下文入口。

## AGENTS.md 应写什么

`AGENTS.md` 不是给人看的 README 替代品，而是给 Agent 的工作说明。建议包含：

- 项目结构：主要目录、入口、核心模块。
- 开发命令：安装依赖、启动、测试、lint、构建。
- 代码规范：命名、分层、错误处理、日志规范。
- 修改边界：哪些文件不能自动改，哪些改动必须先计划。
- 测试要求：不同改动对应的验证命令。
- 安全规则：不要读取或输出密钥，不要改生产配置，不要自动执行危险命令。

## 项目基础结构示例

```text
project/
  AGENTS.md
  opencode.jsonc
  .opencode/
    commands/
      test.md
      review-changes.md
    agents/
      docs.md
      security-auditor.md
```

## 例子：AGENTS.md 片段

```markdown
# Project Instructions

## Commands

- Install: `pnpm install`
- Dev: `pnpm dev`
- Test: `pnpm test`
- Lint: `pnpm lint`

## Rules

- Do not edit `.env*`, deployment manifests, or generated files unless explicitly asked.
- For changes touching authentication, payments, permissions, or data deletion, start in Plan mode.
- After code changes, run the narrowest relevant test first, then broader checks when needed.
```

## 练习

为一个你熟悉的项目写 `AGENTS.md`，至少包含：

- 5 个核心目录说明。
- 4 条开发命令。
- 5 条代码修改规则。
- 3 条安全限制。
- 3 条测试验收规则。

## 验收

- 能安装或选择一种合适的安装方式。
- 能解释为什么 `AGENTS.md` 应该进 Git。
- 能写出让 OpenCode 正确理解项目的初始化说明。
- 能区分个人偏好配置和项目共享约定。

## 重点

- 初始化不是形式动作，它决定 Agent 对项目的第一印象。
- `AGENTS.md` 要写可执行规则，少写空泛价值观。

## 难点

- 对大型项目，`AGENTS.md` 要控制长度。它应该给导航和规则，不要复制完整文档。

## 易错

> **易错：** 在 `AGENTS.md` 中写“代码要优雅、要高质量”。
>
> 正确做法：写具体规则，例如“新增 API 必须补充请求校验和失败测试”。
