# OpenCode 学习资料：Agent 调度、子 Agent 协作和上下文压缩深度解析

[返回索引](../OpenCode学习资料.md)

## 学习目标

- 深入理解 primary agent、subagent 和 hidden agent 的协作边界。
- 能设计多 Agent 工作流，而不是让一个 Agent 包办所有任务。
- 能处理长会话的上下文压缩、摘要漂移和任务丢失。

## 理论导读

Agent 是“职责、提示词、模型、权限和工作模式”的组合。多 Agent 的价值不是“人数变多”，而是职责隔离：

- 主 Agent 管理任务目标和用户交互。
- 子 Agent 做专项探索或审查。
- 隐藏 Agent 做标题、摘要、压缩等辅助工作。

好的多 Agent 设计能提高质量；差的多 Agent 设计会增加成本、重复搜索、上下文污染和责任不清。

## 调度模型

```text
用户任务
  -> Primary Agent 理解目标
  -> 判断是否需要子任务
  -> 调用 subagent
  -> subagent 读取有限上下文并执行专项任务
  -> 返回结构化摘要
  -> Primary Agent 合并结论
  -> 继续计划或执行
```

## 子 Agent 适合什么

| 子任务 | 推荐 Agent | 原因 |
| --- | --- | --- |
| 大范围代码定位 | Explore | 只读、降低误改风险 |
| 外部库文档研究 | Scout | 把外部资料隔离出来 |
| 安全审查 | 自定义 security agent | 权限和关注点明确 |
| 文档更新 | 自定义 docs agent | 限制写入目录 |
| 性能分析 | 自定义 perf agent | 专注瓶颈和测量方法 |

## 子 Agent 不适合什么

- 需求尚不清晰时同时开多个子 Agent。
- 涉及强业务判断的最终决策。
- 需要共享大量隐私数据的任务。
- 需要写生产配置或执行部署的任务。
- 没有明确输出格式的探索任务。

## 输出协议

子 Agent 的输出要可合并。建议要求它返回：

```markdown
## 已确认事实

- ...

## 证据

- 文件：...
- 命令：...

## 假设

- ...

## 风险

- ...

## 建议下一步

- ...
```

> **重点：** 子 Agent 不能只给结论，必须给证据和不确定性。

## 上下文压缩问题

长会话中，OpenCode 可能使用摘要或压缩机制保留重点。压缩有价值，也有风险：

- 细节丢失：边界条件、失败日志、用户禁令被压缩掉。
- 语义漂移：摘要把假设写得像事实。
- 优先级错位：早期高优先级规则被弱化。
- 证据断裂：结论仍在，但文件和命令依据消失。

## 压缩前应保留什么

```markdown
## 当前任务目标
...

## 不可违反规则
...

## 已确认事实
...

## 已修改文件
...

## 已运行验证
...

## 未完成事项
...

## 风险和假设
...
```

## 多 Agent 工作流示例：安全改造

```text
1. Primary Plan：明确改造目标和范围。
2. Explore：定位认证、授权、中间件、路由守卫。
3. Scout：查框架官方安全建议。
4. security-auditor：审查当前方案风险。
5. Primary Plan：合并方案，拆分任务。
6. Build：只实现第一步。
7. security-auditor：审查 diff。
8. Build：补测试。
9. Primary：输出总结和未验证风险。
```

## 练习

为“升级认证库版本”设计多 Agent 流程：

- 哪些步骤只读？
- 哪些步骤查外部文档？
- 哪些步骤可以改文件？
- 哪些步骤需要人工确认？
- 子 Agent 输出格式是什么？

## 验收

- 能解释 primary agent 和 subagent 的职责边界。
- 能设计至少 3 个不同职责的自定义 Agent。
- 能要求子 Agent 输出证据、假设和风险。
- 能处理长会话压缩后的信息丢失问题。

## 重点

- 多 Agent 不是并发越多越好，而是职责隔离和权限隔离。
- 子 Agent 输出必须结构化，否则主 Agent 很难可靠合并。

## 难点

- Agent 之间没有天然一致的“世界观”。主 Agent 必须显式合并冲突事实。

## 易错

> **易错：** 让多个子 Agent 同时修改同一批文件。
>
> 正确做法：子 Agent 负责探索和审查，写操作由主流程串行执行。
