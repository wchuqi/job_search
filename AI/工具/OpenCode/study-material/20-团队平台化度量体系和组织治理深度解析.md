# OpenCode 学习资料：团队平台化、度量体系和组织治理深度解析

[返回索引](../OpenCode学习资料.md)

## 学习目标

- 能把 OpenCode 从个人效率工具推进到团队工程能力。
- 能设计度量体系，判断 AI coding 是否真正提升质量和效率。
- 能制定组织级安全、成本、合规和培训机制。

## 理论导读

团队引入 OpenCode 的核心问题不是“能不能生成代码”，而是：

- 是否缩短交付周期？
- 是否降低认知成本？
- 是否提高测试覆盖？
- 是否减少缺陷？
- 是否引入新的安全风险？
- 是否让新人更快理解项目？
- 是否可审计、可回滚、可治理？

没有度量，就无法判断它是在提效还是制造隐藏成本。

## 平台化分层

| 层级 | 能力 | 产物 |
| --- | --- | --- |
| 个人使用 | 单人提效 | 个人配置、常用 prompt |
| 项目规范 | 团队共享 | `AGENTS.md`、项目命令、权限 |
| 工具平台 | 可复用扩展 | MCP、插件、自定义工具 |
| 治理平台 | 安全合规 | 托管配置、审计、审批 |
| 度量平台 | 持续优化 | 指标、报表、复盘 |

## 度量指标

### 效率指标

- 从 issue 到 PR 的时间。
- 首次可运行实现时间。
- 重复任务自动化比例。
- 新人定位模块时间。
- 文档更新耗时。

### 质量指标

- AI 辅助 PR 的缺陷率。
- review 发现严重问题数量。
- 测试覆盖变化。
- 回滚率。
- 未验证变更比例。

### 安全指标

- 被拒绝的危险命令数量。
- 敏感文件读取尝试数量。
- MCP 外部写入次数。
- 会话分享审查通过率。
- 插件/MCP 审批覆盖率。

### 成本指标

- 每个 PR 的 token 成本。
- 长会话占比。
- MCP 工具调用成本。
- 高价模型使用比例。
- 无效重试次数。

> **重点：** 只看“写代码更快”会掩盖 review、修 bug、回滚和安全成本。

## 团队角色分工

| 角色 | 职责 |
| --- | --- |
| 平台负责人 | 维护组织配置、插件、MCP、版本策略 |
| 项目维护者 | 维护 `AGENTS.md`、命令、项目权限 |
| 安全负责人 | 审查密钥、外部工具、分享和审计 |
| 开发者 | 合理使用 Plan/Build，提交可审查 PR |
| Reviewer | 检查 AI 生成代码、测试和风险说明 |

## 推广路线

### 试点期

- 选择低风险项目。
- 禁止生产写操作。
- 只开放只读、测试、文档和小修复。
- 收集失败案例。

### 标准化期

- 输出项目模板。
- 固化权限基线。
- 建立命令库。
- 建立 PR 模板和培训材料。

### 平台化期

- 接入经过评审的 MCP。
- 建立审计和成本报表。
- 管理版本升级。
- 建立组织级托管配置。

### 优化期

- 根据指标淘汰无效命令。
- 调整模型和成本策略。
- 按团队反馈更新 AGENTS.md。
- 建立最佳实践案例库。

## PR 模板增强版

```markdown
## AI Assistance

- Tool: OpenCode
- Mode used: Plan / Build
- Key prompts or session summary:
- Files changed by AI:
- Human changes after AI:

## Verification

- Tests run:
- Tests not run:
- Manual checks:

## Risk Review

- Auth/security impact:
- Data/migration impact:
- Config/deployment impact:
- Sensitive information checked:
```

## 治理红线

- 不允许 AI 自动部署生产。
- 不允许 AI 自动修改密钥和生产配置。
- 不允许未审查 MCP 访问内部系统。
- 不允许共享包含敏感信息的会话。
- 不允许 AI 生成代码绕过 code review。
- 不允许把权限配置只写成自然语言提示词。

## 练习

为你的团队设计一个 OpenCode 落地方案：

1. 选择试点项目。
2. 制定 5 条安全红线。
3. 写项目模板目录。
4. 设计 10 个度量指标。
5. 制定升级和插件审批流程。
6. 设计新人培训任务。

## 验收

- 能区分个人提效和团队平台化。
- 能用指标评估 OpenCode 的真实价值。
- 能制定组织级安全红线。
- 能把使用经验沉淀成模板、命令、审计和培训。

## 重点

- 团队治理的目标是让 AI 使用可重复、可审查、可度量。
- 平台化必须同时考虑效率、质量、安全和成本。

## 难点

- AI 提效经常带来隐性 review 成本。必须用全流程指标衡量，而不是只看生成速度。

## 易错

> **易错：** 团队推广时只发一份“推荐 prompt”。
>
> 正确做法：同时提供配置模板、权限基线、命令库、PR 模板、审计机制和故障复盘机制。
