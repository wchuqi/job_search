## 一、基础概念

### 1. OpenCode 是什么？

**参考答案：**

OpenCode 是开源 AI coding agent。它不只是回答问题，还能在项目上下文中读取文件、搜索代码、制定计划、修改文件、运行命令，并通过配置、权限、Agent、命令和扩展接入团队工作流。

> **重点：** 关键词是 coding agent、代码库上下文、工具调用、权限控制。

### 2. OpenCode 和普通 AI 聊天有什么区别？

**参考答案：**

普通聊天主要依赖用户粘贴上下文，输出建议。OpenCode 直接运行在项目目录中，可以通过工具读取代码、搜索文件、执行命令和修改文件，所以更适合真实工程任务。区别也带来风险：OpenCode 必须管理权限、回滚、审查和敏感信息。

> **易错：** 只说“OpenCode 更懂代码”。更准确的是：它能访问工程环境并行动。

### 3. 为什么 `/init` 很重要？

**参考答案：**

`/init` 会让 OpenCode 分析项目并生成 `AGENTS.md`。这个文件给 Agent 提供项目结构、开发命令、代码规范、测试要求和安全限制。它应提交到 Git，让团队共享同一套项目约定。

### 4. Plan 和 Build 有什么区别？

**参考答案：**

Plan 用于分析、设计和方案评审，默认更受限，适合不改文件的任务。Build 用于真正开发和执行修改，权限更高。复杂或高风险任务应先 Plan，经人工确认后再 Build。

> **重点：** Plan 降低方向错误风险，Build 负责执行。

### 5. 你会如何让 OpenCode 修改一个已有功能？

**参考答案：**

我会先让它只读分析相关文件和调用链，然后进入 Plan 模式输出修改方案、涉及文件和测试清单。人工确认方案后，只让它实现一个小步骤，并运行相关测试。最后检查 diff、要求总结风险，再走 PR 审查。

## 二、工作流场景

### 6. 如何写一个高质量 OpenCode 请求？

**参考答案：**

高质量请求包含目标、范围、上下文、约束、输出格式和验收标准。例如：

```text
请分析 @src/auth/session.ts 的登录状态管理。
不要修改文件。
输出：职责、调用链、状态变化、潜在风险、建议测试点。
```

### 7. 如果 OpenCode 修改方向错了怎么办？

**参考答案：**

先查看 diff 判断影响范围。如果是最近一次 OpenCode 修改，可以用 `/undo` 撤销。然后补充约束，让它解释错误原因并重新制定更小范围的计划。重要项目仍以 Git 分支、commit 和 code review 作为最终回滚保障。

### 8. 什么时候不应该直接让 OpenCode Build？

**参考答案：**

涉及鉴权、支付、数据删除、迁移、生产配置、安全漏洞、部署发布、大范围重构时，不应直接 Build。应先 Plan，必要时要求只读审查 Agent 参与，再人工确认。

> **易错：** 把“可以回滚”当成“可以随便试”。回滚不能保证没有泄露、外部副作用或数据影响。
