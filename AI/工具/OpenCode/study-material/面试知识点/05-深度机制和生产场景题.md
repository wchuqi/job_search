## 一、执行机制

### 1. 请解释 OpenCode 从用户输入到修改代码的完整链路。

**参考答案：**

一次 OpenCode 执行通常经历：用户输入、会话上下文组装、Agent 规则注入、模型生成下一步、工具调用候选、权限匹配、工具执行、结果回写上下文、模型继续推理、输出 diff 或总结、人工验证或回滚。模型不是直接改文件，真正行动通过工具系统完成，工具调用又要经过权限系统。

> **重点：** 中高级答案要提到“工具调用生命周期”和“权限检查”，不能只说“模型理解后生成代码”。

### 2. 如果 OpenCode 读错文件，为什么后续会连续错？

**参考答案：**

因为工具读取结果会被写回上下文，模型把它当作后续推理的事实来源。如果第一轮搜索或读取定位错了，后续计划、修改和测试都可能围绕错误文件展开。解决方法是要求它先输出已确认事实、证据文件、假设和未确认点，复杂任务先只读探索。

> **难点：** Agent 的错误会在会话状态中累积，不是每一步都从零开始。

### 3. 如何判断一次失败发生在哪个阶段？

**参考答案：**

看现象分类：回答泛泛通常是上下文不足；改错文件是计划和范围问题；命令被拒是权限问题；测试失败可能是实现或环境问题；响应慢可能是上下文或 MCP 工具过多；最终总结不可信通常是缺少验证。排障要回到工具调用、diff 和命令输出。

## 二、配置和权限

### 4. 为什么项目配置里没写某个权限，但它仍然生效？

**参考答案：**

因为 OpenCode 配置是多来源合并，后加载配置覆盖冲突字段，但不会删除不冲突字段。某个权限可能来自全局配置、组织远程配置、内联配置或托管配置。应使用 `opencode debug config` 查看最终配置，而不是只看项目文件。

### 5. 权限通配规则为什么会误放行危险命令？

**参考答案：**

对象规则会按模式匹配，最后一个匹配规则生效。如果先写 `git reset*: deny`，后写 `git *: allow`，危险命令可能被宽泛规则覆盖。安全写法是宽泛规则在前，具体危险拒绝规则在后，并尽量只放行具体只读命令。

### 6. 如何设计生产事故排障模式的权限？

**参考答案：**

生产事故排障应默认只读。允许读代码、搜索、查看日志摘要、运行只读诊断命令；拒绝写文件、部署、重启服务、数据库写入、密钥读取。需要写操作时必须脱离 OpenCode 自动执行，走人工审批和变更流程。

## 三、Agent 和上下文

### 7. 多 Agent 协作的核心风险是什么？

**参考答案：**

风险是职责不清、重复搜索、结论冲突、上下文膨胀和子 Agent 擅自修改。正确做法是让子 Agent 做只读探索或专项审查，输出事实、证据、假设、风险和下一步，由 primary agent 合并结论，写操作保持串行。

### 8. 长会话上下文压缩会带来什么问题？

**参考答案：**

压缩可能丢失细节、把假设写成事实、弱化用户禁令、断开证据链。长任务应在压缩前固定当前目标、不可违反规则、已确认事实、已修改文件、已运行验证、未完成事项和风险假设。

### 9. `AGENTS.md` 应该如何避免过度膨胀？

**参考答案：**

`AGENTS.md` 应放项目导航、关键命令、强约束和高风险规则，不应复制完整 README 或所有规范。细节可以放到命令模板、专门文档或按需文件引用中。过长会增加噪声，反而降低模型遵循能力。

## 四、扩展和供应链

### 10. 为什么 MCP 不是越多越好？

**参考答案：**

MCP 工具会占用上下文，增加模型选择工具的复杂度，也可能扩大外部系统权限和数据泄露风险。应按任务启用最少工具，对每个 MCP 评审数据访问、写操作、凭据、权限、输出过滤、审计和禁用方案。

### 11. 一个安全的自定义工具应该具备哪些特征？

**参考答案：**

应有明确描述、严格参数 schema、allowlist、最小输出、敏感字段过滤、错误可解释、调用审计、默认只读和权限规则。不能把“执行任意命令”封装成自定义工具，否则等价开放 shell。

### 12. 插件为什么比命令更需要审查？

**参考答案：**

插件可能在生命周期 hook 中隐式运行，读取环境变量、注入上下文、发送网络请求或新增工具。它的行为不一定像命令那样显式触发，所以必须审查来源、版本、权限、数据流和禁用方案。

## 五、团队治理和度量

### 13. 如何证明 OpenCode 在团队里真的提效？

**参考答案：**

不能只看生成代码速度，要看全流程指标：issue 到 PR 时间、首次可运行实现时间、review 缺陷率、回滚率、测试覆盖变化、未验证变更比例、危险命令拦截次数、会话成本、无效重试次数。效率、质量、安全、成本要一起看。

### 14. OpenCode 团队落地的最低闭环是什么？

**参考答案：**

最低闭环包括：项目 `AGENTS.md`、项目 `opencode.jsonc` 权限基线、常用命令、只读审查 Agent、PR 模板、敏感信息规则、MCP/插件审批、故障复盘模板和新人培训任务。

### 15. 如果一次 AI 辅助修改引入线上 bug，复盘应该看什么？

**参考答案：**

要看任务输入是否清楚、Agent 是否读对上下文、计划是否经人工确认、权限是否过宽、测试是否覆盖、review 是否检查 AI 变更、是否有未验证风险说明、回滚是否及时。复盘结果要落到权限、命令、`AGENTS.md`、PR 模板和培训，不应只归因于模型。

> **重点：** 中高级候选人应能把 AI failure 当成工程系统 failure 来分析。
