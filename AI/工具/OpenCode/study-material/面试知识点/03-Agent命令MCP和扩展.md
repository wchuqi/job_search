## 一、Agent

### 1. primary agent 和 subagent 有什么区别？

**参考答案：**

Primary agent 是用户直接对话的主 Agent，用于推进主要任务。Subagent 是专项助手，可以由主 Agent 调用，也可以通过 `@` 提及，用于代码探索、外部文档研究、安全审查等子任务。

### 2. Build、Plan、Explore、Scout 分别适合什么？

**参考答案：**

Build 适合实际开发和修改。Plan 适合分析和制定方案。Explore 适合只读代码探索。Scout 适合外部文档和依赖源码研究。选择 Agent 的本质是选择任务边界和权限边界。

### 3. 如何设计一个安全审查 Agent？

**参考答案：**

安全审查 Agent 应该只读，允许读取文件、搜索代码和查看 diff，禁止编辑和执行危险命令。提示词聚焦鉴权、密钥、数据删除、注入、权限绕过和外部调用。

> **重点：** 审查 Agent 默认不修复，只输出发现；修复由开发流程另行确认。

## 二、自定义命令

### 4. 自定义命令解决什么问题？

**参考答案：**

自定义命令把重复 prompt 固化为可版本管理的文件，便于团队复用和审查。例如 review、测试失败分析、生成测试、更新文档、安全审查。

### 5. 命令模板中有哪些动态能力？

**参考答案：**

可以使用 `$ARGUMENTS` 或位置参数接收输入，可以用 `@file` 引用文件，也可以注入 shell 命令输出。shell 输出能力要谨慎使用，因为它会执行真实命令。

### 6. 命令和 `AGENTS.md` 的边界是什么？

**参考答案：**

`AGENTS.md` 写长期项目规则，例如结构、命令、规范和禁止事项。命令写具体任务模板，例如审查当前 diff、分析测试失败。强制安全边界应放权限配置，而不是只写在自然语言规则里。

## 三、MCP、自定义工具和插件

### 7. MCP 适合解决什么问题？

**参考答案：**

MCP 适合接入外部工具或数据源，例如 issue、错误监控、文档检索、代码搜索、数据库元数据。OpenCode 支持本地和远程 MCP。MCP 工具会增加上下文和权限风险，应按需启用。

### 8. 自定义工具和 MCP 有什么区别？

**参考答案：**

自定义工具通常是项目或个人定义的本地函数，适合封装内部脚本或受控查询。MCP 是标准协议，更适合连接外部工具生态。两者都要做参数校验、权限控制和审计。

### 9. 插件适合什么场景？

**参考答案：**

插件适合挂钩 OpenCode 事件，例如发送通知、记录日志、保护 `.env`、注入环境变量、扩展工具。插件可能引入隐式行为和供应链风险，因此应评审来源和版本。

### 10. 如何判断是否应该接入一个 MCP server？

**参考答案：**

看它是否高频、必要、可控、最小权限、输出可验证、上下文成本可接受。如果只是偶尔查资料，直接人工查或临时 web 查询可能更简单。

> **易错：** 把所有 MCP 都打开。工具过多会让模型选择变差，并增加 token 成本和安全风险。
