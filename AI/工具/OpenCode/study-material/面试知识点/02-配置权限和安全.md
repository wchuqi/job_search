## 一、配置

### 1. OpenCode 配置有哪些层级？

**参考答案：**

常见层级包括远程组织配置、用户全局配置、自定义配置、项目配置、`.opencode` 目录、内联配置和管理员托管配置。它们会合并，后加载配置只覆盖冲突字段，不会删除不冲突字段。

> **难点：** 排障时要看最终解析配置，不能只看项目根目录的文件。

### 2. 项目配置和全局配置分别适合放什么？

**参考答案：**

全局配置适合个人偏好，例如默认模型、主题、快捷键。项目配置适合团队共享内容，例如权限基线、项目命令、Agent、MCP 和测试规则。涉及强制安全策略时，应使用组织或托管配置。

### 3. 如何检查最终配置？

**参考答案：**

使用：

```bash
opencode debug config
```

重点检查模型、权限、shell、MCP、插件、server、share 等字段。

## 二、权限

### 4. `permission` 的三种动作是什么？

**参考答案：**

`allow` 表示直接执行，`ask` 表示执行前询问用户，`deny` 表示阻止执行。可以全局设置，也可以按工具和输入模式设置。

### 5. 如何设计安全的 bash 权限？

**参考答案：**

默认 `ask`，允许低风险命令，例如 `git status`、`git diff`、测试命令；拒绝删除、重置、部署、输出密钥等危险命令。

```jsonc
{
  "permission": {
    "bash": {
      "*": "ask",
      "git status*": "allow",
      "git diff*": "allow",
      "pnpm test*": "allow",
      "rm *": "deny"
    }
  }
}
```

### 6. 权限匹配有什么容易踩坑的地方？

**参考答案：**

对象语法中规则按模式匹配，最后一个匹配规则生效。宽泛规则应放前面，具体规则放后面。否则可能被后续宽泛规则覆盖。

> **易错：** 以为第一个匹配规则生效。

### 7. 为什么不能只控制 bash？

**参考答案：**

因为 OpenCode 的文件修改可以通过 `edit`、`write`、`apply_patch` 完成。只限制 bash 不能阻止文件被改写。文件修改要统一控制 `edit` 权限，并配合 Git diff 和 review。

## 三、安全

### 8. 如何防止密钥泄露？

**参考答案：**

不要把密钥写进 `AGENTS.md`、`opencode.jsonc` 或命令模板。限制读取 `.env*`、证书和生产配置。分享会话前检查是否包含敏感信息。MCP 和插件访问外部系统时使用最小权限凭据，并记录审计。

### 9. 什么是外部目录风险？

**参考答案：**

OpenCode 默认在启动目录内工作。如果允许访问外部目录，Agent 可能读取或修改不属于当前项目的文件。应通过 `external_directory` 精确允许路径，并对 read/edit/bash 分别限制。

### 10. 你会给团队设置什么默认权限？

**参考答案：**

默认 `*` 为 `ask`；允许 read、grep、glob；编辑需要确认；测试和只读 Git 命令可允许；删除、部署、重置、密钥读取、生产配置修改全部拒绝或强制审批。高风险项目应默认 Plan。
