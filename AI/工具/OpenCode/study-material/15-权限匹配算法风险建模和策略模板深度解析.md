# OpenCode 学习资料：权限匹配算法、风险建模和策略模板深度解析

[返回索引](../OpenCode学习资料.md)

## 学习目标

- 深入理解 `permission` 的匹配模型。
- 能按风险等级设计权限模板。
- 能把命令、文件、目录、MCP、自定义工具纳入统一安全模型。

## 理论导读

OpenCode 权限不是简单的开关。它是一个基于工具名、输入模式和动作结果的策略系统。策略结果通常是：

- `allow`：无需确认执行。
- `ask`：向用户确认。
- `deny`：拒绝执行。

权限系统的目标不是“阻止 Agent 做事”，而是把行动限制在可接受风险内。

## 权限匹配心智模型

```text
工具调用请求
  -> 识别工具名
  -> 提取输入，如 bash 命令或文件路径
  -> 查找权限规则
  -> 通配符匹配
  -> 后匹配规则覆盖前匹配规则
  -> 得到 allow / ask / deny
  -> 执行、询问或拒绝
```

## 为什么最后匹配重要

示例一：

```jsonc
{
  "permission": {
    "bash": {
      "*": "ask",
      "git *": "allow",
      "git reset*": "deny"
    }
  }
}
```

`git reset --hard` 同时匹配 `*`、`git *`、`git reset*`，最后命中 `deny`，所以拒绝。

示例二：

```jsonc
{
  "permission": {
    "bash": {
      "git reset*": "deny",
      "git *": "allow"
    }
  }
}
```

如果最后匹配规则生效，那么 `git reset --hard` 可能被后面的 `git *` 放行。宽泛规则应放前，危险规则放后。

> **重点：** 权限规则顺序是安全设计的一部分。

## 风险分级

| 风险等级 | 动作 | 推荐策略 |
| --- | --- | --- |
| 低 | 读普通代码、`git status`、`git diff` | allow |
| 中 | 改业务代码、运行测试、安装依赖 | ask |
| 高 | 删除文件、重置 Git、数据库迁移、发布部署 | deny 或强制人工审批 |
| 极高 | 读取密钥、修改生产配置、访问客户数据 | deny |

## 文件风险模型

| 文件类型 | 风险 | 策略 |
| --- | --- | --- |
| `src/**` | 业务逻辑变更 | edit ask |
| `tests/**` | 测试变更 | edit ask 或 allow |
| `docs/**` | 文档变更 | 可按 Agent 放行 |
| `.env*` | 密钥泄露 | read/edit deny |
| `infra/**` | 基础设施 | edit deny 或审批 |
| `migrations/**` | 数据风险 | edit ask，执行 deny |
| generated files | 噪声和覆盖 | edit deny |

## 策略模板：学习只读模式

```jsonc
{
  "$schema": "https://opencode.ai/config.json",
  "permission": {
    "*": "deny",
    "read": "allow",
    "grep": "allow",
    "glob": "allow",
    "lsp": "allow",
    "webfetch": "ask",
    "websearch": "ask"
  }
}
```

适合新人熟悉项目或面试复习。

## 策略模板：日常开发模式

```jsonc
{
  "$schema": "https://opencode.ai/config.json",
  "permission": {
    "*": "ask",
    "read": "allow",
    "grep": "allow",
    "glob": "allow",
    "bash": {
      "*": "ask",
      "git status*": "allow",
      "git diff*": "allow",
      "pnpm test*": "allow",
      "npm test*": "allow",
      "rm *": "deny",
      "del *": "deny",
      "git reset*": "deny",
      "git push*": "deny",
      "kubectl *": "deny",
      "terraform apply*": "deny"
    },
    "edit": {
      ".env*": "deny",
      "infra/**": "deny",
      "migrations/**": "ask",
      "**": "ask"
    }
  }
}
```

## 策略模板：文档 Agent

```jsonc
{
  "agent": {
    "docs": {
      "mode": "subagent",
      "permission": {
        "read": "allow",
        "grep": "allow",
        "glob": "allow",
        "edit": {
          "docs/**": "allow",
          "**": "deny"
        },
        "bash": "deny"
      }
    }
  }
}
```

## MCP 和自定义工具权限

MCP 工具和自定义工具也应进入权限模型。策略原则：

- 工具命名要可识别风险，例如 `github_create_issue`、`db_read_schema`。
- 对只读工具可 `allow`，对写入外部系统工具默认 `ask` 或 `deny`。
- 数据库、工单、云平台、CI/CD 工具要单独审批。
- 对工具输出限制大小，避免把大量敏感数据写入上下文。

## 练习

为以下场景写权限：

1. 新人只读学习。
2. 后端日常开发。
3. 安全审查 Agent。
4. 文档维护 Agent。
5. 生产事故只读排障。

每套策略说明：

- 允许什么。
- 询问什么。
- 拒绝什么。
- 为什么。

## 验收

- 能解释权限匹配顺序。
- 能把命令和文件路径纳入同一风险模型。
- 能为不同 Agent 设计不同权限。
- 能识别通配规则的安全漏洞。

## 重点

- 权限策略必须和任务角色绑定。
- 拒绝规则应覆盖危险命令、敏感文件、外部目录和外部系统写操作。

## 难点

- 通配规则太宽会误放行，太窄会影响效率。要用真实任务日志迭代策略。

## 易错

> **易错：** 写了 `git *: allow`，却忘记 `git reset`、`git push`、`git clean` 都很危险。
>
> 正确做法：低风险只放行具体只读命令，如 `git status*`、`git diff*`。
