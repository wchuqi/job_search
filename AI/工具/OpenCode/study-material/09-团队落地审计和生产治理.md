# OpenCode 学习资料：团队落地、审计和生产治理

[返回索引](../OpenCode学习资料.md)

## 学习目标

- 能设计团队级 OpenCode 使用规范。
- 理解共享会话、权限、托管配置和审计的价值。
- 会把 OpenCode 接入代码审查、测试、CI 和发布流程。

## 理论导读

团队落地 OpenCode 的目标不是让每个人随意让 AI 改代码，而是把 AI 能力纳入现有工程治理。AI 生成代码必须经过同样的 review、测试、安全检查和发布审批。

治理重点包括：配置分层、权限默认值、敏感信息保护、变更审计、模型成本、会话共享边界、团队命令模板、项目初始化质量。

## 团队配置分层

| 层级 | 放什么 | 谁维护 |
| --- | --- | --- |
| 托管配置 | 强制安全策略、Provider 限制、分享限制 | 平台/安全团队 |
| 组织远程配置 | 推荐 MCP、默认权限、模型策略 | 平台团队 |
| 项目配置 | 项目命令、Agent、测试规则、目录限制 | 项目维护者 |
| 个人配置 | 主题、快捷键、个人默认模型 | 开发者 |

## 团队安全基线

- 默认不允许自动读取或输出密钥文件。
- 默认不允许删除、重置、发布、部署、修改生产配置。
- 复杂改动必须先 Plan。
- AI 生成代码必须提交 PR。
- 所有变更必须有测试或明确未测说明。
- MCP 和插件必须经过评审。
- 会话分享前必须确认不含敏感信息。

## PR 审查清单

```markdown
## AI-Assisted Change Checklist

- [ ] 是否说明使用了 OpenCode
- [ ] 是否列出关键 prompt 或会话链接
- [ ] 是否人工检查过核心逻辑
- [ ] 是否运行相关测试
- [ ] 是否检查过安全和权限影响
- [ ] 是否确认没有提交密钥和本地配置
- [ ] 是否更新文档或迁移说明
```

## 生产场景约束

对于以下场景，OpenCode 可以辅助分析，但不应自动完成最终决策：

- 数据删除或数据迁移。
- 鉴权、支付、风控和权限模型。
- 生产部署和回滚。
- 安全漏洞修复。
- 法务、隐私、合规相关代码。
- 基础设施和密钥管理。

## 审计建议

- 保留关键会话摘要，而不是完整泄露上下文。
- 在 PR 中写清 AI 修改范围和人工验证。
- 对自定义工具、MCP、插件做版本和权限记录。
- 对共享命令和 Agent 使用代码评审。
- 定期清理过时模型名、废弃配置和过宽权限。

## 练习

为一个 8 人研发团队制定 OpenCode 使用规范：

- 个人配置模板。
- 项目配置模板。
- 权限基线。
- PR 模板。
- 高风险场景审批规则。
- 新人培训任务。

## 验收

- 能说明 OpenCode 团队落地的最低安全基线。
- 能设计配置分层方案。
- 能把 OpenCode 输出纳入 PR 审查。
- 能识别哪些场景必须人工审批。

## 重点

- OpenCode 不能绕过工程制度，只能进入工程制度。
- 团队应审查 `.opencode/` 目录，就像审查 CI 配置一样。

## 难点

- 治理不能只靠口头规定，必须落到配置、权限、命令、PR 模板和审计记录。

## 易错

> **易错：** 只培训开发者如何“更快生成代码”。
>
> 正确做法：同时培训需求拆解、上下文选择、权限审批、测试验证和安全边界。
