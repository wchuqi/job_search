# OpenCode 学习资料：执行架构、会话状态和工具调用生命周期深度解析

[返回索引](../OpenCode学习资料.md)

## 学习目标

- 从运行机制理解 OpenCode：请求如何变成模型消息、工具调用、文件修改和验证结果。
- 能解释会话状态、上下文压缩、工具权限和回滚之间的关系。
- 能判断一次失败是发生在输入、规划、工具执行、权限、验证还是恢复阶段。

## 理论导读

OpenCode 的核心不是“模型直接改代码”，而是一条受控执行链：

```text
用户输入
  -> 会话上下文组装
  -> Agent 选择和系统规则注入
  -> 模型生成下一步
  -> 工具调用候选
  -> 权限匹配和用户审批
  -> 工具执行
  -> 工具结果写回上下文
  -> 模型继续推理
  -> 文件 diff / 命令结果 / 总结
  -> 人工验证或回滚
```

这条链路中，模型只负责“提出下一步”。真正的行动必须经过工具系统和权限系统。因此工程治理重点不是试图让模型永远不犯错，而是让错误被限制在可观察、可审批、可回滚的范围内。

## 核心心智模型

把 OpenCode 看成一个事件循环：

1. Agent 读取当前会话状态。
2. Agent 决定回答或调用工具。
3. 工具系统检查权限。
4. 工具执行产生观察结果。
5. 观察结果进入上下文。
6. Agent 基于新观察继续。

> **重点：** 每一次工具输出都会改变后续推理。错误日志、误读文件、过长输出、无关 MCP 工具描述都会污染下一步决策。

## 会话状态由什么组成

| 状态类型 | 内容 | 风险 |
| --- | --- | --- |
| 用户指令 | 当前任务、限制、验收 | 指令不完整会导致误解 |
| 项目规则 | `AGENTS.md`、Rules、Agent prompt | 过时规则会误导 |
| 配置状态 | 模型、权限、MCP、插件、shell | 多层配置覆盖难排查 |
| 工具观察 | 文件内容、搜索结果、命令输出 | 输出过多或错误会污染上下文 |
| 修改状态 | diff、创建文件、删除文件 | 可能覆盖用户变更 |
| 会话历史 | 之前计划、回答、审批记录 | 长会话会丢失重点 |

## 工具调用生命周期

### 1. 选择工具

模型根据系统提示、工具描述、任务目标和上下文决定是否调用工具。工具描述越宽泛，越容易被误用；工具越多，选择空间越大。

### 2. 构造参数

模型生成工具参数，例如文件路径、搜索模式、shell 命令。参数不是天然可信的，必须由工具实现、权限系统和用户审批共同约束。

### 3. 权限检查

OpenCode 根据 `permission` 判断是否 `allow`、`ask` 或 `deny`。如果是 `ask`，用户审批结果成为会话事实；如果是 `deny`，模型应改用低风险路径。

### 4. 执行和观察

工具执行后返回结果。结果可能是文件内容、搜索命中、命令输出、LSP 诊断、Web 内容或子 Agent 摘要。

### 5. 继续推理

模型根据工具结果决定下一步。此时最重要的是要求它保持“证据链”：它的结论应能回到具体文件、命令输出或测试结果。

## 文件修改为什么特别危险

文件修改不是一个孤立动作，它会影响：

- Git 工作区状态。
- 用户未提交改动。
- 测试结果。
- 后续模型阅读到的代码版本。
- 生产配置或生成物。
- 自动格式化和构建产物。

因此文件修改应有四个控制点：

```text
修改前：计划和范围
修改中：权限和路径限制
修改后：diff 审查和测试
异常时：undo / Git 回滚 / 手工恢复
```

## 例子：一次 bug 修复的执行链

```text
用户：修复登录失败后错误提示不显示的问题

1. Agent 搜索 login、error、toast。
2. 读取登录组件和 API 调用。
3. 发现 catch 分支吞掉 error。
4. 生成修改计划。
5. 用户确认进入 Build。
6. Agent 修改组件。
7. 运行相关测试。
8. LSP 或测试反馈类型错误。
9. Agent 修复类型。
10. 输出 diff、测试、风险。
```

如果第 2 步读错文件，后续所有判断都可能跑偏。因此复杂任务必须要求它先列出“已确认事实”和“假设”。

## 排障定位表

| 失败表现 | 可能阶段 | 处理方式 |
| --- | --- | --- |
| 回答泛泛而谈 | 上下文组装失败 | `@file` 精确引用，要求先读代码 |
| 修改不相关文件 | 计划和范围失败 | 先 Plan，限制文件路径 |
| 命令被拒绝 | 权限阶段 | 检查 `permission` 和 Agent 权限 |
| 测试反复失败 | 执行或理解失败 | 要求解释失败链路，不允许盲修 |
| 成本突然升高 | 工具选择或上下文膨胀 | 减少 MCP、拆任务、压缩总结 |
| 结果看似正确但未验证 | 验证阶段缺失 | 明确要求运行测试或说明未验证 |

## 练习

拿一次 OpenCode 修改记录，画出生命周期：

- 它读了哪些文件？
- 它用了哪些工具？
- 哪些工具经过审批？
- 哪一步形成关键判断？
- 哪一步做了验证？
- 如果要回滚，回滚点在哪里？

## 验收

- 能画出 OpenCode 从用户输入到工具执行的链路。
- 能解释工具结果如何影响后续推理。
- 能定位一次失败属于哪个阶段。
- 能为文件修改设计前置计划、权限、验证和恢复流程。

## 重点

- Agent 不是一次性回答器，而是工具驱动的状态循环。
- 工具输出是事实来源，也可能是污染源。
- 可靠性来自执行链路控制，不来自盲目信任模型。

## 难点

- 长任务中，上下文会不断变化。你要让 OpenCode显式维护“已确认事实、未确认假设、下一步验证”。

## 易错

> **易错：** 只看最终回答，不看它实际读了什么、改了什么、跑了什么。
>
> 正确做法：以工具调用和 diff 为事实依据，最终回答只是总结。
