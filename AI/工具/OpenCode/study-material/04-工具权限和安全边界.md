# OpenCode 学习资料：工具、权限和安全边界

[返回索引](../OpenCode学习资料.md)

## 学习目标

- 掌握 OpenCode 内置工具类型。
- 理解 `permission` 的 `allow`、`ask`、`deny`。
- 会设计细粒度权限策略，降低误操作和泄密风险。

## 理论导读

工具是 OpenCode 从“会说话”变成“会行动”的关键。内置工具包括 shell、文件读取、搜索、编辑、patch、LSP、Web、任务、技能等。工具越多，Agent 能力越强；权限越宽，风险越大。

OpenCode 使用 `permission` 控制动作是否自动执行、询问用户或直接拒绝。旧的 `tools` 布尔配置仍兼容，但官方文档已将控制重点放在 `permission`。

## 内置工具分类

| 工具 | 作用 | 风险 |
| --- | --- | --- |
| `read` | 读取文件 | 泄露密钥、读错上下文 |
| `grep` | 正则搜索内容 | 搜索范围过大、暴露敏感信息 |
| `glob` | 文件模式匹配 | 包含生成物或私密文件 |
| `edit` | 修改文件 | 改坏代码、覆盖用户变更 |
| `write` | 创建或覆盖文件 | 覆盖现有文件 |
| `apply_patch` | 应用补丁 | 大范围修改不可控 |
| `bash` | 执行命令 | 删除文件、联网、安装依赖、泄密 |
| `lsp` | 代码智能查询 | 实验能力，依赖配置 |
| `webfetch/websearch` | 获取外部资料 | 引入不可信资料 |
| `task` | 调用子 Agent | 放大上下文和行动面 |
| `skill` | 加载技能 | 扩展能力和复杂度 |

## 权限动作

```jsonc
{
  "$schema": "https://opencode.ai/config.json",
  "permission": {
    "*": "ask",
    "read": "allow",
    "grep": "allow",
    "glob": "allow",
    "edit": "ask",
    "bash": {
      "*": "ask",
      "git status*": "allow",
      "git diff*": "allow",
      "pnpm test*": "allow",
      "npm test*": "allow",
      "rm *": "deny",
      "del *": "deny"
    }
  }
}
```

## 匹配规则

权限对象支持通配符。规则按模式匹配，最后一个匹配规则生效。因此建议把宽泛规则放前面，具体规则放后面。

```jsonc
{
  "permission": {
    "bash": {
      "*": "ask",
      "git *": "allow",
      "rm *": "deny"
    }
  }
}
```

## 外部目录

OpenCode 默认工作在启动目录。需要访问外部目录时，应显式配置 `external_directory`，并对读取和修改分别控制。

```jsonc
{
  "$schema": "https://opencode.ai/config.json",
  "permission": {
    "external_directory": {
      "~/projects/personal/**": "allow"
    },
    "edit": {
      "~/projects/personal/**": "deny"
    }
  }
}
```

## 安全策略建议

- 默认 `*` 使用 `ask`，不要一开始全量 `allow`。
- 对 `bash` 做命令级规则，拒绝删除、重置、发布、部署、密钥输出。
- 对 `.env*`、证书、生产配置、迁移脚本设置更严格规则。
- MCP 和自定义工具启用后也要纳入权限管理。
- 高风险任务必须先 Plan，再人工确认。

## 练习

设计三套权限：

1. 只读学习模式：允许读、搜、LSP，拒绝写和 bash。
2. 开发模式：允许读搜，编辑需询问，测试命令允许，危险命令拒绝。
3. CI 辅助模式：只允许固定命令和固定输出，不允许交互式修改。

## 验收

- 能解释 `allow`、`ask`、`deny` 的差异。
- 能说明为什么 `edit` 覆盖多种文件修改行为。
- 能写出通配符权限，并解释最后匹配规则。
- 能识别 OpenCode 常见高风险工具调用。

## 重点

- 权限是 OpenCode 使用中的核心安全机制。
- 允许工具前要考虑数据、文件、命令、网络和外部目录五类风险。

## 难点

- 权限过紧会降低效率，过松会带来事故。团队应按任务类别维护不同配置模板。

## 易错

> **易错：** 只限制 `bash`，忽略 `edit`、`write` 和 `apply_patch`。
>
> 正确做法：把所有文件修改都归入变更审批和 Git 审查流程。
