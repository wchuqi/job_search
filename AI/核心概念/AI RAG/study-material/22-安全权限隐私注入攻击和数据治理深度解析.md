# AI RAG 学习资料：安全、权限、隐私、注入攻击和数据治理深度解析

[返回索引](../AI RAG学习资料.md)

## 学习目标

- 深入理解 RAG 的攻击面和防护边界。
- 掌握权限、隐私、提示注入、数据污染和审计治理。
- 能设计多租户企业 RAG 的安全架构。

## 理论导读

RAG 的安全风险来自“把外部内容送进模型上下文”。外部内容可能是网页、用户上传文档、工单、邮件或工具输出。它们都可能携带恶意指令、敏感信息或错误事实。

## 攻击面

| 攻击 | 入口 | 结果 |
| --- | --- | --- |
| Indirect prompt injection | 被检索文档 | 模型执行文档中的恶意指令 |
| Retrieval poisoning | 知识库投毒 | 检索到错误或恶意证据 |
| Data exfiltration | 用户构造问题 | 泄露无权限内容 |
| Membership inference | 追问是否包含某文档 | 暴露知识库存在性 |
| Citation laundering | 低可信来源伪装权威 | 答案看似有来源 |
| Tool escalation | 证据诱导工具调用 | 越权执行动作 |

## 权限模型

推荐权限字段：

```json
{
  "tenant_id": "t1",
  "resource_id": "doc123",
  "classification": "internal|confidential|restricted",
  "allowed_roles": ["finance"],
  "owner": "finance-dept",
  "region": "CN",
  "expiry": "2026-12-31"
}
```

权限过滤顺序：

```text
用户身份认证 -> 权限上下文生成 -> 检索前 filter -> 检索后硬校验 -> 打包前脱敏 -> 审计
```

## 注入防护

防护不是一句 prompt 能解决，需要多层：

- 数据准入：可信来源、扫描、审核。
- 证据隔离：把 evidence 标成不可信资料。
- 指令层级：外部文本不能覆盖系统规则。
- 工具隔离：模型不能直接执行高风险工具。
- 输出校验：检查是否泄露敏感字段。
- 红队评测：构造恶意文档和恶意问题。

## 数据治理

RAG 数据治理包括：

- 来源可信度。
- 文档 owner。
- 过期策略。
- 删除和撤回。
- PII 和密钥检测。
- 审计日志。
- 数据血缘。

## 练习

设计 10 条攻击样例：

- 5 条文档内注入。
- 2 条越权问题。
- 1 条数据污染。
- 1 条敏感信息诱导。
- 1 条工具升级。

并为每条写出防护层。

## 验收

- 能解释 indirect prompt injection。
- 能设计检索前权限过滤。
- 能处理多租户隔离。
- 能建立文档准入和撤回机制。

## 重点

RAG 安全的底线是：模型只能看到已授权、已治理、已隔离的证据。

## 难点

难点是“相关但无权限”的证据。系统要保证它不进入上下文，也不通过错误提示泄露存在性。

## 易错

> **易错：** 在 prompt 里写“不要泄露敏感信息”就认为安全。
>
> 正确做法：权限、脱敏和工具授权必须由应用层和数据层强制执行。
