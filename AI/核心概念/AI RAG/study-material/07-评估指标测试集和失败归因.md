# AI RAG 学习资料：评估指标、测试集和失败归因

[返回索引](../AI RAG学习资料.md)

## 学习目标

- 掌握 RAG 检索层和生成层的评估指标。
- 能构建 golden set 和失败归因表。
- 能定位 RAG 答错的具体环节。

## 理论导读

RAG 评估必须拆层。只看最终答案会掩盖问题：可能是文档缺失、切分错、召回漏、重排错、上下文太吵、模型不忠实，也可能是问题本身超出知识库。

## 核心心智模型

RAG 评估像检查流水线，每个工位都要有质检点：

```text
数据是否存在 -> 是否能被召回 -> 是否排在前面 -> 是否进入上下文 -> 是否被正确引用 -> 答案是否忠实
```

## 指标体系

### 检索指标

| 指标 | 含义 |
| --- | --- |
| recall@k | 正确证据是否出现在 top-k |
| precision@k | top-k 中相关证据比例 |
| MRR | 第一个正确证据排名是否靠前 |
| nDCG | 排名质量，考虑相关性等级 |
| evidence coverage | 答案所需证据维度是否覆盖 |

### 生成指标

| 指标 | 含义 |
| --- | --- |
| faithfulness | 答案是否忠实于证据 |
| answer relevance | 是否回答用户问题 |
| citation accuracy | 引用是否真实且支持结论 |
| refusal accuracy | 该拒答时是否拒答 |
| groundedness | 关键结论是否有依据 |

## Golden Set 设计

每条测试样例建议包含：

```json
{
  "question": "v2 认证方式有什么变化？",
  "expected_answer_points": ["使用 Authorization header", "query token 废弃"],
  "required_evidence_ids": ["auth-v2-sec-2", "migration-v2-sec-1"],
  "must_not_include": ["仍支持 query token"],
  "question_type": "comparison",
  "difficulty": "medium"
}
```

## 失败归因表

| 层级 | 症状 | 诊断 | 修复 |
| --- | --- | --- | --- |
| 数据 | 知识库没有正确文档 | 搜 source_uri | 补数据和版本治理 |
| 解析 | 表格或代码错乱 | 看原文与 chunk | 换解析器或定制规则 |
| 切分 | chunk 缺上下文 | 看命中片段 | 标题层级或父子 chunk |
| 召回 | 正确证据不在 top-k | 看 recall@k | query rewrite / hybrid |
| 重排 | 正确证据被排后 | 看 rerank 分数 | 调 reranker 或特征 |
| 打包 | 正确证据没进上下文 | 看 packing 日志 | 调预算和覆盖规则 |
| 生成 | 答案不忠实 | 看 evidence vs answer | 强化引用和拒答 |
| 安全 | 越权或注入 | 看权限和来源 | 前置过滤和隔离 |

## 练习

为一个 RAG 项目构建 30 条测试问题：

- 10 条简单事实题。
- 8 条对比题。
- 5 条多跳题。
- 4 条证据不足题。
- 3 条安全或越权题。

每条标注 required evidence。

## 验收

- 能拆分检索评估和生成评估。
- 能构建包含证据标注的测试集。
- 能根据 trace 定位失败层级。
- 能说明每次优化影响了哪个指标。

## 重点

RAG 评估必须评估“模型看到了什么”，不能只评估“模型说了什么”。

## 难点

难点是测试集维护。知识库会变，文档版本会变，golden set 也要版本化。

## 易错

> **易错：** 用少量人工感觉判断 RAG 好坏。
>
> 正确做法：建立可回放评测集，记录检索候选、分数、上下文和答案。
