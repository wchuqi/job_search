# AI RAG 学习资料：总览与心智模型

[返回索引](../AI RAG学习资料.md)

## 学习目标

- 建立 RAG 的整体心智模型。
- 理解 RAG 的输入、状态、证据、检索和生成边界。
- 能用一张数据流图解释 RAG 系统如何工作。

## 理论导读

RAG 的基本思想是：模型参数里没有、可能过期、需要可追溯的知识，不直接要求模型凭记忆回答，而是先从外部知识源检索相关证据，再把证据放入上下文，让模型基于证据生成答案。

它解决的不是“让模型更聪明”这个泛泛问题，而是解决三个具体问题：

- 知识更新：企业制度、API 文档、价格、政策经常变化，模型训练参数无法实时更新。
- 可追溯：很多业务场景要求回答能引用来源，方便复核。
- 私有知识：内部文档、工单、合同、数据库记录不在公开训练语料里。

## 核心心智模型

可以把 RAG 理解成“开卷考试系统”：

- 知识库是教材和资料库。
- 检索器是帮你翻书的人。
- 重排器是判断哪几页最相关的助教。
- 上下文打包器是把关键段落贴到试卷旁边的人。
- LLM 是答题者。
- 评估系统是批卷和复盘的人。

如果“翻书”翻错了，答题者再强也会答错。如果资料过期，回答会引用旧事实。如果试卷旁边贴了太多无关内容，模型会被噪声干扰。

## RAG 标准数据流

```text
用户问题
  -> 查询理解和改写
  -> 多路召回: BM25 / 向量 / SQL / 图 / Web
  -> 权限和元数据过滤
  -> 去重和重排
  -> 证据覆盖检查
  -> 上下文压缩和打包
  -> LLM 基于证据生成
  -> 引用校验和拒答检查
  -> 日志、评估和监控
```

## 知识点详解

### RAG 不等于向量库

向量库只解决“近似相似内容查找”的一部分问题。完整 RAG 还包括文档解析、切分、元数据、权限、检索策略、重排、上下文打包、引用、评估、安全和运维。

### RAG 不等于长上下文

长上下文可以塞入更多内容，但不等于选对内容。RAG 的重点是选择、排序和证据约束。长上下文和 RAG 常常配合使用：先检索候选，再把更大的父文档或结构化上下文交给模型。

### RAG 不等于微调

微调更适合学习输出风格、格式、任务模式和领域语言，不适合频繁变化、要求引用和权限隔离的事实知识。RAG 更适合动态知识和私有知识。

## 例子

场景：内部 HR 问答。

用户问：“试用期员工可以申请几天年假吗？”

错误做法：

```text
直接问模型，让模型根据常识回答。
```

更稳的做法：

```text
1. 检索 HR 制度中包含“试用期”“年假”的条款。
2. 过滤只允许当前员工查看的制度版本。
3. 找到最新生效版本。
4. 把条款作为 E1/E2 证据交给模型。
5. 要求模型引用证据回答；证据没有覆盖时拒答或建议联系人力。
```

## 练习

选择一个你熟悉的场景，例如 API 文档问答、客服知识库、合同审查、运维手册问答。写出：

- 知识源有哪些。
- 哪些知识会过期。
- 哪些知识需要权限控制。
- 用户最常问的 10 类问题。
- 每类问题需要什么证据。

## 验收

- 能画出 RAG 的端到端数据流。
- 能说明 RAG 系统每一层的职责。
- 能区分 RAG、微调、长上下文和搜索引擎。
- 能指出一个 RAG 答案失败可能来自哪一层。

## 重点

RAG 的本质是把“可验证证据”放进模型上下文，让模型在证据边界内回答。

## 难点

难点不在于调用向量库 API，而在于让系统长期稳定地命中正确证据、处理过期和冲突信息，并能被评估和复盘。

## 易错

> **易错：** 把 RAG 设计成“上传文档 -> 向量检索 -> 拼 prompt -> 回答”。
>
> 正确做法：把 RAG 设计成证据流水线，至少包含数据治理、检索、排序、打包、生成、引用、评估和安全。
