# AI RAG 学习资料：生成约束、引用校验、拒答和忠实性深度解析

[返回索引](../AI RAG学习资料.md)

## 学习目标

- 深入理解 RAG 生成阶段的可控性问题。
- 掌握引用校验、无证据结论检测和拒答策略。
- 能设计答案 schema、后处理和人工升级规则。

## 理论导读

RAG 的最后一步是生成，但生成不是自由发挥。模型的任务是把证据转成答案，并显式说明依据、缺口和不确定性。生成阶段常见失败包括：

- 使用证据外常识补全。
- 引用存在但不支持结论。
- 引用粒度过粗。
- 忽略证据中的否定或条件。
- 证据冲突时强行给结论。

## 忠实性拆解

一个 RAG 答案至少要过三道检查：

```text
claim extraction -> evidence support check -> citation alignment
```

### Claim extraction

把答案拆成可验证的原子结论：

```text
答案：v2 开始必须使用 Authorization header，query token 已废弃。

claims:
1. v2 必须使用 Authorization header。
2. query token 已废弃。
```

### Evidence support check

每个 claim 必须被至少一个 evidence 支持。支持不是“主题相关”，而是文本能推出该结论。

### Citation alignment

引用位置要对齐结论。不能在一个段落末尾放一个总引用，却让中间多个结论没有明确依据。

## 拒答决策

推荐拒答矩阵：

| 情况 | 行为 |
| --- | --- |
| 无相关证据 | 拒答并说明知识库无依据 |
| 部分证据缺失 | 给出可回答部分，并列缺口 |
| 证据冲突 | 展示冲突来源，要求人工或权威来源 |
| 权限不足 | 不透露内容，提示无权限 |
| 高风险建议 | 引用证据并建议人工确认 |

## 答案 Schema

```json
{
  "answer": "直接回答",
  "claims": [
    {
      "claim": "v2 必须使用 Authorization header",
      "evidence_ids": ["E1"],
      "support": "supported"
    }
  ],
  "missing_evidence": [],
  "conflicts": [],
  "confidence": "medium",
  "should_escalate": false
}
```

## 生成提示关键规则

```text
如果 evidence 没有覆盖问题，不要猜测。
如果 evidence 冲突，列出冲突并说明无法确定。
不要引用没有使用的 evidence。
每个关键结论都必须带 evidence id。
```

## 练习

准备 10 个答案，让你自己做 claim extraction，并标注：

- 哪些 claim 有证据。
- 哪些 claim 没证据。
- 引用是否对齐。
- 是否应该拒答或追问。

## 验收

- 能把答案拆成可验证 claims。
- 能区分 citation exists 和 citation supports。
- 能设计拒答矩阵。
- 能解释为什么“看起来正确”的答案仍可能不忠实。

## 重点

RAG 生成质量要按 claim 评估，而不是按整段答案感觉评估。

## 难点

难点是引用支持关系。引用相关不等于引用支持，尤其在对比、否定和时间变化问题中。

## 易错

> **易错：** 答案末尾列几个来源就算引用。
>
> 正确做法：关键结论逐条绑定 evidence id，并校验证据能推出该结论。
