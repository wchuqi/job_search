# AI RAG 学习资料：行业场景、架构设计案例库和方案评审清单

[返回索引](../AI RAG学习资料.md)

## 学习目标

- 能把 RAG 能力映射到真实业务场景。
- 掌握不同行业的知识源、风险和架构差异。
- 能评审一个 RAG 方案是否可落地。

## 案例 1：企业制度问答

### 知识源

- HR 制度。
- 财务制度。
- IT 手册。
- 法务公告。

### 架构重点

- 版本和生效时间。
- 员工角色权限。
- 引用制度条款。
- 证据不足时转人工。

### 高风险点

- 旧制度被检索。
- 分公司政策差异。
- 模型给出超出制度的承诺。

## 案例 2：技术文档助手

### 知识源

- API 文档。
- SDK README。
- Changelog。
- Issue 和 FAQ。
- 代码示例。

### 架构重点

- BM25 + 向量混合检索。
- 版本过滤。
- 代码块完整保留。
- 错误码精确匹配。

### 高风险点

- API 版本混淆。
- 代码块被切断。
- 引用旧 changelog。

## 案例 3：客服知识库

### 知识源

- 产品 FAQ。
- 工单。
- 用户手册。
- 政策规则。

### 架构重点

- 意图识别。
- 多轮上下文。
- 低置信度转人工。
- 用户隐私脱敏。

### 高风险点

- 给出错误退款承诺。
- 泄露其他用户信息。
- 只检索 FAQ，不查最新政策。

## 案例 4：运维排障助手

### 知识源

- Runbook。
- 告警文档。
- 日志查询。
- 配置管理。
- 变更记录。

### 架构重点

- 文档 RAG + SQL/日志工具。
- 只读工具默认。
- 操作建议分级。
- 高风险操作人工确认。

### 高风险点

- 把相关性当因果。
- 建议破坏性操作。
- 忽略最近变更。

## 方案评审清单

### 数据

- 知识源是否明确。
- 文档是否有 owner 和版本。
- 是否支持删除和撤回。
- 是否有权限字段。

### 检索

- 是否有评测集。
- 是否比较过 BM25、向量和 hybrid。
- 是否有 rerank。
- 是否处理多跳和对比问题。

### 生成

- 是否要求引用。
- 是否支持拒答。
- 是否有引用校验。
- 是否处理冲突证据。

### 安全

- 是否检索前权限过滤。
- 是否防 prompt injection。
- 是否脱敏。
- 是否审计。

### 生产

- 是否有 trace。
- 是否有监控。
- 是否有灰度和回滚。
- 是否有成本预算。

## 练习

选择一个你要落地的场景，按本文件模板输出一份架构设计，至少包括：

- 知识源。
- 用户问题类型。
- 检索策略。
- 权限策略。
- 评估集。
- 上线指标。

## 验收

- 能为不同业务选择不同 RAG 架构。
- 能指出行业场景的关键风险。
- 能用评审清单发现方案漏洞。

## 重点

RAG 架构必须服务业务风险，不同行业不能套同一个模板。

## 难点

难点是把“答案质量”转成业务可接受标准，例如可引用、可拒答、可转人工、可审计。

## 易错

> **易错：** 所有场景都用同一套 chunk、top-k 和 prompt。
>
> 正确做法：按知识结构、风险等级、权限模型和问题类型定制方案。
