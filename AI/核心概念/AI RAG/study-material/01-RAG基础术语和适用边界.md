# AI RAG 学习资料：RAG 基础术语和适用边界

[返回索引](../AI RAG学习资料.md)

## 学习目标

- 掌握 RAG 的核心术语。
- 能判断何时使用 RAG，何时不适合使用 RAG。
- 能解释 RAG 与微调、搜索、数据库查询和 Agent 的关系。

## 理论导读

RAG 是一种架构模式，不是单一算法。它把信息系统中的检索能力和大模型的语言生成能力组合起来。检索负责找证据，生成负责把证据组织成自然语言答案。

RAG 适合“答案依赖外部事实”的任务，不适合所有大模型任务。比如写邮件风格、分类标签、格式转换通常不需要 RAG；而制度问答、产品文档问答、法律条款摘要、知识库客服、技术排障文档问答通常适合 RAG。

## 核心术语

| 术语 | 含义 | 常见风险 |
| --- | --- | --- |
| Corpus | 原始知识集合 | 混入过期、重复、无权限文档 |
| Chunk | 被切分后用于检索的文本片段 | 切断语义、上下文不足 |
| Embedding | 文本的向量表示 | 模型不匹配领域语言 |
| Retriever | 从知识库找候选证据的组件 | 召回漏掉关键证据 |
| Reranker | 对候选证据重新排序的组件 | 排序慢、成本高 |
| Context | 最终给模型看的上下文 | 噪声过多或证据不足 |
| Citation | 答案引用的证据来源 | 引用不存在或引用不支持结论 |
| Faithfulness | 答案是否忠实于证据 | 模型补全证据外内容 |

## 适用场景

适合 RAG：

- 知识经常更新。
- 答案必须可引用、可复核。
- 有私有知识库。
- 用户问题需要从多个文档中合成答案。
- 需要按权限、版本、地区、产品线过滤知识。

不适合或需要谨慎：

- 答案主要依赖模型推理，不依赖外部事实。
- 知识库质量很差且无法治理。
- 延迟要求极低，无法承受检索和重排。
- 需要强事务一致性的场景，应该直接查询数据库。
- 需要法律、医疗、金融结论时，RAG 只能辅助，必须有审核和责任边界。

## RAG 与相关技术比较

| 技术 | 擅长 | 不擅长 |
| --- | --- | --- |
| RAG | 动态知识、私有知识、引用证据 | 学习新能力本身 |
| 微调 | 固定任务风格、格式、领域表达 | 高频变更事实、权限隔离 |
| 长上下文 | 处理大输入、长文分析 | 自动选择正确证据 |
| 搜索引擎 | 找文档和网页 | 合成答案和按格式表达 |
| SQL 查询 | 精确结构化数据 | 非结构化文档语义理解 |
| Agent | 多步工具执行和规划 | 稳定性和成本控制更难 |

## 例子

```text
问题：某 Java SDK 2.1 版本的认证方式和 1.9 有什么差异？

适合 RAG 的原因：
1. 答案依赖具体版本文档。
2. 需要引用 1.9、2.1 和迁移说明。
3. 模型参数中的知识可能过期。
4. 问题需要对比和合成。
```

## 练习

判断下面任务是否适合 RAG，并写出理由：

- 把中文邮件改写得更礼貌。
- 回答公司 VPN 配置步骤。
- 根据用户画像生成广告文案。
- 查询某订单当前物流状态。
- 总结最近上传的一份合同。

## 验收

- 能解释 RAG 的核心术语。
- 能给出 RAG 适用和不适用场景。
- 能说明为什么“需要知识”不一定等于“需要 RAG”。

## 重点

RAG 适合动态、私有、需引用的外部事实知识。

## 难点

边界判断很难：有些场景需要 RAG + SQL，有些需要 RAG + Agent，有些只需要搜索，有些只需要 prompt。

## 易错

> **易错：** 把所有大模型应用都做成 RAG。
>
> 正确做法：先判断任务是否依赖外部事实、是否要求引用、是否有可治理知识源，再决定是否引入 RAG。
