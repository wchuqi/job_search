# AI RAG 学习资料：查询规划、多跳检索、GraphRAG 和 Agentic RAG 深度解析

[返回索引](../AI RAG学习资料.md)

## 学习目标

- 掌握复杂问题的查询规划和多跳检索。
- 深入理解 GraphRAG 和 Agentic RAG 的收益、成本和失败模式。
- 能设计带预算、回退和证据校验的检索代理。

## 理论导读

复杂 RAG 的关键不是“让模型自己想办法”，而是把问题拆成可检索、可验证、可停止的步骤。多跳检索每增加一步，就增加一次错误传播风险，所以必须有计划、预算和停止条件。

## 查询规划流程

```text
1. 识别问题类型：事实、对比、原因、流程、多跳、结构化查询。
2. 抽取实体、时间、版本、约束。
3. 拆分子问题。
4. 为每个子问题选择检索通道。
5. 定义必须证据和停止条件。
6. 汇总证据并检查冲突。
```

## 多跳检索模式

| 模式 | 例子 | 风险 |
| --- | --- | --- |
| Entity expansion | 先找产品，再找版本文档 | 实体链接错误 |
| Temporal comparison | 对比旧版和新版 | 版本混淆 |
| Causal chain | 故障原因分析 | 相关性被误当因果 |
| Evidence bridging | A 文档指向 B 文档 | 中间证据缺失 |
| Tool routing | SQL + 文档 + 日志 | 权限和一致性复杂 |

## GraphRAG 深入

GraphRAG 的关键不是“用了图数据库”，而是把文档中的实体和关系显式化。

```text
文档 -> 实体抽取 -> 关系抽取 -> 社区/路径构建 -> 图检索 -> 原文证据回填
```

适合：

- 关系密集：组织、人、项目、供应链、代码依赖。
- 需要全局摘要：某类事件的主题、群组、趋势。
- 需要多跳路径：A 与 C 的关系通过 B 连接。

失败模式：

- 实体抽取错误。
- 同名实体未消歧。
- 关系方向错误。
- 图摘要脱离原文。
- 图证据不能回填到可引用文档。

## Agentic RAG 深入

Agentic RAG 让模型决定下一步查什么，但需要外部控制：

```json
{
  "max_steps": 5,
  "max_retrieval_calls": 8,
  "allowed_tools": ["hybrid_search", "sql_readonly", "graph_search"],
  "stop_when": ["requirements_covered", "budget_exhausted", "conflict_unresolved"],
  "human_escalation": ["high_risk", "insufficient_evidence"]
}
```

没有预算和停止条件的 Agentic RAG 容易成本失控、循环检索和证据漂移。

## 练习

为问题“某客户退款失败是否与最近认证变更有关”设计：

- 实体抽取。
- SQL 查询。
- 文档检索。
- 日志检索。
- 证据覆盖。
- 最终回答 schema。

## 验收

- 能拆分复杂问题的检索计划。
- 能说明 GraphRAG 的数据构建和失败模式。
- 能设计 Agentic RAG 的预算、权限和停止条件。
- 能避免把相关性误判为因果。

## 重点

复杂 RAG 的核心是可控查询规划，而不是放任模型自由探索。

## 难点

难点是证据链完整性。多跳答案每一步都要有证据，否则最终结论不可靠。

## 易错

> **易错：** 只要问题复杂就上 Agent。
>
> 正确做法：先判断复杂性来自多实体、多步骤、多数据源还是简单召回不足，再选择最小可控方案。
