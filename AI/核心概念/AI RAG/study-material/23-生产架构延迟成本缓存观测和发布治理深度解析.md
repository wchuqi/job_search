# AI RAG 学习资料：生产架构、延迟成本、缓存观测和发布治理深度解析

[返回索引](../AI RAG学习资料.md)

## 学习目标

- 掌握生产 RAG 的 SLO、成本和发布治理。
- 能设计缓存、限流、降级、灰度和回滚。
- 能用 trace 和指标定位线上问题。

## 理论导读

RAG 系统的生产目标不是“回答最好”，而是在准确性、延迟、成本、安全和可维护性之间达到可接受平衡。你需要为不同业务定义不同 SLO。

## 延迟预算

示例：

```text
总 p95 <= 3000ms
query rewrite <= 300ms
hybrid retrieve <= 300ms
rerank <= 700ms
context pack <= 100ms
LLM generate <= 1400ms
verify <= 200ms
buffer <= 0ms
```

如果 rerank 超时，可以降级为 hybrid top-k；如果 LLM 超时，可以返回“正在处理”或走异步。

## 成本模型

```text
总成本 = embedding 摄取成本 + 存储成本 + 检索成本 + rerank 成本 + LLM token 成本 + 评估成本
```

优化优先级：

- 减少无效检索。
- 降低 rerank 候选数。
- 压缩上下文。
- 缓存稳定结果。
- 对不同问题路由不同模型。

## 缓存失效

缓存 key 应包含版本：

```text
cache_key = hash(query, user_permission, index_version, retriever_version, prompt_version)
```

知识库更新、权限变化、prompt 变更、模型变更都可能导致缓存失效。

## 发布治理

一次 RAG 发布可能包含：

- 新 chunk 策略。
- 新 embedding 模型。
- 新索引参数。
- 新 reranker。
- 新 prompt。
- 新模型。
- 新安全策略。

发布流程：

```text
离线回放 -> 指标对比 -> 小流量灰度 -> 监控告警 -> 扩大流量 -> 保留回滚
```

## 可观测性看板

| 看板 | 指标 |
| --- | --- |
| 检索 | empty rate, recall proxy, top-k 分布 |
| 生成 | citation missing, refusal rate, token |
| 延迟 | p50/p95/p99 分阶段 |
| 成本 | 每请求成本、每日 token |
| 安全 | blocked requests, injection hits |
| 数据 | 索引版本、文档更新滞后 |

## 练习

为“企业客服 RAG”设计：

- SLO。
- 降级策略。
- 缓存 key。
- 灰度发布流程。
- 回滚条件。
- 告警阈值。

## 验收

- 能拆分延迟预算。
- 能设计版本化缓存。
- 能说明一次 RAG 发布影响哪些组件。
- 能设计线上监控和回滚。

## 重点

生产 RAG 的每次回答都是多个版本组件的组合结果。

## 难点

难点是指标互相牵制。提高 recall 可能增加延迟和噪声；降低成本可能损失忠实性。

## 易错

> **易错：** 只优化模型回答质量，不设 SLO 和回滚。
>
> 正确做法：把 RAG 当线上系统治理，质量、成本、延迟和安全一起评估。
