# AI RAG 学习资料：查询改写、混合检索和多路召回

[返回索引](../AI RAG学习资料.md)

## 学习目标

- 掌握 query rewrite、query expansion、multi-query 和 HyDE。
- 理解 BM25、向量检索、SQL、图检索和 Web 检索的互补关系。
- 能为复杂问题设计检索计划。

## 理论导读

用户问题往往不是检索友好的表达。用户会省略术语、使用口语、混合多个意图、引用上下文中的“它”、或者提出需要多份证据合成的问题。查询改写的目标是把“用户表达”转成“检索表达”。

## 核心心智模型

检索像问图书管理员问题。你问“那个新版怎么改了”，管理员很难找；你问“Payment API v2.1 相比 v1.9 的认证方式变更和迁移说明”，命中率会高很多。

## 知识点详解

### 查询改写方式

| 方法 | 说明 | 适用 |
| --- | --- | --- |
| Rewrite | 改写为清晰完整查询 | 多轮省略、口语问题 |
| Expansion | 增加同义词和相关术语 | 术语不一致 |
| Multi-query | 生成多个不同角度查询 | 召回不足 |
| HyDE | 先生成假想答案，再用假想答案检索 | 用户问题太短或抽象 |
| Decomposition | 拆成多个子问题 | 对比、多跳、复杂问题 |

### 混合检索

纯向量检索擅长语义，BM25 擅长关键词、错误码、函数名、专有名词。混合检索常见方式：

```text
score = alpha * dense_score + (1 - alpha) * sparse_score
```

或者先分别召回，再做结果融合：

```text
dense_top_50 + bm25_top_50 -> reciprocal rank fusion -> rerank_top_10
```

### 多路召回

| 通道 | 适用问题 |
| --- | --- |
| Vector search | 语义问答、概念解释 |
| BM25 | 错误码、API 名称、配置项 |
| SQL | 订单、库存、用户状态等结构化事实 |
| Graph | 实体关系、依赖链、多跳路径 |
| Web search | 外部实时信息 |

## 例子

```python
def plan_queries(question, chat_state):
    rewritten = rewrite_with_context(question, chat_state)
    if is_comparison_question(rewritten):
        return [
            {"query": rewritten + " old version", "channel": "hybrid"},
            {"query": rewritten + " new version", "channel": "hybrid"},
            {"query": rewritten + " migration breaking changes", "channel": "bm25"}
        ]
    return [{"query": rewritten, "channel": "hybrid"}]
```

## 练习

为下面问题写检索计划：

```text
我们现在从 v1 升到 v2，鉴权和限流有什么坑？
```

要求列出：

- 子查询。
- 每个子查询使用的检索通道。
- 必须命中的证据类型。
- 如果证据缺失，下一步怎么处理。

## 验收

- 能把模糊用户问题改写成可检索查询。
- 能说明 BM25 和向量检索的互补性。
- 能设计多路召回和结果融合。
- 能处理对比、多跳和上下文省略问题。

## 重点

检索质量常常先由 query 决定，再由索引和排序决定。

## 难点

难点是控制改写不要改变用户意图。改写只能补全和规范表达，不能替用户加入没有依据的约束。

## 易错

> **易错：** 直接用用户原话做一次向量检索。
>
> 正确做法：对多轮、省略、复杂、对比类问题先做查询规划，再多路召回。
