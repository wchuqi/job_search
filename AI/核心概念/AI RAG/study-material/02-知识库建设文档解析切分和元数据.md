# AI RAG 学习资料：知识库建设、文档解析、切分和元数据

[返回索引](../AI RAG学习资料.md)

## 学习目标

- 掌握 RAG 知识库从原始文档到可检索数据的建设流程。
- 理解 chunking 和 metadata 对召回质量、权限和版本治理的影响。
- 能设计一个面向生产的文档摄取 schema。

## 理论导读

知识库建设是 RAG 的地基。很多 RAG 失败不是检索算法不高级，而是文档本身解析错、切分错、版本乱、权限丢失或元数据不足。

RAG 的摄取过程通常是：

```text
原始文件 -> 解析 -> 清洗 -> 结构识别 -> 切分 -> 元数据标注 -> embedding -> 索引 -> 增量更新
```

## 核心心智模型

把知识库建设想成“把书整理成图书馆索引卡”：

- 原始文档是书。
- chunk 是索引卡上的摘录。
- metadata 是书名、章节、版本、作者、权限、出版时间。
- embedding 是索引卡的语义指纹。
- 索引是图书馆的查找系统。

没有元数据的 chunk 就像没有书名、页码和版本的复印件，模型即使引用了也难以复核。

## 知识点详解

### 文档解析

不同来源需要不同解析策略：

| 来源 | 重点 | 风险 |
| --- | --- | --- |
| Markdown | 保留标题层级、代码块、表格 | 标题丢失导致上下文断裂 |
| HTML | 去导航、广告、脚本，保留正文结构 | 把菜单和正文混在一起 |
| PDF | 页眉页脚、表格、换行、扫描 OCR | 顺序错乱、表格变形 |
| Word | 标题、批注、修订、表格 | 隐藏内容混入 |
| 数据库 | 字段含义、主键、时间和权限 | 把结构化查询误做向量检索 |

### 切分策略

| 策略 | 适用 | 风险 |
| --- | --- | --- |
| 固定长度切分 | 快速基线 | 容易切断语义 |
| 标题层级切分 | 文档结构清晰 | 长章节需要二次切分 |
| 语义切分 | 段落和主题边界明显 | 实现和评估复杂 |
| 父子 chunk | 小片段召回，大段落注入 | 存储和去重复杂 |
| 滑动窗口 | 需要保留上下文 | 重复多、成本高 |

### 元数据设计

推荐最小 schema：

```json
{
  "chunk_id": "policy-2026-v3-sec-4-p2",
  "doc_id": "policy-2026-v3",
  "title": "员工报销制度",
  "section": "第 4 节 差旅报销",
  "source_uri": "hr/policy/reimbursement-2026.md",
  "version": "2026.3",
  "updated_at": "2026-05-01",
  "owner": "HR",
  "permission": ["employee", "finance"],
  "language": "zh",
  "content_hash": "..."
}
```

这些字段不是形式主义。它们用于版本过滤、权限控制、引用展示、增量更新、重复检测和审计。

## 例子

文档片段：

```markdown
## 2.1 API 认证

从 v2.0 开始，所有请求必须使用 Authorization header。
Query token 认证方式已废弃。
```

合理 chunk：

```json
{
  "text": "## 2.1 API 认证\n从 v2.0 开始，所有请求必须使用 Authorization header。Query token 认证方式已废弃。",
  "metadata": {
    "product": "payment-api",
    "version": "v2.1",
    "section": "API 认证",
    "source_uri": "docs/payment-api/v2.1/auth.md"
  }
}
```

## 练习

拿一份 Markdown 或 PDF 文档，设计：

- 解析规则。
- 切分规则。
- metadata schema。
- 去重规则。
- 增量更新规则。
- 权限过滤规则。

## 验收

- 能说明每个 metadata 字段的用途。
- 能解释 chunk 大小对召回和生成的影响。
- 能处理文档更新和删除。
- 能追踪答案引用到原始文档。

## 重点

知识库建设要服务于“可检索、可引用、可过滤、可更新、可审计”。

## 难点

难点是结构保真：表格、代码、标题层级、跨页段落和图片说明如果解析错，后面的 embedding 和检索都会继承错误。

## 易错

> **易错：** 只保存 chunk 文本，不保存 source、version、permission。
>
> 正确做法：把元数据当作检索和治理的一等公民，摄取时就写入并验证。
