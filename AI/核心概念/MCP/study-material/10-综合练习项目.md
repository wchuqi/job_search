# MCP 学习资料：综合练习项目

[返回索引](../MCP学习资料.md)

## 学习目标

- 用一个完整项目串联 MCP 的协议、工具、资源、提示、安全和生产治理。
- 能从需求分析到实现、测试、审计、发布形成闭环。
- 能输出一份可面试答辩的 MCP server 设计方案。

## 项目：招聘系统 MCP Server

目标：为招聘 AI 助手实现一个 MCP server，连接招聘系统 ATS，支持候选人搜索、岗位读取、面试安排和状态更新。

## 项目范围

### Resources

```text
ats://job/{job_id}/description
ats://candidate/{candidate_id}/profile
ats://schema/application
ats://policy/hiring-status-flow
```

### Prompts

```text
summarize_candidate_match(candidate_id, job_id)
generate_interview_questions(candidate_id, job_id, focus_area)
explain_application_history(application_id)
```

### Tools

```text
search_candidates(skill, city, min_years, limit)
list_applications(job_id, status, limit)
create_interview(candidate_id, job_id, interviewer_id, start_time, idempotency_key)
update_application_status(application_id, target_status, reason, confirm)
```

## 阶段 1：需求和能力建模

### 学习目标

- 从业务动作中拆分 resource、prompt、tool。
- 标注每个能力的权限和风险等级。

### 任务

1. 为每个 resource 定义 URI、返回字段和权限。
2. 为每个 prompt 定义参数和输出目标。
3. 为每个 tool 定义输入 Schema、输出和错误类型。
4. 标注风险等级。

### 验收

- 没有万能工具。
- 只读上下文没有被错误建成高危工具。
- 写操作有确认、审计和幂等。

## 阶段 2：协议实现

### 学习目标

- 完成 initialize、tools/list、tools/call、resources/read、prompts/get 的基本流程。

### 任务

1. 实现 serverInfo 和 capabilities。
2. 实现 tools/list。
3. 实现 search_candidates 工具。
4. 实现 candidate profile resource。
5. 实现 summarize_candidate_match prompt。

### 验收

- Inspector 能看到工具、资源和提示。
- 工具参数不合法时返回明确错误。
- 权限不足时拒绝访问。

## 阶段 3：安全控制

### 学习目标

- 把 MCP 工具调用绑定真实用户权限和业务状态机。

### 任务

1. `search_candidates` 不返回手机号、邮箱等敏感字段。
2. `update_application_status` 校验合法状态流转。
3. `create_interview` 使用 idempotency key。
4. 高危状态变更要求 `confirm=true`。
5. 记录审计日志。

### 验收

- 普通用户不能查看非所属岗位候选人。
- 重复调用创建面试不会创建两条记录。
- 状态变更有完整审计。

## 阶段 4：调试和测试

### 学习目标

- 建立 MCP server 的测试矩阵。

### 任务

1. 用 Inspector 调用全部工具。
2. 写 Schema 测试。
3. 写权限测试。
4. 写提示注入测试。
5. 写外部系统超时测试。

### 验收

- 每个工具至少有正常、参数错误、权限错误、外部失败四类测试。
- 高危工具必须覆盖确认缺失和确认通过。

## 阶段 5：生产方案

### 学习目标

- 设计 MCP server 上线后的注册、网关、观测和版本治理。

### 任务

1. 写 server manifest。
2. 设计工具级限流。
3. 设计审计字段。
4. 设计版本变更策略。
5. 写应急预案。

### 验收

- 能追踪一次用户请求到外部系统调用。
- 能回滚工具版本。
- 能禁用单个高危工具而不下线整个 server。

## 重点

- 项目要覆盖协议、业务、权限、安全和生产治理。
- 高危工具必须有幂等和审计。
- 面试答辩时要说明为什么这样拆 resource、prompt、tool。

## 难点

- 工具粒度太粗会危险，太细会增加模型选择难度。
- 权限要贯穿 resource、prompt、tool，而不是只在 tool 层做。

## 易错

> **易错：** 先写代码，再想权限。
>
> 正确做法：能力建模阶段就标注权限、风险、确认和审计要求。
