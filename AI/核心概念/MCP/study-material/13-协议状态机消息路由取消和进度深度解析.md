# MCP 学习资料：协议状态机、消息路由、取消和进度深度解析

[返回索引](../MCP学习资料.md)

## 学习目标

- 能用状态机解释 MCP 会话从连接建立到关闭的合法行为。
- 理解 request、response、notification、cancel、progress 的路由关系。
- 能排查“初始化未完成就调用工具”“响应 id 不匹配”“取消后外部副作用仍发生”等问题。

## 理论导读

MCP 入门时通常只记住 `initialize`、`tools/list`、`tools/call`。但生产故障往往发生在状态边界：client 在 initialized 前调用能力，server 返回了错误 id，长任务取消后外部系统仍继续执行，通知被当成可确认动作，或者 server 在未协商能力时反向请求 sampling。

要深入理解 MCP，必须把它看成一个状态机，而不是一组孤立 API。

## 核心心智模型

```text
Disconnected
  -> Transport Connected
  -> Initializing
  -> Initialized / Operational
  -> Closing
  -> Closed
```

每个状态允许的消息不同。协议越清晰，host 越容易做兼容、安全和故障恢复。

## 知识点详解

### 1. 初始化前状态

连接刚建立时，双方还没有协议版本和能力共识。此时 server 不应假设 client 支持 roots、sampling、elicitation；client 也不应直接调用 tools/resources/prompts。

合法动作：

- client 发送 `initialize`。
- server 返回初始化结果。
- client 发送 initialized notification。

风险：

- server 提前发送能力相关请求，client 不知道如何处理。
- client 跳过 initialized，server 状态未准备好。
- 协议版本不兼容却继续运行。

### 2. 运行期状态

进入运行期后，双方只能使用协商过的能力。

运行期消息包括：

- list：发现能力。
- read/get/call：使用能力。
- notifications：能力列表变化、日志、进度等。
- cancel：取消进行中的请求。
- 长任务模式：用进度、取消、资源日志和状态查询工具管理耗时工作。

> **重点：** 初始化返回的 capabilities 是运行期允许行为的依据。

### 3. Request id 路由

JSON-RPC request 带 `id`，response 必须带同一个 `id`。并发请求时，响应可以乱序返回，但必须按 id 匹配。

```text
client -> request id=10 tools/call A
client -> request id=11 tools/call B
server -> response id=11
server -> response id=10
```

这是合法的。client 不能按返回顺序匹配请求。

> **易错：** 用队列顺序匹配响应，导致并发调用时结果错配。

### 4. Notification 的边界

Notification 没有响应。适合：

- initialized。
- 列表变化。
- 日志。
- 进度。
- 取消请求。

不适合：

- 删除资源。
- 创建订单。
- 修改权限。
- 任何需要确认成功的业务动作。

### 5. 取消不等于回滚

Cancel 通知表达“调用方不再需要这个请求结果”或“希望停止”。但外部系统可能已经执行了部分动作。

例子：

```text
tools/call create_interview
  -> server 调用日历 API 成功
  -> client 发送 cancel
  -> server 无法自动删除已创建日历
```

正确设计：

- 长任务在执行前和关键步骤间检查取消信号。
- 有副作用工具设计幂等键。
- 对已发生副作用提供补偿工具。
- 返回“取消时外部状态”的结构化说明。

### 6. 进度和长任务

长任务包括索引构建、批量导入、代码扫描、远程搜索、部署等。进度信息要包括：

- 当前阶段。
- 已完成数量。
- 总数量或未知。
- 可取消性。
- 预计剩余时间可选。

进度是用户体验和运维观测的一部分，不只是 UI 装饰。

### 7. 长任务模式的工程意义

当前 MCP 规范提供进度、取消、日志等 utilities。长时间、异步或可跟踪工作通常要在这些 utilities 之上做工程建模，避免把所有事情都塞进一次阻塞工具调用。

适合：

- 长时间索引。
- 批量分析。
- 数据同步。
- 部署流水线。
- 大型代码库扫描。

设计要点：

- operation id 或 job id 稳定。
- 状态明确：pending、running、succeeded、failed、cancelled。
- 支持查询状态。
- 支持取消。
- 结果和日志可追踪。

## 例子

错误设计：

```text
tools/call index_repository
  -> 阻塞 30 分钟
  -> 中途无进度
  -> 取消无效
  -> 失败后不知道扫到哪里
```

更好的设计：

```text
tools/call start_repository_index
  -> 返回 operation_id
tools/call get_repository_index_status
  -> 查询进度
notifications/cancelled 或 tools/call cancel_repository_index
  -> 请求取消
resource/read index://operation/{operation_id}/log
  -> 读取日志
```

## 练习

1. 画出 MCP 初始化状态机。
2. 构造两个并发 request，说明为什么响应可以乱序。
3. 为一个长任务设计 operation id、状态、进度和取消语义。
4. 说明取消和回滚的区别。

## 验收

- 能说明初始化前、运行期、关闭期的合法消息。
- 能解释 request id 如何解决并发响应乱序。
- 能设计一个可取消但不误导为“可回滚”的长任务。

## 重点

- MCP 是状态协议，不是一组散乱 HTTP API。
- Notification 无响应，不能承载必须确认的业务动作。
- Cancel 是协作式取消，不是外部系统事务回滚。

## 难点

- 长任务要同时处理协议状态、外部系统状态和用户体验。
- 并发请求、取消、重试和断线重连组合后会出现复杂边界。

## 易错

> **易错：** 取消工具调用后，假设外部副作用自动撤销。
>
> 正确做法：明确取消检查点、幂等键、补偿动作和最终状态。
