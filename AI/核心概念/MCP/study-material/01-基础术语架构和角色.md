# MCP 学习资料：基础术语、架构和角色

[返回索引](../MCP学习资料.md)

## 学习目标

- 掌握 MCP 的基础术语和角色职责。
- 理解 host、client、server 为什么要分开。
- 能从业务系统出发设计 MCP 能力边界。

## 理论导读

MCP 的架构像浏览器和网站的关系，但它面向 AI 应用。Host 类似浏览器外壳，MCP client 类似某个站点的连接会话，MCP server 类似提供标准能力的网站。外部系统则是 server 背后的真实数据源或动作执行者。

分层的意义是隔离责任：host 管用户、模型和权限展示；client 管连接；server 管能力声明和执行；外部系统管真实业务状态。职责混在一起时，权限和错误会很难追踪。

## 核心心智模型

```text
Host = 控制台
Client = 一条受控连接
Server = 能力适配器
External System = 真实业务系统
```

## 知识点详解

### 1. Host

Host 是用户直接使用的 AI 应用，例如 IDE、桌面助手、Agent 平台、聊天应用。它负责：

- 展示 MCP server 提供的能力。
- 决定是否把 tool、resource、prompt 暴露给模型。
- 管理用户确认和权限提示。
- 调用模型并处理模型输出。
- 聚合多个 MCP client 的上下文。

> **重点：** Host 是最终用户体验和安全策略的入口。

### 2. Client

MCP client 是 host 内部与某个 server 连接的协议实体。通常一个 server 对应一个 client 会话。它负责：

- 建立传输连接。
- 执行初始化和能力协商。
- 发送请求、接收响应和通知。
- 维护会话状态。
- 将 host 策略转化为协议调用。

### 3. Server

MCP server 是能力提供者。它负责：

- 声明支持的能力。
- 列出 tools、resources、prompts。
- 执行 tool call。
- 读取 resource。
- 处理 server 端权限、参数校验和错误。
- 可选地请求 roots、sampling 或 elicitation。

> **易错：** Server 不应该默认信任模型给出的参数。模型输出只是意图，server 必须做校验和权限判断。

### 4. Tool

Tool 是可执行动作。它可以只读，也可以有副作用。典型字段包括名称、描述、输入 Schema 和输出结果。

设计原则：

- 名称稳定、语义清晰。
- 描述说明何时使用、风险和限制。
- 输入 Schema 尽量严格。
- 高危工具需要确认、审计和回滚。

### 5. Resource

Resource 是可读取上下文。它有 URI，并可由 server 提供列表、模板和读取能力。适合表达文件、文档、表结构、配置、日志片段等。

设计原则：

- URI 要稳定可解释。
- 资源内容要避免过大。
- 敏感资源必须做权限校验。
- 可用 resource template 表达动态资源。

### 6. Prompt

Prompt 是 server 提供的可复用提示模板。它把某类任务的上下文结构、用户输入和指令组织起来。

适合：

- 代码审查模板。
- SQL 优化模板。
- 工单总结模板。
- 候选人面试问题生成模板。

### 7. Roots、Sampling、Elicitation

这三类是 client/host 侧能力：

- Roots：告诉 server 允许访问的根目录或工作区边界。
- Sampling：server 请求 host 代表它调用模型。
- Elicitation：server 请求 host 向用户询问缺失信息。

它们让 server 能更灵活，但也更危险，因为 server 在请求 host 做事。

## 例子

招聘助手 MCP 架构：

```text
Host: 企业招聘 AI 助手
Client A: 连接 ATS MCP server
Client B: 连接 Calendar MCP server
Client C: 连接 Knowledge Base MCP server

ATS Server:
  tool: search_candidates
  tool: update_application_status
  resource: ats://candidate/{id}
  prompt: summarize_candidate_for_interviewer
```

## 练习

1. 为“代码仓库 MCP server”列出 3 个 resource、3 个 tool、2 个 prompt。
2. 判断下面能力应该建成 resource 还是 tool：读取 README、删除分支、生成发布说明、查看 CI 结果。
3. 说明 host 和 server 分别应该做哪些权限检查。

## 验收

- 能解释 host、client、server 的职责差异。
- 能给 tool、resource、prompt 各举 3 个例子。
- 能说明 Roots、Sampling、Elicitation 为什么是客户端能力。

## 重点

- 分层是 MCP 可治理的基础。
- Tool 表示动作，Resource 表示上下文，Prompt 表示任务模板。
- Server 暴露能力，不等于 host 必须全部使用。

## 难点

- 同一业务对象可能同时有 resource、tool 和 prompt 三种表达。
- server 请求 client 能力时，host 必须保留最终控制权。

## 易错

> **易错：** 为了方便，把读取、搜索、更新、删除都做成一个万能 `execute` 工具。
>
> 正确做法：拆成语义明确、权限可控、参数严格、可审计的多个工具。
