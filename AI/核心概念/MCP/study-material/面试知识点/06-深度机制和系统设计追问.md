## 一、深度机制和系统设计追问

### 1. MCP 为什么要做能力协商？如果跳过会怎样？

**参考答案：**

能力协商让 client 和 server 明确协议版本和双方能力。没有协商，client 可能调用 server 不支持的 tools/resources，server 也可能请求 client 不支持的 sampling、roots、elicitation，导致兼容性错误和安全策略失效。

> **重点：** capabilities 是运行期允许行为的依据。

### 2. MCP notification 和 request 的区别是什么？为什么高危动作不能用 notification？

**参考答案：**

Request 有 id，需要 response；notification 没有响应。高危动作必须知道是否成功、失败、被拒绝或需要确认，因此不能用 notification 承载。Notification 更适合 initialized、日志、进度、列表变化等无需直接确认的消息。

> **易错：** 把 notification 当成轻量 RPC。

### 3. Cancel 为什么不是回滚？

**参考答案：**

Cancel 只是请求停止处理或表示调用方不再需要结果。工具可能已经调用外部系统并产生副作用。回滚需要外部系统事务、补偿动作或明确反向工具支持。

> **重点：** 有副作用工具要设计幂等、检查点和补偿。

### 4. Tool annotations 能不能作为权限控制？

**参考答案：**

不能。Annotations 只是提示 host 和模型理解工具性质，例如只读、破坏性、幂等、是否访问开放世界。恶意或错误 server 可能声明错误。权限控制必须由 host policy、gateway、server auth 和外部系统校验共同完成。

> **重点：** annotations 是 UX 和策略辅助，不是安全边界。

### 5. 远程 MCP 为什么要校验 token audience？

**参考答案：**

Audience 确保 token 只用于预期资源或 server。若不校验 audience，一个发给知识库 server 的 token 可能被拿去访问工单 server，形成 token 混用和 confused deputy 风险。

> **重点：** token 校验至少包括 issuer、signature、expiry、audience、scope 和 tenant。

### 6. 你会如何设计 MCP 网关？

**参考答案：**

MCP 网关应包含：

- server registry 和版本路由。
- 用户认证和 tool 级授权。
- 参数策略检查。
- 高危工具确认和审批。
- 限流、熔断、灰度。
- 审计和 trace。
- 数据脱敏和结果大小限制。

它需要理解 MCP 语义，尤其是 `tools/call` 的 tool name、风险等级和参数摘要。

> **难点：** MCP 网关不是普通 HTTP 代理。

### 7. MCP tool description 变更为什么可能是破坏性变更？

**参考答案：**

模型会根据 name、description、Schema 选择工具。描述变更可能改变模型调用概率和调用场景。即使 Schema 没变，模型行为也可能变，因此 description 也是版本化资产。

> **重点：** MCP 兼容性包括模型行为兼容。

### 8. Agent 使用 MCP 时如何避免工具候选集过大？

**参考答案：**

按任务、用户权限、风险等级和当前阶段过滤工具。只暴露当前任务必要工具；高危工具在用户明确请求前隐藏或需要确认；相似工具合并或重命名；对工具分组和路由。

> **重点：** 工具越多不代表 Agent 越强，可能降低选择准确率并提高风险。

### 9. 如何防止 MCP Resource 中的提示注入影响工具调用？

**参考答案：**

Resource 内容必须作为不可信数据处理。Host 应隔离指令层级，server 和 gateway 不应接受 resource 内容授权高危动作。高危工具需要确认，server 仍要做权限和状态校验。

> **重点：** 防提示注入不能只靠“告诉模型不要听”，必须有权限和确认机制。

### 10. 设计一个企业级 MCP 平台，你的核心模块是什么？

**参考答案：**

核心模块：

- MCP server registry。
- MCP gateway。
- OAuth/SSO 授权。
- Policy engine。
- Tool risk classification。
- Audit log。
- Observability and tracing。
- Version and release management。
- Red-team and evaluation framework。

答题时要说明每个模块解决什么风险，而不是只列名词。

> **重点：** 企业 MCP 平台的目标是让工具能力可发现、可控、可观测、可回滚。
