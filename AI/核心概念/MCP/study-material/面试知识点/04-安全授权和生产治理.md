## 一、安全授权和生产治理

### 1. MCP 的主要安全风险有哪些？

**参考答案：**

主要风险包括：

- 提示注入诱导模型调用工具。
- 高危工具无确认。
- token passthrough 导致凭据泄露或权限失控。
- confused deputy，即有权限系统被诱导替攻击者做事。
- server 越权访问资源。
- 工具 Schema 过宽导致注入。
- 审计缺失导致事故不可追踪。

> **重点：** MCP 把模型输出连接到真实系统，安全风险会从“回答错误”升级为“系统副作用”。

### 2. 为什么 token passthrough 危险？

**参考答案：**

把用户 token 或平台 token 直接传给 MCP server，会让 server 拥有过宽权限，且 token 可能被日志、缓存或恶意 server 泄露。它还难以限制用途和审计具体意图。

更好的方式是 OAuth 授权、短期受限 token、明确 audience、细粒度 scope 和下游系统按真实用户校验。

> **易错：** 为了省事给 server 一个超级 token。

### 3. 什么是 confused deputy？MCP 中如何防？

**参考答案：**

Confused deputy 是有权限的中间系统被诱导执行攻击者目的。在 MCP 中，恶意资源可能诱导模型让 host/server 调用高危工具。

防护：

- 授权绑定真实用户、资源、动作和意图。
- 高危操作展示完整参数并二次确认。
- server 校验资源归属和租户。
- host 把 resource 内容视为不可信数据。

> **重点：** 不能只问“用户有没有权限”，还要问“这次操作是否符合用户明确意图”。

### 4. 企业级 MCP 网关应该做什么？

**参考答案：**

MCP 网关负责：

- 统一认证和授权。
- server 注册和发现。
- tool 级策略检查。
- 高危工具审批。
- 限流、熔断和灰度。
- 审计日志和 trace。
- 版本路由和兼容治理。

它不是普通 HTTP 反向代理，需要理解 MCP 的 tool、resource、prompt 语义。

> **重点：** 企业 MCP 的核心难点是治理，不是把请求转发出去。

### 5. Tool 版本变更为什么要评审描述？

**参考答案：**

传统 API 主要关注字段兼容性；MCP tool 还要关注名称和描述，因为模型会根据描述选择工具。描述变更可能导致模型更频繁或更少调用某工具，甚至误用高危工具。

因此 tool name、description、inputSchema、权限 scope、风险等级都应纳入版本评审。

> **难点：** MCP 的兼容性包含“模型行为兼容”，不只是代码接口兼容。
