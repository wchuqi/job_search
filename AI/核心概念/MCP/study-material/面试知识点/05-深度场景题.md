## 一、深度场景题

### 1. 让你设计一个数据库 MCP server，你会怎么避免它变成危险后门？

**参考答案：**

不要暴露任意 `execute_sql`。应按业务场景拆分受控工具，例如查询表结构、搜索候选人、查询慢 SQL 摘要、生成只读报表。每个工具用严格 Schema 限制参数，按真实用户权限过滤数据。

控制措施：

- 默认只读。
- 禁止任意 SQL 或只允许白名单查询模板。
- 限制返回行数和字段。
- 敏感字段脱敏。
- 写操作需要审批和二次确认。
- 记录 SQL 模板、参数、用户、耗时和结果摘要。

> **重点：** 数据库 MCP 的风险不是协议本身，而是工具粒度和权限模型。

### 2. 一个恶意文档要求模型调用删除工具，你怎么防？

**参考答案：**

这是提示注入。防护要分层：

- Host 把 resource 内容标记为不可信数据，不允许其覆盖系统指令。
- 模型选择工具后，host/server 仍做策略检查。
- 删除工具需要明确用户确认，展示目标资源和影响。
- Server 校验用户是否有删除权限。
- 审计记录触发来源、资源内容摘要和确认记录。

> **重点：** 防护不能只靠提示词说“不要听恶意内容”，必须靠权限和确认机制。

### 3. 一个 MCP tool 调用偶发重复创建工单，你如何排查和修复？

**参考答案：**

先查调用链路：host 是否重试、HTTP 是否超时、server 是否收到多次同 id 请求、外部系统是否幂等。然后检查 tool 是否有 idempotency key。

修复：

- inputSchema 增加 `idempotency_key`。
- server 记录 key 和执行结果。
- 外部系统支持幂等创建。
- 超时后先查询状态再决定是否重试。
- 审计中记录 request id 和 idempotency key。

> **易错：** 只把 HTTP 超时时间调大，不处理重复执行的根因。

### 4. 如何设计 MCP server registry？

**参考答案：**

Registry 至少包含：

- server name、description、owner、版本。
- 协议版本、传输方式、endpoint。
- tools/resources/prompts 列表。
- 权限 scope、风险等级、数据分类。
- SLA、限流、审计要求。
- 发布状态、灰度范围、变更记录。

Registry 的价值是让企业知道有哪些 MCP server、谁负责、暴露了什么能力、风险如何、谁可以用。

> **重点：** 没有 registry 的 MCP 平台会变成不可治理的工具集合。

### 5. 多个 MCP server 被 Agent 连续调用时，如何做权限和审计？

**参考答案：**

要建立跨 server trace。每次用户请求生成 trace id，模型选择工具、MCP gateway、server、外部系统调用都带上 trace id。权限上，每个 tool call 都按真实用户、租户、资源和动作单独校验，不能因为前一个工具成功就默认后续工具有权限。

审计要记录：

- 用户请求。
- 模型选择的工具。
- server 和 tool 版本。
- 参数摘要。
- 授权结果。
- 外部系统结果。
- 高危确认记录。

> **难点：** Agent 工作流不是一次调用，权限和审计要覆盖整条链路。

### 6. 如果客户端不支持 Sampling，server 应该怎么办？

**参考答案：**

Server 不能假设 Sampling 可用。初始化时如果 client capabilities 没有 sampling，server 应降级：

- 返回原始 resource 或结构化数据。
- 提示 host 或用户需要手动总结。
- 拒绝依赖模型生成的工具流程，并返回明确错误。
- 提供不需要 sampling 的替代工具。

> **重点：** 能力协商结果必须影响运行时行为。

### 7. 如何评审一个新 MCP server 是否能上线？

**参考答案：**

评审维度：

- 协议：是否正确初始化和声明能力。
- 能力：tool/resource/prompt 是否粒度合理。
- Schema：是否严格。
- 权限：是否绑定真实用户和租户。
- 安全：是否防提示注入、token 透传和 confused deputy。
- 高危工具：是否确认、审批、审计和回滚。
- 观测：是否有日志、指标、trace。
- 版本：是否有 owner、变更记录和灰度策略。
- 测试：是否覆盖参数、权限、错误、注入和超时。

> **重点：** 上线评审不是只看功能 demo，而是看生产治理闭环。
