## 一、协议生命周期和传输

### 1. MCP 初始化流程是什么？

**参考答案：**

MCP 初始化通常包括：

1. Client 发送 `initialize`，包含协议版本、client capabilities、clientInfo。
2. Server 返回协议版本、server capabilities、serverInfo。
3. Client 发送 initialized notification，表示初始化完成。
4. 双方进入运行期，请求 tools/list、resources/read、tools/call 等。

> **重点：** 初始化阶段确定双方能使用哪些能力，运行期不能假设未协商能力存在。

### 2. MCP 为什么使用 JSON-RPC？

**参考答案：**

JSON-RPC 提供了清晰的 request、response、notification 和 error 模型。MCP 可以在其上定义方法名和参数，例如 `tools/list`、`tools/call`、`resources/read`。

请求有 `id`，响应按 `id` 匹配；通知没有响应，适合 initialized、列表变化、日志、进度等无需直接确认的消息。

> **易错：** 用 notification 承载必须确认成功的高危动作。

### 3. stdio 和 Streamable HTTP 怎么选择？

**参考答案：**

stdio 适合本地 server，例如文件系统、Git、CLI 包装工具。Host 启动本地进程，通过 stdin/stdout 交换消息。

Streamable HTTP 适合远程 server，例如企业知识库、工单系统、SaaS。它更适合认证、网关、负载均衡、观测和平台化治理。

> **重点：** 传输方式会影响部署、安全、认证、会话和观测模型。

### 4. 为什么 stdio server 的日志可能造成问题？

**参考答案：**

stdio 的 stdout 通常是 MCP 协议消息通道。如果 server 把普通日志写到 stdout，就可能污染协议流，导致 client 无法解析 JSON-RPC 消息。

正确做法是把日志写到 stderr 或文件，并确保 stdout 只输出协议消息。

> **易错：** 本地调试时随手 `console.log`，导致协议通信失败。

### 5. 远程 MCP 为什么要考虑幂等？

**参考答案：**

远程 HTTP 调用可能因网络中断、超时或重试导致同一请求发送多次。如果工具有副作用，例如创建面试、发送邮件、部署服务，重复执行会造成事故。

解决方案包括 idempotency key、业务状态机、操作前检查、补偿逻辑和审计。

> **重点：** 有副作用工具不能依赖简单网络重试保证正确性。
