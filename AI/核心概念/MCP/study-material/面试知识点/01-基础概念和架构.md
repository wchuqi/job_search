## 一、基础概念和架构

### 1. MCP 是什么？解决什么问题？

**参考答案：**

MCP，全称 Model Context Protocol，是 AI 应用连接外部上下文、工具和工作流的标准协议。它解决的是 AI host 如何发现、调用和治理外部能力的问题，例如文件、数据库、知识库、SaaS、内部平台工具。

它不是模型服务，也不是普通业务 API。它更像 AI 应用的外设协议：host 通过 MCP client 连接 MCP server，server 再访问真实外部系统。

> **重点：** MCP 的价值是标准化能力发现、上下文读取、工具调用和治理边界。

### 2. MCP 的 host、client、server 分别是什么？

**参考答案：**

- Host：用户直接使用的 AI 应用，负责模型调用、用户体验、权限展示和策略控制。
- Client：host 内部与某个 server 连接的协议会话，通常一个 server 对应一个 client。
- Server：外部能力适配器，声明 tools、resources、prompts，并执行具体能力。

例子：AI IDE 是 host，连接文件系统 MCP server 的会话是 client，文件系统 MCP server 是 server。

> **易错：** 把 client 和 host 混为一谈。client 是协议连接，host 是完整应用。

### 3. MCP 和 Function Calling 有什么区别？

**参考答案：**

Function Calling 通常描述模型如何输出结构化工具调用意图；MCP 描述 AI 应用如何发现外部能力、获取工具 Schema、调用工具、读取资源、获取提示模板和治理权限。

二者可以配合：host 从 MCP server 获取 tools，再把可用工具提供给模型；模型选择工具后，host 通过 MCP 调用 server。

> **重点：** Function Calling 更靠近模型接口，MCP 更靠近外部能力协议和运行时治理。

### 4. MCP 和普通 REST API 有什么区别？

**参考答案：**

REST API 面向通用服务调用，MCP 面向 AI host 和外部上下文/工具连接。MCP 具有能力发现、初始化协商、resources、prompts、tools、client capabilities 等 AI 应用特定概念。

MCP server 可以封装 REST API，但它还要补充工具描述、Schema、权限、审计、结构化输出和安全策略。

> **难点：** MCP 不是取代 REST，而是把外部 API 包装成 AI 应用可治理的能力。

### 5. Resources、Prompts、Tools 的区别是什么？

**参考答案：**

- Resources：可读取上下文，例如文件、文档、数据库 schema。
- Prompts：可复用任务模板，例如代码审查模板、SQL 解释模板。
- Tools：可执行动作，例如搜索、创建工单、更新状态。

判断规则：只是读上下文用 resource；组织任务用 prompt；会执行动作或查询逻辑用 tool。

> **易错：** 把所有能力都做成 tool，导致只读上下文也进入高风险执行路径。
