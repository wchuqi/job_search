# MCP 学习资料：生产架构、网关、注册、观测和版本治理

[返回索引](../MCP学习资料.md)

## 学习目标

- 能设计企业级 MCP 平台架构。
- 掌握 MCP server 注册、发现、权限、网关、审计和版本治理。
- 能从延迟、成本、安全和可靠性角度评审 MCP 方案。

## 理论导读

当 MCP server 从个人本地工具变成企业共享能力时，问题会从“如何调用”变成“如何治理”。谁可以发布 server？谁能使用哪个 tool？工具变更如何兼容？调用失败如何追踪？高危操作谁批准？这些都是生产架构必须回答的问题。

## 核心心智模型

```text
MCP Platform =
  Registry + Gateway + Auth + Policy + Observability + Audit + Versioning
```

## 知识点详解

### 1. 架构分层

```text
Host / Agent Platform
  -> MCP Gateway
      -> server registry
      -> authn/authz
      -> policy engine
      -> rate limit
      -> audit log
      -> observability
  -> MCP Servers
      -> external systems
```

### 2. Server 注册和发现

注册信息：

- server 名称、描述、owner。
- 协议版本和传输方式。
- tools/resources/prompts 列表。
- 权限 scope。
- 风险等级。
- 数据分类。
- SLA 和限流。
- 版本和变更记录。

### 3. 网关职责

MCP 网关可以做：

- 统一认证。
- 用户和租户上下文注入。
- 工具级授权。
- 参数策略检查。
- 高危工具审批。
- 调用审计。
- 限流和熔断。
- 版本路由。
- 日志和指标采集。

> **重点：** 网关不只是反向代理。它需要理解 MCP 的 tool、resource、prompt 语义。

### 4. 可观测性

指标：

- initialize 成功率。
- tools/list 延迟。
- tools/call 次数、延迟、错误率。
- 每个 tool 的 P50/P95/P99。
- 权限拒绝次数。
- 用户确认通过率和拒绝率。
- token 成本和 sampling 成本。
- 外部系统错误。

日志：

- request id。
- user id、tenant id。
- server、tool、版本。
- 参数摘要。
- 风险等级。
- 授权结果。
- 执行结果。

追踪：

```text
user request trace
  -> model call
  -> tool selection
  -> MCP gateway
  -> MCP server
  -> external API
```

### 5. 版本治理

版本变更分类：

| 变更 | 兼容性 |
| --- | --- |
| 新增可选参数 | 通常兼容 |
| 新增工具 | 通常兼容 |
| 修改工具描述 | 可能影响模型选择 |
| 删除字段 | 不兼容 |
| 参数含义改变 | 高风险不兼容 |
| 权限 scope 改变 | 安全敏感 |

发布策略：

- 语义化版本。
- 灰度发布。
- 老版本保留窗口。
- 工具变更自动生成评审。
- Prompt/Tool 描述变更也要评估。

### 6. 成本和延迟

MCP 调用会增加：

- 协议往返。
- 工具执行时间。
- 外部 API 延迟。
- 结果进入模型上下文的 token 成本。
- sampling 二次模型调用成本。

优化：

- 缓存只读 resource。
- 工具结果分页。
- 结构化输出减少冗余文本。
- 批量工具替代 N 次小调用。
- 对高频工具做本地索引或预聚合。

## 例子

企业 MCP server 发布流程：

```text
1. 团队提交 server manifest。
2. 平台扫描 tools/resources/prompts。
3. 安全团队评审高危工具。
4. 配置 scope、限流、审计字段。
5. 灰度给部分 host 使用。
6. 观察错误率、延迟、误用率。
7. 全量发布并记录版本。
```

## 练习

1. 为 10 个 MCP server 设计 registry 字段。
2. 为一个高危工具设计网关策略。
3. 设计 MCP 调用链路的 trace 字段。
4. 写一个工具版本变更评审清单。

## 验收

- 能画出企业 MCP 平台架构。
- 能说明网关和注册中心的职责。
- 能设计工具级观测指标。
- 能评估工具版本变更风险。

## 重点

- 生产 MCP 的核心是治理，不只是协议调用。
- Tool 描述、Schema 和权限都是版本化资产。
- 观测要覆盖 host、gateway、server、external system。

## 难点

- 模型选择工具的行为会受描述变化影响，传统 API 兼容思路不够。
- 多 server 组合时，调用链和权限链都要可追踪。

## 易错

> **易错：** MCP server 私下接入 host，没有注册、owner、审计和版本记录。
>
> 正确做法：所有生产 server 进入统一注册和治理流程。
