# MCP 学习资料：总览与心智模型

[返回索引](../MCP学习资料.md)

## 学习目标

- 理解 MCP 的定位：让 AI 应用以标准方式连接外部上下文、工具和工作流。
- 建立 host、client、server、外部系统之间的运行时模型。
- 初步区分 Resources、Prompts、Tools、Roots、Sampling、Elicitation。
- 认识 MCP 的主要风险：权限扩大、提示注入、工具误用、数据泄露和供应链风险。

## 理论导读

AI 应用真正落地时，经常需要访问外部系统：文件、数据库、搜索、知识库、代码仓库、工单、浏览器、CI/CD、云资源。每个系统都有自己的 API、认证方式、权限模型和数据格式。如果每个 AI 应用都单独适配一遍，工程成本和安全风险都会迅速膨胀。

MCP 的思路是把这些外部能力包装成统一协议。AI 应用中的 host 通过 MCP client 连接 MCP server，server 再访问真实外部系统。这样 host 不需要知道每个系统的细节，只要理解 MCP 协议，就能发现可用工具、读取资源、获取提示模板，并在权限允许时执行动作。

## 核心心智模型

把 MCP 想成 AI 应用的“外设接口”：

```text
User
  -> Host（AI 应用、IDE、桌面客户端、Agent 平台）
      -> MCP Client（每个 server 一个连接和协议会话）
          -> MCP Server（把外部能力包装成 MCP 能力）
              -> External System（文件、数据库、SaaS、内部平台）
```

关键边界：

- Host 负责用户体验、模型调用、权限展示和最终决策。
- Client 负责和某个 MCP server 建立协议会话。
- Server 负责声明能力、执行工具、提供资源和提示。
- External System 才是真实数据和副作用发生的地方。

## 知识点详解

### 1. MCP 解决的问题

MCP 解决的是 AI 应用连接外部上下文和工具的标准化问题。它不替代模型，不替代数据库，也不替代业务 API。它提供的是一层协议：如何发现能力、如何描述参数、如何调用、如何返回结构化结果、如何处理权限和错误。

适用场景：

- AI IDE 读取工作区文件、运行测试、查询代码索引。
- 企业助手查询工单、CRM、知识库、审批系统。
- 数据分析助手读取数据表结构、执行受控查询。
- 运维助手查询告警、日志和变更记录。
- RAG/Agent 平台把多个内部系统能力统一暴露给模型。

不适用或需谨慎的场景：

- 低延迟高频内部服务调用，直接 API 可能更简单。
- 高危写操作没有权限审批和审计能力。
- 服务端无法隔离租户或真实用户身份。
- 只需要静态文档检索，不需要工具协议。

### 2. MCP 与相邻概念对比

| 概念 | 解决问题 | 与 MCP 的关系 |
| --- | --- | --- |
| Function Calling | 模型输出结构化工具调用意图 | MCP 可作为工具发现和执行协议 |
| REST API | 普通服务间接口 | MCP server 可封装 REST API |
| OpenAPI | 描述 HTTP API | MCP 更关注 AI host、工具、资源、提示和会话能力 |
| RAG | 检索外部知识补充上下文 | MCP resource/tool 可作为 RAG 数据入口 |
| Agent Framework | 规划、执行、记忆、反思 | MCP 提供可执行工具和上下文连接层 |
| Plugin System | 应用扩展机制 | MCP 是面向 AI 上下文和工具的协议化扩展 |

### 3. 六类核心能力

| 能力 | 方向 | 含义 | 例子 |
| --- | --- | --- | --- |
| Tools | Server -> Client | 可执行动作 | 查询候选人、创建工单、运行测试 |
| Resources | Server -> Client | 可读取上下文 | 文件内容、数据库 schema、文档片段 |
| Prompts | Server -> Client | 可复用提示模板 | 代码审查模板、SQL 解释模板 |
| Roots | Client -> Server | 客户端暴露的访问根范围 | 工作区目录、项目根路径 |
| Sampling | Server -> Client | server 请求 host 代为调用模型 | server 需要模型总结资源 |
| Elicitation | Server -> Client | server 请求用户补充信息 | 部署前询问环境和确认 |

> **重点：** Tools、Resources、Prompts 是 server 提供能力；Roots、Sampling、Elicitation 是 client/host 提供给 server 的受控能力。

### 4. 协议不是安全的自动保证

MCP 标准化了连接方式，但不会自动让工具安全。安全取决于 host、server、外部系统和组织治理共同实现：

- 工具是否按真实用户授权。
- 参数是否校验。
- 高危动作是否二次确认。
- server 是否能越权读取资源。
- 模型是否会被资源内容提示注入诱导调用工具。
- 日志是否记录了谁在什么时候调用了什么。

> **易错：** 认为使用 MCP 就天然安全。
>
> 正确做法：把 MCP 当成协议层，安全仍要靠身份、权限、隔离、审批、审计和红队测试。

## 例子

招聘系统 MCP server：

```text
Resources:
  job://schema/application
  job://candidate/{id}/profile

Prompts:
  explain_candidate_match
  interview_question_generator

Tools:
  search_candidates
  create_interview
  update_application_status
```

同一业务能力的建模选择：

- 候选人简历内容：resource，因为它是上下文。
- 候选人匹配分析模板：prompt，因为它是任务结构。
- 创建面试安排：tool，因为它有副作用。

## 练习

1. 选一个你熟悉的系统，列出哪些能力适合作为 resource、prompt、tool。
2. 画出 host、client、server、外部系统的数据流和权限流。
3. 为一个高危 tool 写出需要二次确认的条件。

## 验收

- 能一句话定义 MCP。
- 能画出 MCP 四层架构。
- 能区分 resource、prompt、tool。
- 能说明 MCP 的安全边界不等于业务权限边界。

## 重点

- MCP 是 AI 应用连接外部能力的标准协议。
- Host 是用户和模型的控制中心，server 是外部能力适配层。
- MCP 的价值来自能力发现、协议统一和安全治理入口。

## 难点

- MCP 是双向协议，不只是“模型调用工具”。
- 工具调用结果会回到模型上下文中，可能影响后续决策。
- 生产中权限必须绑定真实用户、租户、资源范围和操作风险。

## 易错

> **易错：** 把 MCP server 设计成万能后门。
>
> 正确做法：每个 server 只暴露必要能力，工具粒度、参数、权限和审计都要可控。
