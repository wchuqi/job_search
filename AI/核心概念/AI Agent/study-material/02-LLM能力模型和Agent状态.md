# AI Agent学习资料：LLM能力模型和Agent状态

[返回索引](../AI Agent学习资料.md)

## 学习目标

- 理解 LLM 在 Agent 中扮演的角色。
- 掌握 Agent 状态的分类和生命周期。
- 知道模型能力边界如何影响系统设计。

## 理论导读

LLM 在 Agent 中通常负责理解、推理、规划、工具选择、结果解释和自然语言输出。但 LLM 不是数据库、不是权限系统、不是执行环境，也不是最终真相来源。

Agent 系统必须把模型擅长的“语义推理”与传统软件擅长的“确定性控制”结合起来。

## LLM能力模型

| 能力 | 用途 | 风险 |
| --- | --- | --- |
| 语义理解 | 理解用户意图 | 可能误解模糊需求 |
| 规划 | 拆解任务 | 可能计划不可执行 |
| 工具选择 | 决定调用哪个工具 | 可能选错工具 |
| 结果解释 | 总结工具输出 | 可能忽略关键错误 |
| 代码/文本生成 | 生成产物 | 可能幻觉或不安全 |
| 自我反思 | 修正错误 | 可能循环或自信错误 |

## Agent状态分类

| 状态类型 | 内容 | 例子 |
| --- | --- | --- |
| 会话状态 | 当前对话和任务上下文 | 用户目标、约束 |
| 工作状态 | 任务执行进度 | 已查订单、待确认退款 |
| 环境状态 | 外部系统状态 | 工单状态、库存状态 |
| 记忆状态 | 跨会话信息 | 用户偏好、历史决策 |
| 策略状态 | 权限和风险规则 | 是否允许付款 |
| 评估状态 | 测试和验证结果 | 检索命中率、任务成功 |

## 生命周期

```text
初始化 -> 接收目标 -> 构造上下文 -> 计划 -> 执行 -> 观察 -> 更新状态 -> 验证 -> 完成/重试/转人工
```

状态更新必须结构化。只把所有历史对话塞回 prompt，会导致上下文膨胀和错误传播。

## 状态存储策略

- 短任务：内存状态即可。
- 长任务：需要 checkpoint。
- 多会话：需要持久化存储。
- 高风险：状态变更需要审计。
- 多 Agent：需要共享状态或事件总线。

## 例子

客服退款 Agent 状态：

```json
{
  "task_id": "refund-123",
  "user_intent": "request_refund",
  "order_id": "O1001",
  "order_status": "SHIPPED",
  "policy_result": "after_sales_required",
  "next_action": "ask_user_to_submit_after_sales",
  "risk_level": "medium"
}
```

## 练习

为“运维告警诊断 Agent”设计状态字段，至少包含：

- 告警信息。
- 已查询系统。
- 假设列表。
- 已排除原因。
- 下一步动作。
- 转人工条件。

## 验收

- 能说明 LLM 能做什么、不能做什么。
- 能设计 Agent 状态模型。
- 能解释为什么状态不能只靠聊天记录。

## 重点

LLM 是 Agent 的推理组件，不是整个 Agent。状态、工具、权限、验证必须由系统设计承担。

## 难点

难点是状态压缩和一致性。长任务中状态会不断变化，系统必须知道哪些是事实、哪些是假设、哪些是模型生成的中间想法。

## 易错

> **易错：** 把模型输出当作事实状态。
>
> 正确做法：事实必须来自工具、数据库、用户确认或验证器；模型输出只能作为候选判断。
