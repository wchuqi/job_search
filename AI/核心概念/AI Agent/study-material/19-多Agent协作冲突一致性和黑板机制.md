# AI Agent学习资料：多Agent协作、冲突、一致性和黑板机制

[返回索引](../AI Agent学习资料.md)

## 学习目标

- 深入理解多 Agent 系统的协作机制和冲突来源。
- 掌握 Supervisor、黑板、消息队列、共享状态和一致性控制。
- 能判断什么时候多 Agent 会降低质量。

## 理论导读

多 Agent 不是把一个任务拆成很多“人格”。它是分布式系统问题：多个执行体共享目标、状态、工具和资源，会产生冲突、重复工作、责任不清、信息不一致和成本膨胀。

多 Agent 的核心问题：

- 谁分配任务？
- 谁拥有状态？
- 谁解决冲突？
- 谁判断完成？
- 谁对最终结果负责？

## 协作模式

| 模式 | 数据流 | 适合 | 风险 |
| --- | --- | --- | --- |
| Supervisor-Worker | 中央调度 | 任务可拆分 | Supervisor 瓶颈 |
| Blackboard | 共享工作区 | 研究分析 | 状态污染 |
| Pipeline | 顺序传递 | 固定阶段 | 上游错误传播 |
| Debate | 多观点竞争 | 开放问题 | 成本高、难验证 |
| Market | 多 Agent 竞标任务 | 大规模任务池 | 调度复杂 |

## 黑板机制

黑板是共享状态空间：

```text
Blackboard:
  goals:
    - 生成市场报告
  facts:
    - 市场规模数据
  hypotheses:
    - 增长来自海外市场
  artifacts:
    - 图表
    - 草稿
  decisions:
    - 使用 2025 年数据
```

关键规则：

- facts 必须有来源。
- hypotheses 必须标记不确定性。
- decisions 必须有 owner。
- artifacts 必须有版本。
- 冲突必须显式记录。

## 冲突类型

| 冲突 | 例子 | 处理 |
| --- | --- | --- |
| 目标冲突 | 一个 Agent 优化成本，另一个优化质量 | Supervisor 决策 |
| 事实冲突 | 两个数据源数字不同 | 权威源优先 |
| 工具冲突 | 两个 Agent 修改同一文件 | 锁或版本控制 |
| 计划冲突 | 子任务顺序不一致 | 重新规划 |
| 评价冲突 | Critic 和 Worker 互不认可 | 外部 verifier |

## 一致性策略

- 单写者原则：同一资源同一时间只有一个 Agent 可写。
- 乐观并发：提交前检查版本。
- 锁机制：高风险资源加锁。
- 事件溯源：记录每次状态变更。
- 外部验证：测试、规则、人审裁决。

## 多Agent拆分准则

适合拆分：

- 不同工具权限。
- 不同专业知识。
- 可独立验证的中间产物。
- 可并行降低时间。
- 可隔离风险。

不适合拆分：

- 任务简单。
- 状态高度耦合。
- 没有验证器。
- 协作成本高于收益。
- 只是为了模拟组织角色。

## 练习

设计一个“代码开发多 Agent 系统”：

- Requirement Agent。
- Coding Agent。
- Test Agent。
- Review Agent。
- Supervisor。

写出共享状态、资源锁、冲突处理、完成标准和转人工条件。

## 验收

- 能说明多 Agent 的分布式系统问题。
- 能设计黑板数据结构。
- 能处理资源冲突和事实冲突。
- 能判断多 Agent 是否值得。

## 重点

多 Agent 的复杂度来自协调，不来自模型调用。没有状态治理，多 Agent 很容易比单 Agent 更差。

## 难点

难点是责任归属。最终结果必须有一个 owner，否则多个 Agent 的错误会互相推诿。

## 易错

> **易错：** 用多个 Agent 互相聊天代替真实验证。
>
> 正确做法：多 Agent 讨论只能产生候选结果，最终必须由工具、测试、规则或人工验证。
