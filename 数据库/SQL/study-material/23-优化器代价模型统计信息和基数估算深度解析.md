# SQL 学习资料：优化器代价模型统计信息和基数估算深度解析

[返回索引](../SQL学习资料.md)

## 学习目标

- 理解优化器如何基于成本选择执行计划。
- 掌握统计信息、选择性、基数估算、数据倾斜对计划的影响。
- 能解释“为什么有索引却不用”“为什么换个参数计划变了”。
- 能建立执行计划诊断框架。

## 理论导读

数据库优化器不是读 SQL 后凭规则拍脑袋，它会枚举一些可行计划，并用成本模型估算每个计划的代价。成本通常综合 I/O、CPU、内存、排序、哈希、随机读取、顺序读取等因素。计划选择最依赖一个基础判断：每一步会产生多少行。这个判断叫基数估算。估算一旦严重错误，后面的 JOIN 顺序、JOIN 算法、索引选择、内存分配都会跟着错。

## 优化器的简化流程

```text
解析 SQL
  -> 语义分析
  -> 查询重写
  -> 生成候选访问路径
  -> 估算基数和成本
  -> 选择最低成本计划
  -> 执行器执行
```

## 统计信息包括什么

不同数据库实现不同，但常见包括：

- 表行数。
- 页数或块数。
- 列 distinct 值数量。
- NULL 比例。
- 最常见值及频率。
- 直方图。
- 列相关性或扩展统计，部分数据库支持。
- 索引高度、叶子页、聚簇因子等。

## 选择性

选择性是谓词过滤后剩余比例：

```text
选择性 = 满足条件的行数 / 总行数
```

例如 `status = 'open'` 如果匹配 90% 行，单列索引收益可能不高；`email = ?` 如果唯一匹配 1 行，索引收益很高。

## 基数估算错误来源

| 来源 | 例子 | 后果 |
| --- | --- | --- |
| 统计信息过期 | 大批导入后未分析 | 估算仍按旧数据 |
| 数据倾斜 | status 90% 为 open | 平均估算误导 |
| 列相关性 | city 和 department 强相关 | 独立性假设失效 |
| 函数包列 | `DATE(created_at)` | 统计难以使用 |
| 参数敏感 | 热门 job_id 和冷门 job_id 差异大 | 同 SQL 不同参数应选不同计划 |
| 隐式转换 | 字符串列用数字比较 | 索引和统计失效 |

## 例子：数据倾斜

```sql
SELECT *
FROM applications
WHERE status = 'screening';
```

如果 `screening` 占 80%，全表扫描可能比索引扫描更便宜；如果只占 0.1%，索引扫描更合理。同一列不同值的选择性差异很大，优化器需要直方图或最常见值统计才能判断。

## JOIN 顺序为什么依赖基数

三表连接：

```sql
SELECT *
FROM applications a
JOIN candidates c ON c.candidate_id = a.candidate_id
JOIN jobs j ON j.job_id = a.job_id
WHERE j.city = 'Shanghai'
  AND c.years_experience >= 5;
```

如果上海岗位很少，应先过滤 jobs；如果高级候选人很少，应先过滤 candidates。优化器根据统计信息决定驱动顺序。如果估算错，可能先处理巨大中间结果。

## 执行计划诊断问题清单

1. 估算行数和实际行数差多少？
2. 错误从哪一步开始扩大？
3. 使用了什么访问路径：全表、索引、位图、覆盖？
4. JOIN 算法是否匹配实际行数？
5. 是否发生额外排序、哈希溢出或临时表？
6. 统计信息是否过期？
7. 谓词是否可 SARGable？
8. 参数值是否具有强倾斜？

## 优化方向

- 更新统计信息。
- 创建更匹配查询的组合索引。
- 改写不可 SARGable 谓词。
- 拆分热门和冷门参数路径。
- 使用扩展统计或列组统计，按数据库支持。
- 对复杂报表分阶段物化中间结果。
- 必要时使用 hint，但应作为最后手段并记录风险。

## 练习

1. 构造 status 数据倾斜，比较热门值和冷门值执行计划。
2. 修改大量数据后不更新统计，观察计划变化。
3. 写一个两列相关条件，分析独立性估算可能出错。
4. 对一个三表 JOIN 解释优化器为什么选择当前顺序。

## 验收

- 能解释成本模型和基数估算。
- 能指出估算错误对 JOIN 顺序和算法的影响。
- 能给出统计信息相关的慢查询排查路径。

## 重点

- 很多慢查询根因不是“没有索引”，而是优化器估算错或索引不匹配真实选择性。

## 难点

- 成本值不是实际毫秒，不同数据库成本单位不可直接比较。

## 易错

> **易错：** 看到有索引没用就认为优化器错了。
>
> 正确做法：先看选择性、返回比例、随机 I/O 成本、排序需求和统计信息。
