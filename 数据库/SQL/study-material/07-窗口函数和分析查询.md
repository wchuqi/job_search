# SQL 学习资料：窗口函数和分析查询

[返回索引](../SQL学习资料.md)

## 学习目标

- 掌握 `OVER`、`PARTITION BY`、`ORDER BY`、窗口帧。
- 能写排名、分组 TopN、累计和、移动平均、相邻行比较。
- 理解窗口函数和 GROUP BY 的区别。

## 理论导读

窗口函数解决的是“既要保留明细行，又要在相关行集合上计算”的问题。GROUP BY 会把多行压成一行；窗口函数不会减少行数，而是在每行旁边增加一个基于窗口计算出的值。数据分析、报表、去重取最新、分组排名基本离不开窗口函数。

## 基础排名

```sql
SELECT
    a.*,
    ROW_NUMBER() OVER (
        PARTITION BY job_id
        ORDER BY applied_at DESC, application_id DESC
    ) AS rn
FROM applications a;
```

每个岗位最新投递：

```sql
WITH ranked AS (
    SELECT
        a.*,
        ROW_NUMBER() OVER (
            PARTITION BY job_id
            ORDER BY applied_at DESC, application_id DESC
        ) AS rn
    FROM applications a
)
SELECT *
FROM ranked
WHERE rn = 1;
```

## ROW_NUMBER、RANK、DENSE_RANK

| 函数 | 并列时 | 示例排名 |
| --- | --- | --- |
| ROW_NUMBER | 不并列，强行编号 | 1,2,3 |
| RANK | 并列，跳号 | 1,1,3 |
| DENSE_RANK | 并列，不跳号 | 1,1,2 |

## 累计统计

```sql
SELECT
    application_id,
    job_id,
    applied_at,
    COUNT(*) OVER (
        PARTITION BY job_id
        ORDER BY applied_at
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_count
FROM applications;
```

## 相邻行比较

```sql
SELECT
    candidate_id,
    applied_at,
    LAG(applied_at) OVER (
        PARTITION BY candidate_id
        ORDER BY applied_at
    ) AS previous_applied_at
FROM applications;
```

## 窗口帧易错点

很多数据库在有 `ORDER BY` 的窗口聚合中默认帧类似 `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`。这对 `LAST_VALUE` 特别容易出错，因为默认帧的“最后一行”可能只是当前行。

更明确：

```sql
LAST_VALUE(status) OVER (
    PARTITION BY candidate_id
    ORDER BY applied_at
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
)
```

## 练习

1. 查询每个岗位最新一条投递。
2. 查询每个部门薪资最高的 3 个岗位。
3. 计算每个候选人的投递间隔。
4. 统计每个岗位按时间累计投递数。

## 验收

- 能解释窗口函数不减少行数。
- 能区分 `ROW_NUMBER`、`RANK`、`DENSE_RANK`。
- 能说明窗口帧影响 `LAST_VALUE` 和累计统计。

## 重点

- `PARTITION BY` 定义窗口分组，`ORDER BY` 定义窗口内顺序。

## 难点

- 窗口帧默认值因函数和数据库而容易被忽略。

## 易错

> **易错：** 用 GROUP BY 做每组 TopN，导致明细列丢失或结果不确定。
>
> 正确做法：使用窗口函数排名后过滤。
