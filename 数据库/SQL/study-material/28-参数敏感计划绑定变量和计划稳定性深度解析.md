# SQL 学习资料：参数敏感计划绑定变量和计划稳定性深度解析

[返回索引](../SQL学习资料.md)

## 学习目标

- 理解同一 SQL 不同参数可能需要不同执行计划。
- 掌握绑定变量、计划缓存、参数嗅探、通用计划和定制计划的风险。
- 能排查“偶发慢”“某个客户慢”“上线后计划变差”。

## 理论导读

很多生产慢查询不是 SQL 文本本身慢，而是参数分布极不均匀。比如 `job_id = 101` 有 100 万投递，`job_id = 999` 只有 10 条投递。冷门参数适合索引扫描，热门参数可能全表或分区扫描更好。如果数据库缓存了一个为冷门参数生成的计划，再拿去执行热门参数，就可能灾难性变慢。这类问题叫参数敏感计划问题，不同数据库有不同术语和机制。

## 参数敏感例子

```sql
SELECT *
FROM applications
WHERE job_id = :job_id
ORDER BY applied_at DESC
LIMIT 50;
```

如果某些岗位极热门：

- 冷门岗位：索引扫描很快。
- 热门岗位：索引扫描加大量回表可能很慢。
- 如果还要排序，计划选择更复杂。

## 绑定变量的价值和代价

价值：

- 防 SQL 注入。
- 复用解析和计划缓存。
- 减少硬解析开销。

代价：

- 优化器有时不知道具体参数选择性。
- 缓存计划可能不适合所有参数。
- 数据倾斜下容易出现偶发慢。

> **重点：** 参数化是安全必要条件，但参数化后的计划稳定性仍要治理。

## 常见表现

- 同一接口大多数快，少数租户或岗位慢。
- 重启数据库或清计划缓存后短暂恢复。
- 换一个参数执行计划不同。
- 统计信息更新后计划突变。
- ORM 生成 SQL 相同但参数选择性差异大。

## 排查路径

```text
慢查询样本
  -> 记录 SQL 文本和真实参数
  -> 对热门/冷门参数分别 EXPLAIN
  -> 比较估算行数和实际行数
  -> 检查直方图和统计信息
  -> 检查计划缓存或 prepared statement 行为
  -> 设计分流、索引或改写方案
```

## 解决方向

| 方向 | 适用 |
| --- | --- |
| 更新统计信息 | 统计过期 |
| 更合适组合索引 | 查询模式固定 |
| 热点参数特殊路径 | 数据倾斜明显 |
| 分区或分表 | 大租户/大岗位隔离 |
| 强制重新编译或定制计划 | 数据库支持且成本可控 |
| SQL 拆分 | 不同参数走不同 SQL |
| Hint 或 plan guide | 最后手段，需记录风险 |

## ORM 相关风险

ORM 常隐藏 SQL 细节：

- 自动生成 `IN` 大列表。
- N+1 查询。
- 绑定参数类型不匹配导致隐式转换。
- 动态条件导致计划缓存碎片。
- 预加载 JOIN 导致行数放大。

SQL 优化时必须拿到真实 SQL 和参数，而不是只看应用代码。

## 练习

1. 构造热门和冷门 `job_id`，比较执行计划。
2. 用绑定参数和字面量分别观察计划差异。
3. 设计热点岗位特殊查询路径。
4. 找一条 ORM SQL，检查参数类型是否匹配列类型。

## 验收

- 能解释参数敏感计划。
- 能说明绑定变量的安全价值和计划风险。
- 能给出偶发慢查询的参数维度排查方案。

## 重点

- 慢查询必须保存真实参数，否则很多计划问题无法复现。

## 难点

- 不同数据库对计划缓存、参数嗅探、通用计划的策略差异很大。

## 易错

> **易错：** 只用一个测试参数验证 SQL 性能。
>
> 正确做法：覆盖热门、冷门、空结果、大结果、边界时间等参数组合。
