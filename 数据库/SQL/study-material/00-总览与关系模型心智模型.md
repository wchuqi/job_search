# SQL 学习资料：总览与关系模型心智模型

[返回索引](../SQL学习资料.md)

## 学习目标

- 理解 SQL 是声明式、集合式、关系式查询语言。
- 建立表、关系、约束、结果集、逻辑执行顺序和优化器的总体图景。
- 知道 SQL 学习的高风险点：NULL、JOIN、聚合粒度、事务、索引和方言差异。

## 理论导读

SQL 的核心不是“写命令操作数据库”，而是用关系模型描述数据和结果。你告诉数据库“我需要满足这些条件的结果集”，数据库优化器再决定使用哪个索引、先扫哪张表、用什么 JOIN 算法、是否改写子查询。好的 SQL 学习路径要同时建立两层思维：逻辑层保证结果正确，物理层解释性能表现。

关系模型把数据看成一张张有结构的表。表不是 Excel 的随意网格，而是带列类型、约束和业务语义的关系。主键定义实体身份，外键定义实体关系，唯一约束定义业务不变量，检查约束定义取值边界。没有约束的数据模型，很容易把业务正确性推给应用层，最终在并发和异常场景下出错。

## 核心心智模型

```text
业务事实
  -> 表结构和约束
  -> SQL 逻辑查询
  -> 优化器选择物理计划
  -> 执行器读取、连接、过滤、聚合、排序
  -> 结果集
```

## SQL 能力地图

| 领域 | 必须掌握 | 高阶能力 |
| --- | --- | --- |
| 查询基础 | SELECT、WHERE、ORDER BY、分页 | 逻辑顺序、谓词下推、TopN |
| NULL | 三值逻辑、`IS NULL` | `NOT IN` 陷阱、外连接 NULL |
| JOIN | INNER/LEFT/RIGHT/FULL | 半连接、反连接、JOIN 算法 |
| 聚合 | GROUP BY、HAVING、COUNT | 粒度控制、条件聚合、ROLLUP |
| 窗口 | ROW_NUMBER、RANK、SUM OVER | 窗口帧、分组 TopN、累计分析 |
| 写操作 | INSERT、UPDATE、DELETE | UPSERT、幂等、批量写入 |
| 建模 | 主键、外键、唯一、范式 | 反范式、约束驱动正确性 |
| 事务 | ACID、隔离级别 | 丢失更新、写偏斜、锁等待 |
| 性能 | 索引、执行计划 | 基数估算、SARGability、查询重写 |
| 安全 | 参数化查询 | SQL 注入、最小权限、审计 |
| 方言 | 标准 SQL | PostgreSQL/MySQL/Oracle/SQL Server 差异 |

## SQL 的逻辑执行顺序

书写顺序：

```sql
SELECT ...
FROM ...
WHERE ...
GROUP BY ...
HAVING ...
ORDER BY ...
LIMIT ...
```

逻辑顺序更接近：

```text
FROM / JOIN
  -> WHERE
  -> GROUP BY
  -> HAVING
  -> SELECT
  -> DISTINCT
  -> ORDER BY
  -> LIMIT/OFFSET
```

> **重点：** 逻辑执行顺序用于理解结果正确性；实际数据库可能由优化器改写，物理执行顺序不一定如此。

## 例子：同一查询的逻辑解释

```sql
SELECT job_id, COUNT(*) AS application_count
FROM applications
WHERE status <> 'cancelled'
GROUP BY job_id
HAVING COUNT(*) >= 10
ORDER BY application_count DESC;
```

逻辑上：

1. 从 `applications` 取数据。
2. 过滤取消的投递。
3. 按岗位分组。
4. 每组计数。
5. 保留计数大于等于 10 的组。
6. 按计数倒序排序。

## 常见误区

> **易错：** 在 `WHERE` 中使用聚合结果。
>
> 正确做法：`WHERE` 发生在分组前，聚合条件应放在 `HAVING`。

> **易错：** 认为 SQL 按书写顺序逐行执行。
>
> 正确做法：先用逻辑执行顺序保证结果，再用执行计划理解数据库实际怎么做。

## 练习

1. 画出一条含 JOIN、WHERE、GROUP BY、HAVING、ORDER BY 的逻辑执行顺序。
2. 找一个自己写过的 SQL，指出每个条件属于行级过滤还是分组过滤。
3. 写出招聘系统需要的 6 个核心实体和它们之间的关系。

## 验收

- 能解释 SQL 为什么是声明式语言。
- 能说出逻辑执行顺序。
- 能区分逻辑结果正确性和物理执行性能。

## 重点

- SQL 学习先保正确性，再谈性能。
- 约束是数据库层面的业务规则，不是可有可无的装饰。

## 难点

- SQL 结果集是集合/多重集合思维，和面向对象逐个对象处理很不一样。

## 易错

> **易错：** 没有建唯一约束，却以为应用代码查重能保证不重复。
>
> 正确做法：业务不变量尽量落到数据库约束，并在应用层处理冲突。
