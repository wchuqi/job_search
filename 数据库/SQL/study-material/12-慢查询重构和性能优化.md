# SQL 学习资料：慢查询重构和性能优化

[返回索引](../SQL学习资料.md)

## 学习目标

- 掌握慢查询诊断流程。
- 能从 SQL 写法、索引、统计信息、JOIN、排序、分页、聚合角度优化。
- 能写出优化前后对比和验证结论。

## 理论导读

慢查询优化不是把 SQL 改得更短，而是减少不必要的数据读取、连接、排序、聚合和回表。你必须先知道慢在哪里：扫描太多、连接行数爆炸、排序落盘、索引不匹配、估算错误、锁等待，还是网络返回数据太多。没有执行计划和实际指标的“优化”很可能只是改写风格。

## 诊断流程

```text
确认慢查询
  -> 拿原 SQL、参数、频率、耗时分布
  -> 看执行计划
  -> 确认返回行数和扫描行数
  -> 检查索引和统计信息
  -> 判断 JOIN/排序/聚合/分页瓶颈
  -> 改写 SQL 或索引
  -> 对比执行计划和耗时
  -> 回归正确性
```

## 常见优化方向

### 1. 减少扫描

```sql
WHERE applied_at >= '2026-03-01'
  AND applied_at < '2026-04-01'
```

配合索引比函数包列更好。

### 2. 避免 JOIN 放大

先按正确粒度聚合，再 JOIN：

```sql
WITH app_stats AS (
    SELECT job_id, COUNT(*) AS application_count
    FROM applications
    GROUP BY job_id
)
SELECT j.title, s.application_count
FROM jobs j
JOIN app_stats s ON s.job_id = j.job_id;
```

### 3. TopN 优化

合适索引：

```sql
CREATE INDEX idx_applications_job_time
ON applications (job_id, applied_at DESC, application_id DESC);
```

查询：

```sql
SELECT *
FROM applications
WHERE job_id = 101
ORDER BY applied_at DESC, application_id DESC
LIMIT 20;
```

### 4. 深分页改写

OFFSET 深分页改 keyset pagination。

### 5. 只取必要列

避免：

```sql
SELECT *
```

尤其在宽表、JOIN、多行返回、网络传输高的场景。

## 练习

1. 写一个含函数包列的慢查询并改写。
2. 写一个 JOIN 后聚合翻倍的错误查询并修复。
3. 写一个 OFFSET 100000 的分页并改成 keyset。

## 验收

- 能按流程诊断慢查询。
- 能解释优化为什么有效。
- 能提供执行计划和耗时对比。

## 重点

- 优化必须有证据：执行计划、扫描行数、耗时、资源使用。

## 难点

- 同一 SQL 在不同参数下可能选择不同计划，参数敏感性也要验证。

## 易错

> **易错：** 只改 SQL 格式，不看执行计划。
>
> 正确做法：用计划和指标证明优化有效。
