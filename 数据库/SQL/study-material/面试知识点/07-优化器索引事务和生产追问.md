## 一、优化器追问链

### 1. 为什么同一条 SQL 有时快有时慢？

**参考答案：**

可能是参数选择性不同、缓存状态不同、统计信息变化、执行计划变化、锁等待、并发负载、临时排序或哈希溢出导致。深入排查要保留真实 SQL、真实参数、执行时间、执行计划、扫描行数、返回行数和等待事件。

**追问：什么是参数敏感计划？**

同一 SQL 模板在不同参数下数据量差异很大，理想执行计划不同。例如热门岗位有百万投递，冷门岗位只有几条。数据库如果复用一个不适合当前参数的缓存计划，就会出现偶发慢。

**高分判断点：**

- 不只说“加索引”。
- 能提到参数、统计信息、计划缓存、锁等待。
- 能要求真实参数复现。

### 2. 优化器为什么有时不用索引？

**参考答案：**

索引不是必然更快。如果条件选择性低、返回大量行、需要频繁回表、索引不支持排序、统计信息认为全表扫描更便宜，优化器可能不用索引。此外函数包列、隐式转换、前导通配 LIKE 可能让索引不可用。

**追问：怎么判断优化器是不是估错了？**

看执行计划中的估算行数和实际行数。如果某一步估算 10 行实际 100 万行，后续 JOIN 和排序计划很可能都被带偏。再检查统计信息、直方图、数据倾斜和谓词写法。

### 3. 成本值是不是执行毫秒？

**参考答案：**

不是。成本是优化器内部用于比较候选计划的相对估算，综合 I/O、CPU、内存、排序等因素。不同数据库、不同版本成本单位不同，不能把 cost 直接当毫秒。

## 二、索引追问链

### 1. 组合索引为什么讲列顺序？

**参考答案：**

B+Tree 按索引列顺序排序。前导列决定能否快速定位范围。通常等值过滤列放前面，再考虑范围列、排序列和覆盖列。但最终要结合查询频率、选择性和排序需求。

**追问：范围列后面的列是不是完全没用？**

不一定。范围列后的列通常难以继续用于缩小扫描范围，但可能仍用于覆盖查询、过滤下推或部分排序，具体看数据库能力和计划。

### 2. 覆盖索引为什么快？

**参考答案：**

覆盖索引包含查询所需列，执行器可以只读索引，不必回表读取完整行，减少随机 I/O。但覆盖索引会变宽，增加写入和存储成本。

**追问：是不是把所有 SELECT 列都加进索引？**

不是。只对高频关键查询做覆盖，宽索引会拖慢写入、占用缓存、增加维护成本。

### 3. 什么是 SARGability？

**参考答案：**

谓词能否作为搜索参数有效利用索引。`DATE(created_at) = ...`、隐式类型转换、`LIKE '%abc'` 常破坏 SARGability。应改成范围查询、类型匹配和前缀匹配。

## 三、事务追问链

### 1. 事务能解决所有并发问题吗？

**参考答案：**

不能。事务必须结合隔离级别、锁、MVCC、约束和冲突重试。低隔离级别可能出现不可重复读、幻读、丢失更新；快照隔离下还可能出现写偏斜。

**追问：丢失更新和写偏斜有什么区别？**

丢失更新通常是多个事务读同一行后写回覆盖；写偏斜是多个事务分别更新不同的行，但共同破坏一个跨行约束。丢失更新可用原子更新或版本号解决；写偏斜常需要 Serializable、显式锁或重新建模约束。

### 2. 如何防止并发重复插入？

**参考答案：**

必须在数据库层建立唯一约束，然后使用 UPSERT 或捕获唯一冲突。应用层先查再插不能防并发，因为两个事务可能同时查到不存在。

### 3. 死锁如何预防和处理？

**参考答案：**

预防包括统一资源访问顺序、缩短事务、给条件建索引减少锁范围、不在事务中调用外部服务。处理上要捕获死锁错误并重试，同时分析死锁日志定位资源顺序问题。

## 四、复杂报表追问链

### 1. 报表数字翻倍怎么排查？

**参考答案：**

先定义指标粒度，再逐层检查 JOIN 后行数变化。多事实表或一对多明细表连接前要先按业务粒度去重或聚合。例如投递连接多轮面试前，面试表应先按 `application_id` 去重，否则投递漏斗会被面试轮次放大。

### 2. 复杂 SQL 怎么写得可维护？

**参考答案：**

用 CTE 分层：base 明细层、cleaned 清洗层、fact 粒度统一层、agg 聚合层、final 展示层。每层说明粒度和过滤条件，并写校验 SQL。

### 3. 如何证明报表口径正确？

**参考答案：**

写口径文档，明确时间范围、去重规则、状态范围、迟到数据、权限脱敏；构造边界数据；对每层行数和核心指标做校验；与已知样本或人工计算对账。

## 五、生产 SQL 追问链

### 1. 生产执行 UPDATE 脚本前要检查什么？

**参考答案：**

先 SELECT 确认影响范围；确认 WHERE 命中索引；评估锁范围和事务时长；准备备份旧值或反向脚本；小批量执行；记录影响行数；观察慢查询、锁等待和复制延迟。

### 2. DDL 上线为什么危险？

**参考答案：**

DDL 可能锁表、重写表、阻塞读写、造成复制延迟或磁盘压力。大表加字段、加 NOT NULL、建唯一索引、修改列类型都要评估目标数据库在线 DDL 能力和回滚方案。

### 3. 如何建立 SQL 执行计划基线？

**参考答案：**

记录核心 SQL 的指纹、典型参数、执行计划、依赖索引、平均和分位耗时、扫描行数、返回行数。上线前后、统计信息更新后、数据库升级后对比计划。

## 六、评分标准

| 等级 | 表现 |
| --- | --- |
| 初级 | 会写 SELECT、JOIN、GROUP BY |
| 中级 | 能处理 NULL、外连接、窗口函数和基础索引 |
| 高级 | 能读执行计划、解释基数估算、处理事务并发 |
| 资深 | 能做 SQL 生产评审、容量治理、计划稳定性和数据恢复设计 |

## 七、回答禁忌

- 不要慢查询一律回答“加索引”。
- 不要把 `WHERE` 和 `ON` 条件随意移动。
- 不要用 DISTINCT 掩盖 JOIN 粒度错误。
- 不要说“事务能保证并发没问题”而不说明隔离级别。
- 不要给生产 DML 脚本却没有回滚和影响行确认。
