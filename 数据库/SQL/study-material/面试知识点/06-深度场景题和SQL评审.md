## 一、深度场景题和 SQL 评审

### 1. 一个统计 SQL 结果比预期大很多，你怎么排查？

**参考答案：**

先确认统计业务粒度，再检查 JOIN 是否一对多或多对多放大行数。逐步查看每次 JOIN 后行数，确认 `COUNT(*)` 统计的是哪种实体。如果需要统计投递数，不应在连接面试多轮后直接 `COUNT(*)`，可以先按投递聚合，或使用 `COUNT(DISTINCT application_id)`，但要说明 DISTINCT 的成本。

> **重点：** 先定位粒度错误，再谈性能。

### 2. 一个 LEFT JOIN 查询少了未匹配数据，如何判断？

**参考答案：**

检查 WHERE 是否引用了右表列。右表未匹配时列为 NULL，WHERE 条件会过滤 UNKNOWN。若需求是保留左表所有行，应把右表过滤条件放到 ON，或在 WHERE 中显式保留 `right.id IS NULL`。

### 3. 一个查询索引存在但没用，可能是什么原因？

**参考答案：**

可能是函数包列、隐式类型转换、LIKE 前导通配、组合索引列顺序不匹配、返回行比例太高、统计信息过期、OR 条件复杂、排序与索引不匹配。需要看执行计划和谓词条件。

### 4. 如何评审一条生产 SQL？

**参考答案：**

从四个维度评审：

- 正确性：NULL、JOIN 粒度、聚合粒度、边界条件。
- 性能：索引、执行计划、扫描行数、排序、分页、返回列。
- 并发：事务范围、锁、幂等、唯一约束。
- 安全：参数化、权限、敏感字段、导出风险。

### 5. 如何设计“候选人不能重复投递同一岗位”？

**参考答案：**

数据库层建立 `UNIQUE(candidate_id, job_id)`。写入时使用 UPSERT 或捕获唯一约束冲突。应用层可以先查用于用户体验，但不能作为唯一保护。

### 6. 每个岗位最新 3 条投递，如何保证稳定？

**参考答案：**

使用窗口函数，并在排序中加入唯一键作为 tie-breaker。

```sql
WITH ranked AS (
    SELECT
        a.*,
        ROW_NUMBER() OVER (
            PARTITION BY job_id
            ORDER BY applied_at DESC, application_id DESC
        ) AS rn
    FROM applications a
)
SELECT *
FROM ranked
WHERE rn <= 3;
```

### 7. 线上慢查询优化后，如何证明没有改错？

**参考答案：**

用相同参数和边界数据对比新旧 SQL 结果集；对比执行计划、扫描行数、耗时和资源使用；覆盖 NULL、无匹配、多匹配、重复值、边界时间；必要时灰度发布并监控错误率和慢查询分布。

### 8. SQL 面试回答的高分结构是什么？

**参考答案：**

按“直接结论 -> SQL 语义 -> 示例 -> 性能影响 -> 方言边界 -> 常见陷阱”回答。比如讲窗口函数时，不只说语法，还要说它不减少行数、窗口帧影响结果、排序需要成本、不同数据库函数支持有差异。

## 二、SQL 评审清单

| 维度 | 检查项 |
| --- | --- |
| 正确性 | NULL、外连接、聚合粒度、重复行 |
| 性能 | 索引、SARGability、返回列、排序、分页 |
| 并发 | 事务、锁、唯一约束、幂等 |
| 安全 | 参数化、白名单、最小权限、脱敏 |
| 兼容 | 方言函数、分页、UPSERT、时间和排序 |
