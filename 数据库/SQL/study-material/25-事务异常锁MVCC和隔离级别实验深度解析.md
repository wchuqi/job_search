# SQL 学习资料：事务异常锁MVCC和隔离级别实验深度解析

[返回索引](../SQL学习资料.md)

## 学习目标

- 深入理解事务异常不是背名词，而是并发读写时的状态变化。
- 能区分脏读、不可重复读、幻读、丢失更新、写偏斜。
- 能解释锁、MVCC、隔离级别和唯一约束如何共同保护正确性。
- 能设计可复现实验验证并发行为。

## 理论导读

事务隔离的难点在于“两个事务都觉得自己没错，合起来结果错了”。数据库用锁和 MVCC 在并发与一致性之间做取舍。锁让冲突操作等待，MVCC 让读操作看到某个版本快照，约束在写入时保护不变量。不同数据库隔离级别实现差异很大，所以深入学习必须通过实验验证目标数据库。

## 并发异常图谱

| 异常 | 现象 | 常见解决 |
| --- | --- | --- |
| 脏读 | 读到未提交数据 | READ COMMITTED 及以上 |
| 不可重复读 | 同事务两次读同一行值不同 | REPEATABLE READ 或快照 |
| 幻读 | 同事务两次范围查询行集合不同 | 范围锁、Serializable、谓词锁等 |
| 丢失更新 | 两个写覆盖同一值 | 原子更新、锁、乐观版本 |
| 写偏斜 | 分别更新不同行，破坏跨行约束 | Serializable、显式锁、约束建模 |

## 丢失更新实验

错误流程：

```text
T1: SELECT value FROM counters WHERE id = 1; -- 10
T2: SELECT value FROM counters WHERE id = 1; -- 10
T1: UPDATE counters SET value = 11 WHERE id = 1;
T2: UPDATE counters SET value = 11 WHERE id = 1;
```

正确：

```sql
UPDATE counters
SET value = value + 1
WHERE id = 1;
```

或乐观锁：

```sql
UPDATE counters
SET value = value + 1, version = version + 1
WHERE id = 1
  AND version = :old_version;
```

如果影响行数为 0，说明并发冲突，需要重试或返回失败。

## 写偏斜实验

业务规则：至少保留一名面试官可用。

```text
interviewers(id, available)
1, true
2, true
```

T1 看到有 2 人可用，于是把 1 设为不可用。T2 同时看到有 2 人可用，于是把 2 设为不可用。两个事务分别更新不同行，最终 0 人可用，破坏跨行规则。

解决方向：

- 提升到 Serializable，并处理重试。
- 显式锁定规则相关的范围或汇总行。
- 把业务规则转为可被唯一/检查约束保护的模型。
- 使用单行计数器或资源表串行化关键约束。

## MVCC 心智模型

MVCC 让读操作看到符合快照规则的版本，而不是总被写锁阻塞。它提升读写并发，但也带来：

- 旧版本清理成本。
- 长事务保留旧版本。
- 快照读和当前读差异。
- 不同隔离级别快照范围差异。

> **重点：** MVCC 不是“不加锁”，写写冲突和唯一约束仍需要锁或冲突检测。

## 隔离级别要结合数据库

- PostgreSQL 的 Repeatable Read 基于快照隔离，可防很多幻读表现，但写偏斜仍需 Serializable。
- MySQL InnoDB 的 Repeatable Read 配合 next-key lock 在某些当前读范围更新中防幻读，但快照读语义不同。
- SQL Server 可配置锁式读提交或快照隔离。

面试或生产讨论必须说清目标数据库。

## 死锁分析

死锁典型模式：

```text
T1 锁 A，等待 B
T2 锁 B，等待 A
```

预防：

- 统一资源访问顺序。
- 缩短事务时间。
- 给过滤条件建索引，减少锁范围。
- 不在事务内调用外部服务。
- 捕获死锁错误并重试。

## 练习

1. 用两个数据库会话复现丢失更新。
2. 用版本号条件更新修复。
3. 设计写偏斜实验并说明目标数据库表现。
4. 制造死锁并读死锁日志。

## 验收

- 能用时间线解释每种并发异常。
- 能区分丢失更新和写偏斜。
- 能说明 MVCC、锁和约束的职责边界。

## 重点

- 事务正确性要用实验和目标数据库文档验证。

## 难点

- 同名隔离级别在不同数据库中并不完全等价。

## 易错

> **易错：** 认为用了事务就自动没有并发问题。
>
> 正确做法：明确隔离级别、读写模式、约束和冲突处理策略。
