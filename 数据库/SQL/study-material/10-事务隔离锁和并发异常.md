# SQL 学习资料：事务隔离锁和并发异常

[返回索引](../SQL学习资料.md)

## 学习目标

- 理解 ACID、事务边界和隔离级别。
- 掌握脏读、不可重复读、幻读、丢失更新、写偏斜。
- 能用约束、锁、隔离级别和条件更新解决并发问题。

## 理论导读

数据库不是单用户脚本环境。多个事务同时读写同一批数据时，如果没有隔离规则，就会出现读到未提交数据、重复读取结果变化、范围内突然出现新行、两个事务互相覆盖等问题。隔离级别是在一致性和并发性能之间做取舍。SQL 学习必须把“单条语句正确”升级为“并发执行也正确”。

## ACID

| 特性 | 含义 |
| --- | --- |
| Atomicity | 事务内操作要么都成功，要么都失败 |
| Consistency | 事务前后满足约束和业务规则 |
| Isolation | 并发事务互相隔离 |
| Durability | 提交后持久化 |

## 隔离级别

| 隔离级别 | 典型防止 |
| --- | --- |
| READ UNCOMMITTED | 最弱，可能脏读 |
| READ COMMITTED | 防脏读 |
| REPEATABLE READ | 防不可重复读，具体幻读行为依数据库 |
| SERIALIZABLE | 最强，接近串行执行 |

> **重点：** 隔离级别同名不代表不同数据库实现完全相同。MySQL InnoDB、PostgreSQL、SQL Server 的细节不同。

## 丢失更新

错误模式：

```text
T1 读 count=10
T2 读 count=10
T1 写 count=11
T2 写 count=11
```

修复：

```sql
UPDATE counters
SET value = value + 1
WHERE counter_id = 1;
```

或使用版本号：

```sql
UPDATE applications
SET status = 'interviewing', version = version + 1
WHERE application_id = 1003
  AND version = 7;
```

## 写偏斜

两个事务分别检查“至少还有一个面试官可用”，然后各自占用不同面试官，最终违反整体约束。这类问题常见于跨多行约束，可能需要 Serializable、显式锁或把约束转换为可被唯一约束保护的模型。

## 锁等待和死锁

死锁不是锁坏了，而是事务以不同顺序持有和等待资源。预防：

- 统一访问顺序。
- 缩短事务。
- 不在事务中做慢外部调用。
- 合理索引减少锁范围。
- 捕获死锁错误并重试。

## 练习

1. 设计一个并发投递场景，用唯一约束防重复。
2. 写条件 UPDATE 防止状态被覆盖。
3. 解释两个事务如何产生死锁。

## 验收

- 能解释四个隔离级别。
- 能说明丢失更新和写偏斜的区别。
- 能给出并发写入的数据库侧保护方案。

## 重点

- 并发正确性不能只靠应用层先查再写。

## 难点

- 隔离级别的实现依赖数据库，面试和生产都要说明具体数据库。

## 易错

> **易错：** 把事务开得很大，把远程调用也放进事务。
>
> 正确做法：事务只包必要数据库操作，尽量短、可重试、访问顺序稳定。
