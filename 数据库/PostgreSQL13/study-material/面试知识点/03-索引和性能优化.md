## 一、索引和性能优化

### 1. PostgreSQL 常见索引类型有哪些？

**参考答案：**

常见索引类型包括：

- B-tree：默认索引，适合等值、范围、排序。
- Hash：等值查询，实际使用频率较低。
- GIN：数组、`jsonb`、全文检索。
- GiST：几何、范围、排斥约束、全文检索。
- SP-GiST：适合特定空间或分区结构。
- BRIN：超大表且物理顺序与字段相关，例如时间序列表。

> **重点：** 大多数普通查询先考虑 B-tree，不同数据类型和操作符决定可用索引类型。

### 2. 为什么有索引但查询没有走索引？

**参考答案：**

常见原因：

- 查询返回行数占比很高，顺序扫描更便宜。
- 统计信息过旧或估算错误。
- 查询条件和索引表达式不匹配。
- 复合索引列顺序不适合当前谓词。
- 函数或类型转换让索引无法直接使用。
- 部分索引谓词没有被查询条件蕴含。

验证方法：

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT ...
```

> **重点：** 优化器选择成本最低计划，不是看见索引就使用。

### 3. 复合索引的列顺序怎么设计？

**参考答案：**

要结合查询条件和排序：

- 等值条件列通常放前面。
- 范围条件列之后的列较难继续缩小扫描范围。
- 如果查询有排序，索引列顺序和排序方向要能服务 `ORDER BY`。
- 高频查询优先，不为低频组合牺牲核心路径。

```sql
CREATE INDEX idx_application_job_status_created
ON application(job_id, status, created_at DESC);
```

适合 `WHERE job_id = ? AND status = ? ORDER BY created_at DESC LIMIT ?`。

> **难点：** 没有脱离 SQL 和数据分布的固定公式，必须用执行计划验证。

### 4. 什么是部分索引？适合什么场景？

**参考答案：**

部分索引只索引满足谓词的行，适合数据中只有一部分是高频查询目标的场景。例如只查询开放岗位：

```sql
CREATE INDEX idx_job_open_created
ON job(created_at DESC)
WHERE status = 'open';
```

查询条件必须能匹配或蕴含 `status = 'open'`，优化器才会考虑该索引。

> **易错：** 建了部分索引，但查询没有写出匹配条件。

### 5. 什么是覆盖索引和 index-only scan？

**参考答案：**

覆盖索引用索引键加 `INCLUDE` 列覆盖查询需要的字段，减少访问堆表。

```sql
CREATE INDEX idx_application_job_created_include
ON application(job_id, created_at DESC)
INCLUDE (status, candidate_id);
```

但 index-only scan 还需要可见性映射确认页面中行对当前事务可见。如果表频繁更新，仍可能回表。

> **重点：** 覆盖了列不等于一定 index-only scan。

### 6. 如何阅读 `EXPLAIN (ANALYZE, BUFFERS)`？

**参考答案：**

重点看：

- 实际耗时在哪个节点。
- 估算行数和实际行数是否差很多。
- `loops` 是否放大了内层操作。
- `Buffers` 中 shared hit/read 是否显示大量 I/O。
- 是否出现外部排序或 hash batch。

> **重点：** 估算行数偏差大时，优先检查统计信息和数据分布。

### 7. Nested Loop、Hash Join、Merge Join 怎么选择？

**参考答案：**

优化器自动选择。一般来说：

- Nested Loop 适合外表小、内表有索引的情况。
- Hash Join 适合较大等值连接。
- Merge Join 适合两边已有序或排序成本可接受的情况。

如果 Nested Loop 外表实际行数远大于估算值，就可能造成大量重复索引查找。

> **难点：** Join 算法问题常常不是算法本身，而是行数估算错误。

### 8. `work_mem` 调大一定能提升性能吗？

**参考答案：**

不一定。`work_mem` 影响排序、哈希、聚合等操作，但它是每个会话中每个操作节点都可能使用的内存。并发高时全局调大可能导致内存耗尽。

正确做法是针对具体 SQL 验证，必要时在会话或事务局部设置。

```sql
SET LOCAL work_mem = '64MB';
```

> **易错：** 看到排序落盘就全局大幅调高 `work_mem`。
