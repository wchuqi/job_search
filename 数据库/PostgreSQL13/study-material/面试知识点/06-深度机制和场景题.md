## 一、深度机制和场景题

### 1. 一条 UPDATE 在 PostgreSQL 内部大致发生什么？

**参考答案：**

普通 UPDATE 通常不是原地覆盖。PostgreSQL 会把旧 tuple 标记为被当前事务删除或替换，并插入一个新 tuple。旧 tuple 的 `xmax` 记录删除或替换它的事务，新 tuple 的 `xmin` 记录创建它的事务。若更新列涉及索引，索引也需要新增或调整条目；如果没有更新被索引列且同页有空间，可能发生 HOT update，减少索引维护。

同时，修改会产生 WAL。事务提交时提交相关 WAL 必须按 `synchronous_commit` 策略持久化。旧 tuple 后续由 VACUUM 在没有活跃 snapshot 需要它时清理。

> **重点：** UPDATE 会影响 heap、index、WAL、VM 和 VACUUM，不只是改一行字段。
>
> **易错：** 以为 UPDATE 总是原地覆盖，所以不会造成膨胀。

### 2. 为什么覆盖索引存在时，Index Only Scan 仍可能回表？

**参考答案：**

Index Only Scan 需要两个条件：查询所需列都能从索引获得，并且 heap page 的 visibility map 标记为 all-visible。PostgreSQL 仍要保证 MVCC 可见性，如果 VM 位没有证明页面对所有事务可见，就必须访问 heap 检查 tuple 可见性。

频繁 UPDATE、DELETE 或 INSERT 会清除相关页面的 VM 位，因此覆盖索引在高更新表上不一定稳定减少 heap 访问。

> **重点：** 覆盖列只是必要条件，visibility map 决定能否避免可见性回表。

### 3. 长事务为什么会同时影响 VACUUM、膨胀和复制？

**参考答案：**

长事务持有旧 snapshot，PostgreSQL 必须保留该 snapshot 可能看到的旧 tuple。VACUUM 无法清理这些 tuple，导致表和索引膨胀。如果备库启用 `hot_standby_feedback`，备库长查询也可能把类似的保留需求反馈给主库，使主库不能清理旧版本。

长事务还可能导致 `xmin` 推进缓慢，增加 freeze 压力。复制侧如果备库重放和查询冲突，也可能出现查询取消或重放延迟。

> **难点：** 长事务不是只占一个连接，它会拖住 MVCC 清理边界。

### 4. `VACUUM` 为什么执行后磁盘文件没有变小？

**参考答案：**

普通 VACUUM 主要清理 dead tuple，并把空间标记为表内可复用。它通常不会把 relation 文件尾部以外的空洞归还给操作系统。只有当空间位于文件尾部且满足条件时才可能截断一部分。要显著缩小文件，通常需要重写表，例如 `VACUUM FULL`、`CLUSTER`、某些在线重写工具或重建表。

但 `VACUUM FULL` 需要强锁，会阻塞读写，生产必须评估窗口。

> **重点：** 普通 VACUUM 的价值是空间复用、VM 更新和防回卷，不是保证文件立刻变小。

### 5. 为什么某条 SQL 在绑定参数后比直接写常量慢？

**参考答案：**

绑定参数和 prepared statement 可能使用 generic plan。generic plan 不能针对每次具体参数做最佳估算。如果数据分布高度倾斜，例如 `status='active'` 返回 1%，`status='archived'` 返回 95%，不同参数的最佳计划可能不同。generic plan 可能对某些参数很差。

排查方式：

```sql
EXPLAIN (ANALYZE, BUFFERS) SELECT ... WHERE status = 'active';
EXPLAIN (ANALYZE, BUFFERS) SELECT ... WHERE status = 'archived';
SET plan_cache_mode = force_custom_plan;
```

`plan_cache_mode` 更适合诊断或局部调整，不应无脑全局设置。

> **重点：** 参数化 SQL 的计划问题常和数据倾斜、generic plan、统计信息有关。

### 6. 如何解释 `EXPLAIN` 中估算 rows 和 actual rows 差异巨大？

**参考答案：**

这通常表示优化器对选择率估算不准。原因可能包括统计信息过旧、采样不足、数据倾斜、多列相关性、表达式条件没有统计信息、参数化查询导致泛化估算等。

处理方向：

```sql
ANALYZE table_name;
ALTER TABLE table_name ALTER COLUMN col SET STATISTICS 1000;
CREATE STATISTICS st_name (dependencies, mcv) ON col1, col2 FROM table_name;
ANALYZE table_name;
```

> **重点：** 行数估算错误会进一步导致 join 顺序、join 算法和扫描方式错误。

### 7. 线上 WAL 暴涨，你如何判断是归档、复制槽还是写入峰值？

**参考答案：**

先看归档日志和 `archive_command` 是否失败；再查复制槽：

```sql
SELECT slot_name, slot_type, active, restart_lsn, confirmed_flush_lsn
FROM pg_replication_slots;
```

再查复制状态：

```sql
SELECT application_name, state, sent_lsn, replay_lsn, replay_lag
FROM pg_stat_replication;
```

同时结合业务写入量、批处理、索引重建、备份任务和磁盘曲线。归档失败会保留未归档 WAL；复制槽滞后会保留消费者需要的 WAL；写入峰值会自然产生大量 WAL。

> **易错：** 手工删除 `pg_wal` 文件。
>
> 正确做法：修复根因或扩容，确认无用复制槽后才删除 slot。

### 8. promote 备库后，旧主为什么不能直接重新作为备库启动？

**参考答案：**

备库 promote 后会进入新 timeline。旧主可能还停留在旧 timeline，并且可能包含未复制到新主的写入。如果直接接回，历史分支不一致，可能造成数据冲突或损坏。

正确流程是先隔离旧主，确认不会再写入，然后使用 `pg_rewind` 把旧主回退到新主历史线上，或重新做基础备份。`pg_rewind` 需要满足条件，例如有足够 WAL，且启用 data checksums 或 `wal_log_hints`。

> **重点：** 高可用切换的核心风险是脑裂和时间线分叉。

### 9. 如何判断复制延迟发生在哪个阶段？

**参考答案：**

主库看：

```sql
SELECT
  sent_lsn,
  write_lsn,
  flush_lsn,
  replay_lsn,
  write_lag,
  flush_lag,
  replay_lag
FROM pg_stat_replication;
```

判断：

- `sent_lsn` 跟不上当前 WAL：主库发送或网络问题。
- `write_lsn` 落后 `sent_lsn`：备库接收或写入慢。
- `flush_lsn` 落后 `write_lsn`：备库刷盘慢。
- `replay_lsn` 落后 `flush_lsn`：备库重放慢，可能是长查询、I/O 或重放冲突。

> **易错：** 只看时间延迟，不看 LSN 差距。

### 10. 一个大表频繁 UPDATE，你会从哪些方面优化？

**参考答案：**

需要系统评估：

- 是否更新了被索引列，索引维护成本是否过高。
- HOT update 比例如何，fillfactor 是否合适。
- autovacuum 是否触发及时。
- 是否存在长事务阻止清理。
- 是否需要拆分冷热字段，把频繁更新字段放到独立表。
- 是否需要分区或归档历史数据。
- 是否有低价值索引拖慢更新。
- WAL、复制延迟和备份窗口是否受影响。

观察：

```sql
SELECT relname, n_tup_upd, n_tup_hot_upd, n_dead_tup
FROM pg_stat_user_tables
WHERE relname = 'target_table';
```

> **重点：** 频繁 UPDATE 的优化不是只加索引，很多时候是减少索引维护和改善 VACUUM。

### 11. `work_mem` 应该如何设置？为什么不能按单条 SQL 估算？

**参考答案：**

`work_mem` 是每个操作节点可能使用的内存，不是每个连接固定只用一次。一个查询可能有多个排序、hash join、hash aggregate 节点，并发连接也会同时使用。因此要按并发乘法估算风险：

```text
潜在内存 = 活跃连接数 * 每查询内存节点数 * work_mem
```

正确做法是设置保守全局值，对报表或批处理用 `SET LOCAL work_mem` 局部提高，并监控临时文件。

> **易错：** 看到排序落盘就全局把 `work_mem` 调得很大。

### 12. 面试场景：一个 DDL 卡住了大量业务写入，你怎么处理？

**参考答案：**

处理步骤：

1. 用 `pg_blocking_pids` 找阻塞链，确认 DDL 和被阻塞业务。
2. 查看 DDL 已运行多久、是否可取消、是否在事务中。
3. 如果影响大且可回滚，优先 `pg_cancel_backend` 取消 DDL。
4. 若会话 idle in transaction 持锁，评估后 `pg_terminate_backend`。
5. 恢复业务后复盘 DDL 方案：低峰执行、加 `lock_timeout`、使用 concurrent 操作、拆分迁移步骤。

示例：

```sql
SELECT pid, pg_blocking_pids(pid), wait_event_type, wait_event, query
FROM pg_stat_activity
WHERE cardinality(pg_blocking_pids(pid)) > 0;
```

> **重点：** 先找阻塞源，不要批量终止业务会话。
