## 一、安全和生产实践

### 1. PostgreSQL 权限模型如何分层？

**参考答案：**

权限大致分为：

- `pg_hba.conf`：认证入口。
- database：是否允许连接。
- schema：是否允许使用命名空间。
- object：表、序列、函数等对象权限。
- row level security：行级访问控制。

访问 schema 内表通常需要 schema `USAGE` 和表级权限。

> **重点：** GRANT 表权限不代表一定能查询，还要检查 schema 权限。

### 2. 为什么应用账号不能使用超级用户？

**参考答案：**

超级用户可以绕过大多数权限限制，执行高危操作。应用一旦出现 SQL 注入、配置泄露或程序 bug，超级用户权限会把局部问题扩大为实例级事故。

推荐拆分：

- owner 账号或 role：拥有对象。
- migration 账号：执行 DDL。
- app 账号：只做业务读写。
- readonly 账号：只读查询。
- dba 账号：运维管理。

> **易错：** 为了省授权步骤，把 `postgres` 给应用使用。

### 3. 默认权限 `ALTER DEFAULT PRIVILEGES` 有什么坑？

**参考答案：**

默认权限只影响未来创建的对象，不影响已有对象。它还和对象创建者有关，必须在正确 role 的上下文中设置。如果迁移工具使用的账号和你设置默认权限的账号不同，未来对象可能仍没有预期授权。

```sql
ALTER DEFAULT PRIVILEGES IN SCHEMA hr
GRANT SELECT ON TABLES TO readonly_user;
```

> **重点：** 默认权限要结合对象 owner 和迁移账号检查。

### 4. `search_path` 有什么安全风险？

**参考答案：**

`search_path` 决定未限定对象名的查找顺序。如果路径中包含不可信 schema，攻击者可能创建同名函数或对象，让 SQL 调用到非预期对象。函数中尤其要谨慎。

防护方式：

- SQL 中显式写 schema。
- 函数设置安全的 `search_path`。
- 不给普通用户在公共 schema 创建对象的权限。

> **易错：** 在 SECURITY DEFINER 函数中忽略 `search_path`。

### 5. 如何处理连接数过高？

**参考答案：**

先查连接状态：

```sql
SELECT state, count(*)
FROM pg_stat_activity
GROUP BY state;
```

处理方向：

- 检查应用连接泄漏。
- 查找 `idle in transaction`。
- 使用连接池，例如 PgBouncer。
- 缩短事务和查询耗时。
- 不盲目提高 `max_connections`，因为每个连接都有内存和进程成本。

> **重点：** 连接数问题常常是应用行为和慢 SQL 问题，不只是参数太小。

### 6. `VACUUM`、`VACUUM FULL` 和 `ANALYZE` 有什么区别？

**参考答案：**

`VACUUM` 清理 dead tuple，让空间可复用，但通常不把文件缩小。`VACUUM FULL` 会重写表，释放磁盘空间，但需要更强锁，生产慎用。`ANALYZE` 收集统计信息，帮助优化器估算行数和选择计划。

> **易错：** 用 `VACUUM FULL` 当日常维护命令。

### 7. 如何设计 PostgreSQL 13 升级方案？

**参考答案：**

升级方案要覆盖：

- 确认当前 13 小版本和扩展版本。
- 评估目标版本和兼容性。
- 备份和恢复演练。
- 回归测试核心 SQL、函数、触发器和应用。
- 验证复制、权限、连接池和监控。
- 选择升级方式：`pg_upgrade`、逻辑复制迁移或 dump/restore。
- 准备回滚方案和停机窗口。

PostgreSQL 13 已 EOL，升级不是优化项，而是生产风险治理。

> **重点：** 升级前必须验证恢复和回滚。

### 8. 面试场景：线上突然大量慢查询，你怎么处理？

**参考答案：**

处理顺序：

1. 保留现场：连接数、等待事件、慢查询日志、CPU、I/O、复制状态。
2. 判断是否锁等待：查 `pg_stat_activity` 和 `pg_locks`。
3. 判断是否单个 SQL 放大：查 `pg_stat_statements`。
4. 对代表 SQL 做 `EXPLAIN (ANALYZE, BUFFERS)`。
5. 检查统计信息、索引、数据量变化、参数和最近发布。
6. 低风险处理：取消异常 SQL、恢复配置、补 ANALYZE。
7. 复测并补长期修复：索引、SQL 改写、连接池、容量或发布回滚。

> **重点：** 先定位根因和影响面，避免直接重启清掉现场。
