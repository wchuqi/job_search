# PostgreSQL 13 学习资料：故障案例和排障剧本

[返回索引](../PostgreSQL13学习资料.md)

## 学习目标

- 掌握常见 PostgreSQL 生产故障的症状、证据、临时止血和长期修复。
- 能按固定剧本处理慢查询、锁等待、连接耗尽、WAL 暴涨、复制延迟、膨胀和备份失败。
- 学会避免扩大事故的高危操作。

## 排障总原则

1. 先确认影响面：全部业务、单库、单表、单 SQL、单租户还是单应用。
2. 先保留现场：当前活动、等待事件、锁、复制、磁盘、日志、Top SQL。
3. 先止血再根治：取消异常 SQL、限流、切换读流量、扩容磁盘、暂停任务。
4. 所有高危操作前确认回滚路径。
5. 事后必须复盘：触发条件、监控缺口、自动化补救和长期修复。

## 通用现场快照

```sql
SELECT now() AS snapshot_time;

SELECT state, wait_event_type, wait_event, count(*)
FROM pg_stat_activity
GROUP BY state, wait_event_type, wait_event
ORDER BY count(*) DESC;

SELECT pid, usename, application_name, client_addr, state,
       wait_event_type, wait_event,
       now() - xact_start AS xact_age,
       now() - query_start AS query_age,
       query
FROM pg_stat_activity
ORDER BY query_start NULLS LAST
LIMIT 50;

SELECT
  checkpoints_timed,
  checkpoints_req,
  checkpoint_write_time,
  checkpoint_sync_time
FROM pg_stat_bgwriter;
```

## 案例一：突然大量慢查询

### 现象

- 应用接口超时。
- 数据库 CPU 或 I/O 飙升。
- 慢查询日志大量出现。
- 连接数上升，队列堆积。

### 证据查询

```sql
SELECT
  query,
  calls,
  total_exec_time,
  mean_exec_time,
  rows,
  shared_blks_hit,
  shared_blks_read,
  temp_blks_read,
  temp_blks_written
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;
```

对代表 SQL：

```sql
EXPLAIN (ANALYZE, BUFFERS, VERBOSE)
SELECT ...;
```

### 判断路径

- `wait_event_type = Lock`：先按锁等待处理。
- shared read 很高：可能 I/O 或缓存未命中。
- temp 写很多：排序、hash 或聚合落盘。
- actual rows 和估算 rows 差很多：统计信息或数据倾斜。
- loops 很大：Nested Loop 放大。

### 临时止血

- 取消明显异常的报表或批处理 SQL。
- 应用侧限流或降级。
- 临时提高特定报表会话 `work_mem`，不要盲目全局调大。
- 对明显统计过旧的表执行 `ANALYZE`。

### 长期修复

- 优化 SQL 和索引。
- 增加扩展统计。
- 分离报表负载。
- 建立发布前执行计划回归。
- 监控 Top SQL 变化。

## 案例二：锁等待导致全站卡住

### 现象

- 大量会话等待 Lock。
- 某个 DDL 或长事务阻塞后续 DML。
- 应用请求堆积。

### 证据查询

```sql
SELECT
  pid,
  pg_blocking_pids(pid) AS blocking_pids,
  wait_event_type,
  wait_event,
  now() - query_start AS query_age,
  query
FROM pg_stat_activity
WHERE cardinality(pg_blocking_pids(pid)) > 0
ORDER BY query_age DESC;
```

阻塞源：

```sql
WITH blocked AS (
  SELECT pid, unnest(pg_blocking_pids(pid)) AS blocking_pid
  FROM pg_stat_activity
)
SELECT
  b.pid AS blocked_pid,
  blocked.query AS blocked_query,
  b.blocking_pid,
  blocking.state AS blocking_state,
  now() - blocking.xact_start AS blocking_xact_age,
  blocking.query AS blocking_query
FROM blocked b
JOIN pg_stat_activity blocked ON blocked.pid = b.pid
JOIN pg_stat_activity blocking ON blocking.pid = b.blocking_pid;
```

### 临时止血

- 如果阻塞源是误执行 DDL 或报表，优先 `pg_cancel_backend`。
- 如果阻塞源是 `idle in transaction` 且确认可回滚，使用 `pg_terminate_backend`。
- 暂停继续发起同类 DDL 或批处理。

### 长期修复

- DDL 使用低峰窗口和 `lock_timeout`。
- 应用设置 `idle_in_transaction_session_timeout`。
- 批处理分批提交。
- 多资源更新固定顺序，降低死锁。

### 高危操作

不要直接重启数据库作为第一反应。重启会清掉现场，并可能导致恢复时间和业务影响扩大。

## 案例三：连接耗尽

### 现象

- 应用报 `too many connections`。
- 数据库连接数达到 `max_connections`。
- 活跃 SQL 不一定很多。

### 证据查询

```sql
SELECT
  application_name,
  usename,
  client_addr,
  state,
  count(*)
FROM pg_stat_activity
GROUP BY application_name, usename, client_addr, state
ORDER BY count(*) DESC;
```

找事务空闲：

```sql
SELECT pid, application_name, now() - xact_start AS age, query
FROM pg_stat_activity
WHERE state = 'idle in transaction'
ORDER BY age DESC;
```

### 临时止血

- 终止明确异常的 `idle in transaction`。
- 应用侧限流。
- 重启有连接泄漏的应用实例，而不是优先重启数据库。
- 临时扩大连接池前确认数据库资源可承受。

### 长期修复

- 使用 PgBouncer。
- 修复应用连接泄漏。
- 设置连接最大生命周期。
- 设置事务空闲超时。
- 为不同应用设置 `application_name`。

## 案例四：WAL 目录暴涨或磁盘将满

### 现象

- `pg_wal` 或归档目录增长很快。
- 磁盘报警。
- 备份或复制异常。

### 证据查询

```sql
SELECT
  slot_name,
  slot_type,
  active,
  restart_lsn,
  confirmed_flush_lsn
FROM pg_replication_slots;

SELECT application_name, state, sent_lsn, replay_lsn, replay_lag
FROM pg_stat_replication;
```

检查归档日志和 `archive_command`。如果归档失败，PostgreSQL 会保留未成功归档的 WAL。

### 临时止血

- 如果是归档失败，修复归档目标或权限。
- 如果是废弃复制槽，确认无用后删除：

```sql
SELECT pg_drop_replication_slot('slot_name');
```

- 如果磁盘即将满，优先扩容或迁移归档，不要删除 `pg_wal` 中未知文件。

### 长期修复

- 监控复制槽滞后和 WAL 目录大小。
- 设置 `max_slot_wal_keep_size`。
- 归档命令做幂等和告警。
- 为批量任务评估 WAL 峰值。

## 案例五：备库复制延迟

### 现象

- 读备库数据旧。
- 主库 `pg_stat_replication` 显示 replay lag。
- 备库长查询被取消或重放慢。

### 证据查询

主库：

```sql
SELECT
  application_name,
  sent_lsn,
  write_lsn,
  flush_lsn,
  replay_lsn,
  write_lag,
  flush_lag,
  replay_lag
FROM pg_stat_replication;
```

备库：

```sql
SELECT
  pg_last_wal_receive_lsn(),
  pg_last_wal_replay_lsn(),
  now() - pg_last_xact_replay_timestamp() AS replay_delay;
```

### 判断路径

- 接收慢：网络或 WAL sender。
- flush 慢：备库磁盘。
- replay 慢：备库 I/O、长查询冲突、WAL 重放压力。
- 主库写入峰值：备库硬件追不上。

### 临时止血

- 暂停报表查询。
- 把读流量切回主库或其他副本，前提是主库承受得住。
- 处理备库磁盘或网络问题。

### 长期修复

- 报表库和高可用库分离。
- 控制备库长查询。
- 监控 LSN 延迟而不仅是时间延迟。
- 评估同步复制或读一致性要求。

## 案例六：表和索引严重膨胀

### 现象

- 表数据量没明显增长，但磁盘持续增长。
- 查询变慢。
- VACUUM 后文件大小不降。
- autovacuum 频繁运行但追不上。

### 证据查询

```sql
SELECT
  relname,
  n_live_tup,
  n_dead_tup,
  last_autovacuum,
  last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 20;
```

```sql
SELECT
  relname,
  pg_size_pretty(pg_relation_size(relid)) AS table_size,
  pg_size_pretty(pg_indexes_size(relid)) AS index_size,
  pg_size_pretty(pg_total_relation_size(relid)) AS total_size
FROM pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC
LIMIT 20;
```

### 临时止血

- 处理长事务。
- 提高关键表 autovacuum 积极性。
- 对严重索引考虑 `REINDEX CONCURRENTLY`。
- 如果必须释放磁盘，评估窗口内重写表。

### 长期修复

- 改写大批量 UPDATE/DELETE。
- 按时间分区。
- 调整 fillfactor。
- 删除低价值索引。
- 为大表设置 autovacuum 参数。

## 案例七：备份失败或恢复失败

### 现象

- `pg_dump` 中断。
- `pg_basebackup` 中断。
- PITR 找不到 WAL。
- 恢复后权限或扩展缺失。

### 证据

- 备份命令日志。
- PostgreSQL server log。
- 归档目录连续性。
- `pg_dump` 是否包含全局对象。

### 临时止血

- 如果备份失败，不要删除上一份可用备份。
- 立即执行一次替代备份策略。
- 验证归档是否连续。

### 长期修复

- 每次备份后自动恢复验证。
- 单库 dump 加全局对象备份。
- 物理备份加 WAL 归档完整性检查。
- 恢复演练写成脚本。

## 高危操作清单

- 手工删除 `pg_wal` 文件。
- 高峰期 `VACUUM FULL` 大表。
- 未确认阻塞源就批量 terminate 后端。
- 未隔离旧主就 promote 备库。
- 未备份就做大范围 UPDATE/DELETE。
- 在应用中使用超级用户账号。
- PostgreSQL 13 EOL 环境无升级计划。

## 练习

1. 用测试库制造锁等待，写一份排障记录。
2. 用大表制造排序落盘，记录计划和日志。
3. 停止备库或逻辑订阅，观察 WAL 增长。
4. 制造长事务阻止 VACUUM，记录 dead tuple 变化。
5. 模拟误删数据，写 PITR 和数据回灌方案。

## 验收

- 每个故障都能说出现象、证据、止血、根治和风险。
- 能写出现场快照 SQL。
- 能解释哪些操作不能在生产直接执行。
- 能区分恢复可用性、数据一致性和性能优化。

## 重点

- 排障不是背命令，而是按证据缩小范围。
- 先止血再根治，但止血不能破坏恢复能力。
- WAL、锁、连接、VACUUM、复制是生产事故高频主线。

## 难点

- 多个故障会叠加。例如慢查询导致连接耗尽，连接耗尽导致事务堆积，长事务又导致膨胀和 VACUUM 失败。

## 易错

> **易错：** 把所有慢都归因于“数据库性能不行”。
>
> 正确做法：区分锁等待、I/O、CPU、计划错误、连接池、应用取数和复制延迟。

> **易错：** 事故中只做临时处理，不记录证据。
>
> 正确做法：先保存关键系统视图、日志和时间线，便于根因分析。
