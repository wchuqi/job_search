# PostgreSQL 13 学习资料：优化器代价模型和执行计划深度解析

[返回索引](../PostgreSQL13学习资料.md)

## 学习目标

- 理解 SQL 从解析、重写、规划到执行的完整路径。
- 理解 PostgreSQL 优化器如何基于统计信息、选择率、代价参数和 join 搜索选择计划。
- 能深读 `EXPLAIN (ANALYZE, BUFFERS, VERBOSE)`，定位估算错误、I/O 压力、排序落盘和 join 放大。
- 能设计可验证的优化实验，避免凭经验乱建索引或乱调参数。

## 理论导读

PostgreSQL 的优化器不是规则清单，而是代价优化器。它会枚举可行路径，估算每条路径的成本，然后选择估算成本最低的计划。计划是否好，取决于两个核心条件：

- 统计信息是否能准确描述数据分布。
- 成本参数是否大致反映当前硬件和缓存环境。

很多慢查询的本质不是“数据库不会用索引”，而是“优化器基于错误信息做了看似合理但实际很差的选择”。因此优化重点常常是修正估算，而不是强制执行路径。

## 核心心智模型

### 1. 优化器在做选择题

对于一条 SQL，优化器需要选择：

- 从哪个表开始。
- 每个表用顺序扫描、索引扫描、位图扫描还是其他访问方式。
- 多表如何 join，join 顺序是什么。
- 用 Nested Loop、Hash Join 还是 Merge Join。
- 聚合用 HashAggregate 还是 GroupAggregate。
- 排序是否可利用已有顺序，是否需要增量排序。
- 并行计划是否值得。

### 2. 成本不是时间，但要能比较

`cost=10..100` 不是 100 毫秒，而是优化器内部成本单位。默认情况下，顺序读取一个 page 的成本 `seq_page_cost` 是 1，随机读取一个 page 的成本 `random_page_cost` 默认更高。CPU 处理 tuple、操作符和索引 tuple 也有成本。

成本的价值在于比较计划，不在于直接换算时间。

## 知识点详解

## 一、SQL 生命周期

简化路径：

```text
SQL text
  -> parser：语法解析，生成 parse tree
  -> analyzer：名称解析、类型推导、权限检查
  -> rewriter：规则重写，例如视图展开
  -> planner/optimizer：生成候选路径，估算成本，选择 plan
  -> executor：执行计划节点，返回结果
```

这解释了几个现象：

- 同样 SQL 文本在不同 `search_path` 下可能解析到不同对象。
- 视图不是总是先完整执行再过滤，视图定义通常会被纳入整体优化。
- 函数波动性会影响优化器能否提前计算或重排调用。

## 二、统计信息如何影响选择率

`ANALYZE` 会采样表数据，生成列统计信息：

```sql
SELECT
  attname,
  null_frac,
  n_distinct,
  most_common_vals,
  most_common_freqs,
  histogram_bounds,
  correlation
FROM pg_stats
WHERE schemaname = 'public'
  AND tablename = 'application';
```

字段含义：

| 字段 | 含义 | 用途 |
| --- | --- | --- |
| `null_frac` | NULL 比例 | 估算 `IS NULL` |
| `n_distinct` | 不同值数量估计 | 估算等值过滤 |
| `most_common_vals` | 高频值 | 处理倾斜分布 |
| `most_common_freqs` | 高频值频率 | 避免平均分布误判 |
| `histogram_bounds` | 直方图边界 | 估算范围条件 |
| `correlation` | 列值和物理顺序相关性 | 影响索引扫描成本 |

### 数据倾斜问题

如果 `status='archived'` 占 95%，`status='active'` 占 1%，那么：

```sql
WHERE status = 'active'
```

和

```sql
WHERE status = 'archived'
```

应该走不同计划。高频值统计能帮助优化器识别这种差异。如果统计目标太低，稀有值或倾斜分布可能被采样忽略。

提高统计目标：

```sql
ALTER TABLE application ALTER COLUMN status SET STATISTICS 1000;
ANALYZE application;
```

不要无脑全库提高，统计目标越高，ANALYZE 成本和规划信息体积也会上升。

## 三、多列相关性和扩展统计

单列统计无法理解列之间的相关关系。例如：

```text
country = 'CN' 时 city 大多是 Shanghai、Beijing
country = 'US' 时 city 大多是 New York、Seattle
```

如果查询：

```sql
WHERE country = 'CN' AND city = 'Shanghai'
```

优化器可能按独立分布相乘估算，导致行数偏差。

扩展统计：

```sql
CREATE STATISTICS st_candidate_country_city (dependencies, mcv)
ON country, city
FROM candidate;

ANALYZE candidate;
```

类型：

- `dependencies`：函数依赖，用于列之间强相关。
- `mcv`：多列高频组合。
- `ndistinct`：多列 distinct 数量。

## 四、访问路径选择

### Seq Scan

顺序扫描适合：

- 表很小。
- 返回表的大部分行。
- 索引选择率很差。
- 随机 I/O 成本高。

### Index Scan

索引扫描适合：

- 返回少量行。
- 需要按索引顺序返回。
- 过滤条件能有效利用索引。

成本包含索引访问和 heap 回表。如果回表很多且随机，成本可能高于顺序扫描。

### Bitmap Scan

Bitmap Index Scan 先收集匹配 tuple 的位置，再按 page 批量访问 heap。它适合返回行数中等的场景，尤其是多个索引条件组合：

```text
BitmapAnd
  -> Bitmap Index Scan on idx_status
  -> Bitmap Index Scan on idx_created_at
```

Bitmap 可能 lossy，导致需要 recheck 条件。

### Index Only Scan

Index Only Scan 需要：

- 查询字段都在索引中。
- heap page 的 visibility map 显示 all-visible，或需要较少回表。

所以它不是只由索引定义决定。

## 五、Join 顺序和 Join 算法

### Nested Loop

适合外表小、内表查找便宜：

```text
Nested Loop
  -> Index Scan small outer
  -> Index Scan inner by key
```

风险是外表实际行数被低估，内表扫描被重复执行很多次。

### Hash Join

适合等值连接。通常把较小输入构建 hash 表，再扫描较大输入匹配。风险是 hash 表超出 `work_mem` 后分批，出现临时文件 I/O。

计划中关注：

```text
Hash
  Buckets: ...
  Batches: ...
  Memory Usage: ...
```

`Batches > 1` 通常表示 hash 分批，可能有 I/O。

### Merge Join

要求两边按 join key 有序。可以利用索引顺序，也可以先排序。适合大输入且排序成本可接受的场景。

## 六、排序、聚合和内存

排序节点可能显示：

```text
Sort Method: quicksort  Memory: 2048kB
Sort Method: external merge  Disk: 102400kB
```

`external merge` 表示排序落盘。处理方式不一定是全局加大 `work_mem`，而是：

- 看是否有索引能提供顺序。
- 减少排序输入行数。
- 使用更合适的分页方式。
- 对特定会话或任务设置 `work_mem`。

聚合方式：

- HashAggregate：使用 hash 表聚合，可能受 `work_mem` 限制。
- GroupAggregate：输入需要按 group key 有序，可以利用排序或索引。

## 七、并行查询

PostgreSQL 13 支持并行扫描、并行 join、并行聚合等。计划中会看到：

```text
Gather
  Workers Planned: 2
  Workers Launched: 2
```

并行不是越多越好：

- 启动和协调有成本。
- 并发高时并行 worker 会成为争用资源。
- 小查询并行反而慢。
- 某些函数、临时表或计划节点限制并行。

相关参数：

- `max_worker_processes`
- `max_parallel_workers`
- `max_parallel_workers_per_gather`
- `parallel_setup_cost`
- `parallel_tuple_cost`

## 八、Generic plan 和 custom plan

预备语句或绑定参数查询可能涉及 generic plan 和 custom plan 的选择。custom plan 会根据具体参数估算，generic plan 对不同参数复用。

风险场景：

```sql
WHERE status = $1
```

如果 `status='active'` 返回很少，`status='archived'` 返回很多，最优计划可能完全不同。generic plan 可能对某些参数很差。

排查方向：

- 对比直接写常量和绑定参数的计划。
- 检查数据倾斜。
- 在必要场景评估 `plan_cache_mode`，PostgreSQL 13 支持该参数。

```sql
SET plan_cache_mode = force_custom_plan;
```

这个设置应谨慎用于诊断或局部场景，不应盲目全局打开。

## 九、分区表规划

PostgreSQL 13 分区表能进行 partition pruning。关键是查询条件必须能让优化器判断哪些分区不需要访问。

```sql
CREATE TABLE application_log (
  id bigserial,
  created_at date NOT NULL,
  payload jsonb
) PARTITION BY RANGE (created_at);
```

如果查询写成：

```sql
WHERE created_at >= date '2026-06-01'
  AND created_at < date '2026-07-01'
```

更容易裁剪分区。若对分区键包函数：

```sql
WHERE date_trunc('month', created_at) = date '2026-06-01'
```

可能削弱裁剪和索引使用。

## 十、优化流程

标准流程：

1. 明确 SQL、参数、数据量、执行频率和业务目标。
2. 获取 `EXPLAIN (ANALYZE, BUFFERS, VERBOSE)`。
3. 找最耗时节点。
4. 比较估算行数和实际行数。
5. 判断是访问路径、join 顺序、排序、聚合、I/O、锁等待还是网络传输。
6. 只做一个改动：SQL 改写、索引、统计信息、参数或模型调整。
7. 复测并记录前后计划。

## 例子：估算错误导致 Nested Loop 放大

假设 `application.status='active'` 实际有 500000 行，但统计信息估算只有 100 行。优化器可能选择：

```text
Nested Loop
  -> Index Scan application status='active'  rows=100 actual=500000
  -> Index Scan candidate by id              loops=500000
```

处理方向：

```sql
ANALYZE application;
ALTER TABLE application ALTER COLUMN status SET STATISTICS 1000;
ANALYZE application;
```

如果是多列相关：

```sql
CREATE STATISTICS st_application_status_job (mcv, dependencies)
ON status, job_id
FROM application;
ANALYZE application;
```

如果仍然不合适，再评估索引和 SQL 改写。

## 例子：深分页为什么慢

```sql
SELECT id, created_at
FROM application
ORDER BY created_at DESC, id DESC
OFFSET 100000 LIMIT 20;
```

即使有索引，数据库也要跳过前 100000 行。更稳定的 keyset pagination：

```sql
SELECT id, created_at
FROM application
WHERE (created_at, id) < (timestamp '2026-06-06 10:00:00+08', 123456)
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

配套索引：

```sql
CREATE INDEX idx_application_created_id_desc
ON application(created_at DESC, id DESC);
```

## 实操任务

### 任务 1：制造数据倾斜并观察计划

```sql
DROP TABLE IF EXISTS skew_lab;
CREATE TABLE skew_lab(id bigserial PRIMARY KEY, status text NOT NULL, payload text);

INSERT INTO skew_lab(status, payload)
SELECT
  CASE WHEN g <= 990000 THEN 'archived' ELSE 'active' END,
  md5(g::text)
FROM generate_series(1, 1000000) AS g;

CREATE INDEX idx_skew_lab_status ON skew_lab(status);
ANALYZE skew_lab;

EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM skew_lab WHERE status = 'active';

EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM skew_lab WHERE status = 'archived';
```

验收：能解释两个值为什么可能选择不同扫描方式。

### 任务 2：观察排序落盘

在测试环境降低 `work_mem`：

```sql
SET work_mem = '1MB';

EXPLAIN (ANALYZE, BUFFERS)
SELECT *
FROM skew_lab
ORDER BY payload;
```

再提高会话级 `work_mem` 对比。验收：能解释 `Sort Method` 变化。

### 任务 3：验证扩展统计

创建两列相关数据，分别在无扩展统计和有扩展统计时观察估算行数。

验收：能解释单列统计为什么会误估多列相关条件。

## 验收

- 能描述 SQL 从解析到执行的阶段。
- 能解释成本模型不等于实际时间。
- 能读懂 `rows`、`actual rows`、`loops`、`Buffers`、`Sort Method`。
- 能识别统计信息失真、数据倾斜、多列相关和 generic plan 问题。
- 能说明 Nested Loop、Hash Join、Merge Join 的适用场景和失效模式。
- 能给出一个优化方案，并用前后执行计划证明。

## 重点

- 优化器基于估算做选择，估算质量决定计划质量。
- 执行计划分析先看行数偏差，再看耗时节点和 buffer。
- `work_mem` 是每个操作节点可能使用，不是全局总预算。
- 分区裁剪和索引使用都依赖查询条件写法。

## 难点

- 真实生产慢查询通常同时有数据倾斜、参数化 SQL、统计信息、索引设计和并发 I/O 的影响。
- Generic plan 在参数高度倾斜时可能让某些参数表现很差，排查时要对比具体参数计划。

## 易错

> **易错：** 用 `SET enable_seqscan = off` 证明数据库应该走索引。
>
> 正确做法：这只能作为诊断手段，不能作为生产优化方案。应修正统计信息、索引或 SQL。

> **易错：** 只看第一行总 cost，不看实际行数和 loops。
>
> 正确做法：慢点常藏在子节点和被循环放大的内层节点。

> **易错：** 所有慢 SQL 都通过加索引解决。
>
> 正确做法：有些慢来自排序、聚合、join 顺序、统计信息、锁等待、网络传输或应用取数过多。
