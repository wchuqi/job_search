# PostgreSQL 13 学习资料：安全权限和连接管理

[返回索引](../PostgreSQL13学习资料.md)

## 学习目标

- 掌握 role、权限、默认权限、schema 权限和行级安全。
- 理解认证、密码、SSL、`search_path` 和最小权限原则。
- 能规划连接池和应用账号。

## 理论导读

PostgreSQL 权限不是一个开关，而是多层判断：连接认证先通过 `pg_hba.conf`，登录后 role 必须有 database 连接权限、schema 使用权限、对象操作权限。很多“已经 GRANT 了还是不能查”的问题，根源是漏了 schema 的 `USAGE`。

安全设计应把超级用户从日常应用中拿掉。应用账号只应拥有完成业务所需的最小权限，DDL 权限、owner 权限和运维权限应分离。

## 核心心智模型

访问一张表像进一栋楼拿一份文件：

1. `pg_hba.conf` 决定你能不能进大门。
2. database 权限决定你能不能进楼。
3. schema `USAGE` 决定你能不能进楼层。
4. 表权限决定你能不能读取或修改文件。

## 知识点详解

### 1. Role

```sql
CREATE ROLE app_owner NOLOGIN;
CREATE ROLE app_user LOGIN PASSWORD 'change_me';
CREATE ROLE readonly_user LOGIN PASSWORD 'change_me';

GRANT app_owner TO app_user;
```

role 可以是用户，也可以是权限组。推荐用 NOLOGIN role 表示权限集合，再授予登录账号。

### 2. 基础授权

```sql
GRANT CONNECT ON DATABASE job_app TO app_user;
GRANT USAGE ON SCHEMA hr TO app_user;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA hr TO app_user;
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA hr TO app_user;
```

默认权限用于未来新建对象：

```sql
ALTER DEFAULT PRIVILEGES IN SCHEMA hr
GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_user;

ALTER DEFAULT PRIVILEGES IN SCHEMA hr
GRANT USAGE, SELECT ON SEQUENCES TO app_user;
```

### 3. 行级安全 RLS

RLS 可以按行限制访问：

```sql
ALTER TABLE application ENABLE ROW LEVEL SECURITY;

CREATE POLICY application_tenant_policy
ON application
USING (tenant_id = current_setting('app.tenant_id')::bigint);
```

应用连接后设置租户：

```sql
SET app.tenant_id = '1001';
```

RLS 很强，但要小心 owner、超级用户和 bypass RLS 权限。

### 4. 认证和密码

`pg_hba.conf` 决定认证方式。PostgreSQL 13 支持 `md5` 和 `scram-sha-256`。生产环境应优先使用更安全的密码存储和认证方式，并避免共享账号。

```conf
password_encryption = scram-sha-256
```

### 5. SSL

如果数据库连接跨不可信网络，应启用 SSL。客户端也要配置 `sslmode`，例如 `require`、`verify-ca`、`verify-full`。严格生产环境应校验证书，而不是只加密不验证身份。

### 6. 连接池

PostgreSQL 每个连接有进程和内存成本，高并发应用通常需要 PgBouncer 等连接池。

PgBouncer 常见模式：

| 模式 | 特点 |
| --- | --- |
| session | 一个客户端会话绑定一个服务端连接 |
| transaction | 每个事务借用连接，复用率高 |
| statement | 每条语句借用连接，限制较多 |

事务池模式下要避免依赖会话状态，例如临时表、长时间 `SET`、prepared statement 行为等。

## 例子

只读账号：

```sql
CREATE ROLE readonly_user LOGIN PASSWORD 'change_me';
GRANT CONNECT ON DATABASE job_app TO readonly_user;
GRANT USAGE ON SCHEMA hr TO readonly_user;
GRANT SELECT ON ALL TABLES IN SCHEMA hr TO readonly_user;

ALTER DEFAULT PRIVILEGES IN SCHEMA hr
GRANT SELECT ON TABLES TO readonly_user;
```

检查权限：

```sql
SELECT
  grantee,
  table_schema,
  table_name,
  privilege_type
FROM information_schema.role_table_grants
WHERE table_schema = 'hr'
ORDER BY table_name, grantee, privilege_type;
```

## 练习

1. 创建 owner、app_user、readonly_user 三类 role。
2. 让 readonly_user 只能读 `hr` schema 下的表。
3. 验证没有 schema `USAGE` 时，即使有表权限也无法访问。
4. 配置一个 RLS 策略，按 `tenant_id` 限制数据。

## 验收

- 能解释 role 和 user 的关系。
- 能写出 database、schema、table、sequence 的完整授权。
- 能说明默认权限只影响未来对象。
- 能解释连接池事务模式的限制。

## 重点

- 应用账号不应使用超级用户。
- schema `USAGE` 是对象访问路径上的必要权限。
- 默认权限要在对象创建者上下文中配置。

## 难点

- 权限问题常由对象 owner、默认权限、schema 权限和 search path 叠加造成，需要逐层排查。

## 易错

> **易错：** 只 GRANT 表权限，忘记 schema `USAGE`。
>
> 正确做法：访问 schema 内对象至少需要 schema `USAGE` 和对象级权限。

> **易错：** 应用使用 `postgres` 超级用户连接。
>
> 正确做法：拆分 owner、迁移账号、应用账号和只读账号，按最小权限授权。
