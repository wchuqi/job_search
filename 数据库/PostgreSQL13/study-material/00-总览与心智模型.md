# PostgreSQL 13 学习资料：总览与心智模型

[返回索引](../PostgreSQL13学习资料.md)

## 学习目标

- 建立 PostgreSQL 的整体图景，理解它不只是一个 SQL 执行工具，而是一个带有存储、事务、日志、优化器、权限和后台维护机制的数据库系统。
- 区分实例、数据库、schema、表、索引、角色、表空间等层级。
- 知道 PostgreSQL 13 的版本边界和生产风险。

## 理论导读

PostgreSQL 是关系型数据库，但它的关键能力来自多个子系统协作：客户端连接进来后，由服务进程解析 SQL、重写查询、生成执行计划、访问共享缓冲区和磁盘页面；写入产生 WAL，事务提交后由 WAL 保障崩溃恢复；后台进程负责检查点、自动清理、统计信息、复制发送等工作。

可以把 PostgreSQL 想成一个有严格账本制度的仓库。表中的数据是货架，索引是目录，WAL 是操作流水账，事务是一次受控的出入库申请，MVCC 是保留多个历史版本让读者看到一致快照，VACUUM 是定期清理作废版本，优化器则像调度员，根据统计信息选择最便宜的取货路线。

PostgreSQL 13 的重要背景是生命周期。它曾是稳定版本，但现在已经进入 EOL。学习 PostgreSQL 13 仍有价值，因为许多生产环境还在使用它，但新系统应优先使用受支持版本，老系统应规划升级。

## 核心心智模型

| 模型 | 解释 | 典型问题 |
| --- | --- | --- |
| 对象层级模型 | cluster 包含多个 database，database 内有 schema，schema 内有表、视图、函数等对象 | 为什么跨库不能直接 JOIN |
| 事务版本模型 | 一行数据可能有多个版本，不同事务根据快照看到不同版本 | 为什么长事务会导致表膨胀 |
| WAL 账本模型 | 先写日志再修改数据页，崩溃后根据 WAL 重放 | 为什么备份恢复离不开 WAL |
| 代价优化模型 | 优化器基于统计信息估算不同计划的成本 | 为什么有索引但不走索引 |
| 权限边界模型 | role、schema、对象权限和默认权限共同决定访问结果 | 为什么 GRANT 表权限后仍无法访问 schema |

## 知识点详解

### 1. Cluster、Database、Schema

PostgreSQL 的 cluster 是一个由 `initdb` 初始化出来的数据目录和运行环境，不等于业务数据库。一个 cluster 下可以有多个 database，每个 database 彼此隔离。schema 是 database 内部的命名空间，用来组织对象。

```sql
CREATE DATABASE job_app;
\c job_app
CREATE SCHEMA hr;
CREATE TABLE hr.candidate (
  id bigserial PRIMARY KEY,
  name text NOT NULL
);
```

### 2. 进程和后台任务

PostgreSQL 采用多进程架构。每个客户端连接通常对应一个后端进程，后台还有 checkpointer、background writer、WAL writer、autovacuum launcher、stats collector 相关进程、replication sender 等。

生产环境最常见的问题往往不是 SQL 语法，而是后台机制没有被理解：autovacuum 跟不上会膨胀，checkpoint 太频繁会造成 I/O 抖动，复制槽不消费会堆积 WAL，长事务会阻止垃圾回收。

### 3. PostgreSQL 13 需要特别知道的点

- B-tree 索引去重能减少重复键索引体积。
- 增量排序让已经部分有序的数据排序成本降低。
- `VACUUM` 可以并行处理索引，改善大表维护效率。
- 分区表和逻辑复制能力较早期版本有增强。
- PostgreSQL 13 已 EOL，生产环境应制定升级计划。

## 例子

查看当前数据库、schema 和连接信息：

```sql
SELECT current_database(), current_schema(), current_user;

SELECT datname, datallowconn
FROM pg_database
ORDER BY datname;

SELECT schemaname, tablename
FROM pg_tables
WHERE schemaname NOT IN ('pg_catalog', 'information_schema');
```

查看 PostgreSQL 版本：

```sql
SHOW server_version;
SELECT version();
```

## 练习

1. 初始化或连接一个 PostgreSQL 13 实例。
2. 创建 `job_app` 数据库。
3. 在 `job_app` 中创建 `hr` 和 `audit` 两个 schema。
4. 在 `hr` 中建候选人表，在 `audit` 中建操作日志表。
5. 用 `\dn`、`\dt hr.*`、`\d+ hr.candidate` 查看对象。

## 验收

- 能画出 cluster、database、schema、table 的层级关系。
- 能解释 WAL、MVCC、VACUUM、优化器分别解决什么问题。
- 能说明 PostgreSQL 13 在当前生产使用中的版本风险。

## 重点

- PostgreSQL 是系统，不只是 SQL 解释器。
- WAL、MVCC、VACUUM、统计信息和执行计划是理解生产问题的主线。
- 版本生命周期也是生产能力的一部分。

## 难点

- 把 SQL 层现象和内部机制对应起来。例如慢查询可能来自统计信息失真，磁盘膨胀可能来自长事务或 autovacuum 参数不合适。

## 易错

> **易错：** 认为 database 和 schema 只是两个叫法。
>
> 正确做法：database 是连接和隔离边界，schema 是 database 内对象命名空间。

> **易错：** 只学习 SQL，不学习 WAL、锁、VACUUM 和执行计划。
>
> 正确做法：只会 SQL 适合写简单查询，生产排障必须理解内部机制。
