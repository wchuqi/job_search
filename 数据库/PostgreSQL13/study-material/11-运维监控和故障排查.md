# PostgreSQL 13 学习资料：运维监控和故障排查

[返回索引](../PostgreSQL13学习资料.md)

## 学习目标

- 掌握 PostgreSQL 13 日常监控指标和系统视图。
- 能处理慢查询、锁等待、复制延迟、表膨胀、连接耗尽和磁盘增长。
- 能形成生产排障流程。

## 理论导读

数据库运维不是等报警后随机查看命令，而是持续观察数据库状态模型：连接、事务、锁、缓存、WAL、checkpoint、autovacuum、复制、磁盘和查询负载。每个指标都要能回答两个问题：它现在是否异常，它异常时会影响什么。

PostgreSQL 的系统视图提供了大量现场证据。排障时应先保留现场，再做影响面最小的动作，例如取消问题 SQL，最后才考虑终止连接或重启服务。

## 核心心智模型

故障排查按“入口、执行、存储、复制、维护”分层：

- 入口：连接数、认证、连接池。
- 执行：慢 SQL、锁等待、CPU、内存。
- 存储：表膨胀、索引膨胀、磁盘、WAL。
- 复制：发送、接收、重放延迟。
- 维护：autovacuum、ANALYZE、checkpoint。

## 知识点详解

### 1. 连接和事务

```sql
SELECT state, count(*)
FROM pg_stat_activity
GROUP BY state
ORDER BY count(*) DESC;
```

长事务：

```sql
SELECT pid, usename, state, now() - xact_start AS age, query
FROM pg_stat_activity
WHERE xact_start IS NOT NULL
ORDER BY age DESC
LIMIT 20;
```

### 2. 慢查询

生产常用组合：

- `log_min_duration_statement`
- `pg_stat_statements`
- 应用 trace id 或 application_name
- `EXPLAIN (ANALYZE, BUFFERS)`

### 3. 锁等待

```sql
SELECT
  pid,
  wait_event_type,
  wait_event,
  state,
  query
FROM pg_stat_activity
WHERE wait_event_type IS NOT NULL;
```

取消和终止：

```sql
SELECT pg_cancel_backend(12345);
SELECT pg_terminate_backend(12345);
```

优先取消查询，终止连接会回滚事务，可能造成业务影响。

### 4. Autovacuum 和膨胀

查看表统计：

```sql
SELECT
  relname,
  n_live_tup,
  n_dead_tup,
  last_vacuum,
  last_autovacuum,
  last_analyze,
  last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 20;
```

VACUUM 不是简单清空磁盘文件。普通 VACUUM 标记空间可复用，`VACUUM FULL` 会重写表并需要更强锁。生产慎用 `VACUUM FULL`。

### 5. Checkpoint 和 WAL

Checkpoint 会把脏页刷盘，过于频繁可能造成 I/O 抖动。WAL 增长可能来自写入高峰、归档失败、复制槽滞后或备份状态异常。

```sql
SELECT * FROM pg_stat_bgwriter;
```

### 6. 复制监控

主库：

```sql
SELECT application_name, state, sync_state, write_lag, flush_lag, replay_lag
FROM pg_stat_replication;
```

复制槽：

```sql
SELECT slot_name, active, restart_lsn
FROM pg_replication_slots;
```

### 7. 常见故障处理

| 现象 | 优先检查 | 常见处理 |
| --- | --- | --- |
| 连接耗尽 | `pg_stat_activity`、连接池 | 找 idle in transaction、调连接池、优化事务 |
| 慢查询 | `pg_stat_statements`、执行计划 | 索引、统计信息、SQL 改写 |
| 锁等待 | `pg_locks`、阻塞 pid | 取消问题 SQL、固定更新顺序 |
| WAL 暴涨 | 归档、复制槽、写入量 | 恢复归档、处理滞后槽 |
| 表膨胀 | dead tuples、长事务 | 处理长事务、调 autovacuum |
| 备库延迟 | 发送和重放状态 | 检查网络、I/O、长查询 |

## 例子

生成排障快照：

```sql
SELECT now() AS snapshot_time;

SELECT state, wait_event_type, wait_event, count(*)
FROM pg_stat_activity
GROUP BY state, wait_event_type, wait_event
ORDER BY count(*) DESC;

SELECT relname, n_dead_tup, last_autovacuum
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 10;
```

## 练习

1. 开启慢查询日志，执行一个慢查询并在日志中找到它。
2. 制造锁等待，用系统视图找阻塞源。
3. 制造 `idle in transaction`，观察它对 VACUUM 的影响。
4. 停止逻辑订阅或备库，观察复制槽状态和 WAL 增长。

## 验收

- 能写出慢查询排查流程。
- 能区分取消查询和终止连接。
- 能解释 autovacuum 为什么没有立刻释放磁盘空间。
- 能根据复制状态判断延迟位置。

## 重点

- 排障先收集证据，再执行变更。
- 长事务和复制槽是生产隐性风险。
- `VACUUM FULL` 会重写表并加重锁，不能随意执行。

## 难点

- 许多问题有时间维度。当前视图只能看到现场，日志和历史指标能补充事故发生过程。

## 易错

> **易错：** 一遇到数据库慢就重启。
>
> 正确做法：先确认慢查询、锁、连接、I/O、WAL、复制和 autovacuum 状态，重启可能清掉关键现场。

> **易错：** 把 `idle` 连接和 `idle in transaction` 混为一谈。
>
> 正确做法：`idle in transaction` 持有事务快照和可能的锁，风险更高。
