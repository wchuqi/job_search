# PostgreSQL 13 学习资料：实验手册和自测题

[返回索引](../PostgreSQL13学习资料.md)

## 学习目标

- 通过实验验证 PostgreSQL 13 的事务、锁、索引、执行计划、VACUUM、复制和备份机制。
- 把抽象概念变成可观察现象。
- 用自测题检查是否达到面试和生产实践要求。

## 实验环境建议

- 使用测试实例，不要在生产库执行压测或破坏性实验。
- 每个实验前记录 PostgreSQL 版本：

```sql
SELECT version();
SHOW server_version;
```

- 建议创建独立数据库：

```sql
CREATE DATABASE pg13_lab;
```

## 实验 1：READ COMMITTED 和 REPEATABLE READ

### 目标

观察不同隔离级别下 snapshot 刷新时机。

### 步骤

准备：

```sql
DROP TABLE IF EXISTS iso_lab;
CREATE TABLE iso_lab(id int PRIMARY KEY, value text);
INSERT INTO iso_lab VALUES (1, 'v1');
```

会话 A：

```sql
BEGIN ISOLATION LEVEL READ COMMITTED;
SELECT value FROM iso_lab WHERE id = 1;
```

会话 B：

```sql
UPDATE iso_lab SET value = 'v2' WHERE id = 1;
COMMIT;
```

会话 A：

```sql
SELECT value FROM iso_lab WHERE id = 1;
COMMIT;
```

重复一次，把会话 A 改成：

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
```

### 验收

- READ COMMITTED 第二次看到 `v2`。
- REPEATABLE READ 第二次仍看到 `v1`。
- 能解释这不是脏读，而是 snapshot 刷新策略不同。

## 实验 2：行锁和阻塞链

### 目标

观察两个事务更新同一行时的等待，并找出阻塞源。

准备：

```sql
DROP TABLE IF EXISTS lock_lab;
CREATE TABLE lock_lab(id int PRIMARY KEY, value int);
INSERT INTO lock_lab VALUES (1, 100);
```

会话 A：

```sql
BEGIN;
UPDATE lock_lab SET value = value + 1 WHERE id = 1;
```

会话 B：

```sql
BEGIN;
UPDATE lock_lab SET value = value + 1 WHERE id = 1;
```

会话 C：

```sql
SELECT
  pid,
  pg_blocking_pids(pid) AS blocking_pids,
  wait_event_type,
  wait_event,
  query
FROM pg_stat_activity
WHERE cardinality(pg_blocking_pids(pid)) > 0;
```

### 验收

- 会话 B 等待会话 A。
- 能找到 blocking pid。
- 会话 A COMMIT 后，会话 B 继续执行。

## 实验 3：死锁

### 目标

构造死锁并理解固定访问顺序的价值。

准备：

```sql
DROP TABLE IF EXISTS deadlock_lab;
CREATE TABLE deadlock_lab(id int PRIMARY KEY, value int);
INSERT INTO deadlock_lab VALUES (1, 10), (2, 20);
```

会话 A：

```sql
BEGIN;
UPDATE deadlock_lab SET value = value + 1 WHERE id = 1;
```

会话 B：

```sql
BEGIN;
UPDATE deadlock_lab SET value = value + 1 WHERE id = 2;
```

会话 A：

```sql
UPDATE deadlock_lab SET value = value + 1 WHERE id = 2;
```

会话 B：

```sql
UPDATE deadlock_lab SET value = value + 1 WHERE id = 1;
```

### 验收

- 一个事务被 PostgreSQL 中止。
- 能在日志中找到 deadlock 信息。
- 能说明如果都按 id 升序更新，死锁风险会降低。

## 实验 4：HOT update 和 fillfactor

### 目标

观察更新非索引列和索引列对 HOT update 的影响。

准备：

```sql
DROP TABLE IF EXISTS hot_lab;
CREATE TABLE hot_lab(
  id int PRIMARY KEY,
  status text,
  note text
) WITH (fillfactor = 70);

INSERT INTO hot_lab
SELECT g, 'new', repeat('x', 100)
FROM generate_series(1, 50000) AS g;
```

更新非索引列：

```sql
UPDATE hot_lab SET note = note || 'a' WHERE id <= 20000;

SELECT relname, n_tup_upd, n_tup_hot_upd
FROM pg_stat_user_tables
WHERE relname = 'hot_lab';
```

增加索引并更新索引列：

```sql
CREATE INDEX idx_hot_lab_status ON hot_lab(status);
UPDATE hot_lab SET status = 'done' WHERE id <= 20000;

SELECT relname, n_tup_upd, n_tup_hot_upd
FROM pg_stat_user_tables
WHERE relname = 'hot_lab';
```

### 验收

- 能解释为什么更新索引列 HOT 机会降低。
- 能说明 fillfactor 为什么可能改善 HOT。

## 实验 5：数据倾斜和执行计划

### 目标

观察同一列不同值因为选择率不同而走不同计划。

```sql
DROP TABLE IF EXISTS skew_lab;
CREATE TABLE skew_lab(id bigserial PRIMARY KEY, status text NOT NULL, payload text);

INSERT INTO skew_lab(status, payload)
SELECT
  CASE WHEN g <= 990000 THEN 'archived' ELSE 'active' END,
  md5(g::text)
FROM generate_series(1, 1000000) AS g;

CREATE INDEX idx_skew_lab_status ON skew_lab(status);
ANALYZE skew_lab;

EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM skew_lab WHERE status = 'active';

EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM skew_lab WHERE status = 'archived';
```

### 验收

- 能解释 `active` 和 `archived` 的行数差异。
- 能说明为什么返回大量行时顺序扫描可能更便宜。

## 实验 6：排序落盘

### 目标

观察 `work_mem` 对排序的影响。

```sql
SET work_mem = '1MB';

EXPLAIN (ANALYZE, BUFFERS)
SELECT *
FROM skew_lab
ORDER BY payload;
```

再设置：

```sql
SET work_mem = '128MB';
```

重复查询。

### 验收

- 能找到 `Sort Method`。
- 能解释 external merge 和 quicksort 的差异。
- 能说明为什么不应全局盲目提高 `work_mem`。

## 实验 7：长事务阻止 VACUUM

### 目标

验证长事务对 dead tuple 清理的影响。

准备：

```sql
DROP TABLE IF EXISTS vacuum_lab;
CREATE TABLE vacuum_lab(id bigserial PRIMARY KEY, value text);
INSERT INTO vacuum_lab(value)
SELECT md5(g::text)
FROM generate_series(1, 200000) AS g;
```

会话 A：

```sql
BEGIN;
SELECT count(*) FROM vacuum_lab;
```

会话 B：

```sql
UPDATE vacuum_lab SET value = md5(random()::text);
VACUUM vacuum_lab;

SELECT relname, n_live_tup, n_dead_tup
FROM pg_stat_user_tables
WHERE relname = 'vacuum_lab';
```

会话 A 提交后，会话 B 再 VACUUM。

### 验收

- 能说明为什么会话 A 阻止旧版本回收。
- 能用 `pg_stat_activity` 找到长事务。

## 实验 8：部分索引

### 目标

验证部分索引只有在查询条件匹配谓词时才可用。

```sql
DROP TABLE IF EXISTS partial_idx_lab;
CREATE TABLE partial_idx_lab(id bigserial PRIMARY KEY, status text, created_at timestamptz);

INSERT INTO partial_idx_lab(status, created_at)
SELECT
  CASE WHEN g <= 10000 THEN 'open' ELSE 'closed' END,
  now() - (g || ' minutes')::interval
FROM generate_series(1, 500000) AS g;

CREATE INDEX idx_partial_open_created
ON partial_idx_lab(created_at DESC)
WHERE status = 'open';

ANALYZE partial_idx_lab;

EXPLAIN (ANALYZE, BUFFERS)
SELECT *
FROM partial_idx_lab
WHERE status = 'open'
ORDER BY created_at DESC
LIMIT 20;

EXPLAIN (ANALYZE, BUFFERS)
SELECT *
FROM partial_idx_lab
ORDER BY created_at DESC
LIMIT 20;
```

### 验收

- 第一条查询可使用部分索引。
- 第二条查询不能安全使用该部分索引。

## 实验 9：逻辑备份和恢复

### 目标

验证 `pg_dump` 和 `pg_restore`。

命令示例：

```powershell
pg_dump -h localhost -U postgres -d pg13_lab -Fc -f pg13_lab.dump
createdb -h localhost -U postgres pg13_lab_restore
pg_restore -h localhost -U postgres -d pg13_lab_restore pg13_lab.dump
```

### 验收

- 恢复库对象和数据可用。
- 能说明单库 dump 不包含所有全局 role。

### 目标

必须包含：

- 做物理基础备份。
- 插入恢复前标记。
- 执行误删。
- 设置 `restore_command` 和 `recovery_target_time`。

## 重点

- 实验的价值是看到现象并解释机制。
- 每个性能结论都要有执行计划和数据量背景。
- 每个恢复结论都要通过恢复验证。

## 难点

- 同一个实验在不同机器、不同数据分布、不同缓存状态下结果可能不同。重点不是死记结果，而是解释为什么会这样。

## 易错

> **易错：** 跑完实验只记录“快了”或“慢了”。
>
> 正确做法：记录数据量、索引、参数、执行计划、耗时和 buffer。

> **易错：** 在生产环境直接执行教材里的压测或破坏性 SQL。
>
> 正确做法：所有实验在隔离测试库执行，生产只做只读诊断或经过审批的维护操作。
