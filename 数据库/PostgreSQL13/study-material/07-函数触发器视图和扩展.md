# PostgreSQL 13 学习资料：函数、触发器、视图和扩展

[返回索引](../PostgreSQL13学习资料.md)

## 学习目标

- 理解视图、物化视图、函数、存储过程、触发器和扩展的职责边界。
- 能编写简单 PL/pgSQL 函数和触发器。
- 能判断哪些逻辑适合放在数据库中，哪些逻辑应留在应用层。

## 理论导读

PostgreSQL 不只是存储表数据，也提供数据库端编程能力。视图可以封装查询，函数可以复用逻辑，触发器可以在数据变化时自动执行校验或补充操作，扩展可以为数据库增加能力，例如 `pg_stat_statements`、`uuid-ossp`、`pgcrypto`、PostGIS 等。

这些能力很强，但需要控制边界。数据库端逻辑适合放置强一致的数据规则、审计、派生字段维护、权限封装和靠近数据的计算；复杂业务流程、外部 API 调用、长耗时任务通常不适合塞进触发器。

## 核心心智模型

数据库端对象是“靠近数据的工具箱”：

- 视图负责统一读法。
- 函数负责封装可复用计算。
- 触发器负责在写入生命周期中自动执行。
- 扩展负责引入数据库原生没有的能力。

## 知识点详解

### 1. 普通视图

视图保存的是查询定义，不保存数据。查询视图时，优化器会把视图定义纳入整体计划。

```sql
CREATE VIEW open_job_applications AS
SELECT
  j.id AS job_id,
  j.title,
  count(a.id) AS application_count
FROM job j
LEFT JOIN application a ON a.job_id = j.id
WHERE j.status = 'open'
GROUP BY j.id, j.title;
```

### 2. 物化视图

物化视图保存查询结果，适合代价较高且不需要实时的数据。刷新时要考虑锁和可用性。

```sql
CREATE MATERIALIZED VIEW mv_job_daily_stats AS
SELECT
  job_id,
  date_trunc('day', created_at) AS day,
  count(*) AS cnt
FROM application
GROUP BY job_id, date_trunc('day', created_at);

REFRESH MATERIALIZED VIEW mv_job_daily_stats;
```

若要并发刷新，需要物化视图有唯一索引：

```sql
CREATE UNIQUE INDEX idx_mv_job_daily_stats
ON mv_job_daily_stats(job_id, day);

REFRESH MATERIALIZED VIEW CONCURRENTLY mv_job_daily_stats;
```

### 3. PL/pgSQL 函数

```sql
CREATE OR REPLACE FUNCTION touch_updated_at()
RETURNS trigger
LANGUAGE plpgsql
AS $$
BEGIN
  NEW.updated_at := now();
  RETURN NEW;
END;
$$;
```

函数可以设置 `VOLATILE`、`STABLE`、`IMMUTABLE`。这会影响优化器能否缓存或重排调用。

| 波动性 | 含义 |
| --- | --- |
| `VOLATILE` | 结果可能每次都变，默认值 |
| `STABLE` | 单条语句内结果稳定 |
| `IMMUTABLE` | 相同输入永远相同输出 |

### 4. 触发器

```sql
CREATE TRIGGER trg_application_touch_updated_at
BEFORE UPDATE ON application
FOR EACH ROW
EXECUTE FUNCTION touch_updated_at();
```

触发器可按时机分为 `BEFORE`、`AFTER`、`INSTEAD OF`，按粒度分为行级和语句级。行级触发器每行执行一次，大批量更新时成本可能很高。

### 5. 扩展

```sql
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
CREATE EXTENSION IF NOT EXISTS pgcrypto;

SELECT gen_random_uuid();
```

扩展通常安装在数据库级别，不是 cluster 全局自动可用。生产环境安装扩展要确认版本、权限、兼容性和备份恢复策略。

## 例子

审计触发器：

```sql
CREATE TABLE audit_log (
  id bigserial PRIMARY KEY,
  table_name text NOT NULL,
  row_id text NOT NULL,
  operation text NOT NULL,
  changed_at timestamptz NOT NULL DEFAULT now(),
  changed_by text NOT NULL DEFAULT current_user
);

CREATE OR REPLACE FUNCTION audit_application_change()
RETURNS trigger
LANGUAGE plpgsql
AS $$
BEGIN
  INSERT INTO audit_log(table_name, row_id, operation)
  VALUES ('application', COALESCE(NEW.id, OLD.id)::text, TG_OP);
  RETURN COALESCE(NEW, OLD);
END;
$$;

CREATE TRIGGER trg_audit_application
AFTER INSERT OR UPDATE OR DELETE ON application
FOR EACH ROW
EXECUTE FUNCTION audit_application_change();
```

## 练习

1. 创建一个视图，展示开放岗位和投递人数。
2. 创建一个物化视图，统计每日投递数，并尝试并发刷新。
3. 用触发器维护 `updated_at`。
4. 安装 `pg_stat_statements`，查询最耗时 SQL。

## 验收

- 能区分视图和物化视图。
- 能解释触发器的执行时机和粒度。
- 能说明函数波动性对优化器的影响。
- 能判断扩展安装的权限和风险。

## 重点

- 触发器适合强一致数据规则，不适合复杂外部流程。
- 物化视图需要刷新，不能当实时数据看待。
- 函数的 `IMMUTABLE`、`STABLE`、`VOLATILE` 不应乱标。

## 难点

- 数据库端逻辑越多，隐式行为越多。团队需要文档、命名规范和测试，否则排障成本会上升。

## 易错

> **易错：** 在触发器里执行复杂业务流程或远程调用。
>
> 正确做法：触发器只做短小、确定、靠近数据的逻辑，复杂流程交给应用或异步任务。

> **易错：** 把会读取表或时间的函数标成 `IMMUTABLE`。
>
> 正确做法：只有相同输入永远相同输出的函数才标 `IMMUTABLE`。
