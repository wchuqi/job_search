# PostgreSQL 13 学习资料：索引类型和查询规划

[返回索引](../PostgreSQL13学习资料.md)

## 学习目标

- 理解 PostgreSQL 主要索引类型和适用场景。
- 掌握复合索引、部分索引、表达式索引、覆盖索引的设计方法。
- 理解优化器选择访问路径的基本规则。

## 理论导读

索引是访问路径，不是查询加速的万能开关。索引能减少读取的数据量，但也会增加写入、更新、删除、VACUUM 和存储成本。优化器会根据统计信息估算顺序扫描、索引扫描、位图扫描等方案的成本，然后选择成本最低的计划。

PostgreSQL 13 的 B-tree 索引支持重复键去重，对大量重复键场景能减少索引体积。但这不改变索引设计原则：索引仍要服务具体谓词、排序和连接。

## 核心心智模型

把索引想成目录。目录适合按目录项快速定位少量内容；如果一本书大部分页都要读，从第一页顺序翻可能更快。数据库也是一样，如果查询会返回大量行，顺序扫描可能比索引扫描更便宜。

## 知识点详解

### 1. 索引类型

| 类型 | 适用场景 | 示例 |
| --- | --- | --- |
| B-tree | 等值、范围、排序、大多数默认场景 | `email = ?`、`created_at > ?` |
| Hash | 等值查询 | 使用较少，B-tree 通常足够 |
| GIN | 数组、`jsonb`、全文检索 | `profile @> ...` |
| GiST | 几何、范围、排斥约束、全文检索 | 时间段冲突 |
| SP-GiST | 分区空间结构、特定数据类型 | 前缀、空间类查询 |
| BRIN | 超大表且物理顺序和字段相关 | 时间序列表 |

### 2. 复合索引顺序

复合索引顺序要结合等值条件、范围条件、排序和选择率。一般经验：

- 等值条件列通常放前面。
- 范围条件之后的列较难继续用于缩小范围。
- 如果需要同时满足过滤和排序，索引顺序要兼顾 `WHERE` 和 `ORDER BY`。

```sql
CREATE INDEX idx_application_job_status_created
ON application(job_id, status, created_at DESC);
```

适合：

```sql
SELECT *
FROM application
WHERE job_id = 10
  AND status = 'submitted'
ORDER BY created_at DESC
LIMIT 20;
```

### 3. 部分索引

部分索引只索引满足条件的行，适合热点子集。

```sql
CREATE INDEX idx_job_open_created
ON job(created_at DESC)
WHERE status = 'open';
```

适合频繁查询开放岗位，而关闭岗位很多的场景。

### 4. 表达式索引

表达式索引用于查询条件包含函数或表达式的场景。

```sql
CREATE INDEX idx_candidate_email_lower
ON candidate(lower(email));

SELECT *
FROM candidate
WHERE lower(email) = lower('A@EXAMPLE.COM');
```

### 5. 覆盖索引

PostgreSQL 支持 `INCLUDE` 列，让查询可以从索引中拿到额外字段，减少回表机会。

```sql
CREATE INDEX idx_application_job_created_include
ON application(job_id, created_at DESC)
INCLUDE (status, candidate_id);
```

是否能 index-only scan 还受可见性映射影响。如果表频繁更新，仍可能需要访问堆表。

### 6. 查询规划基础

优化器会根据统计信息估算：

- 表有多少行。
- 条件选择率是多少。
- 需要读取多少页面。
- join 的输入大小和连接方式成本。
- 是否需要排序、哈希、聚合。

常见访问方式：

| 访问方式 | 含义 |
| --- | --- |
| Seq Scan | 顺序扫描整表或大部分表 |
| Index Scan | 通过索引定位，再访问表 |
| Index Only Scan | 主要从索引返回结果 |
| Bitmap Index Scan | 先从索引收集位置，再批量访问表 |

## 例子

查看索引：

```sql
SELECT
  schemaname,
  tablename,
  indexname,
  indexdef
FROM pg_indexes
WHERE schemaname = 'public'
ORDER BY tablename, indexname;
```

分析查询计划：

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, candidate_id, status
FROM application
WHERE job_id = 10
  AND status = 'submitted'
ORDER BY created_at DESC
LIMIT 20;
```

## 练习

1. 为 `candidate(email)` 建唯一索引，测试大小写敏感问题。
2. 为开放岗位列表建部分索引，比较前后计划。
3. 为 `jsonb` 技能查询建 GIN 索引。
4. 构造一个低选择率条件，观察优化器为什么选择顺序扫描。

## 验收

- 能说出 B-tree、GIN、GiST、BRIN 的基本适用范围。
- 能根据查询条件设计复合索引顺序。
- 能解释部分索引必须让查询条件匹配索引谓词。
- 能解释为什么 index-only scan 不一定总能发生。

## 重点

- 索引设计必须从查询模式出发。
- PostgreSQL 13 优化器基于统计信息和成本模型选择计划。
- 索引越多，写入和维护成本越高。

## 难点

- 复合索引同时服务过滤、排序和连接时，需要结合真实 SQL 和数据分布验证。

## 易错

> **易错：** 认为有索引就一定会走索引。
>
> 正确做法：用 `EXPLAIN (ANALYZE, BUFFERS)` 验证，并检查选择率、统计信息和返回行数。

> **易错：** 为每个字段都建单列索引。
>
> 正确做法：基于高频查询、写入成本和组合条件设计少量高价值索引。
