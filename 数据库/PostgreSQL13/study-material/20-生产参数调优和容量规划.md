# PostgreSQL 13 学习资料：生产参数调优和容量规划

[返回索引](../PostgreSQL13学习资料.md)

## 学习目标

- 建立 PostgreSQL 参数调优的决策框架，而不是背参数模板。
- 能根据内存、CPU、磁盘、连接数、写入量和查询形态做容量规划。
- 能识别哪些参数适合全局设置，哪些应按会话、角色、数据库或表设置。
- 能给 PostgreSQL 13 制定上线前检查清单。

## 理论导读

参数调优的目标不是追求某个“最佳值”，而是让数据库资源分配匹配业务负载。OLTP、报表、批处理、日志写入、混合负载的瓶颈不同，参数也不同。错误的参数可能比默认值更糟糕，例如盲目增大 `work_mem` 导致并发内存耗尽，盲目增大 `max_connections` 导致进程和内存成本失控。

参数调优应遵循：

1. 先识别负载类型和瓶颈。
2. 先做容量预算。
3. 小步修改。
4. 观测指标。
5. 保留回滚方案。

## 核心心智模型

数据库资源像一个预算表：

- 连接数消耗进程、内存和调度能力。
- `shared_buffers` 是 PostgreSQL 自己的共享缓存。
- 操作系统 page cache 仍然重要。
- `work_mem` 是并发乘法项。
- WAL 和 checkpoint 是写入路径的核心预算。
- autovacuum 是必须预留的后台维护预算。

## 知识点详解

## 一、配置层级

PostgreSQL 参数可能来自多个层级：

| 层级 | 示例 | 说明 |
| --- | --- | --- |
| 配置文件 | `postgresql.conf` | 基础配置 |
| ALTER SYSTEM | `ALTER SYSTEM SET ...` | 写入自动配置文件 |
| database | `ALTER DATABASE job_app SET ...` | 连接到库时生效 |
| role | `ALTER ROLE app_user SET ...` | 指定用户生效 |
| session | `SET work_mem = ...` | 当前会话生效 |
| transaction | `SET LOCAL ...` | 当前事务生效 |
| table storage | `ALTER TABLE ... SET (...)` | 表级 autovacuum、fillfactor 等 |

查看来源：

```sql
SELECT name, setting, unit, context, source
FROM pg_settings
WHERE name IN ('work_mem', 'shared_buffers', 'max_connections');
```

`context` 决定是否需要 restart、reload 或 session 设置。

## 二、内存预算

### shared_buffers

`shared_buffers` 是 PostgreSQL 共享缓冲区。它不是越大越好，因为操作系统缓存也需要内存。常见起点是物理内存的 25% 左右，但要结合系统和负载验证。

观察缓存命中：

```sql
SELECT
  sum(heap_blks_hit) AS heap_hit,
  sum(heap_blks_read) AS heap_read,
  round(100.0 * sum(heap_blks_hit) / nullif(sum(heap_blks_hit) + sum(heap_blks_read), 0), 2) AS hit_pct
FROM pg_statio_user_tables;
```

命中率高不代表没有 I/O 问题，命中率低也不一定只靠加内存解决。要结合工作集大小、索引设计和查询模式。

### effective_cache_size

`effective_cache_size` 不是实际分配内存，而是告诉优化器“系统大概有多少缓存可用”。它影响优化器对索引扫描成本的估计。

如果设置过低，优化器可能低估缓存效果，偏向顺序扫描；设置过高则可能过度乐观。

### work_mem

`work_mem` 是最容易被误调的参数。

近似风险：

```text
最大潜在内存 = 并发连接数 * 每条查询活跃排序/哈希节点数 * work_mem
```

实际不会总是达到最大，但这个乘法说明了为什么不能全局随意调大。

推荐做法：

```sql
-- 针对一个报表事务临时提高
BEGIN;
SET LOCAL work_mem = '128MB';
-- report query
COMMIT;
```

监控临时文件：

```sql
SELECT datname, temp_files, pg_size_pretty(temp_bytes) AS temp_size
FROM pg_stat_database
ORDER BY temp_bytes DESC;
```

日志：

```conf
log_temp_files = 0
```

测试环境可设为 0 记录全部临时文件，生产可设阈值。

### maintenance_work_mem 和 autovacuum_work_mem

影响 VACUUM、CREATE INDEX、ALTER TABLE ADD FOREIGN KEY 等维护操作。大索引创建或 VACUUM 可能受益，但要考虑并发维护任务数量。

## 三、连接数和连接池

`max_connections` 不是业务并发能力。连接太多会增加：

- 后端进程数量。
- 每连接内存开销。
- 锁表和快照管理开销。
- 上下文切换。

建议：

- 应用侧使用连接池。
- 数据库前可用 PgBouncer。
- OLTP 通常让活跃数据库连接数量接近 CPU 和 I/O 能力，而不是请求并发数。

定位连接：

```sql
SELECT
  application_name,
  usename,
  state,
  count(*)
FROM pg_stat_activity
GROUP BY application_name, usename, state
ORDER BY count(*) DESC;
```

## 四、WAL 和 checkpoint 参数

关键目标：写入稳定、恢复时间可控、磁盘不爆。

| 参数 | 调整思路 |
| --- | --- |
| `max_wal_size` | 太小会 checkpoint 频繁，写入库通常需要增大 |
| `min_wal_size` | 保留 WAL 空间，减少反复创建删除 |
| `checkpoint_timeout` | 太短会频繁 checkpoint |
| `checkpoint_completion_target` | 常设较高，让写出更平滑 |
| `wal_compression` | 可减少 full page image WAL 体积，但增加 CPU |
| `wal_keep_size` | 为没有 slot 的备库保留 WAL |
| `max_slot_wal_keep_size` | 限制复制槽保留 WAL，PG13 可用 |

观察：

```sql
SELECT
  checkpoints_timed,
  checkpoints_req,
  buffers_checkpoint,
  checkpoint_write_time,
  checkpoint_sync_time
FROM pg_stat_bgwriter;
```

如果 `checkpoints_req` 经常增长，优先评估 `max_wal_size` 和写入峰值。

## 五、Autovacuum 参数预算

不要只看全局参数。高频更新大表要按表设置：

```sql
ALTER TABLE order_state SET (
  autovacuum_vacuum_scale_factor = 0.02,
  autovacuum_analyze_scale_factor = 0.01,
  autovacuum_vacuum_cost_limit = 4000
);
```

全局参数要保证后台有足够 worker：

```conf
autovacuum_max_workers = 5
autovacuum_naptime = '30s'
```

是否合适要看：

- dead tuple 是否持续积累。
- autovacuum 是否频繁被取消。
- 大表是否接近 freeze 风险。
- 业务高峰是否有 I/O 抖动。

## 六、日志和观测参数

生产建议关注：

```conf
log_min_duration_statement = '500ms'
log_lock_waits = on
deadlock_timeout = '1s'
log_checkpoints = on
log_temp_files = '64MB'
track_io_timing = on
shared_preload_libraries = 'pg_stat_statements'
```

`track_io_timing` 可以帮助区分 I/O 时间，但可能有少量开销。应在目标环境评估。

`pg_stat_statements` 在 PostgreSQL 13 需要 `shared_preload_libraries` 加载并重启，然后在目标库：

```sql
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
```

Top SQL：

```sql
SELECT
  query,
  calls,
  total_exec_time,
  mean_exec_time,
  rows,
  shared_blks_hit,
  shared_blks_read
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;
```

## 七、表设计和容量规划

容量不是只估数据行大小，还要估：

- 主表 heap。
- 索引。
- TOAST。
- WAL 峰值和归档。
- 临时文件。
- 备份空间。
- 膨胀冗余。
- 复制槽滞后空间。

粗略估算：

```text
总空间 = 表数据 + 索引 + TOAST + 膨胀预留 + WAL预留 + 备份/临时空间
```

常见预留：

- OLTP 高频更新表：预留更高膨胀空间。
- 批量导入：预留导入期间 WAL 和临时文件。
- 物理备份：目标端至少接近数据目录大小，压缩另算。
- `CREATE INDEX CONCURRENTLY`：会额外占用新索引空间，期间还要维护旧索引。

## 八、不同负载的调优重点

### OLTP

- 控制连接数。
- 高价值索引，避免过多索引拖慢写入。
- 短事务。
- autovacuum 及时。
- WAL 和 checkpoint 平滑。

### 报表分析

- 合理 `work_mem`。
- 覆盖索引或汇总表。
- 物化视图。
- 读副本隔离报表负载。
- 防止长查询影响 VACUUM 或备库重放。

### 日志和时序写入

- 时间分区。
- BRIN 或少量必要索引。
- 批量写入。
- 按分区归档和删除。
- 控制 WAL 归档和备份窗口。

### 混合负载

- 把 OLTP 和报表隔离。
- 用只读副本或数据仓库承接大分析。
- 应用侧限流和超时。
- 重点防止报表拖垮主库。

## 九、上线前检查清单

| 类别 | 检查项 |
| --- | --- |
| 版本 | PostgreSQL 13 已 EOL，是否已有升级计划 |
| 连接 | 连接池、最大连接、超时、应用名 |
| 权限 | 应用非超级用户、最小权限、默认权限 |
| 备份 | 逻辑备份、物理备份、恢复演练 |
| WAL | 归档、磁盘预留、复制槽监控 |
| VACUUM | 大表表级参数、长事务监控 |
| 性能 | Top SQL、关键索引、统计信息 |
| 日志 | 慢查询、锁等待、checkpoint、临时文件 |
| 高可用 | 故障切换、旧主隔离、回滚流程 |
| 安全 | SSL、密码策略、`pg_hba.conf`、审计 |

## 例子：估算 work_mem 风险

假设：

- PgBouncer 后端允许 80 个活跃数据库连接。
- 一个复杂查询可能同时有 3 个排序或哈希节点。
- 全局 `work_mem = 64MB`。

潜在峰值：

```text
80 * 3 * 64MB = 15360MB
```

这还不含 `shared_buffers`、maintenance、autovacuum、系统缓存和进程其他内存。结论是：全局 64MB 可能太冒险，应改为较保守全局值，并对报表任务局部提高。

## 实操任务

1. 查询当前关键参数的 `context` 和 `source`。
2. 统计连接状态和应用来源。
3. 找出临时文件最多的数据库。
4. 找出 checkpoint 请求触发是否频繁。
5. 为一张大表制定表级 autovacuum 参数。
6. 写一份上线检查清单并逐项验证。

## 验收

- 能解释 `shared_buffers`、`effective_cache_size`、`work_mem` 的区别。
- 能按并发估算 `work_mem` 风险。
- 能说明 `max_connections` 为什么不等于吞吐能力。
- 能根据 `pg_stat_bgwriter` 判断 checkpoint 压力。
- 能写出 PostgreSQL 13 生产上线参数检查表。

## 重点

- 参数调优必须基于负载和指标，不是模板复制。
- `work_mem`、连接数和并发节点数是乘法关系。
- WAL、checkpoint 和 autovacuum 都要预留资源。
- PostgreSQL 13 的版本风险必须进入生产规划。

## 难点

- 同一个参数可能改善一个场景，同时伤害另一个场景。例如增大 `work_mem` 可以减少排序落盘，但高并发下可能造成内存压力。

## 易错

> **易错：** 把 `effective_cache_size` 当成实际分配内存。
>
> 正确做法：它是优化器成本估算参数，不会真的申请内存。

> **易错：** 为了处理连接耗尽直接把 `max_connections` 翻倍。
>
> 正确做法：先查连接泄漏、慢 SQL、事务状态和连接池设置。

> **易错：** 只看数据库目录大小做磁盘规划。
>
> 正确做法：还要预留 WAL、归档、备份、临时文件、索引重建和膨胀空间。
