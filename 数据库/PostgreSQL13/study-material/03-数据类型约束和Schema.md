# PostgreSQL 13 学习资料：数据类型、约束和 Schema

[返回索引](../PostgreSQL13学习资料.md)

## 学习目标

- 掌握 PostgreSQL 常见数据类型及其选择原则。
- 理解约束、默认值、序列、身份列、schema 和 search path。
- 能识别类型选择错误带来的性能、正确性和迁移问题。

## 理论导读

数据类型不是表面格式，而是数据的存储、比较、索引和函数行为的基础。比如 `text` 和 `varchar(n)` 在 PostgreSQL 中性能差异通常不是重点，真正重要的是是否需要长度约束；`timestamp` 和 `timestamptz` 的区别会影响跨时区系统正确性；`jsonb` 很灵活，但滥用会让约束、索引和统计信息变弱。

Schema 是 database 内的命名空间。`search_path` 决定未限定对象名的查找顺序，这既影响便利性，也影响安全。

## 核心心智模型

数据类型像合同，约束像验收规则，schema 像办公区域。合同写清数据是什么，验收规则保证不合格数据不能入库，办公区域防止不同团队或模块的对象名字冲突。

## 知识点详解

### 1. 常见数据类型选择

| 需求 | 推荐类型 | 注意点 |
| --- | --- | --- |
| 自增主键 | `bigserial` 或 `GENERATED ... AS IDENTITY` | 大系统优先 `bigint` |
| 金额 | `numeric(p, s)` | 避免用浮点存金额 |
| 时间点 | `timestamptz` | 保存绝对时间点，显示时按时区转换 |
| 本地日期 | `date` | 生日、入职日等不需要时区 |
| 状态 | `text` 加 `CHECK` 或 enum | enum 修改不如 `CHECK` 灵活 |
| 半结构化数据 | `jsonb` | 可索引，但不应替代核心关系模型 |
| IP 地址 | `inet` | 比 text 更适合比较和索引 |
| UUID | `uuid` | 分布式生成方便，但随机 UUID 对索引局部性不友好 |

### 2. 约束类型

| 约束 | 解决的问题 |
| --- | --- |
| `PRIMARY KEY` | 行标识，隐含唯一和非空 |
| `UNIQUE` | 业务唯一性 |
| `NOT NULL` | 必填 |
| `CHECK` | 字段值合法范围 |
| `FOREIGN KEY` | 引用完整性 |
| `EXCLUDE` | 排斥约束，例如时间段不重叠 |

### 3. `NULL` 的语义

`NULL` 不是空字符串，也不是 0，而是未知或不存在。普通比较中 `NULL = NULL` 结果不是 true，而是 unknown。判断空值必须用 `IS NULL`。

```sql
SELECT 1 WHERE NULL = NULL;      -- 无结果
SELECT 1 WHERE NULL IS NULL;     -- 有结果
```

唯一约束中，PostgreSQL 13 允许多行 `NULL`，因为 `NULL` 不被视作相等。需要“空值也只能有一个”时，要额外设计表达式索引或业务约束。

### 4. Schema 和 `search_path`

```sql
SHOW search_path;
CREATE SCHEMA hr;
CREATE TABLE hr.candidate(id bigint PRIMARY KEY);
```

未限定表名时，PostgreSQL 按 `search_path` 查找对象。生产 SQL 尤其是函数中，建议显式限定 schema 或固定安全的 `search_path`。

### 5. 序列和身份列

`bigserial` 本质上会创建序列并设置默认值。序列取号不参与事务回滚，因此出现 ID 间隔是正常现象。

```sql
CREATE TABLE event_log (
  id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  payload jsonb NOT NULL,
  created_at timestamptz NOT NULL DEFAULT now()
);
```

## 例子

投递表增加更严格约束：

```sql
CREATE TABLE application_period (
  id bigserial PRIMARY KEY,
  candidate_id bigint NOT NULL,
  job_id bigint NOT NULL,
  valid_from timestamptz NOT NULL,
  valid_to timestamptz,
  CHECK (valid_to IS NULL OR valid_to > valid_from),
  UNIQUE(candidate_id, job_id, valid_from)
);
```

JSONB 查询和索引：

```sql
CREATE TABLE candidate_profile (
  candidate_id bigint PRIMARY KEY,
  profile jsonb NOT NULL
);

CREATE INDEX idx_candidate_profile_gin
ON candidate_profile USING gin(profile);

SELECT *
FROM candidate_profile
WHERE profile @> '{"city": "Shanghai"}';
```

## 练习

1. 为候选人表增加手机号字段，思考是否应该用数字类型。
2. 为岗位薪资范围增加 `min_salary` 和 `max_salary`，写出检查约束。
3. 用 `jsonb` 保存候选人技能数组，并查询包含 `PostgreSQL` 技能的人。
4. 创建两个 schema，验证 `search_path` 对同名表查询的影响。

## 验收

- 能解释 `timestamp` 和 `timestamptz` 的区别。
- 能说明为什么金额不应该用 `real` 或 `double precision`。
- 能处理 `NULL` 与唯一约束的边界。
- 能解释 `search_path` 的查找顺序和安全风险。

## 重点

- 类型选择影响正确性、索引、统计信息和未来迁移。
- 约束应表达不可违反的业务规则。
- 序列不保证无间隔，只保证按调用产生值。

## 难点

- `NULL` 的三值逻辑会影响查询、唯一约束和聚合。
- `jsonb` 的灵活性和关系模型的严谨性需要平衡。

## 易错

> **易错：** 用 `timestamp without time zone` 保存跨时区事件时间。
>
> 正确做法：事件发生时刻通常用 `timestamptz`，本地日期才用 `date`。

> **易错：** 手机号用整数类型。
>
> 正确做法：手机号不是数字运算对象，应使用 `text`，并用约束或应用校验格式。
