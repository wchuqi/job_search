# PostgreSQL 13 学习资料：性能优化和执行计划

[返回索引](../PostgreSQL13学习资料.md)

## 学习目标

- 能阅读 `EXPLAIN` 和 `EXPLAIN ANALYZE`。
- 掌握慢查询定位、统计信息维护、连接算法和内存参数影响。
- 能建立性能优化的验证流程。

## 理论导读

性能优化不是凭经验猜索引，而是证据驱动的过程：先确认慢在哪里，再看执行计划和实际行数，判断估算是否准确，最后对 SQL、索引、统计信息、参数或数据模型做最小改动并复测。

PostgreSQL 的优化器会比较不同执行路径成本。成本不是实际毫秒，而是内部估算单位。`EXPLAIN ANALYZE` 会真正执行 SQL，输出实际耗时和行数。对于写操作，要在事务中谨慎测试，必要时回滚。

## 核心心智模型

一条慢 SQL 的成本通常来自四件事：

- 读太多数据。
- 排序或哈希数据太大。
- join 顺序或 join 方法不合适。
- 估算错误导致选错计划。

## 知识点详解

### 1. `EXPLAIN` 关键字段

```sql
EXPLAIN (ANALYZE, BUFFERS, VERBOSE)
SELECT ...
```

| 字段 | 含义 |
| --- | --- |
| `cost` | 估算启动成本和总成本 |
| `rows` | 估算行数 |
| `actual time` | 实际耗时 |
| `actual rows` | 实际行数 |
| `loops` | 节点执行次数 |
| `Buffers` | 共享缓冲区命中和读取情况 |

最重要的排查点是估算行数和实际行数是否偏差巨大。偏差大时，优化器可能选择错误 join 顺序或扫描方式。

### 2. Join 算法

| 算法 | 适用场景 | 风险 |
| --- | --- | --- |
| Nested Loop | 外表结果小，内表有高效索引 | 外表实际很大时会变慢 |
| Hash Join | 等值连接，输入较大 | hash 超出内存会分批，增加 I/O |
| Merge Join | 两边已排序或可利用索引顺序 | 排序成本可能较高 |

### 3. 统计信息

```sql
ANALYZE application;

SELECT attname, n_distinct, most_common_vals
FROM pg_stats
WHERE tablename = 'application';
```

统计信息描述数据分布。数据分布变化后，如果统计信息没有更新，优化器就会基于旧认知做决策。

扩展统计信息用于多列相关性：

```sql
CREATE STATISTICS st_application_job_status
ON job_id, status
FROM application;

ANALYZE application;
```

### 4. 内存参数

`work_mem` 影响排序、哈希、聚合等操作。它不是全局总量，而是每个会话中每个操作节点都可能使用。并发高时盲目增大可能造成内存压力。

```sql
SET work_mem = '64MB';
EXPLAIN (ANALYZE, BUFFERS)
SELECT ...
```

### 5. 慢查询定位流程

1. 打开慢查询日志或使用 `pg_stat_statements`。
2. 找总耗时高或平均耗时高的 SQL。
3. 对具体 SQL 执行 `EXPLAIN (ANALYZE, BUFFERS)`。
4. 比较估算行数和实际行数。
5. 判断是否缺索引、统计信息失真、SQL 写法不合适或参数不足。
6. 做一个改动，复测并记录效果。

### 6. PostgreSQL 13 的增量排序

增量排序可以利用输入数据已经部分有序的事实，只对每个局部组进行排序。它对“前缀已排序，后缀还需排序”的查询有价值。

例如索引已有 `job_id` 顺序，而查询需要 `ORDER BY job_id, created_at`，优化器可能利用已有的 `job_id` 有序性减少整体排序成本。

## 例子

找 Top SQL，需要安装扩展：

```sql
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

SELECT
  query,
  calls,
  total_exec_time,
  mean_exec_time,
  rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;
```

PostgreSQL 13 的 `pg_stat_statements` 常用执行耗时字段是 `total_exec_time` 和 `mean_exec_time`，计划耗时字段是 `total_plan_time` 和 `mean_plan_time`。不同扩展版本字段可能略有差异，可用 `\d pg_stat_statements` 确认。

分页优化示例：

```sql
-- 深分页可能越来越慢
SELECT *
FROM application
ORDER BY id
OFFSET 100000 LIMIT 20;

-- 基于游标条件的分页更稳定
SELECT *
FROM application
WHERE id > 100000
ORDER BY id
LIMIT 20;
```

## 练习

1. 制造一个 100 万行表，分别测试无索引和有索引查询。
2. 对一个 join 查询使用 `EXPLAIN (ANALYZE, BUFFERS)`，记录每个节点实际行数。
3. 调整 `work_mem`，观察排序是否从 external sort 变为内存排序。
4. 创建扩展统计信息，观察多列相关查询的估算变化。

## 验收

- 能解释 `EXPLAIN` 和 `EXPLAIN ANALYZE` 的区别。
- 能根据行数估算偏差定位统计信息问题。
- 能识别 Nested Loop 被错误放大的场景。
- 能提出优化方案并用数据验证。

## 重点

- 优化要以执行计划和实际数据为依据。
- `Buffers` 能帮助判断是 CPU、内存还是 I/O 压力。
- `work_mem` 调整必须考虑并发。

## 难点

- 慢查询往往是多个因素叠加：数据分布、参数、索引、SQL 写法和并发状态都可能影响计划。

## 易错

> **易错：** 只看总耗时，不看计划节点。
>
> 正确做法：定位耗时节点，比较估算行数、实际行数、loops 和 buffer 读取。

> **易错：** 看到排序落盘就全局大幅调高 `work_mem`。
>
> 正确做法：先评估并发和具体 SQL，可以对会话或事务局部设置。
