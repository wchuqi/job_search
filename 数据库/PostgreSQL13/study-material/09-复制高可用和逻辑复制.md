# PostgreSQL 13 学习资料：复制、高可用和逻辑复制

[返回索引](../PostgreSQL13学习资料.md)

## 学习目标

- 理解物理流复制、同步复制、复制槽和热备查询。
- 掌握逻辑复制的发布、订阅和适用边界。
- 能识别高可用方案中的故障切换、脑裂和数据丢失风险。

## 理论导读

复制不是单纯把数据拷贝到另一台机器，而是围绕 WAL 传输和重放建立的一套状态同步机制。物理复制传输 WAL，备库重放后保持与主库相同的数据文件状态；逻辑复制按表和数据变更发布，订阅端把变更应用到目标表。

高可用的核心不是“有从库”，而是故障检测、主从角色切换、客户端路由、数据一致性、恢复旧主和防止双主写入。

## 核心心智模型

物理复制像复制整本账本的每一笔底层记录，目标库必须高度同构。逻辑复制像把业务层的增删改事件发布给订阅者，灵活但不覆盖所有 DDL 和序列状态。

## 知识点详解

### 1. 物理流复制

主库配置：

```conf
wal_level = replica
max_wal_senders = 10
max_replication_slots = 10
```

创建复制用户：

```sql
CREATE ROLE repl WITH REPLICATION LOGIN PASSWORD 'change_me';
```

备库基础备份：

```powershell
pg_basebackup -h primary.example.com -U repl -D D:\pgdata_standby -Fp -Xs -P -R
```

`-R` 会写入备用库连接配置并创建 standby 信号文件。

### 2. 复制槽

复制槽保证主库保留从库还没消费的 WAL，避免从库断开太久后无法追上。但复制槽也有风险：如果订阅者或备库长期不消费，主库 WAL 会堆积。PostgreSQL 13 提供 `max_slot_wal_keep_size` 限制复制槽保留 WAL 的上限。

```sql
SELECT slot_name, slot_type, active, restart_lsn
FROM pg_replication_slots;
```

### 3. 同步复制

异步复制提交快，但主库提交后从库可能还没收到 WAL。同步复制可以等从库确认，降低数据丢失风险，但会增加提交延迟。

```conf
synchronous_commit = on
synchronous_standby_names = 'FIRST 1 (standby1, standby2)'
```

### 4. 热备查询和延迟

备库可以只读查询，但长查询可能和 WAL 重放冲突。常见现象是备库查询被取消，或开启反馈后主库无法清理旧版本。

相关参数：

- `hot_standby`
- `max_standby_streaming_delay`
- `hot_standby_feedback`

### 5. 逻辑复制

发布端：

```sql
CREATE PUBLICATION job_pub
FOR TABLE candidate, job, application;
```

订阅端：

```sql
CREATE SUBSCRIPTION job_sub
CONNECTION 'host=primary.example.com dbname=job_app user=repl password=change_me'
PUBLICATION job_pub;
```

逻辑复制适合：

- 版本升级迁移。
- 部分表同步。
- 数据分发。
- 异构数据处理前的中间同步。

边界：

- 通常不自动复制 DDL。
- 序列值需要额外处理。
- 表需要合理主键或 replica identity。
- 冲突处理能力有限，不适合随意双向写。

PostgreSQL 13 对分区表逻辑复制有增强，可以结合发布参数处理分区发布语义。

### 6. 高可用组件

常见方案会引入 Patroni、repmgr、Keepalived、HAProxy、PgBouncer 等。无论工具如何，必须明确：

- 谁负责故障检测。
- 谁负责提升新主。
- 如何防止旧主继续写。
- 客户端如何连接新主。
- 失败切换后如何重建旧主。

## 例子

查看复制状态：

```sql
SELECT
  pid,
  usename,
  application_name,
  client_addr,
  state,
  sync_state,
  write_lag,
  flush_lag,
  replay_lag
FROM pg_stat_replication;
```

查看备库重放延迟：

```sql
SELECT
  now() - pg_last_xact_replay_timestamp() AS replay_delay;
```

## 练习

1. 搭建一主一备物理复制。
2. 创建复制槽，停止备库一段时间，观察主库 WAL 保留。
3. 配置一个逻辑发布和订阅，只同步 `candidate` 表。
4. 模拟主库不可用，写出手动提升备库和恢复旧主的步骤。

## 验收

- 能区分物理复制和逻辑复制。
- 能解释复制槽的价值和风险。
- 能说明同步复制对延迟和数据丢失的影响。
- 能列出高可用切换必须处理的环节。

## 重点

- 有从库不等于高可用。
- 复制槽必须监控。
- 逻辑复制不等于完整数据库克隆。

## 难点

- 高可用真正困难的是故障状态决策和防止双主写入，不是搭建一个从库。

## 易错

> **易错：** 把异步复制当作零数据丢失方案。
>
> 正确做法：需要降低数据丢失风险时评估同步复制，但要接受提交延迟增加。

> **易错：** 创建复制槽后不监控 WAL 目录。
>
> 正确做法：监控复制槽活跃状态、保留 LSN、WAL 磁盘使用量和 `max_slot_wal_keep_size`。
