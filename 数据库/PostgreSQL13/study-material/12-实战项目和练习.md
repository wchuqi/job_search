# PostgreSQL 13 学习资料：实战项目和综合练习

[返回索引](../PostgreSQL13学习资料.md)

## 学习目标

- 用一个招聘系统数据库串联建模、SQL、事务、索引、备份、权限和运维。
- 通过可验证任务检查是否真正掌握 PostgreSQL 13。
- 建立从开发到生产的数据库交付习惯。

## 项目背景

你要为一个招聘系统设计 PostgreSQL 13 数据库。系统需要管理候选人、岗位、投递、面试、Offer、审计日志和统计报表。

## 阶段 1：建库和权限

### 任务

```sql
CREATE ROLE job_owner NOLOGIN;
CREATE ROLE job_app LOGIN PASSWORD 'change_me';
CREATE ROLE job_readonly LOGIN PASSWORD 'change_me';

CREATE DATABASE job_app OWNER job_owner;
```

在 `job_app` 中创建 `hr` 和 `audit` schema，并为应用账号和只读账号授权。

### 验收

- `job_app` 可以读写业务表。
- `job_readonly` 只能查询，不能插入或更新。
- 应用账号不是超级用户。

## 阶段 2：数据模型

### 任务

创建这些表：

- `hr.candidate`
- `hr.job`
- `hr.application`
- `hr.interview`
- `hr.offer`
- `audit.audit_log`

要求：

- 候选人邮箱唯一。
- 同一候选人对同一岗位只能有一条投递。
- 投递、面试、Offer 状态必须有合法范围。
- 所有核心表有 `created_at` 和 `updated_at`。

### 验收

- 约束能阻止重复投递。
- 外键能阻止引用不存在的候选人或岗位。
- 检查约束能阻止非法状态。

## 阶段 3：SQL 查询

### 任务

写出以下查询：

1. 每个开放岗位最近 7 天投递数。
2. 每个候选人的最近一次面试。
3. 投递漏斗：各状态数量和占比。
4. 每个岗位面试通过率。
5. 重复邮箱或疑似重复候选人检查。

### 验收

- 查询能在测试数据下返回正确结果。
- 使用窗口函数解决最近记录问题。
- 聚合结果包含 0 值岗位，而不是只显示有投递岗位。

## 阶段 4：并发和事务

### 任务

实现 Offer 发放逻辑：

- 同一投递只能有一个有效 Offer。
- 发放 Offer 时更新投递状态为 `offered`。
- 如果并发重复发放，只能成功一次。

参考结构：

```sql
BEGIN;

SELECT id
FROM hr.application
WHERE id = 100
FOR UPDATE;

INSERT INTO hr.offer(application_id, status, offered_at)
VALUES (100, 'active', now());

UPDATE hr.application
SET status = 'offered', updated_at = now()
WHERE id = 100;

COMMIT;
```

### 验收

- 两个会话并发执行时，不会产生重复有效 Offer。
- 出错时事务能完整回滚。
- 能解释锁等待发生在哪里。

## 阶段 5：索引和性能

### 任务

为以下查询设计索引：

- 按邮箱查候选人。
- 按岗位和状态查询投递，按创建时间倒序分页。
- 查询开放岗位。
- 查询候选人的最近面试。
- 查询 `jsonb` profile 中包含某技能的候选人。

### 验收

- 每个索引都有对应查询和 `EXPLAIN (ANALYZE, BUFFERS)` 证据。
- 删除无用索引并说明原因。
- 能解释复合索引列顺序。

## 阶段 6：备份恢复

### 任务

1. 使用 `pg_dump -Fc` 备份 `job_app`。
2. 恢复到 `job_app_restore`。
3. 配置一次物理基础备份。
4. 设计 PITR 演练步骤。

### 验收

- 恢复库中关键表数据量一致。
- 恢复后只读账号权限可用。
- 能说明逻辑备份和 PITR 的差异。

## 阶段 7：监控和排障

### 任务

- 开启慢查询日志。
- 安装 `pg_stat_statements`。
- 制造一个慢查询并优化。
- 制造锁等待并找出阻塞源。
- 查看 autovacuum 统计信息。

### 验收

- 能提交一份排障记录：现象、证据、根因、处理、复测结果。
- 能说明是否需要调整参数或索引。

## 综合自测问题

1. 为什么 `pg_dump` 不能替代 PITR？
2. 为什么 `SELECT count(*) FROM big_table` 可能很慢？
3. 为什么索引没有被使用？
4. 为什么长事务会导致表膨胀？
5. 为什么只 GRANT 表权限还无法查询？
6. 为什么逻辑复制不能简单当作完整高可用？
7. PostgreSQL 13 EOL 对生产有什么影响？

## 重点

- 项目练习必须覆盖正确性、性能和可恢复性。
- 所有优化都要有执行计划证据。
- 所有备份都要有恢复验证。

## 难点

- 综合项目会把多个知识点合在一起。例如并发发 Offer 同时涉及唯一约束、事务、行锁、错误处理和业务状态机。

## 易错

> **易错：** 只完成建表和查询，不做并发、备份、权限和排障。
>
> 正确做法：生产数据库能力必须覆盖数据正确性、性能、恢复和安全。
